2. 本篇 vs 相关论文的 delta

2.1 PrfaaS vs Kimi Linear [2510.26692]

Kimi Linear 贡献一个 27-layer (20 KDA + 7 MLA) 混合 attention 架构, 核心是 KDA (Kimi Delta Attention) 加上 DPLR 约束、ShortConv、NoPE-on-MLA 等；48B MoE (3B active) in-model 达到 75% KV cache reduction.

  • What's new in PrfaaS: 把 Kimi Linear 的 model-side KV reduction 转化为 system-side 部署自由度。Kimi Linear 把 KV 吞吐降了, 但如果继续用传统 PD 部署, 省下来的带宽只是让 单集群内 prefill↔decode transfer 更轻松——并没改变 "prefill 和 decode 绑死同一 RDMA 域" 的基本假设。PrfaaS 把 model side 的 10× 降低视为 "新 regime 开启", 在其上构建跨 DC 部署这个前者不支持的新架构层。
  • What's incremental: 两篇都在 Moonshot AI 的 Mooncake-based KV-centric serving 架构延长线上。
  • What this paper did NOT change: Kimi Linear 的 model architecture；PrfaaS 的 case-study model 是 1T hybrid following Kimi Linear, 不改 layer 结构只改部署。
  • Paradigm shift: Kimi Linear = "make KV smaller in-model"；PrfaaS = "use the smaller KV to move prefill out-of-cluster"。两篇的关系是 enabler → exploiter。

2.2 PrfaaS vs DynaServe [2504.09285]

DynaServe 提出 micro-request abstraction: 单个请求可在任意 token boundary 被切分, 每片段独立调度, chunk-based KV 传输与计算 overlap。Global scheduler + local scheduler 两级。

  • What's new in PrfaaS: 物理边界从 intra-cluster, intra-node 扩展到 inter-cluster, inter-datacenter。DynaServe 的 KV 传输假设是 intra-node RDMA (100s Gbps); PrfaaS 明确处理 commodity Ethernet (~100 Gbps total, 13 Gbps used) 的带宽、拥塞、突发性。
  • What's incremental: 调度粒度选择（分片 vs 请求级别阈值）: DynaServe 在任意 token 边界切；PrfaaS 在请求级别按长度阈值 t 路由 (l > t → PrfaaS)。PrfaaS 的粒度更粗但换来跨 DC 传输的可行性。
  • What contradicts: 不直接矛盾，但应用场景假设不同——DynaServe 假设多路径节点都能相互发送 KV, PrfaaS 假设 "外部 PrfaaS 集群只送给本地 PD 集群, 不反向"（单向 producer）。PrfaaS 的 stateless producer 假设简化了调度但限制了可能的资源复用。
  • Paradigm shift: DynaServe = "slice requests to fit scheduler"；PrfaaS = "slice deployments to fit network boundary"。

2.3 PrfaaS vs DualPath [2602.21548]

DualPath 在 PD-disaggregated inference 中利用decode engine 空闲的 storage NIC 做双路径 KV 加载, 减缓 prefill 端 I/O 瓶颈。

  • What's new in PrfaaS: 瓶颈假设不同——DualPath 解决 storage-side IO (KV 从 cache server 到 prefill engine 的 NIC 带宽), PrfaaS 解决 network-side BW (KV 从 prefill 到 decode 跨集群的 egress 带宽)。同一个 KV 传输问题的两端。
  • What's complementary: 两种技术可叠加——DualPath 优化 prefill 端的 IO, PrfaaS 优化 prefill→decode 的跨集群。理论上 PrfaaS 集群内可用 DualPath 进一步加速 KV 从 cache 到 prefill node 的加载。
  • Contradictory assumption: DualPath 假设 decode NICs 有空闲带宽 (bs=1–8 场景)；PrfaaS 的主要 case study 也在中小 batch (Λ_max 3.24 req/s, 32+64 GPU), regime 类似。
  • Paradigm shift: DualPath = "underutilized hardware (decode NIC) reassigned"；PrfaaS = "underutilized location (compute-dense cluster at different DC) reassigned"。

2.4 PrfaaS vs fabric-lib [2510.27656]

fabric-lib 是 user-space RDMA library for LLM inference, zero-copy p2p, 比 NCCL 在 KV-cache / tensor 传输上快 3.6×。

  • What's new in PrfaaS: fabric-lib 是基础 transport, PrfaaS 是调度策略; 两者可以 compose——PrfaaS 的 intra-cluster 800 Gbps RDMA 路径完全可以用 fabric-lib 实现, 也能作为 PrfaaS 下一版的"准 RDMA-over-Ethernet" candidate。
  • What's complementary, not competitive: 典型的 infrastructure layer (fabric-lib) vs orchestration layer (PrfaaS) 关系。
  • Paper 未提及的实战好处: 用 fabric-lib 做 PrfaaS 的 inter-cluster Ethernet transport, 3.6× gain 可能把 PrfaaS 的 13 Gbps egress 降到 <4 Gbps, 给更大规模 PrfaaS 集群更多 BW headroom。

2.5 PrfaaS vs Sutradhara [2601.12967]

Sutradhara 是 orchestrator-engine co-design, 5 thin API (submit_partial_prefill / extend_prefill / register_streaming_callback / tag_kv_blocks / set_reuse_priority) 让 orchestrator 向 engine 传递 semantic info, 实现 prompt splitting (prefill-tool overlap) + streaming dispatch + workload-aware KV mgmt。

  • What's new in PrfaaS: 协同面不同——Sutradhara 是 single-cluster orchestrator↔engine 协同, 传递 semantic info (哪个 token 是 tool output, 哪部分 KV 值得 pin 住); PrfaaS 是 multi-cluster scheduler ↔ multiple engines 协同, 传递 physical info (哪个集群有什么 prefix cache, 哪个链路多忙)。前者是 "告诉 engine 更多上下文", 后者是 "告诉 scheduler 更多网络状态"。
  • Paradigm overlap: 两篇都是 "把原本隐式的协同显式化为 API / scheduling policy" 的范式——Sutradhara 把 tool orchestration 从 "opaque retry loop" 变成 "5 tagged APIs"; PrfaaS 把 heterogeneous hardware deployment 从 "force-colocate" 变成 "explicit cluster boundary"。
  • Contradiction 点: 无直接矛盾。
  • Potential composition: Sutradhara 的 tag_kv_blocks + set_reuse_priority 可以扩展, 多打一个 "cross-cluster-worthy" 标签, 帮 PrfaaS scheduler 判断哪些 KV 值得跨集群 cache transfer。

2.6 PrfaaS vs Fleet [2604.15379]

Fleet 是 intra-GPU 优化: 在 CUDA/HIP wavefront/workgroup/grid 三层之上加 Chiplet-task, 通过持久化 megakernel + 每 XCD scheduler + M-major 协作 + 两级事件同步, 把 AMD MI350 八颗 XCD 的私有 L2 变成协作复用, Qwen3-8B decode bs=1-16 比 vLLM 快 1.2-1.6×。

  • What's new in PrfaaS: 相反的层级。Fleet 做 intra-GPU(on-chip)决定 L2 的 chiplet 粒度; PrfaaS 做 inter-cluster(off-node) 决定 compute/BW 的 datacenter 粒度。一个往下游钻(L2), 一个往上游走(DC)。
  • Decode vs prefill 镜像: Fleet 明确针对 decode（memory-BW-bound, small batch）；PrfaaS 明确针对 prefill（compute-bound, long context）。两者互补——理想部署是 Fleet 在 decode 端最大化 L2 reuse, PrfaaS 在 prefill 端跨集群最大化 compute density。
  • Paradigm shift 对比: Fleet 把 "隐式的 chiplet L2 约束" → "显式的 Chiplet-task 资源"; PrfaaS 把 "隐式的跨 DC 部署禁忌" → "显式的 selective cluster-boundary routing"。两者都是 "隐式硬件/基础设施约束 → 显式软件可调度资源" 的经典范式样本。
  • Potential composition: PrfaaS 集群里跑 Fleet 做 prefill 的 GEMM, PrfaaS 的 decode 集群继续用 Fleet 做 decode。两者可以一起落地。

3. 可攻击面 (Attack surface — grounded in library evidence)

Notes §6 论证链 的 5 步, 全部 load-bearing。针对每个 load-bearing step 做 adversarial 检查：

3.1 Adversarial rebuttal — library 里的反例检查

Step 1: "Dense $\Phi_{\text{kv}}$ 高 → 传统 PD 的 $B_{\text{out}}$ 超商品 Ethernet"

  • Adversarial from Fleet [2604.15379]: Fleet 的 Qwen3-8B decode 实验里, bs=1-16 的 KV 吞吐并不大（Qwen3-8B 本身参数少 + decode 阶段 KV 不重算）, per-GPU KV 吞吐也远没到 32K-token prefill 的 60 Gbps。Step 1 的攻击面: 论文 §2.1 用的是 32K prefill 阶段 的 benchmark 数字, 不是 serving 稳态下所有请求的平均。实际 serving 中大部分请求是 prefix-cache hit 的短增量, dense 模型在 serving 平均值 上的 $\Phi_{\text{kv}}$ 可能远低于 60 Gbps 的 worst-case。
  • Evidence strength: 中等。这个 rebuttal 只是质疑"数值取了最坏值", 不否认 "dense 模型长上下文 prefill 有 BW 压力" 的事实。
  • Paper 的 pre-empt 程度: Table 3 给了 1K/8K/32K/128K 四个长度, 没给 serving average。部分 pre-empt, 仍有攻击面。

Step 2: "Hybrid attention 降 $\Phi_{\text{kv}}$ 4-13× → cross-DC plausible"

  • Adversarial from Kimi Linear [2510.26692]: Kimi Linear 声称"首个在公平对比下全面超越 Full Attention 的混合线性注意力", 但 KDA 在 100B+ scale 的质量优势 still open question (Kimi Linear open_q 第 1 条). 如果 hybrid attention 在更大 scale 质量下降, PrfaaS 的 "新 regime 已经稳定" 前提就脆弱。
  • Term gate check: PrfaaS 所说的 "hybrid attention" 是否和库内"hybrid"术语一致？—— 是的, Table 1 列出 4 篇 hybrid (Kimi Linear/MiMo-V2-Flash/Qwen3.5-397B/Ring-2.5-1T) 都属 linear-complexity block + full attention block 的混合。术语无歧义。
  • Plausibility gate check: 2026 年 (paper 发表年) 主流 frontier model 里 hybrid attention 占比? 列 4 个已确认 (前述) + Nemotron 3 Nano (NVIDIA), 大约 5 个主要参与者。vs GQA-only 阵营 (MiniMax-M2.5, Qwen3-235B, GPT-OSS 部分) 也是 5 个左右。Hybrid 约占 frontier 阵营一半, plausible 通过, 但不是 overwhelming majority——"新 regime 稳定"是 short-term 假设, 1-2 年窗口安全, 远景不确定。

Step 3: "Naive 全 offload 失败 → 必须选择性 offload"

  • Adversarial from DynaServe [2504.09285]: DynaServe 的 micro-request abstraction 就是 "任意 boundary 切分, 不是全拆也不是不拆"的一种更 fine-grained 的选择性. PrfaaS 的长度阈值 $t$ 是 request-level 粗粒度选择, 而 DynaServe 证明 token-level 切分可进一步提升 goodput。如果把 DynaServe 的 micro-request 用到 cross-DC setting, 一个长请求可以前半在本地 PD 做 prefill、后半跨 DC 到 PrfaaS——超过了 PrfaaS 当前 0/1 路由的能力。
  • Evidence strength: 高。DynaServe 没做跨 DC, 但把它的思想搬过来会给 PrfaaS 一个更优的版本。
  • Implication: PrfaaS 的 "整个请求要么全 local, 要么全 PrfaaS" 是一个可放松的约束；这是 Stage 5 未探索方向的一个线索。

Step 4: "Eq.3-8 + grid search 解出 $t, N_p/N_d$ 唯一最优"

  • Adversarial from preread S-1 (128K 时 $\Phi_{\text{kv}}$ 反而下降): paper §3.4.2 声称 "$\Theta_{\text{prfaas}}/p$ decreases monotonically with $p$", 但这个 monotonicity 假设 $\Phi_{\text{kv}}$ 随长度单调上升。Table 3 的 128K 行数据显示 dense 模型 $\Phi_{\text{kv}}$ 反而下降 (59.93 → 47.82 Gbps), 这是非单调行为。如果这个非单调性存在于 case study 模型的更长 regime, grid search 可能找到 local 而非 global optimum。
  • Evidence strength: 中等。Paper 的 case study 最长 128K，Table 5 只给到 128K, 非单调未在 case study 被检测到。但对想扩展到 256K+ 的部署者, 这是个 warning。
  • Attack on monotonicity argument: "two monotone curves unique intersection" 依赖两条曲线严格单调；如果 $\Phi_{\text{kv}}$ 有反转, $\Theta_{\text{prfaas}}$ 的下降不再严格, intersection 可能多解。

Step 5: "最优配置达 3.24 req/s / 13 Gbps → +54% 吞吐"

  • Adversarial from preread S-6 (Naive Het 的 Mean TTFT 1.74s 胜过 PrfaaS-PD 的 2.22s): Mean TTFT 是用户感知最强的指标, Naive Het 的赢不是小 margin (22%)。paper 不解释这个 22% 来自哪里 — 可能是 cache-aware routing 的 CPU 成本、scheduler 决策延迟、或短请求队列效应。如果部署关心 Mean TTFT 而非 P90, PrfaaS-PD 可能反而弱于朴素方案。
  • Evidence strength: 高。数据在 paper 自己的 Table 6。
  • Paper 处理方式: 无。只强调 P90 和 throughput。

3.2 Ecosystem reality check

对"hybrid attention 持续主流" 的 counterfactual 做 2-gate 检查 (per paper-synthesis §3b):

1. Term gate: paper 的 hybrid = linear/SWA + full-attn 混合, 对应库内 Kimi Linear / MiMo / Qwen3.5 / Ring 共 4 篇, 术语清晰。

2. Plausibility gate: production 部署? 已知: Moonshot 的 K2.6 (kimi-k2-6.md) 即 Kimi Linear 后代, 部署上线; Mistral 的 MiMo 也已发布。但主流 SaaS (OpenAI, Anthropic, Gemini) 仍未公开采用 hybrid。计: 2-3 production deployments。刚好过 plausibility gate (≥2), 不牢固。

结论: "若 hybrid attention 不主流, PrfaaS 失效" 的 counterfactual 非 straw-man, 是真实风险。1-2 年窗口内 bet 安全, 3 年以上窗口需看 Claude/GPT-5 是否 follow hybrid。

3.3 根本性 vs 缓解性 审视 (Root-cause vs mitigation)

PrfaaS 的 54% 吞吐 / 64% P90 TTFT 提升:

  • Regime: conditional — 需要同时满足：

1. Workload 的长度分布是重尾 (log-normal $\sigma=1$ 这样) → 有足够"长请求"值得 offload

2. 跨 DC BW 充足 (100 Gbps class, 利用率 ~13%)

3. Prefill/decode 计算密度差异足够大 (H200 vs H20 的 prefill-compute ratio)

4. Agentic workload 占主导 (prefix cache hit 常见, 增量长度决定路由)

  • Regime 失效时会发生什么:
  • 如果是纯短对话 workload (μ<8K): 大部分请求走本地 PD, PrfaaS 集群空闲, 投入的 H200 成本无回报
  • 如果跨 DC BW 紧张 (<30 Gbps): Eq.3 的 BW-bound 分支成为瓶颈, 等比例削减提升
  • 如果 H200 和 PD 机器算力差距小: Paper §4.4 自己承认等成本下只 +15%, 这是 "compute density delta 小 → 边际收益小" 的直接反映
  • 是否根本性改进: 缓解性, 非根本性。"Prefill 长、decode 带宽" 的本质张力未解, 只是让一侧跑在更适合的硬件上。根本性的改进会是 "用单一硬件同时适合两个阶段" 或 "架构上消除 prefill-decode 分离需求", 都需要硬件/架构层面变革。

3.4 对立观点主动搜索 (Adversarial applicability check)

Notes §7.0 核心壁垒是 layer-wise prefill pipelining; 对应 binding 是 "existing vLLM / SGLang kv_transfer 都不支持 layer-wise streaming"。

对立尝试: 是否存在一种场景, 反而让 非 layer-wise(即 full-prefill-then-send)更适合？

  • 场景 candidate: 极短 prompt + 极长 output (e.g. "讲个长故事" 风格的 agent 任务)。这类请求 prefill 快、KV 小、decode 长; 跨 DC KV 传输时间可以被 decode 完全覆盖; layer-wise streaming 的复杂度 unnecessary。
  • Finding: 有效对立——短 prefill 场景下 layer-wise pipelining 是 over-engineering。但此类场景又不触发 PrfaaS(因为 $l \leq t$, 本地 PD 处理)。所以对立不构成对 PrfaaS 核心设计的反驳, 而是揭示"layer-wise pipelining 只在 long-prefill 子集上 load-bearing"。
  • 结论: binding 成立, 但scope 比声称的窄——只对 PrfaaS 集群的长请求路径是核心壁垒。

Notes §7.4 关键实现细节第 2 条 (PD 集群内同构才能 P↔D swap) 的 adversarial:

  • 场景 candidate: PD 集群内如果异构 (e.g. 一半 H20 一半 H100 老卡), 虽失去 P↔D swap 灵活性, 但换来 更强的 per-request heterogeneity-awareness —— 长请求可以选 H100, 短请求选 H20, 细粒度 BYO。
  • Finding: 这种细粒度方案需要 scheduler 有 per-request hardware-selection 能力, 超出 PrfaaS §3.4 scope。属"未探索方向", 记入 §5。

4. 生态位 (Ecological niche)

4.1 Paradigm-shift positioning

PrfaaS 的 paradigm shift: 把 "跨数据中心部署是禁忌" 这个隐式基础设施约束 转化为 "selective offload with bandwidth-aware routing" 的显式软件调度资源。

对照库内的其他 "隐式约束 → 显式资源" 范式:

Paper隐式约束被消解显式软件资源
PrfaaS [本篇]跨 DC 部署被 KV BW 禁止t 阈值选择性 offload + dual-timescale scheduler
Fleet [2604.15379]多 chiplet L2 物理割裂但 CUDA 隐藏Chiplet-task 抽象 + 每 XCD software scheduler
Kimi Linear [2510.26692]Full attention O(N²) KV 绑死部署混合 layer (linear:full=N:1) 可调比例
DynaServe [2504.09285]请求 atomic 单位被 scheduler 假设micro-request 任意 boundary 切分
Sutradhara [2601.12967]Orchestrator-engine 信息不通5 thin APIs 传递 semantic info
PagedAttention (vLLM)KV cache 必须连续分配page-based explicit KV

trend 观察: 整个库里 6-7 篇顶级 serving/kernel paper 做同样的 paradigm shift。这不是巧合——这代表 2024-2026 年整个 LLM 系统研究的主旋律: "把之前被当成硬件/API 的背景约束, 一条条挖出来变成 scheduler 能感知的第一类软件资源"。PrfaaS 是这条线在 datacenter 尺度 的最新实例。

可推广性: "隐式约束 → 显式资源" 范式是否可继续往更大尺度？

  • 下一层: 多 region / 跨 availability zone? 当前 paper §2.3 说 "loosely coupled clusters, datacenters, or regions", 但具体 regions 间网络复杂度远高于 datacenter VPC (延迟、带宽抖动、成本模型全不同)。PrfaaS 若向 region-level 拓展, dual-timescale 可能需要加第三层(跨 region 再平衡)。
  • 下下层: edge + cloud 混合部署? 极端长尾: edge 做 decode (user latency), cloud 做 prefill。此时 KV 回流到 edge 的 BW 成本更严, selective threshold 的 cutoff 会更高。还没 paper 研究。

4.2 Adoption evidence (cross-paper and cross-framework)

PrfaaS paper 明确提及的 ecosystem partners (§1 Para 1 + referencing notes §7):

  • vLLM [29, §1, Table 5]: case study 使用内部 vLLM fork; §3.2 hybrid prefix pool 基于 vLLM team @ IBM 的 hybrid KV manager blog [28]
  • SGLang [7, Table 3 caption]: Table 3 所有 $\Phi_{\text{kv}}$ 测量使用 SGLang v0.5.9
  • NVIDIA Dynamo [20, §1]: paper §1 Para 1 明确标为 ecosystem 协作方

库内其他 paper 对 PrfaaS 的前向引用: 库中 PrfaaS 是 2026-04 papers 之一, 目前无库内 paper 明确 build on PrfaaS（时间上还太早）。

间接引用 via Mooncake: Mooncake (PrfaaS 的直系前驱, ref [22]) 被库内多篇 paper 引用:

  • Justitia [2510.17015] 明确 cite Mooncake
  • DualPath [2602.21548] 基于 Mooncake 的 KV-centric 架构

生产部署声明 (paper §4.1 + §5): paper 用 Moonshot AI internal 1T model 做 case study, 但未明示外部部署。Moonshot 自己的 Kimi 产品 (K2.5 / K2.6) 已经部署 Kimi Linear 架构, 理论上可以用 PrfaaS 架构部署——paper 没披露是否已上线生产。

4.3 Design bindings with cross-paper grounding

PrfaaS 对下游部署 FORCE 的约束:

1. 强绑定: hybrid attention model。Dense-only 模型 (MiniMax-M2.5 / Qwen3-235B) 用 PrfaaS 直接崩（BW demand 3.8 / 2.1 Tbps 不可行）。

  • Library evidence: Kimi Linear [2510.26692] 说 3:1 hybrid ratio 已经成立; 但 Ring-2.5-1T 7:1 更激进, 质量数据未独立验证。
  • Relaxation path: 无 relaxation——这是 PrfaaS 存在的根本条件。Dense 路线要用其他方案（如 DualPath 的 storage NIC 方案）。

2. 绑定: PD 集群内部同构（notes §7.4 第 2 条）。

  • Library evidence: Helix [2510.xxx, §6 ref] 明确做 heterogeneous GPU within cluster 的 max-flow scheduling; 和 PrfaaS 的约束直接对立。
  • Relaxation path: 在 PD 集群内同样用 PrfaaS-style 的 "长度阈值路由" 继续切分: cluster 内的 H100 做长 prefill (PD-P-long), H20 做短 prefill (PD-P-short) + decode (PD-D)。paper 没探讨 intra-cluster multi-tier。

3. 绑定: agentic workload 占主导 + prefix cache 命中多（§3.3 Para 2）。

  • Library evidence: 库内 7+ 篇 agentic serving paper (Sutradhara / KVFlow / Justitia / ThunderAgent / TokenDance / Scepsy / Concur) 都默认这个前提——和 PrfaaS 一致。这个 binding 是"当下主流"而非"普适"。
  • Relaxation path: 如果未来 agentic 退热、短对话回潮, PrfaaS 价值降低。但库内所有最近 framework 论文都不担心这个, 反映整个 serving 子领域的共同 bet。

5. 未探索方向 (Unexplored directions)

5.1 Trade-off 轴识别

Notes §5.5 Table 6 里的失败数据点: Naive Het PD 的 Mean TTFT (1.74s) < PrfaaS-PD (2.22s), 22%。

  • 这是一根 trade-off 轴还是"没调优"？trade-off 轴。Paper 没说但现象指向: "长请求尾延迟优化" vs "短请求平均延迟" 是根本互斥——PrfaaS-PD 用 cache-aware routing + 调度决策换来了 P90 的改进, 但给每个请求 (包括短请求) 的调度路径加了固定开销。naive het 的决策路径短, 短请求从 routing 决策里得不到好处, 但也避免开销。
  • 轴的名字: "scheduler-aware routing overhead vs tail-latency gain"。
  • 作者未探讨: paper 只讨论 "PrfaaS-PD 全面胜出"（虽然 Table 6 的数字不支持 Mean TTFT 这一点）。

另一个 trade-off 轴 (从 preread S-7): headline throughput gain vs equal-cost gain。54% vs 15% 的 3.6× 差距是 H200 价格溢价的直接代价。

  • 轴的名字: "hardware specialization vs deployment cost"。
  • Library 里的对比: Fleet [2604.15379] 声称 1.2-1.6× decode gain 用的是同一套 AMD MI350 硬件, 没有硬件成本溢价; PrfaaS 的 +54% 需要 H200 (~3× H20 价格)。

5.2 Hybrid / adaptive 方向 (paper 未探索)

基于 §5.1 的两根 trade-off 轴, 提出几个 adaptive 方向:

5.2.1 Adaptive routing overhead

  • 场景: 在 workload 短请求为主时, 动态关闭 cache-aware routing, 降到纯长度阈值, 减少短请求的调度开销。
  • Inspiration: DynaServe [2504.09285] 的 local scheduler 就可以 plug-in。
  • Open research: 最优 "关闭 cache-awareness" 的 workload 特征阈值是多少？

5.2.2 Fine-grained offload (DynaServe-style)

  • 场景: 借 DynaServe 的 micro-request abstraction, 一个长请求的前 N% prefill 在本地, 后 100-N% 跨 DC 到 PrfaaS, 避免 all-or-nothing 的路由二值选择。
  • Why PrfaaS 现在没做: §3.4.1 为 tractability 假设每个请求整体路由 (fraction $p$ 的请求 all go to PrfaaS)。破这个假设需要重做 throughput model。
  • Open research: 跨 DC 续 prefill 的 state migration 成本是否小于节省的 BW?

5.2.3 Intra-cluster heterogeneity + inter-cluster selective offload 组合

  • 场景: 本地 PD 集群内部再分 tier (H100 做长请求 prefill, H20 做短 prefill + decode), 外加 PrfaaS 做跨 DC 选择性 offload。双层 tiered heterogeneity。
  • Inspiration: Helix 的 intra-cluster heterogeneous + PrfaaS 的 inter-cluster scheme。
  • Open research: 两层 scheduler 如何协同？级联 Eq.7/Eq.8 为二级？

5.2.4 Storage-NIC fallback (DualPath + PrfaaS 混合)

  • 场景: PrfaaS 集群内部用 DualPath 做 prefill-engine ↔ KV-cache-server 的存储旁路。
  • Open research: PrfaaS cluster 的 KV cache server 层是否值得单独 scale out?

5.3 Cross-paper open questions

Open questions 汇总:

  • From Kimi Linear: KDA 100B+ scale 的质量 (open_q 1)——这个问题如果 PrfaaS 部署 100B+ 模型就成为 critical 依赖, 直接影响 PrfaaS 的 era positioning。
  • From DynaServe: "micro-request with MoE expert imbalance" (open_q 3)——同样的问题在 PrfaaS 的 cross-DC setting 会放大 (cross-DC MoE expert 路由的通信成本远高于 intra-cluster)。
  • From Fleet: "MoE Chiplet-task 如何适配" (Fleet open_q 2)——如果 PrfaaS 部署的是 MoE 模型 (大部分 hybrid frontier model 是 MoE: Ring-2.5-1T、MiMo-V2-Flash、Qwen3.5), prefill 阶段的 expert 路由和 PrfaaS 的 cross-DC 传输在 MoE 下如何交互?
  • 新增 (来自 PrfaaS × 库组合):
  • 层间 prefill pipelining 是否可和 Fleet Chiplet-task 组合: Fleet 要求 persistent megakernel 占住 SMs, layer-wise pipelining 要求中途把 KV 发出去。两者对 CU/SM 使用的竞争如何裁决?
  • 跨 region 扩展时需要第三时间尺度调度: 短期 (reactive per-request) + 中期 (周期 P↔D 角色转换) + 长期 (跨 region rebalance, e.g. 日间亚洲、夜间美洲各补充 PrfaaS 容量)?

5.4 Era positioning (时代定位)

1. Low-hanging fruit 还是深水区？: 深水区。前驱 Mooncake (2024) 已经把 single-cluster KV-centric disaggregation 的 low-hanging fruit 采走; DistServe / Splitwise (2024) 探索了 goodput 和 cost 的单 DC 视角; DynaServe (2025-04) 进一步 fine-grained。PrfaaS 是第一篇跨 DC 的系统性工作 (注: Mooncake 也暗示过跨 region 但未 formalize)。

2. 哪些 predecessors 被 supersede/extend？: extend Mooncake 的 KV-first-class 范式到跨 DC; supersede naive heterogeneous (通过 Table 6 的 Naive Het PD 对照)。

3. 新 regime 还是老 regime?: 新 regime —— hybrid attention (2025+) 刚 enable 的。预计 1-2 年窗口内 2-3 篇后续工作跟进 (cross-region scheduling、PrfaaS + DualPath/DynaServe 组合、MoE-aware PrfaaS 等)。窗口满后会转入调优/综合期。
