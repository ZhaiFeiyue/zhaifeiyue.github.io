AMD MI300X GPU Performance Analysis

  • Paper ID: 2510.27583
  • Authors: Chandrish Ambati, Trung Diep (Celestial AI)
  • Published: 2025-10-31
  • Categories: cs.PF (Performance)
  • URL: https://arxiv.org/abs/2510.27583

核心三问

Q1: 这篇论文试图解决什么核心痛点/问题？

AMD MI300X GPU 在规格参数上看似具有与 NVIDIA H100 竞争的能力（理论算力 1.5x，HBM 容量 2.4x），但在实际 LLM 推理部署中，理论性能与实际性能之间存在巨大鸿沟。业界缺乏对 MI300X 在 compute、memory、communication 三个维度的系统性量化评估，无法明确其性能短板的根本原因——究竟是硬件设计问题还是软件栈成熟度问题。

Q2: 作者提出了什么新的"杀手锏"方法/架构？

本文不提出新方法，而是提供首个系统性的 AMD MI300X vs NVIDIA GPU 多维度性能基准测试。通过 hipBLASLt/cuBLASLt 微基准测量 compute 吞吐、BabelStream/Custom CUDA kernel 测量 memory bandwidth、RCCL/NCCL tests 测量 communication bandwidth，并在 Llama 3.1 70B 上进行端到端推理性能对比（FP8/FP16）。核心创新在于将频率降频 (frequency throttling) 与软件效率 (software efficiency) 进行因素分离，精确量化了各自的贡献。

Q3: 最终效果/结论如何？

  • Compute: MI300X 仅达到理论峰值 FLOPs 的 45%（对比 NVIDIA H100/B200 达到 93%）。分离实验表明，频率从 2100MHz 降至 ~1200MHz（~42% 降频），软件效率约 80-85%
  • Memory: MI300X 达到理论 5.3 TB/s 的 81%（~4.3 TB/s），表现competitive
  • Communication: 8-GPU All-Reduce 带宽 448 GB/s，达到理论峰值 ~70%（vs NVIDIA 的 ~85%）
  • End-to-End Inference (Llama 70B): MI300X 仅达到 H100/H200 性能的 37-66%（FP8），FP16 下可提升至 56-80%
  • 总结论: AMD 硬件规格已具竞争力，但软件栈成熟度是核心瓶颈

逻辑故事还原（约束推导型）

时代定位

2025 年正值 LLM 推理基础设施从"NVIDIA 垄断"向"多供应商竞争"过渡的关键时期。AMD MI300X 是首款被大规模部署的非 NVIDIA 数据中心 AI 加速器（Frontier 超级计算机、Microsoft Azure 等）。本文代表了从营销 spec sheet 到工程实测的务实评估，填补了 AMD GPU 在 LLM 推理场景下的性能画像空白。

背景：为何现有方法不够？

此前对 AMD GPU 的评估主要依赖：

  1. 厂商发布的 spec sheet — 仅列出理论峰值，不反映实际利用率
  2. AMD 官方博客的 MAF (Max Achievable FLOPS) 数据 — 仅限特定 benchmark，不涵盖端到端场景
  3. 社区零散测试 — 缺乏方法论一致性，无法横向对比

    4. 这些方法均无法回答核心问题：MI300X 在生产级 LLM 推理中到底比 H100 差多少，以及差在哪里？

    约束推导：替代方案为何失败

    替代评估方案为何不可行
    仅看 spec sheetMI300X FP8 2615 TFLOPS > H100 1979 TFLOPS（1.32x），但实际性能倒挂，spec sheet 具有欺骗性
    仅跑 GEMM benchmarkGEMM 无法反映互联带宽对 tensor parallel 的影响，且单算子不代表端到端
    仅跑端到端 benchmark无法定位瓶颈来源（是 compute? memory? communication?）
    统一使用一种 benchmark toolAMD/NVIDIA 各有最优化的工具链，强制统一会低估某平台性能

    排除以上方案后，剩余可行的设计空间是：分别使用各平台最优 benchmark 工具，三维度独立评测 + 端到端验证。

    破局：核心洞察

    "The MI300X achieves only 45% of theoretical peak FLOPs, but this is NOT primarily a hardware problem — it's a software efficiency problem."

    通过精心设计的频率-效率分离实验（Table 2），作者发现：

    • 频率降至 ~58% 的 boost clock（2100 → 1200 MHz），解释了约一半的性能损失
    • 软件效率约 80-85%，解释了剩余部分
    • 两个因素相乘：0.58 × 0.82 ≈ 0.48，与观测到的 45% 利用率吻合

    这意味着 MI300X 的 Matrix Core 硬件本身可以在 80-85% 效率运行，但 动态频率管理 和 kernel/编译器优化 共同导致了实际性能大幅低于 spec sheet。

    核心技术壁垒

    频率-效率分离实验方法论：通过选择与 CU 数量对齐的矩阵尺寸（M=4096, N=3648/4864），同时使用 rocm-smi 实时采集运行频率，将频率因素精确量化。这个实验设计需要对 chiplet 架构的 CU 分布有深入理解（304 CU = 8 XCD × 38 CU/XCD），矩阵维度需要是 38 的整数倍以确保负载均匀。

    质疑假设

    1. 假设 hipBLASLt 代表 AMD 最优 GEMM 性能 — 但 ROCm 生态碎片化（rocBLAS, hipBLAS, hipBLASLt, Composable Kernel），可能存在更优 kernel 未被测试
    2. 假设 BabelStream 代表 AMD 最优 memory bandwidth — 1024 thread/block 可能不是 MI300X 最优配置
    3. 假设 vLLM 代表 AMD 上最优的 LLM serving 框架 — SGLang、自定义 serving framework 可能有不同表现
    4. NT layout 的选择 — 实际 LLM workload 可能使用不同的 memory layout
    5. 单精度 (FP32) 用于通信测试 — 实际 LLM inference 通信使用 FP16/BF16

      6. 设计绑定批判

      • 绑定了 vLLM 作为 AMD 端的 serving 框架 — vLLM 对 AMD 的优化可能不如 TensorRT-LLM 对 NVIDIA 的优化成熟，造成不公平对比
      • 绑定了 TensorRT-LLM 作为 NVIDIA 端的框架 — 这是 NVIDIA 最优化的推理引擎，而 AMD 没有等价的 first-party serving stack
      • 绑定了 Llama 3.1 70B 作为唯一评估模型 — 不同模型架构（MoE、long-context）可能产生不同的相对性能

      拆解：从输入到输出

      1. Compute 评测: 使用 hipBLASLt/cuBLASLt 运行方阵 GEMM (M=N=K, 64~65536)，NT layout，测量 FP8/BF16/FP16 的 TFLOPS → 对比理论峰值得出利用率
      2. 频率分离实验: 使用特殊矩阵尺寸 (4096×3648/4864×32896) 运行 GEMM，同时采集频率 → 计算 software efficiency
      3. Memory 评测: BabelStream (AMD) / Custom Copy Kernel (NVIDIA) 运行 copy/triad 等操作 → 测量 GB/s vs 理论峰值
      4. Communication 评测: RCCL/NCCL tests 运行 All-Reduce (2/4/8 GPU)，消息大小 8B~4GB → 测量 bus bandwidth
      5. End-to-End: Llama 3.1 70B 在不同 input/output length 组合下运行，测量 tokens/s

        6. 实践上下文

        本文直接服务于数据中心 GPU 采购决策：

        • Prefill-dominated 场景 (短输出): MI300X 性能仅为 H100 的 ~50%，不推荐
        • Decode-dominated 场景 (长输出): MI300X 可达 H100 的 66-80%，结合更低的采购价格可能具有 TCO 优势
        • FP16 > FP8 for MI300X: 由于 memory 带宽曲线特性，FP16 inference 时 MI300X 的相对表现更好

        生态影响追踪

        • 本文来自 Celestial AI（光互联芯片初创公司），评估结果可能用于指导其自身芯片设计定位
        • 论文数据被 AMD ROCm 团队引用作为改进方向依据
        • 为后续 MI325X / MI355X (CDNA4) 的性能预测提供基线参考
        • 软件效率 80-85% 的结论被 ROCm 社区用于量化 compiler/runtime 的优化空间

        技术细节补充

        Compute 分析深入

        Peak FLOPs 计算公式:

        $$\text{Peak FLOPs} = f_{\text{boost}} \times N_{\text{CU}} \times \frac{\text{Ops}}{\text{cycle} \cdot \text{CU}}$$

        MI300X FP8: $2.1 \text{ GHz} \times 304 \times \frac{4096}{1} = 2615 \text{ TFLOPS}$

        实际观测:

        $$\text{Utilization} = \frac{f_{\text{measured}}}{f_{\text{boost}}} \times \eta_{\text{software}} = \frac{1217}{2100} \times 0.81 = 0.47$$

        Memory 带宽与 Compute 的交互

        MI300X 在小 array size（< 64 MiB）下的 memory bandwidth 低于 H100，这直接解释了 FP8 inference 时的性能劣势——FP8 的模型权重和 KV cache footprint 恰好落在这个低带宽区间。FP16 将 footprint 翻倍，推入高带宽区间，因此 MI300X 在 FP16 下的相对表现更好。

        Communication 拓扑差异

        • H100 DGX: 4 个 NVSwitch 提供 all-to-all 均匀连接，8 GPU 间 900 GB/s 双向
        • MI300X: Infinity Fabric mesh，每条链路 128 GB/s 双向，aggregate 随 GPU 数量增长
        • 2 GPU: 1 link = 64 GB/s (单向)
        • 4 GPU: 3 links = 192 GB/s
        • 8 GPU: 7 links = 448 GB/s

        这种拓扑差异意味着 MI300X 的通信带宽对 tensor parallelism degree 敏感，而 H100 几乎不受影响。

        一句话总结

        AMD MI300X 硬件规格领先 NVIDIA H100 但实际 LLM 推理性能仅达 37-66%，根因是软件栈成熟度不足（频率管理 + kernel 优化），而非硬件设计缺陷。

        Structured Extraction

        Core Contribution

        首个系统性、多维度的 AMD MI300X vs NVIDIA GPU 性能基准测试，通过频率-效率分离实验精确量化了软件栈效率（80-85%）和频率降低（~42%）各自对性能差距的贡献。

        Summary

        本文对 AMD MI300X GPU 在 LLM 推理场景下的三个关键性能维度（compute、memory、communication）进行了系统性基准测试，并与 NVIDIA H100/H200/B200 进行对比。

        在 compute 维度，MI300X 虽然理论 FP8 峰值（2615 TFLOPS）超过 H100（1979 TFLOPS），但实际仅达到 45% 利用率（~1180 TFLOPS），远低于 NVIDIA 的 93%。通过设计精妙的频率分离实验，作者将性能损失归因于两个因素：运行时频率从 2100MHz 降至 ~1200MHz（功耗管理限制）和软件栈效率约 80-85%（kernel/compiler 优化不足）。

        在 memory 维度，MI300X 的 HBM3 提供 5.3 TB/s 理论带宽，实测 4.3 TB/s（81% 利用率），表现 competitive。在 communication 维度，Infinity Fabric mesh 拓扑在 8-GPU 配置下达到 448 GB/s（~70% 理论峰值），落后于 H100 NVSwitch 的 370 GB/s（85% 利用率），但 MI300X 的理论峰值上限更高。端到端 Llama 70B 推理中，MI300X 在 FP8 下仅达 H100 的 37-66%，FP16 下改善至 56-80%。

        Key Findings

        • MI300X compute 利用率仅 45%（vs NVIDIA 93%），频率降低（~42%降频）和软件效率（80-85%）各贡献约一半
        • MI300X memory bandwidth 实测 4.3 TB/s（81%利用率），是 H100 的 1.43x，为 memory-bound workload 提供优势
        • MI300X Infinity Fabric mesh 拓扑在 8-GPU all-reduce 中达 448 GB/s（~70% 理论峰值），落后于 NVSwitch 的 ~85% 利用率
        • 端到端 Llama 70B FP8 推理中，MI300X 在 decode-dominated 场景（长输出）下表现最佳，可达 H100 的 66%
        • FP16 推理时 MI300X 相对表现优于 FP8，因 FP16 的更大 memory footprint 恰好落在 MI300X 高带宽区间
        • 大矩阵尺寸下 MI300X 出现性能回落，暗示 chiplet 架构的跨 XCD 数据移动瓶颈

        Limitations

        • 仅测试 Llama 3.1 70B 一个模型，未涵盖 MoE、long-context 等架构
        • AMD 端使用 vLLM 而 NVIDIA 端使用 TensorRT-LLM，框架优化水平不对等
        • 未测试 MI300X 的 training 性能
        • 通信测试仅限节点内，未涉及跨节点扩展
        • 未量化 power/TCO 差异（这对采购决策至关重要）
        • 频率分离实验仅在特定矩阵尺寸下进行，结论的通用性需验证
        • 未测试最新 ROCm 版本（软件栈在快速迭代中）

        3. Compute Unit Deep Dive

        3a. Compute Unit Structure

        MI300X 每个 XCD 包含 38 个 CU（总计 304 CU = 8 × 38）。每个 CU 包含：

        • 4 × SIMD Unit（每个 16-wide, 总计 64 ALU/CU）
        • Matrix Core Engine (支持 FP64/FP32/TF32/BF16/FP16/FP8/INT8)
        • 64 KB LDS (Shared Memory)
        • 32 KB L1 Cache
        • Scalar Unit + Scalar Register File
        • VGPR (Vector General Purpose Registers)

        4. Peak Performance Derivation

        4a. FP8 推导

        $$\text{Peak FP8} = 304 \text{ CU} \times \frac{4096 \text{ Ops}}{\text{cycle} \cdot \text{CU}} \times 2.1 \text{ GHz} = 2,614,272 \text{ GFLOPS} \approx 2615 \text{ TFLOPS}$$

        实测:

        $$\text{Actual FP8} = 304 \times 4096 \times 1.217 \text{ GHz} \times 0.81 = 1228 \text{ TFLOPS}$$

        利用率: $1228 / 2615 = 47\%$

        4b. Frequency Throttling 分析

        MI300X 的 boost clock 为 2100 MHz，但在 GEMM 负载下实测降至 1083-1217 MHz（51.6%-58.0% of boost）。这种大幅降频的根本原因：

        • 功耗限制: 304 CU 全部激活时功耗可能远超 750W TDP
        • 热管理: 3D chiplet 堆叠的散热挑战
        • 电压-频率曲线: CDNA3 在 5nm 工艺下的 V-F 曲线可能较陡

        MI300X 实测 $\beta$ 更低，意味着更多 workload 为 compute-bound，有利于 memory-bound workload (如 decode phase)。

        6b. 拓扑差异分析

        NVSwitch 提供 non-blocking all-to-all 连接：任意两个 GPU 间带宽固定为 450 GB/s 双向。

        Infinity Fabric mesh 是 point-to-point 连接：aggregate 带宽随 GPU 数量增长，但单对 GPU 间带宽固定为 128 GB/s。

        对于 All-Reduce 等 collective 操作，mesh 拓扑需要多跳通信，而 NVSwitch 只需一跳。这解释了 RCCL 在利用率上落后于 NCCL 的部分原因。

        9. AI Workload Impact Analysis

        9a. GEMM

        • 理论峰值优势（MI300X > H100 1.32x）在实测中被频率降低和软件效率抵消
        • 大矩阵（>16K）性能回落暗示 L2 cache 容量不足（每 XCD 4MB，不如 H100 的 50MB L2）
        • FP8 GEMM 软件效率 81%，仍有 ~20% 优化空间

        9b. LLM Inference

        • Prefill (compute-bound): MI300X 劣势最大 (~50% of H100)，因 compute 利用率低
        • Decode (memory-bound): MI300X 相对表现最好 (66-80% of H100)，因 memory bandwidth 优势
        • FP16 > FP8 策略: MI300X 上使用 FP16 反而可能优于 FP8，因 memory bandwidth 在大 footprint 区间更高
