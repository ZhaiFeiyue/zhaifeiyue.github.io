2. 本篇 vs 相关论文的 delta #

抽象层级：Halo 占据的独特位置 #

Halo 的核心 delta 在于它是唯一一个在 workflow-plan 层 做全局优化的系统，位于 agent orchestration（LangGraph/AutoGen）和 serving engine（vLLM/SGLang）之间的"中间层"：


                 ┌─────────────────────────────────────────────────────┐
  Agent Layer    │ LangGraph / AutoGen / AgentScope (逻辑编排)          │
                 └───────────────────┬─────────────────────────────────┘
                                     │  (黑盒 API 调用)
                 ┌───────────────────▼─────────────────────────────────┐
  Plan Layer     │ ★ Halo (batch DAG 合并 + epoch DP + coalescing)     │ ← 本篇独创层
                 └───────────────────┬─────────────────────────────────┘
                                     │  (优化后的 request stream)
                 ┌───────────────────▼─────────────────────────────────┐
  Serving Layer  │ vLLM / SGLang + PPD/MFS/DualPath/KVServe            │
                 └─────────────────────────────────────────────────────┘

对比各篇的具体 delta #

vs DualPath [ref:2602.21548#核心三问]：DualPath 解决 PD 分离架构中 prefill engine 的 storage NIC 饱和问题——一个硬件层面的带宽分配问题。Halo 不涉及 storage I/O，而是在软件层面消除 workflow 内的冗余请求，使到达 serving engine 的请求量本身减少。DualPath 的 1.87× 离线吞吐提升来自聚合 SNIC 带宽；Halo 的 3.6× 提升来自消除重复 SQL/API 调用（request coalescing）和全局模型调度。两者解决不同层级的瓶颈，理论上可以叠加。

vs PPD [ref:2603.13358#核心三问]：PPD 基于"append-prefill 对 decode 干扰极小"的微架构观察，per-request 决定是否绕过 prefill 节点。这是 request-routing 粒度的优化——每个请求独立决策，不感知 workflow 结构。Halo 的决策粒度是 epoch-wide batch：它看到 1024 个 workflow DAG 的完整拓扑，可以发现跨 workflow 的 prefix 共享和 tool call 重复。PPD 的 TTFT 降低 48–73% 来自避免 KV 重传；Halo 的加速来自避免重复计算和调度全局模型亲和。

vs MFS [ref:2603.17456#核心三问]：MFS 在网络层面做 flow 调度——它看到的是"哪些网络 flow 该优先传输"，用 RMLQ 近似 Least-Laxity-First。Halo 在计算层面做任务调度——它看到的是"哪些 LLM/tool 节点该在哪个 GPU 上执行"，用 epoch DP 达到 MILP 最优。两者的决策空间完全不重叠（MFS 管网络优先级，Halo 管 compute placement），但都作用于 TTFT 优化。MFS 对 disaggregated MoE 有效，Halo 对 CPU-GPU 混合 agentic workflow 有效。

vs TensorHub [ref:2604.09107#核心三问]：TensorHub 的 Reference-Oriented Storage 消除了 RL 训练中权重传输的 ownership 开销。Halo 的 Request Coalescing 消除了 agentic inference 中 tool call 的 redundancy 开销。两者都是"消除不必要的数据移动/计算"，但作用在不同的 training/inference lifecycle 阶段。TensorHub 6.7× 减少 GPU stall 是在 1024 GPU 大规模下；Halo 3.6× 是在 3×H200 单机下。

vs PrfaaS [ref:2604.15039#核心三问]：PrfaaS 用长度阈值 $t$ 将长请求 offload 到跨 DC 的 compute-dense 集群，其优化函数是 $\Lambda_{\max} = \min(\Theta_{\text{prfaas}}/p, \Theta_{\text{pd-p}}/(1-p), \Theta_{\text{pd-d}})$。Halo 的优化函数是 $\mu \cdot \max_w T_w + (1-\mu) \sum_w T_w$（makespan + cost）。PrfaaS 做 producer-consumer 平衡，Halo 做 DAG 内 CPU-GPU pipeline 平衡。两者解决的是不同 granularity 的资源分配问题。

vs ZeRO-Prefill [ref:2605.02960#核心三问]：ZeRO-Prefill 反转"activation 路由到 expert"为"expert streaming 到 activation"，利用 prefill 大 batch 的 compute window overlap weight transfer。Halo 反转"每个 request 独立 API 调用"为"batch DAG 合并后全局调度"，利用 workflow 结构的冗余。两者的共同范式是利用 workload 的结构性特征（大 batch / 结构化 DAG）来反转传统数据流假设。但 ZeRO-Prefill 局限于 prefill-only MoE，Halo 局限于 static-topology agentic workflow。

vs TileRT [ref:tilert-speed-scaling-law#Q2方法]：TileRT 在最底层（kernel execution）做持久化——一个 persistent engine kernel 贯穿整个 decode 生命周期。Halo 在最高层（workflow plan）做持久化——一个 consolidated plan DAG 贯穿整个 batch 生命周期。两者都追求"消除边界开销"——TileRT 消除 kernel boundary，Halo 消除 request boundary。但优化空间完全不同：TileRT 追求 BS=1 的极致延迟，Halo 追求 BS=1024 的极致吞吐。

Halo 的独有增量 #

  1. Request Coalescing：在所有相关论文中独有。通过归一化 operator signature 发现语义等价的 SQL/API 调用并合并为单次物理执行，贡献 W6 上 +154% 吞吐 [ref:2509.02121#Key Findings]。这不是 prefix cache（只共享 token 前缀），而是 tool call deduplication（共享完整的 tool 执行结果）。
  2. Epoch-based DP with topological frontier pruning：从 MILP 的 NP-complete 到 2s 求解且质量 =oracle [ref:2509.02121#实验与数据]。其他系统要么用贪心（PPD scoring function, MFS RMLQ），要么用 grid search（PrfaaS Eq.7），没有在 DAG 调度上追求 MILP-quality 且保持在线可用。
  3. 三项 cost model $(T_{\text{prep}}, T_{\text{model}}, T_{\text{infer}})$ 的显式建模：DualPath 只考虑 I/O 带宽，MFS 只考虑网络 flow deadline，PPD 只考虑 TTFT/TPOT trade-off。Halo 同时考虑 CPU 预处理、模型切换、推理三个维度的 cost，是当前 agentic serving 中 cost model 最全面的。

    4. 3. 可攻击面 #

    攻击 1：静态 DAG 假设限制了实用性 #

    Halo 要求 workflow 拓扑在编译时确定（YAML 声明式），不支持 ReAct / planner-driven 的动态循环 [ref:2509.02121#Limitations]。而 2025–2026 主流 agent 架构（OpenAI Assistants, Claude Code, Cursor Agent）几乎全部使用动态规划——agent 根据前一步结果决定下一步做什么。Halo 的 DP solver 的"完成集 $D$ + worker 状态 $H$"需要枚举所有可能的执行路径，一旦允许运行时动态增删节点，状态空间爆炸。

    严重程度：高。Halo 论文自身限定为"enterprise SOP 场景下的 static topology"——这实际上是一个 niche 市场（BI 报表、批量审核、模板化检索），不是 agent 的 general case。

    攻击 2：Request Coalescing 的语义等价假设脆弱 #

    Halo 通过"归一化 operator signature"来判断两个 SQL/API 调用语义等价 [ref:2509.02121#Key Innovations]。但：

    • SQL 等价性是 undecidable 的（Rice's theorem），实际实现只能做模板级别匹配（参数化 SQL 的相同模板）
    • 如果两个 workflow 的"参数化 SQL"因上下文不同而有微妙差异（如不同 time window 的 WHERE 条件），coalesce 会产生错误结果
    • 论文未提供 coalescing 正确性的形式化保证或回退机制

    严重程度：中。在 TPC-H 等结构化 benchmark 上不是问题（模板完全确定），但在真实 agentic 系统中 SQL 模板的参数化程度可能不够统一。

    攻击 3：Epoch 粒度的选择缺乏理论分析 #

    论文声称"DP 近线性"但没给出严格的复杂度 bound [ref:2509.02121#生态影响追踪]。topological frontier 的大小取决于 DAG 结构——在高 fan-out 拓扑下（如 W6 的 fanout 结构），frontier 可能在某些 epoch 变得很大（所有 fan-out 分支同时就绪），导致 $B \subseteq \mathcal{F}$ 的枚举空间爆炸。Table 4 只在 W1/W6 两个 workload 上验证了 solver 时间（1–2s），没有给出 worst-case 场景。

    严重程度：中。实际系统中 epoch 长度和 fan-out 度可通过工程手段约束，但缺乏理论保证使系统在 adversarial workload 下可能退化。

    攻击 4：基线对比不完全公平 #

    Halo 的 400× vs vLLM 对比 [ref:2509.02121#Performance Evaluation] 本质上是 "workflow-aware batch system vs request-level serial system" 的对比——任何 agent framework（LangGraph/AutoGen）都不会用 serial vLLM 调用来处理 1024 个 workflow。真正有意义的对比是 vs LangGraph/AgentScope（1.03–3.6×），但这些框架的实现可能未经充分调优。

    论文未对比同期的 Ayo（primitive-level pipelining）和 Murakkab（SLO-aware declarative stack），而这些是更直接的竞品。

    攻击 5：单机 3×H200 限制了 generalizability #

    所有实验在单机 3 GPU 上完成 [ref:2509.02121#Limitations]。多节点部署会引入网络延迟到 cost model 中的 $T_{\text{prep}}$ 和 $T_{\text{model}}$ 项，且 KV migration 跨节点的代价远大于 intra-node。DualPath 明确展示了跨节点 KV 传输是 agentic workload 的真实瓶颈 [ref:2602.21548#核心三问]——Halo 完全回避了这个问题。

    4. 生态位 #

    范式定位 #

    Halo 代表的范式是 "数据库查询优化在 LLM workflow 上的复兴"——将 1970s 的 query plan optimization 思想（cost-based optimizer, plan enumeration, join reordering）移植到 2025 年的 agentic LLM 执行栈上。这不是增量优化，而是一种系统架构观的转变：从"serving engine 优化单个请求"到"workflow optimizer 优化整个 batch plan"。

    采纳证据 #

    • 正向：论文开源了匿名 demo（anonymous.4open.science/r/Halo_Demo-BC86），说明有开源意图 [ref:2509.02121#Infrastructure Impact]
    • 负向：截至 2026-05，尚未被 vLLM/SGLang 正式 adopt；没有生产部署报告；静态 DAG 限制排除了主流 dynamic agent 场景
    • 生态位压力：
    • 下方（serving engine 层）：vLLM/SGLang 的 prefix cache 和 automatic batching 已经在 request 级别做了部分优化，可能侵蚀 Halo 的增量价值
    • 上方（agent framework 层）：LangGraph/AutoGen 正在增加 compile-to-plan 能力（类似 DSPy optimizer），可能从上方侵蚀 Halo 的 workflow optimization 空间
    • 侧面（同层竞品）：Ayo（primitive pipelining）、Parrot（semantic variables）、KVFlow（workflow-aware eviction）都在争夺"workflow-aware serving optimization"这一新子领域

    长期判断 #

    Halo 的 epoch-based DP + request coalescing 更适合 batch analytics pipeline（定时报表、批量审核、数据处理）而非 interactive agentic systems（实时对话、code generation、autonomous agents）。如果 agentic workload 的主要形态变成前者（enterprise SOP 自动化），Halo 的生态位稳固；如果变成后者（动态 agent），Halo 需要根本性扩展才能 remain relevant。

    5. 未探索方向 #

    方向 1：Halo + PPD 的跨层融合 #

    Halo 的 epoch DP 可以把 PPD 的 per-request routing decision 作为 cost model 的一个参数——对于 multi-turn workflow，DP solver 可以预判哪些后续 turn 适合 local append-prefill（$S > 0$），从而在 plan 层提前安排 KV 亲和性。这需要将 PPD 的 scoring function [ref:2603.13358#核心三问] 作为 Halo 的 $T_{\text{infer}}$ 项的修正。

    方向 2：动态 DAG 的 online replanning #

    将 Halo 的 epoch DP 从 "plan-then-run" 扩展为 "plan-adapt-replan"：用 MFS 的 MLU-style 指标 [ref:2603.17456#核心方程物理意义] 监控执行进度，当实际耗时偏离 profiled cost 超过阈值时触发 partial replan（只重新优化未执行的 frontier 子图）。这可以处理 ReAct 式的条件分支——每次 agent 产出新的 sub-task，将其作为新节点插入 frontier 并重新求解。

    方向 3：跨节点 Halo + DualPath 的联合调度 #

    Halo 的 cost model 可以扩展为 $(T_{\text{prep}}, T_{\text{model}}, T_{\text{infer}}, T_{\text{io}})$ 四项，其中 $T_{\text{io}}$ 来自 DualPath 的带宽分析 [ref:2602.21548#作者证明]。在多节点部署中，DP solver 的 placement 决策需要同时考虑 compute affinity 和 storage I/O path 选择（PE read vs DE read）。DualPath 的 Eq.9 bottleneck-free 区间约束可以作为 Halo solver 的 hard constraint。

    方向 4：ZeRO-Prefill 的 frontend-backend co-design 范式在 Halo 上的应用 #

    ZeRO-Prefill 的饱和阈值 $T = t_{\text{EP}} \times F_{\text{GPU}} \times \gamma$ [ref:2605.02960#核心三问] 是一个物理导出的系统参数，frontend 强制 batch FLOPs ≥ T 来保证 backend overlap。类似地，Halo 的 epoch 长度可以用 "最小 overlap guarantee" 物理推导：$T_{\text{epoch}} \geq T_{\text{prep,max}} + T_{\text{model,max}}$，frontend (Parser) 保证每 epoch 的 LLM 节点数 ≥ 某个 compute budget。这将 Halo 的 epoch 选择从 hyperparameter tuning 转为 hardware-aware derivation。

    方向 5：Halo 与 KVServe 的 service-aware 压缩集成 #

    当 Halo 的 DP solver 发现某些 epoch 的 KV 传输量高（跨 worker 迁移 lineage 时），可以调用 KVServe 的 analytical model [ref:kvserve#核心三问] 判断是否值得压缩：B < (1-1/cr)·S 条件成立时启用压缩，否则直接传输。这将 KV 压缩的 go/no-go 决策嵌入到 plan-level cost model 中，而非 reactive 地在 transport 层决定。
