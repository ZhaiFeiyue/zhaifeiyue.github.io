PrefillShare — L3 Per-Paper Synthesis #

§1 相关论文 #

PrefillShare 处于 multi-model agent serving 的交叉路口，连接两个研究集群：(A) 跨模型 KV cache 共享和 (B) agent-aware 调度。

集群定位: PrefillShare 是 KB 中首篇将 模型层面的训练方法（cache-conditioned fine-tuning）与 serving 层面的架构优化（共享 prefill + disaggregated handoff）做端到端联合设计的论文。现有 agent category 的 9 篇论文全部在调度/缓存层面操作，假设模型参数不可变——PrefillShare 打破了这一假设。

§2 本篇 vs 相关论文的 delta #

2.1 PrefillShare vs ICaRus：同构方案的关键分叉 #

两篇论文的核心洞察惊人地一致：将 decoder-only Transformer 分解为 frozen base（KV 生成）+ task-specific decoder（token 预测），从而共享 KV cache。但在 decode 阶段的架构选择上产生了根本分叉：

维度ICaRusPrefillShare
Decode 架构Encoder + Decoder 并行运行，query 沿 head 维度拼接仅 task-specific decode module 运行
Decode 计算开销2× compute（encoder+decoder 同时前向）≈1× compute（仅 decoder）
适配方法LoRA (rank 128, α=256)完整模型权重的 cache-conditioned fine-tuning
Serving 架构单节点 continuous batching (vLLM)Disaggregated prefill-decode（prefill 实例 + decode 实例分离）
最大模型规模Qwen3-32BQwen3-14B
报告的延迟改善11.1× P95 (8 agents)4.5× P95
报告的吞吐改善3.8× (8 agents)3.9×

ICaRus 的 2× decode 计算开销在 memory-bound 场景下可被隐藏（memory access ≈ 1×），但 PrefillShare 通过 disaggregation 彻底消除了这一开销 [2602.12029]。反过来，ICaRus 不依赖 disaggregated serving 基础设施，部署更轻量 [2603.13281]。

关键矛盾: ICaRus 声称 frozen encoder 带来 "implicit regularization"，在 Qwen3 上准确率一致性优于 full-FT [2603.13281]。PrefillShare 的 Table 1 同样报告了 PrefillShare 偶尔超越 Full-FT 的现象（如 HumanEval 48.8 vs 48.2）[2602.12029]。但两者都没有在相同基准上做过 head-to-head 对比，且均未提供理论解释。这种 "frozen 比 full-FT 更好" 的一致性观察值得进一步研究——可能反映了某种过拟合抑制效应，也可能是 benchmark noise。

2.2 PrefillShare vs agent-scheduling 集群（Autellix / Continuum / SAGA） #

PrefillShare 的 delta 在于：它修改了模型本身，而非仅优化调度。

Autellix 的 PLAS/ATLAS 减少 program-level HoL blocking [2502.13965]。Continuum 的 KV TTL 消除排队气泡 [2511.02230]。SAGA 的 WA-LRU 预测 KV 复用概率 [2605.00528]。这三者都在 serving 层操作，假设每个模型的 KV cache 不可跨模型共享。PrefillShare 通过修改训练过程打破了这一前提：$N$ 个模型的 KV cache 从 $O(N)$ 降至 $O(1)$ [2602.12029]。

这意味着 PrefillShare 与上述三者在原理上完全正交——可以叠加使用。例如，PrefillShare 的共享 prefill 实例消除了跨模型冗余 prefill 后，Continuum 的 TTL 可进一步管理 tool-call 间隙中共享 cache 的保留策略；SAGA 的 AEG 可预测哪个 decode module 将被下一步调用。

2.3 PrefillShare vs TokenDance：不同维度的 KV 共享 #

TokenDance 的 KV 共享面向 同一模型 在多 agent 模拟中的 All-Gather 轮次——不同 agent 使用相同模型但不同 prompt，共享部分通过 Position-Independent Caching (PIC) 复用 [2604.03143]。PrefillShare 的 KV 共享面向 不同模型 在同一 prompt 上的冗余——多个 fine-tuned 模型处理相同输入。两者的 KV 冗余结构完全不同：TokenDance 的冗余是 跨 agent 同 prompt 段 的位置偏移，PrefillShare 的冗余是 跨模型同 prompt 的参数耦合。

§3 可攻击面 #

攻击 1：cache-conditioned fine-tuning 的适用边界未探明 #

PrefillShare 声称 cache-conditioned fine-tuning 在 100% KV sharing 下维持 full-FT 精度 [2602.12029]。但实验仅覆盖三类任务（math、coding、tool-calling），且均为 浅层推理（GSM8K 单步、HumanEval 短代码）。对于需要 深层 multi-hop reasoning 的 agent 任务（如 SWE-Bench 的 10+ turn 修复、FRAMES 的跨文档推理），base prefill 的 KV 表征是否仍能捕获足够的 task-specific 信号？论文的 Figure 2 仅展示了 sharing ratio 对单次推理精度的影响，未测量在多轮累积误差放大下的表现。

量化反驳: 如果每 turn 的 base-vs-full-FT 表征偏移为 $\epsilon$，则 $k$ turn 后累积偏移可能达到 $O(k \cdot \epsilon)$ 或更差（取决于是否存在 error amplification）。在 10-turn SWE-Bench agent 中，这个累积效应可能从"不可察觉"变为"task-critical"。

攻击 2：disaggregated serving 的 handoff 瓶颈被低估 #

论文承认 Fig 4 中 PrefillShare 的吞吐在 ~120 concurrent sessions 后下降，归因于 handoff overhead [2602.12029]。但在真实多模型 agent 工作流中，每个 prompt 触发 $N$ 个 decode 请求（一个 prefill → $N$ 次 handoff），handoff 压力是 $N$ 倍。论文的实验使用 $N=4$（ReAct 4-agent），但未报告 $N=8, 16, 32$ 时的退化曲线。ICaRus 在 $N=8$ 下报告了结果且表现稳健 [2603.13281]——因为 ICaRus 不需要 handoff（encoder 和 decoder 在同一 GPU 上运行）。

矛盾: PrefillShare 宣称 disaggregation 是优势（消除 2× decode 开销），但 disaggregation 引入了 handoff bottleneck。ICaRus 的 non-disaggregated 方案在 decode 有 2× 计算但避免了网络传输。在高 $N$ + 高并发场景下，哪种 trade-off 更优是一个未回答的问题。

攻击 3：与 LoRA 生态的兼容性缺失 #

ICaRus 使用 LoRA adapter 作为 task-specific decoder——这意味着 adapter 的热切换、合并、分发可以利用成熟的 LoRA serving 生态（vLLM LoRA、S-LoRA 等）[2603.13281]。PrefillShare 的 decode module 是完整模型权重的 cache-conditioned fine-tuning——每个 task 需要一份完整的模型权重。在 $N$ 个 task 下，PrefillShare 的权重存储是 $O(N \times M_{decode})$ vs ICaRus 的 $O(M_{base} + N \times M_{adapter})$，后者远小于前者（LoRA adapter 通常 <1% of base parameters）。

攻击 4：实验 workload 与真实 multi-model agent 的代表性 #

PrefillShare 使用 HotpotQA + ReAct/Reflexion 作为 agent workload [2602.12029]。但 HotpotQA 是一个 静态 QA 数据集，agent pipeline 的 tool-call 模式高度可预测。真实 multi-model agent 工作流（如 SWE-Bench 的 coding + testing + reasoning 联合管线）的 prompt 结构、tool-call 间隙、模型调用序列远比 HotpotQA 复杂。SAGA 和 Continuum 均在 SWE-Bench 上验证 [2605.00528] [2511.02230]——PrefillShare 未在同一 benchmark 上评估，使得直接对比不可能。

攻击 5：无 failure mode 分析 #

L2 的 agent-specific checks 已指出 [2602.12029]：PrefillShare 未分类 multi-agent pipeline 的 failure modes。当 base prefill 的 KV 表征与某个特定 task decoder 严重不匹配时（例如该 task 需要 prompt 中的某些 fine-grained 语义信号），系统会静默降级而非报错。对比 SideQuest 的 non-completion rate 检测和 Continuum 的 TTL 过期恢复路径，PrefillShare 缺乏运行时的质量监控机制。

§4 生态位 #

范式定位：从 "scheduling the inference" 到 "training for shared inference" #

现有 agent-serving 研究几乎全部在 serving 层 操作：给定 $N$ 个不可变模型，优化调度、缓存、路由。PrefillShare（及 ICaRus）代表了一个新范式——通过修改训练过程消除 serving 层的根本限制。这类似于 quantization-aware training vs post-training quantization 的关系：前者改变模型以适应部署约束，后者在不改变模型的前提下优化部署。

在 agent category 的 taxonomy 中，PrefillShare 占据一个当前为空的格子：

LayerScopeMechanism
模型训练 + KV-Cache多模型 agent 工作流训练时对齐 (cache-conditioned FT)

现有 9 篇论文均在 "KV-Cache 管理 / 调度与执行 / 架构与形式化" 层面操作。PrefillShare 新增了 "模型训练" 这一层次，且是唯一同时涉及训练和 serving 的论文。

采纳证据 #

  • 代码未开源 [2602.12029]——阻碍社区采纳。ICaRus 同样未开源。
  • vLLM disaggregated serving 依赖: PrefillShare 建立在 vLLM 的 disaggregated serving 之上，但该功能在 vLLM 主线中仍为实验性质，增加了实际部署的不确定性。
  • 训练成本: cache-conditioned fine-tuning 需要对每个 task 的训练数据重新运行 base model prefill 生成 $C_{base}$，再训练 decoder。这比标准 LoRA fine-tuning 的额外开销需要量化但论文未给出。

与已有综述的关系 #

L3 category/agent.md 当前覆盖的 9 篇论文均假设模型不可变。PrefillShare 的加入需要扩展 taxonomy 的 Layer 维度——增加 "模型训练" 层。同时，PrefillShare 的 "共享 prefill + disaggregated decode" 架构为综述的 Gap #1（跨 turn KV 管理 + CPU 调度联合优化）提供了新的可能：共享 prefill 天然简化了 KV 管理（因为只有一份 cache），但 handoff 引入了新的 CPU/网络开销。

§5 未探索方向 #

方向 1：PrefillShare + LoRA adapter = 轻量化 decode modules #

PrefillShare 使用完整模型权重做 cache-conditioned fine-tuning，ICaRus 使用 LoRA。两者的训练目标本质相同（在 frozen base KV 上训练 decoder），唯一区别是参数化方式。一个自然的混合方向是 LoRA-based cache-conditioned fine-tuning + disaggregated serving：保留 PrefillShare 的 disaggregation 优势（decode 无 2× 开销），同时用 LoRA 减少 per-task 权重存储和切换开销。这需要验证 LoRA 在 cache-conditioned 训练范式下的表现是否与完整 fine-tuning 一致。

方向 2：动态 prefill module selection #

PrefillShare 假设所有 task 共享同一个 frozen base model 的 prefill。但当 task 差异巨大时（如 coding vs 多模态），单一 base 的 KV 表征可能无法同时服务所有 task。一个自适应方案是维护 2–3 个 "domain base models"（如 code-base、chat-base、multimodal-base），根据输入 prompt 的语义选择 prefill module。这将 PrefillShare 的 $O(1)$ KV 提升到 $O(K)$（$K \ll N$），但 $K$ 远小于完全独立的 $N$。

方向 3：PrefillShare 与 Continuum TTL 的联合 #

PrefillShare 消除了跨模型 KV 冗余，但未解决 tool-call 间隙的 KV 保留问题——共享 $C_{base}$ 在 tool call 期间应保留多久？Continuum 的 TTL 机制可直接应用于共享 cache：$\tau^*(C_{base})$ 的 Benefit 项应乘以 $N$（因为一份 cache 服务 $N$ 个 decoder），而 Cost 项不变（只占一份显存）。这使得共享 cache 的最优 TTL 严格大于 per-model cache 的 TTL——从理论上支持 PrefillShare 在 agent workload 下的进一步优势。

方向 4：cache-conditioned fine-tuning 的理论分析 #

PrefillShare 和 ICaRus 都观察到 "frozen base 有时优于 full-FT" 但缺乏理论解释 [2602.12029] [2603.13281]。一个可行的理论方向是分析 $\|C_{base} - C_{full-FT}\|$ 的上界作为 frozen layer 数量的函数，并证明当 decoder 容量足够时，这个表征差距可以被完全补偿。这将为 frozen-ratio 的选择提供理论指导，而非当前的 "全冻结或不冻结" 的二元选择。

方向 5：SAGA 的 AEG 指导 decode module 预加载 #

在 PrefillShare 的 disaggregated 架构中，decode instance 需要加载 task-specific 权重。SAGA 的 Agent Execution Graph 可预测下一步将调用哪个 task model [2605.00528]，从而提前预加载对应的 decode module 权重——将 decode module switching 的延迟从关键路径上移除。这将 SAGA 的工作流预测能力从 KV cache eviction 扩展到 weight management。
