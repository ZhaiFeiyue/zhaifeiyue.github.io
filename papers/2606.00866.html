L3 Synthesis · MORI (2606.00866) — 相对空闲度驱动的 agentic KV-cache 分层调度 #

1. 相关论文 #

MORI [2606.00866] 位于 agent serving 系统这条主线的最新一层。8 篇相关论文按「与 MORI 的机制距离」由近及远排列：

核心近亲 (同一 program-aware KV 调度问题, 同一 Berkeley/Stoica 谱系)

  • Continuum (2511.02230) — 最强直接对手，也是 MORI 的 predecessor 之一。同样处理 multi-turn agent 的 tool-call 间隙 KV 管理，但用的是 KV cache TTL（cost-benefit 定时 pin）而非连续空闲度排序 [2511.02230]。两者共享 program-level FCFS 血统。
  • Autellix (2502.13965) — MORI 引用为 program-aware 调度奠基者之一。把 OS 的 Least-Attained-Service 提升到 program 级 (PLAS/ATLAS)，解决 program-level HoL blocking [2502.13965]。MORI 与它正交：Autellix 排队列优先级，MORI 排内存层放置。
  • TokenCake (2510.18586) — 机制上最像 MORI 的对手：同样把 tool-call idle window 当作 offload 机会，同样做 GPU KV 分区，但用 双调度器 (Temporal + Spatial) + 预测式 upload，而非单一空闲度排名 [2510.18586]。

同赛道不同刀法

  • Speculative Tool Calls (2512.15834) — 与 MORI 目标相同（消除 tool-call 造成的 GPU 空转），但手段相反：MORI 接受空闲窗口并用它 offload；投机工具则试图消灭空闲窗口（提前异步执行工具，让生成与工具重叠）[2512.15834]。
  • KVCOMM (2510.12872) — multi-agent KV 复用，但攻击的是跨 agent 前缀的 prefill 重复，而非跨时间的分层放置 [2510.12872]。与 MORI 的 subagent 场景有交集（MORI 把 subagent 当一次长 tool call）。

上下文与边界

  • CPU-Centric Perspective (2511.00739) — 特征刻画论文，揭示 agent E2E 延迟高达 88% 花在 CPU 工具执行上 [2511.00739]。这为 MORI 的整篇动机（tool-call gap 真实且巨大）提供独立佐证，但也构成攻击面（见 §3）。
  • RouteLLM (2406.18665) — 只在「agent serving 成本优化」这个宽泛类目下相关；它是 query→model 路由分类器，无 KV、无多轮、无 tool-call 状态 [2406.18665]。与 MORI 几乎正交，仅作 taxonomy 对照。
  • Agent Interoperability Survey (2505.02279) — 协议层综述 (MCP/A2A/ACP/ANP)，与 MORI 的系统调度层无技术交集 [2505.02279]。仅说明 agent 生态的另一维度。

2. 本篇 vs 相关论文的 delta #

新颖点 (MORI 独有):

  1. 把空闲度从「二值/阈值」重构为「连续、相对」的排名。 这是 MORI 相对所有前作的本质 delta [2606.00866]。Continuum 用 per-request TTL 定时器 [2511.02230]，TokenCake 用 per-agent 静态/动态优先级分数 [2510.18586]，Autellix 用累计 attained service [2502.13965]——三者都是绝对量。MORI 的 $\iota$ 排名让 GPU:CPU 分界线自动滑到任意硬件容量比，无需 per-hardware 调参 [2606.00866]。
  2. 单一 sort-and-fill 同时充当两层 admission control。 「从最忙一端填满每个 tier」这个动作本身就是 GPU 和 CPU 双层准入 [2606.00866]。相比之下 Continuum 只管 GPU 侧（TTL 是 GPU 显存上界）[2511.02230]，MAS (2511.00739) 用独立弹性队列上限做准入 [2511.00739]。

    3. 增量点 (站在前作肩上):

    1. coordinated 两层放置 vs 前作的单层/uncoordinated。 MORI 的直接基线 TA+O 正是「scheduler 管 GPU + HiCache 独立 LRU 管 CPU」的 uncoordinated 版本，MORI 证明协调 CPU tier 能把额外 DRAM 真正用起来（2× CPU 让 MORI TTFT 降但 TA+O 不动）[2606.00866]。这与 TokenCake 的「shared pressure snapshot 让两调度器一致」是同一洞见的不同实现 [2510.18586]。
    2. 多 replica affinity 覆盖 CPU tier。 MORI 追踪程序 KV 在 GPU 或 CPU 的驻留并路由回原 replica，把 churn 从 14–15% 降到 0.3–2.9% [2606.00866]。Autellix 的 locality-aware 路由只到 primary engine 级别、multi-engine 收益仅 ~1.5× [2502.13965]。

      3. 矛盾/张力点:

      1. 「空闲窗口是资源」vs「空闲窗口是敌人」。 MORI 的整个设计前提是 tool-call idle window 值得利用来 offload [2606.00866]；Speculative Tool Calls 则视 tool 等待为纯浪费，用小模型投机执行去消灭它 [2512.15834]。二者不冲突但暗含不同世界观——若投机把长 idle 窗口普遍缩短，MORI 可 offload 的窗口就变少（详见 §3）。

        2. 3. 可攻击面 #

        攻击 1 — 空闲窗口的「可利用性」建立在工具无法被投机/加速的假设上。

        MORI 的 §3.2 论证：短 tool call 的 offload 传输成本超过 gap 本身，只有长 idle 才值得 offload [2606.00866]。但 Speculative Tool Calls 表明无状态工具 (web search / file read / ls) 可提前异步执行、与生成重叠 [2512.15834]。若投机工具普及，MORI 赖以 offload 的长 idle 窗口会被压缩——尤其 §3.3 里那 13% 长 tool call（占 58% tool 时间）中的一部分（如 pytest、compile）恰恰是投机难覆盖的有副作用工具，但 human-input / subagent 等可提前调度的窗口会缩水。MORI 未讨论与投机执行共存。

        攻击 2 — 88% 的 CPU 工具瓶颈可能意味着 offload 目标本身受争用。

        CPU-Centric 论文测得 tool 执行占 E2E 高达 88%，且 CPU 在 BS=128 已 over-subscribe [2511.00739]。MORI 把 idle 程序的 KV offload 到同节点 CPU DRAM，并假设 CPU-to-GPU PCIe reload 便宜 [2606.00866]。但如果同一批 CPU 核心正被 tool 执行 (FAISS/LexRank) 打满，offload/upload 的 host 侧 memcpy 会与工具执行争 CPU/内存带宽——MORI 的成本模型只算了 PCIe 传输，没算 host 端 CPU 争用。这在 RAG-heavy 或 ChemCrow 类 CPU-bound workload 下可能让 offload 净收益转负。

        攻击 3 — 5 秒 tick + k=5 窗口对短命程序反应太慢。

        MORI 的 sticky 策略默认 5 s 控制 tick、k=5 循环窗口 [2606.00866]。Continuum 的实测显示 tool call 中位 925 ms (SWE-Bench)，且改进随 turn 数放大——per-turn 排队气泡是累积量 [2511.02230]。对一个只有 6–11 个 turn 的短程序，5 s tick 可能在其大半生命周期里只调度 1–2 次，来不及把它从 busy 阶段的正确 tier 里放好。MORI 在 20 程序低并发下仅 +2% (546 vs 534) [2606.00866] 正是这个副作用的征兆——它承认低并发无优势，但没量化 tick 粒度对短程序尾延迟的伤害。

        攻击 4 — 「idleness 是未来行为的可靠代理」缺形式化，且与 Autellix 的 SRPT gap 呼应。

        MORI 用「phase transition 稀疏 → 近期行为预测未来」为 windowed $\iota$ 辩护 [2606.00866]，但和所有同类一样无形式化证明。Autellix 明确量化了 non-clairvoyant 调度相对 clairvoyant SRPT 的固有 gap [2502.13965]。MORI 的 $\iota$ 本质也是 non-clairvoyant 预测器，它相对「已知每个 tool 时长的 oracle 放置」的 competitive gap 完全没测——这是可攻击的空白，Continuum 的 memoryfulness factor $\eta$ [2511.02230] 至少把「顺序是否有用」量化了，MORI 没有对应的可诊断标量。

        4. 生态位 #

        范式定位：从「二值 pin」到「连续相对排名」的调度器进化第三代。

        • 第一代：program-aware 但只管 GPU 队列优先级（Autellix PLAS/ATLAS，2025-02）[2502.13965]。
        • 第二代：引入 KV 保留 + CPU offloading，但决策是绝对量（Continuum TTL 定时器 2025-11 [2511.02230]；TokenCake 双调度器 2025-10 [2510.18586]）。
        • 第三代 = MORI (2026-05)：把放置决策统一为一条连续相对排名，让分界线随硬件自适应，并把两层 admission 折叠进 sort-and-fill [2606.00866]。

        采用证据 / 工程可信度： MORI 建在 ThunderAgent + SGLang v0.5.10 之上，仅 ~3300+500 LoC，client 侧只需一个 program_id [2606.00866]。这个「最小侵入 + 单字段 API」路线与 Continuum 的「vLLM 模块化回调插件」[2511.02230] 是同一采用哲学。但 MORI 代码未公开 [2606.00866]，而 Continuum 已开源 (github.com/Hanchenli/vllm-continuum) [2511.02230]、RouteLLM 已开源 [2406.18665]——在可复现性上 MORI 目前落后于最直接的对手。

        独特生态位： MORI 是本簇里唯一把「硬件容量比自适应」作为一等目标的系统——它明确针对 1:1.6 vs 1:3.1 GPU:CPU 比的差异 [2606.00866]。其余系统都假定固定拓扑。这让 MORI 在异构集群 / 多种 DRAM 配置的部署场景里有独占价值。

        5. 未探索方向 #

        方向 1 — MORI × 投机工具执行的混合调度。

        若把 Speculative Tool Calls 的 client-side 投机 [2512.15834] 作为 MORI 的输入信号：投机模型对「某程序下一个 tool 是短还是长」的预测，可以让 MORI 的 $\iota$ 从后验窗口平均升级为带前瞻的估计，缩短 §3 攻击 3 里的反应延迟。反过来，MORI 的 tier 状态可告诉投机器「哪些程序反正要 offload，不值得投机」。

        方向 2 — 自适应 tick / 窗口。

        把 CPU-Centric 的静态→动态批判 [2511.00739] 迁移到 MORI：k=5 和 5 s tick 是固定的 [2606.00866]。可按程序当前 turn 速率自适应窗口——高频短程序用小 k 快反应，长 idle 程序用大 k 抗抖动。Continuum 的 memoryfulness factor $\eta$ [2511.02230] 是天然的自适应触发信号。

        方向 3 — 把 subagent 从「一次长 tool call」提升为一等调度对象。

        MORI 现在把 spawned subagent 当作父程序的单次长 tool call [2606.00866]。KVCOMM 揭示 subagent 间大量共享上下文可跨前缀复用 KV [2510.12872]。混合方向：父程序 offload 到 CPU 期间，用 KVCOMM 的 anchor 偏移近似让 subagent 复用父程序 CPU 侧 KV，避免父程序 promote 回来时的全量 reload。

        方向 4 — CPU-tier 感知的成本模型。

        把攻击 2 转成设计：MORI 的 offload 决策应把 host CPU 争用（tool 执行占用的核心/带宽 [2511.00739]）纳入 $\iota$ 或纳入 demotion 门槛——当 CPU 正被 CPU-heavy 工具打满时，offload 到该节点 DRAM 应被惩罚，转而走 §7.1 讨论的 SSD / 跨节点 RDMA tier。

        方向 5 — 分层调度的形式化 competitive 分析。

        借 Autellix 的 SRPT-gap 框架 [2502.13965]，给 MORI 的 $\iota$-ranking 放置建立相对 clairvoyant oracle（已知每个 tool 时长）的 competitive ratio，填补 §3 攻击 4 的空白。这是整簇（MORI/Continuum/TokenCake/Autellix）共同缺失的理论方向，宜留待 L4/topic 处理其根本难度。
