2606.00866 Cross-Paper Synthesis #

相关论文 #

2606.00866（MORI）处在“程序级 agent serving + KV 分层驻留”主线上，但与同线方法在调度信号与优化目标上有明显分工。Autellix 将调度单位提升到 program 并用 attained service 解决 program-level HoL，本质是公平与等待时间导向 [2502.13965][2502.13965]；Continuum 用 TTL 在 turn 间保留 KV，核心是把 queueing cost 显式写进保留时长决策 [2511.02230][2511.02230]；SAGA/HexAGenT 进一步把单位抬到 workflow/online-DAG，在集群层面优化任务完成与 SLO attainment [2605.00528][2605.16637]。ConServe 则把“会话级两阶段”作为第一原则，通过一次转移+长期 pin 避免每 turn 预测 [2606.01839]。

与 MORI 在同一“内存层级管理”面向上，TokenCake强调时间/空间双调度协同，IntentKV/Leyline强调“缓存内容与布局本身可编辑/可裁剪”，PrefillShare强调多模型共享 prefill 以避免重复建 cache，Orla强调 workflow 控制面分离与跨 stage 生命周期管理 [2510.18586][2606.09916][2606.01065][2602.12029][2603.13605]。因此 MORI 的邻域并非单一“offloading 论文群”，而是 agent serving 中“调度信号设计”与“KV 生命周期控制”两条线的交叉点。

delta #

MORI 的主要增量不在“引入 CPU offload”本身，而在把 idleness 定义为连续相对量并用排名来对齐 GPU/CPU 容量比，避免固定 busy/idle 阈值在不同硬件比下失配 [2606.00866]。这与 Continuum 的“每请求 TTL 最优化”形成互补：后者是绝对时长决策，前者是全局相对排序决策 [2511.02230]。与 Autellix 的 attained-service 范式相比，MORI 优先优化“谁更可能马上消耗 GPU”，而不是“谁已获得多少服务” [2502.13965]。

第二个 delta 是双层准入与类型一致性：MORI 同时管理 GPU HBM 与 CPU DRAM 准入，并把 busy/idle/inactive 类型下传到引擎层驱逐顺序，减少调度器意图和块级驱逐策略之间的不一致 [2606.00866]。这比仅在上层做队列优先级（如部分 workflow 调度工作）更“端到端” [2605.00528][2605.16637]。

第三个 delta 是多副本亲和约束下的 CPU-tier continuity：MORI 把“CPU 上仍有可用 KV”纳入路由亲和，而不是只看 GPU 命中，这一点直接针对了 TA/TA+O 在高并发下的 churn 问题 [2606.00866]。对比 ConServe 的“会话固定绑定解耦预测”与 TokenCake 的“时间-空间联动”，MORI 处于中间地带：比 per-turn 方法更程序感知，但仍以单节点两层内存为主 [2606.01839][2510.18586]。

可攻击面 #

第一，MORI 的 idleness 估计依赖固定窗口 k=5 与阶段持续性假设；当工作负载出现更高频 phase flip（例如短 burst 与微工具交替）时，窗口平均可能滞后于真实状态，导致“错层驻留” [2606.00866]。这类场景在 workflow 高分支/高动态设置里更常见，SAGA 与 HexAGenT 通过结构信号（AEG 或 online DAG）规避了仅靠时长统计的盲点 [2605.00528][2605.16637]。

第二，MORI 的证据主体来自 coding-agent 回放；虽然覆盖多模型/多并发，但对开放环境（browser agent、多工具异步失败重试）的外推性仍有风险。SAGA 在 SWE-bench + WebArena 双场景给出 cluster 级对照，ConServe 则展示了会话级策略在错误预测率变化下的结构稳定性，这两类证据维度当前在 MORI 中偏弱 [2605.00528][2606.01839][2606.00866]。

第三，MORI 把主要优化放在“谁驻留哪一层”，但不直接处理“驻留内容是否可压缩/可编辑”；当上下文爆炸且 CPU 也吃紧时，仅迁移层级而不改变内容体量会触及硬上限。IntentKV 与 Leyline 的结果提示：语义相关裁剪或指令化 splice 能进一步减少最坏 KV 读写压力，这是 MORI 当前未覆盖的攻面 [2606.09916][2606.01065]。

生态位 #

MORI 的最佳生态位是“工具调用重尾显著、并发中高、且单机/少量副本内 GPU-CPU 分层明显”的 agent serving 场景：它不要求工作流先验图，也不依赖高精度时长预测，部署约束相对温和 [2606.00866][2606.00866]。

在更强调 workflow deadline/SLO 的异构集群场景，HexAGenT/SAGA 的 workflow-level 风险或公平目标更直接；在“会话两阶段结构非常稳定”的场景，ConServe 的会话级 pin 可能更低复杂度；在“多模型同 prompt”与“跨 stage 编排”场景，PrefillShare 与 Orla 的收益点更靠前 [2605.16637][2605.00528][2606.01839][2602.12029][2603.13605]。

因此，MORI 更像“程序感知内存层级调度器”：它在不改 agent 语义层的前提下，吃到 idle window 红利并兼顾多副本亲和；但它不是全栈解（不负责 DAG 级 SLO 优化，也不直接改变 KV 内容表示）[2606.00866]。

未探索方向 #

  1. Relative rank + absolute TTL 混合策略：在 MORI 排名框架中加入 Continuum 风格的显式 TTL 截止，可把“谁更闲”与“最多可占多久”同时纳入，降低 CPU 层长期滞留风险 [2606.00866][2511.02230]。
    1. 两层调度分解：集群间用 HexAGenT/SAGA 的 workflow 风险做粗分配，节点内再用 MORI 的 idleness 排名做细粒度 GPU/CPU 驻留，实现“结构感知 + 行为感知”组合 [2605.16637][2605.00528][2606.00866]。
      1. 内容感知下沉：对高 idleness 程序先执行 IntentKV 式跨 turn 相关裁剪或 Leyline 式定向 splice，再下沉 CPU，可减少迁移体积与回迁成本，缓解 CPU 容量压力 [2606.09916][2606.01065]。
        1. 多模型与跨后端扩展：把 MORI 的相对 idleness 概念推广到 PrefillShare/Orla 式多模型多 backend pipeline，让“驻留决策”覆盖 shared-prefill、stage 切换和 cache-affinity，而非仅限单模型两层内存 [2602.12029][2603.13605]。
