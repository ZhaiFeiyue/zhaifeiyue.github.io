§2 本篇 vs 相关论文的 delta #

2.1 FP4/FP6 — 从理论精度到实测硬件行为 #

2512.02189 首次报告 FP4 在 B200 上达到 7700 TFLOPS (96.2% 峰值) [ref:2512.02189#实验与数据]，延迟仅比 FP16 多 1.4 cycle (12.6 vs 11.2)。这一数据直接重塑了 2502.01563 的量化策略建议：该论文发现 Q/K 中的 massive values 对上下文理解至关重要，保护 massive values 的量化方法 (AWQ/SmoothQuant) 显著优于不保护的方法 (GPTQ) [ref:2502.01563#实验与数据]。但 2502.01563 的实验局限于软件量化 (W4A16, W8A8)，从未考虑 FP4 作为 原生硬件格式 的场景。

2512.02189 报告 FP4 推理的 perplexity 代价为 +8.2% (Mistral-7B) [ref:2512.02189#实验与数据]。这个退化幅度与 2502.01563 报告的 GPTQ 在 contextual knowledge 任务上的约 25% 归一化精度下降 [ref:2502.01563#实验与数据] 处于完全不同的量级——暗示原生 FP4 tensor core 的精度保持远优于后量化方案。但两者的实验维度不同（perplexity vs task-specific accuracy），直接对比需谨慎。

Delta: 2512.02189 提供了 FP4/FP6 的 硬件地板价（延迟、吞吐、功耗），将量化研究从"软件精度保持"推进到"硬件-精度-吞吐三角权衡"。

2.2 TMEM — 模型架构的新内存层 #

TMEM (256KB/SM, 16 TB/s read, 420 cycle cache-miss) 是 Blackwell 最具架构影响的创新 [ref:2512.02189#实验与数据]。对相关论文的影响分三个层次：

直接受益者——KDA 状态矩阵 (2510.26692)：Kimi Linear 的 KDA 每 head 维护 128×128 recurrent state (16K floats = 64KB per head) [ref:2510.26692#模型架构互动图]。32 heads 的完整 KDA 状态 ~524K floats ≈ 2MB per layer，虽然超过单 SM 的 TMEM 容量 (256KB)，但单 head 的 state update完全适配 TMEM 的 64×64 最优 tile [ref:2512.02189#实验与数据]。chunkwise kernel 在 TMEM 上可避免 $S_t$ 回写 HBM 的 12 TB/s 流量——这正是 2512.02189 为 chained GEMM 场景量化的收益。

间接受益者——MLA latent attention (2507.20534, 2602.02276)：K2/K2.5 的 MLA 将 KV 压缩到 576 floats/token [ref:2602.02276#K2 LLM Decoder Layer]。TMEM 的 accumulator 驻留能力意味着 kv_b_proj 的展开 (512→16384) 可在 TMEM 中完成 chained GEMM，不经 global memory 回写。2512.02189 估算 chained GEMM 每 SM 避免约 12 TB/s 数据移动 [ref:2512.02189#实验与数据]。

需要 kernel 重写——所有 workload：2512.02189 明确指出 TMEM 要求使用全新的 tcgen05.cp/ld/st 指令族，传统 ldmatrix/wmma.load 无法访问 [ref:2512.02189#实验与数据]。这意味着 2510.26692 的 KDA Triton kernel、2510.22200 的 BSA Triton kernel 都需要完全重写才能利用 TMEM。

2.3 恒定延迟 tensor core — 空间阵列对 kernel 设计的影响 #

2512.02189 最重要的微架构发现是 tcgen05 的 ~11 cycle 恒定 MMA 延迟（tile 从 m64→m256 不影响延迟），区别于 Hopper wgmma 的线性增长 (32→128 cycle) [ref:2512.02189#实验与数据]。这暗示 Blackwell 采用空间阵列而非时域流水线。

对 2510.26692 (Kimi Linear) 的启示：KDA 的 chunkwise kernel 使用 chunk_size=64 [ref:2510.26692#核心训练壁垒]。空间阵列意味着增大 chunk_size 到 128 或 256 不会增加单 MMA 延迟，但会线性提升吞吐——这为 KDA kernel 的 Blackwell 移植提供了清晰的优化方向。

对 2510.22200 (LongCat-Video) 的启示：BSA 的 Triton kernel 对视频 latent 做 3D block attention，当前选择 chunk shape [4,4,8] [ref:2510.22200#架构 / 方法图]。空间阵列使更大 tile 成为可能而无延迟惩罚，BSA 可以在 Blackwell 上使用更粗粒度的 block 选择而不牺牲单 block 内 attention 的延迟。

2.4 算力-带宽比的 workload 迁移 #

2512.02189 量化了 Blackwell 不同精度下的 Ops/Byte 比率：FP16 = 241, FP8 = 481, FP4 = 963 [ref:2512.02189#实现 cross-reference]。实测验证：FP16 推理 BW utilization 67.3% (memory-bound) → FP4 降至 47.6% (转向 compute-bound) [ref:2512.02189#实验与数据]。

这一转变对 2509.17765 (Qwen3-Omni) 和 2503.20215 (Qwen2.5-Omni) 有直接含义：它们的 prefill 阶段是 compute-bound，decode 阶段是 memory-bound。FP4 会将 decode 阶段也推向 compute-bound，使得 Qwen3-Omni 的 234ms 首包延迟目标 [ref:2509.17765#实验与数据] 在 Blackwell 上更可能通过增大 batch 来改善，而非仅靠降低单请求延迟。

2.5 训练加速分解的可信度 #

2512.02189 将 GPT-1.3B 训练加速分解为 SM 增量 (1.09×) × CTA pairing (1.27×) × TMEM (1.26×) = 1.75，但实测仅 1.55× [ref:2512.02189#实验与数据]。13% 的 gap 暗示这三个因子非独立——存在未量化的 diminishing returns。

对比 2507.20534 (Kimi K2)：K2 在 15.5T tokens 上的训练使用 16PP+16EP+ZeRO-1 分布式策略 [ref:2507.20534#Infrastructure Impact]。在 MoE 训练中，all-to-all 通信 (384 experts, top-8) 会部分抵消 TMEM 和 CTA pairing 的单卡计算加速——因此 2512.02189 报告的 1.55× 训练加速上界在实际 MoE 分布式场景中可能进一步缩减至 1.3-1.4×。

§3 可攻击面 #

3.1 FP8 推理加速仅 1.16× — Blackwell 的 FP8 增量优势极小 #

2512.02189 报告 Mistral-7B FP8 推理的 B200/H200 加速仅 1.16× (57,125 vs 49,200 tok/s)，远低于 FP16 的 1.97× [ref:2512.02189#实验与数据]。这意味着 H200 的 FP8 路径已被充分优化，Blackwell 在 FP8 上的增量优势主要来自新精度 (FP4/FP6) 而非 FP8 本身。

对 2509.17765 (Qwen3-Omni) 的冲击：Qwen3-Omni 未披露训练/推理的精度配置 [ref:2509.17765#实验与数据]，但 MoE 模型通常使用 FP8 或 BF16。如果 Qwen3-Omni 在 H200 上已用 FP8 推理，迁移到 B200 的 FP8 加速仅 ~1.16×——远低于架构宣传的 "generation leap"。真正的加速需要重新量化到 FP4，但这会引入 2502.01563 警示的 massive value 破坏风险。

3.2 STREAM Triad 低效率暴露双 die 开销 #

B200 在 4-16GB 工作集的 STREAM Triad 上仅达 52% 峰值带宽，低于 H200 的 60% [ref:2512.02189#实验与数据]。论文归因于 "H200 更适配小工作集" 但未解释机理。更可能的解释是 NV-HBI 双 die 互联引入的 NUMA 效应——尽管对软件透明，但跨 die 访问的额外延迟在小工作集 (fit in single die L2) 场景中无法被 pipeline 掩盖。

对 2603.15031 (Attention Residuals) 的含义：AttnRes 每层增加 ~5.5d/token I/O [ref:2603.15031#Limitations]。在 Blackwell 的双 die 架构上，如果 block cache 分布在两个 die 上，online softmax merge 的跨 die 访问可能进一步放大 I/O 开销——AttnRes 的 "< 4% training overhead" 数字可能在 Blackwell 上不再成立。

3.3 DE 的 output-bounded 设计限制了模型压缩 ROI #

DE 输出吞吐稳定在 170-220 GB/s，不随压缩率提升 [ref:2512.02189#实验与数据]。对高压缩率数据 (245×)，compressed input 仅 0.85 GB/s。这意味着 DE 对 中等压缩率 (2-4×) 最有价值，对极端压缩反而浪费 input 带宽。

对 2507.20534 (Kimi K2) 的 MoE expert 权重压缩：如果使用 DE 存储压缩后的 expert 权重以减少 HBM 占用，output 带宽上限 (220 GB/s) 会成为 expert activation 的新瓶颈——远低于 HBM3e 的 8 TB/s 峰值。DE 更适合数据库/检索场景，而非模型权重解压。

3.4 FP64 被排除在 TMEM/tcgen05 之外 #

FP64 DGEMM 不走 tcgen05.mma 或 TMEM 路径，使用独立的 DMMA 单元 [ref:2512.02189#实验与数据]。虽然 FP64 吞吐因翻倍单元获得 1.32× 加速，但无法享受 TMEM 的延迟和带宽优势。论文未讨论这一 gap 是否会在后续架构中弥合——这对科学计算社区是一个长期关切。

§5 未探索方向 #

5.1 TMEM + 线性注意力 kernel 协同设计 #

2510.26692 的 KDA 状态更新 $S_t = (I - \beta_t k_t k_t^\top) \cdot \text{Diag}(\alpha_t) \cdot S_{t-1} + \beta_t k_t v_t^\top$ [ref:2510.26692#Core Contribution] 中，$S_{t-1}$ 可驻留 TMEM 而非 RF 或 SMEM。结合 2512.02189 报告的 TMEM 16 TB/s read bandwidth [ref:2512.02189#实验与数据]，状态矩阵的 read-modify-write 可在 TMEM 内闭环完成。这需要将 KDA 的 Triton chunkwise kernel 重写为 tcgen05 PTX——目前无人尝试。

5.2 FP4 推理 + massive value 保护的硬件感知量化 #

2502.01563 证明 massive values 对上下文理解至关重要 [ref:2502.01563#§2 Q1 / Q2 / Q3]，但其量化分析仅覆盖 W4A16 和 W8A8 [ref:2502.01563#实验与数据]。2512.02189 提供了 FP4 的硬件性能数据 (7700 TFLOPS, +8.2% PPL) [ref:2512.02189#实验与数据]。一个自然的混合策略是：对 Q/K 的 massive value 维度使用 FP8 (QMMA)，non-massive 维度使用 FP4 (OMMA)，利用 Blackwell 的 per-instruction precision routing。这需要 CUTLASS 层面的 mixed-precision tile 支持。

5.3 DE 加速 MoE expert offloading #

2507.20534 (K2) 的 384 experts × 44M params/expert ≈ 17B routed params [ref:2507.20534#Architecture]，在推理时仅 top-8 active (32B)。对于 memory-constrained 部署，inactive experts 可压缩存储在 HBM 并通过 DE 按需解压。2512.02189 报告 LZ4 DE 吞吐 173 GB/s [ref:2512.02189#实验与数据]，一个 44M expert (BF16 ≈ 88MB) 的解压延迟约 0.5ms——可能满足 per-token expert switching 的延迟要求。但 output-bounded 特性意味着 2× 以上压缩率下 input 吞吐急剧下降，需要精心选择压缩格式。

5.4 多模态 chained attention 的 TMEM pipeline #

2503.20215 和 2509.17765 的 Thinker-Talker 架构中，Thinker 的 attention block (Q×K→softmax→V) 是 chained GEMM 的典型场景 [ref:2503.20215#架构 / 方法图]。2512.02189 估算 chained GEMM 中间结果驻留 TMEM 可每 SM 避免 ~12 TB/s 数据搬运 [ref:2512.02189#实验与数据]。结合 Qwen3-Omni 的 MoE Thinker (30B-A3B) [ref:2509.17765#§2 Q1 / Q2 / Q3]，TMEM-aware attention kernel 可能将 234ms 首包延迟进一步压缩——前提是 MoE 路由的 all-to-all 通信不成为新瓶颈。

5.5 AttnRes 深度缓存的 TMEM 驻留 #

2603.15031 的 Block-AttnRes 需要缓存 N-1 个完成的 block 表示用于 online softmax merge [ref:2603.15031#核心壁垒]。在 Kimi Linear (27 层, block_size=6) 中约 9 个 block，每个 block 表示为 [B, T, 2304]——对长序列，缓存大小可观。TMEM 的 256KB/SM 不足以存储完整 block cache，但可以存储 当前 partial block + 上一个 finalized block，将 online merge 的 V stack + softmax 操作保持在 TMEM 带宽内而非走 L2。这需要 AttnRes 的 forward pass 重构为"两块交替 TMEM buffer"模式。
