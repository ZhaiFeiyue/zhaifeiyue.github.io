NVIDIA H100 Tensor Core GPU Architecture White Paper

Hardware   NVIDIA  ·  2022-03  ·  arXiv ↗
gpu-architecturetensor-coreh100hopperfp8transformer-engine
Summary

Core Contribution

第四代Tensor Core引入FP8精度和Transformer Engine自动精度管理，配合TMA硬件加速和Thread Block Clusters编程模型，实现LLM训练30x加速。

Summary

H100基于TSMC 4N定制工艺(非标准N4)，die面积814mm²，集成80B晶体管。SXM5版本配备132个SM、16,896个CUDA Cores(128/SM，相比A100的64/SM翻倍)和528个第四代Tensor Core。最核心创新是FP8 Tensor Core——支持E4M3和E5M2两种FP8变体，峰值吞吐达1,978.9 TFLOPS(3,957.8 sparse)，相比A100 FP16 TC的312 TFLOPS提升6.3倍。与FP8配套的Transformer Engine是软硬件联合设计——在每一层前向/反向传播中自动选择FP8或FP16精度，通过per-tensor动态缩放因子(dynamic scaling factor)维持训练收敛性，使用户无需手动调整混合精度策略。

数据搬运方面，H100引入TMA (Tensor Memory Accelerator)——一个专用DMA引擎，可直接将多维tensor从global memory搬运到shared memory(或反向)，支持任意stride、padding和坐标变换，将原本需要数十条指令的地址计算和数据拷贝压缩为单条TMA指令，大幅降低数据搬运的指令开销和寄存器压力。编程模型上新增Thread Block Clusters层级——允许多个Thread Block跨SM协作，配合Distributed Shared Memory实现SM间直接load/store shared memory内容(无需经过L2)，这对GEMM的tile分解和通信-计算overlap至关重要。

互联方面，NVLink 4.0提供18条链路共900GB/s(A100的1.5x)。更革命性的是NVLink Switch System——通过外部NVLink Switch芯片实现最多256个GPU的NVLink域，突破了DGX 8-GPU的NVLink拓扑限制，为大规模模型并行(TP/EP跨节点)铺平道路。

Limitations

  • 80GB显存容量与A100相同，在LLM推理场景不足(Llama-2 70B FP16需>140GB)——H200通过升级到141GB HBM3e部分缓解
  • 700W TDP使散热设计成为挑战，SXM5需要专用散热方案
  • FP8训练需要Transformer Engine支持，框架集成和模型适配有学习曲线
  • Thread Block Clusters编程模型较复杂，需CUDA Toolkit 12+和新编程范式
  • NVLink Switch System需要额外硬件(NVSwitch芯片)，部署成本高
  • Distributed Shared Memory带宽有限，不适合大量跨SM数据交换
  • FP8精度对某些模型(尤其是fine-tuning小数据集)可能不够稳定

Infrastructure Impact

  • 定义LLM训练基础设施: H100成为GPT-4/Claude/Gemini等大模型训练的主力GPU
  • FP8生态推动: Transformer Engine推动PyTorch/JAX原生FP8支持，催生了大量FP8量化训练/推理研究
  • DGX H100系统: 8x H100 SXM5 + 4x NVSwitch，系统FP8算力达15.8 PFLOPS
  • HGX H100: OEM平台标准，AWS p5、Azure ND H100、GCP a3等云实例基于HGX H100
  • 256-GPU NVLink域: NVLink Switch System使TP/EP可扩展到节点间，改变了大规模并行策略
  • 推动vLLM/TensorRT-LLM发展: H100的FP8能力直接推动了推理引擎的FP8优化路径
  • MLPerf统治: H100/DGX H100在MLPerf Training v3.0+中多项打破记录
Details

Deep Analysis (hardware)

1. Architecture Overview

H100基于GH100 GPU，采用TSMC 4N定制工艺——这是NVIDIA与TSMC合作的定制节点，基于N4但针对HPC GPU进行了优化(更高密度标准单元库和更优的大die良率)。完整GH100 die包含8个GPC(Graphics Processing Cluster)、72个TPC(Texture Processing Cluster)、144个SM和576个Tensor Core。H100 SXM5产品启用132个SM(禁用12个用于良率优化)、528个TC。每个GPC包含9个TPC，每个TPC包含2个SM——这是与A100相同的GPC→TPC→SM层级结构。

3. SM / Tensor Core Deep Dive

  • Full GH100: 8 GPCs, 72 TPCs, 144 SMs, 576 4th-gen Tensor Cores
  • H100 SXM5 Product: 132 SMs, 528 Tensor Cores
  • SM内部: 4个Processing Block, 每Block:
  • 32 FP32 CUDA Cores (16 FP32 + 16 FP32/INT32 dual-issue)
  • 1个4th-gen Tensor Core
  • 1个Warp Scheduler + Dispatch Unit
  • 8 LD/ST units, 4 SFU
  • 第四代Tensor Core增强:
  • 新增FP8 (E4M3 + E5M2) 支持，吞吐为FP16的2x
  • TF32吞吐相比A100 3rd gen TC提升~2.6x (考虑SM数增加)
  • 保持FP64 TC支持
  • Tensor Core矩阵尺寸:
  • FP16/BF16: m16n8k16 (warp-level MMA)
  • FP8: m16n8k32 (K维度翻倍)
  • TF32: m16n8k8
  • Warp Group MMA (WGMMA): 新指令，4个warp协作执行更大矩阵运算，提升TC利用率

4. Peak Performance Derivation

FP8 Tensor Core:

  • 每TC: FP8 throughput = 512 FMA/cycle = 1,024 FP8 FLOPS/cycle
  • 每SM: 4 TC × 1,024 = 4,096 FP8 FLOPS/cycle
  • 全芯片 (132 SM): 132 × 4,096 = 540,672 FP8 FLOPS/cycle
  • Boost Clock: ~1.83 GHz (SXM5)
  • 峰值: 540,672 × 1.83 × 10⁹ ÷ 10¹² = 989.4 × 2 = 1,978.9 TFLOPS

(每个FP8 FMA = 2 FLOPS)

实际上更直接的推导:

  • FP16 TC = 989.4 TFLOPS (和A100的推导方法相同，但SM更多、频率更高、每TC吞吐更大)
  • FP8 TC = 2 × FP16 TC = 1,978.9 TFLOPS (FP8操作数宽度减半，吞吐翻倍)
  • Sparsity 2x: FP8 sparse = 3,957.8 TFLOPS

FP64 TC:

  • 66.9 TFLOPS = 2 × FP64 CUDA (33.5 TFLOPS)
  • vs A100: 66.9 / 19.5 = 3.43x

5. Memory Subsystem

  • HBM3: 80GB, 5 stacks, 3,350 GB/s (3.35 TB/s)
  • 首颗采用HBM3的GPU (JEDEC HBM3 standard)
  • vs A100 HBM2e: 带宽1.64x, 容量不变
  • H200升级到141GB HBM3e @ 4.8 TB/s
  • L2 Cache: 50MB
  • vs A100的40MB: +25%
  • 分为多个partition，MIG模式下各实例获得独立切片
  • Shared Memory / L1: 256KB per SM
  • vs A100的192KB: +33%
  • 支持更大的GEMM tile，配合WGMMA和TMA可实现更高TC利用率
  • 配置选项: 最多228KB shared memory
  • Register File: 256KB per SM (与A100相同)
  • TMA加速: Tensor Memory Accelerator直接搬运tensor到shared memory，省去地址计算指令

6. Transformer Engine

  • 核心思想: 在Transformer的每一层(Linear/Attention)动态选择FP8或FP16精度
  • 工作流程:

1. 前向传播: 输入FP16/BF16 → 量化到FP8 (E4M3) → TC计算 → 输出FP16/BF16

2. 反向传播: 梯度FP16 → 量化到FP8 (E5M2, 更大动态范围) → TC计算

3. Per-tensor scaling: 每个tensor维护一个FP32 scaling factor，动态跟踪值分布

4. Delayed scaling: 基于前几个iteration的统计量预测当前最佳scale

  • 硬件支持: TC内部集成FP8↔FP16转换逻辑，零开销精度转换
  • 精度保证: 主权重(master weights)和优化器状态始终保持FP32
  • 框架集成: NVIDIA提供transformer_engine Python库，集成到PyTorch/JAX

7. TMA (Tensor Memory Accelerator)

  • 问题: 传统CUDA中，从global memory加载tensor到shared memory需要：
  • 计算多维索引 → 物理地址(多条指令)
  • 逐元素或逐向量拷贝(多条load/store)
  • 处理boundary padding
  • 总计可能需要50+条指令
  • TMA方案: 单条异步指令完成整个tensor tile的搬运
  • 支持1D~5D tensor
  • 支持任意stride
  • 硬件处理边界检查和zero-padding
  • 异步执行，与计算重叠
  • 支持shared memory ↔ global memory双向
  • 性能影响: 减少数十条地址计算指令 → 释放register和warp scheduler资源 → 间接提升TC utilization

8. Thread Block Clusters & Distributed Shared Memory

  • Thread Block Cluster: CUDA编程模型新层级
  • 层级: Thread → Warp → Block → Cluster → Grid
  • 一个Cluster包含多个Thread Block，调度在相邻SM上
  • Cluster内的Block可以通过Distributed Shared Memory直接通信
  • Distributed Shared Memory (DSMEM):
  • 一个SM可以直接load/store另一个SM的shared memory
  • 无需经过L2 Cache
  • 延迟低于L2路径
  • 对GEMM分解: 不同SM处理不同tile，可直接交换中间结果
  • 应用场景: Flash Attention、GEMM split-K、跨SM reduction

9. Interconnect

  • NVLink 4.0:
  • 18条链路 (A100为12条)
  • 每链路: 50 GB/s双向 (25 GB/s per direction)
  • 总带宽: 18 × 50 = 900 GB/s 双向
  • 信号速率: 50 Gbps PAM4 per lane
  • NVSwitch 3.0: 第三代NVSwitch芯片
  • 每芯片: 64个NVLink 4.0端口
  • 交换带宽: 3.2 TB/s
  • 支持in-network计算(AllReduce in switch)
  • NVLink Switch System (全新):
  • 外部NVLink Switch机箱，通过铜缆或光缆连接多个DGX H100
  • 实现最多256个GPU的NVLink域
  • 意义: TP/EP可以跨节点扩展而不需要走InfiniBand
  • PCIe Gen 5 x16: 64 GB/s per direction (128 GB/s bidirectional)
  • DGX H100: 8x H100 SXM5 + 4x NVSwitch 3.0
  • 任意GPU对: 900 GB/s
  • 系统FP8算力: 15.8 PFLOPS (31.6 PFLOPS sparse)

10. MIG Enhancements

  • 保持A100的最多7个GPU Instance设计
  • 新增Secure MIG: 支持Confidential Computing
  • 每个MIG实例可运行在TEE (Trusted Execution Environment)中
  • GPU显存加密
  • CPU-GPU通信加密
  • 对多租户云部署(如医疗、金融)提供硬件级安全保证
