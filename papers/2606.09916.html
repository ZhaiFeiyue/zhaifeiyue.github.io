相关论文 #

以机制维度看，相关工作可分成三组。第一组是 workflow 调度/排队侧：Autellix、Continuum、SAGA、HexAGenT、Orla，核心在于把调度单位从 request 提升到 program/workflow，并借助 session affinity、TTL、online DAG 风险排序等手段减少跨轮等待和跨机抖动 [2502.13965] [2511.02230] [2605.00528] [2605.16637] [2603.13605]。第二组是 KV 保留/淘汰策略侧：PBKV 与 SideQuest 都在做"语义或预测驱动"的缓存价值判断，而不是仅依赖 recency/attention 启发式 [2605.06472] [2602.22603]。第三组是多模型共享 prefill 侧：PrefillShare关注跨模型冗余 prefill 的消除，而非单模型会话内跨轮保留 [2602.12029]。

IntentKV 与这些工作共享同一问题域（agent 长轨迹导致 KV 读写瓶颈），但其切入点是"单会话内 token 保留 + 布局不搬移"二元共设：QueryMemory 负责跨 turn 相关性，dead-slot 负责不破坏 prefix 身份 [2606.09916]。

delta #

相对 Autellix/Continuum/SAGA/HexAGenT，IntentKV 的新增量不在 cluster 调度，而在 token 级 retained set 与 slot-map 语义的一体化：它直接解决"同一请求内部哪些 token 该留"和"留存后如何继续命中 radix prefix"两个耦合点，而前者多偏 request/program 排队与资源分配 [2606.09916] [2502.13965] [2511.02230] [2605.00528] [2605.16637]。

相对 PBKV，IntentKV 不做显式 workflow 未来调用预测与 host-GPU 预取，而是把跨 turn 意图压缩为 QueryMemory，并用零初始化残差头做小幅语义修正；这使其更靠近"在线轻改造 scorer + 原内核不变"路径 [2606.09916] [2605.06472]。相对 SideQuest，IntentKV 不引入并行辅助推理线程与删除指令协议，而是把决策嵌入同一推理链路，工程表面积更小，但语义可解释性弱于 SideQuest 的显式删除理由 [2606.09916] [2602.22603]。

相对 PrefillShare，IntentKV 是"单模型会话内跨轮保留"，PrefillShare 是"多模型共享 prefill"，两者正交且可叠加：前者减少每轮有效 live KV 与读取，后者减少跨模型重复 prefill [2606.09916] [2602.12029]。

可攻击面 #

第一，IntentKV 的核心收益高度绑定于 Qwen 系列与 BCP/FRAMES 式工具调用分布，跨骨干普适性尚未实证闭环；论文自己也指出非 Qwen 开源骨干在该 harness 下 agentic 行为不稳定，这会把"压缩策略增益"与"agent 能力差异"耦合 [2606.09916]。对比地，Autellix/HexAGenT/SAGA 更强调对上层 agent 逻辑透明，主张 backbone-agnostic 的系统层收益 [2502.13965] [2605.16637] [2605.00528]。

第二，dead-slot 方案依赖 slot-map/allocator/radix 细节配合，若 serving 栈不具备同等别名安全与前缀保护语义，迁移成本可能高于论文表述。Orla 与 Continuum 的收益更多来自控制面与 TTL 策略，对底层 attention kernel/slot-map 语义侵入较轻，工程迁移风险更可控 [2606.09916] [2603.13605] [2511.02230]。

第三，IntentKV 的监督信号来自 future tool arguments 的字符串匹配，可能低估隐式语义证据；SideQuest 与 PBKV 分别走"辅助语义推理"和"多步预测 + 退化界"路径，在错删鲁棒性叙事上更直接 [2606.09916] [2602.22603] [2605.06472]。

生态位 #

IntentKV 的最佳生态位是"单模型、多轮 tool-use、prefix 复用可用、但显存/带宽受限"的在线代理服务层：它在不改主模型权重与 attention kernel 的前提下，把 token 级压缩与 prefix 复用兼容性同时拉起来 [2606.09916]。在这个位点上，它比纯调度法（Autellix/HexAGenT/SAGA）更细粒度，比纯语义删改法（SideQuest）更轻量，比多模型共享法（PrefillShare）更聚焦会话内长轨迹 [2502.13965] [2605.16637] [2605.00528] [2602.22603] [2602.12029]。

可预期的组合式部署形态是三层叠加：上层用 HexAGenT/SAGA/Autellix 处理 workflow 与多租公平，中层用 Continuum 管 turn-gap TTL，下层用 IntentKV 做 token 留存与 layout-preserving eviction；多模型代理再叠加 PrefillShare 的共享 prefill [2605.16637] [2605.00528] [2502.13965] [2511.02230] [2602.12029]。
