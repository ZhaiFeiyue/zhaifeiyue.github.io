ThunderAgent: A Simple, Fast and Program-Aware Agentic Inference System #

Hao Kang, Ziyang Li, Xinyu Yang, Weili Xu, Yinfang Chen, Junxiong Wang, Beidi Chen, Tushar Krishna, Chenfeng Xu, Simran Arora | 2026-02 | https://arxiv.org/abs/2602.13692 Category: framework | Tags: agentic-inference, kv-cache-scheduling, program-abstraction, rl-rollout, tool-orchestration Read: 2026-04-18

Core Contribution #

ThunderAgent introduces an agentic program as a first-class scheduling unit that persists across multiple LLM calls and tool executions, enabling a program-aware scheduler (global waiting queue, state-aware pause/restore with a provably optimal exponential time-decay caching function) and a lifecycle-aware tool resource manager to eliminate KV-cache thrashing, cross-node memory imbalance, and tool-environment leakage in multi-turn agent serving and RL rollout.

Summary #

现代 Agent 工作流（coding agents、SWE-Agent、OpenHands、ToolOrchestra）由多轮 LLM 调用和外部工具执行交错组成。主流部署方式是把一个 LLM 推理引擎（vLLM / SGLang）松耦合地拼到一个通用工具编排器（Kubernetes）上。两个组件各自以 per-request 粒度调度，对整个 workflow 的拓扑、未来 reuse、工具生命周期毫无知觉。作者在 8×H100 上用 GLM-4.6 + SWE-Bench Lite 跑实测，发现这种组合方式在高并发下会发生三类系统性失败：(1) KV-cache thrashing——工具执行期间旧请求的 KV 被激进驱逐，工具返回后必须重跑整个历史 prefill，端到端延迟放大最高 7.14×；(2) 跨节点内存不均衡——vLLM KV-aware router 把同一 workflow 的所有请求钉到同一节点，由于 agent 上下文快速不可预测地增长，出现某些节点爆满、另一些节点空闲，90 分钟 rollout 中峰值内存差达 51%；(3) 工具生命周期无感——完成的 workflow 留下的 Docker sandbox / 端口占用不会被回收，磁盘线性上涨直至系统崩溃；同时每个新 workflow 必须等环境 ready 才能开始推理。

ThunderAgent 的解决方案是把整个 agent workflow 抽象成一个 Agentic Program 元组 $P=\langle \mathit{ID}, c, \mathcal{T}, \mathcal{L}, \tau, s\rangle$，包括程序 ID、上下文 token 数、所需工具环境集、后端放置位置、执行阶段（Reasoning / Acting）和调度状态（Active / Paused / Terminated）。基于这个抽象，scheduler 把调度问题公式化为一个 cost model：$\text{Cost}_{\text{total}} = \text{Cost}_{\text{decode}} + \text{Cost}_{\text{prefill}} + \text{Cost}_{\text{recompute}} + \text{Cost}_{\text{unused}} + \text{Cost}_{\text{caching}}$，目标是最大化 decode/prefill 两项生产性开销、最小化后三项浪费。具体机制包括：周期性 thrashing 检测（每 $\Delta t$ 查询各 DP backend 内存）、State-aware pausing（优先 pause 在 Acting 状态的 program，保留 Reasoning program 的 KV）、Shortest-first eviction（数学证明重计算代价 $\propto c_i^2$，故 pause 最短 program 最优）、时间衰减函数 $f(t)$（作者证明在 memoryless 的工具延迟假设下，最优衰减函数必为指数形式 $e^{-\lambda t}$）、Global program-aware waiting queue（跨 DP 节点共享队列，允许 paused program 在任何节点 restore 以缓解不均衡）、以及 生命周期感知的工具资源管理器（显式 release 信号驱动 GC、与 reasoning 重叠异步准备环境）。

实验使用 GLM-4.6 MoE 在 8×H100 上对比 vLLM+Kubernetes、Continuum、Autellix 等 SOTA 系统。在 Serving 场景，ThunderAgent 在 ToolOrchestra/SWE-Agent/OpenHands 三种 agent 上实现 1.48–3.58× 吞吐提升；在分布式 RL Rollout 场景实现 1.79–3.92× 提升（mini-SWEAgent: 375 → 672 req/h；OpenHands: 69 → 271 req/h）；磁盘占用节省 4.2×。Ablation 证明时间衰减函数和 $\Delta t$ 周期对结果稳健，主要吞吐增益来自 prefill/decode latency 的下降，而 tool resource management 额外贡献约 10% 延迟收益。

Key Findings #

  • KV-cache thrashing 是 agent serving 的头号瓶颈：per-request 调度完全忽视 workflow 内强 prefix 依赖，工具等待期 KV 被驱逐，导致重 prefill 放大端到端延迟 7.14×。
  • Session-pinning 的 KV-aware 路由在 agent 场景下反而有害：vLLM 把同一 workflow 钉在同一节点 → 两节点 DP 场景 90 min 内有 37 min 内存差 >20%，峰值 51%；SGLang 前缀路由在共享 system prompt 场景下会把几乎所有请求送到同一节点。
  • Shortest-first eviction 是最优 pause 策略（数学证明）：重计算成本 $\text{Cost}_{\text{recompute}} \propto c_i^2$，且目标函数 $x^2$ 严格凸 + super-additive，必然选择最短 context 程序可获全局最优。
  • 指数衰减是唯一合法的缓存衰减函数（理论结果）：在 memoryless（工具剩余时间与已等待时间独立）假设下，满足时间同质性和边界条件 $f(0)=1, f(\infty)=0$ 的所有 $f$ 必为 $e^{-\lambda t}$ 或离散几何 $x^{-k}$。
  • KV cache hit rate 并非万灵指标：在工具延迟不稳定的场景，Continuum 的 TTL pin 策略 hit rate 更高，但 caching cost 被放大，吞吐反而低于 ThunderAgent——证明生产性/非生产性内存占用的 tradeoff 必须显式建模。
  • PD 分离在 agent workload 下退化：把集群切成 prefill-only / decode-only 后，单侧 HBM 池变小，在更低并发就触发 thrashing，与 chatbot 场景下的优化直觉相反。
  • KV cache offloading (LMCache) 无法救场：PCIe 带宽不足以承接 agent 高频上下文切换，swap-in/out 抵消掉扩容收益。
  • Low-touch API：只需 3 处改动（请求体加 program_id、工具调用传 program_id、程序结束显式 POST /programs/release），OpenAI 接口完全兼容。

Limitations #

  • 硬绑定 OpenAI 接口：所有集成假设 LLM backend 提供 OpenAI 兼容的 chat_completion endpoint，自定义私有 API 或 streaming-only backend 需要额外 adapter。
  • 需要应用层显式 release：程序结束必须 POST /programs/release，若开发者忘记发送，仍会出现资源泄漏——相比纯被动 GC 不够"简单"。
  • Memoryless 假设在真实远程 API 上可能不成立：指数衰减的最优性依赖工具延迟 memoryless，但论文自己在 Appendix C 承认远程 API 呈重尾（p99 远大于中位数），最优衰减形式在重尾分布下是否仍是指数是 open question。
  • 未评估多模型混部：所有实验用单一 GLM-4.6 模型，多租户多模型场景的 program routing 未涉及。
  • 未对比更新的 agent 系统（如 MegaFlow / RollArt）的端到端数据**，仅在 Related Work 中定性提及。
  • $\lambda_{\min} = \lambda_{\max} = 1$ 的选择：论文说"共享 prompt 已提供 buffer"，但若 system prompt 短或 workflow 异质，无 hysteresis 窗口可能导致震荡。

Infrastructure Impact #

  • Algorithm: 为 async RL rollout 提供基础设施底座——rollout 吞吐 ≈ policy-lag 削减，直接提升 PPO/GRPO 类 agent-RL 训练的 convergence speed 和 final reward。
  • Kernel: N/A —— 不涉及 kernel/numerical format 改动；完全工作在调度层，与 FlashAttention / FP8 GEMM 正交。
  • Framework: 直接竞品/增强层。作为 vLLM/SGLang 的中间件，在其上包装 program-level 语义。可视为 Autellix、Continuum、Sutradhara 等 agent-serving 系统的下一代——把"进程表"细化为"程序元组 + 全局队列 + 证明最优的衰减函数"。
  • LLM: N/A（模型无关；任何 multi-turn + tool 的模型都可受益）。
  • Agent: 重要。为 agent 框架（SWE-Agent / OpenHands / ToolOrchestra）提供 throughput-optimal 的 runtime；显式建模了 agent 的双态（Reasoning / Acting）——今后 agent 框架设计会更明确地上暴露这两个状态给 runtime。
  • Code: 开源仓库（见 https://github.com/HaoKang-Timmy/ThunderAgent / arXiv 摘要同时提到 Agentic-Kinetics/ThunderAgent）——值得深入阅读 program_abstraction、global_waiting_queue、time_decay 三个关键模块的实现。

Inherited Phase 2 Deepening #

时代定位 (Era Positioning) #

2026 年的 LLM 推理系统栈中，chatbot serving 的 low-hanging fruits 已被 paged attention、continuous batching、chunked prefill、PD 分离、MoE EP、speculative decoding 等技术采摘干净。vLLM/SGLang/TRT-LLM 三巨头的性能差距缩小到 <20%。此时真正的瓶颈从单请求效率转向multi-turn agent workload 的调度语义——当一个 workflow 会在 LLM 与外部工具间往返 10–50 次、上下文从 2k 增长到 100k 时，过去所有"per-request 视角"的优化都失效。ThunderAgent 代表了推理系统进入 program-aware 深水区的第一批工作之一（与 2026 年前后的 MegaFlow、Sutradhara、Continuum 并列），其核心主张是：workflow 不能是 runtime 的无名气一等公民。

核心技术壁垒 (Core Technical Barrier) #

ThunderAgent 真正难以复制的不是"全局队列"或"state machine"——这些是工程量。壁垒是 §4.3.1 中那个时间衰减函数 $f(t)$ 的最优性证明，它把 engineer 可能随手写的"衰减 heuristic" 升格为唯一合法的指数形式（memoryless 假设下的 Cauchy 函数方程闭解）。这使得 ThunderAgent 在理论上击败所有非指数衰减的 TTL/LRU/热度分策略。实操意义是：同行再造类似系统时，如果选了别的衰减形式（如线性、多项式），论文作者可以直接用 Theorem E.1 解释为何他们次优。这是论文的"护城河"。

质疑假设 (Challenge Assumptions) #

  • Memoryless 假设可能过强：Appendix C 图 9 显示 HLE-search / HLE-enhance-reasoning 这类远程工具的延迟是重尾 + p99 拉到百秒级。如果延迟符合 Pareto 或 Log-Normal，条件概率 $P(T > t+\Delta | T > t)$ 会随 $t$ 变大（老化效应），此时最优衰减可能不是指数。论文没有在重尾数据上重新求解最优 $f$，这是一个可被后续工作攻击的点。
  • $\text{Cost}_{\text{recompute}} \propto c_i^2$ 依赖 chunked prefill 线性累积：如果 backend 使用非均匀 chunk（如按 MoE expert balance 动态切），二次关系可能退化为 $O(c_i^{1+\alpha})$，shortest-first 仍最优但比例系数变。
  • Global waiting queue 的延迟开销被默认为 0：跨节点决策每 $\Delta t$ 查询所有 backend，当 DP 扩到数百节点时，查询带宽和调度器本身的 CPU 处理能力可能成为新瓶颈——论文实验只到 2 节点 16 GPU。

设计绑定批判 (Design Binding Critique) #

ThunderAgent 强制绑定以下前提：

  1. backend 必须支持 pause/restore 语义：要求 vLLM/SGLang 暴露"驱逐某个 program 的 KV 并保留能力"——当前 vLLM 的 abort_request 可用但 SGLang 的类似接口未必成熟。
  2. 应用层必须正确发 release 信号：假设所有 agent 开发者都记得调 POST /programs/release，否则 tool resource GC 会退化为 timeout-based（和 K8s 没区别）。
  3. KV cache 粒度是整个 program 的 context：无法对 program 内部做细粒度 KV 驱逐（例如仅保留 system prompt 部分）。这让 ThunderAgent 与 Shadow-KV / Pensieve 等 sparse-KV 方案不正交。
  4. OpenAI 接口必须兼容：pure streaming 或多模态 embedded protocol 需要写 adapter。
  5. DP 而非 PD 架构：Fig 7b 已经承认 PD 分离和 ThunderAgent 不兼容。

    6. 生态影响追踪 (Ecosystem Influence) #

    ThunderAgent 在 2026 初发布，生态影响可观察以下路径：

    • 短期：vLLM / SGLang 会吸收其"global program queue + release signal"设计到 router 层（预计 vLLM v0.8+ 的 agent-router、SGLang gateway v2）。
    • 中期：agent 框架（SWE-Agent / OpenHands）将上暴露"reasoning vs acting"的显式状态字段到 OpenAI 兼容的 extra_body 中，形成事实标准。
    • 长期：program-aware 成为下一代 agent-serving system 的默认姿态，类似 2023–2024 paged attention 从可选优化变成默认项。
    • 横向竞争：与 MegaFlow（更重的多组件编排）、Sutradhara（orchestrator-engine co-design）形成"轻量 middleware vs 重型 co-design"两条路线。

    1. System Scope #

    • 主要目标：Agent workflow 的 多轮 serving 以及 RL rollout 两个场景的高吞吐推理服务。
    • 单机 vs 分布式：既适配单节点也适配多节点 DP（实验 1×H100 到 2×8H100，即 16 GPU）。
    • 在线 vs 离线：偏向吞吐优先（论文明确说 agentic workload 由 sustained throughput 决定；human-in-the-loop 由 tail latency 决定）——选择面向前者。
    • 工作负载：LLM agent（SWE-Agent / OpenHands / ToolOrchestra），模型用 GLM-4.6 MoE，涉及 coding、computer-use、scientific discovery、routing 四类任务。

    2b. Data Movement Hotspots #

    1. KV cache 在 Pause 时被丢弃（不做 offload），Restore 时需要重 prefill。这是每次 Pause/Restore 的主要成本，与 $c^2$ 成正比——论文选择"接受重算 + 精选 pause 对象"而非 offload（因为 PCIe 带宽不够）。
    2. 跨节点 program migration 的 state transfer：一旦 program 被 Pause，其 KV 被驱逐 → migration 实际上只迁移"元数据 + 待 prefill 的 context"，数据量极小，这是全局队列能工作的关键前提。
    3. Tool environment 准备期的 Docker 镜像拉取：数 GB 级别，与 LLM reasoning 重叠后 hidden（异步执行）。

      • Batch formation：仍然依赖底层 backend（vLLM continuous batching），ThunderAgent 在其上做 哪些 program 的请求被允许进入 backend 的决策。
      • Memory management：显式维护每个 backend 的 active_program_tokens 和 C_total；λ_max=1（实测共享 prompt 已经提供隐式 buffer）。
      • GPU 利用率：通过 pause acting program → 把 HBM 让给 reasoning program，保证 decode/prefill 生产性开销最大化。
      • 多租户：每个 program 对应一个 isolated tool env（Docker sandbox）；GC 保证 cross-program 不相互影响。
      • SLO-aware：论文未显式建模 SLO；专注 throughput 最大化。P99 latency 在 ablation 有间接显示。

      7c. Bottleneck Shift #

      
Baseline: thrashing-bound (KV 驱逐风暴)
  → 引入 program queue + state-aware pause: → 变为 memory-bound（仍受 C_total 限制，但稳定）
  → 引入 time-decay f(t):                    → 变为 tool-latency-bound（最优 trade-off 后由远程工具 p99 决定）
  → 引入 tool lifecycle GC + async prep:     → 变为 RL / workflow-logic-bound（真实业务瓶颈）

      剩余瓶颈：远程工具 p99 延迟 和 单 program 的 context 长度增长速度——这两者都是应用层问题，推理层已无进一步优化空间（论文在 §A.2 明示 offload 和 PD 都不可行）。

      7d. Baselines & Fairness #

      • 对比对象：vLLM + Kubernetes（工业默认）、Continuum（TTL pin）、Autellix（进程表跟踪）、vLLM + LMCache（offload）、vLLM + PD 分离。
      • 公平性：同模型同硬件同 benchmark，确实公平。但未对比 MegaFlow / RollArt（仅在 Related Work 讨论）。
      • ThunderAgent 是否存在 lose case？：Continuum 在短 + 可预测的 tool 调用下 hit rate 更高（Fig 5），但 throughput 仍低——作者已诚实披露。

      Open Questions #

      • 当远程工具延迟是重尾分布（非 memoryless）时，最优衰减函数 $f(t)$ 是否仍为指数？或需要 Pareto-aware 修正？
      • Shortest-first eviction 会不会让长 program 饥饿？是否需要公平性 guard？
      • ThunderAgent 与 sparse KV（ShadowKV / Pensieve）能否组合？program-aware + sub-program KV 裁剪的协同空间？
      • 当 DP 扩展到数百节点时，global waiting queue 的中心化调度器是否会成为新瓶颈？
      • 多租户多模型场景下，program 的 routing 在模型维度如何处理？
