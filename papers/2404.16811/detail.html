Make Your LLM Fully Utilize the Context (FilM-7B / IN2 Training)

Shengnan An, Zexiong Ma, Zeqi Lin, Nanning Zheng, Jian-Guang Lou (Microsoft / 西安交大 / 北大) | 2024-04 | https://arxiv.org/abs/2404.16811 Category: algorithm | Tags: long-context, lost-in-the-middle, data-synthesis, instruction-tuning, RoPE, evaluation Read: 2026-04-19

Core Contribution

用一个纯数据驱动的方案（IN2 训练）治好了 lost-in-the-middle 病：把 GPT-4 合成的"问题答案藏在长上下文中任意一个 128-token 段"的 1.75M 条 QA 数据，喂给 Mistral-7B-Instruct-v0.2 做 instruction tuning，得到 FilM-7B —— 在 32K 上下文中所有位置上的检索性能都接近平的，与 GPT-4-Turbo 同台竞争且不损害短上下文能力。

Summary

动机: 现代长上下文 LLM（Mistral、LongAlign、InternLM2 等）虽然训练窗口扩到 32K+，但普遍在"中间位置"的信息提取上失败 —— 即 lost-in-the-middle。作者提出一个反直觉的假设：这不是模型架构或注意力机制的锅，而是训练数据自身带有不可见的位置偏置。原因有二：(1) 自回归预训练的下一 token 损失被近距离 token 主导；(2) SFT/对齐阶段，system message 总是放在开头，让模型学到"重要信息在开头/结尾"。

方法: IN2 (Information-INtensive) 训练。从 C4 realnewslike 语料里，先把每篇文章切成 128-token 段（"最小信息单元"）；让 GPT-4-Turbo 针对单段（细粒度 awareness）或多段（多跳 reasoning）生成 QA pair；然后把"目标段(s)"+ 大量随机段拼成 4K~32K 的长上下文 Lᵢ，并通过 reject sampling 让长度均匀分布、目标段位置随机。最终数据集：1.1M 细粒度 + 0.3M 多跳 + 0.15M 短上下文 + 0.2M OpenOrca 混合 instruction tuning 数据，共 1.75M 条。Loss 只在 answer 部分；16 节点 ×8 张 A100 训练约 300 GPU-day。

结果: 在作者新设计的 VaL Probing 上（涵盖文档/代码/结构化数据 × 前向/后向/双向检索 6 种组合），FilM-7B 平均分 85.9，超过 GPT-4-Turbo (79.0)，且 min-max gap 仅 13.9（GPT-4-Turbo 32.1）。在 9 个真实 LongBench 任务上从 30.6 → 39.9，NarrativeQA F1 从 23.5 → 26.9。短上下文 8 个 benchmark 平均仅有微小波动（MMLU 59.3 → 59.2）。论证了"合成长上下文数据可以泛化到真实任务，且不损害基础能力"。

Key Findings

  • Lost-in-the-middle 是数据问题，不是架构问题 —— 仅靠数据 + instruction tuning 就能修复，不改注意力、不改 position encoding 即可（虽然 RoPE base 调到默认 10× 还能再增 ~5 分）。
  • 128-token 是合适的"最小信息单元" —— 既小到可以在 32K 中藏成"针"，又大到能形成完整问答事实。
  • VaL Probing 揭示 Needle-in-the-Haystack 高估了模型 —— LongAlign / InternLM2 在 NIH 上几乎满分，但在 VaL（特别是 backward / bi-directional 模式）上仍呈明显 U 形曲线。
  • Sliding window 与长上下文目标根本冲突 —— 不论是预训练还是 IN2 阶段加 4K SW，超出窗口距离的信息检索精度直接归零（Fig. 6）。
  • 训练时拉高信息密度需要更大的 RoPE base θ —— 在 IN2 训练中把 θ 从 1e6 提到 1e7（10×）能把 VaL avg 从 80.3 → 84.9。
  • 合成数据 → 真实数据的迁移成立 —— 训练数据全部由 GPT-4 + C4 生成，但真实 LongBench 任务（NarrativeQA、Qasper、HotpotQA 等）也获得 SOTA-级提升。

Limitations

每条 limitation 显式标注它攻击 §3.0 论证链的哪一步 (per-critique chain anchor 强制规则):

  • 依赖 GPT-4-Turbo 生成数据 [攻击 step 4 premise]：数据成本高且开源不友好；更关键的是论文未做 QA quality control 报告 — 如果 GPT-4 生成的 (q, a) 没有 highly specific 到对应 segment, IN2 就在训"错误 alignment", 整个 step 4 的"仅靠数据即可修"会被悄悄削弱。如果 backbone 已经接近 GPT-4 水平，IN2 数据可能会给 backbone 带来"distillation 上限"。
  • 只在 32K 长度内验证 [攻击 step 5 premise]：没有展示 64K/128K 表现，"IN2 能泛化到真实长任务"的 conclusion 在 32K 之外是未验证的；能否外推到更长仍是开放问题。
  • C4 realnewslike 单一数据源 [攻击 step 4 premise]：所有合成数据来自新闻类文本，对代码/数学/对话风格上下文的 IN2 数据多样性不够；VaL 上的 code/database 任务也是用合成上下文，与真实代码长仓库 Q&A 的分布差距可能仍大 — 这弱化了"数据驱动方案"的普适性论点。
  • 未报告训练曲线 / loss spike [范围外]：论文几乎没披露训练稳定性细节，复现者要踩多少坑无从评估；这是工程复现性问题, 不直接攻击论证链。
  • Reject sampling 把长度强制做成均匀分布 [攻击 step 5 premise]：真实工作负载（聊天、RAG）的长度分布是长尾，模型在极短样本上可能会有次优行为（论文未做相关分析）— 这意味着 step 5 的"不损害 base capability" conclusion 在长尾负载下可能站不住。
  • VaL 仍是合成上下文的"造句子" [攻击 step 5 premise]：文档任务从 arXiv abstract 抽句、代码任务从 StarCoder 抽函数，仍属"针在草堆"的简化版本，离真实多文档推理还有距离 — 这意味着 VaL 上的 +6.9pp over GPT-4-Turbo 这个"超越"数字含金量受限, step 5 的论证强度被削弱。
  • 短上下文回归被 understate [攻击 step 5 conclusion]：HellaSwag -4.5 / ARC-C -3.4 (Table §6b) 已经不是论文宣称的 "minor variance" — 这两项是常识/推理类标准 benchmark, 4-5pp 是显著退化。step 5 的"不损害 base capability"在常识推理这一类目下其实不成立, 论文的 framing 偏乐观。
  • 1.75M 数据规模无 scale ablation [攻击 step 4 premise]：作者给出 1.75M 条这一数字, 但没做"用 0.5M / 1M 是否够"的消融 — 数据驱动方案的最小必要规模未知, 这弱化了"数据是关键"的精确性 (究竟是"数据"还是"足够多的数据"才是关键)。
  • 复现成本对学术 lab 是不小的门槛 [范围外]：~\$60K-\$120K (GPT-4 调用 + 128×A100×56h) 是工程依赖, 不直接攻击论证链 (step 4 的 GPT-4 可被其他 strong LLM 替代是论证可塑性, 但论文未做)。

Infrastructure Impact

  • Algorithm: 提供了"修复 lost-in-the-middle 的标准 recipe"——后续 long-context SFT 都可以参考 IN2 的"reject sampling 让 gold position 均匀"+"细粒度+多跳 QA 二分"的设计。
  • Kernel: N/A — 不引入新的算子。
  • Framework: N/A — 标准 instruction tuning + FSDP；但提示训练框架在 reject sampling 阶段对长度均衡的 batching/packing 需求会变高。
  • LLM: 不改架构。提示后续 LLM 在 SFT 阶段把 system prompt 偶尔放到 context 中间也许有助于削弱位置偏置（论文未做但隐含建议）。
  • Agent: 直接利好长上下文 agent 应用 —— 让 LLM 真正能"读完整段历史"再决策，而不是只看末几条 message。
  • Code: 微软已开源 https://github.com/microsoft/FILM 包含数据生成代码、VaL 数据集与评估脚本。

2. Problem Formulation

目标函数（标准 instruction tuning + answer-only loss）：

$$\mathcal{L}_{\text{IN2}} = -\mathbb{E}_{(\mathcal{L}_i, q_i, a_i) \sim \mathbb{D}} \sum_{t=1}^{|a_i|} \log p_\theta\big(a_{i,t} \mid \mathcal{L}_i, q_i, a_{i,

其中 $\mathbb{D}$ 是 IN2 数据集，$\mathcal{L}_i$ 是 4K-32K 长上下文，$q_i$ 是问题，$a_i$ 是答案。关键的隐式约束 由数据构造保证：

$$\Pr[\text{position}(s_i) \in [k, k+\Delta] \mid |\mathcal{L}_i| = L] = \frac{\Delta}{L}, \quad \forall k \in [0, L-\Delta]$$

即"目标段位置在长上下文中近似均匀"。这是 IN2 的核心显式监督，也是和普通 long-context SFT 的根本区别。

假设: (a) GPT-4 生成的 QA 对真的 highly specific 到对应 segment（如果 GPT-4 出错，IN2 就训出"错误 alignment"）；(b) 短上下文能力不会被 catastrophic forgetting 抹掉（用 9% 短 QA + 11% OpenOrca 做"防遗忘 anchor"）；(c) 合成数据可以泛化到真实长文档（结果验证了这个假设）。

改进自: 隐式上对比 LongAlign（用真实长指令数据）、Fu et al. 2024（用 data engineering 平衡长度分布）。前者样本量小且 gold 位置非均匀；后者只解决长度分布、不解决位置分布。

3. Method Core


# IN2 数据生成（pseudo-code）
def build_in2_dataset(C, target_count, length_bins=[4K..32K, step=4K]):
    D = []
    bin_counters = {b: 0 for b in length_bins}
    while sum(bin_counters.values()) < target_count:
        Ci = sample(C)  # raw text from C4 realnewslike
        if 10gram_overlap(Ci, eval_set):  continue   # data contamination filter

        # Path A: fine-grained
        si = random_segment(Ci, n_tokens=128)
        qi, ai = gpt4_turbo(prompt=I_f, segment=si)

        # Path B: multi-hop
        segs = split(Ci, 128_tokens)
        qi, ai = gpt4_turbo(prompt=I_r, segments=segs)

        # 拼长度并 reject sample 让长度均衡
        L_target = pick_underrepresented_bin(bin_counters)
        rj_pool = sample_random_segments(C, n=L_target/128 - len(target_segs))
        Li = shuffle_concat(target_segs + rj_pool)
        if len(Li) ≈ L_target:
            D.append((Li, qi, ai)); bin_counters[L_target] += 1
    return D

# Training (standard SFT)
for batch in DataLoader(D ∪ ShortQA ∪ OpenOrca):
    L, q, a = batch
    logits = model(format("[INST] {L} {q} [/INST]"))
    loss = ce(logits[answer_mask], a[answer_mask])  # answer-only loss
    backward(); step()

Key insight 与 prior work 的区别:

  • prior: 把长上下文 SFT 数据直接拿去训，gold 位置受文章结构影响（往往集中在前/后）。
  • this: 把"信息位置"作为一个独立的 randomization 维度，硬塞进数据生成 pipeline。

敏感超参:

  • 128-token segment size: 太小 → GPT-4 没法生成有意义 QA；太大 → 长上下文里"针"太显眼。
  • 比例 (63% 单段 + 17% 多段 + 9% 短 + 11% 通用): 多段太多则 single-segment 检索退化，反之亦然。
  • RoPE base θ: 默认 1e6 → IN2 推荐 1e7（10×）。

复杂度: 训练阶段就是普通 SFT，O(L²) attention，单步 ~32K tokens × bs128。数据生成是大头：1.4M 条 GPT-4-Turbo 调用，按 2024 价格 (~\$0.01/1K input) 估算光 LLM 调用费 \$50K-\$100K 量级。

5. Convergence & Stability

  • 收敛保证: 标准 SFT，无任何 convergence theory。
  • 学习率 / batch size 行为: 论文给定 LR 1e-6 + bs=128，没有扫描；从经验上 bs 拉到 256/512 应该可以减少 noise。
  • 失败模式: (a) 如果 forget 短上下文能力 → 论文用 9% 短 QA + 11% OpenOrca 兜底；(b) 如果 GPT-4 生成的 QA 答案错了 → 模型学到错的 alignment（论文没有 quality control 报告）。
  • Curriculum / warmup: cosine + 3% warmup，未做 length curriculum（直接混合 4K~32K）。

6c. Strength & Weakness

Strengths:

  • 长上下文检索（VaL/Long-Bench）压倒性提升，归功于 IN2 数据。
  • 数学 (GSM8K +4.1, MATH +2.6) 反而上升 —— 推测是 multi-hop QA 训练带来的通用推理增强。

Weaknesses:

  • 常识/推理类 (HellaSwag -4.5, ARC-C -3.4) 略降 —— 这是 catastrophic forgetting 的痕迹，OpenOrca anchor 没完全兜住。论文承认 "almost comparable with minor variances"，但这两项 -4.5 / -3.4 已经不算 minor。
  • 总结类任务 (GovRep, QMSum, MultiNews) 几乎无提升 —— IN2 数据是 QA 风格，对长文摘要的格式 transfer 有限。

Suspicious / 缺失:

  • 没有 IFEval（instruction following 现代标准）—— 短上下文能力评估不全。
  • 没有 GPQA / MMLU-Pro —— 高难推理类没覆盖。
  • VaL 任务用 word-level recall（doc）+ exact-match（code/db），评估标准较宽松；如果换成 BLEU/ROUGE 可能差距会缩小。
  • 没有 Chinese / 多语种长上下文评估。

9. Position in Research Landscape

最相关的 5 篇前作:

  1. Lost in the Middle (Liu et al. 2024, TACL) —— 第一个 systematic 揭示 U 形位置偏置；本文是 follow-up 的 fix。
  2. Mistral-7B (Jiang et al. 2023) —— backbone，IN2 没改架构。
  3. LongAlign (Bai et al. 2024) —— 同样关注长上下文 SFT，但用真实数据；本文证明 synthetic + 强位置随机化 > real but biased。
  4. Data engineering for 128K context (Fu et al. 2024) —— 解决长度分布平衡；本文进一步解决 位置 分布平衡。
  5. Needle-in-the-Haystack (Kamradt 2023) —— 本文 VaL probing 直接对标并扩展。

    6. 下一步 enables:

    • 更长 context (128K, 1M): 原 IN2 框架天然可以扩展，只要 reject sampling 把 bin 拉到 128K/1M。
    • 多模态长上下文: 把 segment 换成 image patch / audio chunk，IN2 思路天然适用 video / long-audio LLM。
    • Agent 长 trajectory training: agent 历史天然就是"段拼接"，IN2 可以直接迁移训练 agent 的"长 trace 内任意位置 retrieval"。
    • 后续工作: NVIDIA RULER (2024)、DeepSeek-V2/V3 的长上下文 SFT 都借鉴了类似思路。

    • [step 4 premise] 单一 GPT-4-Turbo 数据源是否引入了风格 bias？同样的 IN2 pipeline 换成 Llama-3 70B 或 DeepSeek V3 生成 QA, 还能保持 VaL 85.9 / Gap 13.9 这个数字吗? 如果能, step 4 的"GPT-4 不是 essential, IN2 框架本身才是 key" 论点才彻底成立。

    • [step 4 premise] Multi-hop 比例 17% 是不是最优？为什么不是 50/50？没有比例 ablation, IN2 数据组合的"必要 vs 充分"边界不明确。

    • [step 3 premise] 论文把 LITM 归因于"训练数据位置偏置"两个机制 (next-token loss 近距离主导 + system message 开头放置), 但没做 controlled study 单独验证哪个机制贡献更大。如果只修一个 (比如随机化 system message 位置) 是否就够? 这能进一步精确化 step 3 的 attribution。
    • [范围外] 工程依赖: 把 GPT-4-Turbo 替换成开源 strong LLM (Llama-3 70B / DeepSeek V3 / Qwen 2.5 72B) 的成本/质量曲线是什么样? 这是工程化必答题, 不直接削弱论证链。
    • [范围外] IN2 数据集本身是否会被开源? 论文只放了构造代码和 VaL probing 数据集 (§8 Reproducibility), 1.75M 数据未公开 — 这是社区复现门槛, 不构成对论证链的攻击。
    • [step 5 conclusion 的边界] 在 chat / RAG 等长度长尾的真实负载下, "均匀长度分布的 IN2 训练"是否仍 dominant? 论文未做 ShareGPT / 真实 RAG 流量评估, 这是 step 5 conclusion 的一个未覆盖盲区。
