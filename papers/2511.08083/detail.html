HipKittens: Fast and Furious AMD Kernels

William Hu, Drew Wadsworth, Sean Siddens, Stanley Winata, Daniel Y. Fu, Ryan Swann, Muhammad Osama, Christopher Ré, Simran Arora | 2025-11 | https://arxiv.org/abs/2511.08083 Category: kernel | Tags: AMD GPU, tile-based DSL, CDNA4, GEMM, attention kernel Read: 2026-04-16

Core Contribution

HipKittens (HK) is the first systematic C++ embedded tile-based programming framework for AMD GPUs, providing reusable primitives (explicit register scheduling, bank-conflict-free swizzles, 8-wave ping-pong scheduling, chiplet-aware cache optimization) that match or outperform AMD's hand-optimized assembly kernels across GEMM, attention, and memory-bound workloads on CDNA3/CDNA4 hardware.

Summary

AMD GPUs offer competitive peak compute (MI355X: 2.5 PFLOPs BF16, 5.0 PFLOPs FP8) and memory bandwidth (8 TB/s) versus NVIDIA B200, but peak-performance AMD kernels require raw assembly—an approach that doesn't scale across the breadth of AI workloads. The "CUDA moat" persists not due to hardware deficiency but due to the lack of mature, reusable programming abstractions for AMD. Compilers like Triton on AMD struggle with register lifetime tracking and fail to lower to the most performant intrinsics, while existing C++ DSLs (ThunderKittens, CuTe DSL, Gluon) only target NVIDIA hardware.

HipKittens addresses this gap by extending the tile-based programming paradigm from ThunderKittens to AMD GPUs. The key insight is that the tile abstractions (typed tiles in register/shared/global memory, PyTorch-style bulk operators like mma, exp, add) are general across vendors, but the algorithms instantiating those abstractions must be fundamentally rethought for AMD. HK introduces: (1) developer-controlled register scheduling that bypasses HIPCC's limitations on AGPR usage, (2) heterogeneous swizzle strategies for AMD's non-compositional matrix instruction layouts, (3) an 8-wave ping-pong scheduling pattern (and a 4-wave interleave variant) that replaces NVIDIA's wave-specialization paradigm, and (4) a chiplet-aware XCD grid scheduling algorithm (Algorithm 1) that jointly optimizes L2 and LLC cache reuse.

Validated on MI325X (CDNA3) and MI355X (CDNA4), HK kernels compete with AMD's hand-optimized assembly (AITER) on BF16/FP8 GEMM and attention forward, while outperforming all baselines by 1.2–2.4× on workloads where assembly support is incomplete (d=64 attention, GQA backwards, memory-bound kernels). HK consistently outperforms Triton by 1.3–3.0× on GEMM and 2× on attention. Correctness is validated by matching perplexity when pretraining Llama 1B and BERT 110M.

Key Findings

  • Wave specialization (producer-consumer) underperforms on AMD: achieves only 80% of peak BF16 GEMM performance due to static register allocation—producer waves waste registers without contributing computation, limiting output tile size and arithmetic intensity
  • 8-wave ping-pong schedule (2 waves/SIMD alternating compute/memory roles) suffices to match assembly-level performance for BF16 GEMM, FP8 GEMM, and attention forward — remarkably simple yet effective
  • Explicit register pinning (bypassing HIPCC to directly use AGPRs as MFMA inputs) closes the gap with assembly kernels on attention backward — 855→1024 TFLOPS improvement
  • AMD matrix instruction layouts lack NVIDIA's compositional structure (no single core matrix building block), requiring per-instruction swizzle strategies with undocumented phase orderings
  • Chiplet-aware XCD scheduling (Algorithm 1) improves GEMM performance by up to 19% over naive row-major by jointly optimizing L2 (75-79% hit rate) and LLC (55-93% hit rate) cache reuse
  • HK outperforms all AMD baselines on GQA backwards by 1.8–2.5× — a workload where assembly doesn't scale

Limitations

  • Only validated on AMD CDNA3 (MI325X) and CDNA4 (MI355X); no RDNA or other architectures
  • 4-wave interleave pattern significantly increases code complexity (4× more LoC) for modest performance gains
  • No end-to-end inference or training throughput benchmarks — only kernel microbenchmarks
  • Chiplet scheduling algorithm requires manual tuning of W and C parameters per problem size
  • FP6 support is preliminary (Appendix F case study only)
  • No comparison with NVIDIA on identical model architectures at system level
  • Register pinning mechanism bypasses compiler safety checks — may be fragile across ROCm versions

Infrastructure Impact

  • Algorithm: Enables rapid prototyping of new AI algorithms on AMD hardware by providing familiar tile-based abstractions; attention backward support unlocks full training pipelines on AMD
  • Kernel: Establishes the first systematic set of AMD-specific kernel optimization principles (8-wave scheduling, chiplet-aware cache, register pinning) that can serve as the foundation for future AMD kernel libraries
  • Framework: HK kernels can be integrated into PyTorch via Python bindings, potentially replacing suboptimal SDPA/Triton backends for AMD; paves the way for vendor-agnostic framework backends
  • LLM: Competitive GEMM and attention performance on MI355X means LLM training/inference workloads can achieve near-peak hardware utilization on AMD, reducing the CUDA moat
  • Agent: Lower kernel latency on AMD could enable real-time agent inference on AMD hardware, though this is not directly demonstrated
  • Cluster: Chiplet-aware scheduling insights apply to multi-GPU AMD clusters where hierarchical cache behavior impacts collective communication patterns

Deep Analysis (kernel)

1. Target Operation

HipKittens targets the core computational primitives of modern AI workloads:

Primary operations:

  • GEMM (General Matrix Multiplication): BF16 and FP8, square matrices M=N=K from 1024 to 14592
  • Attention (forward and backward): Multi-head (MHA) and Group-query (GQA), causal and non-causal, head dimensions 64 and 128, sequence lengths 1024-8192
  • Memory-bound kernels: Fused dropout-residual-LayerNorm, Rotary Positional Encoding (RoPE)

Data types: BF16, FP8 (MXFP8), FP6 (preliminary), FP32 accumulators

Theoretical arithmetic intensity:

  • BF16 GEMM (M=N=K=8192): ~4096 FLOPs/byte — deeply compute-bound
  • FP8 GEMM: ~8192 FLOPs/byte — even more compute-bound
  • Attention forward (d=128, seq=4096): ~256 FLOPs/byte — compute-bound
  • Attention backward: highly register-intensive, requires mixed MFMA shapes
  • LayerNorm/RoPE: memory-bound (arithmetic intensity < 10)

Roofline position:

  • GEMM and attention sit firmly in the compute-bound regime on MI355X (2.5 PFLOPs BF16 peak, 8 TB/s bandwidth)
  • Memory-bound kernels (LayerNorm, RoPE) are bandwidth-limited

2. Hardware Model

Target architectures:

  • AMD CDNA4 (MI355X, gfx950): 256 CUs, 8 XCDs (chiplets), 4 SIMDs/CU, 512 registers/SIMD
  • AMD CDNA3 (MI325X, gfx942): similar organization with 38 CUs/XCD

Hardware units exploited:

  • Matrix Cores (MFMA): 16×16×32 (BF16), 32×32×16 — primary compute unit
  • Vector ALU (VALU): for online softmax operations (max, exp2, accumulate) in attention
  • LDS (Local Data Share): shared memory per CU, used for tile staging
  • Direct HBM→LDS loads (buffer_load_dword): bypass register file, analogous to NVIDIA TMA

Memory hierarchy exploitation:

  • Registers: 512 per SIMD, split into 256 VGPRs + 256 AGPRs (single wave/SIMD). HK enables explicit pinning to control which tiles live where
  • LDS (shared memory): Per-CU, with 64 banks (4-byte wide for ds_read_b128). HK provides per-instruction swizzle patterns to avoid bank conflicts
  • L2 cache: Per-XCD (4MB per XCD), not shared across chiplets. XCD scheduling (Algorithm 1) maximizes reuse
  • LLC: Shared across all XCDs, sits between L2 and HBM. Joint L2+LLC optimization needed
  • HBM: 288 GB, 8 TB/s bandwidth

Theoretical peak throughput (MI355X):

  • BF16: 2.5 PFLOPs
  • FP8: 5.0 PFLOPs
  • FP6/FP4: 10.1 PFLOPs
  • Memory bandwidth: 8.0 TB/s

5. Numerical Considerations

  • Uses FP32 accumulators for BF16 and FP8 GEMM — standard for numerical stability
  • Online softmax in attention forward with proper rescaling (max-subtract, exp2, accumulate)
  • Validates correctness by pretraining Llama 1B and BERT 110M to matching perplexity over 10B tokens
  • No special handling of NaN/Inf/subnormals discussed
  • FP6 support uses ds_read_b96 instruction, which has different banking behavior than BF16 instructions

8. Key Source Code Walkthrough

8a. Core Kernel Source Analysis

The paper provides code listings in Appendix E for the GEMM and attention forward kernels. The key architectural pattern is the 8-wave ping-pong:


// 8-wave ping-pong GEMM hot loop (conceptual, from paper):
// Each SIMD has 2 waves. Waves alternate compute/memory roles.

// Wave group A (waves 0-3, one per SIMD):
if (wave_in_group_A) {
    __builtin_amdgcn_s_setprio(3);  // high priority for compute
    hk::mma(acc, a_tile, b_tile);    // bulk MFMA over tiles
    barrier.arrive_and_wait();        // swap roles
    __builtin_amdgcn_s_setprio(0);
    hk::load(smem_a, gmem_a_next);   // prefetch next tiles
    hk::load(smem_b, gmem_b_next);
    barrier.arrive_and_wait();        // swap back
}

The key insight: only 48 lines of code for the FP8 GEMM hot loop (Table 3), compared to 183 lines for 4-wave interleave.

8b. Pseudocode Annotation — Algorithm 1 (XCD Swizzle)

Grid level (entire GPU):

  • Lines 3-6: Flatten 2D grid → linear block ID. Compute full cycle boundaries (nXCD × C aligned)
  • Lines 10-14: XCD grouping — de-interleave round-robin assignment to group C consecutive blocks per XCD. This is a register-level computation per thread block. Memory level: purely index computation, no memory access.
  • Lines 15-23: Windowed traversal — map linear block IDs to 2D (row, col) using windows of height W. Row = fast index (within window), Col = slow index. This controls L2 locality (blocks in same window share rows of A matrix).

Synchronization: None — purely a remapping of block IDs, computed independently per thread block.

Compute vs. memory: Entirely compute (integer arithmetic), negligible cost. The optimization impact is on subsequent GEMM tile loads.

8c. Data Layout & Memory Access Patterns

Register tiles:

  • Default: smallest MFMA shape (16×16×32 for BF16) for maximal scheduling control
  • Each thread owns elements in a pattern dictated by the MFMA instruction (Figure 3) — unlike NVIDIA, AMD has no compositional core matrix structure
  • Row-major vs column-major layout determines which MFMA instruction is used for loading

Shared memory (LDS):

  • 64 banks (4-byte wide) for ds_read_b128
  • Per-instruction phase orderings (Table 5): ds_read_b128 = 4 phases × 16 threads, ds_read_b96 = 8 phases × 8 threads, ds_write_b64 = 4 phases × 16 threads
  • Swizzle strategy: XOR-based (offset ^= ((offset % 512) >> 7) << 3) applied to HBM addresses (not LDS addresses like NVIDIA)
  • No single universal swizzle — different instructions require different patterns. HK identifies commonly co-occurring layouts and provides joint swizzles

Figure 4: Bank-Conflict-Free Swizzle Pattern

解读: 此图展示了16×32 BF16 shared memory tile的swizzle策略。左侧是未swizzle的布局，存在2-way bank conflict（同一phase的线程访问同一bank）。右侧是swizzle后的布局——从第8行开始，前8列和后8列交换，消除了ds_read_b128的bank conflict。关键在于，同一个swizzle同时也消除了ds_read_b64_tr_b16（列major读取）的bank conflict，这使得同一块shared memory可以同时服务于行和列两种register layout的读取——这对attention backwards这类需要混合MFMA形状的kernel至关重要。

Register allocation:

  • 512 registers/SIMD, split 256 VGPR + 256 AGPR at single occupancy (1 wave/SIMD)
  • With 2 waves/SIMD (8-wave pattern): 128 VGPR + 128 AGPR per wave
  • Attention backward uses pinned registers to bypass HIPCC's restriction on AGPR→MFMA input
