TensorHub: Scalable and Elastic Weight Transfer for LLM RL Training

Chenhao Ye et al. (ByteDance Seed, UW-Madison) | 2026-04 | https://arxiv.org/abs/2604.09107 Category: framework | Tags: RL-training, weight-transfer, RDMA, reference-oriented-storage, fault-tolerance, elastic-scaling, cross-datacenter, pipeline-replication

§1 TL;DR

TensorHub 提出 Reference-Oriented Storage (ROS) 抽象，消除 RL 权重传输中的数据所有权开销——引用服务器仅追踪轻量级引用，数据在 GPU 间 RDMA 直传无中间拷贝。配合 pipeline replication 和 topology-aware 调度，在 1024 GPU 上实现 6.7× stall 降低，已部署于字节跳动生产 RL 训练。

§2 核心三问

Q1 痛点：RL 大规模权重传输中 flexibility 与 efficiency 的根本矛盾

LLM RL 训练依赖高频权重传输：trainer 每步产出新权重，数千 rollout worker 需拉取最新权重进行推理。权重规模可达 TB 级，传输效率直接决定训练吞吐。

生产环境进一步要求支持三种 rollout 策略：standalone（专用 GPU）、elastic（可抢占 spot 实例）、cross-datacenter（跨数据中心异构资源）。现有方案均无法同时满足五个设计目标：

  • NCCL：静态通信组，任何节点故障导致全集群崩溃，全局 barrier 中断所有 worker
  • UCX：无全局拓扑视图，fan-out 下 sender 带宽成为瓶颈
  • Distributed Storage：push-then-pull 模式将网络流量翻倍，Ray Plasma 40 GB 传输需 32s vs GPU-direct RDMA ~0.2s（~160× 差距），且 >300 GB 时常 OOM 崩溃
  • 根因：data ownership——storage 必须先接管数据再提供读取，这是额外搬运和存储开销的本质来源

Q2 方法：Reference-Oriented Storage — 消除数据所有权的存储抽象

核心洞察：RL 训练中模型权重具有三个可利用属性——(1) 数据并行导致高度冗余复制，(2) 推理期间不可变，(3) 版本短暂（只关心最近几个）。这些属性使得 GPU 上已有的权重副本天然构成多个对等数据源，无需额外存储。

ROS 直接复用 GPU 上已有权重 in place，通过集中式引用服务器追踪哪些 worker 持有哪些版本，由客户端之间 RDMA 直连传输数据。三个核心原语：

  • publish：trainer 向引用服务器注册其 GPU 权重的引用（不搬运数据）
  • replicate：rollout 查询服务器获取源引用，通过 RDMA 从源 GPU 直读到目标 GPU
  • unpublish：mutation 前显式撤销引用，服务器等待 in-flight 传输完成后确认

两个保障机制：

  • Mutability Contract：publish/unpublish 纪律确保引用有效期间数据不被修改，任何 concurrent transfer 都能读到完整数据
  • Retention Protocol：当最后一个 replica 要 unpublish 保留版本时，offload 一份到 CPU 内存（~48 GB/s PCIe，<2s 延迟）。此 fallback 极少触发——rollout 通常会及时 replicate

TensorHub 将 ROS 扩展为生产级系统：

  1. Pipeline Replication：部分复制完成的 worker 立即可作为源，利用 RDMA NIC 全双工特性在接收同时提供数据。每个 replicating worker 维护 progress counter，新请求者读取 counter 前缀并重复直到完成。将 N-way fan-out 转化为 bandwidth-amplifying DAG，总 stall 从二次方降至线性
  2. Topology-Aware Scheduling：集中式服务器维护每 replica 的 in-flight 引用计数，优先同 DC 最低负载 replica。Cross-DC 时使用两阶段复制——先 TCP seeding 后 RDMA pipeline
  3. Transactional Consistency：对 model-parallel group 内的请求在引用服务器上执行事务语义，首个 worker 的请求开启事务并缓存状态，后续 shard 请求消费缓存保证一致视图
  4. Fault Tolerance：heartbeat 超时检测故障，replica 粒度驱逐。引用服务器仅持有软状态，故障后新 server 无需恢复任何状态——等待下轮 publish 即可

    5. 核心技术壁垒：pipeline replication 的设计将传统 fan-out broadcast 问题转化为 self-amplifying 分发网络。关键在于识别 RDMA NIC 全双工特性（接收时上行空闲）可以零额外成本将接收节点转变为中继。Progress counter 机制使部分复制的 worker 能在数据仍在流入时就开始服务，时间维度上的流水化是带宽放大的关键——这不是简单的 BitTorrent chunk 分发，而是 tensor 粒度的 streaming relay。

    §3 架构与方法

    3.1 System Scope

    • Stage coverage：仅覆盖 RL 训练循环中的权重传输阶段（train → transfer → rollout），不涉及训练内核或推理引擎本身
    • Serving or training：专为 RL 训练设计（PPO 等算法的 weight sync loop），也支持故障恢复时的 weight pull
    • Parallelism：model parallelism 下每 GPU 持有一个 shard，TensorHub 原生支持分片（naming: model → version → replica → shard）。不涉及 TP/PP/EP/DP 的具体实现，仅处理跨 group 的权重同步
    • Deployment mode：multi-node，支持 standalone rollout（同 DC 专用 GPU）、elastic rollout（跨 stable + spot GPU）、cross-datacenter（TCP + RDMA 混合）

    3.2 ROS 数据通路

    sequenceDiagram participant Trainer as Trainer (GPU) participant Server as Reference Server participant R1 as Rollout-1 (GPU) participant R2 as Rollout-2 (GPU) Note over Trainer: 训练完成, 产出 version N Trainer->>Server: publish(version=N, ref=GPU_addr) Note over Server: 记录: version N → Trainer replica R1->>Server: replicate(version="latest") Server->>Server: 解析 "latest" → N, 选最低负载源 → Trainer Server-->>R1: source_ref = Trainer.GPU_addr R1->>Trainer: RDMA Read (GPU→GPU 直传) Note over R1: 完成后自动成为 version N 的 replica R2->>Server: replicate(version="latest") Server->>Server: 选最低负载源 → R1 (pipeline replication) Server-->>R2: source_ref = R1.GPU_addr R2->>R1: RDMA Read (pipeline, 读 progress counter 前缀) Note over R2: R1 仍在接收时已可提供已完成的前缀

    3.3 核心组件

    Reference Server：集中式轻量协调器，以 Ray named actor 实现，首次 open() 时惰性实例化。仅持有软状态（引用映射），不存储或转发任何权重数据。故障后无需状态恢复——新 server 等待下轮 publish 重建。单 server 可支撑数千 worker。

    Transfer Engine：每客户端嵌入的数据平面组件，基于 Mooncake transfer engine 构建。三种传输模式：

    • RDMA Direct：长期存活的 tensor 直接注册到 RNIC，远端单边 RDMA Read，零拷贝绕过 CPU
    • RDMA Copy：频繁重分配场景下，后台线程分片拷贝到预注册 buffer 后 RDMA Write
    • TCP：跨 DC 无 RDMA 时 fallback

    Hardware affinity-aware：GPU tensor 选择最近 NIC，RDMA Copy 使用同 GPU device 的预注册区域。

    Tiny-Tensor Compaction：LLM 大量 <2 MB 小 tensor 合并到连续 buffer 统一注册传输。36B 模型约 500 tensor 中一半为小 tensor，合并后仅需 ~3 MB 额外内存。

    Cross-node communication：同 DC 内 RDMA（400 Gbps/NIC × 4 NIC/node），跨 DC 走 VPC NIC TCP（200 Gbps/node）。Pipeline replication 仅在同 DC RDMA 上生效，跨 DC 采用两阶段 seeding。

    3.4 Pipeline Replication 机制

    RDMA NIC 全双工，接收数据时上行链路空闲。每个 replicating worker 维护 tensor 粒度的 progress counter：

    1. Rollout-B 请求 version N，服务器选 Rollout-A（正在接收中）作为源
    2. B 读取 A 的 progress counter，获取已完成的 tensor 前缀
    3. B 通过 RDMA 读取该前缀
    4. B 重复步骤 2-3 直到所有 tensor 完成

      5. N 个 rollout 同时请求时，第一个从 trainer 获取（2.2s/50GB），后续每个加入都增加系统总带宽。总 stall time 线性增长而非二次方。

      3.5 Cross-Datacenter Replication

      两阶段模式：

      1. Seeding：远端 DC 第一个 replica 通过 TCP 拉取全量权重
      2. Local pipeline：seeding 完成后，该 DC 内后续 replica 通过 RDMA pipeline replication 获取

        3. 优化：

        • Smart Skipping：polling replica 发现 seeding replica 正在 TCP 传输中时，视新版本为暂时不可用，避免序列化在慢速 TCP 后面
        • Offload Seeding：将 TCP 接收缓冲放到 CPU 内存并后台传输，update() 立即返回让 GPU 继续计算。以有限 CPU 内存换取 GPU stall 消除

        3.6 Consistency Model

        Model-parallel group 内 N 个 shard worker 必须看到一致的版本状态（否则 SPMD 程序会 diverge）。服务器在 replica 粒度执行事务：第一个到达的 shard 请求开启事务、执行查询并缓存结果（如 "latest = 12"），后续 shard 请求消费缓存值。事务按开始顺序串行化。

        3.7 Fault Tolerance

        故障类型处理方式
        Client 故障（heartbeat 超时）整个 replica 驱逐，释放关联资源
        接收方故障服务器废弃不完整 replica，释放引用计数
        源方故障接收方检测传输失败报告服务器，服务器标记源故障、选替代源
        Spot instance 抢占retention 计数排除 spot replica；最后非 spot replica 故障时返回优雅错误
        Reference Server 故障Client 重置为 unpublished 连接备用 server，新 server 无需恢复状态

        5.2 关键实验结果

        RDMA 带宽效率：TensorHub 50 GB 传输 2.2s，22 GB/s（88% 理论 25 GB/s）。NCCL 18.8 GB/s，UCX 18.1 GB/s。Ray Plasma 比直接传输慢两个数量级且 >35 GB 崩溃。

        Pipeline Replication 扩展：1 trainer → 1-8 rollout groups，有 pipeline replication 时每 shard 固定 2.2s（不随 replica 数增加），总 stall 线性增长。无 pipeline 时所有 replica 争抢 trainer 带宽，总 stall 二次方增长。

        故障掩盖：Pipeline 中 Rollout-A 故障后 Rollout-B 检测 RDMA timeout（~4s）并由服务器重路由到 trainer 完成传输。无数据丢失，延迟增量有限。

        Standalone Rollout（最多 1024 GPU）：TensorHub 的 publish 为轻量引用传递，trainer 立即恢复；NCCL/UCX 要求 Ray driver 全局协调阻塞至传输完成。1T 模型理想 66 GB 传输应 2.6s，NCCL stall 全部 1024 GPU 5.3s，UCX 4.0s，TensorHub mean 3.1s 且仅 standalone GPU stall → 总 stall 降低 6.7×。

        Elastic Rollout（Spot Instance）：260B 模型，1 standalone + 3 elastic machines。UCX 下 elastic 必须等 standalone 先从 trainer 拉取，bandwidth contention 导致阶梯状 CDF，尾部 GPU 等待 7.2s。TensorHub 下所有 rollout 可从任意源获取，stall 恒定 ~1.5s 不随 elastic GPU 数变化，达 4.8× 加速。

        Cross-DC Rollout：9B 模型，2 DC（trainer DC + rollout DC），仅 200 Gbps VPC TCP 连接。UCX 每 GPU 等待 7.8s（所有 flow 争抢单 VPC NIC）。TensorHub + smart skipping 仅 seeding replica 等 2.5s，其余 RDMA 传输 0.45s。加 offload seeding 后 seeding stall 也可隐藏，总计 19× stall 降低。

        5.4 瓶颈转移分析

        
Baseline (NCCL/UCX):
  全局协调 + straggler amplification (所有 GPU stall)
  → TensorHub: publish 为轻量引用, 仅 rollout stall

Fan-out contention (UCX, elastic):
  所有 elastic 排队争抢 standalone 出口带宽 → 阶梯状延迟
  → TensorHub: pipeline replication 将每个新 replica 转化为新数据源

Cross-DC bandwidth (UCX):
  每个 replica 独立跨 DC TCP 传输 → VPC NIC 拥塞
  → TensorHub: 仅 1 次 TCP seeding, 后续全部 RDMA

残余瓶颈:
  - 1T 模型 TensorHub mean 3.1s vs ideal 2.6s → ~19% overhead (尾部 straggler)
  - Cross-DC seeding replica 仍需 2.5s TCP (offload seeding 可隐藏但消耗 CPU mem)
  - Reference server 单点 (论文提出分区方案作为 future work)

        5.5 Scheduling & Resource Management

        调度粒度：以 version（整个模型权重的一个版本）为调度单位。服务器为每个 replicate 请求选择源 replica（least-loaded, same-DC-first）。Pipeline replication 下每 replica 同时至多服务 1 个请求。

        Preemption/admission：无显式 preemption。服务器通过 reference count 限制并发。Retention protocol 提供隐式 admission——offload 仅在最后一份被 unpublish 时触发。

        Memory management：

        • GPU 侧：tensor buffer 由用户管理，通过 register/unregister 告知 TensorHub
        • CPU 侧：retention offload 的 CPU buffer 为临时创建，offload 完成后自动释放
        • RDMA 注册：长期 tensor 直接注册（RDMA Direct），频繁重分配用预注册 buffer 池（RDMA Copy）
        • Tiny-tensor compaction buffer：<2 MB tensor 合并的临时 buffer，总开销 ~3 MB

        §7 实现与工程细节

        7.1 代码 Cross-Reference

        [实现未公开] — TensorHub 为字节跳动内部系统，3.7K Python + 2.4K C++，代码未开源。

        依赖的开源组件：

        • Mooncake transfer engine (Qin et al., 2025, FAST'25)：RDMA 传输引擎
        • Ray (Moritz et al., 2018, OSDI'18)：reference server 实现为 Ray named actor, client-server 通信走 Ray RPC
        • veRL (Sheng et al., 2025b, EuroSys'25)：评估使用的 RL 训练框架

        7.2 核心技术壁垒

        Pipeline replication 中将 progress counter 暴露为外部可读状态，使正在接收的 worker 能同时作为数据源。传统 BitTorrent 式 P2P 需等待完整 chunk 后才分发；TensorHub 利用 tensor 粒度的 progress tracking 实现时间维度的流水化，本质上是将接收窗口内的"已到达前缀"实时暴露为可服务状态。

        这要求精确控制 RDMA 传输的数据一致性边界：只有 progress counter 指示"已完成"的 tensor 才可被远端读取。Counter 的更新必须在 RDMA completion 之后原子执行，否则可能暴露部分写入的 tensor 数据。

        7.3 关键实现细节

        1. FoundationDB 风格确定性测试：单进程模拟多客户端的请求交错，注入故障为停止特定 client 请求。无需 RDMA NIC 或 GPU，1.3K 行并发测试覆盖 model-parallel consistency 和故障场景。
          1. End-to-end checksum：publish 时计算每 tensor checksum 附到引用上，replicate 完成后验证。GPU 上 checksum 计算快且可与 RDMA 传输重叠。同时用于验证 mutability contract 未被违反。

            2. 7.4 API & Usability

            • 对外接口：Python API — open / register / publish / unpublish / replicate / update / list / wait / close，统一支持 CPU 和 GPU tensor
            • 配置复杂度：低。主要参数为 model_name、replica_name、num_shards、shard_idx、retain 策略
            • 集成成本：极低——~40 行代码，vs NCCL ~450 行 / UCX ~1200 行。无需修改 RL 框架的调度逻辑
            • 迁移路径：从 NCCL/UCX 迁移仅需替换权重传输代码段，trainer 端改为 publish/unpublish，rollout 端改为 replicate/update

            7.5 Adoption & Ecosystem

            • 开源状态：否。TensorHub 为字节跳动内部生产系统
            • 生产部署：是。论文声明 "TensorHub has been deployed in ByteDance production to support cutting-edge RL training"
            • 上游合并：未合并到任何开源框架。核心思想（ROS 抽象、pipeline replication）可独立实现
            • 下游要求：需要 RDMA 网络（InfiniBand/RoCE），跨 DC 场景需 TCP fallback。对 GPU 类型无硬性要求（评估使用 NVIDIA Hopper）

            7.6 Deployment Context

            • 覆盖阶段：RL 训练循环中的权重同步阶段，不涉及训练或推理本身
            • 并发规模：评估覆盖 16-1024 GPU，设计目标为数千 worker。单 reference server 已验证可支撑 1024 GPU
            • 硬件亲和：最大收益在大 RDMA 带宽集群（400 Gbps × 4 NIC/node），跨 DC 场景下 TCP 带宽为瓶颈。NIC 拓扑感知（每 GPU 选最近 NIC）对多 NIC 节点很关键
            • Ecosystem integration：以 client library 形式嵌入 RL 框架，reference server 为独立 Ray actor。与 veRL 集成仅需 ~40 行改动；可嵌入 OpenRLHF、AReaL 等任何支持 Ray 的框架
            • 迁移阻力：最小——纯 library 层面替换，不要求修改模型定义或训练逻辑

            7.7 Software → Hardware Reverse Implication

            TensorHub 的 pipeline replication 假设 RDMA NIC 全双工（接收时上行空闲）——这是带宽放大的硬件基础。如果未来 NIC 架构改变全双工特性（如共享带宽调度），pipeline replication 的效果会降低。

            Cross-DC 依赖 TCP 的低带宽是系统的主要瓶颈之一。硬件层面的改进方向：

            • RDMA over WAN：跨 DC 部署 RoCE v2 或类似 RDMA 传输可大幅降低 seeding 延迟
            • NIC-integrated versioning：NIC 原生支持版本化数据读取（类似 RDMA extended atomics），progress counter 检查可在硬件层面完成
            • Smart NIC 辅助调度：将 least-loaded replica 选择下沉到 DPU/SmartNIC，消除 reference server 作为软件中间层的延迟
