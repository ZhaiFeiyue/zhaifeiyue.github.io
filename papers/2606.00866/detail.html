Idleness is Relative — MORI (L2) #

1. TL;DR #

Agentic programs alternate between busy phases (bursts of short tool calls) and idle phases (blocked on long tool calls). MORI ranks all live programs by a continuous idleness metric, places the busiest in GPU HBM and the most idle in CPU DRAM, slides the tier boundary to fit any hardware ratio, and enforces per-tier admission control — yielding 20–71% higher throughput and 18–43% lower TTFT than the best offloading baseline.

2. Q1 / Q2 / Q3 #

Q1 — 痛点. Serving agentic (ReAct-loop) workloads accumulates per-program KV cache across tool-call steps that quickly exceeds GPU HBM, forcing offload to CPU DRAM. But tool-call durations vary by 3+ orders of magnitude and are unpredictable, so re-placing KV on every call is impractical: transfer cost dominates short gaps. LRU and static context-length eviction are phase-blind — they may evict a program about to resume while keeping a long-blocked one GPU-resident. A binary busy/idle label also fails: the busy:idle demand ratio rarely matches the fixed GPU:CPU capacity ratio, so one tier is oversubscribed while the other sits idle.

Q2 — 方法. Define each program's idleness $\iota$ as the fraction of recent wall-clock time spent in tool calls (Acting) vs. total (Acting + Reasoning) over a sliding window of the last $k{=}5$ cycles. Rank all active programs on this continuous spectrum; a sticky control loop (5 s tick) demotes the highest-$\iota$ programs from GPU→CPU→Waiting and promotes the lowest-$\iota$ programs back up, filling each tier exactly to its hardware capacity (which is what enforces admission control). Typed offloading hints (busy / idle / inactive) propagate to the engine, whose LRU eviction is re-sorted by type with reversed priority per tier. In multi-replica mode, an affinity-aware balancer routes a returning program back to the replica holding its cache (GPU or CPU).

  • 核心技术壁垒: reframing idleness from a binary label into a continuous, relative ranking whose partition boundary auto-adapts to any GPU:CPU capacity ratio — no per-hardware tuning, and it unifies GPU + CPU admission control into a single sort-and-fill operation.

Q3 — 结果. At 80 concurrent programs (DP=1): 20–71% higher output throughput and 18–43% lower TTFT than TA+O (offloading baseline), 1.6–2.1× throughput and up to 2.8× lower TTFT vs non-offloading systems. DP=3: 54–79% higher throughput than TA+O while sustaining 99%+ GPU utilization (vs 59–76% for phase-oblivious schedulers that collapse at high concurrency). Program churn drops from 14–15% to 0.3–2.9%.

3. 架构 / 方法图 #

3.1 Agent scope #

  • Task class: open-ended coding agents (Claude Code / Codex / Cursor), ReAct-style dynamic workflows — not fixed-graph function calling.
  • Interaction pattern: multi-turn, long-horizon (tens of inference steps interleaved with tool calls over minutes per session).
  • Autonomy: fully autonomous with occasional human-in-the-loop tool calls (approval waits are one source of idle phases). MORI itself is infrastructure and needs only a program_id per request — no phase hints from the client.

3.2 System overview #

Paper's Figure 1: MORI (1) categorizes programs on a continuous idleness spectrum, (2) an adaptive offloading scheduler places busy (low-$\iota$) programs in HBM and idle (high-$\iota$) programs in DRAM with two-tier admission control, and (3) an affinity-aware balancer routes to engines that already cache each program's KV state. This figure is the load-bearing summary: the three numbered stages map exactly to the idleness metric (§4.2 of paper), the tiering policy (§4.3), and the multi-replica router (§4.1).

Paper's Figure 2: a program alternates shaded inference steps with tool-call gaps; the KV cache grows monotonically because each invocation shares the prefix of all preceding ones. This is the unit-of-scheduling premise: the gap is inside one logical execution, so the cache stays valuable during it — the whole reason offloading (not eviction) is the right move.

3.3 The agent-loop state model #

One turn cycles through two instantaneous statuses whose sustained pattern defines the phase:

stateDiagram-v2 [*] --> Reasoning: request arrives Reasoning --> Acting: response emitted (tool call starts) Acting --> Reasoning: tool result returns (next request) Reasoning --> GPUqueue: iota low (busy) Acting --> CPUqueue: iota high (idle) & GPU over capacity CPUqueue --> Waiting: CPU over capacity Waiting --> GPUqueue: capacity frees (full prefill recompute) CPUqueue --> GPUqueue: promote (PCIe reload) GPUqueue --> CPUqueue: demote (offload)
  • Memory model: short-term = KV cache in HBM (busy); "long-term" spill = KV in CPU DRAM (idle, per-replica for affinity); discarded = Waiting queue (must recompute via full prefill). There is no vector-DB / episodic store — memory here means the KV working set only.
  • Tool invocation: opaque to the scheduler; a tool call simply flips the program to Acting for an unknown duration. Parallel tool calls are collapsed into one gap = max of individual durations. A spawned subagent is a separate program; from the parent's view its whole execution is one long tool call.
  • Error recovery / fallback: if a program cannot be admitted to GPU or CPU, it falls back to the Waiting queue and loses its cache entirely (recompute on resume). Gated (queued) time is excluded from $\iota$ so the scheduler's own delays don't pollute the phase signal.

3.4 Planning & scheduling style #

  • Planning style: not agent planning — this is a predetermined control-loop scheduler over program placement. Sticky rebalancing: a program stays put until a capacity violation forces demotion or free capacity allows promotion.
  • Decomposition: top-down capacity fitting — rank by $\iota$, fill GPU from the busiest end, spill the rest to CPU, overflow to Waiting.
  • Budget: 5 s control tick; $k{=}5$-cycle idleness window; GPU/CPU/Waiting admission bounded by physical HBM/DRAM.
  • Backtracking: promotion undoes demotion; because transitions cost offload/reload, the sticky policy deliberately minimizes churn.

4. 作者证明 #

无形式化作者证明 — 仅实证. As is typical for agent-serving systems, there are no convergence or optimality guarantees. The only formal object is the idleness definition (Eq. 1); everything else is validated empirically across a hardware/model/concurrency sweep.

Notation table #

SymbolMeaning
$\iota$idleness of a program, $\in [0,1)$
$T_{\text{acting}}^{(k)}$wall-clock time in tool-call (Acting) status over last $k$ cycles
$T_{\text{reasoning}}^{(k)}$wall-clock time in GPU inference (Reasoning) status over last $k$ cycles
$k$sliding-window length in inference–tool-call cycles ($k{=}5$)

The one equation and its physical meaning #

$$\iota = \frac{T_{\text{acting}}^{(k)}}{T_{\text{reasoning}}^{(k)} + T_{\text{acting}}^{(k)}}$$

$\iota$ is the fraction of recent time a program left its KV cache sitting idle in HBM while blocked on tools. $\iota \to 1$ = idle phase (offload it); $\iota \to 0$ = busy phase (keep it GPU-resident). It is a proxy for wasted HBM, not a duration prediction.

6 minimum checks #

  1. Success-rate / performance sweep — swept over (GPU tier × model size × concurrency × CPU:GPU ratio): {H200-80GB/7B, H200/30B-MoE, B200/70B-TP2} × {20,50,80 programs} × {1×, 2× CPU}. Monotonicity: MORI throughput rises monotonically with concurrency (e.g. B200 136→191→189 tok/s) where TA+O is non-monotonic (147→181→146). The gain over TA+O widens with memory pressure (2% at 20 programs → up to 71% at 80).
  2. Latency budget per turn — TTFT decomposes into queue + reload + prefill; MORI's GPU queue prioritizes already-resident programs, cutting reload stalls. No "interactive latency" claim at 80 programs (TTFT 33–38 s on B200), but it is the lowest of all systems.
  3. Failure-mode classification — the paper isolates three baseline failure classes: (a) phase-blind eviction (LRU/context-length keeps idle caches on GPU), (b) uncoordinated CPU LRU (extra DRAM gives TA+O no benefit), (c) affinity break on eviction (cross-replica churn). MORI is targeted at all three; the dominant one under load is (a) → throughput collapse.
  4. Backbone sensitivity — method is backbone-agnostic (7B Qwen-2.5, 30B Qwen-3 MoE, 70B Llama-3.1 all benefit); it depends on timing behavior, not model capability.
  5. Overhead check — Table 2: MORI adds 2.3 ms CPU scheduling (21.5→23.8 ms) but this is fully overlapped with the ~32 ms GPU step, so zero critical-path cost.
  6. *What could have been bounded* — the reload-vs-recompute cost tradeoff could be formalized (transfer cost linear in KV size vs. gap duration), giving a competitive-ratio bound; the paper argues it informally in §3.2 instead.

    7. 5. 实验与数据 #

    Workload: 186 replayed Claude Code traces on SWE-bench Pro (claude-sonnet-4-6, high effort), closed-loop replay for 1 hour per run. Baselines: SMG (prefix-aware gateway), TA (program-aware, no offload), TA+O (TA + uncoordinated HiCache CPU offload).

    Paper's Table 1: three GPU configs spanning H200-80GB/7B, H200/30B-MoE (DP=1 and DP=3), and B200/70B-TP2. The sweep deliberately spans a loose regime (7B leaves ample KV room) to the tightest (70B TP=2), so the widening MORI advantage tracks memory pressure rather than a single lucky config.

    Paper's Figure 3: tool-call durations span from sub-second file I/O to minutes (human input / subagents). This heavy tail is the entire motivation: because per-call duration is unpredictable and spans decades, per-call placement is hopeless and coarser phase-level signal is required.

    Paper's Figure 5: median busy-phase durations of ~4 s / 20 s / 41 s across thresholds; robust to threshold choice. This is the load-bearing evidence that phases persist (median 20 s at the 2 s threshold), long enough that sticky placement won't oscillate — and that 13% of long calls hold 58% of tool-call wall-time.

    Paper's Figure 9: the tightest memory regime; TA+O throughput is non-monotonic (collapses at 80 programs) while MORI grows monotonically and reaches the largest relative gain (+71%). This is the case-study config: it most cleanly separates phase-aware from phase-oblivious placement because eviction thrashing is unavoidable.

    Paper's Figure 10: multi-replica setting; MORI sustains 99%+ GPU utilization and 54–79% higher throughput while phase-oblivious baselines suffer throughput collapse (falling from 1147→798 tok/s as concurrency rises). It shows memory tiering and cross-replica load balancing can be co-optimized rather than trading off.

    Honest tie: at 20 programs on H200-80GB/1× CPU, MORI 546 vs TA+O 534 tok/s (+2%) — when the working set fits in CPU, phase-awareness is marginal.

    7. 实现 cross-reference #

    [实现未公开] — no public repository is cited in the L1 source; citations below are to the paper's own implementation description.

    • Built on ThunderAgent + SGLang v0.5.10: ~3,300 LoC Python added to ThunderAgent's scheduler + ~500 LoC to SGLang's cache subsystem via HiCache (§5).
    • User API: OpenAI-compatible chat/completions augmented with a single program_id field; a spawned subagent uses a distinct id. This is the only client-side requirement — no tool-call annotations or phase hints (§5).
    • Scheduler: async control loop in the ThunderAgent router, default 5 s tick; request handlers for demoted programs block until re-promotion so no request reaches the engine before its KV is GPU-resident (§5).

    核心技术壁垒 (dedicated paragraph). The single hardest-to-replicate insight is treating idleness as relative rather than absolute. A binary or threshold classifier is easy; the non-obvious move is that ranking programs against each other makes the busy/idle partition boundary slide automatically to whatever the local GPU:CPU capacity ratio happens to be — so the same code serves a 1:1.6 and a 1:3.1 node with zero retuning, and the act of "filling each tier to capacity from the busiest end" simultaneously is the admission-control mechanism for both tiers. Getting this right requires the windowed $\iota$ (responsive to phase change, robust to a single outlier long call) and the sticky policy (adapt without churn) working together.

    关键实现细节 (easy-to-miss tricks).

    1. Reversed per-tier LRU sort key (§4.3.2): the engine keeps plain LRU but adds the type label (busy/idle/inactive) as a higher-priority sort key, and the priority order is inverted between tiers — GPU evicts inactive→idle→busy (busy last), CPU evicts inactive→busy→idle (idle last) — so each tier preferentially retains exactly the programs the scheduler assigned to it.
    2. Excluding scheduler-gated time from $\iota$ (§4.1/§4.2): time a program spends waiting in the CPU/Waiting queue is subtracted from both $T_{\text{acting}}$ and $T_{\text{reasoning}}$, so the metric measures the program's intrinsic behavior and never feeds the scheduler's own queuing delay back into its placement decision.
