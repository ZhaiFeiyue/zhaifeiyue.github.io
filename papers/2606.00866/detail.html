Idleness is Relative: Exploiting Tool-Call Idle Windows for Offloading in Agentic Systems with MORI — L2 #

1. TL;DR #

MORI把agent程序的“空闲”建模为连续相对指标而非二值标签，按全局idleness排序在GPU HBM/CPU DRAM/Waiting三层之间做粘性重平衡与双层准入控制，在真实编码agent回放中实现更高吞吐和更低TTFT。

2. Q1 / Q2 / Q3（痛点 / 方法 / 结果） #

Q1 痛点：工具调用时长高方差让“每次调用都决策”不可行 #

  • Agent程序由推理与工具调用交替构成，KV缓存跨步增长，显存压力随并发和上下文长度迅速上升。
  • 工具调用呈重尾分布：同类工具时长可跨数量级，短调用时频繁搬移KV会得不偿失，长调用时继续占GPU又浪费HBM。
  • 传统LRU或仅按上下文长度的策略不感知“程序阶段”，会把即将恢复推理的busy程序赶走，导致reload/recompute抖动。
  • 仅做busy/idle二分类也不够，因为实时busy与idle工作集比例会漂移，常与固定硬件容量比例不匹配，造成一层拥塞一层闲置。

Q2 方法：连续idleness排序 + 三层粘性调度 + 分层类型化驱逐 #

  • 定义最近$k$步窗口的程序空闲度：

    • $$\iota=\frac{T_{\text{acting}}^{(k)}}{T_{\text{reasoning}}^{(k)}+T_{\text{acting}}^{(k)}}$$

    其中$T_{\text{acting}}^{(k)}$与$T_{\text{reasoning}}^{(k)}$只统计程序自身行为，不计调度等待时间；$k=5$用于实验。

  • 调度核心不是预测单次工具调用时长，而是基于近期行为估计“相对更闲/更忙”，并在容量约束触发时做最小必要迁移（sticky placement）。
  • 每个副本维护GPU队列与CPU队列，集群共享Waiting队列；GPU满时优先降级高$\iota$且处于Acting的程序，GPU空出时优先提升低$\iota$且等待推理的程序。
  • 调度器把busy/idle/inactive类型下发给推理引擎，GPU层与CPU层采用相反的类型优先级驱逐，保证“该留在本层的程序块最后被驱逐”。
  • 核心技术壁垒：在不可靠预测工具调用时长的前提下，把“程序阶段信号”稳定转化为跨两级容量约束和多副本亲和性的统一排序机制，同时抑制频繁搬移带来的传输开销。

Q3 结果：单副本与多副本都证明“相对空闲排序”在高压下更稳 #

  • 单副本80并发下，相对最强offloading基线TA+O，MORI输出吞吐提升20-71%，TTFT降低18-43%。
  • 相对非offloading系统（SMG/TA），吞吐可达1.6-2.1x，说明仅有程序感知但无协同分层管理仍受GPU容量硬约束。
  • 多副本（DP=3）时，基线在高并发出现吞吐回落与GPU利用率下降；MORI保持99%+ GPU利用率并显著降低program churn。
  • 调度CPU开销略增但被GPU步长掩蔽，未进入关键路径。

这个循环对应论文实现中的控制tick：观察程序状态与容量后做最小迁移，验证类型标签与准入一致性，再滚动更新窗口统计。

3.2 Planning / memory / tool interface要点 #

  • Planning风格：固定周期控制环而非树搜索；每tick做全体排序并在容量违规或可提升时动作。
  • Memory模型：短期记忆是每程序最近$k$步的Reasoning/Acting时间窗，长期状态由程序ID绑定在副本GPU/CPU队列与全局Waiting队列。
  • Tool接口契约：调度器接收program_id与请求边界事件；程序在请求完成后进入Acting，下一次请求到来前可能被降级或保持原层。
  • 失败恢复：若目标层无容量，降到更低层；若两层都无法准入则进Waiting并在恢复时重算KV。

4.1 记号与物理含义 #

记号含义
$\iota$程序近期空闲度，越大表示更多时间阻塞于工具调用
$T_{\text{acting}}^{(k)}$最近$k$个推理-工具周期中的工具调用总时长
$T_{\text{reasoning}}^{(k)}$最近$k$个周期中的GPU推理总时长
GPU/CPU/Waiting三层驻留状态：HBM、DRAM、无缓存待重算

方程的物理意义是“GPU占用有效性比率”的互补刻画：$\iota$高表示把该程序留在HBM的机会成本高，应该优先下沉。

7.1 核心技术壁垒（最难复现） #

最难复现的是把“程序级相对空闲排序”与“引擎内块级驱逐”做一致化闭环：上层必须稳定地产生跨副本可比较的idleness排序，下层必须在GPU与CPU两层执行相反类型优先级驱逐且保持亲和；任何一侧失配都会退化为局部最优并引入搬移抖动。

7.2 关键实现细节（易忽略） #

  • 计时口径：等待队列造成的调度阻塞时间必须从$T_{\text{acting}}$与$T_{\text{reasoning}}$剔除，否则idleness会把“系统拥塞”误判为“程序本性更闲”。
  • 懒降级策略：当只剩Reasoning程序可降级时，需允许其完成当前推理步再迁移，避免中断活跃批次并放大尾延迟。

7.3 Agent系统落地约束（backbone / cost / production） #

  • Backbone要求：方法不依赖特定模型家族的推理技巧，但依赖稳定可观测的程序边界事件与可控KV offloading接口。
  • 成本形态：收益来自减少无效HBM占用与重算；成本来自GPU<->CPU传输、控制环开销和潜在亲和牺牲，实际最优点随并发与模型大小变化。
  • 生产要点：需要程序ID贯穿客户端到调度器、跨副本缓存驻留追踪、以及对CPU层同等严格的准入控制，否则高并发下会在CPU侧重演thrash。
