Core Contribution

K2.5 把 K2 那个 1.04 T total / 32 B active 的 MoE LLM 复用为多模态主干，前面接一个 27 层 MoonViT-3D 视觉塔（NaViT 打包 + 4 帧时空压缩），通过 patchmerger MLP 投影到 7168 维 LLM 嵌入流。最关键的工程创新不在于架构（架构基本是 K2 + MoonViT 的组合），而在于：

1. 早期融合 + 低视觉比例 — 联合预训练阶段 vision : text = 10 : 90，挑战 "视觉数据越多越好" 的常识

2. 零视觉 SFT — text-only SFT 配 IPython 程序代理图像操作就能激活视觉能力，避开高质量 visual SFT 数据的稀缺

3. PARL / Agent Swarm — 把 multi-agent orchestration 形式化为 RL 问题，编排器 RL 训练 + 子 agent 冻结，3-4.5× 延迟降低

4. DEP (Decoupled Encoder Process) — 让多模态训练吞吐达到纯文本 ~90%

5. Toggle 解码预算控制 — 交替 budget-limited 和 full-length 训练，~25-30% token 节省

Summary

K2.5 的设计哲学是 "LLM 主干别动，把所有创新放到对接层和训练管线上"。LLM 主干完全继承 K2：61 层 dense-skip + MoE，d_model=7168，MLA（DeepSeek V3 风格 latent attention，q_lora_rank=1536, kv_lora_rank=512, qk_nope_head_dim=128, qk_rope_head_dim=64, v_head_dim=128），384 routed experts top-8 + 1 shared expert，expert hidden=2048（SwiGLU），首层 dense（first_k_dense_replace=1），vocab=163 840，YaRN 扩展到 262 144 context。

视觉塔 MoonViT-3D：基于 SigLIP-SO-400M 初始化，27 层，d=1152，16 heads，head_dim=72（继承 SigLIP 的非 2 幂设计），FFN=4304，patch=14×14，2D RoPE，NaViT 风格 patch 打包同时处理多分辨率。视频路径在编码器后做 空间 2×2 merge + 时间 4× pool，最后一个 patchmerger MLP 投到 7168 维直接替换文本流里的 <|kimi_k25_video_placeholder|> token。

训练分四大阶段：(1) K2 文本预训 15T tokens（继承）；(2) ViT 阶段 ~1T，先训 ViT 再训 projector 桥接（caption CE only）；(3) 联合预训 ~15T，vision : text = 10 : 90，DEP 让吞吐 ~90% of text-only；(4) mid-train 长上下文 500B → 200B（论文 Table 3），seq 32K → 262K，YaRN factor=64。后训练：zero-vision SFT → joint multimodal SFT → joint multimodal RL（按 ability 而非 modality 组织），RL 用 token-level log-ratio clipping + KL 惩罚 + MuonClip。Agent 变体走 PARL 路径：λ₁·r_parallel + λ₂·r_finish + r_perf，λ 退火到 0；CriticalSteps 度量。

Key Findings

  • 架构 = K2 backbone + MoonViT-3D：61L · d=7168 · 384E top-8 + shared，MLA 不是 GQA / MHA
  • MLA 关键 latent rank：q_lora_rank=1536，kv_lora_rank=512 — 单 token KV cache 仅 512+64=576 floats（vs 标准 GQA 数千）
  • MoE 配置精选：384 routed × 2048 hidden + 1 shared，sigmoid scoring，noaux_tc top-8，routed_scaling_factor=2.827
  • YaRN 64×：4K 原生 → 262K 部署 context（factor=64, beta_fast=32, beta_slow=1）
  • Vision tower 相对小：27 × 1152 vs 主干 7168，靠 NaViT packing + 2×2 merge + 4× temporal pool 控制 token 数
  • Early fusion 10:90 比 conventional 50:50 更好 — 早期混入但比例低，避免 LLM 偏向视觉
  • Zero-vision SFT 比 text+vision SFT 更好 — 文本 SFT + IPython 图像操作代理足够激活视觉能力
  • Vision RL 反向提升 text — 跨模态正向迁移（GPQA +2.1, MMLU-Pro +1.7）
  • DEP 让多模态训练 ~90% throughput of text-only — 否则视觉 stage-0 PP load spike 会拖慢一切
  • PARL Agent Swarm：3-4.5× 端到端延迟下降，子 agent 用 frozen K2，编排器单独 RL
  • Toggle：alternating budget / full-length 训练，~25-30% token 节省，accuracy 几乎不掉

Limitations

  • 论文几乎全部 LLM 主干维度都让你去查 K2 报告，正文非常 light on hyperparams
  • mid-train 那个 "500B→200B" 的表述含混，需要核对 PDF 附录
  • 总 GPU-hours、节点数、互联类型完全没披露
  • precision (BF16/FP8)、optimizer 细节、aux-loss 权重等都没说，只说 "继承 K2"

Infrastructure Impact

  • Algorithm: 早期融合 + 低视觉比例策略；Zero-vision SFT；按 ability 组织的 joint multimodal RL；PARL；Toggle
  • Kernel: MLA kernel (DeepSeek V3 family)；MoE routing；MoonViT NaViT packing kernel
  • Framework: vLLM + SGLang 都已支持（KimiK25ProcessingInfo, kimi_k25.py, 复用 DeepseekV3ForCausalLM）
  • LLM: 证明 "主干不动 + 多模态加挂" 在 1T 规模仍可行；YaRN 64× 长上下文部署有效
  • Agent: PARL 是首个把 multi-agent orchestration 形式化为 RL 问题的开源工作

MLA (Multi-head Latent Attention) 细节：

  • 每 token 通过 q_a_proj 压到 q_lora_rank=1536，再 q_b_proj 展开到 64×(128+64)
  • KV 路径：kv_a_proj 压到 latent (512+64) → kv_b_proj 展开到 64×(128+128)
  • Q split: Q_nope [64, 128] + Q_rope [64, 64]，RoPE 只施加在 Q_rope 部分
  • K 同样 split；K_rope 这一段是 跨 head 共享的
  • 真实 attention 是 64 heads × head_dim=192 effective（128 nope + 64 rope）
  • KV cache 仅 576 floats / token（512 latent + 64 rope k）— 这是 1T 级模型能跑的关键

MoE 细节：

  • Router: Linear(7168→384), sigmoid scoring (不是 softmax)，noaux_tc top-8 选择
  • routed_scaling_factor=2.827 是 routed 输出的乘性放大
  • 384 expert × 2048 hidden × SwiGLU = 每 expert ≈ 44M 参数
  • 1 shared expert 始终激活，hidden 同 2048
  • 首层 (layer 0) 是 dense 不走 MoE（first_k_dense_replace=1），稳定性考虑
  • aux_loss_alpha=0.001, seq_aux=true — 序列级辅助 loss 控制 expert 负载均衡

2d. 后训练

Thinker（K2-style 三段）：

1. Zero-Vision SFT — 全部 text，IPython program 代理图像操作；论文实验显示 text+vision SFT 反而比纯 text SFT 差（先验对齐已够强）

2. Joint Multimodal SFT — 合成 + 人工验证

3. Joint Multimodal RL — 按 ability 组织（数学 / 代码 / 视觉推理 / 通用对话），不按 modality

RL 算法（不是 PPO）：

  • Token-level log-ratio clipping 作为 gradient mask，区间 [α, β]
  • KL 惩罚项系数 τ
  • MuonClip 优化 RL objective
  • Truncated importance sampling
  • Dynamic KL + dynamic minibatch size

奖励设计：

  • Rule-based: 数学 / 代码 / 指令遵循（可验证）
  • Vision-specific: IoU、edit distance、计数、mask、LLM-as-judge
  • GRMs (General Reward Models): 跨行为通用 RM
  • Toggle: 每 m 步交替 budget-limited (budget(x) = ρ-percentile of correct rollout lengths) 和 full-length

Agent 变体 (PARL):

  • 编排器 RL 训练，子 agent 冻结（K2 frozen）
  • 奖励 = λ₁·r_parallel + λ₂·r_finish + r_perf
  • λ_i 退火到 0，最终只剩 r_perf
  • CriticalSteps 指标：度量哪些 step 是任务关键节点

2f. 核心训练壁垒

最难复现的一件事：zero-vision SFT 的可行性。它依赖于联合预训阶段（10% vision + 90% text，~15T tokens）已经把 LLM 表征空间彻底"沾染"了视觉特征，使得后续不需要再做视觉 SFT 也能正确响应视觉任务。如果联合预训规模或比例不对，zero-vision SFT 会立刻退化。论文承认实验成本太高没办法在多个 scale 上验证。

第二壁垒：DEP 实现。让多模态训练吞吐达到纯文本 90% 是工程奇迹——意味着 vision encoder 的 fwd/bwd 几乎完全 hide 在 backbone 后面。论文只描述三相位，不公开实际 schedule、buffer 大小、recompute 策略。
