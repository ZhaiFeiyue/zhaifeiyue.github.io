CacheWise — 面向编程 Agent 的 KVCache 管理层 #

1. TL;DR #

编程 agent 是长时运行、闭环、工具驱动的 LLM 会话，其请求反复复用巨大且不断增长的前缀。CacheWise 在 vLLM 里加一层 KVCache 管理：前缀感知调度（优先派发所需新增 block 最少的请求）+ 预测式淘汰（用工具调用元数据预测下次复用时间，替代 LRU）。在真实轨迹上把淘汰次数降 2–2.6×、会话完成时间降至最多 ~3.5×。

2. Q1/Q2/Q3 #

Q1 — 痛点：为什么 chatbot 的 serving 策略在编程 agent 上失效 #

编程 agent 与 chatbot 是结构上不同的 serving workload。一个用户任务会展开成一条闭环链：LLM 生成工具调用 → 环境执行 → 结果回灌成新请求，每一轮都在一个不断增长的前缀上追加上下文。三条实测特征使既有系统崩溃：

  • 闭环主导：工具触发的请求在中位数上是用户触发请求的 20×，性能目标从 per-request 延迟变成 session 完成时间。
  • 前缀巨大且增长 + 长会话：会话中位时长 36 min、尾部 >2.6 h；prefill:decode token 比约为 chatbot 的 21×。大前缀必须常驻 XPU 内存以便跨轮复用。
  • 工具时间可变但有结构：工具执行时长跨数量级（grep 短而紧凑，bash/pytest 长而重尾），导致 KVCache 复用间隔极不规则。

vLLM/Mooncake 默认的 FCFS 调度 + LRU 淘汰是 workload-agnostic 的：FCFS 交错多会话、撑大工作集，容易驱逐马上要用的前缀；LRU 只看历史访问时间，分不清"工具马上返回"和"还要很久"的会话，导致 KVCache 抖动（thrashing），压低 token goodput。

Paper's Figure 1（caption: "Chat and coding-assistant interactions."）。 左侧 chat 三轮上下文只从 1K→1.5K；右侧编程 agent 三轮通过 Glob/Read/Edit 工具调用把上下文从 10K 涨到 80K，直观说明"闭环 + 前缀爆炸"这一根本差异。

Q2 — 方法：两个正交技术 + 元数据预测器 #

CacheWise 是一层 agent-aware KVCache 管理层，不改 agent 本身、也不与 agent 框架耦合，只改 serving 系统：

  1. 前缀感知调度：在时刻 $t$ 选择所需新增 block 最少的请求 $\arg\min_i a_i(t)$ 派发，直接减少需回收/回填的 block，同时近似 shortest-job-first 降低整体排队。
  2. 预测式淘汰：不按 recency，而按预测的"下次复用时间"$\mathbb{E}[\tau_i(t)]$ 从大到小淘汰，逼近 Belady 最优。关键洞见——只需预测相对顺序（甚至只需准确找出 $\tau$ 最大的那个会话）就够了。
  3. 轻量预测器：从工具调用元数据 $m=(\text{tool\_name},\text{tool\_args})$ 出发，用 TF-IDF 对参数做语义嵌入、KMeans 聚类，用每簇历史时长分布（而非每工具分布）估计剩余执行时间。

    4. 核心技术壁垒：把"KVCache 淘汰"重述为一个只需相对序、且只需 top-1 正确的在线预测问题，并证明可以用工具调用元数据（函数名 + 参数的 TF-IDF 聚类）这一"免费"信号把它做到接近 ground-truth oracle 的精度——无需触碰 agent、无需精确回归绝对时长。这是难以复制的判断：把预测目标从"准确预测每个 $\tau$"松弛成"识别最该保留的会话"，是全篇效果的支点。

    Q3 — 结果 #

    在 2×H200（TP=2，282GB）上用 Qwen2.5-Coder-32B、回放 CATraces：高负载（$N>10$）下相对 vLLM/InferCept 会话完成时间低 2.7×–3.5×，token goodput 高 1.64×–2×，淘汰次数与 KVCache 传输量降 2–2.6×，且性能逼近拥有真值时延的 CacheWise*。代价：P99 请求尾延迟更高、CPU 调度开销 2.4–3×（但被 GPU 执行时间的下降远远抵消）。

    请求生命周期：同一会话的请求经负载均衡器的前缀匹配路由（把请求送到常驻可复用前缀最大的节点）→ 通常落到同一推理节点；节点内 CacheWise 接管两处决策：(B) 调度器按 $a_i(t)$ 升序成批，最大化命中常驻 KVCache；(C) KVCache 管理器用预测复用概率而非 LRU 决定淘汰哪个 block。下面的时序图刻画闭环回灌如何驱动调度/淘汰：

    sequenceDiagram participant U as User participant LB as Load Balancer (prefix-match) participant SCH as Prefix-aware Scheduler participant KV as KVCache Manager (pred. eviction heap) participant ENV as Tool Env U->>LB: task request LB->>SCH: route to node w/ largest resident prefix loop closed-loop turns SCH->>SCH: pick argmin a_i(t) (fewest new blocks) SCH->>KV: need a_i blocks; if W(t)+a_i > M evict KV->>KV: evict block with max E[τ] (not LRU) SCH-->>ENV: LLM emits tool call (tool_name,args,T_i) ENV-->>SCH: result → new request extends prefix end

    内存管理单元沿用 vLLM 的 block 级分页；淘汰在"引用计数归零"的未引用 block 上进行，block 保留其会话元数据并进入淘汰堆，堆的优先级即预测的 $\mathbb{E}[\tau_i(t)]$。

    4. 作者证明 #

    本篇无形式化定理证明，但给出一个 load-bearing 的轻量资源模型（§4），用于推导 FCFS/LRU 失效。下面复现记号、方程物理意义与检查。

    记号表 #

    符号含义
    $M$节点扣除模型权重与运行时开销后的 KVCache block 总容量
    $\mathcal{S}_t$时刻 $t$ 同驻节点的活跃会话集合
    $d_i$会话 $S_i$ 的完整前缀 block 需求（单调增长）
    $k_i(t)$$S_i$ 在时刻 $t$ 常驻 XPU 的 block 数（可复用前缀重叠）
    $a_i(t)$请求 $r_i$ 需新增分配的 block 数
    $W(t)$活跃 KVCache 工作集
    $\tau_i(t)$$S_i$ 常驻 block 的下次复用时间
    $m,\,T_i$工具元数据 $(\text{tool\_name},\text{tool\_args})$；工具调用生成时刻
    $N_{rebuild}$淘汰堆重建周期（engine 迭代数），实验取 3

    方程与物理意义 #

    工作集与预算不变式：

    $$W(t) = \sum_{i \in \mathcal{S}_t} d_i, \qquad W(t) \leq M$$

    只要总需求不超预算 $M$ 就无需淘汰。（注意：§4 正文写 $W(t)=\sum_i d_i$，而 Figure 10 标注 $W(t)=\sum_i k_i(t)$，二者在内存压力下不等价——见 §7 开放点。）

    新增 block 与准入条件：

    $$a_i(t) = d_i - k_i(t)$$

    $$\text{admit } r_i \iff W(t) + a_i(t) \leq M$$

    前缀复用 $k_i(t)$ 直接缩小分配足迹；超预算时须从其他会话淘汰恰 $a_i(t)$ 个 block。这正是前缀感知调度取 $\arg\min_i a_i(t)$ 的物理动机。

    理想（Belady 式）淘汰选择：

    $$j^{*} = \arg\max_{j \in \mathcal{S}_t,\, j \neq i} \tau_j$$

    驱逐"最久才会被用到"的会话，最小化即将到来的重算/搬运惩罚。

    预测器的条件期望估计量：

    $$\mathbb{E}\left[\tau_i(t) \mid \tau_i(t) - T_i > t - T_i\right]$$

    给定工具已运行 $t-T_i$、尚未完成，用同元数据（同用户/同项目）历史样本估计剩余时长——一个 survival-style 残余时间估计。

    检查（≥6） #

    1. 量纲：$M,d_i,k_i,a_i,W$ 皆为 block 数；$a_i=d_i-k_i$ 量纲一致。✔
    2. 边界 $k_i=0$：无常驻前缀时 $a_i=d_i$，退化为全量 prefill，符合"冷会话最贵"。✔
    3. 边界 $k_i=d_i$：前缀全命中则 $a_i=0$，请求应被调度器最高优先，与 $\arg\min a_i$ 一致。✔
    4. 准入单调：固定 $W(t)$ 下 $a_i$ 越大越易触发淘汰，符合 FCFS 撑大工作集→抖动的论证。✔
    5. Belady 退化：若所有 $\tau_j$ 相等，$\arg\max\tau_j$ 无偏好，退化为任意/LRU，说明预测收益仅来自 $\tau$ 的方差（正是工具时长跨数量级的场景）。✔
    6. 松弛自洽：淘汰只需 $\tau$ 的相对序、甚至只需 top-1，故预测器只需保序而非无偏回归——与"轻量元数据估计器≈oracle"的实测（§6.2、§6.5）自洽。✔
    7. 首阶映射：Table 3 显示 bash 参数簇的 P50 从 python one-liner 的 0.1s 到 mypy 的 10s（P99 182s），跨两个数量级，正是 per-tool 分布不足、需 per-cluster 分布的直接证据。✔

      8. 5. 实验与数据 #

      环境：2×H200（TP=2，282GB），AMD EPYC 9534；Qwen2.5-Coder-32B-Instruct；chunked-prefill（512 tokens/chunk）。80% 会话离线训练预测器，其余回放评测；人类空闲视为会话边界。基线：vLLM（LRU+FCFS）、InferCept（FCFS + 移动平均时长预测）；消融：CacheWise*（真值时延 oracle）、vLLM(Prefix-aware)、InferCept(Prefix-aware)。

      Paper's Figure 13（caption: "Session completion time (in seconds) for different KVCache management systems under the coding agent workload (sampled from CATraces)."）。 $N\le10$ 各系统相当（无内存竞争）；$N>10$ CacheWise 完成时间低 2.7×–3.5×，且逼近甚至偶尔略超 CacheWise*（归因于淘汰与调度的耦合动态）。

      Paper's Figure 14（caption: "Serving efficiency... (a) Goodput... (b)-(e) LLM request latency distribution... (f) LLM request throughput."）。 $N>10$ 下 token goodput 高 1.64×–2×、吞吐高 1.5×–2×；P50/P90 请求延迟低 13–14×，但 P99 更高——前缀感知调度会推迟无历史的新请求，这是被有意接受的取舍（session 完成时间才是用户可见指标）。

      Paper's Figure 16（caption: "Impact of prefix-aware scheduling policy. For each system, we compare the FCFS variant with the corresponding Prefix-aware variant."）。 仅加前缀感知调度：$N=30$ goodput ~1.38×–1.64×、$N=40$ ~1.6×–1.7×；会话完成时间降 ~1.8×–2.66×。说明两项优化相互独立，且不依赖准确的复用预测即生效。

      Paper's Figure 17（caption: "KVCache movement incurred (in TB of data) during the duration of each experiment..."）。 通过降低淘汰频率，CacheWise 把跨层搬运量降 ~2×–2.6×，对 LMCache/Mooncake 这类分层 offload 系统是互补收益。物理搬运（GPU↔CPU over PCIe，$N=30$）配置下会话完成时间仅低 ~1.19×，因为 PCIe 传输远快于大前缀重算，改进空间本就更小。

      Paper's Figure 19（caption: "Eviction count and session completion time... CX=Tool argument based clustering with X clusters. Hint=Trace replay with ground truth tool durations."）。 粒度从 Point Estimate→Tool-Name→C20/C50/C100 递增，淘汰数与会话完成时间单调下降；C100 最佳，会话完成时间最多改进 19%；超过 100 簇数据不再支持更细粒度（预测器的数据上限）。

      6. 论证链 #

      #步骤依据（篇内）
      1采集真实 CATraces，量化编程 agent 三特征：闭环（工具:用户=20×）、长会话（中位 36min）、大增长前缀（prefill:decode ~21×）§3, Table 1, Fig 3–8
      2用资源模型形式化 $W(t)\le M$、$a_i=d_i-k_i$，推出 FCFS 撑大工作集→抖动、LRU 忽略未来复用两条失效§4, Fig 10
      3由 $\arg\min a_i$（调度）与 $\arg\max\tau_j$（Belady 淘汰）给出理想策略§4→§5.1/§5.2
      4关键松弛：淘汰只需 $\tau$ 相对序 / top-1 正确，故可用工具元数据 TF-IDF+KMeans 轻量预测§5.2
      5在 vLLM 落地（~2500 LoC，block 附会话元数据，淘汰堆，$N_{rebuild}=3$）§5.3
      6端到端评测：完成时间 2.7×–3.5×、淘汰/搬运 2–2.6×、逼近真值 oracle§6.2–§6.4
      7消融证明两技术独立生效；预测粒度 C100 单调最优（19%）§6.3, §6.5

      7. 实现 cross-reference #

      • 实现规模与位置：~2,500 行 Python，改动 vLLM 的 batch scheduler 与 KVCache block manager；给每个 block 附加会话级元数据 tool_name, tool_args, T_i（L1 §5.3 para 1）。
      • 淘汰堆机制：引用计数归零的未引用 block 保留元数据、按预测 $\mathbb{E}[\tau_i]$ 作优先级入堆，替换默认 LRU；同会话多 block 共享一次预测，避免逐 block 调用预测器（L1 §5.3 para 2）。
      • 预测器细节（易漏）：(1) TF-IDF 向量上限 5000 lexicons、scikit-learn ℓ2 norm 加权；(2) KMeans 簇数上界取很大以最大粒度，实测数据在 100 簇饱和（L1 §5.2 footnote 2/3、§6.5）。这两点是"轻量却接近 oracle"的落地关键。
      • 预测陈化处理：$\mathbb{E}[\tau_i]$ 随工具执行推进而陈化，故每 $N_{rebuild}=3$ 次 engine 迭代重建整个淘汰堆以刷新估计（L1 §5.3 para 3）——freshness 与开销的可调平衡点。
      • 开源：数据集与代码 github.com/cachewise-project/cachewise-coding-traces（L1 §0/§8）；系统源码具体文件行号 [实现未公开于本文正文]。

      8. 开放问题 #

      • 工作集定义不一致：§4 正文 $W(t)=\sum d_i$ vs Figure 10 $W(t)=\sum k_i(t)$，内存压力下不等价，篇内未澄清。
      • 超越自身 oracle：CacheWise 偶尔略胜拥有真值时延的 CacheWise*，归因于淘汰与前缀调度的耦合动态。
      • P50 巨幅改进但 P99 回退：13–14× 中位改进伴随尾延迟恶化，取舍需在 session 指标下才成立。
      • SGLang 未直接对比：以 InferCept(Prefix-aware) 消融代替 SGLang 场景，代表性存疑。
      • 分布漂移：预测器依赖历史轨迹，仓库/工具演化后需周期重训，鲁棒性留待未来工作。
