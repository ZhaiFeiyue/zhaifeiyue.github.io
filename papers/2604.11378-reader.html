From Agent Loops to Structured Graphs: A Scheduler-Theoretic Framework for LLM Agent Execution

Hu Wei · arXiv 2604.11378 · 2026-04-13 agentframeworkschedulingDAGposition paper
一句话论文：Agent Loop = $|\mathcal{U}|\le1$ 的 scheduler，SGH 用静态 DAG 把它升维到 $|\mathcal{U}|\ge 1$

核心论点

Agent 执行可统一为 scheduler 问题，ready-set 基数 $|\mathcal{U}|$ 和 policy 显式性 是关键坐标。

核心方法

静态不可变 DAG + 三层分离 (planner/runtime/recovery) + 三级升级 recovery 协议。

核心结论

Position paper，无实验；给出 7 组对照协议把总增益分解为 $G_{\text{plan}},G_{\text{scaffold}},G_{\text{graph}},G_{\text{patch}},G_{\text{replan}}$。

🗺 研究背景与现状

A · 时代定位

从 "Agent Loop 野蛮生长" 到 "Graph orchestration 沉淀" 的拐点

2025-2026 年 LangGraph/AutoGen/CrewAI/TDP/DynTaskMAS 各家往 graph 结构收敛，但每家只解决局部问题。此时缺一个能把这些系统放在同一尺度比较的理论框架。SGH 不是新系统，而是 命名和坐标系——把 "Agent Loop 不够用" 从工程直觉升格为 $|\mathcal{U}|\le 1$ 的形式化命题。

B · 领域定位

LLM agent runtime / orchestration 层

介于 LLM serving (vLLM/SGLang) 和 app framework (LangChain/AutoGen) 之间，专门负责 "下一步该执行哪个单元、怎么处理失败、怎么审计"。现实痛点：coding agent 跑到一半 LLM 决定换策略，结果没人知道最终执行的是哪个计划。

C · 核心痛点

Agent Loop 的三大结构性缺陷

①依赖隐式 步骤间依赖只存在于 context，LLM 忘了就乱序；②recovery 无界 LLM 自主决定重试/跳过/replan，没有上限；③plan 可被静默改写 LLM 中途换策略，旧计划被 context 覆盖，事后无法 audit。survey 显示 60% 项目都是 Agent Loop 变体。

D · 本文目标

能否把 agent 执行改造成可审计、有界、可并行的 scheduler？

要同时满足：(1) 多就绪单元并发 $|\mathcal{U}|\ge 1$；(2) 确定性调度策略；(3) recovery 有界；(4) plan 不可变。以前的系统每次只满足其中 1-2 个。

E · 设计空间约束

"为何不可 X" 的穷举

为何不可 LangGraph 式动态图？ Runtime 改图 → audit trail 成移动目标。
为何不可 first_of 竞争并行？ Loser cancellation 需补偿协议，引入新失败模式。
为何不可递归子图展开？ $|V|$ 无界破坏 bounded termination。
为何不可 parent-chain rollback？ 违反 plan invariant。
挤出的唯一可行点 = 静态 DAG + all_of/any_of + 三级 recovery + plan versioning。

F · 提出框架

Structured Graph Harness (SGH)

通过 把控制流从 context 提取到显式静态 DAG 实现可审计的 agent 执行。三个设计承诺：(1) plan 在 version 内不可变；(2) planning/execution/recovery 三层分离；(3) recovery 严格升级 `local_retry → local_patch → request_replan`，跳级禁止。

G · 技术挑战

非确定 LLM 节点下的形式保证

挑战 1：contract 验证本身可能错 → 用 $p_v$ 显式给 validation gap；
挑战 2：节点可能死循环 → $\tau_v$ 超时 + $b_v$ 预算 + terminal absorbing 证 bounded termination；
挑战 3：failure 历史污染下一步 → $\mathcal{C}_{\text{exec}}\cap\mathcal{C}_{\text{diag}}=\emptyset$ context 分区。

H · 设计绑定 (强制前提)

SGH 不是免费午餐

① 任务依赖必须 upfront 枚举 → exploratory 任务不适用；
② 必须有可靠 contract 语言 → 创意生成类无 schema；
③ 必须有可分类的 side-effect → `run_shell` 这种模糊 tool 难归类；
④ planner 必须比 executor 强 → 和 "弱 executor 省钱" 思路冲突。

🏗 核心架构图解

Why 不能用 LangGraph 动态图？Runtime add_node/add_edge 让 plan 版本成为移动目标，audit trail 无法重建"哪个版本的 plan 管了哪些 action"。SGH 用 plan version immutable 换掉这个能力，换来可审计性。
Why 静态 DAG 是正确的承诺？DAG 保证三件事：acyclicity → ready set 始终能前进；finite |V| → 有界终止；拓扑序 → canonical schedule，$\mathcal{P}$ 可以 refine 但不能 violate。这些都是 compile-time 可验证的。
核心技术壁垒
把 non-deterministic LLM 节点 + 可能错的 contract validation 塞进一个仍然能证 bounded termination + conditional soundness 的状态机。$\tau_v$/$b_v$ 上界 + terminal absorbing 是 termination 证明的机械护栏；$\Pr\ge\prod p_v$ 是 soundness 的诚实边界。
Assumption ChallengeContract validation 的 $p_v$ 被假设独立——但在 LLM 做 semantic validation 时，验证 LLM 和执行 LLM 同源（都是 GPT-4），独立假设可能破，乘积 bound 会严重高估真实可靠性。

🔬 第一性原理

A · 约束推导

为什么 "给 Agent Loop 加 planner" 不够？vs. "Plan-and-Act / Routine 这类系统已经解决了吧？"

在 scheduler 框架下，Plan-and-Act / TDP 这类系统的 plan 是给 LLM 的 提示，但执行层每一步仍然由 LLM 自主决策 "下一个做什么"。$\mathcal{P}$ 仍然是 LLM 推理这个非确定关系，$|\mathcal{U}|$ 仍然 ≤ 1——plan 的存在没有改变 scheduler 类型。真正的多 ready 单元并发需要 plan 成为 调度结构 而不仅仅是 prompt 内容。

"加 planner" 只在 policy 显式性轴上前进，不能跨到多 ready 单元象限。
A · 约束推导

为什么 first_of 竞争性并行必须被排除？vs. "并发跑多个方案、取第一个成功的不是很香？"

first_of 要求 loser cancellation——已经开始跑的节点要中途停掉。两个硬问题：① LLM generation 是 stateful 的，mid-generation 取消需要 compensation 协议（回滚副作用、处理部分输出），这本身是个分布式事务级的坑；② commit-point 歧义：A 先成功了，但 B 可能质量更高，语义上算谁赢不明确。

any_of 的 "all-dispatch + 后来者 skip" 是控制性更高的近似，代价是"输家继续跑完"。
B · 假设审计

"planner 能识别任务并行" 这个前提在什么条件下崩？vs. "规划是 LLM 擅长的事，应该没问题吧？"

作者自己在 §9.3 坦白：survey 的 70 个项目里只有 30-40% 任务有 natural parallelism，其余 60-70% 本质是线性链。对线性任务，SGH 退化为 $|\mathcal{U}|=1$ scheduler，多 ready 单元的所有好处消失，只剩 bounded recovery + audit 两项附加价值。笔者也不是十分确定作者的 30-40% 估计本身——样本是开源仓库，有 coding-agent 偏差，真实 enterprise workload 比例可能更低。WorfBench 显示即使 GPT-4 在 graph planning 上也比 sequence planning 低 15%，说明 planner 的并行识别能力是 bottleneck。

SGH 的收益上限由 planner 识别并行的能力决定。
B · 假设审计

"Validation 错误可独立" 假设在 LLM 验证时崩vs. "对每个节点做 contract 验证不就完事了？"

Theorem 6.3 的 $\Pr[\text{all correct}]\ge \prod_v p_v$ 要求 各节点验证错误独立。在 syntactic validation (JSON schema) 时这没问题，因为是代码在判。但当验证由 同一个 LLM 做（"判断 fix 是否正确"），执行 LLM 和验证 LLM 共享相同的 failure mode——比如都理解错了一个 edge case。这时 $p_v$ 的独立性不成立，真实 joint 概率远低于乘积。

用 LLM 验证 LLM 是相关误差，不是独立误差。
C · 设计合理性

为什么 context 严格分区 C_exec ∩ C_diag = ∅？一个细节里的 insight

常见 Agent Loop 的失败模式是：节点 A 失败 → 把失败信息拼到 context → 节点 B 读到这个失败信息 → B 的行为被"失败 narrative"扭曲（比如变得过度谨慎或拟合失败模式）。SGH 把 failure history 锁在 $\mathcal{C}_{\text{diag}}$ 里，只给 recovery 层看；execution 节点只能看 $\mathcal{C}_{\text{exec}}$ 的合法输入/artifact。这让每个节点的执行 只依赖 DAG 结构化输入，而不是"前面发生了什么故事"。

诊断信息只能通过 planner 生成新 plan version 才能进入 execution 路径。
C · 设计合理性

为什么 recovery 升级必须在 API 边界强制，而不是"约定"？从 unsafe 类比得到的工程智慧

论文把 recovery API 只暴露 3 个入口 `attempt_retry / attempt_patch / request_replan`，每个入口 check `recovery_state[v]` 是否到位。这不是"写在 doc 里希望开发者遵守"，是 机械护栏——你 call `request_replan` 时只要有任何节点还没 `patched`，API 直接拒。类比 Rust 的 `unsafe`：只要代码在 safe API 边界内，invariant 必然成立；破坏 invariant 必须显式绕过 API（这时 SGH 不再对正确性负责）。

规范如果不可机械强制，就不是规范，是愿望。

📑 关键段落双栏批注

§1.3We observe that the Agent Loop is, at its core, a single-ready-unit scheduler: at any point during execution, at most one executable unit (tool invocation, sub-task, or reasoning step) is active, and the choice of the next unit is the output of an opaque LLM inference. This reframing lets us place Agent Loops and graph-based execution engines on a single semantic continuum.
段落功能 · 核心 reframing

全文的"阿基米德点"。这一句话把 Agent Loop 从一个工程模式降维成一个 scheduler 分类里的特殊情形——$|\mathcal{U}|\le 1$ + 非确定 $\mathcal{P}$。一旦接受这个视角：

① 半世纪的 DAG scheduling 文献直接可用；
② "Agent Loop vs Graph" 变成 "$|\mathcal{U}|=1$ vs $|\mathcal{U}|\ge 1$" 的可测量对比；
③ 各种中间形态（加 planner / 加 scaffold / 加 parallel tool calls）被精确定位到 continuum 的不同点。

⚠️ 这是 reframing 而非新方法——论文 80% 的价值在这个命名里。

§4.2In the face of uncertain benefit, prioritize the predictability and verifiability of the execution process. [...] Our qualitative analysis of 70 agent systems showed an observed pattern: failure-loop behavior was frequently observed among the graph/flow orchestration systems in our dataset (3 out of 4 projects), while such behavior was rarely observed in state-machine-based systems (0 out of 7 projects).
段落功能 · Principle 1 的证据链

Principle 1 "controllability first" 的经验基础。3/4 vs 0/7 的对比是作者最 aggressive 的经验 claim，但样本太小（4 个 + 7 个）且分类主观。

💡 真正可信的是后半句 "面对不确定收益时，优先保证可预测性" 的决策原则——这是经典的软件工程取舍，和数据库的 "优先 correctness 再谈 performance" 同构。

🔍 交叉验证：这个原则和 Temporal / Cadence workflow engine 的设计哲学一致。

§6.2The escalation invariant is enforced structurally through a per-node recovery state counter: recovery_state[v] ∈ {pristine, retried, patched}. The recovery layer exposes exactly three entry points [...] A call to attempt_patch is rejected unless recovery_state[v] ≥ retried; a call to request_replan is rejected unless all failed nodes have recovery_state[v] ≥ patched. This makes the escalation invariant mechanically enforceable rather than merely normative.
段落功能 · 从"规范"到"机械护栏"的升级

这是 SGH 工程落地时最可复用的 pattern。很多系统声称有 "retry 预算" 或 "escalation 规则"，但实际是个配置参数——开发者可以 bypass。

SGH 的做法：把规则编码到 API 入口参数校验里。调用 `request_replan` 时，runtime check 所有 failed node 的 recovery_state 到 patched 为止。没到？直接 throw。

🔑 类比：这是 Rust type system 的思路——把不变量交给类型系统而不是程序员的自律。

§9.8SGH is a multi-ready-unit scheduler only when the plan's DAG contains independent paths. If the planner produces a linear chain, then $|\mathcal{U}|\le1$ at every state, and Graph Harness degenerates to a single-ready-unit scheduler. [...] SGH's value is bounded by planner quality.
段落功能 · 作者罕见的自嘲式坦白

整篇论文最诚实的一段。SGH 号称的多 ready 单元优势只在 DAG 有并行分支 时成立。如果 planner 只会画线性链——而 60-70% 任务本来就是线性的——SGH 退化为 $|\mathcal{U}|=1$，剩下的好处只有：

  • bounded recovery (三级升级)
  • audit trail (plan version)
  • context 隔离 (C_exec / C_diag)

这些仍然有价值，但不是论文的"主打功能"。⚠️ 实际工程决策时要先看你的任务分布，别被 Figure 1 这种精心设计的并行 DAG 误导。

📐 核心指标解读 (Protocol 定义)

G_graph · 图结构增益 Perf(G4) − Perf(G3)

💡 生活类比：把单车道改多车道后的通勤时间节省。G3 是"计划好了但只能单线跑"，G4 是"计划好了可以多线并发跑"。差值就是多车道的净贡献。

含义：单独衡量从单 ready 单元升级到多 ready 单元带来的收益，剔除规划本身的贡献。方向：↑ 越大说明并行调度越值得投入。论文理论预测 $G_{\text{graph}}>0$ 且随任务复杂度增长。现状：未实验。

G_replan · 重规划增益 Perf(G6) − Perf(G5)

💡 生活类比：在 Level 1 retry + Level 2 patch 基础上加"可以整个计划重来"这一招的价值——只有在任务失败率高时才有用。

含义：Level 3 replan 相对"只有 retry + patch"的净收益。方向：↑ 但论文预测只在 failure-prone task 上 > 0，顺利任务可能为 0。

Failure-loop rate · 失败循环频率 不是一个 rate，是一个事件

💡 生活类比：刻舟求剑——LLM 每次都用同样方法失败，context 越堆越多但没新信息。Agent Loop 的"经典翻车场面"。

含义：系统进入非生产性重试循环的频率。方向：↓ 越低越好。SGH 的三级 recovery 理论上能把这个 rate 压到 0（有限 budget + 有限 replan 次数）。观察：survey 中 graph/flow 系统 3/4 被观察到 failure-loop，state-machine 0/7。

Validation gap · 验证鸿沟 Pr[all correct] ≥ ∏ p_v

💡 生活类比：10 个环节的质检流水线，每环节准确率 95%，整条线真正全对的概率只有 60%。SGH 让这个数字"可见"。

含义：即使所有节点都通过 contract 验证，整体正确概率仍受验证器本身可靠性拖累。意义：Theorem 6.3 把这个通常被工程掩盖的事实显式写进 soundness bound，是 SGH 的诚实之处。

七组对照实验 (G0-G6) · 归因分解实验

类型归因设计 (attribution design)

怎么设计7 个 group，每 group 相对前一个 只加一个 feature：G0 SOTA Loop (Claude Code 级) · G1 Naive Loop · G2 +planner · G3 +scaffold · G4 +graph · G5 +patch · G6 +replan。任务集：50 个跨编程/数据分析/运维的任务。

🚀 科研改进方向

这是一篇 position paper，改进方向天然开放。以下 6 个方向按 "top-venue 可发表性" 排序。

① 真正做 7 组对照实验 · Empirical validation

⭐⭐

问题：论文最大黑洞——所有理论预测都没验证。G_graph 到底多大？在什么任务上退化？Agent Loop + prefix cache 是否已经吃掉了多 ready 单元的一半好处？

  • 方案 1：基于 LangGraph 的 runtime 强制加 plan invariant + 三级 recovery + contract validation，对比原版
  • 方案 2：在 SWE-bench / WebArena / GAIA 上跑 G0-G6 七组
  • 方案 3：curate 新 benchmark 专门用于测 DAG 并行识别能力（补 WorfBench 空白）
  • 方案 4：消融 $p_v$——故意降低 contract 验证准确率，观察 system-level 降级曲线

② Planner 并行识别能力提升 · Parallelism-aware planning

⭐⭐⭐

问题：SGH 收益上限由 planner 决定。WorfBench 显示 GPT-4 graph planning 比 sequence planning 低 15%。如何训一个专门识别任务并行结构的 planner？

  • 参考：G-Designer (2025) 用 VGAE 自动设计 topology
  • 参考：AFlow 用 MCTS 搜索 workflow graph
  • 方案：SFT + RL 专训 "parallel vs sequential" 分类头，plan 输出阶段加 DAG-level reward

⭐⭐⭐

  • 方案：引入 "plan version epoch"——允许在 epoch 边界 commit 新 subgraph（有界深度）
  • 方案：parent-chain rollback 只允许在同 epoch 内，跨 epoch 必须走 replan
  • 类比：Git 的 merge vs rebase 边界

④ Contract validation 的可靠性建模 · Validation-gap aware orchestration

⭐⭐

问题：Theorem 6.3 假设 $p_v$ 独立，LLM validating LLM 时假设破。

  • 方案 1：用 Ensemble validator（3 个不同 base model 投票）估算真实 $p_v$
  • 方案 2：在线 calibrate——随机抽样人工审核，更新 $p_v$ 估计
  • 方案 3：联合建模执行-验证的相关误差矩阵而非独立乘积

⑤ SGH on serving stack · 多 ready 并发的 KV cache 策略

⭐⭐

问题：一个 plan 内多节点并发发起 LLM 请求，但它们共享大量 system prompt。现有 prefix cache 不感知 "同 plan family"，会重复计算。

  • 方案：vLLM / SGLang 加 plan-id tag，同 plan 内请求共享 KV 页
  • 方案：预测 DAG 接下来的 ready 节点，preemptive allocate KV 给可能并发的 request
  • 参考：Sutradhara (2601.12967) 讨论过 orchestrator-engine co-design，可结合

⑥ Hybrid routing · 简单任务走 loop，复杂任务走 SGH

⭐

问题：H4 假设 SGH 在简单任务上有固定开销。如果验证成立，生产系统需要自动路由。

  • 方案：task complexity classifier（LLM-based，~100ms）→ route 到 Agent Loop 或 SGH
  • 方案：投机执行——同时起 loop 和 SGH planner，谁先出结果用谁

B · 实践部署上下文

适合场景：coding agent (Claude Code / Cursor / Codex)、CI/CD 自动修复、数据分析 pipeline、运维 incident response、合规要求强的企业工作流。不适合：deep research agent、creative writing、open-ended 探索。
Serving 栈位置：SGH 是 agent runtime 层，下面套 vLLM/SGLang 做 LLM inference（需要 prefix cache、continuous batching、function calling JSON mode），上面套 LangChain/AutoGen 做 app。
硬件偏好：多 ready 并发对 LLM serving 的 并发能力 要求高；对 GPU 架构无特殊偏好（不是 kernel 级工作）。
成本画像：token 上可能节省（TDP 同思路报告 82% 削减），因为 context 隔离；但 peak QPS 更高，对 serving 基础设施要求更高。

C · 生态影响预测

Paper 2026-04 发布，目前无下游采用。影响路径预测：
① 学术：scheduler continuum 语言被后续 graph agent 论文引用，成为标准坐标。
② 工业：Claude Code / Codex / Devin 这种 coding agent 借鉴 plan-versioning 做可审计 trace。
③ 框架：LangGraph 可能引入"immutable plan"模式作为高可靠选项；Temporal / Inngest 这种 workflow engine 可能往 LLM-native 侧融合。
📌 关键看点：作者承诺的 companion paper（EGA with prototype）如果出来，会决定 SGH 是变成标准还是停留在 paper。

D · 资深教授判断

"未来 5 年 agent 基础设施的主干道不是'多 agent 辩论'，而是可审计的 agent runtime——就像 OS 从 DOS 走到多任务 + 隔离 + 审计日志。SGH 的历史地位不取决于它自己的 prototype 表现，而取决于它命名的 scheduler continuum 是否成为下一代 agent 框架的默认词汇。"

📝 Worksheet 答题卡

1

Q1 · 定义理解：ready-set 基数 $|\mathcal{U}|$ 是什么？为什么它比"是否有 planner"更本质？

$|\mathcal{U}|$ 是某一时刻可被调度的 executable 单元数。Planner 只改变任务描述方式，不改变 runtime 每一步能选择的单元数；只有显式 DAG 才让 $|\mathcal{U}|>1$。
  • §3.2 Definition 3.2：$|\mathcal{U}(s)|\le 1$ 定义 single-ready-unit
  • §3.3 Definition 3.3：存在 state 使 $|\mathcal{U}(s)|>1$ 定义 multi-ready-unit
  • §3.4 Figure 2：加 planner / scaffold 只在 policy 轴前进，$|\mathcal{U}|$ 不变

⚡ 反例：OpenAI parallel tool_calls 有 $|\mathcal{U}|\ge 1$ 但策略非确定，仍归为 Agent Loop 变体

2

Q2 · 方法对比：SGH vs LangGraph 的本质区别是什么？

不是结构性差异而是 哲学差异：LangGraph 追求最大灵活性（runtime mutable graph），SGH 追求最大可控性（plan immutable within version）。表达力 LangGraph 更强，审计性 SGH 更强。
  • §2.4 Table 2：plan mutability / recovery / routing / attribution 四维对比
  • §9.2：作者自承"这是互补设计点"，SGH 不比 LangGraph 好，是不同
  • 适用场景：engineering task → SGH；exploratory task → LangGraph
3

Q3 · 架构设计：三层分离为什么必须严格？能不能让 runtime 直接 call planner？

不能。一旦 runtime 能在执行中 trigger planner，plan 就从承诺变成"随时可改的建议"，违反 Plan Invariant，audit trail 成为移动目标。
  • §5.3 三层分离架构 + Figure 3
  • §5.4 Context Partition：$\mathcal{C}_{\text{exec}}\cap\mathcal{C}_{\text{diag}}=\emptyset$，诊断信息不得进入执行 context
  • 唯一 replan 通道是 recovery 层 request_replan，生成新 plan version
4

Q4 · 关键机制：三级 recovery 为什么必须强制升级、禁止跳级？

避免"LLM 一遇到失败就 request 全盘重规划"这种 failure loop。Level 1 retry 解决网络/timeout 这种轻微问题；Level 2 patch 解决 contract 级配置问题；Level 3 replan 是最后手段，改动大、成本高、可审计影响大。
  • §6.2 Definition 6.3 + Proposition 6.4 Escalation Invariant
  • 机械护栏：recovery_state[v] ∈ {pristine, retried, patched}，API 入口强制 check
  • survey 观察：Agent Loop 普遍无 recovery 上限 → failure loop 常见
5

Q5 · 实验解读：为什么 G0 = "SOTA Loop" 被放进 baseline 里？缺了会怎样？

如果只对比 SGH vs naive loop，任何改进可能被解释为"你给了模型更多上下文"而不是"结构起了作用"。加 G0 (Claude Code / Codex 级) 保证对手不是 strawman。
  • §8.1：G1−G0 isolates information-vs-structure effect，若 G1-G6 全低于 G0 → 结构无用
  • G2−G1 isolates planner；G4−G3 isolates graph；G6−G5 isolates replan
  • $G_{\text{total}}=G_{\text{plan}}+G_{\text{scaffold}}+G_{\text{graph}}+G_{\text{patch}}+G_{\text{replan}}$
6

Q6 · 多任务逻辑：static DAG 在什么任务上必然失败？

依赖结构本身 emergent 的任务——deep research、creative writing、"调查 outage 修坏的东西"这种 fix depends on diagnosis depends on investigation 的链式任务。这些任务 upfront 无法枚举节点，SGH 会频繁触发 Level 3 replan。
  • §9.7 Boundary of Applicability: exploratory / dynamic goal / creative generation 三类不适用
  • §10.4 Limitation 2: Static DAG Assumption
  • 作者建议这些任务用 LangGraph 或 Agent Loop with inline replan
7

Q7 · 性能分析：$G_{\text{graph}}$ 什么情况下会 ≈ 0 甚至为负？

(1) 任务本质线性（survey 估 60-70%），planner 生成线性链，SGH 退化为 $|\mathcal{U}|=1$；(2) Simple 任务，SGH 的 DAG validation / WAL / 状态机开销 >= 并行节省（H4 假设）；(3) Planner 错误识别并行，spurious dependency 反而强制了不必要的 serialization。
  • §3.7 Planning Failures 列 5 种错误
  • §9.8 "Degenerate single-ready-unit Scheduler"
  • §10.2 H4 Fixed-Overhead Hypothesis → 可能需 dual-path routing
8

Q8 · 迁移价值：这篇 position paper 对 不做 agent runtime 的工程师有什么迁移意义？

两点最可复用：① 用 API 边界强制 invariant——把规则编码到参数校验而非 doc 和自律里，类 Rust unsafe 思路；② 用 scheduler 语言看"下一步做什么"类问题——不只是 agent，任何"ad-hoc 决策 vs 显式策略"的系统都可以用 $|\mathcal{U}|$ + $\mathcal{P}$ 类型参数化。
  • §6.2 Mechanical Enforcement pattern 可迁移到分布式事务、权限系统、限流器
  • §3.4 Scheduler Continuum 是一个 命名工具——给任何有选择的系统一个坐标
  • §5.4 Context Partition pattern：执行信息 vs 诊断信息分离，可迁移到 logging / tracing 架构
