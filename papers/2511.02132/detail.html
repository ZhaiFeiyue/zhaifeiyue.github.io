Optimizing Attention on GPUs by Exploiting GPU Architectural NUMA Effects

Mansi Choudhary, Karthik Sangaiah, Sonali Singh, Muhammad Osama, Lisa Wu Wills, Ganesh Dasika | 2025-11 | https://arxiv.org/abs/2511.02132 Category: kernel | Tags: attention, NUMA, MI300X, chiplet, L2-cache, swizzling, FlashAttention Read: 2026-04-16

Core Contribution

在 AMD MI300X 的多 chiplet (8 XCD) 架构上，识别出 FlashAttention2 的 workgroup 调度未考虑 NUMA 效应导致 L2 cache 失效，提出 Swizzled Head-first Mapping 策略——将同一 attention head 的所有 workgroups 映射到同一 XCD，使 K/V 数据在单 XCD 的 L2 cache 中被复用。以极小的代码改动（~15 行 swizzle 逻辑）实现最高 50% 性能提升和 80-97% L2 cache hit rate。

Summary

现代 AI GPU 正从单 die 向多 chiplet 架构演进（MI300X 8 XCD、Blackwell 双 die、Rubin Ultra 四 die），引入了显著的 NUMA（Non-Uniform Memory Access）效应：不同 XCD 拥有独立的 L2 cache 和 HBM 控制器，跨 XCD 访问延迟和带宽显著劣化。然而，现有的 FlashAttention2 实现假设 GPU 内存访问是 uniform 的，其 workgroup 调度策略（round-robin 分配到各 XCD）将同一 attention head 的 workgroups 分散到不同 XCD，导致 K/V 数据在每个 XCD 的 L2 cache 中只有部分命中，产生大量冗余 HBM 访问。

本文提出 Swizzled Head-first Mapping：将 attention 的计算网格按 head-first 顺序迭代，并通过 swizzle 逻辑将同一 head 的所有 row blocks 强制映射到同一 XCD。这样，每个 XCD 在某一时刻只服务一个 Attention Compute Cluster (ACC)，K/V 数据只需从 HBM 加载一次到该 XCD 的 L2 cache 中，被该 head 的所有 workgroups 共享复用。

在 MI300X 上的实验覆盖 MHA（DeepSeek-V3 prefill, H=128）和 GQA（Llama 3 系列），Swizzled Head-first 在 MHA H=128 / seqlen=128K 下比传统 Block-first 映射快 50%，L2 cache hit rate 从 ~1% 提升到 90-97%。Forward 和 backward pass 均有效。

逻辑故事还原（约束推导型）

时代定位

2024-2025年，GPU 架构正经历从单 die 到多 chiplet 的范式转换。A100/H100 是单 die 统一 L2 cache，所有 CU 对 L2 的访问是 uniform 的。MI300X（8 XCD chiplet）和 Blackwell（2 die）打破了这个假设——L2 cache 被物理分散到各 die/XCD 中，NUMA 效应出现。GEMM 领域已经通过 CTA swizzling（Tensile, hipBLASLt）解决了这个问题（L2 hit rate 从 43% → 92%），但 attention kernel 领域还没有人系统性地解决 NUMA-aware 调度问题。本文将 GEMM swizzling 的思路迁移到 attention 领域，是一个 "低垂但重要的果实"。

约束推导（为何不可X？）

为何不可继续用 Naive Block-first (round-robin) 调度？

→ Round-robin 将 wg0 分到 XCD0, wg1 分到 XCD1, ...，同一 head 的 workgroups 被均匀打散到 8 个 XCD。每个 XCD 只服务该 head 的 1/8 row blocks，但需要完整的 K/V 数据。因此 K/V 在每个 XCD 的 L2 cache 中只有 1/8 的复用率，其余 7/8 是冗余加载。当 head 数多（H≥64）、序列长（≥32K）时，K/V 数据远超单 XCD 的 4MB L2 cache，命中率暴跌至 ~1%（Figure 13）。

为何不可只用 Swizzled Block-first？

→ Swizzled Block-first 在 GQA（KV heads 数 = XCD 数 = 8）时效果好（每个 GQA group 恰好映射到一个 XCD），但在 MHA（如 DeepSeek-V3 的 H=128）时，128 个独立 head 需要 128 个 ACC，每个 XCD 同时服务 128/8 = 16 个 ACC，L2 cache 仍被 16 个不同 head 的 K/V 数据竞争，命中率在长序列下暴跌。

为何不可只用 Naive Head-first (无 swizzle)？

→ Head-first 迭代顺序对了（先完成一个 head 的所有 blocks 再到下一个 head），但 round-robin XCD 分配仍将同一 head 的 blocks 打散到所有 XCD。在短序列时尚可（K/V 小，各 XCD 的 L2 都装得下），但长序列（128K）时 K/V 超出单 XCD L2，打散导致命中率从 90% 跌至 40-60%（Figure 13）。

为何 NVIDIA Blackwell 不需要这个优化？

→ Blackwell 在硬件层面维持了两个 die 之间的 full cache coherency（Figure 1b），对软件隐藏了 NUMA 效应。而 MI300X 显式暴露了 NUMA 特性，需要软件层面的 NUMA-aware 调度。这是一个重要的 AMD vs NVIDIA 架构差异：NVIDIA 选择硬件兜底，AMD 选择给程序员更多控制。

核心技术壁垒

ACC (Attention Compute Cluster) 到 XCD 的 1:1 映射。核心 insight 极其简单——"同一 head 的所有 blocks 放在同一 XCD"——但需要精确的 swizzle 逻辑来在 round-robin 硬件调度器之上实现这个映射。Figure 11 的 ~15 行 Triton 代码是全部实现。

真正的壁垒不在代码，而在 对 MI300X 内存层次结构的深度理解：(1) L2 cache 是 per-XCD 私有的 4MB；(2) 硬件 WG 调度器使用 chunked round-robin（chunk=1）；(3) 这个调度策略是 driver-level、跨代 GPU 可能变化的；(4) 因此 swizzle 必须做成 kernel-level 的 mutable 逻辑而非依赖固定的硬件行为。

质疑假设

  1. MI300X 的 chunk=1 round-robin 调度假设：论文指出这是 "current hardware" 的行为，未来 driver 更新可能改变。如果 AMD 在驱动中实现了更智能的 NUMA-aware 调度，这个优化可能变得不必要。
  2. L2 cache capacity 假设：方法的有效性取决于 "单个 ACC 的 K/V 数据能装入单 XCD 的 L2 cache (4MB)"。对于 HEAD_DIM=128, seqlen=128K: K 或 V 的大小 = 128K × 128 × 2B(FP16) = 32MB，远超 4MB L2。但 FA2 的 tiled access pattern 确保了在任一时刻只有 active tiles 在 L2 中，而非全部 K/V。这个隐含假设是否在所有 tile 大小和序列长度下都成立，论文未详细分析。
  3. Forward-only 评估为主：Backward pass 仅在 Section 4.6 简要评估（~10% speedup），远低于 forward 的 50%。论文暗示 backward pass 有其他瓶颈（scalar operations）限制了收益。
  4. 仅在 MI300X 上验证：MI350X (CDNA4) 有不同的 chiplet 拓扑和 L2 大小，结论是否迁移？Blackwell 因硬件 coherency 不需要此优化，但 Rubin Ultra (4 die) 可能重新需要。

    5. 设计绑定批判

    • 绑定 AMD MI300X 架构: 核心优化依赖 8 XCD + per-XCD L2 cache 的特定拓扑。迁移到 MI350X 需要调整 NUM_XCD 参数；不适用于 NVIDIA GPU（Blackwell 硬件 coherent）
    • 绑定 Triton 实现: 代码用 Triton 编写，依赖 tl.program_id 和 swizzle 逻辑。CUDA/HIP native kernel 需要不同的 swizzle 实现
    • 绑定 FA2 算法结构: Swizzle 逻辑与 FA2 的 grid 结构（batch × heads × blocks）紧耦合。其他 attention 算法（FA3、PagedAttention、MLA）需要重新设计 swizzle mapping
    • 绑定 head 数 ≥ XCD 数: 当 H < 8（如 MQA with H_Q=1），每个 head 的 blocks 不足以填满一个 XCD，方法退化

    生态影响追踪

    • AITER: AMD 的高性能 AI 算子仓库已部署了 Swizzled Block-first (Section 3.2.2)，本文的 Swizzled Head-first 是更优方案，可能被 AITER 后续采纳
    • Triton: 论文在 Triton 上实现，代码改动极小，可直接 upstream 到 Triton 的 FA 默认实现
    • 与 2511.02168 (Three Taxes) 的关联: 同为 AMD GPU inference 优化，Three Taxes 解决 inter-GPU 通信融合，本文解决 intra-GPU NUMA 调度。两者在同一集群上互补
    • CDNA4/MI350X 适用性: MI350X 保持了 chiplet 架构但 XCD 数和 L2 大小可能不同，核心方法论（ACC-to-XCD mapping）仍适用

    Key Findings

    • MI300X 上 FA2 默认调度导致 L2 cache hit rate 在高 head 数 + 长序列下暴跌至 ~1%
    • Swizzled Head-first Mapping 将 L2 hit rate 提升到 80-97%，实现最高 50% 性能提升
    • MHA (H=128) 比 GQA (H_KV=8) 收益更大——GQA 因 KV groups=XCD 数而天然 cache 友好
    • Forward pass 收益显著 (50%), backward pass 收益有限 (~10%)
    • DeepSeek-V3 prefill (MHA H=128, D=56) 是最佳 case study

    Deep Analysis (kernel, extends Phase 2)

    1. Target Operation

    • Primitive: Multi-Head Attention (forward + backward) 的 FlashAttention2 实现
    • I/O shapes: $Q,K,V \in \mathbb{R}^{B \times H \times N \times D}$, $D$=56-128, $N$=2K-128K
    • Arithmetic intensity: FA2 forward 的 FLOP = $O(BHN^2D)$, 内存访问 = $O(BHND)$ (tiled) → compute-bound for large $N$
    • Roofline: 在 MI300X 上，FA2 在长序列时是 compute-bound（MFMA 瓶颈），但在短序列或 cache miss 严重时退化为 memory-bound → 本文通过提升 L2 hit rate 将瓶颈从 HBM bandwidth 移回 compute

    2. Hardware Model

    • Target: AMD MI300X (CDNA3), 8 XCDs, 304 CUs total
    • Compute units: Matrix Cores (MFMA instructions), 每 XCD 38 CUs
    • Memory hierarchy: L1 (16KB/CU) → L2 (4MB/XCD, 私有) → HBM3 (192GB, 5.3TB/s)
    • NUMA topology: 8 个 XCD 各有独立 L2 和 HBM 控制器。跨 XCD L2 访问需通过 Infinity Fabric，延迟和带宽显著劣化
    • WG scheduler: 硬件 round-robin, chunk=1（每个 XCD 依次分配 1 个 WG）

    3. Design Space & Constraint Analysis (extends Phase 2)

    见上方约束推导——覆盖了 4 种 mapping 策略的 feasibility matrix 和 NVIDIA vs AMD 架构差异分析。

    核心 constraint: MI300X 的 L2 是 per-XCD 私有且不 coherent across XCDs → 任何跨 XCD 的数据共享必须通过 HBM 重新加载。这迫使 kernel 设计者将数据共享的 workgroups 物理放在同一 XCD。

    10. Key Source Code Walkthrough

    10a. Swizzled Head-first Mapping (Figure 11)

    
# Grid: total WGs = batch * num_q_heads * ceil(seqlen_q / BLOCK_M)
grid = lambda META: (batch * num_q_heads * (seqlen_q / META["BLOCK_M"]))

wid = tl.program_id(0)                         # [Line 5] 获取线性 WG ID
wid_per_batch = wid // BATCH                    # [Line 6] 去掉 batch 维度
heads_per_xcd = NUM_Q_HEADS // NUM_XCD          # [Line 7] 每 XCD 分配的 head 数 (e.g. 128/8=16)
blocks_per_head = ceil(SEQLEN_Q / BLOCK_M)      # [Line 8] 每 head 的 row block 数
chunk_size = NUM_XCD * blocks_per_head           # [Line 9] 一个 "chunk" = 8 × blocks_per_head

# [Line 12] head_offset: 通过 wid % NUM_XCD 确定目标 XCD，
#   再乘以 heads_per_xcd 映射到该 XCD 负责的 head 范围
head_offset = ((wid_per_batch % NUM_XCD) * heads_per_xcd
               + wid_per_batch // (NUM_XCD * blocks_per_head))

# [Line 13] block_offset: 在目标 head 内的 row block 编号
block_offset = (wid_per_batch % chunk_size) // NUM_XCD

# [Line 14] batch_offset: batch 维度 index
batch_offset = (wid // (blocks_per_head * NUM_Q_HEADS)) % BATCH

    核心机制: wid % NUM_XCD 决定 XCD 编号 → heads_per_xcd 个 head 被绑定到该 XCD → 同一 head 的所有 blocks 共享该 XCD 的 L2 cache。极简但有效。

    10c. Data Layout & Memory Access Patterns

    • Q/K/V: 标准 [B, H, N, D] layout (row-major)
    • 同一 head 内：FA2 的 tiled access → 每个 WG 处理一个 Q row block (BLOCK_M=128 rows)，需访问完整 K/V
    • Swizzled Head-first 保证同 head 的 WGs 在同一 XCD → K/V 的 tiled blocks 在 L2 中被逐块缓存和复用
    • 无额外 bank conflict 或 coalescing 问题（纯调度优化，不改数据布局）
