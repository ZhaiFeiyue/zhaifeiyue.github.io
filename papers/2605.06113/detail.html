BalanceRoute: Practical Online Routing for DP Load Balancing in LLM Serving #

§1 TL;DR #

BalanceRoute 利用分段线性 F-score 捕获 barrier 同步下 safe-margin fill 与 overflow 之间的 $(G{-}1)$ 倍不对称性，通过两阶段算法（贪心填充 + 精细分配）在毫秒级预算内实现 DP 负载均衡。无需预测的 BR-0 即可将不均衡降低 4.1×、吞吐提升 +11.8%；加上轻量二元终止分类器的 BR-H 达到 +15.4%，优势随 $G$ 超线性增长。

§2 Q1 / Q2 / Q3 #

Q1 痛点 #

DP decode 的 barrier 同步使每步延迟由最重 worker 决定；KV-cache 随生成单调增长且迁移代价极高（sticky assignment），加上到达非平稳、决策窗口 < 100 ms，现有通用启发式（Random / RR / P2C / JSQ）对 LLM decode 结构无感知。Chen et al. (2026) 报告生产部署中 >40% 加速器时间因 barrier 空闲而浪费。

Paper Figure 1, verbatim (caption: "Data-parallel decode under barrier synchronization").

图 1 直观对比：均衡分配时所有 worker 近乎同时到达 barrier（利用率 88–94%）；不均衡时 straggler 将步延迟拉高，轻载 worker 空闲等待（Worker 3/4 仅 32–42% 利用率）。每步空闲在 request 生命期内逐步累积。

问题形式化:

  • 目标函数: 最小化累积不均衡 $\sum_k I(k)$，其中 $I(k) = G \cdot M(k) - \sum_{g} L_g(k)$
  • 假设: sticky assignment（非抢占）、KV-cache 线性增长 $w_i^{(j)} = s_i + j - 1$、barrier 同步使步延迟 = $\max_g L_g(k)$
  • 算法单步输入: 等待池 $R_{\text{wait}}(k)$（含已知 prefill 长度 $s_i$），每个 worker 的活跃集 $\mathcal{A}_g(k)$ 与负载 $L_g(k)$
  • 算法单步输出: 对每个 worker 的 admission 子集 $Q_g \subseteq R_{\text{wait}}(k)$

Q2 方法 #

核心机制 — 分段线性 F-score: 度量将候选子集 $Q$ 分配给 worker $g$ 对单步不均衡 $I(k)$ 的边际效应：

$$F_g(Q) = \Delta_s(Q) - G \cdot (\Delta_s(Q) - m_g)_+$$

其中 $\Delta_s(Q) = \sum_{i \in Q} s_i$ 为候选总 prefill 负载，$m_g = M(k) - L_g(k)$ 为 safe margin。

区域条件F-score 行为物理意义
Safe regime$\Delta_s(Q) \leq m_g$$F_g(Q) = \Delta_s(Q)$每单位负载减少 1 单位 deficit
Overflow regime$\Delta_s(Q) > m_g$$F_g(Q) = G \cdot m_g - (G{-}1) \cdot \Delta_s(Q)$每单位溢出增加 $(G{-}1)$ 单位不均衡

交叉点处不对称性为 $(G{-}1)$ 倍：$G{=}16$ 时一单位溢出产生 15 倍不均衡代价。

Before (JSQ/P2C/RR)After (BR-0 / BR-H)

BR-H 扩展: 将 F-score 推广到 horizon 折扣形式：

$$F_g(Q) = \alpha \, (\mathbf{1}^\top \mathbf{d}) \, \Delta_s(Q) - \beta \, (\Delta_s(Q) \, \mathbf{1} - \mathbf{m}_g)_+^\top \mathbf{d}$$

其中 $\mathbf{d} = (1, \gamma, \ldots, \gamma^H)^\top$ 为折扣向量，$\mathbf{m}_g$ 为 horizon margin 向量。BR-0 是 $H{=}0, \alpha{=}1, \beta{=}G$ 的特例。BR-H 仅需二元终止分类器（request 是否在 $H$ 步内结束），而非完整长度回归——论文论证二元分类器比完整回归更可靠，因为噪声增长快于信号。

核心技术壁垒: 分段线性 F-score 将 barrier 同步的物理约束（max-load 决定步延迟）精确编码为一个具有尖锐交叉的目标函数，使 safe-margin fill 与 overflow 的不对称代价可在 $O(1)$ 内求值。这个洞见非平凡——它将看似需要整数规划的在线调度问题化简为可在毫秒级求解的贪心 + 子集选择，且理论上给出了最优结构（overflow 代价随 $G$ 线性放大）。现有通用负载均衡文献中没有类似的 barrier-aware 分段评分函数。

Q3 结果 #

  • BR-0（无预测）: 不均衡降低 4.1×（Proprietary）/ 1.94×（Azure），吞吐 +11.8% / +8.8% over JSQ
  • BR-H deployed (ExactMatch): 吞吐 +15.4% / +14.5% over JSQ，同时 TPOT P95 最低
  • 随 $G$ 增长优势超线性放大：BR-H 在 $G{=}4/8/16$ 分别 +13.4% / +21.5% / +34.5%
  • 部署在 144-NPU Ascend 910C 集群，验证硬件无关性

§3 架构 / 方法图 #

Paper Figure 2, verbatim (caption: "System architecture. A stateful proxy maintains the cluster snapshot and runs the BR-H rule.").

系统采用三层架构：控制面（Stateful Python Proxy）、Prefill Tier（kv_producer）、Decode Tier（kv_consumer）。

关键数据流：① Client request 经 Payload Mutator 改写（设 max_tokens=1 触发 prefill-only）→ ② Prefill 完成后返回 KV handle，request 进入 Prompt Pool → ③ BR-H Dispatcher 从 Cluster Snapshot $\sigma(k) = \{\mathcal{A}_g(k), L_g(k), \hat{c}_i(k)\}$ 读取状态，运行 horizon F-score 选择最优 (worker, subset) 分配，重写 payload 发送到 decode worker → ④ SSE telemetry 回传 per-token 进度，增量更新 snapshot。

三个设计原则：集中式状态（horizon F-score 需要跨 worker 视图）；Above-the-engine 集成（通过 payload mutation 和 SSE 解析，零引擎修改）；Pool 解耦（prefill 完成的 request 缓存在 prompt pool 中，等待 decode dispatcher 以全局视图调度）。

两阶段算法流程:

  1. Stage 1（充裕容量）: 当总空闲槽 $S_{\text{tot}} > S_{\text{greedy}}$ 时，选容量最大的 worker，将 F-score 最高的 request 贪心分配。此时必在 safe regime，$F_g(\{i\}) = s_i$，等价于将最大 request 发给最轻 worker。
  2. Stage 2（稀缺容量）: 剩余槽位按 $(\texttt{cap}_g, \min_h m_{g,h})$ 优先队列排序。对每个 worker 考虑 $R_{\max}$ 个候选 request 的最优子集 $Q^\star$，最大化 $F_g(Q)$。若最优子集非正分，仍分配单个 request 避免饥饿。

    方程物理意义 #

    Eq. (1) — BR-0 F-score: $F_g(Q) = \Delta_s(Q) - G \cdot (\Delta_s(Q) - m_g)_+$

    F-score 分解为 reward（deficit 填充收益）减去 penalty（overflow 代价乘以 $G$）。Safe regime 下 penalty 为零，每单位负载一单位收益。Overflow regime 下每单位溢出产生 $(G{-}1)$ 单位不均衡代价——因为新的 max-load 抬高了 barrier，所有 $G$ 个 worker 都必须等到这个新 max。

    Eq. (2) — BR-H horizon F-score: $F_g(Q) = \alpha \, (\mathbf{1}^\top \mathbf{d}) \, \Delta_s(Q) - \beta \, (\Delta_s(Q) \, \mathbf{1} - \mathbf{m}_g)_+^\top \mathbf{d}$

    将单步 F-score 在 $H$ 步 horizon 上求和，几何折扣 $\gamma$ 使近期预测权重更高。$\alpha, \beta$ 从理论值 $(1, G)$ 解耦为可调参数，补偿预测噪声。逻辑上：未来 $H$ 步中每一步的 overflow 都按该步的 margin 向量独立判断是否激活惩罚。

    6 项检验 #

    #检验项判定
    1F-score 在 $m_g$ 处的连续性✓ — 在 $\Delta_s = m_g$ 处左右极限一致 $= m_g$，但导数不连续（分段线性交叉点）
    2BR-0 → BR-H 特例归约✓ — $H{=}0$ 时 $\mathbf{d} = (1)$, $\mathbf{m}_g = (m_g)$，取 $\alpha{=}1, \beta{=}G$ 恢复 Eq. (1)
    3极端情况 $G{=}1$✓ — $G{=}1$ 时 $I(k) = 0$ 恒成立，F-score 退化为常数，路由无意义
    4极端情况 $m_g{=}0$（满载 worker）✓ — F-score $= \Delta_s - G \cdot \Delta_s = -(G{-}1)\Delta_s < 0$，正确表示任何 admission 都纯增不均衡
    5Stage 2 starvation guard✓ — 最优子集非正分时仍分配单 request，保证 waiting pool 不无限增长
    6目标一致性：$F_g$ 最大化 ↔ $I(k)$ 最小化✓ — $F_g$ 定义为 $-\Delta I(k)$（不均衡减少量），故最大化 F-score 等价于最小化单步不均衡增量

    算法特定证明分析 #

    • 无形式化收敛定理: 论文不含收敛定理或近似比证明。F-score 是一阶贪心的评分函数，论文未给出其与全局最优 $\min \sum_k I(k)$ 的近似比界。理想情况下应有 competitive ratio bound（如 $O(\log G)$ 竞争比），但论文选择了纯实证路线。
    • 方差/复杂度界: 论文未分析 BR-H 预测噪声对 F-score 质量的影响界。$\gamma$ 的选择隐式控制了预测误差放大，但无显式 bias-variance 分析。
    • 损失分解: $F_g$ 自然分解为 reward 项（deficit fill）和 penalty 项（overflow cost），物理意义清晰。Penalty 系数 $G$（BR-0）或 $\beta$（BR-H）控制两项权衡。

    §5 实验与数据 #

    实验设置 #

    • 硬件: 144-NPU Ascend 910C 集群
    • 模型: DeepSeek-V3 671B (W8A8)；验证也在 Qwen3-30B-A3B (MoE) / Atlas 910B
    • 拓扑: 默认 4P1D（4 prefill nodes × 16 NPUs TP=8，1 decode node × 32 NPUs DP=8 TP=4），$G{=}8$
    • 软件栈: vllm-ascend V1 scheduler, continuous batching, PagedAttention, MooncakeConnector
    • 数据集: Proprietary production trace (8k requests, mean prompt 3,197 / output 1,185) + Azure-2024 conversation split (10k requests, mean prompt 4,652 / output 1,052, filtered output > 1000)
    • 基线: Random, Round-Robin, P2C, JSQ (vLLM default)

    Paper Table 1, verbatim (caption: "Main results on Proprietary Data and Azure-2024. G=8, heavy load.").

    §7 实现 cross-reference #

    代码开放状态: [实现未公开] — 论文部署在 vllm-ascend（Ascend-NPU port of vLLM）之上，但 BalanceRoute proxy 代码未开源。

    最近开源参考: 基线实现基于 vLLM 的 router 策略（vllm.entrypoints.openai.api_server 的 Random / RR / P2C / JSQ 选项）。

    核心技术壁垒展开: F-score 的实现本身是轻量的（一个 piecewise-linear 函数求值），真正的工程壁垒在于维护毫秒级更新的 Cluster Snapshot $\sigma(k)$：

    1. 需要通过 SSE stream inline parsing 逐 token 追踪每个 request 的 decode 进度，增量更新 $L_g(k)$ 和 $\hat{c}_i(k)$
    2. Cross-tier KV transfer 的 late-binding 设计（Prompt Pool 缓存 + decode dispatch 解耦）需要精确的 PrefillerBudget 背压控制
    3. Stage 2 的子集选择（在 $R_{\max}$ 候选中搜索最优子集）需要在 < 1 ms 内完成，论文在 Appendix D 中描述了计算优化但未公开实现

      4. 关键实现细节:

      1. BR-0 pool-bypass path: 当预测基础设施不可用时，BR-0 可以跳过 Prompt Pool 缓冲，直接在 request 到达时路由——这个快速路径避免了池化引入的额外延迟，但牺牲了 Stage 2 的全局子集选择能力。
      2. $(\beta, \gamma)$ 调参: Table 1 显示两组超参 $(43, 0.86)$ 和 $(14.67, 0.64)$ 产生相近结果（不均衡 24.0k vs 23.6k），论文未报告系统化超参搜索方法。$\gamma{=}0.64$ 对应 $H{=}80$ 步后权重衰减至 $0.64^{80} \approx 10^{-16}$，实质 effective horizon 远小于名义 $H$。
