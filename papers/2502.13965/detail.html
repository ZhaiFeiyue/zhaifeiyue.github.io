Autellix: An Efficient Serving Engine for LLM Agents as General Programs — L2 #

§1 TL;DR #

LLM serving引擎对agentic program存在program-level head-of-line blocking。Autellix将OS的Least Attained Service调度提升到program级别（PLAS/ATLAS），按program累计服务时间排优先级，配合反饥饿和locality-aware load balancing，在相同延迟下实现4-15×吞吐提升。

§2 Q1 痛点 / Q2 方法 / Q3 结果 #

Q1 痛点 #

现有LLM serving系统（vLLM、SGLang）将每个LLM call视为独立请求调度，忽略同一agentic program内call之间的依赖关系。这导致两层head-of-line (HoL) blocking：

  1. Call-level HoL blocking：长decode的call阻塞短call——已知问题，MLFQ可缓解。
  2. Program-level HoL blocking：长program（含多个call）的新call在MLFQ中被放入最高优先级队列，持续阻塞短program——MLFQ在program级别反而可能比FCFS更差。

    3. 实测表明，中等负载下program大部分时间花在等待（Fig. 5），短program的等待/执行时间比显著高于长program（Fig. 6）。

    Paper's Figure 5, verbatim (caption: "Program execution and wait times, over different programs and system loads. With moderate loads, programs spend the most time waiting.").

    Fig. 5直观展示了在Chatbot、ReAct、MCTS三类workload下，wait time随负载增加而迅速主导program总延迟。这是Autellix设计的核心motivation：减少wait time既降低延迟又提升吞吐（完成更快→后续call更早到达→稳态并发更高）。

    Q2 方法 #

    Problem formulation. 给定 $N$ 个program以未知模式到达serving系统，每个program $i$ 是一个动态DAG，由LLM call和external interrupt（tool call/human input）交替组成。DAG结构仅在运行时逐步揭露。系统目标：

    $$\min \frac{1}{N}\sum_{i=1}^{N} W_i$$

    其中 $W_i$ 是program $i$ 所有LLM call的累计等待时间。系统同时追求GPU利用率最大化和公平性（p95/p99延迟可控）。

    假设：Non-clairvoyant——不知道program到达模式、DAG结构、call时长分布。Programs是dynamic（不同prompt产生不同执行模式）、non-deterministic（何时终止未知）、distributed（可并行fork/join）。

    Inputs/outputs of one scheduling step：

    • Input：新到达的LLM call $c_j$，附带 $c_j.\text{pid}$（program ID）；全局process table中该program的累计service time
    • Output：$c_j$ 在MLFQ中的队列分配（即优先级）

    Method core — 单一核心机制：将Least Attained Service (LAS) 从per-request提升到per-program。新call的初始优先级不是最高，而是继承program已消耗的累计服务时间。

    AspectBefore: LAS/MLFQ per-requestAfter: PLAS/ATLAS per-program
    Priority unit单个LLM callProgram（所有call共享state）
    新call初始队列最高优先级 $Q_1$按program累计service分配
    Service metric本call已用时间PLAS: $\sum t_k$; ATLAS: $\max s_i$
    跨call状态无Global process table

    PLAS（单线程program）：

    $$p(c_j) = \sum_{k < j,\; c_k.\text{id} = c_j.\text{id}} t_k$$

    物理意义：program消耗的总服务越少，优先级越高——偏好短/新program。

    ATLAS（多线程program）：

    $$p(c_j) = \max_{T_i \in \text{program}(c_j)} s_i$$

    物理意义：以critical path（最长线程）的累计服务作为program优先级；同时在program内部优先调度critical path上的call以缩短makespan。

    Anti-starvation：当program的等待/服务比超过阈值 $\beta$，call被提升到 $Q_1$：

    $$\frac{W_{\text{total}}}{S_{\text{total}}} \geq \beta \implies \text{promote to } Q_1$$

    核心技术壁垒：将LAS理论从单请求扩展到program级需要两个非trivial的insight：(1) 新call必须继承program累计service而非从零开始——这打破了传统MLFQ的基本假设；(2) 多线程program的priority必须用critical-path max而非sum——sum会过度惩罚高并行度program。这两个选择的组合使得ATLAS在MCTS等高并行workload上获得了15×的吞吐提升，而naive扩展（如对所有线程sum）无法达到同样效果。

    Q3 结果 #

    • Single-engine throughput：在相同median latency下，Autellix vs vLLM FCFS = 4-15×（Chatbot ~4×, ReAct ~4-5×, MapReduce ~6×, MCTS ~15×）。
    • Multi-engine throughput：locality-aware load balancer vs round-robin = up to 1.5×。
    • KV-cache swap优化：batched transfer vs vLLM default = 18× fewer swap operations, 3-7× swap time reduction, ~1.3× throughput。
    • Tail latency：anti-starvation机制使p95/p99可控，无starvation。
    • ATLAS vs PLAS on multi-threaded：ATLAS在MapReduce和MCTS上优于PLAS，验证critical-path优先级设计。

    §3 架构 / 方法图 #

    Paper's Figure 8, verbatim (caption: "Autellix's system architecture. Users run their programs locally, which initiates a stateful session and submits LLM calls to Autellix's backend. Autellix leverages a global process table to track sessions and better inform its custom load-balancer and scheduler.").

    Autellix的核心架构：用户端program通过stateful session API与Autellix后端通信。后端维护global process table，记录每个program的累计service time、线程元数据、等待时间等。Load balancer根据input length将call路由到合适的engine（长call → program's primary engine保持locality，短call → least-loaded engine均衡负载）。每个engine内的scheduler使用MLFQ + program-level priority进行调度。

    Paper's Figure 1, verbatim (caption: "Execution workflows for Agentic Programs. Agentic programs are highly dynamic execution workflows that follow a directed acyclic graph (DAG). It consists of LLM calls from one or more LLM agents and external interrupts (i.e. tool calls, humans).").

    Fig. 1展示了四类代表性agentic program的DAG结构：(a) Chatbot是最简单的线性链，(b) ReAct在LLM call和tool call之间交替，(c) Map-Reduce在某步fork出并行线程后join，(d) MCTS形成树状结构，包含大量并行expand/evaluate/backpropagate操作。这些DAG的dynamic和non-deterministic特性是non-clairvoyant scheduling的根本原因。

    Paper's Figure 9, verbatim (caption: "Critical path for multi-threaded programs. (Left) Example of a critical path through a DAG. (Right) Best-case scenario makespan, 14 units, is achieved by prioritizing calls on the critical path.").

    Fig. 9说明ATLAS的critical-path逻辑：多线程program的makespan取决于最长线程（critical path）。ATLAS通过 $\max s_i$ 识别critical path，并在program内部优先调度该path上的call，从而在不增加total compute的情况下缩短program完成时间。右图展示了最优调度（14 units）vs 非critical-path-aware调度的对比。

    方程物理意义 #

    PLAS priority:

    $$p(c_j) = \sum_{k < j,\; c_k.\text{id} = c_j.\text{id}} t_k$$

    累计已消耗的服务越多，优先级越低。这直接借鉴了OS中的Least Attained Service (LAS) 调度，但作用域从单个job扩展到了program（多个sequential calls）。在DHR (Decreasing Hazard Rate) 分布下（如Pareto/log-normal），LAS最小化mean response time。

    ATLAS priority:

    $$p(c_j) = \max_{T_i \in \text{program}(c_j)} s_i$$

    用program中最长线程的累计服务作为优先级。如果用sum代替max，高并行度program（如MCTS有数十线程）的priority会被过度膨胀，导致不公平。Max准确捕捉了program的"进度"——critical path决定completion time。

    Process table update:

    $$\text{pd.service} \leftarrow \max(\text{pd.service},\; c.\text{service} + c.\text{model\_time})$$

    对单线程program退化为简单累加；对多线程program取max以追踪critical path。

    Anti-starvation condition:

    $$\frac{W_{\text{total}}}{S_{\text{total}}} \geq \beta \implies \text{promote to } Q_1$$

    其中 $W_{\text{total}} = \text{pt}[c.\text{pid}].\text{wait} + c.\text{wait}$，$S_{\text{total}} = \text{pt}[c.\text{pid}].\text{service} + c.\text{model\_time}$。Promote后reset wait和service，防止反复promote。

    6 minimum checks #

    1. PLAS定义一致性：§4.2.1定义的 $p(c_j)$ 与Algorithm 1中queue assignment一致——新call被放入对应program累计service的queue，而非 $Q_1$。✓
    2. ATLAS max vs sum：§4.2.1明确说明用max（critical path），Algorithm 1 line 4中 pd.service = max(pd.service, c.service + c.model_time) 确认实现一致。✓
    3. Anti-starvation不破坏correctness：promote到 $Q_1$ 后reset service/wait（Algorithm 1 lines 29-30），避免一次promote后永久高优先级。✓
    4. MLFQ demotion与program priority正交：call在queue内按MLFQ quantum demotion处理call-level HoL，跨queue的初始assignment处理program-level HoL——两层机制互不干扰。✓
    5. Load balancer threshold无循环依赖：input token count是call自身属性，不依赖scheduler状态。✓
    6. Non-clairvoyant假设一致性：所有scheduling决策仅依赖已完成call的runtime（past），不使用任何future预测。✓

      7. 理论连接 #

      无独立新定理——Autellix不证明新的optimality result。但其理论基础来自已知结果：

      • Rai et al. (2003): LAS在DHR分布下最小化mean response time（non-clairvoyant optimal）。
      • Kim et al. (HPCA 2010, ATLAS): per-thread attained service用于DRAM调度——Autellix的ATLAS命名和max-across-threads思路直接受此启发。

      期望但缺失的理论保证：(1) program-level LAS（跨sequential calls）的formal optimality条件——当call数量也服从DHR时是否仍optimal？(2) ATLAS在DAG调度下的competitive ratio。Paper选择了纯实证路线。

      §5 实验与数据 #

      实验设置 #

      • Hardware：NVIDIA A100-80GB GPU
      • Models：LLaMA-3.1-8B（单卡），LLaMA-3.1-70B（多卡）
      • Baselines：vLLM FCFS, vLLM + MLFQ (FastServe-style), round-robin load balancer
      • Workloads：Chatbot（单线程，多轮对话）, ReAct（单线程，tool call交替）, Map-Reduce（多线程fork-join）, MCTS（多线程树搜索）
      • Metrics：program-level throughput（相同latency下完成program数/单位时间）, end-to-end latency (median/p95/p99), wait time breakdown

      HoL blocking实证 #

      Paper's Figure 6, verbatim (caption: "Ratio of Waiting to Execution Time for LLM Calls and Programs. Head-of-line blocking occurs when short LLM calls and programs wait significantly longer than their execution times.").

      Fig. 6是论文最重要的diagnostic figure之一。横轴按decode steps（call级）或LLM call数（program级）分组，纵轴是wait/exec ratio。关键观察：(a)(c) MLFQ在call级别改善了HoL（短call ratio下降），但 (b)(d) 在program级别MLFQ ≈ FCFS甚至更差——短program仍然被长program的高优先级新call阻塞。这直接证明了program-level scheduling的必要性。

      Data locality #

      Paper's Figure 7, verbatim (caption: "Prefix cache hit rates for LLM calls within and across programs. LLM calls within the same program often share KV cache, whereas LLM calls across programs typically do not.").

      Fig. 7验证了load balancer设计的前提假设：intra-program cache hit rate在各input length下均>90%（call共享conversation history），而inter-program hit rate随input length指数衰减。这解释了为什么长call应路由到program's primary engine（高cache hit），短call可分散（主要是system prompt，hit rate对engine选择不敏感）。

      主要吞吐结果 #

      核心结果：Autellix在相同median latency下的throughput improvement。

      WorkloadAutellix vs FCFSAutellix vs MLFQATLAS vs PLAS
      Chatbot~4×~4×N/A (single-threaded)
      ReAct~4-5×~4-5×N/A (single-threaded)
      Map-Reduce~6×~6×ATLAS > PLAS
      MCTS~15×~15×ATLAS > PLAS

      MCTS获得最大提升因为：(1) 单engine仅能处理~0.2 programs/sec，program-level blocking极其严重；(2) 每个program含数十到数百个call，长program的新call反复挤占短program。

      Ablation #

      Paper's Figure 19, verbatim (caption: "KV-cache swap optimization results showing Autellix's batched transfer vs vLLM's kernel.").

      Preemptive scheduling导致频繁KV-cache swap。Autellix通过batching parallel block transfers将swap operations减少18×、swap time减少3-7×。这是使preemptive scheduling实际可行的关键实现优化——没有此优化，preemption的overhead可能抵消scheduling收益。

      Anti-starvation threshold $\beta$ 敏感性：$\beta$ 过低→频繁promote→退化为FCFS；$\beta$ 过高→长program被无限期饿死。经验有效范围：wait/service ratio 3-7。

      MLFQ队列数 $K$：更多queue提供更细粒度preemption，$K=8\text{-}16$ 后收益递减。

      Steady-state行为 #

      Paper's Figure 4, verbatim (caption: "Number of LLM calls in serving engine during steady state over 1 hour. Optimizing programs' wait times increases the volume of LLM calls at steady state.").

      Fig. 4展示了一个有趣的正反馈：Autellix通过减少program wait time使program更快完成，更快的完成意味着后续call更早提交，稳态下GPU同时处理的call数量增加约10个。这说明program-level scheduling的收益不仅仅是延迟改善，还通过提高到达率间接提升了GPU利用率。

      §7 实现 cross-reference #

      Autellix作为vLLM上层实现，论文未开源完整代码。

      关键实现细节：

      1. 新call的MLFQ queue assignment（Algorithm 1的核心修改）：传统MLFQ将新请求放入 $Q_1$，Autellix根据process table中program的累计service time决定初始queue。这一行代码级的改动是program-level scheduling生效的关键——没有它，长program的新call仍会享受最高优先级。
        1. Batched KV-cache block transfer：vLLM默认逐block swap，preemptive scheduling下频繁swap成为bottleneck。Autellix将同一call的多个KV block打包为单次PCIe transfer，减少18× operations——这是使preemption实际可行的实现前提。

          2. 核心技术壁垒（详述）：

          co-design scheduler与program-level metadata tracking。Process table必须在call completion时同步更新（Algorithm 1 line 4），且update必须用max（for ATLAS）而非sum——错误的聚合函数会导致高并行program被过度惩罚。Anti-starvation的reset逻辑（promote后清零）也是subtle的：不reset则一次promote后service归零→再次被promote→oscillation。这些设计选择的组合空间大，paper的贡献在于找到了正确的组合。
