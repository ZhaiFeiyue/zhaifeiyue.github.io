Efficient LLM Serving for Agentic Workflows: A Data Systems Perspective (Helium)

Noppanat Wadlom, Junyi Shen, Yao Lu (National University of Singapore) | 2026-03 | https://arxiv.org/abs/2603.16104 Category: framework | Tags: agent-serving, prefix-caching, query-optimization, scheduling, batch-agentic-workflows Read: 2026-04-18

Core Contribution

Helium reframes batch agentic LLM workflows as query plans with LLMs as first-class operators, then applies database-style logical-plan rewriting, proactive KV/prompt caching, and a cost-based cache-aware scheduler over a Templated Radix Tree (TRT) to eliminate cross-call redundancy and maximise prefix reuse — achieving up to 1.56× speedup over KVFlow and 1.34× on a realistic trading workflow.

Summary

Agentic workflows — sequences of interdependent LLM calls, often running speculatively or in parallel — have become a dominant AI-serving workload. Engines like vLLM optimise each LLM call in isolation (continuous batching, PagedAttention, LRU prefix cache) and are workflow-agnostic and reactive: they can only reuse a KV prefix if two matching requests happen to arrive close together. Orchestration frameworks (LangGraph, AgentScope, Spark UDFs) sit above the engine but treat the LLM as a black-box UDF, so they are inference-agnostic and blind to KV cache, prefill/decode asymmetry, or prefix structure. Helium closes this abstraction gap by borrowing classic data-systems machinery: a workflow is parsed into a DAG, run through a query optimizer (operator pruning, common-subgraph elimination, prompt-cache substitution via CacheFetch), then compiled into a Templated Radix Tree that captures both static prefix structure and inter-call dependencies across the whole batch.

On top of the TRT, Helium runs two cooperating mechanisms. Proactive caching precomputes KV cache for static prompt prefixes on first execution and keeps them pinned; a separate global prompt cache stores outputs of deterministic operators so entire LLM calls can be replaced by a lightweight CacheFetch. Cache-aware scheduling formulates the problem as NP-hard multi-worker makespan minimisation with token-step cost model and precedence delays, then solves it with a DFS critical-path greedy that produces nested sequences: inner sequences group calls sharing the same static prefix (scheduled query-by-query to reuse per-query prefix), outer sequences alternate inner groups (operator-by-operator to reuse the static prefix).

On five primitive workflows (MapRed, Debate, Reflect, Iterative, Parallel) across MMLU/TAT-QA/Amazon datasets with Qwen3-8B, Helium beats vLLM by 38–101×, OpWise by up to 1.58×, LangGraph by 1.83×, Parrot by 2.21×, and KVFlow by 1.56×. On the complex 88-operator Trading workflow, it achieves 1.34× over KVFlow and up to 39.5× over vLLM. Ablations show plan pruning (−23.4%) and cache-aware scheduling (−17.7%) are the largest contributors; MILP comparison shows the scheduler is within 0.9% of the theoretical optimum on average versus 14–72% gaps for baselines.

Key Findings

  • Workflow-level visibility unlocks reuse that per-call engines cannot see. vLLM's passive LRU prefix cache cannot guarantee reuse when related queries are separated by unrelated requests; Helium's offline TRT analysis converts opportunistic reuse into deterministic reuse.
  • Nested scheduling structure is essential. A flat per-call schedule thrashes the cache when inner prefixes (per-query context) and outer prefixes (static system prompt) compete. Grouping calls by inner-sequence maximises both prefix tiers simultaneously.
  • High hit rate ≠ fast execution. On Qwen3-14B Trading, vLLM matches Helium's 42.9% hit rate but is 38× slower because it runs queries sequentially. End-to-end speed requires scheduling AND caching, not either alone.
  • Cost-based scheduling approaches optimality. Helium's heuristic achieves 0.9% average optimality gap vs MILP; LSPF (SGLang-style online prefix sorting) sits at ~14.5%, proving that global visibility matters more than local prefix awareness.
  • Metadata footprint scales with workflow structure, not batch size. TRT uses 552 KiB at 16 branches vs SGLang RadixCache's 14.8 MiB — a 27× advantage because TRT is keyed on the static template, not individual request tokens.
  • Plan pruning is the single largest contributor (23.4%). Common-subgraph elimination is more valuable than any caching trick because redundant operators break prefix identification downstream, compounding the loss.

Limitations

  • DAG abstraction is rigid: cannot natively express dynamic control flow (conditional loops, fan-out-depends-on-output, runtime mapping); the paper explicitly flags this for future work and gestures at TensorFlow-style control-flow primitives.
  • Closed world: simplifying assumptions restrict agents to local LLM calls + local data ops. No remote tool/API calls, no quality-aware approximation (proxy models), no model heterogeneity (all agents share the same base LLM).
  • Requires offline profiling: the cost model needs average output-length statistics per operator. New workflows incur a one-time profiling pass on a representative query set.
  • Determinism-bound caching: prompt cache only applies when operators are deterministic (temperature 0); any sampling-based operator cannot be bypassed.
  • Memory pressure sensitivity: on larger models (Qwen3-14B) the pinned proactive KV prefixes compete with active requests and the advantage shrinks (Helium's advantage is smaller on 14B than 8B).
  • Single-node scope: two H100s, single machine. Multi-node scheduling (network cost, cross-host KV transfer) is not addressed.

Infrastructure Impact

  • Algorithm: Repositions agent workflows as query plans amenable to classical DB optimization (CSE, plan rewriting, cost-based ordering). Opens the door to treating sampling/prompt selection as algebraic rewrites.
  • Kernel: No new kernels; relies on vLLM's PagedAttention. However, augments vLLM with a pin mechanism for KV blocks tied to static prefixes, which future kernel work could specialise.
  • Framework: Biggest impact layer. Introduces the TRT data structure, proactive vs passive cache taxonomy, and operator-level (vs request-level) scheduling granularity. Directly contests the design of Parrot, SGLang, KVFlow, and LangGraph.
  • LLM: Orthogonal to model architecture; any decoder-only LLM with standard KV cache works. Does require deterministic sampling for prompt-cache hits.
  • Agent: Provides a DSL that records the workflow as a symbolic DAG, enabling global optimization. This could become a compilation target for higher-level agent frameworks (LangGraph, AgentScope) if they expose structure rather than UDF black boxes.
  • Cluster: N/A at single-node scope; cross-node KV migration is out of scope.
  • Hardware: N/A — no hardware-specific optimizations.
  • Code: Open-source prototype at github.com/mlsys-io/helium_demo, built on vLLM v0.16.0 with IPC-based worker engines.

Deep Analysis (framework)

0. 时代定位、约束推导、核心壁垒（Phase 2 补深）

时代定位: 2024–2025 年单次 LLM 调用的优化已经被 vLLM / SGLang / FlashInfer / DistServe 等工作采摘殆尽（continuous batching、PagedAttention、chunked prefill、disaggregated P/D）。另一条战线——agentic workflow 编排（LangGraph/AgentScope/AutoGen）——则发展出了越来越复杂的 DAG，但完全把 LLM 当黑盒 UDF。2025 年出现的一批"workflow-aware serving"论文（Parrot, Autellix, KVFlow, Halo, Ayo）开始缝合这两层，但都停留在"反应式"（reactive）策略：只在请求到达时才匹配 KV cache。Helium 的位置是这条线的下一站——把数据库社区 30 年积累的 query-plan 重写与 cost-based 调度直接搬过来，把 agentic 优化从"启发式缝合"推进到"系统化 query optimization"。

约束推导（为何不可 X？）

  • 为何不可完全复用 SGLang 的在线 radix tree 调度？因为它以每个 LLM call 为调度单元，复杂度随 batch size 增长（无界），且不了解 workflow 级别依赖——会在跨 agent 的 system-prompt 切换点反复重算前缀。Helium 把调度单元降到 operator 级，复杂度 $O(|V_{int}| \cdot c_{max}^3 + |E'| \cdot d_{max})$ 只依赖 workflow 结构。
  • 为何不可复用 Parrot 的 Semantic Variables？因为 Semantic Variables 只在单次 LLM call 内部建模 prompt 结构，无法跨多 agent 的共享前缀层级建模；它形成的是"依赖 DAG"而不是"全局前缀层级"，调度被迫退化为 reactive。
  • 为何不可在 Spark/Dask 的 UDF 内优化？UDF 是黑盒，framework 看不到 KV cache、prefill/decode 不对称成本；continuous batching 无法跨 UDF 边界工作，导致每个 UDF 结束点都形成"流水线塞车"。
  • 为何不可完全静态编译？因为 LLM call 的输出长度动态、runtime batching 行为复杂，offline cost model 不可能完全准确——所以 Helium 把输出设为 "soft schedule"（嵌套 operator 序列），允许 worker 运行期 reorder。

核心技术壁垒: Templated Radix Tree (TRT) + nested-sequence schedule 的双层前缀复用机制。TRT 把 static prompt 前缀和 dynamic placeholder 编码在同一棵树里，且叶子之间显式连有依赖边——这让 scheduler 同时看到"谁和谁共享静态 system prompt"与"谁依赖谁的输出"。更精妙的是 nested sequences：内层序列 query-by-query 走相同 operator（复用 per-query context prefix），外层 operator-by-operator 换 operator（复用 static system prompt）。任何只看一层前缀的 scheduler（LSPF、Parrot、RadixCache）在两层都共享的场景下必然会 thrash 其中一层。表 4 里 LSPF 的 hit rate 仅比 Random 高 1.5% 就是铁证——它优化了单个 prefix 但牺牲了另一个。

质疑假设:

  • "offline profile 的输出长度统计足够准" —— 对于 reasoning 模型（思维链长度方差巨大）可能失效。论文只在 Qwen3 / Llama3.1 上验证，对 reasoning 长输出鲁棒性未知。
  • "greedy sampling 够用" —— 真实 agentic 系统大量使用 temperature>0 做探索（ToT/GoT），此时 prompt cache 完全失效，只能依赖 KV 前缀缓存。论文的 prompt cache 贡献（13.56%）在 sampling 场景下会消失。
  • "workload 结构稳定且可 profile" —— 真实 agent trajectory 常常 early-stop 或条件分叉（ReAct 里 "action = ..." 的分支），DAG 抽象强制将这些展开为静态结构。论文在 §9 里诚实地承认了这点。

设计绑定批判: Helium 绑定了（1）batch agentic（query 必须成 batch 出现）、（2）deterministic operators（非 greedy 即失去 prompt cache）、（3）DAG-expressible workflows（无 dynamic fan-out）、（4）单一 base LLM（不支持 model-heterogeneous 的 router / mixture agents）。这意味着 Helium 天生不适合 online-serving（单查询流）、exploration-heavy RL trajectories、或 Mixture-of-Agents with heterogeneous backends。

生态影响追踪: Helium 是 2026 年初的新作品，尚未见下游采纳。但它对 SGLang / Parrot / KVFlow 是直接挑战：如果 SGLang v0.5+ 引入 workflow-level 前缀 hint API，或 vLLM 加入 "pinned KV prefix" 机制，Helium 的思路将被整合。其理论贡献（MILP-validated cost model + nested-sequence schedule）很可能被后续 workflow-aware 工作（尤其同实验室的 FlowMesh 2510.26913）吸收为 baseline 或子模块。

1. System Scope

  • Primary goal: Batch agentic workflow serving — optimizing wall-clock latency for a batch of structurally-identical queries through a multi-LLM DAG.
  • Scale: Single-node, multi-GPU (evaluated on 2×H100 NVL 94GB). Does not address multi-node or disaggregated P/D.
  • Online/offline: Offline / batch-oriented. Not designed for latency-sensitive online serving (scheduler is offline-computed, proactive cache assumes repeated structure across batches).
  • Workload: Agentic workflows over a single base LLM. Primitive patterns (MapRed, Debate, Reflect, Iterative, Parallel) and composite (Trading: 19 agents, 88 operators).

2. Architecture & Data Flow

Request flow:

  1. User writes DSL (Python lazy DAG: ops.placeholder, ops.llm, ops.fmt, ...)
  2. graphs.build(...).compile(input_batch) binds placeholders, materialises a batch DAG
  3. helium.invoke(graph):
  4. Query optimizer (CPU): prune dead ops → CSE → bottom-up traversal probes prompt cache, replaces hits with CacheFetch operators
  5. TRT construction (CPU): topological sort; for each LLM op generate prefix template, insert into radix tree; add dependency edges between leaves
  6. Cache-aware scheduler (CPU): Algorithm 1 — partition ops across workers, DFS with critical-path heuristic, produce nested sequence soft schedule
  7. Dispatch: push schedules to per-worker queues; workers issue to vLLM engine via IPC, best-effort release when dependencies + precedence delays satisfied
  8. vLLM (GPU) prefills pinned static prefixes once, reuses across batches; runs continuous batching on dynamic portions

    9. Control vs data plane: Clean separation. Control plane = optimizer + scheduler (CPU, milliseconds). Data plane = vLLM engines (GPU). Communication via IPC message queues.

    Stateful: vLLM KV cache (including pinned proactive prefixes), global prompt cache (deterministic operator outputs).

    Stateless: Each invocation rebuilds TRT & schedule from the compiled DAG + current cache state.

    Failure handling: Not discussed. Paper silent on engine crashes, block-eviction edge cases under memory pressure (relies on vLLM's native LRU as fallback).

    2b. Data Movement Hotspots

    1. KV cache residency in HBM — static prompts (e.g. 200+ token system prompts across 8 trader agents) pre-computed and pinned once, then reused across all batches. Saves O(num_batches × num_prefix_tokens × num_layers × head_dim × 2) compute, but competes with active-request cache blocks (LRU eviction pressure under memory constraint).
    2. IPC between scheduler and engine workers — one message per scheduling decision. Measured at 89 μs per request routing — negligible.
    3. HTTP OpenAI-API round-trip — baselines pay ~3.1 ms per call; Helium avoids this by direct IPC. For 1,408 LLM calls on Trading, that's ~4.4 s of pure routing overhead baselines eat.

      5. Scheduling & Resource Management

      • Batch formation: offline soft schedule (nested operator sequences per worker); runtime dispatcher executes best-effort, buffering ready ops to saturate GPU while preserving cache-friendly order.
      • Memory: pinned KV for static prefixes + vLLM native paged attention for dynamic portions + LRU eviction as safety valve. Prompt cache LRU-evicted at operator granularity.
      • GPU utilization: nested-sequence schedule interleaves dependent and independent ops so that workers rarely stall (1.18× larger avg inference batch than LangGraph, Figure 12).
      • Multi-tenancy: not addressed (single tenant batch).
      • SLO-aware: not a design goal; makespan minimisation only.

      7c. Bottleneck shift

      
Before: scheduling-bound (passive cache thrash, serial execution)
 → after plan pruning: compute-bound for redundant ops dissolved
 → after cache substitution: compute shifted to remaining dynamic LLM calls
 → after cache-aware scheduling: cache-reuse maximised, GPU utilisation ↑
 → remaining bottleneck on Qwen3-14B: memory pressure (pinned prefix vs active blocks)

      At 14B the advantage narrows — the remaining bottleneck is HBM capacity, suggesting future work in hierarchical (HBM↔DRAM↔NVMe) KV offload for agentic workflows.

      7d. Baselines & fairness

      • Same hardware (2×H100 NVL), same vLLM v0.16.0 backend, same model (Qwen3-8B/14B), greedy sampling for all.
      • Baselines interact via OpenAI API + lightweight load balancer; profiled overhead is 89 μs + 3.1 ms RTT — small compared to multi-second inference.
      • Parrot uses its own runtime; KVFlow uses SGLang backend per its paper — Helium doesn't force-replace their engines, which strengthens the fairness claim.
      • Baselines "win" only on prefix cache hit-rate with Qwen3-14B vLLM (+0.3%), but vLLM is 38× slower end-to-end due to sequential execution — decisively losing on the primary metric.

      8. API & Usability

      • API: Custom Python DSL (helium.graphs.build(...).compile(...)). Not OpenAI-compatible — users write workflows in Helium's lazy dataflow style rather than LangGraph. Migration cost is non-trivial.
      • Model format: Inherits vLLM support (HuggingFace, SafeTensors).
      • Deployment: On-prem, multi-GPU, vLLM engine per worker in separate process, IPC communication.
      • Knobs: worker count, KV cache size, profiling sample size. Few user-facing knobs for the scheduler itself.

      11. Adoption, Maturity & Ecosystem

      • Open source: Yes, github.com/mlsys-io/helium_demo. Research prototype; demo-grade.
      • Community: Early-stage (2026-03 publication). No independent deployments reported.
      • Production readiness: Requires users to rewrite workflows in Helium DSL; missing dynamic-control-flow & external-API support limits direct drop-in for LangGraph-based apps.
      • Adoption path: Most promising as a backend for LangGraph/AgentScope if those frameworks expose workflow structure via a compilation API. Alternatively, the ideas (TRT, CacheFetch, nested scheduling) can be upstreamed into SGLang v0.5+ or a vLLM "workflow-aware" extension.
      • Downstream influence: Too early to trace. Within the same NUS group, FlowMesh (2510.26913) and Halo (2509.02121) share authorship and likely reuse core ideas.
