🗺 研究背景和现状

A · 时代定位

2024-2026 年，LLM 从单轮问答演进到 agentic 架构（ReAct/AutoGen/LangChain），推理系统优化仍聚焦单次 LLM 调用。Sutradhara 代表从"优化单次推理"到"优化多步推理流水线"的范式转变。

B · 领域定位

方向：LLM 推理系统 × agentic workload
现实问题：企业 AI 助手（M365 Copilot）中用户等待数秒才看到第一个 token

C · 核心痛点

现有方法可以优化单次 LLM 调用延迟，但无法优化跨迭代的 agentic 端到端延迟：

  • ①延迟层：tool 执行占 FTR 的 30-80%（尾部），CV > 100%
  • ②并行层：50-80% prefill 不依赖 tool 输出却被迫等待
  • ③内存层：LRU 驱逐导致级联 thrashing

D · 设计空间约束

替代方案为何不可行
静态 DAG (Parrot/Murakkab)Agent 执行图运行时生成，不可预知
预测 tool 延迟 (Continuum)CV > 100%，预测精度不足
Partial execution (Conveyor)生产 tool 需完整参数
纯 orchestrator/engine 优化信息不对称，单侧只能获部分收益

结论：可行空间 = orchestrator-engine co-design + thin API

E · 提出框架

Sutradhara：通过 5 个 API（submit_partial_prefill / extend_prefill / register_streaming_callback / tag_kv_blocks / set_reuse_priority）实现三大优化——prefill-tool 并行、decode-tool 并行、语义 KV cache 管理。本质区别：让 orchestrator 的语义知识指导 engine 的调度决策。

F · 技术挑战

  • partial prefill KV cache 保活 → set_reuse_priority() pin ✅
  • tool output 正确拼接 → extend_prefill() continuation handle ✅
  • streaming JSON 容错 → 流式 parser ⚠️部分缓解
  • 最优驱逐策略 → 手动优先级链 ⚠️未自动学习

语义 KV 驱逐优先级

$$\text{RESPONSE} \longrightarrow \text{TOOL\_OUTPUT} \longrightarrow \text{USER\_QUERY} \longrightarrow \text{SYSTEM\_PROMPT} \longrightarrow \text{PARTIAL\_PREFILL}$$
从先驱逐（左）到后驱逐（右）：RESPONSE 无复用价值最先驱逐，PARTIAL_PREFILL 最后驱逐

"Why not static DAG?"

Tool-based agent 的 DAG 运行时生成：$\text{DAG}_{i+1} = f(\text{decode}_i)$。Parrot/Murakkab 的静态 DAG 假设对动态 agent 不成立。

"Why co-design?"

优化信息分散：orchestrator 知道 prompt 结构，engine 知道 cache 状态。5-API thin interface 是最小侵入的信息共享方案。

核心技术壁垒

Partial Prefill KV 生命周期管理——在 continuous batching 中维护"半完成"请求，pin KV blocks 数百毫秒，处理 extend 拼接和失败 cleanup。

假设挑战

50-80% tool-independent 假设依赖固定 prompt template。若 orchestrator 动态重组 system prompt，partial prefill 的 KV cache 可能完全无用。

🔬 第一性原理

约束推导① 为什么不能预测 tool 延迟？

Figure 4(b) 实证：top 7 tools 延迟 CV > 100%，标准差大于均值。Tool 延迟取决于外部因素（查询复杂度、服务负载），与 LLM 计算完全解耦。95% 置信区间宽到无法指导实时调度。

"当变量的不确定性大于其均值时，预测不如适应——与其猜 tool 何时返回，不如让系统在返回时即时响应。"

约束推导② Engine 能自动识别 prompt 结构吗？

Engine 只看到 token ID 序列，不知道哪些是 system prompt、tool output、user query。自动识别需要内嵌 prompt parser，破坏模块化。Co-design API 是最小信息传递方案。

"信息不对称是系统低效的根源——最小化接口传递关键语义比全面融合更优雅。"

假设挑战：tool-independent 比例是否稳定？

生产环境中：orchestrator 可能动态切换 system prompt；multi-agent 场景中 split point 不明确；RAG 结果可能插入 prompt 中间。若比例降到 30%，prompt splitting 收益大幅缩水。

"Prompt splitting 的收益是 prompt template 设计的函数——template 越固定收益越大。"

为什么用语义标签而非学习型驱逐？

Orchestrator 知道哪些请求正在等 tool 返回，能预测 KV cache 即将被复用。语义标签将"有限的未来知识"编码为优先级——基于确定性 workflow 结构，比统计推断更可靠。

"当你有部分确定性的未来信息时，规则策略优于学习策略——确定性不需要泛化。"

JSON 的自闭合特性使 streaming dispatch 成立

JSON object 在 } 闭合时就完整可解析，不需要等待整个 array 的 ]。每个 tool call 可被独立提取和分发。若输出非 JSON 格式则需重写 parser。

"JSON 的自闭合属性是 streaming dispatch 的数学基础——每个 {} pair 是独立语义单元。"

Intra-request vs Inter-request 优化权衡

Request-aware scheduling 优先处理旧请求，可能牺牲新请求的公平性。在高并发下这是零和博弈——论文未分析不同 QPS 下的公平性影响。

"任何优先级调度都是零和博弈——问题不是能否加速某些请求，而是被减速的请求是否可接受。"

📐 核心指标解读

FTR — First Token Rendered ↓

生活类比：去餐厅点大菜，FTR = 从下单到第一口菜上桌的时间。厨房需多次去仓库取食材（tool call），每次取完继续做（LLM 迭代）。

$$\text{FTR} = \sum_{i=1}^{N-1} (t_{\text{prefill}_i} + t_{\text{decode}_i} + t_{\text{tools}_i}) + t_{\text{prefill}_N} + t_{\text{first\_token}_N}$$
$N$ = 总迭代次数，前 $N-1$ 次产生 tool calls，第 $N$ 次产生用户回答

🚀 科研改进方向

Sutradhara 开辟了 co-design 新方向，但评估规模和适用范围留下大量探索空间。

1. Disaggregated P-D Serving 下的 Co-design ⭐⭐⭐

Partial prefill 假设 P/D 同 GPU。DistServe/Splitwise 分离 P/D 后，KV cache 需跨 GPU 传输。技术路径：预测 extend 时机提前迁移 KV / RDMA 异步传输 / 重评 break-even point。

2. 自动化 Split Point 发现 ⭐⭐

当前需手动标注。方案：静态分析 prompt template AST / runtime profiling / NLP 依赖分析。

3. 多 GPU Agentic-Aware Scheduling ⭐⭐⭐

单 GPU → 多 GPU TP/PP：KV priority 需跨 GPU 协调。方案：层级化 cache 管理 + synchronized partial prefill。

4. 学习型 KV Cache 驱逐 ⭐⭐

手动优先级链 → 在线学习最优策略。方案：RL agent / multi-armed bandit / workload-aware strategy selection。

改进方向选择矩阵

方向难度工程量适合
Disaggregated P-D⭐⭐⭐高系统方向博士生
自动 Split Point⭐⭐中NLP+Systems 交叉
多 GPU Scheduling⭐⭐⭐高分布式系统
学习型驱逐⭐⭐中ML+Systems

资深教授判断

"未来 5 年 LLM serving 核心挑战将从'优化单次推理'转向'优化多步推理流水线'。Sutradhara 的 co-design 思想是正确起点，但真正突破需解决 disaggregated serving + dynamic prompt + 多 GPU 协调的交叉。"

📝 答题卡

1FTR vs TTFT 的区别是什么？

查看答案

FTR = 从请求到 final response 首 token 的累积时间（含多次 LLM + tool）。TTFT 仅衡量单次 LLM 调用。§3.3 → §3.5 → §5.2

2Prompt splitting 依赖什么前提？

查看答案

50-80% prompt 是 tool-independent；orchestrator 知道 split point；engine 支持 partial prefill + extend。若 prompt 动态重组则收益缩水。§3.5 Finding 2 → §4.2 → Table 2 (7.57%)

3LRU 为什么在 agentic workload 下 thrashing？

查看答案

LRU 按访问时间排序，不知道哪些 blocks 即将被 tool 等待中的请求复用。Iteration 1 的 blocks 在 tool 等待期间变为"最旧"被驱逐→重算→级联。§3.5 Finding 3 (Fig 6) → §4.4 (Fig 8)

45 个 API 各解决什么？

查看答案

partial_prefill→prefill-tool overlap / extend→拼接 / callback→decode-tool overlap / tag→语义驱逐 / priority→防 thrashing。最小侵入设计。§4.1 Table 1

5vs Parrot/Conveyor 的核心区别？

查看答案

Parrot=静态DAG（agent不适用），Conveyor=partial execution（生产tool不支持）。Sutradhara 不假设 DAG 可知，不要求 partial execution，而是利用 prompt structure 和 JSON 自闭合。§6

6E2E 改进为什么小于 FTR？

查看答案

E2E = FTR + final decode。Final decode 是纯自回归，无法被 PS/DS 加速，且 final decode token 数是 intermediate 的 ~5x。

7tool-independent 比例降到 30% 还值得做 PS 吗？

查看答案

收益从 ~7.57% 降到 ~3%。此时 streaming dispatch (4.2%) 成为主要贡献者。需 cost-benefit 分析：3% 收益 vs 实现复杂度。

8如何迁移到 disaggregated P-D serving？

查看答案

Partial prefill 在 prefill GPU，decode 在 decode GPU → KV cache 需跨 GPU 传输。方案：(1) decode GPU 上做 partial prefill (2) RDMA 异步传输 (3) 评估 break-even point。开放研究问题。
