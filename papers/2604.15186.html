Scepsy vs 相关论文：跨篇综合 #

相关论文 #

本篇选取 8 篇 framework 类别的相关实体，覆盖 PD 分离 I/O 优化、多轮对话路由、网络流调度、RL 权重传输、跨 DC 推理、MoE prefill 执行、kernel 级延迟优化和 KV-cache 压缩，与 Scepsy 在"agentic workflow 的多 LLM GPU 编排"这一核心问题上形成多维对比。

DualPath (2602.21548) — 共享"agentic workload 改变推理基础设施假设"的 motivating context。DualPath 发现 agentic 工作负载使 prefill 从 compute-intensive 退化为 I/O-intensive（KV-Cache hit 率 98.7%），因此用双路径 KV-Cache 加载 + CNIC 流量隔离消除 storage I/O 瓶颈 [2602.21548]。Scepsy 同样以 agentic workflow 为靶点，但攻击的是上层问题——多 LLM 之间的 GPU 分配和拓扑放置 [2604.15186]。两者形成 vertical stack：DualPath 优化单 LLM 实例的 I/O 路径，Scepsy 优化多 LLM 实例的 GPU 编排。DualPath 部署于 DeepSeek（1152 GPU），Scepsy 评估于 16 GPU——规模差距一个数量级。

PPD (2603.13358) — 共享"steady-state 统计替代精确建模"的方法论。PPD 观察到 append-prefill 的 decode 干扰仅 2%（vs full prefill 的 48%），据此设计 per-request 动态路由（$S > 0$ 则 D 节点本地执行），将不稳定配置变为 100% 可用 [2603.13358]。Scepsy 观察到 LLM 调用的相对份额比绝对值稳定 4×，据此设计 Aggregate LLM Pipeline 做 GPU 分配 [2604.15186]。两者都把 dynamic system 用 statistical shortcut 近似：PPD 用 offline profiled lookup table，Scepsy 用 $(n_m, p_m)$ 聚合统计量。但 PPD 是 single-LLM per-request 路由，Scepsy 是 multi-LLM cluster-level 分配——粒度和对象不同。

MFS (2603.17456) — 共享 disaggregated serving 的资源争用问题，但攻击不同层。MFS 解决 MoE serving 中 KV-cache 复用、collective comm、P2D 传输三阶段的网络流争用，用 Reverse MLFQ 实现 Defer-and-Promote 调度 [2603.17456]。Scepsy 解决多 LLM workflow 中 GPU fraction × TP × replica 的联合分配问题 [2604.15186]。MFS 在 network layer 做 per-flow 调度（ms 级决策），Scepsy 在 orchestration layer 做 per-deployment 分配（分钟级决策）——时间尺度差两个数量级。MFS 的 RMLQ 可以成为 Scepsy 下层 engine 的 transport 优化组件。

TensorHub (2604.09107) — 共享"利用空闲 NIC 带宽的 DAG 传输"设计模式，但面向完全不同的场景。TensorHub 用 Reference-Oriented Storage + pipeline replication 优化 RL 训练的 trainer→rollout 权重传输 [2604.09107]。Scepsy 的 beam search workload（generator + verifier 循环）正是 inference-time scaling 的典型实例——TensorHub 训练出的模型用 Scepsy 部署到生产 [2604.15186]。但两者在技术栈上几乎无交集：TensorHub 优化 RDMA 级别数据搬运，Scepsy 优化 Kubernetes 级别 pod 编排。

PrfaaS (2604.15039) — 共享"异构资源 + 选择性 offload"的调度范式。PrfaaS 把长 prefill 选择性 offload 到跨 DC 的 compute-dense 集群，用长度阈值 $t$ 路由 + 双时间尺度调度实现 +54% 吞吐 [2604.15039]。Scepsy 用 latency ratio 驱动的分数 GPU 分配实现 +2.4× 吞吐 [2604.15186]。两者都用 steady-state throughput model 求解最优配置（PrfaaS 的 Eq.7 ∩ Eq.8 vs Scepsy 的三维剪枝枚举），但 PrfaaS 面向 single-LLM 跨 DC，Scepsy 面向 multi-LLM 单集群。PrfaaS 的形式化程度更高——有完整的 closed-form throughput model 和 case study 数字一阶映射。

ZeRO-Prefill (2605.02960) — 共享"反转数据流方向以消除通信瓶颈"的设计哲学。ZeRO-Prefill 反转 EP 的 activation-to-expert 路由为 expert-weight-to-GPU 的 AsyncEP，用后台 AllGather 替代同步 AllToAll [2605.02960]。Scepsy 反转"用户手动分配 GPU"为"系统自动搜索最优分配"，用 Aggregate Pipeline 替代 per-LLM 独立扩缩 [2604.15186]。ZeRO-Prefill 的 frontend-backend co-design（饱和阈值 T 由 backend 定义、frontend 强制）与 Scepsy 的 profiler-scheduler co-design（per-LLM profile 驱动搜索器）有结构相似性。ZeRO-Prefill 仅适用于 prefill-only 判别式任务，Scepsy 面向完整的 agentic workflow（含 decode）。

TileRT (tilert-speed-scaling-law) — 攻击栈的另一端。TileRT 把整个模型 AOT 编译为单个 Persistent Engine Kernel，消除 kernel 间的 launch gap 和 synchronization overhead [tilert-speed-scaling-law]。Scepsy 在 Kubernetes 层面编排多个这样的 engine 实例。TileRT 优化 BS≈1 decode latency（面向 Agent/TTS/voice），Scepsy 优化多 LLM workflow 的 throughput + latency。两者可垂直叠加——TileRT 加速单 engine 实例，Scepsy 决定该实例获得多少 GPU fraction。TileRT 缺乏定量基线对比，无法判断与 Scepsy 使用的 vLLM/SGLang engine 的性能差距。

KVServe (kvserve) — 共享 PD 分离栈，但正交互补。KVServe 压缩 PD 间的 KV-cache 传输（up to 10× 压缩比），用 service-aware controller 动态选择压缩策略 [kvserve]。Scepsy 不触及 KV-cache 传输，而是管理多 LLM 的 GPU 分配 [2604.15186]。KVServe 以 vLLM external connector 零侵入接入；Scepsy 扩展 NVIDIA device plugin + MPS，侵入 Kubernetes 层。两者可叠加：Scepsy 决定 LLM 副本数和 GPU fraction，KVServe 优化副本间的 KV 传输效率。但 KVServe 不支持 MLA 架构，限制了在 DeepSeek 类模型上的适用性。

本篇 vs 相关论文的 delta #

Scepsy 的核心 delta 是 Aggregate LLM Pipeline 抽象 ——将不可预测的 agentic workflow 用两个静态统计量 $(n_m, p_m)$ 压缩为 pipeline 模型，配合 $(d_m, TP_m, \text{frac}_m)$ 三维联合搜索和 NVLink-aware hierarchical placement，在 multi-LLM × workflow-aware × auto-allocation × framework-agnostic 四维上唯一勾满 [2604.15186]。

Scepsy 的独有增量：

  1. 唯一的 multi-LLM workflow-level GPU 编排——所有其他 8 篇都在 single-LLM 或 single-stage 维度优化。DualPath/PPD/MFS/PrfaaS/KVServe 处理 PD 分离的 data movement；TensorHub 处理训练权重传输；ZeRO-Prefill 处理 MoE 并行执行；TileRT 处理 kernel 延迟。没有任何一篇同时解决"多个异构 LLM 如何分配 GPU × TP × fraction"的问题 [2604.15186]。
    1. "相对份额稳定性"是 Scepsy 方法论的 empirical cornerstone——PPD 的 offline profiled lookup table 和 PrfaaS 的 throughput model 都依赖 steady-state 近似，但 Scepsy 是唯一在论文中用 Figure 3 的 500-request 实验显式证明"相对份额比绝对值稳定 4×"的 [2604.15186]。PPD 和 PrfaaS 假设 workload 稳态但未量化稳定度。
      1. Fractional GPU + topology-aware placement 在 Kubernetes 上的端到端落地——DualPath/TensorHub 依赖 RDMA 硬件栈（InfiniBand VL QoS、CNIC loopback），ZeRO-Prefill 依赖 NVLink AllGather，PrfaaS 依赖跨 DC Ethernet。Scepsy 选择了最"commodity"的路径：扩展 NVIDIA device plugin + MPS，在标准 Kubernetes 上落地 [2604.15186]。这降低了部署门槛但也限制了性能天花板（MPS 的 SM/memory 静态限制是 coarse grained）。

        2. Scepsy 的相对劣势：

        1. 形式化程度最低——PrfaaS 有完整的 Eq.1-8 throughput model + case study 一阶映射 [2604.15039]；DualPath 有 Eq.1-9 closed-form BW 约束分析 [2602.21548]；MFS 有 MLU 公式 + 6 项检查 [2603.17456]；ZeRO-Prefill 有饱和阈值 T 的物理推导 [2605.02960]。Scepsy 的 pipeline 公式 $L_w, T_w$ 是简单算术，搜索是暴力枚举+剪枝——没有 optimality guarantee 或 bottleneck-free 区间分析。
          1. 评估规模最小（与 production 系统对比）——DualPath 1152 GPU 生产验证，TensorHub 1024 GPU 生产验证，PrfaaS 模拟 96 GPU（但基于 Moonshot AI 生产模型参数）。Scepsy 的 16×A6000 配置中 NVLink 仅为 2-pair bridge，完全不同于 H100/H200 的 NVSwitch 拓扑——TP domain 上限为 2，搜索空间比 H100 节点简单得多 [2604.15186]。
            1. Baseline 公平性存疑——Ayo 使用旧版 vLLM v0.2；Aegaeon 在 beam search 场景天然吃亏（无 prefix cache）[2604.15186]。相比之下 DualPath 的 Basic→Ours 是 same-implementation-base 增量对比 [2602.21548]，PPD 所有配置使用相同 vLLM codebase [2603.13358]。

              2. 可攻击面 #

              Attack 1: "相对份额稳定"假设在 tool-heavy agent 中失效 (L2 §Key Findings)

              Scepsy 的整个 Aggregate LLM Pipeline 建立在"每个 LLM 的相对延迟占比稳定"的 empirical 观察上——Figure 3 证明 beam search 下 GEN/VER 的相对份额比绝对值稳定 4× [2604.15186]。但这个结论来自两个高度 LLM-dominated 的 workload（beam search 和 RAG+reranker），non-LLM tool 时间 "[2604.15186]。

              在真实 agentic workflow 中，tool 调用时间可能与 LLM 推理时间量级相当甚至更长：web search（秒级）、代码执行（秒到分钟级）、数据库查询（百毫秒到秒级）。当 tool 时间不可忽略时，LLM 的"相对份额"不再由 LLM 本身决定，而是被 tool 延迟所调制——如果某个 tool 耗时剧烈波动（例如代码执行可能成功 100ms 或失败后 retry 10s），LLM 相对份额会跟随 tool 延迟而波动，Figure 3 的 "4× more stable" 结论直接失效。

              DualPath 的 production trace 提供了间接反证：157 轮 coding agent 中 tool call latency 导致 working set 以 $r^2$ 扩展 [2602.21548]——说明 tool 延迟在真实 agentic 场景中是 dominant factor。Scepsy 论文自己也承认 bimodal load distribution 是未解决问题 [2604.15186]，但未评估任何 tool-heavy workload，这是一个显著的评估盲区。

              Attack 2: A6000 NVLink 2-pair 拓扑使搜索空间过于简单 (L2 §Deep Analysis)

              Scepsy 在 16×A6000（4 节点 × 4 GPU，NVLink pair + PCIe4）上评估 [2604.15186]。A6000 的 NVLink 仅为 2-way bridge，意味着 TP 上限为 2——搜索空间中 TP 维度几乎退化为二值选择。在 H100/H200 节点上 NVLink 域为 8-way NVSwitch，TP 可达 8，搜索空间从 $\binom{16 \times 10 + 5 - 1}{5-1} \approx 29M$ 可能膨胀到数十亿级——论文声称的"<35s 搜索时间"可能完全不适用 [2604.15186]。

              更关键的是，NVSwitch 拓扑下的 placement 约束完全不同：A6000 的 hierarchical heuristic（NVLink pair 内做 TP → 跨 pair 做 DP）在 NVSwitch 下变成"8-way NVSwitch 域内任意 TP → 跨域走 IB"——允许的 TP 组合呈组合爆炸。论文外推到 128 GPU 的搜索时间 (<70s) 是 microbenchmark 而非真实部署 [2604.15186]。PrfaaS 使用 H200 + H20 异构集群评估 [2604.15039]，DualPath 使用 Hopper GPU 集群 [2602.21548]——Scepsy 的硬件平台是 8 篇中最弱的。

              Attack 3: MPS 的 noisy-neighbor 效应被系统性低估 (L2 §Limitations)

              Scepsy 用 NVIDIA MPS 在同一 GPU 上实现 fractional 隔离，按 SM 百分比和 memory 上限限制每个 LLM 实例 [2604.15186]。但 MPS 的 SM 限制是 soft guarantee——L2 cache 和 memory bandwidth 无法被 MPS 隔离。当两个 LLM 共享 GPU 时：(1) L2 cache thrashing 导致两者 cache hit 率下降；(2) HBM bandwidth 争用导致 memory-bound decode 变慢；(3) SM 调度器的 occupancy-based 分配在 mixed workload（一个 compute-bound prefill + 一个 memory-bound decode）下行为不确定。

              论文的消融实验（Figure 10）证明 co-location 对 RAG 吞吐"最关键" [2604.15186]——但如果 noisy-neighbor 效应使 co-located 实例的延迟方差增大，Aggregate LLM Pipeline 的 steady-state 预测精度就会恶化（预测基于独立 profiling 的 per-LLM T-L 曲线，而非 co-located 场景的实测）。论文明确未讨论 QoS 保障 [2604.15186]。ZeRO-Prefill 的 frontend-backend co-design 通过物理量 T 保证 overlap [2605.02960]，PrfaaS 的 layer-wise pipelining 保证 compute/transfer 重叠 [2604.15039]——两者都有明确的性能保证机制，而 Scepsy 的 MPS co-location 缺乏类似的 formal guarantee。

              Attack 4: Aggregate Pipeline 的"pipeline 近似"在 fan-out workflow 中系统性误差 (L2 §Limitations)

              Scepsy 的延迟模型 $L_w = \sum_m L_m(\lambda_w \cdot n_m) \cdot n_m/p_m$ 假设 workflow 可近似为串行 pipeline——每个 LLM 依次贡献延迟，bottleneck LLM 决定吞吐 [2604.15186]。论文自己承认"如果 workflow 在不同 LLM 之间 fan-out，Aggregate Pipeline 会错误建模为串行" [2604.15186]。

              这不是边角 case——multi-agent 系统（AutoGen、CrewAI）的核心模式就是多个 LLM 并行推理然后聚合：debate 架构中 3-5 个 LLM 同时生成观点，judge LLM 聚合评分。在这种 parallel fan-out 下，$L_w$ 应为 $\max_m L_m$（而非 $\sum$），吞吐模型也需要考虑 GPU 的并发共享而非顺序占用。Scepsy 的搜索空间假设每个 LLM 独立占用 GPU fraction——但 parallel fan-out 意味着多个 LLM 同时需要 GPU，MPS 的 concurrent kernel 能力和 GPU memory 需要同时满足所有并行 LLM。PPD 通过 per-request 级别的动态路由可以适应请求级别的并行度变化 [2603.13358]；Scepsy 的 ahead-of-time allocation 无法在请求级别响应 fan-out degree 变化。

              Attack 5: 与 K8s HPA 的对比可能存在 under-tuning (L2 §Deep Analysis)

              Scepsy 的核心实验对比 K8s HPA（autoscaler）声称 2.4× 吞吐和 27× 延迟改善 [2604.15186]。但 HPA 的 scaling 策略参数（scale-up/scale-down 速度、冷启动时间、target utilization）"未说明" [2604.15186]——可能 under-tuned。HPA 在"扩上去 → 队列空 → 缩回来 → 队列堆积 → 再扩上去"的振荡中损失严重 [2604.15186]，但这一振荡行为高度依赖 HPA 参数配置（cooldown period、scale-down stabilization window）。

              更公平的对比应使用 KEDA（Kubernetes Event-Driven Autoscaling）或 Knative 的 queue-based autoscaler，而非 metrics-based HPA。此外 AIBrix/llm-d/vLLM Production Stack 等 LLM-aware 调度器已经内置了比 HPA 更智能的 autoscaling 策略——论文 Table 1 列出了这些系统但未实际评估任何一个 [2604.15186]。如果 AIBrix 的 LLM-aware autoscaler 能减少振荡，Scepsy 的相对优势可能从 2.4× 缩水到更modest 的数字。

              生态位 #

              范式定位：Scepsy 代表 LLM serving 从"per-LLM 独立优化"向"workflow-level 联合优化"的跃迁。类比于数据库领域从 single-query optimization 到 workload management 的演进，Scepsy 提出"scheduler 看 workflow、engine 看请求"的分层接口 [2604.15186]。这个定位在 2026 年的 agentic 浪潮中有明确的时代必要性：beam search/MCTS inference-time scaling、RAG pipeline、multi-agent debate 都是多 LLM workflow，需要跨 LLM 联合调度。

              但 Scepsy 的"ahead-of-time static allocation"范式与下层系统的"online dynamic scheduling"范式存在张力。PPD 的 per-request 路由 [2603.13358]、MFS 的 per-flow 调度 [2603.17456]、PrfaaS 的 dual-timescale scheduler [2604.15039] 都在 request/flow 级别做动态决策。Scepsy 在 deployment 级别做静态决策——profiling → search → deploy → 稳定运行。这意味着 Scepsy 无法响应 request-level 的负载突变（如 beam search 在某些 prompt 下 fan-out degree 剧烈变化），只能依赖下游 engine 的 continuous batching 吸收波动。

              采纳信号与障碍：

              • 正面：(1) 完全透明接入——workflow 用原 framework（LangChain 等）编写，HTTP proxy 无侵入 [2604.15186]；(2) 面向 K8s 生态——扩展标准 device plugin + MPS，不需要 RDMA/InfiniBand；(3) 对已有 vLLM/SGLang 栈低迁移成本
              • 障碍：(1) 代码仍在 anonymized 状态 [2604.15186]；(2) 仅在 A6000 评估，H100/H200 适用性未验证；(3) 依赖代表性 profiling trace，production drift 时需重新 profiling [2604.15186]；(4) MPS 在生产环境的 noisy-neighbor 风险

              未探索方向 #

              方向 1: Scepsy + Autellix 两层 scheduler。Scepsy 做 deployment-level GPU 分配（分钟级），Autellix/JITServe 做 request-level 优先级调度（ms 级）[2604.15186]。将两者叠加：Scepsy 的 Aggregate Pipeline 确定每个 LLM 的 GPU fraction/TP/replica，Autellix 在每个 LLM 实例内部做 workflow-aware 请求优先级排序。关键接口：Scepsy 的 profiler 捕获 per-LLM 调用统计量 $(n_m, p_m)$，Autellix 的 request-level scheduler 需要知道这些统计量以做跨 LLM 的优先级决策——两者需要共享 workflow trace 数据。这是 Scepsy 自己在 Deep Analysis 中预测的"下一步"方向。

              方向 2: Aggregate Pipeline + PrfaaS 的 selective offload。Scepsy 假设所有 LLM 在同一集群内 [2604.15186]。但 agentic workflow 中小模型（generator）和大模型（verifier）的 compute 需求差异巨大——可以把 verifier offload 到 PrfaaS 式的 compute-dense 远程集群，generator 留在本地 bandwidth-optimal 集群。PrfaaS 的长度阈值 $t$ 路由 [2604.15039] 可以推广为"LLM 角色阈值"——根据 LLM 在 workflow 中的角色（compute-heavy vs latency-sensitive）决定是否 offload。Scepsy 的 Aggregate Pipeline 模型需要扩展为 cross-cluster 版本，Eq.6 的 $\Lambda_{\max} = \min(\Theta_i/n_i)$ 变成 PrfaaS 式的双 producer 模型。

              方向 3: 动态 Aggregate Pipeline 配合 DualPath 式 I/O 优化。Scepsy 的 pipeline 模型假设 non-LLM 时间可忽略 [2604.15186]。当 agentic workflow 含有长 tool 调用时（DualPath 的 production trace 显示平均 157 轮 coding agent [2602.21548]），pipeline 模型需要将 tool 时间纳入——$L_w = \sum_m (L_m + T_{\text{tool},m}) \cdot n_m/p_m$，其中 $T_{\text{tool},m}$ 从 trace 中提取。进一步地，tool 执行期间 GPU 处于空闲——可以用 DualPath 的思路在 tool 等待期间预取下一轮的 KV-cache 或预热下一个 LLM。Scepsy 的 HTTP proxy trace 已经捕获了 tool 调用延迟，只需在 Aggregate 模型中增加 tool stage。

              方向 4: Frontend-backend co-design 借鉴 ZeRO-Prefill。ZeRO-Prefill 的饱和阈值 $T = t_{\text{EP}} \times F_{\text{GPU}} \times \gamma$ 是 backend 定义、frontend 强制的物理量 [2605.02960]。Scepsy 可以引入类似机制：backend（vLLM/SGLang engine）报告每个 LLM 实例的"最小有效 batch size"（低于此值 GPU utilization 坍塌），frontend（Scepsy scheduler）在搜索时将此作为硬约束而非仅优化吞吐-延迟曲线。当前 Scepsy 的 profiling 独立测量每个 LLM 的 T-L 曲线 [2604.15186]，但不建模 co-located 干扰——ZeRO-Prefill 的"T 必须被满足"范式可以防止 Scepsy 把 GPU fraction 分得太小导致 engine 性能坍塌。

              方向 5: KVServe 压缩 + Scepsy multi-LLM 的联合优化。Scepsy 目前不触及 data-plane 优化，但 multi-LLM workflow 中不同 LLM 之间可能存在 KV-cache 传递（如 RAG pipeline 中 embedding 模型的 output 作为 generation 模型的 input 前缀）。KVServe 的 service-aware controller [kvserve] 可以为每个 LLM pair 的 KV 传输选择最优压缩策略，Scepsy 的 Aggregate Pipeline 可以将传输带宽约束纳入搜索——当 co-located LLM 通过 GPU memory 直传 KV 时不需要压缩，当跨节点传输时 KVServe 自动启用。关键挑战：KVServe 不支持 MLA 架构 [kvserve]，而 Scepsy 的 beam search workload 可能使用 DeepSeek 类 MLA 模型。
