Pancake vs 相关论文：跨篇综合 #

相关论文 #

本篇选取 8 篇相关实体，覆盖 agentic 推理基础设施的全栈——从 KV-cache I/O（DualPath）、multi-turn 路由（PPD）、网络调度（MFS）、RL 权重传输（TensorHub）、跨 DC prefill（PrfaaS）、MoE prefill 优化（ZeRO-Prefill）、低层执行引擎（TileRT）到 KV 压缩（KVServe），与 Pancake 在"agent memory 子系统"这一独特定位上形成多维交叉。

DualPath (2602.21548) — Pancake 最近的"agentic workload 同族"。两者都面向 agentic LLM 推理、都依赖 GPU-CPU 协同、都在 DeepSeek/UCSD 级别的大规模集群上验证。但层级完全不同：DualPath 解决 PD 分离架构中 KV-cache 从存储到 GPU 的 I/O 带宽瓶颈 [2602.21548]，Pancake 解决 agent 外部记忆数据库的 ANN 查询/更新瓶颈 [2602.21477]。两者在数据通路上串联而非竞争：DualPath 管 LLM 自身的 KV-cache，Pancake 管 agent 调用的外部 memory。DualPath 的 agentic trace 分析（157 轮、429 token/轮、98.7% 命中率）[2602.21548] 恰好是 Pancake 所需的"agent 行为统计"的上游数据源。Kind: related。

PPD (2603.13358) — 与 Pancake 共享 multi-turn 场景但优化完全不同的子系统。PPD 发现 append-prefill 对 decode 的干扰仅 2%（vs full prefill 48%）[2603.13358]，据此在 decode 节点本地执行 Turn 2+ prefill，减少 KV 传输。Pancake 的 multi-turn 关注点是每轮 agent 生成时的 memory search/insert 交错——PPD 的"Turn 2+ 本地化"实际上使 Pancake 的 memory 操作更频繁（decode 节点上 prefill+decode 混合执行，memory query 与 LLM compute 更紧密耦合）。PPD 的 session affinity（Turn 2+ 路由到同一 D 节点）[2603.13358] 与 Pancake 的 per-agent FSM state 存在协同空间——同一 D 节点可以同时维护 KV cache 和 agent memory FSM。Kind: related。

MFS (2603.17456) — 两者面向 agentic workload 但优化正交的层面。MFS 解决 disaggregated MoE serving 中三阶段通信（KV 复用 + collective comm + P2D 传输）的网络争用 [2603.17456]，是网络调度层的优化。Pancake 是计算层的 memory 子系统优化。MFS 的 agent workload 特征化（短序列、高 KV 复用、突发性）[2603.17456] 正是 Pancake 面临的同一类 workload——但 MFS 关心的是这些特征对网络争用的影响，Pancake 关心的是对 ANN 索引 locality 的影响。两者可以同时部署在同一 agentic serving 集群中互不干扰。Kind: related。

TensorHub (2604.09107) — 与 Pancake 共享"GPU-CPU 协同"和"异步数据搬运"的系统设计哲学。TensorHub 的 Reference-Oriented Storage 通过 RDMA 直接引用 GPU 上已有权重 [2604.09107]；Pancake 的 GPU-CPU coordinated dynamic index 通过 async cluster migration 在 GPU 和 CPU 间搬运热点向量簇 [2602.21477]。两者面对相同的工程挑战：如何在不中断前台服务的前提下完成后台数据搬运（TensorHub 的 pipeline replication overlap、Pancake 的 GPU cache migration overlap）。但场景截然不同：TensorHub 面向 RL 训练中 TB 级权重的低频传输，Pancake 面向 agent 推理中 MB 级向量簇的高频搬运。Kind: related。

PrfaaS (2604.15039) — PrfaaS 把 prefill 选择性 offload 到跨 DC 的 compute-dense 集群 [2604.15039]，其 hybrid attention 模型将 KV 吞吐压到 ≈3 Gbps 使跨 DC 传输可行。Pancake 的 agent memory 在 PrfaaS 架构下面临新问题：如果 prefill 在远端 PrfaaS 集群执行，agent 的 memory search 应发生在 PrfaaS 端（靠近 LLM compute）还是 PD 端（靠近 memory DB）？PrfaaS 的长度阈值路由（长请求走 PrfaaS、短请求走本地 PD）[2604.15039] 暗示 Pancake 也需要类似的"memory query 路由"策略——长 context agent step 的 memory 查询送到远端与 prefill 共定位，短 step 的在本地解决。Kind: related。

ZeRO-Prefill (2605.02960) — 与 Pancake 共享"反转传统数据流方向"的设计哲学。ZeRO-Prefill 将 MoE 的"按 activation 路由到 expert"反转为"按 weight 流入到 activation" [2605.02960]；Pancake 将传统 IVF 的"把 insert 分配到 nearest centroid"反转为"根据 agent 访问模式重组 cluster hierarchy"。两者的核心 insight 都是：prefill/大 batch 的长 compute window 创造了足够的 overlap 机会。ZeRO-Prefill 利用这个 window overlap weight AllGather，Pancake 利用它 overlap GPU cluster migration。ZeRO-Prefill 的 frontend-backend co-design（饱和阈值 T 由 backend 定义、frontend 强制）[2605.02960] 对 Pancake 的 FSM-based scheduling 有启发——Pancake 缺乏这种物理量导出的保证。Kind: related。

TileRT (tilert-speed-scaling-law) — 两者在优化层级上差距最大。TileRT 用 AOT 编译将整个模型展开为单个 Persistent Engine Kernel，消除 inter-kernel idle [tilert-speed-scaling-law]。Pancake 的 GPU-side 计算（hot cluster search、K-means splitting）是独立的 CUDA kernel，与 LLM 推理的 kernel 交替执行。如果 LLM 推理采用 TileRT 的 persistent kernel 模式，Pancake 的 GPU kernel 如何被调度成为新问题——persistent kernel 独占 GPU SM，Pancake 的 ANN search kernel 需要 SM 时间片。TileRT 的 heterogeneous worker 设计（GPU0 做 Sparse Indexer、GPU1–7 做 MLA）[tilert-speed-scaling-law] 提示了一种可能：把 Pancake 的 GPU ANN 搜索指定到专用 GPU worker，与 LLM compute workers 并行。Kind: related。

KVServe (kvserve) — 与 Pancake 共享"数据压缩降低传输瓶颈"的策略空间但应用层不同。KVServe 压缩 PD 间的 KV-cache 传输 [kvserve]；Pancake 的 multi-level cache 本质上也是一种"压缩"——用 L0/L1/L2 三级 subset 代替全量索引扫描，减少实际需要的向量计算量。KVServe 的 service-aware controller（analytical model + bandit 在线选择 compression profile）[kvserve] 与 Pancake 的 FSM-based pattern table 都是 workload-adaptive 的决策机制，但 KVServe 的理论基础更强（有 benefit condition theorem），Pancake 的 FSM 更偏启发式。KVServe 的 vLLM external connector 零侵入接入方式 [kvserve] 对 Pancake 的集成设计有参考价值——Pancake 目前只给 LangChain/LlamaIndex/Mem-GPT 做了 adapter，但没有 vLLM/SGLang 级别的 serving engine 集成。Kind: related。

本篇 vs 相关论文的 delta #

Pancake 的核心 delta 是在 framework 类别中开辟了一个独特的子问题空间：agent 外部长期记忆的 ANN 索引系统。其他 7 篇论文+1 个代码项目全部聚焦 LLM 推理/训练本身的基础设施（KV-cache、通信、权重传输、执行引擎、压缩），Pancake 是唯一把优化目标放在 LLM 之外的 memory 子系统上的。

Pancake vs DualPath — 同一 agentic serving 集群的两个 I/O 子系统。DualPath 聚合 DE 空闲 SNIC 带宽加速 KV-cache 从 3FS 到 PE 的加载 [2602.21548]，Pancake 在 GPU 上缓存热点向量簇加速 agent memory search [2602.21477]。两者发现了 agentic workload 的两个不同瓶颈：DualPath 发现"KV-cache 命中率 98.7% 使 prefill 从 compute-intensive 退化为 I/O-intensive"；Pancake 发现"memory 操作占 end-to-end 时间 82%"。DualPath 发表后 Pancake 的价值反而增大——DualPath 把 KV-cache I/O 瓶颈消除后，Pancake 要解决的 memory 操作瓶颈成为下一个 bottleneck。两者组合构成一条完整的 agentic serving I/O 优化栈。但 DualPath 的工程深度（CNIC-centric data path、InfiniBand VL QoS 配置、1152 GPU 生产验证）[2602.21548] 远超 Pancake（合成 trace、无生产部署、代码未公开）。

Pancake vs PPD/MFS — 同一 agentic workload 的不同系统层。PPD 在请求路由层优化 multi-turn TTFT [2603.13358]，MFS 在网络调度层优化 disaggregated MoE 的 TTFT SLO [2603.17456]，Pancake 在应用数据层优化 agent memory 操作延迟。三者正交可叠加：PPD 减少 KV 传输 → MFS 在剩余传输上做优先级调度 → Pancake 在 LLM compute 间隙做 memory 操作。但 Pancake 的 FSM-based scheduling 缺乏 PPD 和 MFS 的理论严格性——PPD 有 offline profiling + scoring function 的完整决策框架 [2603.13358]，MFS 有 MLU 单调性保证的 RMLQ [2603.17456]，Pancake 的 FSM 是纯启发式的 state merge + d_merge 门限 [2602.21477]。

Pancake vs TensorHub — GPU-CPU 协同的两种形态。两者在"如何在 GPU 上缓存热数据、在 CPU 上暂存动态数据、用 async 搬运解耦两者"这一核心设计问题上高度同构。TensorHub 的 pipeline replication 利用 RDMA NIC 全双工使正在接收的 worker 立即向下游服务 [2604.09107]；Pancake 的 GPU-CPU search 利用 GPU 算 cached cluster + CPU 算 insertion buffer 并行 merge [2602.21477]。但 TensorHub 的工程壁垒远更深——progress counter 并发安全、partial-replica 故障恢复、tiny-tensor compaction [2604.09107]，有 1.3K 行 concurrency tests 做保障。Pancake 的 GPU-CPU consistency window 问题（cluster 搬迁过程中 search 如何保证 recall）[2602.21477] 缺乏同等严格的工程验证。

Pancake vs ZeRO-Prefill — "反转数据流方向"的设计模式。ZeRO-Prefill 反转 EP 数据流（activation 不再路由到 expert，weight 流入到 activation 所在 GPU）[2605.02960]，获得 1.35–1.37× 吞吐提升。Pancake 反转 IVF 索引策略（不再把 insert 分配到 nearest centroid，而是根据 agent 访问模式重组 cluster）[2602.21477]，获得 4.29× 端到端吞吐提升。ZeRO-Prefill 有物理量导出的饱和阈值 T = t_EP × F_GPU × γ 保证 overlap [2605.02960]，Pancake 的 CPU insertion buffer 大小 Binsert=128 只是经验临界点 [2602.21477]。ZeRO-Prefill 的 frontend-backend co-design 通过三接口（T 定义/prefix affinity/true-FLOPs）建立物理一致的系统 [2605.02960]；Pancake 的 FSM + hybrid graph + GPU cache 三组件之间缺乏类似的形式化联结。

Pancake 的独有增量：

  1. 唯一面向 agent 外部记忆的系统。8 个对比实体全部聚焦 LLM 推理/训练本身的计算、存储或通信，Pancake 首次将 ANN 索引系统设计为 agent serving 的一等组件。FSM 建模 agent 访问模式、hybrid graph 统一多 agent coarse 索引、per-cluster agent profile——这些抽象在其他论文中不存在。
  2. 唯一同时处理 search+insert 交错的动态索引。其余论文的优化目标（KV-cache、权重、网络 flow）都是单向数据流（写一次然后多次读），Pancake 面对的是 interleaved search-insert workload——每步 LLM 生成之间插入 ANN 查询和更新，这是更复杂的 consistency 问题。
  3. 唯一做 multi-agent coordination 的系统。DualPath 面向多 agent 但不做 agent 间协调（只是共享存储带宽）；Pancake 的 hybrid graph 和 agent profile 实现了"一次 BFS 跨多 agent scope 搜索"，是真正的 multi-agent memory sharing。

    4. 可攻击面 #

    以下攻击针对 Pancake L2 中的具体声明。

    Attack 1: "memory 操作占 82% 端到端时间"的痛点定量在 agent 框架快速演进中可能已过时

    Pancake 的核心 motivation 是 memory 操作占端到端 82% 以上 [2602.21477]。但这一数字来自 Mem-GPT/A-Mem/LangChain/LlamaIndex 的 naive Python 实现——这些框架的 memory 操作是同步 Python + CPU Faiss，没有任何 GPU 加速或异步执行。对比 DualPath 的 agentic trace：157 轮、429 token/轮 append、KV-cache hit 98.7% [2602.21548]——如果 agent memory 操作也能利用 prefix caching（同一 user 的 memory 高度重叠），82% 的占比可能大幅缩减。更关键的是，2026 年的 agent 框架（AutoGen、LangGraph、CAMEL）正在从"每步都查 memory"转向"选择性查询"——agent 学会了何时需要记忆、何时不需要，降低了 memory 操作频率。Pancake 应讨论 memory 操作占比对不同 agent 行为模式的敏感性，而非假设 82% 是稳态。

    Attack 2: FSM 建模 agent 访问模式缺乏理论保证，对比同类系统的形式化差距明显

    Pancake 的 FSM pattern table 用 d_merge 门限做 state merge、αet 做 early termination [2602.21477]，全部是启发式参数。对比同期 framework 论文的理论基础：DualPath 有 Eq.1–9 的 closed-form BW 约束分析 [2602.21548]；MFS 有 MLU 单调性保证 RMLQ 不抖动 [2603.17456]；PrfaaS 有 Eq.3–8 的吞吐模型和 monotonicity 保证唯一交点 [2604.15039]；ZeRO-Prefill 有物理量导出的 T [2605.02960]；KVServe 有 benefit condition theorem [kvserve]。Pancake 是 8 个对比实体中唯一一个核心调度机制完全没有形式化保证的。FSM 对"长 chain-of-thought 或嵌套 sub-task" agent 是否退化成每 request 一个 FSM？FSM table 规模与 agent 行为复杂度的关系是什么？这些问题论文自己也承认未解答 [2602.21477]。

    Attack 3: 4.29× 端到端吞吐提升的 baseline 极弱，与 vector DB 本身的优化对比不充分

    Pancake 的 headline 数字 1.12×–26.18×（平均 4.29×）是相对 Mem-GPT/A-Mem/LangChain/LlamaIndex 的 [2602.21477]。但这些 baseline 是 agent framework 而非 vector DB——它们的 memory 操作是 naive Python + synchronous CPU Faiss，Pancake 胜出的一大原因只是"用了 GPU"。更公平的对比应是：相同 agent framework + 不同 vector DB backend（Faiss-GPU / Quake / SPFresh / DiskANN）。论文确实做了独立 vector DB 对比（query 吞吐 1.9×–4.2×），但这只是 vector DB 层面的 microbenchmark，没有接入 agent framework 的端到端验证 [2602.21477]。如果把 LangChain 的默认 Faiss-CPU 换成 Faiss-GPU flat index（无 IVF 分簇），纯暴力 GPU 搜索在 8M 向量 × 1024d 上的延迟是多少？这个中间 baseline 的缺失使我们无法判断 Pancake 4.29× 中有多少来自"GPU 加速"（trivial 贡献）vs "多级缓存+FSM+hybrid graph"（创新贡献）。

    Attack 4: 合成 workload trace 与真实 production agent trace 之间的 gap

    Pancake 的 4 种 access pattern（One-Search-One-Insert / Step-Search-Then-Insert / Search-Then-Step-Insert / Search-Only）是手工枚举 [2602.21477]。对比 DualPath 使用 DeepSeek 生产 coding agent trace（157 轮、429 token/轮、98.7% KV hit 率）[2602.21548]、PPD 使用 ShareGPT 和 WildChat 真实数据集 [2603.13358]、MFS 使用 QwenA-Conv 和 QwenB-agent 两种源自生产的 workload [2603.17456]——Pancake 是唯一完全没有真实 agent trace 的系统论文。合成 trace 对 FSM 更友好（pattern 规律且可预测），但真实 agent 行为可能高度随机——ChatGPT Memory、Claude Projects、Copilot 的实际 memory 查询模式从未被公开研究。论文自己也在 limitations 中承认这一点 [2602.21477]。

    Attack 5: 多 agent 扩展只到 20 个，degradation <10.2% 的声明面临真实规模挑战

    Pancake 声称 20 agent 时 degradation <10.2%，hybrid graph 将 coarse search cost 降低 >20× [2602.21477]。但 20 agent 是极小规模——AI Town 等 multi-agent 框架动辄 50-100+ agent，enterprise 级 multi-tenant 场景可能有数千个 agent 共享 memory。hybrid graph 的 inter-connection portal 节点数量随 agent 数增长，portal edge 的概率 1/ef_connect [2602.21477] 在 100+ agent 时是否导致 graph 过度密集？agent profile per static cluster [2602.21477] 的空间开销随 N_agents 线性增长，100 agent × 100K cluster 的 profile table 大小是多少？论文未探索这些 scale-up 问题，且 8M vectors 是"中等规模" [2602.21477]，真实的 enterprise agent memory 可能到数十亿级。

    Attack 6: GPU-CPU async migration 的 recall consistency 问题未量化

    Pancake 的 GPU cache migration 过程中"search 会分别打到新/旧区域"[2602.21477]，论文给出"merge 结果"描述但没有量化 recall 漂移。TensorHub 面临类似挑战（pipeline replication 中 partial-replica 的数据完整性）但用了 progress counter + epoch-based reclamation 做精确保证 [2604.09107]。Pancake 未说明在 migration window 内是否可能出现：(1) 同一向量被新旧两个区域都返回导致 top-k 重复；(2) 正在迁移的向量在短窗口内既不在新区也不在旧区导致 recall 掉点。对于要求 recall >0.95 的 agent memory 场景，几十毫秒的 consistency 窗口可能造成不可接受的服务质量波动。

    生态位 #

    范式定位：Pancake 代表了 LLM serving 基础设施的一个新子领域——agent memory 子系统。传统 LLM serving 栈是 model→KV-cache→scheduler→network，Pancake 在这条栈旁边开辟了 agent→memory-DB→ANN-index→GPU-cache 的平行管线。这不是对现有栈的优化，而是新增一个一等组件。类比 database 领域：传统关系型引擎 + 搜索引擎是两套独立系统，Pancake 是"为 LLM agent 定制的搜索引擎"首次被设计为 serving 框架的内置模块。

    与 LLM serving 栈的集成定位：

    Pancake 已给出 LangChain/LlamaIndex/Mem-GPT 的接入接口 [2602.21477]，但缺乏与 vLLM/SGLang 级别引擎的集成。对比 KVServe 通过 vLLM external connector 零侵入接入 [kvserve]、PPD 作为 vLLM scheduler-level patch [2603.13358]、MFS 作为 NCCL/Mooncake adapter [2603.17456]——Pancake 的集成层级最高（application-level adapter），优点是对下游无侵入，缺点是无法利用 serving engine 内部信息（如 KV cache pressure、scheduler queue depth）做协同调度。

    采纳信号：

    • 正面：(1) 首次系统性定义了 agent memory 的性能问题，其"memory 操作占 82%"的发现对社区有教育价值；(2) 多级缓存+FSM 的方案概念清晰，实现原理可移植到任何 IVF 索引；(3) hybrid graph 的 multi-agent coarse search 是独创贡献
    • 障碍：(1) 代码未公开——最大阻碍 [2602.21477]；(2) 缺乏真实 agent trace 和生产验证，对比 DualPath/TensorHub 的千 GPU 生产部署差距巨大；(3) 约 8 个需要调的配置旋钮（Np, NS, dmerge, αet, αic, efconnect, efsearch, Binsert）[2602.21477] 且无 auto-tuning；(4) 绑定 GPU-CPU 异构平台和 IVF 索引，不适用于纯 CPU 或 pure-graph 索引

    定位判断：Pancake 在"multi-agent + large-scale external memory + interleaved search-insert"的三重交集场景下价值最大。但 2026 年中 agent memory 仍是新兴领域——大多数 production agent 只用简单的 RAG pipeline（向量检索 + 重排），不需要 Pancake 的复杂多级索引。Pancake 的实际采纳取决于 agent memory 规模是否真的到达 8M+ 且 multi-agent 共享成为刚需——这在当前以单 agent 为主的生产环境中尚未普遍发生。

    未探索方向 #

    方向 1: Pancake + DualPath 联合优化 agentic serving 的完整 I/O 栈。DualPath 解决 KV-cache I/O [2602.21548]，Pancake 解决 memory ANN I/O [2602.21477]，两者在同一 serving 集群中串联运行。未探索的协同点：DualPath 的 decode engine 已经用 CNIC loopback 做 GPU↔DRAM 数据搬运 [2602.21548]——Pancake 的 GPU cluster migration 可以复用这条已建立的 RDMA 数据通路，而非独立管理 PCIe async copy。更深层的协同：DualPath 的 layerwise streaming（逐层加载 KV-cache 与 prefill compute overlap）[2602.21548] 创造了 GPU idle window——每层 KV-cache 在 DRAM→HBM 搬运时 GPU 不做计算，这个 window 恰好可以用于 Pancake 的 memory search GPU kernel。将 Pancake 的 hot cluster search 嵌入 DualPath 的 layerwise pipeline，实现"KV-cache 搬运时做 memory search、prefill compute 时做 KV-cache 搬运"的三级流水。

    方向 2: 用 ZeRO-Prefill 的饱和阈值 T 替换 Pancake 的启发式 Binsert。Pancake 的 CPU insertion buffer 大小 Binsert=128 是"CPU 搜 128 条的时间 ≈ GPU 搜单个 cluster 的时间"的经验值 [2602.21477]。ZeRO-Prefill 的 T = t_EP × F_GPU × γ 提供了一个更优雅的范式——从 GPU 的物理能力（FLOP rate）和传输延迟（AllGather/PCIe）直接推导阈值 [2605.02960]。应用到 Pancake：T_buffer = t_migration × F_GPU_search × γ，其中 t_migration 是 cluster 从 CPU→GPU 的 async copy 延迟，F_GPU_search 是 GPU 上 brute-force search 的向量吞吐。只要每 batch memory search 的 GPU compute ≥ T_buffer，migration 就被完全 overlap。更进一步，ZeRO-Prefill 的 frontend-backend co-design（frontend 保证 batch ≥ T）[2605.02960] 可以移植到 Pancake——agent controller 将多个 agent 的 memory query batch 起来直到 FLOPs ≥ T_buffer 再发给 GPU。

    方向 3: 借鉴 KVServe 的 service-aware controller 做 Pancake 的 adaptive 层级策略选择。Pancake 的 L0/L1/L2 三级 cache 和 FSM 参数是静态配置的 [2602.21477]，但 agent workload 会漂移（从 planning-heavy 转为 execution-heavy）。KVServe 的 analytical model + ε-greedy bandit 两层控制器 [kvserve] 提供了在线自适应的参考架构：Tier-1 用 Pancake 的"搜索次数 vs recall"分析模型快速筛选可行的 cache 层级配置（L0-only、L0+L1、全三级），Tier-2 用 bandit 在运行时学习哪个配置在当前 workload 下最优。关键技术对接：KVServe 的 benefit condition B < (1−1/cr)·S 的等价形式——"当 agent memory 查询频率低于阈值时，维护 L0 cache 的开销超过收益，退化到 L2 直接搜索更好"。

    方向 4: 将 Pancake 的 hybrid graph 与 PPD 的 session affinity 结合做 multi-agent memory-aware 路由。PPD 的 Turn 2+ 请求被路由到持有 KV cache 的特定 D 节点 [2603.13358]。如果扩展 session affinity 到 Pancake 的 agent memory——同一 agent 的所有请求路由到持有该 agent FSM state 和 L0 cache 的节点——可以实现 PPD 和 Pancake 的协同优化：D 节点同时维护 KV cache（PPD 的 Turn 2+ append-prefill）和 agent memory L0 cache（Pancake 的 hot subset）。PPD 的 scoring function S(ψ; π, w) [2603.13358] 可以扩展为 S'(ψ, m; π, w)，新增 m = agent memory cache hit ratio 作为路由因素——memory cache miss 意味着需要到远端查全量索引，此时路由到 P 节点做 full memory search 可能更优。

    方向 5: TileRT 的 persistent kernel 模式下的 agent memory 执行模型。如果 LLM 推理采用 TileRT 的 persistent Engine Kernel（GPU 持续驻留，不释放 SM）[tilert-speed-scaling-law]，Pancake 的 GPU ANN search kernel 无法通过常规 CUDA launch 获得 SM 时间。需要两个方向之一：(a) 将 Pancake 的 ANN search 编译为 TileRT Engine Kernel 内部的 tile task，在 LLM 层间边界执行——类似 MFS 在 layer 边界做 flow priority promotion [2603.17456]；(b) 利用 TileRT 的 heterogeneous workers 设计，指定一个 GPU 专用于 agent memory 操作（类似 TileRT 的 GPU0 = Sparse Indexer）[tilert-speed-scaling-law]。方案 (a) 更高效但要求 Pancake 的 ANN 算法适配 tile 粒度；方案 (b) 浪费一个 GPU 但实现简单。
