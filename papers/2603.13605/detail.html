Orla: A Library for Serving LLM-Based Multi-Agent Systems — L2 #

§1 TL;DR #

Orla 是面向 multi-agent LLM 系统的 workflow 级 serving 库，将请求执行与 agent 级策略分离——通过 stage mapper（异构模型路由）、workflow orchestrator（DAG 调度）、memory manager（跨 stage KV cache 生命周期管理）三组件，在 SWE-bench Lite 上实现 38–60% 延迟降低与 35% 成本节约。

§2 核心三问 #

Q1 痛点 #

当前 agentic 系统的 serving 栈是松散拼凑的：开发者手动将编排代码（LangGraph / AutoGen）与 request 级 LLM 引擎（vLLM / SGLang）粘合。这些引擎通过 prefill/decode 调度、continuous batching 和 PagedAttention 优化单请求执行，但对 workflow 结构毫无感知。两个核心缺口：

  1. 异构性（heterogeneity）——同一 workflow 的不同 stage 需要不同规模模型（路由/分类用小模型，综合/代码生成用大模型）、不同后端（SGLang on GPU vs. Ollama on edge）、不同性能目标。
  2. 跨 stage 协调（coordination）——需要决定每个 stage 用哪个模型、在哪执行、如何调度共享资源和传播 context，但现有 request 级引擎缺乏跨 stage 协调机制。

    3. Q2 方法 #

    核心 insight：将 request 执行与 workflow 级策略分离，在现有推理引擎之上插入一层 serving 控制面。三个组件：

    1. Stage mapper——将 workflow DAG 中的每个 stage 映射到具体 model + backend + 推理参数。支持静态显式映射和动态路由（内置 OneBitStageMapper：用轻量 LLM 将请求分为 simple/complex 再路由到不同规模模型）。
    2. Workflow orchestrator——两级调度：(a) stage scheduler 在 per-backend queue 中选择服务哪个 stage（FCFS / priority / 可扩展 SJF），(b) request scheduler 在 stage 子队列中选择请求。Stage 完成后通过 prompt builder 函数将输出传播为下游 stage 的 context。
    3. Memory manager——在 workflow 级管理 KV cache 生命周期，使用 orchestrator 的结构信号（stage 依赖、backend 切换、workflow 完成）决定 preserve / flush。三种策略：小增量保留、workflow 边界 flush、内存压力驱逐。

      4. 核心技术壁垒：将 serving 层定位为 workflow 级控制面——介于上层编排框架（LangGraph / AutoGen）与下层推理引擎（vLLM / SGLang）之间——利用 workflow 结构信号（stage 转换、DAG 依赖、workflow 完成）做出两层都无法独立做出的 serving 决策（跨 backend 路由、跨 stage cache 生命周期、结构感知调度）。这种"夹层"定位需要同时理解上下两层的接口契约，是单纯优化 request 级引擎或单纯扩展编排框架都无法替代的。

      Agent-specific 检查：

      • Success-rate model: 不适用——Orla 是 serving 层，不改变任务成功率，仅改变延迟/成本。
      • Latency budget per turn: stage mapping 实验报告了 per-request completion time CDF，但未分解为 tool call / LLM inference / memory read 各阶段延迟。
      • Failure mode classification: 论文未讨论故障模式、错误恢复或 agent 失败场景。
      • 可以被形式化但未做的：给定 DAG 结构和各 stage 的到达率/服务率，建立 multi-class queueing model 分析各调度策略下的平均延迟——这对 Mitzenmacher 的背景来说是自然延伸。

      §5 实验与数据 #

      Stage mapping（SWE-bench Lite） #

      Paper Figure 2, verbatim (caption: "(a) Wall-clock time (b) Cost (c) Completion time.").

      实验设置：单机 NVIDIA RTX PRO 6000 Blackwell Max-Q 96 GB VRAM, CUDA 13.0。vLLM 同时 serve Qwen3-8B（60% GPU 内存）和 Qwen3-4B（35%），max context 32,768 tokens, temperature 0, max output 4,096 tokens。

      OneBitStageMapper 将 956/2,294 请求分类为 simple 并路由至 Qwen3-4B，其余走 Qwen3-8B。三项指标均有显著改善：wall-clock −38%、mean completion −60%、cost −35%。Completion time CDF 在 Orla 配置下全分布左移，说明改善不仅来自平均值而是全局性的。

      注意：SWE-bench Lite 实验使用简化的单 stage prompt（提取 gold-patch 文件 → 生成 diff），并未使用多 stage 工作流——这意味着 stage mapping 评估实际未行使 Orla 的 workflow 编排能力。

      Cache management（DAG-MATH） #

      Paper Figure 3, verbatim (caption: "Cache management (request vs. workflow flush).").

      实验设置：SGLang serve Qwen3-8B，temperature 0, max output 256 tokens。比较两种 KV cache 策略：flush-per-request（每次 LLM 调用后清空）vs. flush-per-workflow（Orla 默认，跨 stage 保留 cache，workflow 完成时 flush）。

      结果：flush-per-workflow 在整个 TTFT 分布上优于 flush-per-request。但实验仅使用 DAG-MATH 的前 5 个问题（共 2,894 个），且未报告绝对 TTFT 数值、cache hit rate 或内存节省量。

      Memory manager workflow 示意 #

      Paper Figure 8, verbatim (caption: "In the customer support workflow, classify runs on a lightweight backend while later stages run on a heavier backend...").

      Memory manager 行为示意：classify（轻后端）→ policy_check（重后端）时 flush cache（backend 切换），policy_check → reply（同一重后端，共享 context）时 preserve cache，workflow 完成时 reclaim 所有 cache。显式 per-stage hint SetCachePolicy("flush"/"preserve") 可覆盖默认策略，但 Orla 在 backend 切换时会自动识别并 flush。

      未评估的维度 #

      • Workflow orchestrator 的调度组件（FCFS vs. priority vs. SJF）未做定量评估。
      • 无 multi-agent 并发场景下的 throughput 实验。
      • 无 ThunderAgent 对比（尽管定位为最近相关工作）。

      §7 实现 cross-reference #

      开源仓库：github.com/dorcha-inc/orla（Go 语言实现）

      • 核心 API 定义在 Appendix A（Figure 5）：orla.NewOrlaClient, orla.NewSGLangBackend, stage 定义, wf.AddDependency, wf.Execute(ctx)
      • Stage mapper interface 支持自定义 mapper，内置 OneBitStageMapper
      • Scheduling 策略通过 SetSchedulingPolicy("priority") / SetRequestSchedulingPolicy("fifo") 配置
      • Cache 策略通过 SetMemoryPolicy(orla.NewFlushAtBoundaryPolicy()) / SetCachePolicy("flush"/"preserve") 配置
      • Backend 统一通过 OpenAI-compatible HTTP API 通信，orla.NewSGLangBackend / orla.NewOllamaBackend / orla.NewSimulatedBackend 三种后端

      核心技术壁垒实现：workflow 控制面的"夹层"定位体现在代码架构上——Orla 不修改底层引擎，仅通过 HTTP API 交互；也不替代上层编排逻辑，仅通过 stage/workflow 抽象暴露 serving 决策点。这使得同一 workflow 定义可跨硬件环境运行（Docker Compose 一键部署，GPU path 或 CPU path）。

      关键实现细节：

      1. Simulated backend 用参数化 prefill/decode 延迟模型替代真实 LLM 推理，隔离编排行为与推理方差——研究者无需 GPU 即可进行 scheduling policy 实验。
      2. Builder 函数模式——stage 间 context 传播通过 SetPromptBuilder(func(upstream map[string]*orla.StageResult) (string, error)) 实现，每个 stage 声明如何从上游输出构造 prompt，而非全局状态传递。
