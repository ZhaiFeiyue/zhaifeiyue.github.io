ConServe — Conversation-Level Disaggregated Scheduling for Agentic Serving #

1. TL;DR #

Agentic serving is fragile because the scheduling unit is the turn: per-turn placement must predict unobservable decode-side cost (decode length, tool calls, KV growth). Raising the unit to the conversation collapses turn irregularity into one compute-bound turn-1 prefill + one long memory-bound tail, so placement reduces to two directly-observable signals — first-turn input length and per-decoder KV occupancy. ConServe routes turn-1 prefill to a prefiller, transfers KV once, and pins the conversation to one decoder for life. Result: p95 TTFET −51.08% and +7.51% energy vs a per-turn prediction baseline, +22.75% more energy on heterogeneous tiers, with zero learned cost model.

2. 痛点 / 方法 / 结果 (Q1/Q2/Q3) #

Q1 痛点 (what breaks). In single-shot serving the scheduled unit (a request = one prefill+decode) coincides with the unit the user values, so per-turn optimization is harmless. Agentic workloads break this coincidence: a user task is a stateful multi-turn ReAct program whose total cost is unknown at arrival, and most intermediate turns emit only tool calls the user never reads. This produces a scheduling–value mismatch — the system optimizes per-turn latency while the user waits for the conversation's final externally-meaningful output. Prior multi-turn systems (AMPD, PPD) keep the turn as the decision unit and, for each turn, decide whether to disaggregate by predicting/profiling its cost. But the cost driver (decode length, whether a tool is called, KV growth) is unobservable at the moment the scheduler must act, so any predictor eventually misroutes. The paper's sharpest claim: this dependence on prediction is imposed by the scheduling unit, not the workload.

Q2 方法 (the mechanism). Schedule the conversation, not the turn. At conversation granularity the irregularity collapses into a stable two-phase structure: (1) an initial compute-bound turn-1 prefill that materializes task state, (2) a memory-bound trajectory (first decode + every subsequent append-prefill/decode turn) where appends are tiny relative to accumulated KV. This restores the classical prefill–decode abstraction at conversation level: one compute phase, one memory phase, one KV transfer at the boundary. ConServe instantiates it with a minimal policy — route turn-1 prefill unconditionally to the prefiller, transfer KV exactly once, bind the conversation to the least-KV-occupied decoder, and pin it there for its entire tail (no re-evaluation, no learned decode-cost model). Placement reads only two observable signals: the offline-profiled deterministic prefill-latency curve (from input length) and live per-decoder KV occupancy.

核心技术壁垒 (the single hardest-to-replicate insight). The load-bearing move is deliberately making the prefiller the throughput bottleneck by overprovisioning decoders, so the system's ceiling sits on the side where the driving signal (input-token rate) is observable at admission and maps deterministically to utilization via the offline prefill curve. This converts capacity management, admission, and autoscaling from forecasting problems into reading-current-state problems. It is counterintuitive (one normally avoids hard bottlenecks) and is what lets the whole scheme run with no cost model — the part a re-implementer is most likely to get wrong by "balancing" the tiers instead.

Q3 结果 (what it buys). Against the per-turn AMPD baseline: p95 TTFET −51.08% (gmean −19.17%), +7.51% tokens/J, zero SLO violations up to the 1.634 conv/s saturation point, while last-turn TBT and SLOs are preserved. Mapping the two phases onto heterogeneous tiers (300W prefiller / 200W decoders) adds a further +22.75% tokens/J with essentially unchanged p95 latencies. SLO violations and energy of the per-turn baseline degrade linearly in prediction-error rate, while ConServe holds constant by construction.

3. 架构 / 方法图 #

Paper's Figure 9, verbatim (caption: "ConServe System Architecture.").

The system runs a prefill-decode disaggregated topology: one prefiller node (compute-bound, processes turn-1 prompts) and N decoder replicas (memory-bound tail). The ConServe Scheduler box consumes two monitoring signals — Prefiller Context Monitoring (against an offline Prefill Latency Modeling) and Decoder KV-Cache Utilization Monitoring — shown as dashed "Monitoring Signal" arrows distinct from the solid "Data Path". A new conversation's turn-1 prompt flows to the prefiller; on Prefill Completion its KV cache is transferred once to a bound decoder (KV Occupancy decides which), and the conversation is then "pinned" there. The reader should notice the asymmetry the diagram encodes: exactly one compute-bound hop per conversation, then a permanently local memory-bound tail — no arrow ever routes turn 2+ back to the prefiller.

Request lifecycle as a state machine:

stateDiagram-v2 [*] --> Turn1Prefill: new conversation (read L_in) Turn1Prefill --> KVTransfer: prefill done on prefiller KVTransfer --> Bind: transfer KV once Bind --> Tail: pin to least-KV-occupied decoder Tail --> Tail: turn 2+ append-prefill + decode (local, prefix-cache reuse) Tail --> [*]: conversation ends (decrement KV occupancy)

The binding is persistent: once Bind fires, every later turn re-enters Tail on the same decoder, so KV state never crosses the network again after the single transfer. Scheduling decisions occur only on the [*] --> Turn1Prefill and Bind edges — nowhere inside the tail loop.

4. 作者证明 #

The paper offers a lightweight analytical provisioning model (no learned model, no convergence theorem), plus load-bearing empirical fits. The formalization lives in §4.1.

Notation table:

SymbolMeaning
$N$number of decoder replicas
$T_d$per-decoder token throughput
$R$conversation arrival rate
$L_d$mean per-conversation token volume handled by decoders (turn-1 decode + all turn 2+ prefill & decode)
$B$concurrent conversation slots per decoder (GPU mem ÷ per-conversation peak KV footprint)
$W$mean wall-clock conversation lifetime (incl. external tool-call time)
$T_p$prefill input throughput
$L_{in}$mean first-turn input length
$R^{*}$arrival rate at which the prefiller saturates

Equations and their physical meaning:

Throughput constraint — aggregate decoder throughput must keep up with token generation demand:

$$N \cdot T_d \geq R \cdot L_d$$

Memory constraint — total pinned-conversation slots must hold all concurrently live conversations (Little's law, $R \cdot W$):

$$N \cdot B \geq R \cdot W$$

Prefill saturation point — dividing prefiller throughput by mean input length gives conversations/sec the prefiller absorbs:

$$R^{*} = \frac{T_p}{L_{in}}$$

ConServe provisions $N$ as an integer that more than satisfies both inequalities at $R = R^{*}$. The design intent: force the binding constraint onto the prefill side, where $L_{in}$ is observable at admission and $T_p$ maps deterministically to utilization.

Load-bearing empirical fits (the "observability" claim rests on these):

Turn-1 prefill latency is a near-deterministic quadratic in input length (attention-dominated at agentic scales):

$$\text{TTFT}_{\text{prefill}}(L) \approx 1.47\,\text{ns}\cdot L^{2} + 10.5\,\mu\text{s}\cdot L + 13.8\,\text{ms} \quad (R^{2}=1.000,\; L \geq 1024)$$

With a prefix-cache hit, turn 2+ prefill collapses to near-constant $\approx 27\,\text{ms}$ regardless of $L$ — the justification for running appends locally on the decoder. KV-transfer cost is linear in $L$ and therefore dwarfed by quadratic prefill exactly in the agentic long-input regime, justifying "transfer once":

$$\text{KV-transfer}(L) = \begin{cases} 4.8\,\text{ms} & L < 1024 \\ 0.98\,\mu\text{s/tok}\cdot L + 5.3\,\text{ms} & L \geq 1024 \end{cases} \quad (R^{2}=0.981)$$

6 minimum checks:

  1. Notation reproduced — all 9 symbols above ($N,T_d,R,L_d,B,W,T_p,L_{in},R^{*}$) defined verbatim from §4.1.
  2. Why ≥ not = — both constraints are floors; overprovisioning $N$ pushes the equality (bottleneck) onto the prefill side deliberately.
  3. Monotonicity / boundary — both constraints are increasing in $R$; the operating point is the boundary $R=R^{*}$, chosen so the prefiller (not decoders) is the active constraint. The optimum is a boundary, not interior, by construction.
  4. First-order mapping (verify reported numbers) — §5.1 gives $T_p \approx 25\text{k}$ input tok/s, $T_d \approx 1\text{k}$ output tok/s, $B \approx 300\text{k}$ KV tokens/decoder; a conversation averages ~15k input + ~1k output tokens. Throughput floor: $R L_d/T_d$ with $L_d\approx$ (1k decode + append work) yields the paper's stated "at least 1.67 decoders per prefiller," so $N=3$ (integer, exceeds both floors at $R^*$) — matches the reported config and the "prefiller saturates first" guarantee.
  5. *$R^{}$ sanity* — $R^{}=T_p/L_{in}=25\text{k}/15\text{k}\approx 1.67$ conv/s, matching the 1.634 conv/s synthesized saturation rate used in §5.2/§5.3.
  6. No learned model — the entire policy conditions on $L_{in}$ (observable) and live KV occupancy (observable); no decode-side quantity is forecast, consistent with eqs (1)–(2) containing only measured aggregates.

    7. 5. 实验与数据 #

    Setup: SWE-bench bm25 13K traces via swe-agent (generated with Qwen3-Coder-30B-A3B), replayed on a 4× NVIDIA A40 machine serving Qwen3-0.6B (bf16); 1 prefiller + 3 decoders; implemented on vLLM + LMCache. Baselines: Collocated, Full Disaggregation, and a best-effort AMPD (per-turn prediction, default 10% wrong-prediction rate). SLO = 5× single-request baseline latency.

    Paper's Figure 10, verbatim (caption: "Normalized agentic performance results over multiple request arrival rates (lower is better). AMPD has 10% wrong prediction rate.").

    ConServe beats AMPD by up to 19.17% (gmean) and 51.08% (p95) on TTFET. At saturation (1.5–1.634 conv/s) ConServe holds steady while Collocated and AMPD degrade sharply at p95 — Collocated from prefill-decode contention, AMPD from accumulated routing-error queue pressure on the prefiller. Full Disaggregation is uncompetitive everywhere (gmean TTFET/E2E >10× baseline) because routing every turn through the prefiller forfeits cross-turn KV reuse. The honest loss row: last-turn TBT inverts — Full Disagg (1.35× baseline) beats ConServe (2.49×, i.e. 91.85% higher) since its decoders never touch prefill; but that penalty is dwarfed by Full Disagg's 18.78× vs ConServe's 2.53× gmean TTFET (→4.7× worse E2E). Collocated has the worst p95 TBT at saturation (7.52×, 72% worse than ConServe).

    Paper's Figure 11, verbatim (caption: "Overall TTFT and TBT distribution at RPS 1.634.").

    The TTFT CDF resolves into three regimes: below the 0.7 percentile (turn 2+ prefills) ConServe/AMPD/Collocated overlap; between 0.7–0.8 (turn-1 prefills) ConServe pulls ahead because Collocated batches them with ongoing appends and AMPD queues behind wrongly-disaggregated turn 2+ prefills; above 0.8 the order flips as AMPD's accumulated prefill queue stretches into the tail. The TBT CDF shows ConServe ≈ AMPD throughout, and Collocated winning below 0.7 only because it dedicates 4 GPUs to decode vs ConServe's 3 (one-third more aggregate bandwidth) — an advantage that inverts once long turn-1 prefills contend with decode.

    Paper's Figure 12, verbatim (caption: "Comparison between ConServe and AMPD over different wrong prediction rate.").

    This is the load-bearing case study. At 0% wrong-prediction rate AMPD routes all appends locally and reduces exactly to ConServe — i.e. the per-turn baseline's best case is ConServe's policy. At just 5% wrong predictions, SLO violations already hit ~7.8% (TTFET) and 6.3% (E2E); by 50% both gmean/p95 latencies grow linearly and violations exceed 50%. The linearity is mechanistic: each wrong decision injects one unit of unanticipated load into a system running at saturation, so queue delay scales with cumulative excess load. Panel (d) shows energy: at saturation with 10% wrong predictions AMPD takes 15% longer to drain and delivers 7.5% fewer tokens/J, declining monotonically toward 58 tokens/J at 50%; ConServe is a flat line — it makes no per-turn decision, so it has no degradation curve.

    Paper's Figure 13, verbatim (caption: "P95 agentic performance and energy efficiency on heterogeneous GPUs.").

    Testing the phase-fidelity hypothesis: homogeneous (300W/300W) vs heterogeneous (300W prefiller / 200W decoders). Capping decoders to 200W raises tokens/J by 22.75% while p95 TTFET, last-turn TBT, and E2E stay essentially unchanged — the memory-bound tail absorbs the power cap because its throughput is gated by memory bandwidth, not the compute the cap removes. Baselines can't exploit it: Collocated's p95 TTFET worsens 11.15% (the cap starves its co-batched prefill), and Full Disagg misses SLO in both configs.

    7. 实现 cross-reference #

    The paper reports an implementation but does not release code in the L1 source; concrete integration points are stated in prose (§5.1), so replication rests on published components:

    • Serving engine: vLLM (Kwon et al., 2023) — [实现未公开] at file:line granularity; prefiller configured with vLLM standard 8192 chunked-token size per batch (§5.1 para 3).
    • PD-disaggregation / KV manager: LMCache (Cheng et al., 2025, arXiv:2510.09665) used as the prefill-decode disaggregation manager and the KV-transfer path (§5.1 para 3).
    • AMPD baseline: reconstructed best-effort — bidirectional KV transfer simulated (latency modeled + requests stalled), traces post-processed with offline profiling; default 10% wrong-prediction rate (§5.1 para 3). This is a fairness caveat to flag, not a defect.
    • Collocated baseline: 2944 chunked-token size, derived so iteration latency stays within 5× SLO per DynamoLLM practice (§5.1).

    关键实现细节 (easy-to-miss tricks):

    1. *KV-occupancy signal must be active, not allocated. The scheduler decrements per-decoder occupancy on conversation termination* so the signal reflects only live working-set KV, never allocated-but-idle memory (§4.2 para 1). Binding to the "least-KV-occupied" decoder is only correct if this bookkeeping is exact — a re-implementer using allocated memory would mis-bind under churn.
    2. *Provision $N$ as an integer that strictly exceeds both eqs (1)+(2) at $R=R^{}$**, not a balanced value — the "1.67 decoders needed → deploy 3" jump is intentional slack so the prefiller (observable side) saturates first (§4.1, §5.1). Losing this slack silently moves the bottleneck to the unobservable decoder side and reintroduces the prediction problem the design eliminates.
