SAGA: Workflow-Atomic Scheduling for AI Agent Inference on GPU Clusters — L2 #

§1 TL;DR #

SAGA treats entire agent workflows—not individual LLM calls—as first-class schedulable units. Using Agent Execution Graphs to predict KV cache reuse across tool-call boundaries, it achieves 1.31× of Bélády's optimal offline eviction and reduces task completion time by 1.64× (geo-mean) over vLLM+APC on a 64-GPU cluster serving SWE-bench and WebArena agents.

§2 Q1 / Q2 / Q3 #

Q1 痛点 #

Agent scope: SAGA targets open-ended multi-turn autonomous agents (coding agents on SWE-bench, browser agents on WebArena) running iterative ReAct loops with 10–150 LLM calls per task, interleaved with external tool invocations of highly variable latency (45 ms file ops to 45 s web API calls at P99).

GPU cluster schedulers (vLLM, SGLang, Orca) treat each LLM call independently: KV cache is evicted via LRU during tool-call idle periods, requests from the same session scatter across workers, and fairness is defined per-request rather than per-task. Instrumentation on a 32-GPU cluster shows 38% of total execution time spent regenerating discarded KV cache, 42% average GPU memory utilization, and 6.0× end-to-end latency inflation vs. inference-only baseline. The mismatch is quadratic: a $k$-step agent with per-step cache $c$ incurs $O(k^2 c)$ regeneration under request-level scheduling vs. $O(c)$ with perfect workflow awareness.

Paper's Figure 1, verbatim (caption: "Inefficiencies in serving agent workloads with request-level scheduling").

Three sub-panels quantify the gap: (a) 38% time in KV regeneration for vLLM vs. 8% for SAGA; (b) GPU memory utilization rising from 42% to 71%; (c) latency overhead dropping from 6.0× to 1.5× on log scale. The figure makes the case that the overhead is dominated by cache churn during tool-call gaps, not by compute itself.

Q2 方法 #

Three mechanisms unified under the "workflow-as-unit" thesis:

  1. Agent Execution Graphs (AEGs) + WA-LRU eviction — formalize agent workflows as DAGs $G=(V,E,P,\phi)$ with transition probabilities and tool-type mappings. WA-LRU scores eviction candidates by a weighted combination of recency, predicted reuse probability (computed from AEG successor structure), and size: $P_{evict}(s) = 0.3\hat{R}(s) + 0.5(1 - P_{reuse}(s)) + 0.2\hat{S}(s)$. Tool-call-aware TTL fits log-normal distributions to per-tool-type latency histories and scales retention by memory pressure.
    1. Session-affinity batching with work stealing — routes requests to the worker holding the session's KV cache unless that worker exceeds 80% load. Randomized work stealing (Blumofe–Leiserson) redistributes load with mean 230 ms migration overhead via Llumnix, narrowing worker utilization from 23–94% to 68–79%.
      1. Agent Fair Share (AFS) — defines per-tenant urgency as remaining-work-over-remaining-time ($AFS_i = \sum_{t \in \mathcal{T}_i} work_{remain}(t) / (deadline(t) - t_{now})$), allocates capacity proportionally, and preempts via KV migration. Lyapunov drift analysis proves bounded deviation from proportional completion times (Theorem 2).

        2. 核心技术壁垒: The reuse probability computation $P_{reuse}(s) = \sum_{u \in succ(v_s)} P(v_s \to u) \cdot overlap(s, u)$ is the single mechanism that closes the gap to Bélády-optimal. It requires both structural knowledge (AEG successor set and transition probabilities) and semantic knowledge (per-tool-type observation length estimates via exponential moving averages). Without it, the system degrades to prefix caching (+96% TCT, Table 4 ablation). No prior system computes this cross-tool-call reuse signal.

        §3 架构 / 方法図 #

        System architecture #

        Paper's Figure 2, verbatim (caption: "SAGA architecture. Layer 1 captures workflows ... Layer 2 routes each AEG as a single schedulable unit ... Layer 3 runs extended vLLM workers under workflow-aware LRU eviction").

        The architecture cleanly separates concerns: Layer 1 (Agent Interface) constructs AEGs from framework annotations or infers them from request patterns (87% accuracy without hints). Layer 2 (Global Scheduler) holds three coordinating engines—Affinity Router, AFS Engine, Work Stealer—sharing cluster state updated with bounded staleness of one epoch (100 ms). Layer 3 (Worker Pool) extends vLLM with WA-LRU eviction per worker. Cross-layer state is read-mostly; the only write-heavy path is AFS urgency updates.

        Agent Execution Graph structure #

        Paper's Figure 3, verbatim (caption: "Concrete AEG for a SWE-bench coding agent. Nodes are LLM inference steps; forward edges carry transition probabilities, backward edges encode retry loops").

        The AEG for a typical SWE-bench task is a 5-node linear chain with backward retry edges. Tool latencies span 53× (45 ms read_file to 2.4 s run_test), which is the regime where fixed-TTL fails and adaptive TTL is essential. Transition probabilities decay along the chain (0.95 → 0.85 → 0.70 → 0.60), reflecting increasing termination probability as the agent approaches a solution.

        Agent loop (state machine) #

        stateDiagram-v2 [*] --> Prefill: new task admitted Prefill --> Decode: KV cache populated Decode --> ToolDispatch: action ≠ "finish" Decode --> TaskComplete: action = "finish" ToolDispatch --> ToolWait: tool invoked ToolWait --> CacheCheck: tool returns observation CacheCheck --> Prefill: cache MISS (regenerate) CacheCheck --> Decode: cache HIT (append obs) ToolWait --> TTLExpiry: TTL exceeded TTLExpiry --> CacheEvict: memory pressure > threshold_low CacheEvict --> Prefill: next request arrives TaskComplete --> [*] note right of ToolWait WA-LRU retains cache TTL = P95(tool_type) × pressure_factor end note

        Planning style: The paper does not propose a new planning method; it schedules agents that follow ReAct (linear chain) or tree-of-thought (branching DAG) patterns. Decomposition is bottom-up from observed request patterns. Budget is implicit: tasks terminate when the agent emits action = "finish" or hits the benchmark's step limit. No backtracking support at the scheduler level; retry loops are captured as backward edges in the AEG.

        Memory model: Short-term memory is the KV cache (2–12 GB per session for 70B model at 32K context). Long-term memory management is outside SAGA's scope (delegated to agent frameworks). SAGA's contribution is preserving short-term memory across tool-call boundaries.

        方程物理意义 #

        WA-LRU eviction score (Eq. 1): $P_{evict}(s) = \alpha \hat{R}(s) + \beta(1 - P_{reuse}(s)) + \gamma \hat{S}(s)$ — a linear combination that approximates Bélády's "farthest future reuse" by replacing oracle knowledge with AEG-predicted reuse. The $\beta$-weighted reuse term dominates ($\beta = 0.5$), so entries with high predicted reuse survive even when stale. Sensitivity: ≤8% TCT variation across tested ranges.

        Reuse probability (Eq. 4): $P_{reuse}(s) = \sum_{u \in succ(v_s)} P(v_s \to u) \cdot overlap(s, u)$ — expected fraction of cached tokens reused in the next step, weighted by AEG transition probabilities. For linear ReAct chains, this simplifies to $(1 - p_{term}) \cdot n_{curr}/(n_{curr} + \hat{n}_{obs})$. The overlap estimation uses per-tool-type exponential moving averages.

        Lyapunov drift (Eqs. 10–14): The Lyapunov function $V(t) = \sum_i (S_i(t) - \mu_i t)^2$ measures cumulative deviation of actual service from proportional fair share. AFS creates negative drift: underserved tenants get higher urgency → more allocation → gap shrinks. The key lemma $\mathbb{E}[(a_i(t+1) - \mu_i) \cdot e_i(t)] \leq -\eta \cdot e_i(t)^2$ guarantees quadratic restoring force with $\eta = C / (N \cdot (deadline_{max} - t)^2)$.

        Competitive ratio (Theorem 3, Eq. 16): $CR_{empirical}(\text{WA-LRU}) \leq 1 + \epsilon \cdot k_{max} + O(\sigma_{tool}/TTL_{adaptive})$ — worst-case bound is extremely loose ($\approx 20.5$ for SWE-bench parameters) because it assumes all mispredictions cause maximum-length regeneration. Empirical 1.31× shows average-case behavior is far better.

        §5 实验与数据 #

        End-to-end performance (Table 3) #

        Paper's Table 3, verbatim. TCT = Task Completion Time (seconds), Mem = GPU memory utilization (%).

        SAGA achieves 203.4 s mean TCT on SWE-bench vs. 352.1 s for vLLM+APC (1.73×) and 298.4 s for KVFlow (1.47×). The closest competitor is KVFlow, which has workflow-aware eviction but lacks distributed scheduling and fairness. The gap widens on SWE-bench (longer tasks, more cache reuse opportunity) vs. WebArena (shorter tasks, higher tool-call variance).

        Ablation (Table 4) #

        Paper's Table 4, verbatim. Each row removes one component from the full system.

        Session affinity is the dominant contributor (+96% TCT when disabled), followed by workflow-aware eviction (+54%) and tool-call TTL (+42%). AFS contributes only +8% in single-tenant mode but becomes essential under multi-tenant load (§9.6). Speculative prefetching (+19%) provides diminishing returns when cache is already retained by TTL. The ablation reveals a clear importance hierarchy: routing > eviction > TTL > prefetch > fairness (for single-tenant).

        Multi-tenant fairness (Table 6) #

        Paper's Table 6, verbatim. SLO = fraction of tasks completing within 1.5× expected TCT.

        The most dramatic result: light tenants' SLO attainment jumps from 43.2% (vLLM) to 98.7% (SAGA), a 55.5 pp improvement. vLLM's FCFS scheduling causes starvation for light tenants whose short tasks get queued behind heavy tenants' long-running sessions. AFS's urgency-proportional allocation directly addresses this. The 99.1% attainment for heavy tenants (vs. 89.4% for vLLM) shows that fairness does not come at the cost of heavy-tenant performance.

        Tool & environment interface #

        SAGA intercepts requests at the agent framework level and does not modify tool execution. Tool catalog is whatever the agent framework provides (code execution, file ops, web/API calls, database queries per Table 1). All tools are opaque to SAGA—it observes only invocation time and return time. Side effects: SAGA is read-only with respect to tool execution; all writes are within the scheduling layer (cache management, routing state). Error surface: if a tool call times out beyond $TTL_{max}$ (300 s), the KV cache is evicted and must be regenerated on the next LLM call. The agent framework handles tool-level retries.

        Execution strategy tradeoff (Table 8) #

        The BFS-DFS spectrum is fundamental: pure BFS maximizes throughput (12.4 tasks/min) but evicts 78% of cache; pure DFS minimizes eviction (3%) but serializes tasks (4.2 tasks/min). SAGA's hybrid achieves optimal TCT (203.4 s) at 8.7 tasks/min—the ~30% throughput reduction vs. BFS is the explicit price paid for latency optimization.

        Parameter robustness #

        All 10 tunable parameters show <8% TCT variation under single-axis perturbation (Table 9). The most sensitive are $\beta$ (reuse weight, <8%) and $T_{idle}$ (steal trigger, <7%). Robustness is structural: session affinity (the largest contributor) is binary, and the continuous parameters enter as smoothly weighted contributions to normalized scores.

        §7 实现 cross-reference #

        Implementation details #

        8.5K lines Python + 1.2K lines C++/CUDA extending vLLM v0.6.0 (V1 engine). Four components: Workflow Analyzer, Distributed Scheduler (Ray + gRPC, P99 <5 ms), KV Cache Manager (PagedAttention + WA-LRU + TTL + prefetch on separate CUDA streams), Fairness Module (AFS computation). Runs as a standalone intercepting service—no agent code modifications required.

        [实现未公开] — code is not released as of the paper date. Implementation details are described at the design level only; no repository URL, file paths, or reproducible build instructions are provided.

        核心技术壁垒 (expanded) #

        The hard-to-replicate insight is that AEG-predicted reuse probability (Eq. 4) must be computed online, per-session, using both structural (successor set, transition probabilities from historical traces) and semantic (per-tool-type observation length EMAs) information. This requires:

        1. A framework integration layer that extracts workflow metadata from LangChain callbacks / AutoGen message logs / CrewAI hooks, or infers it from bare request sequences.
        2. Online maintenance of per-tool-type latency distributions (log-normal fit) and observation length distributions (exponential moving averages).
        3. Tight coupling between the cache manager and the scheduler: eviction decisions (Eq. 1) depend on predicted reuse (Eq. 4) which depends on AEG state managed by the scheduler.

          4. No existing system computes this cross-tool-call reuse signal. KVFlow has workflow-aware eviction but lacks the distributed scheduling and tool-type-specific TTL prediction that complete the picture.

          关键実装细节 #

          1. Contention mitigation: Thread-local update buffering with 10 ms / 100-update batched flush reduces coordinator overhead by 12× vs. per-update synchronization. Lock-free session tables use atomic CAS; 64-byte cache-line alignment avoids NUMA false sharing.
            1. Speculative prefetching on separate CUDA streams: When a node completes inference and begins tool execution, the most likely successor's prefix KV cache is prefetched asynchronously. This overlaps cache loading with tool execution at no compute cost (uses spare memory and idle CUDA streams).

              2. LLM backbone requirements #

              • Evaluated on Llama-3-70B-Instruct only (4-way tensor parallelism, GQA with $n_{kv} = 8$).
              • KV cache size scales linearly with model size: ~1.5 GB (8B) → ~10.7 GB (70B) → ~50 GB+ (405B). SAGA's benefit is proportional to cache size—larger models benefit more.
              • MoE architectures require routing-aware extensions (stated limitation).
              • No backbone sensitivity study across model families (Mistral, Qwen, DeepSeek not tested).
              • Required capabilities: standard autoregressive generation with tool-call output parsing. No special structured output or chain-of-thought format requirements from the serving layer.

              Evaluation benchmarks #

              • SWE-bench (500 verified tasks): coding agent, mean 37 steps, max 150. 2–4K prompt + 100–500 output tokens per step.
              • WebArena (812 tasks): browser agent, mean 18 steps. 4–8K prompt + 50–200 output tokens per step.
              • BurstGPT-derived: synthetic multi-tenant (10 tenants: 3 heavy / 4 medium / 3 light) with Poisson arrival at ~80% peak throughput.
              • Metrics: TCT, GPU memory utilization, throughput (tasks/min), SLO attainment (1.5× expected TCT deadline).
              • Baselines: vLLM v0.6.0, vLLM+APC v0.15.1, SGLang v0.5.8, Llumnix v1.2, TRT-LLM+Scaffolding, vLLM+KVFlow.

              Multi-agent considerations #

              The paper does not evaluate multi-agent coordination (AutoGen-style multi-agent conversations). §1.5 notes degradation on dynamic multi-agent frameworks requiring AEG re-inference per epoch. Topology, coordination, and failure isolation are not addressed.

              Production readiness #

              • Sandboxing: Tool execution is outside SAGA's scope (delegated to agent framework). SAGA itself runs as a standalone service with no direct tool access.
              • Observability: Coordinator logs session routes, AFS scores, migration events. No explicit trace logging for reproducibility of non-deterministic runs.
              • Cost controls: AFS provides per-tenant fairness; $TTL_{max} = 300$ s caps memory retention. Memory pressure (Eq. 6) provides graceful degradation.
              • Limitations for production: Single-datacenter only; no geo-distributed support. Behavior under >95% memory utilization not empirically validated (degrades to standard LRU per Eq. 6). Workflow observability degrades 12–18% TCT without framework hints.
