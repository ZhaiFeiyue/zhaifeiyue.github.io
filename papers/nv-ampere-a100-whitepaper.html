NVIDIA A100 Tensor Core GPU Architecture White Paper

Hardware   NVIDIA  ·  2020-05  ·  arXiv ↗
gpu-architecturetensor-corea100amperemigsparsity
Summary

Core Contribution

第三代Tensor Core支持全数据类型(含FP64 TC和TF32)，引入MIG多实例GPU和2:4结构化稀疏，奠定数据中心GPU标准。

Summary

A100基于TSMC 7nm工艺，die面积826mm²，集成54.2B晶体管，配备108个SM、6,912个CUDA Cores和432个第三代Tensor Core。核心创新包括：(1) TF32——一种19-bit浮点格式(8-bit exponent + 10-bit mantissa)，兼容FP32输入却能在Tensor Core上以8x FP32的速度运算，使现有FP32 DL模型无需代码修改即可获得10x V100 FP32的加速；(2) FP64 Tensor Core——首次将矩阵加速扩展到双精度浮点，为HPC科学计算(如分子动力学、气候模拟)提供19.5 TFLOPS的FP64矩阵吞吐，达V100 FP64的2.5倍；(3) 2:4结构化稀疏——硬件原生支持细粒度稀疏模式(每4个权重中2个为零)，Tensor Core吞吐直接翻倍而精度损失极小。

显存子系统方面，A100配备80GB HBM2e显存(带宽2,039 GB/s)，L2 Cache从V100的6MB大幅扩展至40MB(6.7x)，配合L2 Residency Control可将热点数据钉在L2中，显著降低HBM访问次数。每SM拥有192KB的可配置Shared Memory/L1(V100为128KB，1.5x提升)，支持更大的tile size以提升TC利用率。互联方面采用第三代NVLink，12条链路共600GB/s(V100的2倍)，支持8-GPU全互联DGX A100系统。

MIG(Multi-Instance GPU)是A100的革命性特性——可将单颗GPU硬件级隔离为最多7个独立实例，每个实例拥有独立的SM、L2 Cache切片、显存控制器和显存容量，实现QoS保证和安全隔离。这使数据中心能在同一GPU上混合运行不同规模的推理任务和小规模训练任务，大幅提升GPU利用率。

Key Findings

  • 108 SM, 6,912 CUDA Cores (64 FP32/SM), 432 Tensor Cores (4/SM, 3rd gen)
  • TF32 TC: 156 TFLOPS (312 TFLOPS with 2:4 sparsity)，无需代码修改即可加速FP32 DL workload
  • FP64 TC: 19.5 TFLOPS (2.5x V100 FP64)——首次Tensor Core支持双精度
  • FP16/BF16 TC: 312 TFLOPS (624 sparse)，2.5x V100 FP16 TC
  • INT8 TC: 624 TOPS (1,248 sparse)；INT4 TC: 1,248 TOPS (2,496 sparse)
  • 2:4 Structured Sparsity: 硬件原生支持，所有TC数据类型吞吐翻倍
  • 80GB HBM2e, 2,039 GB/s (2.27x V100)
  • 40MB L2 Cache (6.7x V100的6MB)，支持L2 Residency Control
  • 192KB Shared Memory/L1 per SM (1.5x V100的128KB)
  • MIG: 最多7个GPU Instance，硬件级隔离(SM、L2、显存独立)
  • NVLink 3: 12 links, 600GB/s total (2x V100)
  • PCIe Gen 4 x16
  • TDP: 400W (SXM4)
  • Async Copy: 直接从Global Memory到Shared Memory，绕过RF
  • 全芯片(GA100): 8 GPCs, 128 SMs, 512 TCs——A100产品启用108 SMs

Limitations

  • MIG的7-instance上限和固定分区粒度限制了灵活性，不支持任意比例划分
  • 2:4 Sparsity要求严格的结构化模式(每4元素恰好2个零)，并非所有模型/层都能轻松满足
  • INT4 TC虽有吞吐但实际AI模型很少使用INT4量化(精度损失大)
  • 单die设计(826mm²)接近光刻极限，良率压力大
  • 80GB显存在大型LLM推理场景逐渐显得不足(GPT-3 175B需多卡)
  • FP64 TC虽然是突破，但19.5 TFLOPS仍低于AMD MI250X的95.7 TFLOPS FP64 Matrix
  • NVLink 600GB/s对于AllReduce密集型大规模训练仍是瓶颈

Infrastructure Impact

  • 奠定数据中心GPU标准: A100定义了"AI加速器"的完整功能集——TC全精度覆盖、结构化稀疏、多实例隔离、大L2 Cache
  • MIG改变部署模式: 允许云厂商在同一GPU上提供不同大小的虚拟GPU实例，直接影响AWS/Azure/GCP的GPU实例产品线
  • TF32降低迁移门槛: 现有FP32训练代码无需修改即可获得显著加速，加速了从CPU到GPU的workload迁移
  • DGX A100系统: 8x A100 + 6x NVSwitch构成5 PFLOPS AI系统，成为AI训练的"标准单元"
  • 推动MLPerf基准: A100在MLPerf Training/Inference多项benchmark中长期霸榜
  • 2:4 Sparsity推动模型压缩研究: 催生了大量结构化剪枝算法研究
Details

Deep Analysis (hardware)

1. Architecture Overview

A100基于GA100 GPU，采用TSMC N7(7nm)工艺。完整的GA100 die包含8个GPC(Graphics Processing Cluster)、128个SM和512个Tensor Core，但A100产品因良率考虑禁用了部分单元，最终启用7个完整GPC + 1个部分GPC = 108个SM、432个Tensor Core。每个SM内部结构：4个Processing Block(又称Sub-partition)，每个Block包含16个FP32 CUDA Core、1个Third-Gen Tensor Core、1个Warp Scheduler和一个Dispatch Unit。A100在Ampere架构中移除了V100的INT32/FP32共享路径，改为全部64个Core都可执行FP32——因此FP32 CUDA吞吐达19.5 TFLOPS(与FP64 TC相同数值纯属巧合)。

核心变化是Tensor Core从"FP16加速器"变为"全精度矩阵引擎"——这使GPU从AI-only加速器变为AI+HPC通用矩阵加速器。

3. SM / Tensor Core Deep Dive

  • Full GA100 Die: 8 GPCs, 128 SMs, 512 Tensor Cores
  • A100 Product (SXM4): 7 GPCs + 1 partial, 108 SMs, 432 Tensor Cores
  • SM内部: 64 FP32 CUDA Cores/SM (4 blocks × 16 cores)
  • 每SM 4个Tensor Core, 每个TC在一个时钟周期内执行:
  • FP16/BF16: 256 FMA ops (= 512 FLOPS)
  • TF32: 128 FMA ops (= 256 FLOPS)
  • FP64: 32 FMA ops (= 64 FLOPS)
  • INT8: 512 ops; INT4: 1024 ops
  • 第三代TC vs 第一代 (V100):
  • V100: 每SM 8个TC，但每个TC只做4×4×4 FP16矩阵乘 (64 FMA/cycle)
  • A100: 每SM 4个TC，但每个TC重新设计，吞吐大幅提升
  • 净效果: 相同SM数下FP16 TC吞吐2.5x

4. Peak Performance Derivation

以FP16 Tensor Core为例:

  • 每TC: 256 FMA/cycle = 512 FP16 FLOPS/cycle
  • 每SM: 4 TC × 512 = 2,048 FP16 FLOPS/cycle
  • 全芯片 (108 SM): 108 × 2,048 = 221,184 FP16 FLOPS/cycle
  • Boost Clock: ~1.41 GHz
  • 峰值: 221,184 × 1.41 × 10⁹ = 311.9 TFLOPS ≈ 312 TFLOPS

TF32推导:

  • 每TC: 128 FMA/cycle = 256 TF32 FLOPS/cycle (TF32操作数更宽，吞吐减半)
  • 全芯片: 108 × 4 × 256 × 1.41 GHz = 156 TFLOPS

FP64 TC推导:

  • 每TC: 32 FMA/cycle = 64 FP64 FLOPS/cycle
  • 全芯片: 108 × 4 × 64 × 1.41 GHz = 19.5 TFLOPS

Sparsity效果: 2:4稀疏跳过零值计算，等效吞吐翻倍 → TF32 sparse: 312 TF, FP16 sparse: 624 TF

5. Memory Subsystem

  • HBM2e: 80GB, 5 stacks, 2,039 GB/s (= 2.04 TB/s)
  • V100为32GB HBM2 @ 900 GB/s，A100显存容量2.5x，带宽2.27x
  • L2 Cache: 40MB (40,960 KB)
  • V100仅6MB，A100 L2扩大6.7x
  • 分为多个slice，MIG模式下各partition获得独立L2 slice
  • L2 Residency Control: 通过cudaAccessPolicyWindow API将数据钉在L2中
  • 对于working set < 40MB的kernel，可显著减少HBM流量
  • Shared Memory / L1: 192KB per SM (可配置)
  • V100: 96KB shared + 不可配置L1 → A100: 192KB总量可配置split
  • 配置选项: 0/16/32/64/100/132/164 KB shared memory，剩余为L1
  • 更大shared memory → 更大tile size → TC利用率更高
  • Register File: 256 KB/SM (65,536 × 32-bit registers)

6. MIG Architecture

  • 最多7个GPU Instance (GI)，每个GI包含:
  • 独立的SM集合 (14 SM 最小粒度)
  • 独立的L2 Cache切片
  • 独立的Memory Controller / Memory分区
  • 独立的显存容量配额
  • 隔离级别: 完整的硬件级隔离——错误隔离、性能隔离、地址空间隔离
  • GI内可创建Compute Instance (CI)，进一步细分SM资源
  • 典型配置:
  • 1g.10gb: 1个slice (14 SM, ~10GB) — 小型推理
  • 2g.20gb: 2个slice — 中型推理
  • 3g.40gb: 3个slice — 较大推理/小训练
  • 7g.80gb: 全卡 — 训练/大推理
  • 限制: 分区方案固定，不支持任意比例划分；不同size实例可混合但有约束

7. Interconnect

  • NVLink 3.0:
  • 12条链路 (V100为6条)
  • 每链路: 50 Gbps/signal pair → 25 GB/s 单向 (50 GB/s双向)
  • 总带宽: 12 × 50 = 600 GB/s 双向
  • 支持NVSwitch v2: 实现8-GPU全互联 (DGX A100)
  • PCIe Gen 4 x16: 31.5 GB/s per direction (64 GB/s bidirectional)
  • DGX A100系统: 8x A100 + 6x NVSwitch
  • 任意GPU对之间600 GB/s
  • 系统总AI性能: 5 PFLOPS FP16 TC
  • 8x 200Gbps InfiniBand for inter-node
