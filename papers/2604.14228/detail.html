Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems

Jiacheng Liu, Xiaohan Zhao, Xinyi Shang, Zhiqiang Shen | 2026-04 | https://arxiv.org/abs/2604.14228 Category: agent | Tags: code, framework, agent-architecture, safety, context-management Read: 2026-04-20

TL;DR

首次基于源码 (v2.1.88, ~512K 行 TypeScript) 解剖生产级 AI coding agent (Claude Code) 的完整架构——揭示 "1.6% 决策逻辑 + 98.4% 确定性基础设施" 的范式, 追踪 5 values → 13 principles → implementation choices, 并通过 OpenClaw 对比展示部署上下文如何塑造 agent 架构。对所有 agent 系统设计者的必读架构参考。

Core Contribution

首次基于源码的生产级 AI coding agent 设计空间分析, 识别 5 values → 13 design principles → implementation choices 的完整映射, 揭示 minimal-scaffolding + maximal-harness 范式, 通过 OpenClaw 对比展示同一设计问题在不同部署上下文下的差异化解答。

Q1: 这篇论文试图解决什么核心痛点？

尽管 Claude Code 等 production coding agent 已被广泛使用, 但 Anthropic 只发布用户文档不发布架构描述。学术界缺少对这些系统内部架构决策的系统分析——现有工作要么关注 benchmark 评测, 要么关注 agent framework 的 API 设计, 没有人从源码级别追踪 "价值观 → 设计原则 → 实现选择" 的完整链条。

Q2: 作者提出了什么方法？

通过对 Claude Code v2.1.88 TypeScript 源码 (~1,884 文件 / ~512K 行) 的逆向工程分析, 构建三层分析框架: (1) 识别 5 个驱动架构的人类价值观; (2) 追踪至 13 条设计原则和 6 个重复性设计问题; (3) 与 OpenClaw (开源多通道 AI 助理网关) 在 6 个维度上对比。三级证据体系: Tier A (产品文档) / Tier B (代码验证) / Tier C (重构推理)。

Q3: 最终效果/结论如何？

揭示 agent 系统的核心范式是 "最小脚手架 + 最大操作基座" (1.6% 决策逻辑 vs 98.4% 基础设施), 识别出权限 (7 modes) / 扩展 (4 mechanisms) / 上下文 (5 layers) / 委派 (subagent isolation) / 持久化 (append-only JSONL) 五大子系统的设计空间, 并通过 OpenClaw 对比证明设计问题是稳定的但答案取决于部署上下文。提出 6 个未来方向。

Summary

Motivation: Production coding agent (如 Claude Code) 已从 autocomplete 进化到可自主规划、执行 shell 命令、读写文件的 agentic 系统, 但内部架构决策从未被系统分析。27% 的 Claude Code 辅助任务是 "没有此工具就不会尝试" 的全新工作, 说明架构赋能了质变而非量变。

Method: 对 Claude Code v2.1.88 的 TypeScript 源码进行设计空间分析, 按 6 个 recurring design questions 组织 (reasoning location / execution engine / safety posture / binding constraint / extensibility / delegation)。识别 5 个 human values (authority, safety, reliability, capability, adaptability) 和 13 条 design principles (deny-first, graduated trust, defense-in-depth, context-as-scarce-resource 等), 追踪至具体源码文件 (query.ts, permissions.ts, tools.ts 等)。进一步与 OpenClaw 在 6 个维度上进行架构对比。

Results: 核心发现是 agent 系统由一个极简的 while-loop (queryLoop AsyncGenerator) 加上庞大的确定性基础设施构成。具体包括: 7 层安全管道 (tool pre-filtering → deny-first rules → permission modes → auto-mode classifier → shell sandbox → no-restore-on-resume → hook interception); 4 种按 context cost 分层的扩展机制; 5 层渐进式上下文压缩管道; summary-only 返回的子代理隔离。OpenClaw 对比揭示同样的设计问题在 gateway-level 部署中产生了根本不同的答案 (perimeter access control vs per-action safety, embedded runtime vs central loop)。

Key Findings

  • Agent 系统遵循 minimal scaffolding + maximal harness 范式: 1.6% 决策逻辑 vs 98.4% 确定性基础设施
  • 5 values → 13 design principles → implementation choices 的完整追踪映射
  • 7 种 permission modes 构成单调递减安全梯度 (plan → default → acceptEdits → auto → dontAsk → bypassPermissions → bubble)
  • 4 种扩展机制按 context cost 分层: hooks(零) → skills(低) → plugins(中) → MCP(高)
  • 5 层 context compaction pipeline: budget reduction → snip → microcompact → context collapse → auto-compact
  • Subagent 采用 summary-only return + sidechain transcript + worktree isolation, 团队模式 token 消耗约 7×
  • 93% permission prompt 被批准 → approval fatigue 使交互确认不可靠 → 架构必须独立于人类警觉性维持安全
  • Agent 设计空间是分层的 (gateway-level vs task-level harness 可组合), 不是扁平分类

Deep Analysis (agent)

1. Agent Architecture

Agent type: 单代理 + 可选的 hierarchical multi-agent (parent-child delegation)。核心是单一 while-loop, 通过 AgentTool 可生成子代理, 但子代理是 subordinate workers 而非 peer agents。

Core loop: ReAct 模式的 queryLoop AsyncGenerator——model 生成 reasoning + tool_use blocks, harness 解析/权限检查/分发/收集结果, 结果反馈下一轮迭代。没有显式规划图或状态机, 不使用 backtracking 或 tree search。停止条件: 无 tool_use blocks (主要) / maxTurns / context overflow / hook intervention / explicit abort。

Memory mechanism: 两层分离设计:

  • Working memory: 对话 context window (200K-1M tokens), 受 5 层 compaction 管理
  • Long-term memory: CLAUDE.md 4 级文件层次 (managed → user → project → local) + auto-memory entries, 全部 file-based, 无 embedding index

Tool use protocol: Structured tool_use blocks in model response (Anthropic API format), 54 built-in tools (19 unconditional + 35 conditional), 通过 assembleToolPool 合并 MCP tools, buildTool factory 统一构建。Model 从不直接访问文件系统/shell/网络——必须通过 tool_use protocol, harness 验证后执行。

Tool invocation decision: Model 自主决定, harness 不约束调用顺序或选择。Model 在 tool pool 中看到所有可用 tools 的 schema, 自行判断何时/是否使用 tools vs 直接回答。ToolSearch 机制允许延迟加载, 减少 context 消耗。

2. Planning & Reasoning

Planning granularity: Plan permission mode 提供显式规划 (model 先创建计划, 用户批准后执行), 但 Claude Code 的 plan tool 被观察为 "基本上是 no-op"——其价值在于保持 agent on track 而非外部计算。没有 task decomposition 或 goal hierarchy 机制。

Backtracking/revision: 无显式 backtracking。当 permission 被拒绝时, denial reason 反馈给 model, model 修正方法并在下一轮尝试更安全的替代方案。这是 recovery-oriented 设计——permission enforcement shapes behavior rather than halting it。

Reasoning approach: Pure chain-of-thought (model 内部 reasoning), 支持 thinking configuration (extended thinking) 但 harness 不指导推理结构。不使用 ToT/GoT/self-consistency/explicit planners。

Ambiguity handling: 依赖 model 的 contextual judgment + CLAUDE.md 指令。Values-over-rules 原则: 不用刚性规则, 培养 "good judgment and sound values that can be applied contextually" (引自 Anthropic Constitution)。

3. Tool & Environment Interface

Available tools: 54 built-in tools (BashTool, FileReadTool, FileEditTool, FileWriteTool, AgentTool, SkillTool, WebFetchTool, WebSearchTool 等), 加 MCP servers 提供的外部 tools, 加 plugins 注册的 tools。Tool pool assembly 5 步: base enumeration → mode filtering → deny rule pre-filtering → MCP integration → deduplication。

Tool description format: JSON schema (Anthropic API tool_use format)。每个 tool 有 name, description, input schema (JSON Schema), maxResultSizeChars, isEnabled runtime check。MCP tools 以 mcp\_\_server\_\_tool 命名, server-level rules 匹配该 server 所有 tools。

Error handling: StreamingToolExecutor 的 sibling abort controller——任何 Bash tool 出错立即终止其他进行中的子进程。Tool results 按请求顺序缓冲和发出。Max output tokens escalation (最多 3 次恢复尝试)。Reactive compaction 当 context 接近容量时触发。prompt_too_long 错误触发 context-collapse overflow recovery。

Environment: Dynamic, partially observable——agent 只通过 tool results 观察环境, 没有全局状态视图。Context window boundary 限制了同时可见的代码库范围。

Sandboxing: 双层——permission system (逻辑层, 评估 "是否允许") + shell sandbox (隔离层, 限制文件系统和网络访问)。两者在不同轴上操作: authorization vs isolation。命令可以 permission-approved 但仍被 sandboxed。

4. LLM Backbone Requirements

Base model: Claude (Anthropic API), 支持 200K context (旧模型) 到 1M context (Claude 4.6 series)。支持 fallback model 切换 (primary 失败时)。

Fine-tuning: Auto-mode classifier 是一个独立的 LLM 调用 (用 Claude 评估 tool safety), 不需要特殊 fine-tuning。Model 通过 system prompt + CLAUDE.md 指令引导, 不需要 agent-specific SFT/RLHF。

Prompt format: System prompt (包含 output style, environment info) + user context (CLAUDE.md hierarchy, 作为 user message 而非 system instruction) + conversation history + tool results。CLAUDE.md 以 user message 传递是关键设计选择: 模型对这些指令的遵从是 probabilistic 而非 guaranteed, permission rules 提供 deterministic enforcement。

Context requirements: Context window 是 binding resource constraint——5 层 compaction pipeline + lazy CLAUDE.md loading + deferred tool schemas + summary-only subagent return 全部为了管理这个约束。

Latency sensitivity: 支持 streaming (queryLoop 是 AsyncGenerator, 边生成边 yield events)。Speculative classifier 在 BashTool 场景可 race pre-started classification vs timeout 实现即时批准。Interactive vs headless 由 surface layer 决定, core loop 相同。

5. Evaluation

Benchmarks: 论文不提供 benchmark 评测。引用了 Anthropic 内部调查 (132 engineers/researchers) 显示 27% 任务是 "无此工具不会尝试" 的新工作。引用 auto-mode 分析 (93% approval rate)。引用 longitudinal autonomy 数据 (auto-approve 从 20%→40%+ over 750 sessions)。

External evidence: 引用 Cursor 采用的 807 repo 因果分析 (代码复杂度 +40.7%, 初始速度飙升月三归基线); 304,000 AI-authored commits 审计 (约 1/4 AI 引入问题持续到最新版本); 16 开发者 RCT (AI 工具使 19% 变慢, 尽管 perceived +20%); EEG 研究 (LLM 用户显示持续的 neural connectivity 弱化)。

Failure analysis: 识别出 14 种 agent failure modes (system-design, inter-agent misalignment, task verification), 78% AI failures invisible, quality (非 cost) 是 production 首要障碍。

6. Multi-Agent (applicable)

Agent count: 6 种 built-in subagent types (Explore, Plan, General-purpose, Claude Code Guide, Verification, Statusline-setup) + custom agents via .claude/agents/*.md + plugin-contributed agents。

Communication protocol: Summary-only return——子代理只返回最终 response text + metadata 给父级, 完整对话 history 永远不进入父级 context window。Sidechain transcript (.jsonl + .meta.json) 独立存储。

Coordination: Parent-child hierarchy (非 peer-based), Agent tool 在 main agent loop 中作为一个 tool 被调用。Multi-instance coordination (agent teams) 使用 file locking (非 message broker), tasks 通过 lock-file mutual exclusion 从 shared list 认领。

Isolation modes: Worktree (git worktree, filesystem-level separation, 零外部依赖), Remote (internal-only, 远程环境), In-process (default, 共享文件系统但隔离对话 context)。

Overhead: Agent teams 消耗约 7× standard session tokens (plan mode), 使 summary-only return 更加关键。每个 subagent 的 21 参数 lifecycle (runAgent.ts) 覆盖 agent definition, prompts, permissions, tools, model settings, isolation, callbacks。

8. Production Readiness

Reliability: 论文不直接报告成功率, 但文档了多种 recovery 机制: max output token escalation (3 attempts), reactive compaction, prompt-too-long handling (context-collapse → reactive compact → terminate), streaming fallback, fallback model。

Safety: 7 层独立安全机制 (tool pre-filtering → deny-first rules → permission modes → auto-mode classifier → shell sandbox → no-restore-on-resume → hook interception)。93% approval rate 暴露 approval fatigue 问题, 架构通过 sandboxing 将 permission prompts 减少约 84%。已知漏洞: 2 个 pre-trust initialization ordering CVE (hooks/MCP 在 trust dialog 前执行)。

Observability: Append-only JSONL session transcripts 提供完整审计轨迹。27 种 hook events 提供 lifecycle 可观测性。但 context compaction 对用户不可见 (context collapse 是 read-time projection, 用户无法检查什么被丢失)。

Scalability: 单进程架构, multi-instance 通过 file locking 协调。无 distributed coordination service。Agent teams 的 7× token 消耗限制了并行子代理数量。

9. Impact on AI Infra

LLM serving 影响: Agent sessions 是 long-lived, stateful, multi-turn——与传统 stateless completion 有根本区别。5 层 compaction pipeline 揭示了 serving 系统需要: (1) prompt cache awareness (microcompact 的 cache-aware 路径); (2) very long context support (200K-1M tokens); (3) streaming tool dispatch (边生成边执行); (4) abort/recovery semantics (sibling abort, reactive compact)。

Model capability push: 推动更长 context window (减少 compaction 压力), 更好的 tool_use format compliance, 更强的 contextual judgment (values-over-rules 需要模型自身有好的判断力), 更好的 cache compatibility (compaction 不应 invalidate prompt cache)。

Kernel requirements: Interactive tool use 需要快速 decoding (streaming 响应), 低延迟 tool_use block parsing, 可能需要 speculative execution (PASTE 式 tool 预执行)。

Production deployment: 需要 CLI/IDE/SDK 多入口 surface, 统一的 query loop, 可配置的 permission system, MCP server 基础设施, 文件系统访问 (CLAUDE.md + transcripts), 可选的 shell sandbox, git worktree 支持 (subagent isolation)。

Logic Flow (逻辑故事还原)

时代定位 (Era Positioning)

2025-2026 年 AI coding agents 从 autocomplete 进入 fully agentic 阶段——agent 不再只是建议代码, 而是自主规划、执行 shell、读写文件、迭代输出。GitHub Copilot → Cursor → Claude Code 的演进代表了从 suggestion 到 autonomous action 的范式转移。27% 的 Claude Code 辅助任务是用户 "没有工具就不会尝试" 的全新工作, 说明架构已赋能质变。然而, 尽管广泛使用, 这些系统的内部架构从未被系统分析——学术界关注 benchmark 和 framework API, 忽略了 "价值观如何变成代码" 的链条。

背景 (Context)

现有 agent 系统分析方法的缺陷: (1) Benchmark 导向 (SWE-Bench, HumanEval) 只看结果不看机制; (2) Framework 导向 (LangGraph, AutoGen) 关注 API 设计不关注 production 实现; (3) Product 导向 (用户文档) 只说能做什么不说怎么做。缺少从源码级别追踪 "为什么这样设计而不是那样设计" 的分析。

约束推导 (Constraint Derivation)

核心设计问题: "在安全、可靠、可扩展的约束下, 如何让 agent 最大化有效自主行动?"

替代方案为何不可
显式规划图 (LangGraph-style state machines)约束模型推理——越强的模型越不需要推理脚手架, 决策图反而限制了模型的 contextual judgment
单一扩展机制 (unified tool API)不同扩展有不同 context cost——zero-cost event handler 和 schema-heavy tool server 无法共享一个 API 而不迫使 extension authors 做不必要的 trade-off
单一安全边界 (Docker-only like SWE-Agent)提供 environment-level 隔离但无法 per-action 精细控制——当需要 "npm test 允许, rm -rf / 拒绝" 时, container 粒度太粗
Git-based rollback (Aider-style)只能恢复文件系统变更, 无法回滚网络请求/API 调用/数据库操作——安全 must be preventive, not just recoverable
单一 compaction 策略截断丢信息, summarization 丢细节, cache-aware 太激进——需要 graduated pipeline 在 cost-benefit 连续谱上操作

破局 (Insight)

核心 insight: "模型推理, harness 执行" (Model reasons, harness enforces)。将决策权交给 LLM, 但用庞大的确定性基础设施 (98.4%) 创造让模型 "能做出好决策的条件"。这不是 "信任模型" 也不是 "限制模型", 而是 "为模型创造可信赖的操作环境"。

类比: 传统 agent framework 像 "给飞行员写详细的飞行手册并要求逐条执行", Claude Code 像 "给飞行员一架仪表完备的飞机——飞行员自己判断怎么飞, 但仪表(harness)确保他能看到所有必要信息, 自动驾驶(recovery)在紧急时接管, 安全系统(permissions)在危险操作前询问"。

作者证明

No formal 作者证明 — empirical/analytical architecture analysis. 本文不提出新算法或数学模型, 而是一篇架构分析论文。其 "证明" 机制是:

  1. Source code evidence (Tier B): 所有架构 claim 直接引用 TypeScript 源文件和函数名 (query.ts, permissions.ts, tools.ts 等)
  2. Design-space analysis methodology: 识别设计问题 → 列举替代方案 → 追踪 Claude Code 的具体选择 → 用 OpenClaw 对比校准
  3. Three-tier evidence system: Tier A (product docs) / Tier B (code-verified) / Tier C (reconstructed), 每个 claim 明确标注证据级别
  4. External empirical validation: 引用 4 个独立研究 (Cursor 807-repo analysis, 304K commits audit, 16-developer RCT, EEG study) 佐证架构预测

    5. 可攻击面: (1) 逆向工程只能揭示 "是什么" 不能确认 "为什么"——design intent 可能与实现不同; (2) v2.1.88 是静态快照, feature flags 可能在生产中 flip; (3) 1.6% vs 98.4% 的比例来自社区分析 (Tier C) 而非 Anthropic 官方确认

    核心技术壁垒 (Core Technical Barrier)

    五层上下文压缩管道 (5-layer context compaction pipeline) 是整个系统最 sophisticated 的工程:

    • Budget reduction (per-tool-result size limits, 豁免机制, 恢复引用) → Snip (轻量老历史修剪, token 计数需要 explicit 传递因为 usage field 幸存 snip) → Microcompact (时间路径 always on + cache-aware 路径 gated, deferred boundary marker 使用 actual cache_deleted_input_tokens) → Context collapse (read-time virtual projection, "nothing is yielded; the collapsed view is a read-time projection over the REPL's full history") → Auto-compact (full model-generated summary via compactConversation, 只在前 4 层不够时触发)

    这个 pipeline 的壁垒在于 5 层之间的交互复杂性: snip 的 token savings 对 auto-compact 不可见需要 explicit 传递; microcompact 的 boundary messages 需要 defer 到 API 响应后才能获得 actual cache 数据; context collapse 不修改存储只修改视图, 需要与 persist/resume 协调。重现这个管道需要深刻理解 prompt caching 的经济学和 append-only transcript 的不变量。

    质疑假设 (Challenge Assumptions)

    1. "1.6% decision logic" 的定义 (攻击 Step 3 premise): 这个比例来自社区分析 (Tier C), 不是 Anthropic 官方数据。"Decision logic" 的边界如何划定? System prompt 构造算不算? Permission rule 配置算不算? 如果把 "影响模型决策的代码" 都算进去, 这个比例可能显著更高。
      1. "Minimal scaffolding" 的因果方向 (攻击 Step 3 conclusion): 这是 Anthropic 的设计选择还是 Claude 模型能力的结果? 如果换成弱模型 (如 Llama-8B), minimal scaffolding 是否仍然 work? 论文假设越强的模型越不需要 scaffolding, 但这个假设没有实验验证。
        1. OpenClaw 对比的代表性 (攻击 Step 4 premise): OpenClaw 是一个 multi-channel personal assistant gateway, 与 coding agent 的差异是否太大以至于对比不够 informative? 为什么不对比 SWE-Agent, OpenHands, Aider 等更直接的 competitor?
          2. 生态现状: SWE-Agent/OpenHands 的源码也是公开的, 对比它们可能更有架构参考价值。论文选择 OpenClaw 的原因是要展示 "部署上下文如何改变答案", 但这个选择削弱了 within-category 的精细对比。

          设计绑定批判 (Design Binding Critique)

          1. 绑定 Anthropic Claude 模型 (攻击 Step 3 load-bearing): 整个 values-over-rules 哲学假设模型有 "good judgment"——这直接依赖 Claude 的 alignment 质量。换成开源模型 (Llama, Qwen) 是否需要更多 scaffolding? 如果需要, "minimal scaffolding" 不是通用范式而是 Anthropic 特定的设计点。
            2. 生态现状 (2026-04): Claude 在 coding agent 领域仍是 top-tier, 但 GPT-4.1, Gemini 2.5 Pro 也在竞争。Codex CLI (OpenAI) 的架构可能更 scaffolding-heavy, 这支持 "minimal scaffolding 依赖模型能力" 的假设。短期 bet 安全。
            1. 绑定 context window 作为 binding constraint: 如果 context window 扩展到 10M+ tokens (Gemini 方向), 5 层 compaction pipeline 的必要性大幅降低, 大部分工程投资可能被 bypass。论文自己在 §11.6 承认 "longer context windows would reduce compaction pressure, potentially simplifying the graduated pipeline"。
              1. 绑定 single-repository session scope: Claude Code 是 per-repo ephemeral CLI process, 所有设计假设围绕 "单仓库单会话"。Cross-repo, multi-project, organizational-level agent 需要完全不同的设计 (更接近 OpenClaw 的 gateway 模式)。

                2. 拆解 (Deconstruction)

                1. User 通过任一 interface (CLI/IDE/SDK) 提交 prompt
                2. Prompt 进入 queryLoop → context assembly (6 层来源) → 5 层 pre-model context shapers
                3. Model call (streaming) → 返回 text + tool_use blocks
                4. 如有 tool_use: StreamingToolExecutor 或 runTools 处理 → 每个 tool request 过 permission pipeline (7 层)
                5. Tool results → append to conversation → loop continues
                6. 如无 tool_use: turn complete → yield final response
                7. State 持久化: append-only JSONL transcript, sidechain for subagents

                  8. 实践上下文 (Deployment Context)

                  • Inference stage: 完全在 decode/generation 阶段, 每个 turn 是一次完整的 LLM call (prefill context + decode response)
                  • Context window: 200K-1M tokens, 是 binding constraint
                  • Compute-bound vs memory-bound: Long context prefill 是 compute-bound, 但 interactive decoding 需要低延迟 (memory-bound)
                  • GPU architectures: 需要大 HBM 支持 long KV cache (MI300X/H100/B200), 但 production 通过 API 调用 Anthropic 云
                  • Framework integration: Claude Code 是 standalone CLI, 但通过 MCP 与外部系统集成; OpenClaw 可将 Claude Code 作为 external harness 组合

                  生态影响追踪 (Ecosystem Influence)

                  • MCP 协议: Claude Code 是 MCP 的旗舰应用, MCP 已捐赠给 Linux Foundation AI & Data Foundation, 成为 de facto 标准 (thousands of community servers across 26+ major directories)
                  • Agent SDK pattern: Claude Code 的 queryLoop AsyncGenerator 模式影响了 OpenAI Agents SDK 的设计方向
                  • CLAUDE.md / AGENTS.md pattern: 文件基础的 agent 指令层次被 Cursor (Rules), Windsurf 等竞品借鉴
                  • Subagent delegation: AgentTool + worktree isolation 模式被 coding agent 社区广泛讨论
                  • Open questions influence: 论文提出的 6 个未来方向 (observability gap, cross-session persistence, harness evolution, horizon scaling, governance, human capability preservation) 为后续研究设定了议题
