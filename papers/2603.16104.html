L3 Per-Paper Synthesis: Helium (2603.16104) #

Efficient LLM Serving for Agentic Workflows: A Data Systems Perspective vs 8 peers in category framework

1. 相关论文 #

Helium 的核心命题——将 batch agentic workflow 视为 query plan 并施以 DB 式全局优化——横跨三个问题面：(1) agentic workload 下的 KV-cache 管理与调度，(2) PD 分离架构中的传输/调度协同，(3) serving 框架的执行模型设计。以下 8 个 entity 分别从不同切面与之对话。

2. 本篇 vs 相关论文的 delta #

2.1 Helium 的独特贡献 #

Helium 是目前唯一将完整 DB query optimization pipeline（parse → logical rewrite → cost-based scheduling → execute）映射到 agentic LLM serving 的工作。其四项核心 delta：

Delta 1: Templated Radix Tree (TRT) — 双层前缀的统一表示

所有 peer 系统中没有任何一个同时表达 static prompt prefix hierarchy 与 inter-operator dependency。SGLang 的 RadixCache 仅追踪 token-level prefix [2603.16104]，KVFlow 的 Agent Step Graph 追踪 agent 间依赖但不建模 prefix hierarchy，Parrot 的 Semantic Variables 建模单 call 内依赖但不跨 agent。TRT 将这两个维度统一到一棵树中，且以 template 粒度（workflow 结构）而非 token 粒度（request 内容）建索引——metadata 仅 552 KiB vs SGLang 的 14.8 MiB（27× 缩减）[2603.16104]。

Delta 2: Nested-sequence schedule — 同时优化两层 cache locality

Helium 的 nested-sequence schedule（inner sequence = query-by-query 复用 per-query context prefix；outer sequence = operator-by-operator 复用 static system prompt）在设计上优于所有 peer 的单层 prefix 调度。LSPF（SGLang 式在线 prefix sorting）在 MILP 对比中有 14.5% optimality gap，Helium 仅 0.9% [2603.16104]。PPD 的 scoring function 优化 TTFT/TPOT trade-off 但不建模 prefix 层级 [2603.13358]。DynaServe 风格的 P/D 分割也不涉及 workflow 内部的 prefix 依赖。

Delta 3: Logical plan rewriting (CSE + CacheFetch) — 从根源消除冗余

Helium 的 plan pruning（common-subgraph elimination + dead-code elimination）在 ablation 中贡献最大的 23.4% [2603.16104]。这种 DB 式 logical rewrite 在 LLM serving 领域没有先例——其他系统只能 cache 计算结果（被动），Helium 可以在 DAG 层面消除整个 operator（主动）。CacheFetch 将 deterministic operator 的输出直接替换为缓存查找，跳过 LLM call 本身。

Delta 4: 从 reactive 到 proactive 的范式转变

DualPath 的 KV-cache prefetch 是 reactive（请求到达时才加载），Helium 的 proactive KV caching 是 offline 的（第一次执行时预计算并 pin 住 static prefix）。PrfaaS 的层间 pipelining 也是 per-request reactive [2604.15039]。Helium 利用 batch workflow 的结构重复性（同一 DAG 跑不同 query batch），将机会主义的 prefix reuse 转化为确定性的 prefix reuse [2603.16104]。

2.3 增量 vs 范式 #

Helium 是 范式引入者（将 DB query optimization 映射到 LLM serving），而非已有范式的增量改进。但其范式本身建立在强假设上（batch、deterministic、DAG-expressible、single LLM），这使得适用面受限。对比 DualPath 和 PPD 均基于更弱假设工作（DualPath 只需 PD 分离 + RDMA，PPD 只需 vLLM disaggregated serving），Helium 的方法论贡献大于工程适用性。

3. 可攻击面 #

3.1 "高 hit rate ≠ 快执行" 的论据反转 #

Helium 声称 vLLM 在 Qwen3-14B Trading 上 "matches Helium's 42.9% hit rate but is 38× slower" [2603.16104]，以此论证 scheduling 比 caching 更重要。但这一论据的参照基线是 sequential execution under vLLM（逐个 query 串行），不是 vLLM 的最佳实践。vLLM 在生产环境中使用 continuous batching + 并发请求，不会串行执行 80 个 query。如果将 vLLM 的基线从串行替换为合理的并发（如 LangGraph 使用的 asyncio 并发发送），38× 的差距将大幅缩小——Table 2 中 LangGraph 仅比 Helium 慢 1.28× 平均即为佐证。论文以最弱基线（sequential vLLM）放大差距、以次强基线（KVFlow 1.56×）展示真实优势，存在 framing bias。

3.2 MILP 验证的规模缩水 #

Helium 声称其 scheduler "within 0.9% of MILP optimum" [2603.16104]，但 MILP 求解仅在 "scaled-down configs"（2–4 agents × 2–4 queries）上完成，6 小时 timeout 才覆盖 7 个配置点。在 88-operator Trading workflow 的完整规模下，MILP 无法求解——因此 Helium 在真实工作负载上的 optimality gap 未知。0.9% 这个数字仅在 toy-scale 下成立，可能随规模增长恶化（NP-hard 问题的近似比通常与实例规模相关）。论文自身承认这一限制但未量化 scaling behavior。

3.3 Determinism 假设与生产现实的矛盾 #

Helium 的 prompt cache（贡献 13.56%）要求 greedy sampling (temperature=0) [2603.16104]。但主流 agentic 框架（ReAct、Tree-of-Thought、GoT）大量使用 temperature>0 以实现探索性推理。在 sampling 场景下：

  • Prompt cache 完全失效（−13.56%）
  • Plan pruning 的 CSE 也部分失效（non-deterministic output 使相同 input 的不同 run 产生不同结果，subgraph hash 碰撞率上升）
  • 只剩 proactive KV caching（3.55%）和 cache-aware scheduling（17.66%）

综合来看，sampling 场景下 Helium 的实际优势可能从 1.56× over KVFlow 缩水到 ~1.2×，与 LangGraph（1.28×）的差距进一步收窄。论文在全文 greedy sampling 设置下评估，回避了这一关键场景。

3.4 DAG 抽象的表达力限制 #

Helium 要求 workflow 在执行前被表达为静态 DAG [2603.16104]。但 agentic workflow 的本质特征之一是 runtime-dependent control flow——ReAct 的 action 分支取决于 LLM 输出、条件 loop 的迭代次数取决于中间结果、dynamic fan-out 的并行度取决于数据。论文在 §9 诚实承认了这一点并 gesture at "TensorFlow-style control-flow primitives"，但未提供任何解决方案或 feasibility 分析。

对比：PPD 的 per-request scoring function 天然适应动态 turn 数（每轮独立决策）[2603.13358]。DualPath 的 adaptive request scheduler 基于运行时 queue 状态（三级分类 + token count proxy），无需预先知道 workflow 结构 [2602.21548]。Helium 的 DAG 假设使其无法处理这两个系统可以处理的动态场景。

3.5 内存压力下的性能退化 #

Helium 在 Qwen3-14B 上的优势明显小于 Qwen3-8B [2603.16104]——pinned proactive KV prefixes 与 active requests 竞争 HBM，导致 LRU eviction 更频繁。这一趋势在大模型（70B+）上只会恶化。DualPath 的 layerwise prefill 通过逐层流式处理将 HBM 需求降至单层 KV [2602.21548]，PrfaaS 的跨 DC offload 将 prefill 负载从 decode 节点完全卸载 [2604.15039]——两者在内存约束下均比 Helium 的 pin-in-HBM 策略更可持续。

4. 生态位 #

4.1 范式定位 #

Helium 代表了 LLM serving 领域一个新范式方向的萌芽：将 agentic workflow 视为数据库 query plan 并施以系统化优化。这是一个从 "per-call reactive" 到 "per-workflow proactive" 的质变，与 DualPath 等系统的 "per-request adaptive" 形成对比。

但这一范式目前处于 概念验证阶段。纵览 8 个 peer：

  • 已投入生产的：DualPath（DeepSeek 内部 [2602.21548]）、TensorHub（ByteDance 生产 [2604.09107]）、TileRT（GLM-5/5.1 上线 [tilert-speed-scaling-law]）
  • 有生产级设计但未部署的：PrfaaS（Moonshot AI [2604.15039]）、MFS（需 switch 配置 [2603.17456]）
  • 学术原型：Helium（helium_demo）、PPD、KVServe、ZeRO-Prefill

Helium 与 DualPath/TensorHub/TileRT 的生产成熟度差距最大。

4.2 采纳路径分析 #

Helium 的采纳面临三重阻力：

  1. DSL 迁移成本：用户必须用 Helium 的 Python DSL（helium.graphs.build(...).compile(...)）重写 workflow，不兼容 LangGraph/AgentScope/AutoGen。对比 PPD 继承 vLLM 的 OpenAI-compatible API（零迁移）[2603.13358]、KVServe 的 vLLM external connector（zero-fork）[kvserve]，Helium 的采纳门槛远高于 peers。
    1. Closed-world 假设：不支持 remote tool call、model heterogeneity、dynamic control flow [2603.16104]。真实 agentic 系统（如 LangGraph + 外部 API + 多模型 router）无法直接适配。
      1. 单节点限制：2×H100 单机——对于需要大规模 serving 的场景不可用。DualPath 已验证 1152 GPU 扩展 [2602.21548]，PrfaaS 已论证跨 DC 可行性 [2604.15039]。

        2. 4.3 最可能的影响路径 #

        Helium 的最大价值不在于系统本身被采纳，而在于其方法论被吸收：

        • TRT 数据结构 → SGLang/vLLM 的 prefix cache 可以从 per-token radix tree 扩展为 template-aware radix tree，在 workflow hint API 的辅助下实现确定性 prefix pinning
        • CacheFetch / plan pruning → 上层编排框架（LangGraph v2、AgentScope）如果暴露 DAG 结构给下层 serving engine，CSE 可以作为 middleware 优化 pass
        • Nested-sequence schedule → cost-based scheduling 可以被其他 workflow-aware 系统（KVFlow、Halo）吸收为调度策略之一
        • MILP-validated cost model → 作为后续工作的 benchmark baseline（尽管只在 toy-scale 验证）

        同实验室的 FlowMesh (2510.26913) 和 Halo (2509.02121) 与 Helium 共享核心作者，很可能已在内部复用 TRT 和 nested-sequence schedule 的思路。

        5. 未探索方向 #

        5.1 Helium + PD Disaggregation #

        Helium 在单一 vLLM engine 内运行，完全不涉及 PD 分离。将 Helium 的 workflow-level scheduling 叠加到 PD 分离架构上（如 PPD 的 dynamic routing 或 DualPath 的 dual-path loading）是一个未被探索的高价值组合。具体地：TRT 可以指导哪些 workflow operator 应该 batch 在同一 prefill engine 上（prefix 共享最大化），PPD 的 scoring function 可以决定 multi-turn 请求在 prefill vs decode 节点上的路由——两者的 scheduling 决策在不同粒度上互补。

        5.2 Dynamic DAG + Partial Plan Rewriting #

        Helium 的 DAG 必须在执行前静态确定 [2603.16104]，但 agentic workflow 天然包含 runtime-dependent 分支。一个混合方向：允许 DAG 中的 "conditional subgraph" 在运行时展开，已执行部分的 TRT 和 schedule 保持不变，新展开部分做增量 plan rewriting + re-scheduling。这类似于数据库的 adaptive query execution（如 Spark AQE），但目前没有 LLM serving 系统尝试过。

        5.3 Workflow-aware KV Compression #

        Helium 的 TRT 精确知道哪些 prefix 是 static（跨 batch 不变）、哪些是 per-query context（跨 query 不同）。这一信息可以指导 KVServe 式的 KV 压缩策略：static prefix 的 KV cache 应高精度保持（被频繁复用，压缩/解压开销被放大）；per-query dynamic portion 的 KV 可以更激进地压缩（仅用一次）。当前 KVServe 的 head-importance-based 压缩 [kvserve] 与 Helium 的 workflow-position-based 压缩在正交维度上，两者的交叉优化空间未被探索。

        5.4 Proactive Caching + Weight Streaming for MoE Agentic Workflows #

        ZeRO-Prefill 的 AsyncEP 证明 MoE expert weights 可以在 prefill compute window 中被后台 AllGather [2605.02960]。Helium 的 proactive KV caching 证明 static prefix 可以预计算并 pin 住。对于 MoE 模型上的 agentic workflow，一个未被探索的方向是：Helium 的 TRT 提供 workflow 结构信息 → 预测每个 operator 将触发哪些 expert（基于 routing 历史 profiling）→ ZeRO-Prefill 式 weight prefetch 与 Helium 式 KV prefix prefill 在同一 GPU 上双流水线执行——同时消除 expert 通信冗余和 prefix 重算冗余。

        5.5 Cross-node Helium via DualPath's Dual-path Loading #

        Helium 的最大部署限制是单机 [2603.16104]。如果将 Helium 扩展到多机，TRT 的 proactive KV caching 需要跨节点 KV 分发——此时 DualPath 的 dual-path loading（聚合所有节点的 SNIC 带宽 [2602.21548]）和 CNIC-centric traffic isolation 可以作为 transport layer。但需要解决一个新问题：Helium 的 nested-sequence schedule 假设所有 worker 共享同一 KV cache 地址空间（vLLM 内 IPC），跨节点需要 distributed cache coherence 或 TRT-guided prefetch plan。这连接了 Helium（scheduling）、DualPath（transport）和 PrfaaS（cross-DC 调度）三篇的 insight，是一个有野心但技术上可行的综合方向。
