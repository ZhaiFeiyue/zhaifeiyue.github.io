DualPath: Breaking the Storage Bandwidth Bottleneck in Agentic LLM Inference

Framework   Yongtong Wu, Shaoyuan Chen, Yinmin Zhong, Rilin Huang, Yixuan Tan  ·  2026-02  ·  arXiv ↗

粗读 Overview

Core Contribution

DualPath 通过引入 dual-path KV-Cache loading（让 decode engine 的空闲 storage NIC 参与 KV-Cache 加载，经 RDMA 计算网络转发至 prefill engine），打破了 PD 分离架构下 prefill 端存储带宽的单点瓶颈，在 agentic workload 下实现了最高 1.87× 的离线推理吞吐提升和平均 1.96× 的在线服务吞吐提升。

核心三问

Q1: 这篇论文试图解决什么核心痛点？

在 agentic LLM 推理中，多轮对话导致 KV-Cache 命中率极高（≥95%），prefill engine 需要从远端存储大量加载 KV-Cache，而其 storage NIC 带宽成为系统瓶颈。与此同时，decode engine 的 storage NIC 几乎完全空闲，造成严重的带宽利用不均。这个问题在 DeepSeek-V3 级别模型上尤为突出——cache-compute ratio 达到 22 GB/PFLOP，GPU 大部分时间在等 I/O。

Q2: 作者提出了什么新的"杀手锏"方法？

DualPath 打破了"KV-Cache 只能由 prefill engine 从存储直接加载"的思维定式，增加了一条 storage → decode engine → (RDMA) → prefill engine 的新数据通路。配合 CNIC-centric traffic management（通过 InfiniBand VL 隔离 KV-Cache 流量与模型推理通信）和 adaptive scheduler（动态平衡两条路径的负载），将存储 I/O 从单点瓶颈转化为全局可调度资源池。

Q3: 最终效果如何？

在 DeepSeek-V3.2 660B 上，离线推理吞吐提升最高 1.87×，在线服务吞吐平均提升 1.96×（不违反 SLO）。大规模实验（1152 GPU）验证了近线性扩展能力：48P96D 配置下 JCT 与 2P4D 几乎持平（3201s vs 3167s），调度器 CPU 开销低于 10 核。

逻辑故事还原

背景：

现有 PD 分离推理架构中，prefill engine 独占所有 KV-Cache 存储读取——这在传统短对话场景下尚可接受，但 agentic workload 彻底改变了游戏规则。Agent 动辄 100+ 轮交互，上下文累积至数万 token，每轮只追加几百 token，KV-Cache 命中率高达 98.7%。这意味着 prefill 的工作从"计算"变成了"搬数据"，而 storage NIC 带宽增长远跟不上 GPU 算力增长（Ampere 到 Blackwell，I/O-compute ratio 下降 14.4×）。Mooncake 用 DRAM pool 缓解，但 DRAM 成本高、RL training 时 DRAM 被 offloaded state 占用。

破局：

核心洞察类似于"双车道高速公路"——如果一条车道（PE storage NIC）堵死了，而旁边另一条车道（DE storage NIC）完全空闲，为什么不让货物（KV-Cache）先走空闲车道，再通过立交桥（RDMA 计算网络）转到需要的地方？计算网络带宽远大于存储网络（每 GPU 一张 400Gbps CNIC vs 每节点一张 400Gbps SNIC），且模型推理的集合通信是脉冲式的（sub-millisecond 级别），有大量空闲窗口可以利用。

拆解：

1. Dual-path data path：PE read path（传统路径：storage → PE buffer → PE HBM）和 DE read path（新路径：storage → DE buffer → RDMA → PE HBM），两条路径逐层流水线化与计算重叠

2. CNIC-centric traffic management：所有 GPU 进出数据都走 CNIC（包括本地 H2D/D2H），通过 InfiniBand Virtual Lane 的 WRR 策略将模型推理通信分配 ~99% 带宽优先级，KV-Cache 流量走低优先级通道

3. Adaptive scheduler：三级分类调度（overloaded / short-queue / long-queue），PE 调度以 token count 为 proxy 平衡 GPU 负载和 NIC 负载，DE 调度两阶段（组间平衡 + 组内 HBM 感知），KV-Cache 读取路径选择基于 disk reading queue 长度

Key Details & Takeaway

1. CNIC-centric data path 是生产级的关键决策：所有 GPU 数据传输（包括本地 H2D/D2H）都走 CNIC + RDMA Write，而非 cudaMemcpyAsync 或 GPUDirect Storage。原因是现有 GPU 不支持 PCIe QoS，无法隔离 KV-Cache 流量与模型推理的集合通信。RDMA work submission 仅 ~1μs（vs cudaMemcpyAsync 的 5-7μs），且可以 doorbell batching 进一步摊薄。这个设计"看似绕路实则唯一可行"，体现了 DeepSeek 在大规模部署中的工程经验。

2. Bottleneck-free 条件推导：论文给出了完整的带宽分析（Eq. 1-9），在 g=8, s=1 的典型配置下（8 GPU/node，1 SNIC/node），bottleneck-free P/D 比范围为 1/7 ≤ P/D ≤ 7/2，覆盖了几乎所有实际配置。这个分析让 dual-path 从"看起来能work"提升到"可以证明不引入新瓶颈"。

一句话总结：DualPath 通过"让空闲的 decode engine storage NIC 帮忙读 KV-Cache 再走 RDMA 转发"这个简洁洞察，将 PD 分离架构中的存储带宽单点瓶颈转化为全局可调度资源，是 DeepSeek 内部 agentic RL training infrastructure 的核心优化之一。

Summary

DualPath 解决的是 PD 分离架构下 agentic LLM 推理的核心性能瓶颈：prefill engine 的 storage NIC 带宽饱和。在 agentic workload 中，多轮交互导致上下文持续累积（平均 32.7K tokens），每轮仅追加少量 token（~429），KV-Cache 命中率高达 98.7%。这意味着 prefill 阶段的工作从计算密集转变为 I/O 密集——GPU 算力严重浪费在等待 KV-Cache 加载上。与此同时，decode engine 的 storage NIC 却几乎完全空闲。

DualPath 的核心创新是引入 dual-path KV-Cache loading：除了传统的 storage → prefill engine 路径，新增 storage → decode engine → RDMA → prefill engine 路径。这聚合了整个集群的 storage NIC 带宽，消除了不对称瓶颈。为了避免新数据路径干扰模型推理的集合通信（如 EP AllToAll），DualPath 采用 CNIC-centric traffic management，将所有 GPU 数据传输统一走 CNIC，利用 InfiniBand VL 机制实现严格的流量隔离（模型通信 ~99% 优先级）。配合 adaptive scheduler 在 token count、disk queue length、HBM capacity 三个维度动态平衡负载，系统实现了高效的资源利用。

实验基于 DeepSeek 真实生产环境（3FS 分布式存储 + InfiniBand），在三个模型上验证：DS 660B 离线推理最高 1.87× 加速，在线服务平均 1.96× 吞吐提升，大规模实验（1152 GPU）验证了近线性扩展。值得注意的是，这是 DeepSeek 内部 inference framework 的工程实践总结，而非纯学术系统——论文使用的 3FS、FlashMLA、DeepGEMM、DeepEP 全部来自 DeepSeek 生态。

Key Findings

  • 存储 I/O 已成为 agentic LLM 推理的主导瓶颈：在 98.7% KV-Cache 命中率下，DeepSeek-V3.2 的 cache-compute ratio 达到 22 GB/PFLOP，GPU 算力被 storage NIC 带宽限制而严重浪费
  • PD 分离架构存在固有的带宽不均：prefill engine 的 SNIC 持续饱和，decode engine 的 SNIC 几乎空闲——简单增加 PE 端带宽成本高且不现实
  • Dual-path loading 可以将存储带宽从单点瓶颈转化为全局可调度资源，在 1/7 ≤ P/D ≤ 7/2 的广泛配置范围内不引入新瓶颈
  • 流量隔离是 dual-path 可行的前提条件：通过 CNIC-centric data path + InfiniBand VL QoS，KV-Cache 流量对模型推理通信的影响几乎为零
  • Ablation 证明 dual-path loading 是最大贡献者（-38.19% JCT），layerwise prefill（-17.21%）和 scheduling（-45.62% 总计）分别是前置条件和锦上添花
  • 系统具备生产级可扩展性：48P96D（1152 GPU）配置下 JCT 仅比 2P4D 增长 1.07%，调度器 CPU 开销 <10 核

Limitations

  • 存储后端强绑定 3FS：DualPath 的评估完全基于 DeepSeek 自研的 3FS 分布式存储系统，论文未讨论在其他存储后端（如 Lustre、GPFS、S3）上的适用性和性能
  • 基线对比不够公平：与 SGLang (HiCache + Mooncake) 的对比存在实现差异（论文自己承认"Comparing DualPath and SGL(MC) is unfair due to implementation differences"），只报告了 Basic→Ours 的改善而非绝对性能
  • SGLang 基线频繁崩溃：多个大配置标记为 N/A（运行出错），削弱了对比的说服力
  • 未探索 split-read 优化：论文承认"将单个请求拆分到两条路径分别读取可能更好"但作为 future work 搁置——这可能是进一步提升的低垂果实
  • 在线服务的 working set 分析有局限：论文承认零 tool call latency 的假设使 working set 偏小，真实场景下 DRAM cache 可能更重要
  • DS 27B 的 TPOT 问题：小模型下 Basic 和 DualPath 的 TPOT 都显著高于 Oracle，说明基础 P-D transfer 开销在小模型上不可忽略，但论文仅标注"leave it as future work"
  • 调度算法相对简单：KV-Cache 读取路径选择仅基于 "which side has shorter reading queue"，没有考虑网络拓扑、请求优先级等因素
  • DeepSeek 内部系统的可复现性：基于 in-house inference framework，5K 行修改，外部研究者无法复现

Infrastructure Impact

  • Algorithm: 直接支撑了 agentic RL training 的 rollout 阶段，其中 DRAM 被 optimizer state 占用，KV-Cache 必须走 SSD 存储。这是 DualPath 的原始驱动力
  • Kernel: 使用现有 kernel（FlashMLA、DeepGEMM、DeepEP），无需 custom kernel。但 CNIC-centric H2D/D2H 依赖 GPUDirect RDMA API 和 io_uring-like 接口
  • Framework: 对 PD 分离推理框架的核心数据通路做了架构级重构。思想可以移植到 vLLM、SGLang 等开源框架，但需要深入理解其网络拓扑和 QoS 配置
  • LLM: 对 MLA 模型（DeepSeek-V3）和 GQA 模型（Qwen）均有效。MLA 的低 KV-Cache per-token 反而降低了 DualPath 的绝对收益（因为 cache-compute ratio 本身低），非 MLA 模型收益更大
  • Agent: 直接针对 agentic workload 设计，trace 数据来自真实 coding agent。对所有多轮、长上下文、高 KV-Cache reuse 的 agent 场景都有价值
精读 Deep Analysis

Deep Analysis (framework)

1. System Scope

  • Primary goal: Inference serving optimization，专注于 PD 分离架构下 agentic workload 的 KV-Cache I/O 加速
  • Distributed: 多节点分布式，评估规模从 2 节点（1P1D）到 144 节点（48P96D, 1152 GPU）
  • Both online and offline: 离线批量推理（RL training rollout）和在线服务（SLO-aware，TTFT ≤ 4s, TPOT ≤ 50ms）
  • Target workload: Agentic LLM inference——多轮、短追加、长上下文、高 KV-Cache 命中率。也适用于 agentic RL training 的 rollout 阶段

2. Architecture & Data Flow

组件拓扑：

  • Central Scheduler: 全局请求调度器，接收客户端请求，分配到 (PE, DE) pair，选择 read path。CPU 开销 <10 核
  • Prefill Engines (PE): 加载 KV-Cache + prefill 计算。每个 PE 管理一个 GPU
  • Decode Engines (DE): autoregressive decoding。接收 PE 完成的完整 KV-Cache
  • Traffic Manager: 每个 engine 内部模块，管理所有数据搬运（H2D/D2H、PE↔DE transfer、storage I/O）
  • 3FS: 分布式存储后端，无内部 DRAM cache，可饱和 400Gbps SNIC 带宽

Control plane vs Data plane:

  • Control plane: Central scheduler 通过 Leader Engine（每组 rank 0）与 engine groups 通信，FIFO queue + 三级分类调度
  • Data plane: 双路径 KV-Cache 传输，统一走 CNIC（RDMA），通过 VL 隔离 QoS

State management:

  • Stateful: KV-Cache in 3FS（持久化）、DE buffer / PE buffer in DRAM（临时）、HBM 中的计算状态
  • Stateless: Scheduler 仅维护 (seq_e, tok_e, read_q) 等轻量级指标

Failure handling: 论文未提及，这是一个明显的遗漏——大规模部署中 GPU/node 故障是常态

#### 2b. Data Movement Hotspots

Top 3 data movement bottlenecks:

1. Storage → DRAM (KV-Cache loading)：最大瓶颈。每节点仅 1× 400Gbps SNIC，而 8 GPU 共享。64K 上下文下 DeepSeek-V3.2 单请求 KV-Cache 约数百 MB。频率：每个 prefill request 一次，但 agentic workload 下每秒大量请求。DualPath 前：仅 PE 端加载，SNIC 饱和；DualPath 后：PE + DE 均参与，缓解瓶颈。这是本文解决的核心热点。

2. DRAM ↔ HBM (H2D/D2H)：每层 KV-Cache 的 H2D 传输，每个 forward batch 重复 n_layer 次。走 CNIC loopback，通过 VL 隔离不影响推理通信。PE 端 DRAM 压力 = 2sB，DE 端 = (3+2P/D)Bs。在典型配置下不构成瓶颈但在极端 P/D 比下可能。与计算重叠，非阻塞。

3. PE → DE (KV-Cache transfer via RDMA)：prefill 完成后完整 KV-Cache 传到 DE。走 CNIC compute network，带宽高（每 GPU 400Gbps）。频率：每个请求一次。重叠 + 高带宽，通常非瓶颈，但 DS 27B 小模型上 P-D transfer overhead 显著（TPOT 远高于 Oracle）。

**Dual-path KV-Cache loading**增加 storage → DE → RDMA → PE 新路径，聚合所有 engine 的 SNIC 带宽消除 PE 端 SNIC 单点瓶颈，吞吐提升最高 1.87×DE 端额外 DRAM buffer 开销、更复杂的数据编排、DE DRAM/CNIC 压力增加
**CNIC-centric traffic management**所有 GPU 数据传输统一走 CNIC + RDMA Write，利用 InfiniBand VL 隔离KV-Cache 流量不影响模型推理通信（~99% 优先级给推理），同时 RDMA submit 延迟低（~1μs vs cudaMemcpy 5-7μs）放弃了 GPUDirect Storage 的直连路径，看似"绕路"；依赖 InfiniBand VL 或 RoCE TC 的硬件 QoS 支持
**Adaptive request scheduler**三级分类调度（overloaded/short-queue/long-queue），PE 以 tok_e 为 proxy 平衡 GPU+NIC，DE 两阶段（组间全局平衡 + 组内 HBM 感知），路径选择基于 queue lengthStorage NIC 负载均衡从 1.53 降到 1.18（Max/Avg），attention 执行时间 Max/Avg 低至 1.06调度器是中心化单点，虽然目前 <10 核但超大规模下可能成为限制；路径选择未考虑 split-read
**Layerwise prefill (adopted)**每层独立分配/释放 KV-Cache in HBM，仅存当前层Token batch size 扩大约 n_layer 倍，提升 GPU 利用率需要更多细粒度数据传输（Layer Block vs Full Block 两种布局增加复杂度）

4. Scheduling & Resource Management

Batch formation: Dynamic continuous batching for DE（所有请求进 forward batch）。PE 端采用 compute-quota-based FIFO packing：逐个添加请求直到预估 attention 层执行时间达到上限（300ms），溢出请求做 chunked prefill（binary search 找最大可塞入的 bsz'）。

Memory management: PE buffer + DE buffer 在 DRAM 预分配（DualPath 每节点 80GB for DS models，320GB for Qwen 32B）。HBM 采用 paged 管理，DE 调度考虑 remaining HBM 来决定能否接纳新请求。与 SGL(MC) 对比，SGL(MC) 使用 1.5TB DRAM/node（Mooncake Store），DualPath 仅需 80GB——DRAM 效率高 18.75×。

GPU utilization:

  • Layerwise prefill 避免 HBM 容量限制，允许更大 batch
  • Compute quota 确保同一 EP group 内各 GPU 的 attention 执行时间对齐（Max/Avg ≤ 1.06），减少同步等待 bubble
  • Dual-path loading 让 KV-Cache 读取与计算重叠，减少 GPU idle

Multi-tenancy: 未提及

Priority/SLO-aware:

  • Online serving 场景 SLO: TTFT ≤ 4s, TPOT ≤ 50ms
  • 通过 overloaded engine 排除（tok_e > β）间接实现 SLO 感知
  • 无显式 per-request priority 机制

#### 6c. Bottleneck Shift Analysis


Before (Basic): Storage I/O bound (PE SNIC saturated, GPU idle waiting for KV-Cache)
  → + Layerwise prefill: Storage I/O bound (HBM 不再限制 batch size, 但 SNIC 仍饱和)
  → + Dual-path loading: Approaching compute bound (DE SNIC 参与, 总 SNIC BW 翻倍)
  → + Scheduling: Compute bound (SNIC 均衡利用, Ours ≈ Oracle 说明 I/O 已非瓶颈)
  
Remaining bottleneck after all optimizations:
  - DS 660B: GPU compute (Ours ≈ Oracle)
  - DS 27B: P-D KV-Cache transfer overhead (TPOT > Oracle, 小模型下 transfer time 占比高)
  - Large-scale: 可能出现 scheduler queuing (论文承认 TTFT percentile 还有优化空间)

#### 6d. Baselines & Fairness

Baselines:

  • Basic: 自家 in-house framework 未修改版。公平（same implementation base）
  • SGL(MC): SGLang + HiCache + Mooncake Store + 3FS。不公平（论文自认："Comparing DualPath and SGL(MC) is unfair due to implementation differences"）。SGL(MC) 用 1.5TB DRAM/node 做 Mooncake Store，DualPath 仅 80GB——Mooncake 的 DRAM cache 优势被忽略。SGL(MC) 多个配置报 N/A（运行出错），但论文未解释原因
  • Oracle: 零 I/O 上界（bypass all disk/transfer），公平（基于 DualPath 实现）

隐藏的不公平:

1. DualPath 使用 DP attention（数据并行），SGL(MC) 使用 TP=8（张量并行）——attention 层计算效率不同

2. Mooncake Store 的 DRAM cache 命中率未报告——如果大部分 KV-Cache 命中 DRAM，Mooncake 可能不需要走 SSD

3. 仅报告 Basic → Ours 的改善比，不报告绝对性能——我们无法知道 DualPath vs SGL(MC) 的绝对吞吐

4. 数据集来自 DeepSeek 内部 RL training trace，可能对 DeepSeek 模型有利

DualPath 可能不占优的场景:

  • Working set 较小、DRAM 充足：Mooncake 的 DRAM pool 命中率高时，根本不需要走 SSD
  • 短上下文、低 KV-Cache 命中率：compute-bound 场景下 DualPath 无优势
  • 单轮或少轮对话：无 KV-Cache reuse，传统架构不存在 I/O 瓶颈

7. API & Usability

  • API compatibility: 未提及——in-house system，无公开 API 规范
  • Model format: 支持 DeepSeek MLA 系列（V3、V3.2、27B downscaled）和 HuggingFace 标准模型（Qwen2.5-32B）
  • Deployment: 需要 InfiniBand（或 RoCE）双网隔离的集群、3FS 分布式存储、NVIDIA Hopper GPU + 400Gbps RDMA NIC
  • Configuration complexity: 高。需要配置：P/D ratio、InfiniBand VL QoS 参数（qos_max_vls, qos_high_limit, qos_vlarb_high/low）、scheduler 参数（α, β, compute quota）、DRAM buffer size、block size。论文 Appendix A.1/A.4 有详细配置

10. Adoption & Maturity

  • Open source: 否。DualPath 是 DeepSeek 内部 inference framework 的一部分，~5K 行修改。代码未公开
  • Production deployment: 是——论文明确提到 "widely adopted in our production deployment"（CNIC-centric approach），数据来自 "production agentic RL training workloads"
  • 依赖的开源组件: 3FS（已开源）、FlashMLA（已开源）、DeepGEMM（已开源）、DeepEP（已开源）
  • Community: 无独立社区。属于 DeepSeek 内部基础设施
  • Adoption barrier:

1. 需要 InfiniBand/RoCE 双网隔离的数据中心网络——大多数机构不具备

2. 需要 3FS 分布式存储系统——部署成本高

3. 需要深入理解 InfiniBand QoS VL 配置（论文 A.1 给出了具体参数）

4. Scheduler 参数（α, β）需要根据硬件和工作负载 profiling 调优

5. 核心思想（dual-path + VL isolation）可以移植到其他框架，但工程量巨大

与 DeepSeek-V3 的关系: DualPath 的作者团队与 DeepSeek-V3 高度重合（Shaoyuan Chen、Mingxing Zhang 出现在 DeepSeek-V3 infra 论文 Zhao et al. 2025a 中）。DualPath 使用的全部底层组件（3FS、FlashMLA、DeepGEMM、DeepEP）均为 DeepSeek 自研并开源。DualPath 本质上是 DeepSeek inference stack 的关键组成部分，论文描述的是其内部生产系统在 agentic workload 兴起后的架构演进。3FS 作为 DeepSeek 的分布式存储系统，其"无 DRAM cache、可饱和 400Gbps"的设计特点恰好匹配 DualPath 的需求——二者是协同设计的关系，而非独立系统。
