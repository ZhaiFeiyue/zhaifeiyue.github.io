L3 Synthesis: HexAGenT (2605.16637) — Workflow-Aware Heterogeneous Scheduling for Agentic LLM Serving #

§1 相关论文 #

HexAGenT 处于 agentic workflow serving 的核心——它是迄今唯一同时解决 online-revealed DAG scheduling + heterogeneous P-D placement + decode KV 容量约束的系统。以下 8 篇论文从不同层次与之关联：

Serving-layer 调度——直接可比 #

  • Continuum (2511.02230)：同属 agent serving 优化，但攻击不同维度。Continuum 解决的是多轮 tool call 间隙的 KV cache 保留（TTL + program-level FCFS），优化单个 agent program 的 job completion time [2511.02230]。HexAGenT 解决的是多个并发 workflow 在异构 P-D 集群上的 联合 placement + 排序，优化 workflow-level SLO attainment [2605.16637]。二者处于 serving stack 的同一层但解决正交问题——Continuum 管理时间维度（何时保留/驱逐 KV），HexAGenT 管理空间维度（在哪个 instance 上执行）。
  • PASTE (2603.18897)：在 agent 和 tool system 之间作为中间件，通过 pattern-aware 投机执行将 tool stall 与 LLM generation 重叠，减少 48.5% 端到端延迟 [2603.18897]。PASTE 优化的是 tool 执行 的等待时间，假设 LLM inference 由底层 serving 引擎处理；HexAGenT 优化的是 LLM inference 本身在异构集群上的调度。二者可叠加：PASTE 减少 tool→LLM 间隙，HexAGenT 减少 LLM call 在 P-D 集群上的排队延迟。

KV cache 与内存管理——互补层 #

  • SideQuest (2602.22603)：用 LRM 自身的辅助推理线程做 KV cache 语义驱逐，56–65% peak token 降低 [2602.22603]。SideQuest 攻击的是 单个 agent session 内部 的 context 线性膨胀；HexAGenT 的 decode KV 容量约束 $m(c) \leq \text{Cap}(d)$ 是 跨 session 的 instance-level 资源管理 [2605.16637]。SideQuest 的 eviction 可以减少每个 call 的 KV 需求，从而放松 HexAGenT 的 decode capacity 约束。
  • Continuum (2511.02230) 的 TTL 机制和 Agent Memory (2603.04428) 的 Q4 持久化都解决 KV cache 的 跨 turn 生命周期 问题——前者在 GPU 上用 TTL pin [2511.02230]，后者在边缘设备上用 disk persistence [2603.04428]。HexAGenT 的 KV-transfer stage 假设 KV 在 prefill→decode 间通过网络传输，这些 KV 管理方案影响的是传输数据量和 decode instance 的 KV 压力。

Workflow 抽象——概念共振 #

  • SGH (2604.11378)：将 agent 执行统一为 scheduler 模型 $\mathcal{E}=(\mathcal{S},\mathcal{U},\mathcal{P},\mathcal{O},\Delta)$，用 ready-set 基数 $|\mathcal{U}|$ 区分 Agent Loop 和 graph executor [2604.11378]。HexAGenT 也用 DAG 建模 workflow，但两者的 DAG 语义不同：SGH 的 DAG 是 执行控制图（决定哪些 node 可并行），HexAGenT 的 DAG 是 调度输入（告诉 scheduler call 之间的依赖关系）。SGH 是 position paper 无实验数据；HexAGenT 是完整系统有端到端评估。二者在"agent 执行 = DAG scheduling"这一洞察上高度共振。
  • CMV (2602.22402)：将 session 历史建模为 DAG（快照为节点、分支为边），实现 context 版本控制 [2602.22402]。与 HexAGenT 的 DAG 不同——CMV 的 DAG 是 用户级 session lineage（手动 snapshot/branch），HexAGenT 的 DAG 是 runtime 级 call dependency（自动 online reveal）。但两者共享 DAG 作为 agentic 工作流的自然抽象这一认知。

瓶颈刻画——互相校验 #

  • CPU-Centric Perspective (2511.00739)：揭示 CPU 工具执行占 agentic AI 端到端延迟高达 88% [2511.00739]。这与 HexAGenT 的隐含假设形成张力：HexAGenT 假设 LLM inference 是瓶颈（优化 prefill/decode placement），但如果 CPU tool 执行占主导，workflow 级 SLO 可能主要被 tool 延迟而非 inference 延迟决定。PASTE 的 profiling 数据也显示 tool 执行占 35–61% [2603.18897]。HexAGenT 对此的回应是：其 workflow horizon $H_w(t)$ 包含了 tool 执行时间（作为 call 间隔的一部分），但 scheduler 只能控制 LLM call 的 placement——tool 延迟被建模为不可控的外生变量。

架构分析——设计空间定位 #

  • Claude Code Analysis (2604.14228)：揭示生产级 coding agent 是 "1.6% AI 决策逻辑 + 98.4% 确定性基础设施" [2604.14228]。HexAGenT 恰好就是这 98.4% 基础设施中的一个关键组件——它不改变 agent 的 planning/reasoning，只优化 inference 调度。Claude Code 的 ReAct 循环产出的 LLM call 序列，正是 HexAGenT 的调度输入。

§2 本篇 vs 相关论文的 delta #

核心创新：projected scaled-SLO risk + joint heterogeneous P-D placement #

HexAGenT 的独特贡献在于将三个此前独立处理的维度联合优化：

  1. Workflow-level SLO（而非 per-call latency）：所有 peer works 要么不关注 SLO（SideQuest、CMV、Agent Memory），要么定义在 per-call 或 per-program 层面（Continuum 的 job completion time）。HexAGenT 的 $\min_\pi \alpha$ s.t. $\frac{1}{|\mathcal{W}|}\sum \mathbb{1}[C_w^\pi \leq \alpha H_w] \geq \tau$ 是目前唯一的 workflow-level scaled-SLO 形式化 [2605.16637]。
    1. Online-revealed DAG：SGH 提出 DAG 执行框架但假设 DAG 在执行前由 planner 一次性生成 [2604.11378]。HexAGenT 处理的是 DAG 在运行时逐步揭示的场景——到达时只有源节点可见，子节点在父节点完成后才出现 [2605.16637]。这更符合 ReAct 风格 agent 的实际行为（Claude Code 的 queryLoop 就是典型的 online-revealed 模式 [2604.14228]）。
      1. Heterogeneous P-D placement：在 A100/H100/H200 混合集群上联合选择 prefill instance + decode instance，考虑跨硬件类型的 KV 传输带宽差异。没有任何 peer work 处理此维度——Continuum/SideQuest/Agent Memory 都在同构环境下工作。

        2. 与 Continuum 的互补性 #

        Continuum 通过 TTL + program-level FCFS 减少 per-turn queueing delay（scheduling bubbles 占总延迟 58.2%）[2511.02230]。HexAGenT 通过 projected ratio 排序 + 异构 placement 减少 workflow tail SLO violation。两者在 priority signal 的构造上差异显著：

        维度ContinuumHexAGenT
        Priority signalProgram-level FCFS（近似 SRTF）Projected ratio $R_s(c,t) = \frac{(t-a_w)+\Delta_s(c,t)}{H_w(t)}$
        KV cache 决策主动管理（TTL pin/evict）被动约束（$m(c) \leq \text{Cap}(d)$ admission control）
        收益来源消除排队气泡（减少 idle time）优化 placement + 排序（减少 critical-path delay）
        目标 metricJob completion time（单个 program）SLO attainment rate（所有 workflow 的 tail）
        评估规模单机（vLLM on H100）异构集群（8–10P + 8–10D，A100/H100/H200）

        二者完全可叠加：Continuum 的 TTL 减少 per-turn KV 重计算，HexAGenT 的 projected-ratio 减少 workflow-level SLO violation。一个管理 KV cache 生命周期，另一个管理 call placement。

        与 PASTE 的正交性 #

        PASTE 优化 tool→LLM 间隙（通过投机执行减少 67% tool stall）[2603.18897]；HexAGenT 优化 LLM call 在集群上的执行。二者攻击不同延迟来源：

        • PASTE 的 characterization 显示 tool 执行占 35–61% E2E 延迟
        • HexAGenT 的 characterization 显示 per-call FCFS 下 workflow Req99 可达 26.89（Qwen-BFCL），说明 LLM scheduling 低效也是巨大延迟来源 [2605.16637]

        在一个部署了 HexAGenT + PASTE 的 agentic serving stack 中，HexAGenT 负责"LLM call 放在哪里执行、以什么顺序"，PASTE 负责"下一个 tool call 能否提前投机执行"。

        与 SGH 的关系：从理论到实践 #

        SGH 论证了 Agent Loop 是 $|\mathcal{U}|=1$ 的 scheduler，graph executor 才能 $|\mathcal{U}|\geq 1$ [2604.11378]。HexAGenT 的实践恰好验证了这一理论——它把多个 workflow 的 ready calls 组成 $|\mathcal{U}| > 1$ 的就绪集合，用 projected-ratio 排序做全局调度。不同之处在于：SGH 讨论的是 single-agent 内部的多节点并行，HexAGenT 讨论的是 多 agent workflow 之间的调度竞争。SGH 的 bounded termination 证明与 HexAGenT 的 progressive horizon refinement 有概念联系——两者都需要保证系统不会无限循环。

        §3 可攻击面 #

        攻击 1：Tool 延迟被忽视 #

        HexAGenT 假设 workflow 端到端延迟主要由 LLM inference scheduling 决定。但 CPU-Centric Perspective (2511.00739) 证明 tool 执行占 E2E 高达 88% [2511.00739]，PASTE 也确认 35–61% [2603.18897]。如果一个 workflow 的 tool 执行占主导，HexAGenT 的优化空间会大幅缩小——因为 scheduler 只能控制 LLM call 的 placement，对 tool 延迟无能为力。

        矛盾根源：HexAGenT 的 benchmark workloads（ShareGPT、BFCL-v3、LATS）可能恰好是 LLM-heavy 的——ShareGPT 纯对话无 tool，BFCL-v3 的 function calling 通常很快（<100ms），LATS 的 tool 调用也较轻量。在 tool-heavy 场景（如 SWE-agent 的 bash 执行平均 925ms [2511.02230]，RAG 检索占 88%）下，HexAGenT 的收益会显著下降。HexAGenT 的 standalone horizon $H_w$ 包含 tool 延迟但 scheduler 无法缩短它——projected ratio 排序只能改善 LLM call 的排队和 placement。

        攻击 2：KV cache 管理缺失 #

        HexAGenT 对 KV cache 仅做 admission control（$m(c) \leq \text{Cap}(d)$），不主动管理其生命周期。Continuum 证明了 per-turn queueing delay 占总延迟 58.2%，仅靠排序无法消除 [2511.02230]——当 agent 在 tool call 期间 KV 被驱逐，返回后必须重新排队，这个延迟独立于 HexAGenT 的 placement 决策。SideQuest 进一步证明 heuristic KV eviction 可导致模型崩溃（non-completion rate 60%+）[2602.22603]。HexAGenT 的 capacity check 只防止 OOM，不优化 KV 复用或保留策略。

        HexAGenT 可能的辩护：其 SGLang 实现的 RadixAttention 已提供 prefix caching，且 P-D 分离架构中 decode instance 的 KV 是 prefill instance 传输来的（总是新鲜的），tool call 间隙的 KV 保留问题主要在非 P-D 架构中更严重。

        攻击 3：无形式化保证 #

        HexAGenT 的 projected-ratio greedy scheduling 没有竞争比、近似比或收敛性证明 [2605.16637]。SGH 至少证明了 bounded termination 和 conditional soundness [2604.11378]。对于在线调度问题，经典理论（如 SRPT、makespan minimization on unrelated machines 的 2-approximation）提供了丰富的分析工具。HexAGenT 作为 $R_s(c,t)$ 的 argmax greedy，在 workflow 是纯链时应可证明为 SRPT 的推广，但论文未尝试。

        攻击 4：评估缺乏真实 agentic trace #

        HexAGenT 的 workload（ShareGPT、BFCL-v3、LATS、Mixed）是从 benchmark 构造的模拟 trace，不是真实 production agentic workflow 的 trace。Continuum 使用了 Company A 的真实 SWE-agent 测试床（达到 8.18× 改善 vs 模拟的 1.12–3.66×）[2511.02230]。Claude Code 分析表明生产级 agent 的 tool 调用模式（27 种 hook events、多种 permission modes）远比 benchmark 复杂 [2604.14228]。HexAGenT 的 SLO reduction 数字在真实 production trace 下可能不同。

        攻击 5：非单调鲁棒性未解释 #

        Table 5 中 Qwen 的估计误差-SLO 退化关系是非单调的（10% 误差时 Req99 退化 9.5%，但 30% 误差时只有 5.4%）[2605.16637]。论文承认但未解释。可能的原因：噪声估计偶尔打破 near-tie call 的排序而碰巧改善贪心决策——但这说明 projected-ratio 在 tie 附近的决策是脆弱的。

        §4 生态位 #

        在 agent serving stack 中的位置 #

        HexAGenT 占据一个独特的生态位：workflow-level 的 GPU-side scheduling layer，位于 agent runtime（Claude Code queryLoop [2604.14228]、SGH 的 plan-commit-dispatch [2604.11378]）之下，LLM serving engine（vLLM/SGLang 的 per-call batching）之上。

        这个位置的价值在于：它是目前唯一在 workflow 粒度 做调度的系统。Continuum 在 program 粒度做 FCFS/TTL，SGLang 在 call 粒度做 batching——HexAGenT 填补了 workflow-level 的调度空白。

        采用路径 #

        HexAGenT 基于 SGLang v0.5.9 实现，代码未公开。其采用的主要障碍是：(1) 需要异构 P-D 集群（大多数部署仍是同构 GPU），(2) 需要 workflow metadata（agent runtime 须将 DAG 结构传递给 scheduler）。同构集群上仍有 24–37% 的 Req99 改善 [2605.16637]，说明即使没有异构 placement，workflow-aware 排序本身也有显著价值。

        §5 未探索方向 #

        1. HexAGenT + Continuum 的联合系统：HexAGenT 做 workflow-level P-D placement，Continuum 做 per-turn KV TTL 管理。HexAGenT 的 projected ratio 可以为 Continuum 的 TTL 决策提供 workflow-level urgency signal——如果一个 workflow 的 $R_s > 1$，其 KV cache 应该优先保留。这需要在 scheduler 和 serving engine 之间传递 workflow 优先级。
          1. Tool-aware projected ratio：将 PASTE 的 pattern prediction 信息注入 HexAGenT 的 projected-ratio 计算。如果 PASTE 预测下一个 tool call 将很快返回（高概率短延迟 tool），HexAGenT 可以降低当前 workflow 的 urgency（因为 tool 不会拖延太久）。反之，如果预测到长延迟 tool（如 bash 执行、web fetch），HexAGenT 应提高 urgency 确保 LLM call 尽快完成以释放 GPU 资源。
            1. Projected-ratio 的 competitive ratio 分析：在 chain workflow + 同构集群的简化设定下，projected-ratio greedy 应可证明为 SRPT 的推广。SGH 的 bounded termination 证明 [2604.11378] 提供了方法论参考——通过节点状态机的结构性保证推导系统性质。为 HexAGenT 提供理论保证将显著增强其学术影响力。
              1. 边缘设备的 workflow scheduling：Agent Memory (2603.04428) 证明了边缘设备上多 agent 推理的可行性（Q4 KV persistence，136× TTFT 加速）[2603.04428]。HexAGenT 的 projected-ratio 排序可以适配到边缘场景——将 heterogeneous P-D placement 替换为 heterogeneous device placement（如 M4 Pro CPU vs GPU vs ANE），在固定内存预算下优化多 agent workflow 的端到端延迟。
                1. CPU-GPU co-scheduling for agentic workflows：CPU-Centric Perspective (2511.00739) 揭示了 CPU tool 执行的瓶颈 [2511.00739]，HexAGenT 只调度 GPU 上的 LLM call。一个完整的 agentic workflow scheduler 应该联合调度 CPU 工具执行和 GPU LLM 推理——将 COMB 的 CPU 微批调度与 HexAGenT 的 GPU P-D placement 统一到一个 framework 中。
