Chimera — 异构 LLM 集群上的多智能体工作流预测式调度 #

1. TL;DR #

Chimera 是 vLLM 之上的中间件调度层，为跑在异构 LLM 集群（不同尺寸/家族的模型）上的多智能体工作流联合优化端到端延迟与任务性能：语义路由给每个请求打各模型置信分，QRF 预测整条工作流剩余输出 token 数作 STJF 优先级，活动监视器用在途预测 token 量估计每引擎拥塞。相对 vLLM/MLFQ/LTR 延迟降 1.2–2.4×、性能升 8.0–9.5pp，调度开销 ≤2.2%。

2. Q1 / Q2 / Q3 #

Q1 — 痛点 #

现有 LLM serving 有两个不匹配异构多智能体工作流的假设：(1) serving 系统假设同构集群（同一模型的一个或多个副本），只在单请求粒度优化，不管异构容量如何分配给难度不同的请求；(2) 路由方法把延迟当作模型的静态属性，而非排队与资源竞争下的涌现结果。两者叠加导致一个内在权衡（Figure 1）：把请求路由到更强但更慢的模型能提升性能，却放大 GPU 竞争、拖慢其他工作流的延迟。工作流的端到端延迟由跨多个依赖阶段的排队交互涌现，使请求级调度次优（Luo et al., 2025）。

Q2 — 方法 #

Chimera 用三个预测信号驱动模型选择与队列优化，构成四模块中间件（Figure 2）：

  1. Semantic Router：ModernBert-large 微调的多标签编码器，为每个请求输出各候选模型独立 sigmoid 置信分 $q[m]\in[0,1]$（成功概率）。
  2. Length Predictor：CPU 上的 Quantile Random Forest（QRF）预测工作流级剩余总输出 token 数 $Y_r$ 的条件中位数，作为调度优先级（STJF）。
  3. Activity Monitor：program table 记录每请求预测 token 量，聚合成每引擎在途工作量 $I[m]$。
  4. Load Balancer：把 $q[m]$ 与由 $I[m]$ 算出的 TTLT 负载 $L[m]$ 结合——先找最快模型 $m_{\text{fast}}$，再在延迟 slack $\tau$ 内、置信提升超过 margin $\Delta s$ 的前提下选置信最高的模型；随后以 $\hat y$ 为优先级入队。

    5. 核心技术壁垒：把「模型路由」与「拥塞感知负载均衡」通过一个单一延迟-性能感知的 dispatch 规则（同模型的置信分与由预测 token 量导出的 TTLT 在同一比较循环里做取舍）耦合起来，并让这个耦合在负载上升时产生单调回退（congested 大模型 $L[m]$ 上升 → 流量自动转向更空闲模型）——这是难以复制的联合设计，而非把路由器和调度器简单串联。同一工作流跨阶段复用路由决策既省 GPU（少调路由器）又促进 KV-cache 复用，是让该耦合在线可行的关键。

    Q3 — 结果 #

    在 APPS（代码生成）与 MATH（数学推理）、多种异构模型组合与 RPS 下，Chimera 始终占据最好的操作点（同时最低 E2E 延迟 + 最高性能）。相对 vLLM 平均延迟降 1.2–2.4×、性能升 8.0–9.5pp；APPS Qwen1.5+7B 上对 vLLM 达 3.4× 加速 + 16.0pp 提升。调度器（路由+预测+其他）合计仅占 E2E 延迟 ≤2.2%。延迟 slack $\tau$ 是可调旋钮：收紧偏低延迟、放宽解锁高性能。

    sequenceDiagram participant App as Multi-agent App participant LB as Load Balancer participant R as Semantic Router participant P as Length Predictor participant AM as Activity Monitor participant E as Chosen Engine (priority queue) App->>LB: request r=(p,i,x_r,meta) alt program_id 已分配模型 LB->>LB: 复用 A[program_id]=m* else 新工作流 LB->>R: q[m]=σ(ℓ_{r,m}) for m∈M LB->>AM: 读 I[m], 算 L[m]=P_m·decode_ms/max_batch LB->>LB: m_fast=argmin L[m]; 选 slack τ 内且 q≥q*+Δs 最高分模型 m* end LB->>P: ŷ=Ŷ(meta, m*) (STJF 优先级) LB->>AM: I[m*][r.id]=ŷ (记在途负载) LB->>E: Dispatch(r, priority=ŷ, arrival) Note over E: 本地 min-heap 按 (starvation_level, priority, arrival) 出队

    调度器只阻塞在 dispatch 所需的最小输出上（路由置信分 + 预测总长），路由器与预测器作为异步批处理服务在独立 worker pool 运行（进程隔离），推理引擎仍负责 continuous batching / kernel / KV-cache（§3.8）。

    4. 作者证明 #

    无形式化作者证明 — 仅实证。全篇没有排队论/收敛性定理，但给出了一组闭式估计公式 + 一个 dispatch 规则，其物理意义与自洽性可核验。

    符号表 #

    符号含义
    $r=(p,i,x_r,\text{meta\_info}_r)$请求：工作流 id $p$、阶段 $i$、阶段输入 $x_r$、元信息
    $N_p$工作流 $p$ 的有序阶段数
    $C_p,\ S_p$工作流端到端完成时间 / 最终输出的任务性能分
    $q[m]=\sigma(\ell_{r,m})$模型 $m$ 对请求 $r$ 的独立成功置信分 $\in[0,1]$
    $Y_r=\sum_{j=i}^{N_p}\text{OutTokens}(p,j)$从当前阶段到末阶段的剩余总输出 token
    $\hat y\approx\text{median}(Y_r\mid\phi(r),m^\star)$QRF 预测的条件中位数（优先级）
    $P_m,\ L[m]$模型 $m$ 在途预测 token 之和 / 其 TTLT 估计
    $\tau,\ \Delta s$延迟 slack / 置信 margin 两个旋钮
    $S,\ Q$反饥饿的 starvation 阈值 / running quantum

    关键方程与物理意义 #

    TTLT（time-to-last-token）负载估计：

    $$ L[m] = \frac{P_m \cdot \Pi[m].\text{decode\_ms\_per\_token}}{\Pi[m].\text{max\_batch\_size}}, \qquad P_m = \sum_{t \in \text{Values}(I[m])} t $$

    分子把在途 token backlog $P_m$ 乘离线 profile 的每 token 解码耗时，转成时间；除以 max batch size 近似并行服务（一个 batch 内多请求并发解码）。dispatch 规则：

    $$ m_{\text{fast}} = \arg\min_{m \in M} L[m]; \quad \text{选按 } q[m] \text{ 降序第一个满足 } L[m]\le(1+\tau)L_{\text{fast}} \ \wedge\ q[m]\ge q^\star+\Delta s \text{ 的 } m $$

    最小核验（6 条） #

    1. 为何除以 max_batch_size 而非乘：解码是批并行的，backlog 被 batch 并发消化，故有效等待时间随批容量下降 → 除法正确。
    2. 为何用中位数而非均值：输出长度重尾（APPS Qwen1.5B 均值 447 但 std 1276；MATH std 2587，§4.1），均值被尾部拉偏，QRF 分位数对重尾稳健。
    3. 单调回退性：$m^\star=m_{\text{fast}}$ 起步，只有满足 $L[m]\le(1+\tau)L_{\text{fast}}$ 才升级到更强模型；当强模型拥塞其 $L[m]$ 增大越出 slack，即被自动排除 → 负载上升时流量单调转向空闲模型，边界由 $\tau$ 决定（内点/边界解随 $\tau$ 连续移动）。
    4. $\Delta s$ 的作用：要求置信提升超过 margin 才换模型，避免为边际收益频繁切换、破坏 KV-cache 复用 → 迟滞（hysteresis）项。
    5. 为何丢掉 prefill token：长输出下解码时间主导 prefill，作者实测加权 prefill「no measurable improvement」→ 优先级只用 $\hat y$ 自洽。
    6. STJF 优先级正确性：按 $Y_r$（工作流剩余总量）而非单请求长度排序，才对齐 makespan 最短；Figure 3 实测 STJF 比 SJF 再降 15–34% 排队时间，佐证工作流级信息有增益。

      7. 一个形式化模型（如 M/G/1 排队 + 路由概率）本可澄清 $\tau$ 与吞吐/尾延迟的定量关系及稳定性条件，本文以实证 Pareto 前沿代替。

      5. 实验与数据 #

      评测在 NVIDIA RTX A6000（49GB）上，模型覆盖 Qwen2.5-1.5B/3B/7B/14B、Ministral8B、Llama3B；指标为 (每 token E2E 延迟, 任务分)，越靠右下越好；baseline 为 vLLM、MLFQ、LTR，另有 oracle 参照 vLLM(STJF)。

      Paper's Figure 5, verbatim（caption: "Latency vs. performance on heterogeneous model combinations. ... Chimera demonstrates lower latency and higher performance compared to other serving systems."）. (a1)–(a4) 为 APPS、(b1)–(b4) 为 MATH。要注意的是：Chimera 的点整体位于所有 baseline 的右下方外沿，即同时更快更准；而 MLFQ/LTR 只在延迟轴上移动、性能几乎不变——因为它们不联合优化性能。APPS Qwen1.5+7B(a1) 对 vLLM 达 3.4× + 16.0pp；在 Llama3B+Ministral8B(a3) 这种超长输出配置里，vLLM(STJF) oracle 也只有 1.2× 空间，Chimera 已近 oracle。

      Paper's Figure 6, verbatim（caption: "Pareto frontiers of Chimera with and without the semantic router. ... Using the router yields a markedly better Pareto frontier across all RPS values."）. 这是最关键的消融：去掉路由器后系统只能用固定启发式（如 slack 允许时总选更强模型），无法按 query 自适应选模型，需要更高延迟才能换来相当性能。它直接支撑「per-query 置信路由 × 预测调度」的联合设计价值。

      7. 实现 cross-reference #

      调度主循环以伪代码形式在 L1 §3.2 Algorithm 1 给出（SCHEDULEREQUEST：置信分循环 → TTLT 累加 → slack+margin 选模型 → 预测总长作优先级 → 记在途负载 → Dispatch）。整体实现基于 vLLM 并暴露与 OpenAI chat/generation 兼容的 API（§3.8）。路由器与长度预测器作为异步批处理服务运行：请求入队轻量 feature 对象，后台 collector 按最大 batch 或 timeout 成批，批推理在独立 worker pool（进程隔离）执行，调度器只阻塞在 dispatch 所需最小输出上；推理引擎仍负责 continuous batching / kernel / KV-cache（§3.8）。

      关键实现细节（易漏）：

      1. 同工作流跨阶段复用路由分配（§3.6）：若 program id 已有分配则不再调路由器——既省 GPU（路由器占卡），又保 KV-cache 复用，还降低路由器训练复杂度（无需复杂训练流程）。
      2. 反饥饿的三元队列 key（§3.7）：本地 min-heap 按 (starvation_level, priority, arrival_time)；请求连续未被选中达阈值 $S$ 则 starvation_level 减一临时提升，经 $Q$ 次调度后再降回，既保 STJF 又界定最坏等待。
