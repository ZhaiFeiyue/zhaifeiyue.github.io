PrefillShare: Shared-Prefill Disaggregated Serving for Multi-Model Agent Workloads — L2 #

§1 TL;DR #

PrefillShare freezes a shared base-model prefill module and cache-conditioned fine-tunes task-specific decode modules, letting all models in a multi-agent workflow reuse one KV cache per prompt — yielding 4.5× lower p95 latency and 3.9× higher throughput on vLLM disaggregated serving.

§2 痛点 / 方法 / 结果 #

Q1 — 痛点：多模型 Agent 工作流中的 KV 冗余 #

Multi-model agent systems (e.g. ReAct-style pipelines with 4 specialized models) invoke multiple fine-tuned LLMs over a shared prompt context. Because KV caches are tightly coupled to each model's parameters ($\theta_i \neq \theta_j \Rightarrow C_i \neq C_j$), every model must independently prefill the same prompt and maintain its own cache. This causes:

  1. Latency blow-up — repeated prefill raises TTFT and intensifies prefill–decode contention.
  2. Memory explosion — duplicated KV caches saturate GPU memory, forcing evictions that kill prefix-cache hit rates and further degrade throughput.

    3. Disaggregated serving (DistServe, Splitwise) separates prefill and decode onto different GPUs to reduce contention, but does not eliminate the cross-model redundancy: $N$ models still run $N$ prefills for one prompt.

    Q2 — 方法：Shared Prefill + Cache-Conditioned Fine-Tuning #

    PrefillShare introduces a two-role factorization:

    • Base Prefill Module ($M_\text{base}$, frozen): processes prompt $X$ once → shared KV cache $C_\text{base}$.
    • Specialized Decode Modules ($M_\text{dec}^{(i)}$, trainable per task): generate output tokens conditioned on $C_\text{base}$.

    The key training procedure is cache-conditioned fine-tuning: freeze $\theta_\text{base}$, compute $C_\text{base}$ for each training example, detach gradients through the cache, and minimize

    This aligns training with inference: decode modules learn to generate from an "alien" base-model cache rather than their own, eliminating the feature mismatch that causes accuracy collapse under naive sharing.

    Paper's Figure 2, verbatim. Naive sharing (blue) collapses as the KV cache sharing ratio increases, while PrefillShare (orange) maintains near full-FT accuracy even at 100% sharing. Cache-conditioned fine-tuning is the critical enabler: without it, increasing the fraction of base-model KV cache used during decode leads to progressive accuracy degradation. With it, the decode module adapts to consume the foreign cache representation without loss.

    核心技术壁垒: The insight that a frozen base-model KV cache can serve as a universal prompt representation across heterogeneous fine-tuned models — provided the decode modules are explicitly trained against that cache — is non-obvious. Prior work (ICaRus) required running the encoder alongside decoding; PrefillShare eliminates this by recognizing that the decode module alone has sufficient capacity to bridge the feature gap when the training loss correctly conditions on the shared cache.

    A routing and cache-handoff mechanism built on vLLM disaggregated serving completes the system: after the shared prefill instance produces $C_\text{base}$, it is handed off to whichever task-specific decode instance the router selects, avoiding redundant transfer or recomputation.

    无形式化作者证明 — 仅实证

    No formal convergence or approximation-error bound is provided. The paper's argument is entirely empirical. Below are the 6 minimum checks:

    #CheckVerdict
    1Assumptions stated?Implicit: all task-specific models share the same base architecture and tokenizer. The prefill module must produce compatible KV representations for all decoders.
    2Variables defined?Yes. $\theta_\text{base}$ (frozen), $\theta_\text{dec}$ (trainable), $C_\text{base}$ (shared cache), loss $\mathcal{L}(\theta_\text{dec})$ all formally defined in §3.
    3Steps logically follow?Yes. The factorization → cache-conditioned training → shared inference pipeline is internally consistent. Freezing prefill enables sharing; conditioning training on the frozen cache aligns train/inference regimes.
    4Edge cases addressed?Partially. Table 2 shows robustness across model sizes (1.7B–14B). No discussion of what happens when task-specific adaptations require prompt-level feature differences that the shared prefill cannot capture.
    5Bounds / guarantees?None. Accuracy parity is demonstrated empirically but not bounded. A useful bound would be the approximation gap $\\hat{C}_\text{dec} - C_\text{full-FT}\$ as a function of the number of frozen layers.
    6Counter-examples?Not shown. The paper does not present a task where PrefillShare underperforms Full-FT. All reported numbers are comparable or better.

    Agent-specific checks:

    • Success-rate model: Empirical only — no sweep over (task difficulty, planning depth, tool set, backbone). Accuracy reported per-task but not as a function of agent pipeline complexity.
    • Latency budget per turn: Not decomposed. The paper reports aggregate p95 and TTFT but does not break down per-turn tool-call vs LLM-inference vs memory-read latency.
    • Failure mode classification: Not addressed. The paper does not classify failure modes of the multi-agent pipeline under PrefillShare.

    §5 实验与数据 #

    5.1 Accuracy: PrefillShare matches Full-FT #

    Paper's Table 1, verbatim. Accuracy on math (GSM8K/GSM+), coding (HumanEval/HumanEval+), and tool calling (BFCL Simple Python/Multiple) for LLaMA 3.1-8B and Qwen3-8B-Base. Models fine-tuned on MetaMathQA-40K, EvolInstruct-Code-80K, and xLAM-function-calling-60K respectively. PrefillShare achieves accuracy comparable to Full-FT across all six benchmarks on both backbones, with KV sharing fully enabled. Notable: PrefillShare sometimes slightly exceeds Full-FT (e.g. HumanEval 48.8 vs 48.2 on LLaMA, HumanEval 86.6 vs 83.5 on Qwen3-8B).

    Paper's Table 2, verbatim. GSM8K and GSM+ accuracy across Qwen3-1.7B/8B/14B. PrefillShare maintains comparable accuracy to Full-FT at all three scales, demonstrating that the cache-conditioned fine-tuning approach is robust to model size. The gap is consistently within ~1 point.

    5.2 Serving performance under multi-model agent workloads #

    Paper's Figure 3, verbatim (caption: "Serving performance under multi-model agent workloads"). Six panels compare p95 latency, throughput, and TTFT for ReAct (top) and Reflexion (bottom) patterns as session arrival rate increases from 1 to 4 sessions/s. PrefillShare's advantage widens dramatically at higher arrival rates: at 4 sessions/s under ReAct, throughput reaches ~1050 tok/s vs ~280 for baseline (3.75×), while p95 latency is ~25s vs ~185s (7.4× improvement). The divergence is driven by the baseline's prefix-cache miss rate rising under load, triggering redundant re-prefill.

    5.3 Prefix cache hit ratio and scalability #

    Paper's Figure 4, verbatim (caption: "Prefix cache hit ratio and throughput under varying max concurrent sessions"). Top: the baseline's prefix cache hit ratio drops from ~60% at 20 sessions to near 0% at 80+ sessions, while PrefillShare sustains ~90% across the entire range. Bottom: throughput tracks the hit ratio — baseline plateaus at ~200 tok/s and never recovers, while PrefillShare peaks at ~1100 tok/s around 110 sessions before declining due to handoff overhead at extreme concurrency. The ~40-session cliff in the baseline is where KV cache evictions begin cascading.

    5.4 Generalization to larger backbone (Qwen3-14B) #

    Paper's Figure 5, verbatim. Replicates the Fig 3 experiment with Qwen3-14B. The pattern holds: PrefillShare achieves substantially lower p95 latency and higher throughput under both ReAct and Reflexion. At 3 sessions/s under ReAct, PrefillShare reaches ~380 tok/s vs ~100 for baseline, and p95 latency drops from ~500s to ~30s. The larger model amplifies the benefit because per-model KV caches are proportionally larger, making redundancy elimination more impactful.

    §7 实现 cross-reference #

    [实现未公开]

    The paper describes a vLLM-based disaggregated serving system with custom routing and cache-handoff, but no public code repository is referenced.

    关键实现细节:

    1. Cache-handoff zero-copy path: The shared $C_\text{base}$ is produced by the prefill instance and must be transferred to decode instances. In a vLLM disaggregated setup, this likely leverages NCCL or shared-memory IPC between prefill and decode GPU workers. The efficiency of this handoff is critical — Fig 4 shows PrefillShare's throughput drops at extreme concurrency (~120+ sessions), suggesting handoff becomes the bottleneck when prefill-to-decode transfer saturates the interconnect.
      1. Frozen prefill module = base model checkpoint: The prefill module is simply the unmodified base model (LLaMA 3.1-8B or Qwen3-base). No architectural modification is needed for prefill — only the decode modules require task-specific weights. This means a single prefill instance can be shared across an arbitrary number of task-specialized decoders without any model surgery.
