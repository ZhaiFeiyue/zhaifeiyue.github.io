Phase 2: 综合分析

2a. 核心三问

Q1: 这篇论文试图解决什么核心痛点/问题？

Agentic LLM 推理中，用户感知延迟（First Token Rendered, FTR）被严重放大。与传统单轮推理不同，agentic workload 在产出最终回复前需经历多轮 LLM 调用 + 工具执行的迭代链。论文通过生产级合成 trace 分析揭示三大瓶颈：(1) 工具调用占 FTR 延迟的 30-80%（尾部高达 83%）；(2) 尽管存在大量上下文复用，KV cache 命中率因 LRU 驱逐策略崩溃；(3) 顺序编排浪费了请求内部的并行化潜力。这些瓶颈的根本原因是编排器（orchestrator）和 LLM 引擎作为解耦黑盒运行，阻止了跨层优化。

Q2: 作者提出了什么新的"杀手锏"方法/架构？

提出 SUTRADHARA——一个协同设计的 agentic 推理系统，通过一个轻量 API 层将编排逻辑与 LLM 引擎调度紧密集成。核心三项优化：

1. Prompt Splitting + Tool Overlap：将 iteration $i+1$ 的 prompt 拆分为工具无关部分和工具依赖部分，在 iteration $i$ 的工具执行期间投机性地开始 prefill，工具结果到达后增量扩展

2. Streaming Tool Dispatch：在 decode 阶段通过流式 JSON 解析器识别完整的工具调用对象，立即派发执行，无需等待完整输出

3. Orchestrator-Aware Cache Management：通过语义标签（SYSTEM_PROMPT / USER_QUERY / TOOL_OUTPUT / RESPONSE / PARTIAL_PREFILL）和优先级驱动的驱逐策略，替代 workload-agnostic 的 LRU

五个新 API：submit_partial_prefill(), extend_prefill(), register_streaming_callback(), tag_kv_blocks(), set_reuse_priority()

Q3: 最终效果/结论如何？

基于 vLLM v0.11.0 实现（<3500 行代码），在 A100-80GB GPU 上使用 Qwen3-14B 和 Gemma-12B 评估：

  • 中位 FTR 延迟降低 15%，端到端延迟降低 10%（Qwen3-14B, Tool-heavy trace）
  • Gemma-12B: FTR 降低 10.74%, E2E 降低 6.39%（Tool-heavy）
  • Ablation: KV-pinning → +PS 降 7.57% FTR → +DS 再降至 11.77% FTR
  • 不牺牲吞吐量，无需修改模型架构、训练流程或推理 kernel

2b. 逻辑故事还原 — 约束推导型

#### 时代定位

2025-2026年是 LLM 从单轮推理向 agentic 多轮推理演进的关键时期。传统推理优化（continuous batching, PagedAttention, prefix caching, chunked prefill）已趋成熟，但这些优化均将每次 LLM 调用视为独立请求。当 LLM 进入 agentic 模式——迭代式 LLM+工具调用——传统指标（TTFT, TPOT）不再能反映用户体验，FTR 成为新的关键指标。本文是首个系统性分析生产级 agentic 推理工作负载的研究，标志着 serving 优化从"单请求优化"进入"多迭代请求级优化"的新阶段。

#### 背景：为什么现有方法无法解决？

现有 agentic serving 架构由三层组成：LLM Engine、Orchestrator、Tool Execution Layer。三者通过不透明的 request-response 接口通信：

  • Engine 无法知道某个请求属于哪个 agentic 工作流的哪一轮迭代
  • Orchestrator 知道 prompt 结构、工具依赖、迭代边界，但这些知识被孤立
  • 结果：engine 和 orchestrator 无法做出全局最优决策

#### 约束推导：替代方案为何失败？

替代方案为何不可行
静态 DAG 编排（Parrot, Murakkab）Tool-based agentic 应用的 DAG 结构事先未知，工具调用由 LLM 运行时动态决定
非透视调度器（Autellix）仅优化吞吐量，假设工具调用对延迟贡献可忽略——生产环境中不成立（30-80% FTR 来自工具）
工具执行时间预测（Continuum）工具执行时间变异系数 >100%，预测不实际
部分工具执行（Conveyor）生产中许多工具无法在参数不完整时执行，部分执行不可行
多 agent KV 共享（DroidSpeak, KVFlow）聚焦 multi-agent 场景的 KV 共享，不处理单 agent 多迭代的工具延迟问题
纯 CPU offload KV（LMCache, Mooncake）解决存储层级问题但不解决编排层的顺序执行瓶颈

排除以上方案后，剩余可行设计空间是：在编排器和引擎之间建立轻量协调接口，让双方共享语义信息，在不改变模型/kernel 的前提下实现跨层优化。

#### 破局："啊哈"时刻

关键洞察：50-80% 的 iteration $i+1$ 的 prompt 在 iteration $i$ decode 完成时就已可用。系统指令、对话历史、模板等不依赖工具输出，只有最后 20-50% 需要工具结果。这创造了一个自然的分割点——可以在工具执行时投机性地开始 prefill。

同时，LLM 生成工具调用时使用 JSON 数组格式，第一个 JSON 对象 } 闭合后就可以立即派发该工具，无需等待整个数组生成完毕。

#### 核心技术壁垒

Prompt Splitting 的正确性与效率协调：将 prompt 拆分为工具无关/工具依赖两部分后，引擎必须：(1) 知道在哪里拼接工具输出（正确性）；(2) 在工具执行期间保持部分 prefill 的 KV cache 不被驱逐（效率）。这需要 continuation handle 机制和 KV 块优先级标记的精确配合——partial prefill 的 KV 块被赋予最高优先级（PARTIAL_PREFILL），防止在等待工具结果期间被 LRU 驱逐。

#### 质疑假设

1. 假设1：工具无关 prompt 和工具依赖 prompt 可以清晰分割。风险：如果编排器的 prompt 模板设计不支持这种分割（如工具结果散布在 prompt 各处），则 prompt splitting 无法应用。

2. 假设2：工具调用以结构化 JSON 格式生成。风险：如果模型使用非标准格式或自由文本描述工具调用，streaming dispatch 的 JSON 解析器将失效。

3. 假设3：KV cache 容量有限，驱逐是常见操作。风险：如果 GPU HBM 足够大（如未来的 GPU），或使用 KV offload 到 CPU/SSD，驱逐压力降低，cache management 优化的收益减小。

4. 假设4：生产 trace 的特征（2 次中位迭代、20K token prompt、高系统 prompt 占比）具有代表性。风险：不同 agentic 应用可能有截然不同的迭代模式。

#### 设计绑定批判

  • 绑定1：要求编排器对 prompt 结构有完整感知——知道哪些部分依赖工具输出、哪些不依赖。通用的"黑盒" LLM API 调用模式不兼容
  • 绑定2：要求工具调用以可流式解析的 JSON 格式生成。绑定特定的 tool calling 格式
  • 绑定3：要求修改 LLM 引擎的调度器和 KV cache 管理器。虽然改动量小（<3500 LOC），但与现有引擎的抽象边界冲突
  • 绑定4：当前实现基于 vLLM v0.11.0 的 v1 scheduler，迁移到 SGLang/TRT-LLM 需要重新适配

#### 拆解：从输入到输出的具体步骤

1. 用户提交 agentic 请求 → orchestrator 接收

2. Orchestrator 构建 iteration 1 prompt（系统指令 + 用户查询）

3. 通过 tag_kv_blocks() 标记 prompt 各段语义标签，提交 LLM engine

4. Engine 执行 prefill → decode，生成工具调用 JSON

5. Orchestrator 通过 register_streaming_callback() 流式接收 decode tokens

6. 流式 JSON 解析器识别第一个完整工具调用对象 → 立即派发执行（Streaming Tool Dispatch）

7. 后续工具调用对象依次完成、依次派发

8. 同时，orchestrator 识别 iteration 2 prompt 的工具无关部分

9. 通过 submit_partial_prefill() 提交部分 prefill，返回 continuation handle（Prompt Splitting）

10. 通过 set_reuse_priority() 将 partial prefill KV 块标记为最高优先级

11. 工具结果陆续返回 → orchestrator 构建工具依赖后缀

12. 通过 extend_prefill() 拼接工具输出到 pinned KV cache

13. Engine 完成 prefill → decode → 可能继续迭代或产出最终回复

14. 最终回复的 decode token 标记为 RESPONSE（最低优先级，率先驱逐）

#### 实践上下文

  • 部署场景：生产级 agentic LLM 应用，如企业搜索、代码生成、文档检索等涉及多工具调用的场景
  • 硬件平台：A100-80GB GPU（当前评估），可迁移到其他 GPU
  • 集成点：替换现有 orchestrator（如 LangChain、AutoGen）与 LLM engine 之间的通信层，需要编排器端支持 prompt 分析能力
  • 适用条件：工具调用延迟占比显著（>30% FTR）、多工具并发（fan-out ≥2）、KV cache 压力大的场景

#### 生态影响追踪

  • 对 vLLM 的影响：直接基于 vLLM v0.11.0 实现，提出的 5 个 API 如果被上游采纳，将为 vLLM 添加原生 agentic serving 能力
  • 对编排框架的影响：LangChain、AutoGen、DSPy 等框架如果整合这些 API，可以实现从"黑盒调用"到"协同优化"的升级
  • 对 MCP 的影响：MCP（Model Context Protocol）等工具调用协议需要支持流式/增量工具执行的接口
  • 对行业的影响：Microsoft 自身的 agentic 平台（基于生产 trace 的分析）可能率先采用，推动 agentic serving 成为独立优化方向
  • 开放问题：P-D 分离架构下 prompt splitting 如何适配？多节点场景如何处理？

2d. 技术细节补充与一句话总结

#### 技术细节补充

1. KV 块驱逐优先级：$\text{RESPONSE} \rightarrow \text{TOOL\_OUTPUT} \rightarrow \text{USER\_QUERY} \rightarrow \text{SYSTEM\_PROMPT} \rightarrow \text{PARTIAL\_PREFILL}$（从先驱逐到后驱逐）

2. Workload-aware Scheduling：基于 agentic 请求的初始到达时间排序，而非单个 LLM 调用到达引擎的时间。先到的请求的后续迭代优先于后到请求的首次迭代

3. 实现规模：仅 3500 行 Python 代码，基于 asyncio 事件驱动架构

4. 心跳机制：orchestrator 和 engine 使用心跳协议做成员管理，可扩展为 Raft 等分布式一致性协议

5. 中间迭代 vs 最终迭代：中间迭代生成 token 数约为最终迭代的 1/5，中间迭代更 prefill-bound，最终迭代更 decode-bound

#### 一句话总结

Sutradhara 通过在 LLM 引擎和编排器之间建立 5 个语义协调 API，实现 prompt splitting 并行执行、流式工具派发和语义感知 KV 驱逐，将生产级 agentic 推理的 FTR 延迟降低 15%。

2e. 结构化提取

1. Core Contribution (one sentence):

提出 orchestrator-engine 协同设计范式，通过轻量 API 层实现 prompt splitting 并行、streaming tool dispatch 和语义 KV cache 管理，系统性降低 agentic 推理的 FTR 延迟。

2. Summary:

本文首先对大规模云提供商的生产级 agentic 推理工作负载进行了系统性的实证分析，揭示了三个关键挑战：工具执行占 FTR 延迟的 30-80%、KV cache 因 LRU 驱逐策略崩溃、顺序编排浪费并行潜力。这些问题源于编排器和 LLM 引擎之间的"黑盒"隔离。

作者提出 SUTRADHARA，通过 5 个新 API 实现编排器-引擎协同。三项核心优化——prompt splitting 将 prefill 与工具执行重叠、streaming dispatch 在 decode 期间增量派发工具、语义标签驱动的 KV cache 管理——协同工作降低延迟。

基于 vLLM 实现的评估显示，在 A100 GPU 上使用 Qwen3-14B 和 Gemma-12B 模型，Sutradhara 在 tool-heavy 和 iteration-heavy 两种 trace 上均取得 4-16% 的 FTR 和 E2E 延迟改善，且不牺牲吞吐量。

3. Key Findings:

  • 工具执行在尾部（p90）占 FTR 延迟的 68%，p99 高达 83%，远非"轻量 I/O"
  • 50-80% 的 iteration i+1 prompt 在 iteration i decode 完成时即可用，为并行执行提供空间
  • LRU 驱逐在并发 agentic 请求下导致级联 cache 失效，语义感知驱逐可解决
  • Prompt splitting 单独贡献 7.57% FTR 降低，加上 streaming dispatch 达 11.77%
  • 方法可泛化到不同模型（Qwen3-14B, Gemma-12B）和不同工作负载类型
  • 5 个 API 的协同设计足以实现跨层优化，无需修改模型或 kernel

4. Limitations:

  • 仅在 A100-80GB 单 GPU 上评估，未验证多 GPU / 多节点场景
  • 使用合成 trace 而非真实生产流量，虽与生产验证但不完全等价
  • 工具延迟通过比例缩放模拟，非真实工具执行
  • Prompt splitting 假设 prompt 可清晰分割为工具无关/依赖部分，依赖编排器的模板感知能力
  • 仅评估 60 个请求子集，样本量有限
  • 未与最新的 agentic serving 系统（如 Autellix, Circinus）进行直接对比
  • 吞吐量在高 QPS 下优势减弱（引擎饱和时所有系统受等待时间主导）

Phase 5: Deep Read — Framework Analysis

1. System Scope

  • Primary Goal: Serving — 降低 agentic LLM 推理的用户感知延迟（FTR）
  • Scale: 单 GPU（A100-80GB），设计可扩展但未验证多节点
  • Latency Model: Online, latency-sensitive（优化 FTR 和 E2E latency）
  • Target Workload: Tool-based agentic LLM inference（多迭代 LLM + 工具调用）

2. Architecture & Data Flow

#### 2b. Data Movement Hotspots

1. KV Cache 分配与驱逐：每次迭代的 prefill 产生大量 KV 块（~20K tokens），并发请求竞争 HBM 容量。频率：每次迭代。当前是阻塞的（驱逐后才能分配），Sutradhara 通过优先级驱逐减少不必要的重算。

2. 工具结果注入 Prompt：工具输出从 CPU 传输到 GPU 进行 tokenization 和 prefill extension。频率：每次迭代（1-20 次工具调用）。通过 extend_prefill() 增量注入，避免从头重算。

3. Streaming Token 回调：每个 decode token 从 GPU 传回 CPU 供 orchestrator 解析。频率：每个 token。轻量但高频，需要低延迟回调通道。

3. Design Space & Constraint Analysis

#### 3a. Alternative Approaches

方案可行性原因
静态 DAG 优化（Parrot, Murakkab）❌ 不可行Tool-based agent 的执行路径运行时动态决定，无法预先构建 DAG
工具延迟预测（Continuum）❌ 不实际生产工具延迟 CV > 100%，预测精度太低无法指导调度
部分工具执行（Conveyor）❌ 不可行生产工具需要完整参数才能执行，不支持 partial execution
纯 KV offload（LMCache, Mooncake）⚠️ 正交解决存储层级但不解决编排瓶颈，可作为补充方案
投机式解码⚠️ 不直接适用不解决工具执行延迟问题，仅加速 decode
纯调度优化（Autellix）⚠️ 部分有效优化吞吐但不解决工具延迟占比高和顺序编排问题

#### 3b. Constraint Derivation

  • 为何不能预测工具延迟？ 工具延迟是外部服务响应时间的函数，受网络条件、服务负载、查询复杂度影响。Figure 4b 显示 7 种工具的 CV > 100%，同一工具在不同请求上的延迟差异可达 10x+。
  • 为何不能预编译执行图？ Agentic 请求的迭代次数、每轮调用的工具类型和数量都是 LLM 运行时决定的（中位迭代 2 次，尾部 7 次；中位 fan-out 2，尾部 20）。
  • 为何 LRU 驱逐不工作？ 并发请求的第一轮 KV cache 在工具执行期间是"将被复用"的状态，但 LRU 仅看访问时间，导致级联驱逐（Figure 6）。

#### 3c. Assumption Audit

假设成立条件可能失效场景
Prompt 可分为工具无关/依赖部分编排器使用模板化 prompt 构建自由格式 prompt，工具结果散布各处
工具调用以 JSON 格式生成使用 function calling API自由文本格式的工具描述
KV cache 容量有限当前 GPU HBM（80GB）超大 HBM GPU 或高效 KV 压缩
系统 prompt 跨请求共享同类工具调用的请求高度个性化的系统 prompt
工具执行延迟显著30-80% FTR 来自工具轻量/本地工具调用

#### 3d. Core Technical Barrier

核心壁垒：Continuation Handle 机制 + 优先级驱逐的精确配合

Prompt splitting 的关键难点不在于"拆分"本身，而在于：

1. Engine 必须为 partial prefill 的 KV 块维护一个 continuation handle

2. 这些 KV 块必须被 pinned（最高优先级），在工具执行期间不被驱逐

3. 当 extend_prefill() 到达时，engine 必须找到 pinned 的 KV 块并在正确位置拼接新内容

4. 如果工具执行失败或超时，必须清理 pinned 资源

这要求在 vLLM 的 PagedAttention KV manager 中添加"部分完成请求"的状态管理——现有引擎设计中没有这个概念，因为每个请求要么在执行要么已完成。

#### 3e. Design Binding Critique

设计绑定失去的灵活性
需要编排器感知 prompt 结构无法使用通用 prompt 构建库
需要结构化 JSON 工具调用格式不兼容自由文本工具描述

5. Scheduling & Resource Management

  • Batch Formation: 使用 vLLM 的 continuous batching，chunk size = 256
  • Memory Management: PagedAttention + 语义标签优先级驱逐（替代 LRU）
  • GPU Utilization: 通过 prompt splitting 将工具等待期间的 GPU 闲置转化为 speculative prefill
  • Multi-tenancy: 通过请求到达时间排序实现隐式公平性
  • Priority Scheduling: 基于 agentic 请求级别（非单次 LLM 调用级别）的优先级

#### 7c. Bottleneck Shift Analysis


Before: 工具延迟 (30-80% FTR) + 顺序编排 + KV cache thrashing
→ After Semantic KV: KV thrashing 消除，但工具延迟仍是瓶颈
→ After PS: 部分 prefill 与工具执行重叠，但工具延迟仍在关键路径
→ After DS: 工具派发提前，减少关键路径工具时间
→ Remaining bottleneck: 不可压缩的工具执行时间（外部服务延迟）+ 高 QPS 下的引擎饱和

#### 7d. Baselines & Fairness

  • Baselines: (1) vLLM 标准调度 + LRU；(2) vLLM + request-aware scheduling
  • 公平性: 同一硬件（A100-80GB）、同一模型、同一 trace、chunk size=256
  • 规模效应: 在低-中 QPS 下优势明显，高 QPS（引擎饱和）下优势减弱
  • 框架开销: 5 个 API 的调用开销、流式 JSON 解析器开销——论文未量化
  • Baseline 可能更优的场景: 工具延迟极短（本地工具）、单迭代请求（退化为传统推理）

8. API & Usability

  • API 兼容性: 非标准 OpenAI 接口，5 个自定义 API 调用
  • 模型格式: 使用 vLLM 支持的任何格式（HuggingFace）
  • 部署: 需要改造 vLLM scheduler 和自定义 orchestrator
  • 配置复杂度: 需要设置语义标签规则、prompt 分割策略——较高
