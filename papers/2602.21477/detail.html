Pancake: Hierarchical Memory System for Multi-Agent LLM Serving

Zhengding Hu, Zaifeng Pan, Prabhleen Kaur, Vibha Murthy, Zhongkai Yu, Yue Guan, Zhen Wang, Steven Swanson, Yufei Ding (UCSD) | 2026-02 | https://arxiv.org/abs/2602.21477 Category: framework | Tags: agent-memory, multi-agent, ANN, vector-db, ivf, gpu-cpu-coordination, rag, memgpt Read: 2026-04-18

Core Contribution

Pancake 是首个针对 多 agent 长期记忆 设计的多层级动态 ANN (approximate nearest neighbor) 索引系统——通过 (i) 基于 FSM 的多级 cluster 缓存、(ii) 跨 agent 的 hybrid-graph 统一粗索引、(iii) GPU-CPU 协同的热点簇管理，把 agent 记忆操作从占端到端 82% 的开销压缩到 3.2%，端到端吞吐平均提升 4.29×。

Summary

Motivation. LLM agent 越来越依赖一个"外部长期记忆"数据库（Mem-GPT、A-Mem、LangChain、LlamaIndex 都有），里面存着 user profile、action history、外部知识等 embedding 向量。每一步生成都要对这个数据库做 ANN 搜索 + 插入。问题是：① 记忆规模大（MS MARCO 8M 条 passage，>100GB）；② 更新频繁（interleaved search-insert，批次极小）；③ 多 agent 并存（每个 agent 维护自己的 memory，但也要 search 别的 agent 的）。现有 vector DB（Faiss、Milvus、SPFresh、Quake、DiskANN）都是为静态 / batch-update 的传统数据库场景设计的，在 agent 场景下跑到 memory op 占 82% 以上执行时间。

Method. Pancake 的设计是"多层级 × 多 agent × 多设备"三个正交维度的协同：

  1. Pattern-Driven Multi-Level Index Cache (§4.2)：把 IVF cluster 组织成 3 层 L0/L1/L2 缓存，上层是下层的 subset 但用更贴合 agent 访问模式的方式聚类；用 FSM (Finite-State Machine) 对 agent 的 memory access 序列建模（state = semantic cluster，transition = step 间跳转），据此做 reordering、prefetching、early termination。
  2. Multi-Agent Hybrid Graph Indexing (§4.3)：把每个 agent 的独立 HNSW coarse index 通过 跨 graph 的 inter-connection portal 合并成一个 hybrid graph，一次 BFS 就能遍历多个 agent 的 coarse index；每个静态 cluster 上加 agent profile（per-agent 访问过的 vector ID 有序列表）处理非均匀 fine-search 访问模式。
  3. GPU-CPU Coordinated Dynamic Index (§4.4)：GPU 缓存 hotspot cluster，但由于 CUDA 不擅长动态 list 扩展，Pancake 把 insert 先放到 CPU-side per-cluster buffer（≤128 条），search 时 GPU（缓存部分）+ CPU（buffer）并行算然后 merge，buffer 满就异步把整个 cluster 搬到新 GPU 区再释放旧区；还把 K-means cluster splitting 也 onload 到 GPU。

    4. Results. 在 UltraChat / UltraFeedback / Prm800k / Gsm8k / APIGen / AgentGym 六个数据集 + Llama3.1-8B/70B + GPT-5 上，相比 Mem-GPT / A-Mem / LangChain / LlamaIndex，end-to-end 吞吐提升 1.12×–26.18×（平均 4.29×）；memory-op 时间占比从 >82% 降到平均 3.2%。对比独立 vector DB（Quake/SPFresh/DiskANN）query 吞吐提升 1.9×–4.2×，加 GPU 再 2.2×（合计 3.9×）。多 agent 扩展到 20 个 agent 时 degradation <10.2%；coarse search cost 比独立 index 降低 >20×。

    Key Findings

    • 高维 shell 效应导致 in-place insert 散布：insert 100 个请求的记忆项会散落到最多 175 个 cluster 里，38%–100% 的这些 cluster 访问频率 <5%（§3.1, Fig 4a）。
    • Agent 记忆有两层 locality：intra-agent（同一 agent 的 memory 语义一致）和 inter-request step-wise（不同 request 的同一 reasoning step 访问同一 memory 区）——单独一个 dedicated cluster per agent 能处理前者但抓不住后者（§3.1, Fig 5）。
    • 多 agent coarse search 成本随 agent 数线性上涨：独立索引 20 个 agent 时 coarse search 占总 latency 80%+（§3.2, Fig 7a）。
    • GPU vs CPU 的 cluster size 交叉点约 256–512：cluster <256 时 CPU 快（kernel launch overhead 主导），≥512 时 GPU 有 3× 以上 speedup（§3.3, Fig 8a）。
    • CPU-side insertion buffer 大小 = 128：这是"CPU 搜 128 条的时间 ≈ GPU 搜单个 cluster 的时间"的临界点，保证并行执行不增加端到端延迟。
    • 静态 index 跑 search-update 混合时 recall 显著掉：IVF 在 search-update 模式下要扫 128 个 cluster 才能 recall>0.9，因为 insert 散布破坏 locality。

    Limitations

    • Embedding 模型固定：所有实验用 E5-1024，未验证不同 embedding 维度 / 模型下 αet、αic、Binsert 这些 magic number 是否需要重新调。
    • 8M 条静态知识库是中等规模：文中提到"often over 100 GB"但实验只做到 MS MARCO 8M；>100GB + 20 agent 的真实 scale 没有实测。
    • FSM 构造启发式：state merge 用 d_merge 门限、新 FSM 创建走"逐 access 一个 state + merge"，没做理论保证；pattern 急剧漂移的 agent 上可能退化成每个 request 一个 FSM。
    • GPU-CPU async 路径的一致性窗口：cluster 在搬迁过程中的 search 会分别打到新 / 旧区域，论文给出"merge 结果"描述但没量化短窗口内 recall 漂移；也没讨论多 agent 同时触发 migration 时的 GPU memory 碎片。
    • 工作负载 trace 是合成：4 种 access pattern 是手工枚举 + 现有数据集映射的；没有来自真实 production agent（e.g. ChatGPT Memory / Claude Projects / Copilot）的 trace。
    • 没有和 RAG-style cache（RAGCache / HedraRAG）头对头比：虽然 cite 了，但 Fig 15 里主要和 vector DB 本身比，agent-serving 综合栈的对比缺。
    • 多层 cache 的空间开销没有明确给：L0/L1/L2 三份（部分）数据在 CPU 上占多少额外内存，没有 table。

    Infrastructure Impact

    • Algorithm: 把"FSM 状态机建模 agent 访问"提出来作为一个可学习对象——未来可以把 FSM 替换成学出来的 transformer/markov 模型；也给 online RL memory 提供了"低开销状态摘要"的 baseline。
    • Kernel: 需要一个 GPU K-means kernel（on-GPU cluster split）、一个 multi-stream async GPU-CPU copy path、以及一个 CPU-GPU merge kernel（batch 合并 top-k 结果）。整体 kernel 工程量不大，但 async migration 的 stream 排布是关键。
    • Framework: 直接对 LLM serving 框架（vLLM / SGLang）提出一个新插槽——"agent memory manager"，与 KV cache manager 并列。Pancake 已经给出了 LangChain / LlamaIndex / Mem-GPT 的接入接口，未来 vLLM 的 agent-mode 扩展可以把这个层吸纳进去。
    • LLM: 对 LLM 本身无改动；但对"memory-augmented generation"类 agent（Mem-GPT, A-Mem, GMemory, Mem0）有直接替换价值。
    • Agent: 为 multi-agent 框架（AutoGen / CAMEL / AI Town / LangGraph）提供了真正可 scale 的 shared-memory 后端；之前这些框架的 memory 是玩具级实现。
    • Code: 论文未公布代码，但实现上需要：FSM table with lock-free update、HNSW 改造支持 cross-graph portal、CPU 侧 lock-free insertion buffer、GPU 侧 ring allocator + dirty-bit tracking。

    Deep Analysis (framework)

    1. System Scope

    • Primary goal: online serving of memory-augmented LLM agents（非 training，也不是 offline batch）。是一个介于"vector DB library"和"LLM serving framework"之间的中间件。
    • Scale target: 单节点 CPU-GPU（AMD EPYC 64c + 8×H100 80GB），可扩展到多 agent（论文到 20 个），但没有多节点。
    • Online, latency-critical：每步 LLM generation 之间插入 ANN 查询，memory op 直接影响 end-to-end latency。
    • Workload: memory-based agent inference。包括 multi-turn dialog、long-CoT、tool-calling、persona-driven generation。LLM 本身不是本论文关注点，跑 vLLM + Llama / GPT-5 API 就行。

    2b. Data Movement Hotspots

    1. CPU-GPU cluster migration（on buffer full）：每 128 条 insert 触发一次 async copy，整个 cluster（几百-几千条 × 1024-d × 4B ≈ MB 量级）过 PCIe。通过 async allocate + overlap with serving 隐藏。
    2. 每次 search 的 query embedding 传 GPU（1024 × 4B = 4KB）：量小但频次高，kernel launch 开销主导（这也是为什么 cluster <256 时不如 CPU）。
    3. Cluster split on GPU：K-means 时需要把 cluster 所有向量留在 GPU（假设已在），避免 CPU → GPU 再搬；论文指出 "split 的 cluster 通常正好是 hotspot 已经缓存在 GPU 上的" —— 这是 locality 赋予的隐性收益。

      3b. Constraint Derivation（extends Phase 2）

      为何不可单 IVF + in-place insert？

      • 高维欧氏空间的 shell 效应（Beyer 1999）：大量点集中在 hypersphere 表面附近，点到 centroid 的距离分布方差很小，细微语义差别转成大距离差。所以即使两个向量 cosine 相似度 0.9，插入时它们计算出的 nearest centroid 可能不同，分散到几十个 cluster。
      • 后果：search 时要 scan 几十倍的 cluster 才能找回相关向量，recall 还掉。论文观测：100 请求散到 up to 175 cluster，38%-100% cluster 访问率 <5%。

      为何不可 per-agent dedicated cluster（intra-agent locality naive 解法）？

      • Step-wise locality 的存在：tool-calling agent 的 planning / tool-call / reflection 三步会访问 memory 空间的三个不同区域。单一 dedicated cluster 的 centroid 变成"三个区的加权平均"，对所有步都不代表；PCA 可视化（Fig 5b）出现 3 个明显 sub-cluster。
      • 所以至少要 3 级层级——L0 抓 step-wise 热点，L1 抓 intra-agent 扩展邻域，L2 抓全局 coarse。

      为何不可独立 HNSW per-agent（多 agent）？

      • HNSW greedy descent 单图 O(log N)，N 图就 O(N log N)。但 N 只有 20 这个常数因子就已经让 coarse search 占到 80%+ 总 latency（Fig 7a）。更关键：跨 agent 的 top-k 要 N 个图各自算 top-k 再合 merge，无法做 early termination 跨图传播。
      • 所以需要把多图缝成一张图——hybrid graph。

      为何不可把 index 全放 GPU？

      • Vector DB 100GB 量级（论文 cite Wikipedia dump），H100 只有 80GB；LLM Llama-70B 权重 + KV cache 再吃掉 70GB+。剩余 GPU memory 不够装全量 index。
      • 所以 GPU 只能当 hotspot cache。

      为何不可 GPU 上直接动态 list 扩展？

      • CUDA malloc 是 per-stream 的同步原语，无 lock-free 动态 list primitive；hotspot cluster 一天插入几万次，每次 GPU malloc+copy 代价巨大。
      • 所以 insert 必须 stage 在 CPU（insertion buffer），async 批量迁移。

      为何 Binsert = 128？

      • Fig 8a 给出 cluster <256 时 CPU 比 GPU 快。Binsert 设置为"CPU 搜 Binsert 条的时间 = GPU 搜一个 cluster 的时间"——保证 search 时 CPU buffer 部分的计算被 GPU 的 cluster 计算完全 cover，不拖慢端到端。
      • 若 Binsert>128：CPU 部分成为瓶颈；<128：async migration 频次过高，transfer 成本回升。

      3c. Assumption Audit

      假设依据何时可能失效
      Agent 访问存在 intra-agent + step-wise localityFig 4b, Fig 5a 的 similarity 分布随机探索 agent（纯 search, 无 step structure）；topic 频繁漂移的 dialog
      FSM 有 Np=小常数个即可覆盖所有 pattern经验值，论文未给 ablation多模态 agent、workflow 分支极多时 FSM 表爆炸或频繁 merge 抖动
      Hot cluster 重合于要 split 的 cluster基于"split 总是发生在高频插入 cluster"的假设如果 insert 是均匀的而 search 集中在少数 cluster，split 可能发生在 cold cluster
      αet = 0.6~0.8 给出好 trade-off经验值recall 要求 >0.95 时可能需要 αet 更高，失去 early termination 收益
      GPU cache 可以静态识别 hotspot论文用"预定义 GPU budget + access frequency"workload 冷启动 / 剧烈漂移时 cache miss 率飙升

      笔者对 FSM 能"online 识别 step-wise pattern"这一点持谨慎态度——论文的 sim 公式只是"prefix alignment × δ/(1+|c-v|)" 的加权求和，不是真正的序列学习，面对长 chain-of-thought 或嵌套 sub-task 可能退化成不如简单 LRU cache。

      3d. Core Technical Barrier

      最核心的壁垒是：per-cluster CPU insertion buffer + async GPU cluster migration 的实现正确性 + overlap 技术。

      这不是高层 idea（高层 idea 是"CPU 负责 dynamic，GPU 负责 hot static"），而是下面这个工程细节链条：

      1. buffer 要 lock-free（insert 和 search 并发）
      2. async migration 期间要维持"search 看到的 buffer + cached 数据"的逻辑一致（新 vector 不能被跳过也不能被重复）
      3. CPU 搜 buffer + GPU 搜 cluster 的结果 merge 要 deterministic（避免 top-k 在"已迁移一半"的 window 里漂移）
      4. 旧 GPU cluster 释放的时刻要所有 in-flight search 都完成——需要一个 reference counting 或 epoch-based reclamation

        5. 这一整套是 systems-engineering 级别的工程壁垒，做"idea 级别复现"容易，做到"真正不掉 recall 不掉 throughput"要几个月。类似 vLLM PagedAttention 的 copy-on-write 也是这一类壁垒。

        3e. Design Binding Critique

        Pancake 强制绑定了几件事：

        • GPU-CPU 异构平台：纯 CPU 或纯 GPU 服务器用不上 §4.4。
        • IVF / 可划分 cluster 的索引：HNSW + 纯 graph 的 vector DB（不分 cluster）不兼容这套 cache 层级。
        • Embedding 已由外部 model 产出：内嵌编码不在本系统范围——embedding drift 不归它管。
        • Agent 访问有 locality：无 locality 的 workload（e.g. 纯随机 knowledge probing）收益退化到 baseline。
        • LLM 和 memory 共享同一 GPU：跨节点 disaggregation（like Mooncake 那种分离式）下 GPU-CPU 协同的拓扑要重做。

        若 "LLM serving" 本身改成 P-D disaggregation（prefill 和 decode 不同 GPU），Pancake 要决定：memory GPU cache 放 prefill 节点还是 decode 节点，或者再搞一个专用 memory GPU，不平凡。

        5. Scheduling & Resource Management

        • Batch formation: 相邻同类型 op（search / insert）在 API 层 group 成 batch，提升 GPU 利用率。
        • Memory management: 三级 index cache 是 CPU-side 缓存；GPU 是 hotspot cluster cache + LLM KV cache 共存；async migration 用 ring allocator 避免 fragmentation。
        • GPU utilization: search 时 GPU 算 cached cluster, CPU 算 buffer，两者并行——避免 GPU idle。
        • Multi-tenancy: 通过 agent profile 做 per-agent 隔离；shared read / exclusive write 锁保证并发正确性。
        • Priority / SLO: 论文未明确讨论 SLO-aware scheduling，也没有 priority queue。这是 framework 层留给下游的口子。

          7c. Baselines & Fairness

          • Agent-side baselines（A-Mem / Mem-GPT / LangChain / LlamaIndex）都是现成 Python lib，和 Pancake 共享 vLLM backend，公平。
          • Vector DB baselines（Quake / SPFresh / DiskANN）用 Pancake 生成的 trace feed，公平但给 Pancake "主场"——trace 本身就是 agent access pattern，对 Pancake 的 FSM 更友好。
          • 输入 lengths: 没有在表里明确给，是否所有 workload 同一 request rate 没说。
          • 硬件: 8 × H100 80GB + EPYC 64c，充足；但 Pancake 单节点设计，多节点实验完全缺。
          • baseline 什么时候 win？论文没给——应给一个"什么 workload 上 Pancake 不赢 / 甚至输"的反例。

          8. API & Usability

          • Python 接口：search / insert / update / delete + memory scope specifier。兼容 Faiss 初始化。
          • 无 OpenAI-compatible HTTP server（Pancake 是 library，不是 serving server）。
          • LangChain / LlamaIndex / Mem-GPT 接入：现成 adapter。
          • 配置旋钮：Np (pattern num), NS (FSM states), dmerge, αet, αic, efconnect, efsearch, Binsert, GPU cache budget——大约 8 个需要调，不轻量。
