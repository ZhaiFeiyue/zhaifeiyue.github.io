Sarathi-Serve (2403.02310) — L3 跨论文综合 #

Target: Taming Throughput-Latency Tradeoff in LLM Inference with Sarathi-Serve Mode A（per-paper synthesis）。本篇是一篇在线推理调度系统论文，其真正的 直接对手只有一个（Splitwise）。related 集合里其余 7 篇同属 category: model 但分属不同 sub-topic（基础模型、agent 范式、KV/attention 机制、安全、长上下文、 MoE）。本文诚实标注这个"相关度梯度"，把泛泛类比的稀释降到最低。

1. 相关论文（谁相关、为什么相关） #

按与本篇的机制耦合强度分三层，而非平铺：

强相关（同一问题：prefill/decode 两阶段异构 → 调度） #

  • Splitwise (2311.18677) — 唯一真正的正面对手。二者共享同一根命题：prefill

    • 是 compute-bound、decode 是 memory-bound，二者混跑会互相干扰

    [2311.18677]。但解法方向相反：Sarathi-Serve 在**同一

    replica 内**用 chunked-prefill + stall-free batching 把两阶段"融合"到一个

    hybrid batch [2403.02310]；Splitwise 把两阶段"拆开"到

    不同机器池并通过 InfiniBand 传 KV-cache [2311.18677]。

    本篇 L1 §6 明确把 Splitwise 归为"disaggregated"第三类调度器并把定量对比留给

    future work [2403.02310]。这是 delta 分析的主战场（见 §2）。

中相关（decode 阶段的显存/带宽约束，与本篇 memory-bound 论点同源） #

  • StreamingLLM (2309.17453) — 从 decode 侧的 KV-cache 约束切入。本篇靠

    • "decode 是 memory-bound、有算力 slack"立论 [2403.02310]，

    StreamingLLM 则靠"decode 的 KV footprint 随序列线性增长"立论

    [2309.17453]。两者是同一物理瓶颈（HBM 带宽/容量）的两面：

    Sarathi 关心"如何在 memory-bound iteration 里塞更多计算"，StreamingLLM 关心

    "如何让 KV 不无限膨胀"。正交且可叠加。

  • DeepSeek-V2 / MLA (2405.04434) — 从模型架构侧压 KV-cache（93.3% 压缩）

    • [2405.04434]。本篇 L1 §2.2 已把 GQA/MQA 列为"缓解

    KV 瓶颈、放大 batch"的互补技术 [2403.02310]；MLA 是这一路线的

    更激进版本。二者正交：MLA 让每 token KV 更小 → 同显存下 batch 更大 → 本篇的

    decode slack 论证更成立。

弱相关（同 category model，但 sub-topic 不同，仅提供对照坐标系） #

  • GPT-3 (2005.14165) — 定义了本篇要服务的工作负载形态：dense

    • decoder-only、autoregressive、prefill+decode 两阶段的鼻祖，KV-cache

    ~4.7 MB/token [2005.14165]。是"被服务对象"而非

    "同类方法"。

  • ReAct (2210.03629) — agent 范式，多步 Thought/Action/Observation 会显著

    • 拉长上下文、制造长 prefill + 多轮 decode [2210.03629]，

    恰是本篇 stall-free 调度最能获益的下游负载模式。属"负载生成者"。

  • 2404.16811 (FilM / 长上下文) — 训练出真正能用满 32K 上下文的模型，直接

    • 推高 prefill 长度与 decode KV footprint [2404.16811]，

    加剧本篇要解决的 generation-stall 问题。属"负载放大器"。

  • 2403.06833（指令/数据隔离）、2404.13208（指令层级） — 安全/对齐论文，与

    • 推理调度无机制交集。仅共享 category 标签。本篇对其无可引用的技术断言，

    下文不强行类比（避免稀释 evidence chain）。

本篇的增量：把前作 Sarathi（纯 throughput 优化）的 chunked-prefill 机制升级为

stall-free batching——先装满所有 decode、再放一个 ongoing prefill chunk、

最后在 token budget τ 内接纳新请求，保证 decode 永不被打断

[2403.02310]。这是"从吞吐单目标 → 吞吐+TBT 双目标"的跨越。

互补而非互斥的坦白：本篇 L1 §6 亲口承认 disaggregation 的 TTFT 更好

（full prefill 比 chunked prefill 快），只是需要 KV 迁移 + 浪费 prefill 机的显存

[2403.02310]。Splitwise 侧也承认在 prompt:output 比例均衡时，

mixed pool 会占主导、系统"退化成 iso-count baseline"[2311.18677]。

两篇的真实关系是"低带宽/单 replica 选 Sarathi、高带宽/prompt-heavy 选 Splitwise"，

而非谁淘汰谁。

vs StreamingLLM / MLA —— 正交叠加，非竞争 #

本篇的 decode-slack 论证依赖"decode 是 memory-bound"[2403.02310]。

  • MLA 把每 token KV 从 $2n_h d_h$ 压到 $(d_c+d_h^R)\approx4.5d_h$

    • [2405.04434] → 同显存能容纳更大 decode batch → decode

    的算术强度更低（更 memory-bound）→ Sarathi 能 piggyback 的 prefill token

    更多。即 MLA 让本篇论点更强。

  • StreamingLLM 让 KV footprint 恒为 4+L[2309.17453]，

    • 同样放大可容纳的并发 → 与本篇正交。

没有矛盾：三者从不同层（调度 / 架构 / cache 管理）攻击同一 memory-bound

现实，可同时启用。

vs GPT-3 / ReAct / FilM —— 本篇是它们的"运行时" #

这三篇制造/服务本篇要处理的负载：GPT-3 定义两阶段推理形态、ReAct 制造多轮长

上下文、FilM 让 32K 上下文真正被用满。本篇没有对它们做技术改进，delta 方向是

"我服务你们"而非"我改进你们"。
