HexAGenT: Efficient Agentic LLM Serving via Workflow- and Heterogeneity-Aware Scheduling #

You Peng, Youhe Jiang, Wenshuang Li, Xu Xu, Ke Zhou, Jiawei Jiang, Chen Wang, Binhang Yuan | 2026-05 | arXiv 2605.16637 Category: algorithm | Tags: agentic-serving, workflow-scheduling, dag-scheduling, heterogeneous-cluster, pd-disaggregation Read: 2026-05-25

§1 TL;DR #

HexAGenT 把 agentic workflow 建模为 online-revealed DAG，用 projected scaled-SLO risk 排序 + 联合 prefill/decode placement，在异构 A100/H100/H200 P-D 分离集群上平均降低 Req95 20.1%、Req99 33.0%，最大分别达 45.0%/80.5%。

§2 Q1 / Q2 / Q3 #

Q1 痛点 #

Agentic LLM 应用已从单请求转为多步 workflow（planning → tool-use → branching → synthesis），用户感知的是 workflow 端到端延迟而非单次 LLM 调用延迟。但现有 serving 系统（vLLM、SGLang）和 P-D 分离架构（DistServe、Splitwise）仍以 per-call 粒度调度，忽视三个交叉挑战：

  1. 在线不确定性——workflow DAG 在运行时逐步揭示，到达时只有源节点可见
  2. 资源异构性——同一 LLM 调用在不同 GPU 类型上 prefill/decode 延迟差异显著，KV 传输带宽也不同
  3. workflow 结构冲突——per-call FCFS / load-balancing 优化的是局部目标，可能阻塞 workflow 关键路径

    4. Paper Figure 1：三种 agentic workflow 模式——链式 agent-tool-agent、bounded self-refinement、parallel fan-out。只有源节点在 workflow 到达时可见，后续节点在父节点完成后才揭示。这一 online-revealed 特性使得经典 DAG 调度不可直接套用。

    Q2 方法 #

    核心机制：projected scaled-SLO risk prioritization + joint heterogeneous P-D placement。

    具体分为三层：

    1. Online workflow horizon $H_w(t)$：为每个 workflow 维护 standalone 完成时间估计（若独占集群），随新节点揭示和真实运行时间逐步修正
    2. Projected ratio $R_s(c,t)$：将每个 ready call 的排序指标统一为归一化的 SLO 风险值——值越大表示 workflow 越接近超时
    3. Greedy joint planner：贪心地选择 projected risk 最高的 call，枚举所有可行 prefill-decode pair，选择使 projected decode finish 最早的组合，并更新模拟状态后继续选下一个 call

      4. 核心技术壁垒：projected-ratio surrogate 将异构 workflow 的紧迫度归一化到同一量纲。不同 workflow 有完全不同的 DAG 结构和 standalone 完成时间，直接比较 absolute slack 或 remaining time 不公平；除以各自的 $H_w(t)$ 后，scheduler 可以在混合 workload 中正确识别哪个 workflow 真正「快超时了」。这个归一化 + 在线修正的组合是 baseline（FCFS、LLF、ATLAS）都没有的。

      Before (per-call / workflow-unaware)After (HexAGenT)
      按 call 到达顺序排队（FCFS）或按 attained service 排队（ATLAS）按 workflow 级 projected ratio $R_s(c,t)$ 排队
      Prefill/decode placement 按 load-balance 分配联合枚举 P-D pair，选 projected decode finish 最早的
      Workflow 结构对 scheduler 不可见维护 online DAG + progressive horizon refinement

      Q3 结果 #

      • 异构集群（A100/H100/H200 8P+8D / 10P+10D）：avg Req95 减少 13.0%，avg Req99 减少 24.5%；最强单点 Qwen-Mixed Req99 从 8.96 降至 3.94（56.0%）
      • 同构集群（H200 4P+4D / A100 4P+4D）：avg Req95 减少 24.0%，avg Req99 减少 35.4%——说明 workflow-aware 排序本身是主要收益来源
      • 鲁棒性：30% 估计误差下 Req99 退化 ≤1.5%（Llama）/ ≤9.5%（Qwen），非单调
      • 开销：per-invocation ≤23.7 ms，异步 apply 不阻塞在飞 call

      §3 架构 / 方法图 #

      Paper Figure 2：HexAGenT 四组件架构。Workflow front-end 接收请求并维护 online DAG 状态；Global Scheduler 包含 State Collector → Estimator → Joint Planner → Plan Dispatcher 四模块；P-D Serving Cluster 由异构 prefill pool 和 decode pool 通过 KV Transfer Network 连接；External Services 提供 LLM weights 和 tool API。事件驱动的反馈环路使 scheduler 在每次 prefill/decode/transfer 完成时重新评估。

      调度算法数据流 #

      graph TD A[Event: workflow arrive / prefill done / decode done] --> B[Update queues & workflow horizons H_w] B --> C{Stage triggered?} C -->|Prefill| D[Enumerate ready calls in prefill queue] C -->|Decode| E[Enumerate ready calls in decode queue] D --> F["For each call c: project best P-D pair, compute R_P(c,t)"] E --> G["For each call c: compute R_D(c,t) with capacity check"] F --> H["Greedy: pick c* = argmax R, assign, update sim state"] G --> H H --> I{Queue empty?} I -->|No| F I -->|Yes| J[Async apply plan to still-waiting calls]

      关键设计决策 #

      • No preemption/migration：一旦 call 开始 prefill 或 decode，placement 固定不可撤销。收益全部来自初始决策质量。
      • Joint P-D planning at prefill time：prefill scheduler 同时选定 decode instance，建立 KV 传输路由。在异构集群中这避免了跨硬件类型高延迟传输（除非 service-time 收益 justify）。
      • Asynchronous plan application：scheduler 在后台计算，serving 不停；plan 只 apply 到仍在等待的 call。

      方程与物理意义 #

      Eq 1 — Scaled-SLO 目标：

      $$\min_{\pi}\;\alpha \quad \text{s.t.} \quad \frac{1}{|\mathcal{W}|}\sum_{w \in \mathcal{W}} \mathbb{1}[C_w^\pi \leq \alpha H_w] \geq \tau$$

      物理意义：在目标 attainment $\tau$ 下找到最小的 SLO 放大系数 $\alpha$。$H_w$ 是 workflow $w$ 独占集群时的 makespan，$\alpha H_w$ 是实际 SLO deadline。$\alpha = 1$ 意味着即使共享集群也能在独占时间内完成——不现实；$\alpha$ 越小说明调度越好。与固定 SLO 不同，此 metric 自动归一化了不同 workflow 的内在复杂度差异。

      Eq 2 — Projected ratio：

      $$R_s(c,t) = \frac{(t - a_w) + \Delta_s(c,t)}{H_w(t)}$$

      物理意义：分子 $(t - a_w) + \Delta_s(c,t)$ 是 workflow 已经消耗的时间加上当前 call 的预计剩余耗时，分母 $H_w(t)$ 是当前 standalone horizon。$R_s > 1$ 意味着 workflow 已经超出目标——极度紧急。这个 ratio 在不同 DAG 深度、不同 standalone makespan 的 workflow 间提供了可比较的紧迫度量纲。

      Eq 3-4 — Decode 容量约束：

      $$m(c) = L_\text{in}(c) + \widehat{L}_\text{out}(c)$$

      $$m(c) \leq \text{Cap}(d)$$

      物理意义：admission control——call 的总 KV-cache 需求（输入 + 预测输出）不能超过 decode instance 的剩余容量。这防止 OOM 并引导 scheduler 将长输出 call 分配到 HBM 更大的 instance。

      六项检查 #

      1. 目标函数完备性：Eq 1 直接优化 workflow-level tail attainment，不是 average latency 或 per-call metric ✓
      2. 变量可控性：$\pi$ 控制 call 排序和 P-D placement，这是 scheduler 实际可操控的决策变量 ✓
      3. 假设合理性：$H_w(t)$ 假设 standalone execution 可近似，用 roofline 模型估计——合理但有精度限制（Table 5 量化了误差影响）✓
      4. Surrogate 一致性：$R_s(c,t)$ 作为 Eq 1 的 surrogate——贪心 argmax 不保证全局最优，但经验上有效 ⚠️ 无最优性证明
      5. 约束完整性：Eq 3-4 的 decode capacity 约束在 greedy placement 内强制执行 ✓
      6. Progressive refinement 收敛性：$H_w(t)$ 单调增（新揭示的 call 只能延长 makespan）且最终收敛到真实 $H_w$——论文未明确证明单调性但逻辑上成立 ✓

        7. Algorithm-specific 分析 #

        无形式化作者证明 — 仅实证。论文没有给出关于 projected-ratio greedy 的竞争比、近似比、或收敛性定理。理想情况下应提供：(a) 在 workflow DAG 为 chain 的特殊情况下，greedy 相对于 offline optimal 的 competitive ratio bound；(b) $H_w(t)$ 估计误差对最终 $\alpha$ 的 sensitivity bound；(c) greedy joint P-D assignment 的近似比（类似于 makespan minimization on unrelated machines 的已知 2-approximation）。

        §7 实现 cross-reference #

        实现基础 #

        • 基于 SGLang v0.5.9 的 P-D disaggregated serving feature，scheduling 逻辑在 gateway/worker 路径上集成
        • 主要修改集中在 router 侧（workflow 元数据解析、stage-state 构建、异步 plan apply、completion feedback）和 worker 侧（暴露 P-D snapshots、接受 priority update、completion telemetry）
        • Event-driven simulator：~4.6K 行 Python，覆盖完整 call lifecycle（waiting prefill → prefill → KV transfer → waiting decode → decode → completion），用 roofline-style 延迟模型

        代码状态 #

        [实现未公开]——论文未提供公开代码仓库链接。最接近的可参考实现：

        • SGLang 的 P-D disaggregation（sglang/srt/）提供 substrate
        • Autellix/ATLAS（luo2025autellix）提供 program-aware baseline 的开源参考
        • Proxy model-based output length prediction（qiu2024proxy）用于 $\widehat{L}_\text{out}(c)$

        关键实现细节 #

        1. Asynchronous plan application：scheduler 和 serving 解耦——plan 计算中 serving 继续执行，plan 完成后只 apply 到 仍在等待的 call（已开始 prefill/decode 的不受影响）。这避免了 lock contention 和 head-of-line blocking，但引入了 plan 过时的风险（部分 call 在 plan 计算期间已被 dispatched）。
          1. Progressive horizon refinement：$H_w(t)$ 在每次新 call 揭示或真实运行时间更新时重计算。由于新揭示的 call 只能延长 critical path，$H_w(t)$ 是非递减的——workflow 的 SLO 目标只会放松，不会变紧。这个单调性是 scheduler 稳定性的隐含保证。

            2. 算法分类特定分析 #

            Problem formulation #

            目标函数：

            $$\min_{\pi}\;\alpha \quad \text{s.t.} \quad \frac{1}{|\mathcal{W}|}\sum_{w \in \mathcal{W}} \mathbb{1}[C_w^\pi \leq \alpha H_w] \geq \tau$$

            假设：

            • Workflow 独立到达（no inter-workflow dependency）
            • DAG 结构事先未知，online revealed（非 Markov——依赖完整历史 $G_w(t)$）
            • Service time 可用 roofline 模型近似估计
            • KV-cache 容量是硬约束（不可 swap/evict）
            • 一旦 call 开始执行不可 preempt/migrate

            一步算法输入/输出：

            • Input：当前所有 workflow 的 revealed frontier calls $\{c : c \text{ is ready}\}$ + 集群状态 snapshot $S(t)$
            • Output：每个 ready call 的 (prefill instance, decode instance, priority rank) 三元组

            Reproducibility #

            • 调度代码未开源。~4.6K 行 Python simulator 也未发布
            • 最接近的开源参考：SGLang P-D disaggregation + Autellix ATLAS baseline
            • 社区尚无复现报告（论文刚发布）
            • 核心算法（Algorithm 1）伪代码完整，可独立实现
