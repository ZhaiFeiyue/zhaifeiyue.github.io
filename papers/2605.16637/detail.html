HexAGenT: Efficient Agentic LLM Serving via Workflow- and Heterogeneity-Aware Scheduling #

You Peng, Youhe Jiang, Wenshuang Li, Xu Xu, Ke Zhou, Jiawei Jiang, Chen Wang, Binhang Yuan | 2026-05 | arXiv 2605.16637 Category: agent | Tags: agentic-serving, workflow-scheduling, online-dag, heterogeneous-cluster, pd-disaggregation, workflow-slo Read: 2026-05-25

§1 TL;DR #

HexAGenT 把 agentic workflow 建模为 online-revealed DAG，用 projected scaled-SLO risk 排序 + 联合异构 P-D placement，在 A100/H100/H200 集群上平均降低 Req95 20.1%、Req99 33.0%，最大 45.0%/80.5%。

Agent Scope #

  • Task class: 本文不是 agent 本身，而是 agent workflow 的 serving-side scheduler。服务对象覆盖 open-ended（ReAct 链式推理、LATS 树搜索）和 closed（BFCL-v3 function calling）两类 agentic workload。
  • Interaction pattern: multi-turn workflow，每个用户请求触发一条由多个 LLM 调用组成的 DAG；DAG 结构在运行时 online revealed（父节点完成后子节点才可见）。
  • Autonomy: scheduler 本身是全自动的——event-driven 无人干预；它所服务的 agent 可以是 fully autonomous（LATS）或 human-in-the-loop（conversational chains）。
  • Scope boundary: HexAGenT 不涉及 agent 的 planning/reasoning/tool-selection 策略，只关心「给定 agent 产出的 LLM call 序列，如何在异构 P-D 集群上调度以最小化 workflow 端到端 SLO violation」。

§2 Q1 / Q2 / Q3 #

Q1 痛点 #

Agentic LLM 应用的基本单元已从单请求变为多步 workflow（planning → tool-use → branching → synthesis），用户感知的是 workflow 端到端延迟而非单次 LLM 调用延迟。但现有 serving 系统（vLLM、SGLang）和 P-D 分离架构（DistServe、Splitwise）仍以 per-call 粒度调度，忽视三个交叉挑战：

  1. 在线不确定性——workflow DAG 在运行时逐步揭示，到达时只有源节点可见
  2. 资源异构性——同一 LLM 调用在不同 GPU 类型上 prefill/decode 延迟差异显著，KV 传输带宽也不同
  3. workflow 结构冲突——per-call FCFS / load-balancing 优化的是局部目标，可能阻塞 workflow 关键路径

    4. Paper Figure 1：三种 agentic workflow 模式——链式 agent-tool-agent、bounded self-refinement、parallel fan-out。只有源节点在 workflow 到达时可见，后续节点在父节点完成后才揭示。这一 online-revealed 特性使得经典 DAG 调度不可直接套用，也让 attained-service 等公平性指标（如 Autellix-ATLAS）与 SLO 风险脱钩。

    Program-aware serving 系统（Parrot、Hermes、Autellix、Continuum）开始向 application-level 调度靠拢，但没有任何现有系统同时处理 (i) online-revealed workflow DAG, (ii) 异构 P-D instance 的 joint placement, (iii) decode KV 容量约束, (iv) 异步 runtime overhead。

    Q2 方法 #

    核心机制：projected scaled-SLO risk prioritization + joint heterogeneous P-D placement。

    三层设计：

    1. Online workflow horizon $H_w(t)$：为每个 workflow 维护 standalone 完成时间估计（若独占集群），随新节点揭示和真实运行时间逐步修正。$H_w(t)$ 是非递减的——新揭示的 call 只能延长 critical path。
    2. Projected ratio $R_s(c,t) = \frac{(t - a_w) + \Delta_s(c,t)}{H_w(t)}$：将每个 ready call 的排序指标统一为归一化的 SLO 风险值。$R_s > 1$ 意味着 workflow 已超出 standalone horizon——极度紧急。
    3. Greedy joint planner：贪心地选择 projected risk 最高的 call，枚举所有可行 prefill-decode pair，选择使 projected decode finish 最早的组合，更新模拟状态后继续选下一个 call。

      4. 核心技术壁垒：projected-ratio surrogate 将异构 workflow 的紧迫度归一化到同一量纲。不同 workflow 有完全不同的 DAG 结构和 standalone 完成时间，直接比较 absolute slack 或 remaining time 不公平；除以各自的 $H_w(t)$ 后，scheduler 可以在混合 workload 中正确识别哪个 workflow 真正「快超时了」。这个归一化 + 在线修正的组合是 baseline（FCFS、LLF、ATLAS）都没有的——FCFS 无紧迫度信号，LLF 的 slack 没有 horizon 归一化，ATLAS 的 attained-service 是公平性指标而非 SLO 风险指标。

      Q3 结果 #

      • 异构集群（A100/H100/H200 8P+8D / 10P+10D）：avg Req95 减少 13.0%，avg Req99 减少 24.5%；最强单点 Qwen-Mixed Req99 从 8.96 降至 3.94（56.0%）
      • 同构集群（H200 4P+4D / A100 4P+4D）：avg Req95 减少 24.0%，avg Req99 减少 35.4%——证明 workflow-aware 排序本身是主要收益来源，异构 placement 是额外收益
      • Characterization ablation：Workflow-FCFS 相比 per-call FCFS avg 减少 Req95 31.4%、Req99 23.3%；HexAGenT 在此基础上再减 Req95 26.9%、Req99 42.6%
      • 鲁棒性：30% 估计误差下 Req99 退化 ≤1.5%（Llama）/ ≤9.5%（Qwen），非单调
      • 开销：per-invocation ≤23.7 ms，异步 apply 不阻塞在飞 call

      Planning & Reasoning（Scheduler 的决策逻辑） #

      • Planning style: greedy online——每次事件触发时对当前所有 waiting calls 做一次完整排序和 assignment。不做 tree search、不做 lookahead（没有显式的 future DAG 预测）。
      • Decomposition: bottom-up——从单个 ready call 的 projected risk 出发，贪心地逐个 assign，每次 assignment 后更新模拟状态再选下一个。
      • Budget: 无显式的 max steps/tokens/tool calls 限制。Scheduler 每次处理当前 waiting set 中的所有 call，复杂度随 queue depth 线性增长。对大 queue 有 one-pass fallback（不做 recomputation）。
      • Backtracking: 无。一旦 call 开始 prefill 或 decode，placement 固定不可撤销——收益全部来自初始决策质量。这是一个强设计约束：没有 preemption/migration 使系统简单但限制了纠错能力。

      关键设计决策 #

      • No preemption/migration：一旦 call 开始执行，placement 固定。这简化了 KV-cache 管理（无需跨 instance 搬迁 KV state），但意味着 scheduler 的初始决策必须足够好。
      • Joint P-D planning at prefill time：prefill scheduler 同时选定 decode instance，建立 KV 传输路由。在异构集群中这避免了跨硬件类型高延迟传输（除非 service-time 收益 justify）。
      • Asynchronous plan application：scheduler 在后台计算，serving 不停；plan 只 apply 到仍在等待的 call。引入 plan 过时风险但避免 head-of-line blocking。
      • Progressive horizon refinement：$H_w(t)$ 单调非递减，新揭示的 call 只能延长 critical path。SLO 目标只会放松不会变紧——这是 scheduler 稳定性的隐含保证。

      方程与物理意义 #

      Eq 1 — Scaled-SLO 目标：

      $$\min_{\pi}\;\alpha \quad \text{s.t.} \quad \frac{1}{|\mathcal{W}|}\sum_{w \in \mathcal{W}} \mathbb{1}[C_w^\pi \leq \alpha H_w] \geq \tau$$

      物理意义：在目标 attainment $\tau$ 下找到最小的 SLO 放大系数 $\alpha$。$H_w$ 是 workflow $w$ 独占集群时的 makespan，$\alpha H_w$ 是实际 SLO deadline。$\alpha = 1$ 意味着共享集群也能在独占时间内完成——不现实；$\alpha$ 越小说明调度越好。与固定 SLO 不同，此 metric 自动归一化了不同 workflow 的内在复杂度差异。

      Eq 2 — Projected ratio：

      $$R_s(c,t) = \frac{(t - a_w) + \Delta_s(c,t)}{H_w(t)}$$

      物理意义：分子 $(t - a_w) + \Delta_s(c,t)$ 是 workflow 已消耗时间加上当前 call 的预计剩余耗时，分母 $H_w(t)$ 是当前 standalone horizon。$R_s > 1$ → workflow 已超出目标——极度紧急。这个 ratio 在不同 DAG 深度、不同 standalone makespan 的 workflow 间提供了可比较的紧迫度量纲。

      Eq 3-4 — Decode 容量约束：

      $$m(c) = L_\text{in}(c) + \widehat{L}_\text{out}(c)$$

      $$m(c) \leq \text{Cap}(d)$$

      物理意义：admission control——call 的总 KV-cache 需求（输入 + 预测输出）不能超过 decode instance 的剩余容量。防止 OOM 并引导 scheduler 将长输出 call 分配到 HBM 更大的 instance。

      基础检查（6 项） #

      1. 目标函数完备性：Eq 1 直接优化 workflow-level tail attainment，不是 average latency 或 per-call metric ✓
      2. 变量可控性：$\pi$ 控制 call 排序和 P-D placement，这是 scheduler 实际可操控的决策变量 ✓
      3. 假设合理性：$H_w(t)$ 假设 standalone execution 可用 roofline 模型近似——合理但有精度限制（Table 5 量化了误差影响）✓
      4. Surrogate 一致性：$R_s(c,t)$ 作为 Eq 1 的 surrogate——贪心 argmax 不保证全局最优，但经验上有效 ⚠️ 无最优性证明
      5. 约束完整性：Eq 3-4 的 decode capacity 约束在 greedy placement 内强制执行 ✓
      6. Progressive refinement 收敛性：$H_w(t)$ 单调增（新揭示的 call 只能延长 makespan）且最终收敛到真实 $H_w$——论文未明确证明但逻辑成立 ✓

        7. Agent-specific 检查 #

        • Success-rate model：本文用 SLO attainment rate 代替 agent success rate，在 $(model, workload, cluster, baseline)$ 四维上做全面 sweep。attainment 沿 $\alpha$ 单调递增（更宽松的 SLO → 更高 attainment），符合预期。但没有 sweep backbone model size 或 planning depth——因为 scheduler 对这些维度透明。
        • Latency budget per turn：scheduler 每次 invocation 10–15 ms（Table 6），远小于 LLM call 的 prefill（10–200 ms）+ decode（秒级）延迟。异步 apply 确保 scheduling overhead 不在 serving 关键路径上。可以认为 scheduler 本身满足「interactive latency」约束。
        • Failure mode classification：论文未显式分类 workflow SLO violation 的 failure modes。隐含的 failure modes 包括：(a) critical-path 阻塞（长 call 占用快 instance，短 call 被排到慢 instance）；(b) decode KV 容量耗尽导致等待；(c) 估计误差导致 risk misjudgment。HexAGenT 通过 projected-ratio 针对 (a)，通过 capacity check 针对 (b)，对 (c) 则依赖 workflow-level 归一化的平滑效应。
        • 无形式化作者证明 — 仅实证。论文没有给出关于 projected-ratio greedy 的竞争比、近似比或收敛性定理。理想情况下应提供：(a) 在 workflow DAG 为 chain 的特殊情况下 greedy vs offline optimal 的 competitive ratio bound；(b) $H_w(t)$ 估计误差对最终 $\alpha$ 的 sensitivity bound；(c) greedy joint P-D assignment 的近似比（类似于 makespan minimization on unrelated machines 的已知 2-approximation）。可能可以 bound 的 metric：在 chain workflow + 同构集群的简化设定下，projected-ratio greedy 应可证明为 SRPT 的推广，继承其已知 competitive ratio。

        Tool & Environment Interface（Agent 视角） #

        HexAGenT 的 scheduler 与 agent workflow 之间的接口契约：

        • Tool catalog：scheduler 不直接管理 tool——tool 调用在 LLM call 之间的间隙由 workflow front-end 执行。Scheduler 只看到 tool 返回后新揭示的 LLM call。
        • Side effects：scheduler 的所有操作都是对 serving 集群内部状态的修改（queue priority、instance assignment），不产生对 agent 可见的 side effect。
        • Error surface：当 decode instance 的 KV 容量不足时，call 被阻塞等待容量释放（不报错给 agent）。Scheduler 的 capacity check $m(c) \leq \text{Cap}(d)$ 是硬约束。
        • Environment contract：P-D 集群状态在 scheduling invocation 之间是确定性更新的（每次事件都精确更新 queue/running/capacity 状态）。但异步 plan application 意味着 scheduler 看到的 snapshot 和 plan apply 时的真实状态可能不同步——论文通过「只 apply 到仍在等待的 call」来处理这个一致性问题。

        §7 实现 cross-reference #

        实现基础 #

        • 基于 SGLang v0.5.9 的 P-D disaggregated serving feature，scheduling 逻辑在 gateway/worker 路径上集成
        • Router 侧修改：workflow 元数据解析、stage-state 构建、异步 plan apply、revision check for safe queue mutation、bootstrap metadata injection、completion feedback accounting
        • Worker 侧修改：暴露 P-D snapshots、接受 priority/reassignment update for still-waiting requests、completion telemetry
        • Event-driven simulator：~4.6K 行 Python，覆盖完整 call lifecycle（waiting prefill → prefill → KV transfer → waiting decode → decode → completion），用 roofline-style 延迟模型

        代码状态 #

        [实现未公开]——论文未提供公开代码仓库链接。最接近的可参考实现：

        • SGLang P-D disaggregation（sglang/srt/）提供 substrate
        • Autellix/ATLAS 提供 program-aware baseline 的开源参考
        • Proxy model-based output length prediction 用于 $\widehat{L}_\text{out}(c)$

        关键实现细节 #

        1. Asynchronous plan application：scheduler 和 serving 解耦——plan 计算中 serving 继续执行，plan 完成后只 apply 到仍在等待的 call（已开始 prefill/decode 的不受影响）。这避免了 lock contention 和 head-of-line blocking，但引入了 plan 过时的风险（部分 call 在 plan 计算期间已被 dispatched）。
          1. Roofline-style latency estimator：prefill/decode 时间估计不基于 profiling lookup 而是 roofline 模型——给定 prompt 长度、model config、GPU spec 计算理论 prefill FLOP 和 decode memory bandwidth bound。KV transfer latency 从 KV size × measured effective bandwidth 计算。在异构集群中，cross-hardware transfer 的较低带宽被显式建模。

            2. LLM Backbone Requirements #

            • Minimum model size：论文验证了 Llama3.1-70B（dense）和 Qwen3-235B-A22B（MoE），未测试更小模型。由于 HexAGenT 是 serving-side scheduler，与 backbone 解耦——原则上适用于任何 size 的模型。
            • Required capabilities：HexAGenT 对 backbone 无特殊要求（不需要 tool-call format、structured output、CoT）。它只需要 serving engine 暴露 P-D disaggregation 接口。
            • Sensitivity to backbone：Qwen（MoE）比 Llama（dense）获得更大的异构 placement 收益——因为 MoE 的 TP 策略在不同 GPU 上差异更大（TP8 vs TP4），placement 决策的 impact 更强。
            • Serving cost：论文未报告 tokens/workflow 或 cost-per-million-tokens。Scheduler 本身的 overhead 可忽略（ms 级 vs 秒级 LLM inference）。主要成本节省来自更少的 GPU 资源即可达成相同 SLO——降低 $\alpha$ 意味着在相同 hardware 预算下可以接受更紧的 deadline 或服务更多 workflow。

            Evaluation（Agent 视角） #

            • Benchmarks：ShareGPT（conversational chains, from Vicuna dataset）、BFCL-v3（function calling, Berkeley benchmark）、LATS（tree search on HotpotQA）、Mixed（三者交替）。这些覆盖了 agent workflow 的三大类型：sequential chain、parallel fan-out、tree search。
            • Task difficulty distribution：未显式分为 easy/medium/hard。Workflow 复杂度由 DAG 深度和 fan-out 隐含分层：ShareGPT 最简单（纯 chain），LATS 次之（bounded tree），BFCL-v3 最复杂（高 call count with tool dependencies）。
            • Metric：scaled-SLO attainment（Req95/Req99）——比 average latency 或 pass@k 更适合 serving 场景，直接反映 tail 用户体验。
            • Baselines：SGLang-FCFS（per-call）、Workflow-LLF（least-laxity-first, workflow-aware）、Autellix-ATLAS（attained-service fairness, program-aware）。缺失的 baseline：Hermes（multi-turn scheduling）和 Parrot（semantic variable scheduling）——可能因为它们不支持 P-D disaggregation。

            Production Readiness #

            • Sandboxing：论文未讨论 tool 执行隔离——HexAGenT 不管理 tool，tool 在 workflow front-end 侧执行。Scheduler 本身运行在 gateway 进程内，无独立沙箱。
            • Secrets & auth：未讨论。Scheduler 需要访问所有 P-D instance 的 queue/capacity 状态，相当于 cluster-admin 权限。
            • Observability：event-driven 架构天然产生事件日志（每次 scheduling invocation 的输入状态 + 输出 plan）。论文的 simulator 提供完整 lifecycle trace 但实际部署中的 observability 未讨论。
            • Cost / rate limit / concurrency controls：scheduler 不限制并发 workflow 数量。Admission control 仅通过 decode KV capacity 隐式实现——当所有 decode instance 满时新 call 排队等待。缺乏显式的 backpressure 或 admission control 机制来防止 overload 下的 cascading SLO violation。
