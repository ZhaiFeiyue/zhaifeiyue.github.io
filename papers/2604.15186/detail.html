Scepsy: Serving Agentic Workflows Using Aggregate LLM Pipelines

Marcel Wagenländer, Otto White, Britannio Jarrett, Pedro Silvestre, Yanda Tao, Guo Li, Huanzhou Zhu, Llúis Vilanova, Peter Pietzuch (Imperial College London) | 2026-04 | arXiv:2604.15186 Category: framework | Tags: agentic-workflow, serving, multi-LLM, GPU-scheduling, fractional-GPU, topology-aware-placement Read: 2026-04-18

Core Contribution

Scepsy 提出 Aggregate LLM Pipeline —— 利用"workflow 的端到端延迟变化剧烈，但每个 LLM 在总执行时间中的相对占比高度稳定"这一经验观察，将不可预测的 agentic workflow 抽象为一个简单的轻量级 pipeline 吞吐/延迟预测器，并以此驱动"分数 GPU × 张量并行度 × 副本数"三维联合搜索，在 16-GPU 集群上实现比 Kubernetes 自动扩缩、Aegaeon 多模型复用与 Ayo 工作流感知调度最多 2.4× 吞吐、27× 延迟 的联合提升。

Summary

当前 agentic workflow（beam search、RAG+reranker、MCTS、multi-agent）将多个异构 LLM 与外部工具编排起来，其端到端执行路径高度数据依赖（循环、分支、fan-out），加之 LLM 数量往往多于可用 GPU，造成 GPU 严重超订购。现有方案要么将每个 LLM 独立自动扩缩（Kubernetes/AIBrix/vLLM Production Stack）—— 忽略跨 LLM 依赖，导致振荡；要么多模型时空复用（AlpaServe/MuxServe/Aegaeon）—— 对 agentic 的执行不确定性束手无策；要么 workflow-aware 但只支持单 LLM（Parrot/Ayo/Autellix/JITServe）。所有这些系统都把"给每个 LLM 分多少 GPU"丢给用户，且不能同时处理 异构多 LLM + 任意编程模型 + 动态控制流 的组合。

Scepsy 的 key insight 是：虽然绝对延迟抖动巨大，但每个 LLM 占 workflow 总执行时间的相对份额在大量请求下保持稳定。 这一稳定性允许系统用一个"聚合 pipeline"近似整个 workflow 的 steady-state 性能：每个 LLM 被抽象成一个 stage，stage 的参数（平均调用次数 $n_m$、平均并行度 $p_m$、吞吐-延迟曲线 $T_m/L_m$）通过轻量级 workflow tracing + 独立 per-LLM profiling 获得，无需做任何静态程序分析。基于此抽象，Scepsy 做三件事：(1) 在 $(d_m, TP_m, \text{frac}_m)$ 三维搜索空间用剪枝枚举找最优分配；(2) 用 latency ratio 驱动的 hierarchical heuristic 把分配映射到 NVLink 拓扑约束下的真实 GPU；(3) 通过扩展的 NVIDIA device plugin + MPS 在 Kubernetes 上落地分数 GPU 隔离。

实验在 16×A6000（4 节点 × 4 GPU，NVLink pair + PCIe4 + 100 Gbps IB）上用 RAG+reranker、beam search 两个真实 workflow 评估：beam search 相对 K8s autoscaler 4/8/16 GPU 分别拿到 2.4×/1.5×/1.8× 吞吐和最高 10.4× 延迟降低；相对 Aegaeon 拿到 7.3×/6.8× 吞吐、14.1×/12.9× 延迟降低（Aegaeon 缺少 prefix caching + prefill 实例切换开销是其主要痛点）；相对 Ayo 拿到 3.2×/8.2× 吞吐（Ayo 在低负载小优势，但吞吐饱和早）。消融表明 co-location（分数 GPU）对 RAG 吞吐最关键、TP 对 beam search 延迟最关键。调度器搜索时间在 16 GPU + 10 fraction + 5 LLM 下 <35s，128 GPU 下 <70s。

Key Findings

  • 相对份额稳定性假设：在 500 个 beam search 请求上，generator/verifier 的 绝对延迟 分布跨度极大（24~844 次 generator 调用/请求，9~264s 端到端），但每个 LLM 的 相对延迟占比 分布集中度比绝对分布高 4×（Fig. 3）。这是整个 Aggregate LLM Pipeline 的 empirical 基础。
  • Pipeline 近似是合理的：由于 non-LLM tool/orchestration 时间通常
  • 三维搜索空间 (fraction, TP, replica) 是 throughput-latency co-optimization 的最小必要维度：data parallelism (replica) 加吞吐、tensor parallelism 减延迟、fractional GPU 提高利用率，三者缺一不可。
  • 分数 GPU × 拓扑放置 不能被现有 orchestrator（Kubernetes、KAI-scheduler）同时支持；Scepsy 用 ahead-of-time 层次化启发式 + MPS 强制 SM/memory 限制补全此空白。
  • Workflow-aware 比 multi-LLM-serving 和 single-LLM-in-agentic-context 都强：对 Ayo（workflow-aware 但单 LLM）拿到 2.1-8.2× 吞吐，对 Aegaeon（多 LLM 但无 workflow 感知）拿到 2.5-7.3× 吞吐。
  • 搜索复杂度可控：5 LLM / 16 GPU / 10 fraction 的完整枚举为 $\binom{16 \cdot 10 + 5 - 1}{5-1} \approx 2.9 \times 10^7$，通过"按 latency ratio 排序 LLM → 只连续分配 → 限制 TP 在高带宽域内"三步剪枝后 <35s 完成。

Limitations

  • 跨 LLM 的 fan-out 被错误建模：如果 workflow 在 不同 LLM 之间 fan-out（而不是同一 LLM 的并行调用），Aggregate LLM Pipeline 会把它当成串行 pipeline，高估延迟。论文在 Discussion 中承认需要跟踪重叠 trace 来修复。
  • 双模负载分布：同一个 LLM 在同一 workflow 中扮演不同角色（或跨多个 workflow 共享）时，会产生 bimodal load distribution，单一 aggregate 分布会误判。论文仅给出"按 pattern 分类后合并"的构想，未实现。
  • 长 tool 执行时间：Scepsy 基于"non-LLM 组件延迟 < few ms"假设。对于含有长运行 tool（浏览器、代码执行、训练 job）的 workflow，此假设失效，需要额外建模。
  • 预测精度依赖 profiling 阶段的代表性：需要 workflow-level 的"代表性请求序列"来生成 trace。如果生产流量分布漂移（domain shift），重新 profiling 的触发机制未说明。
  • 评估规模较小：16 GPU × A6000 × 2 workflow，未触及大规模（>100 节点、MoE、多 workload interference）。搜索时间外推到 128 GPU 是 micro-benchmark，不是真实部署。
  • 只依赖 MPS 做 isolation：MPS 的 SM/memory 静态限制在生产环境下有 noisy-neighbor 风险，论文未讨论 QoS 保障。
  • TP 被限制在高带宽域内：放弃了跨 NVLink 域的 TP 方案，这是工程妥协而非根本性解。

Infrastructure Impact

  • Algorithm: N/A — 不提出新的训练/推理算法，但为 inference-time scaling (beam search/MCTS) 提供了可行的生产化部署路径，间接助推这类算法的大规模应用。
  • Kernel: N/A — 使用现成 vLLM/SGLang engine，不触及 kernel。
  • Framework: 直接影响 —— 在 Kubernetes + vLLM/SGLang 生态之上提供 workflow-aware 的编排层，定义了"scheduler 见 workflow、engine 见请求"的分层接口。推动 K8s device plugin 向 fractional + topology-aware 方向演进（vGPU、MPS 配置自动化）。
  • LLM: N/A — 对模型架构无假设，理论上支持任意 vLLM/SGLang 兼容模型。
  • Agent: 核心影响 —— 为 LangChain/LangGraph/AutoGen/Camel 等 agent framework 提供了"不修改代码即可享受 GPU 自动分配"的运行时，是 agentic 自建部署栈的关键缺失拼图。与 Autellix/JITServe 等"workflow-aware 但单 LLM"方案形成互补：Autellix 管 request-level 优先级，Scepsy 管 LLM-level GPU 分配。
  • Cluster: 引入"GPU fraction + NVLink-aware placement"的新需求，对 K8s scheduler 和 gang scheduling 方案（如 Kai-scheduler、Volcano）提出新的约束求解需求。
  • Code: 开源（post-acceptance at github.com/anon/Scepsy），~25K LOC Python，提供可被 AIBrix/llm-d/production-stack 吸收的 reference implementation。

Deep Analysis (framework)

0. Phase 2 深度扩展

时代定位

2025-2026 是 agentic workflow 从 demo 走向生产部署的窗口期。早期 (2023-2024) 的 LLM serving 优化几乎全部围绕 单 LLM 的 request-level 问题 展开：continuous batching (Orca)、paged attention (vLLM)、chunked prefill (Sarathi-Serve)、P/D disaggregation (DistServe/Mooncake)、speculative decoding。这些 low-hanging fruit 在 2025 年已经基本采摘殆尽。下一代问题是 multi-LLM × workflow-level × GPU-allocation，属于 agent serving 的"深水区"。Scepsy 所在的位置非常具体：


2023  Orca (continuous batching)
2024  vLLM (paged attention)  Sarathi (chunked prefill)  DistServe (P/D)
2025  Autellix / JITServe (single-LLM workflow-aware scheduling)
      Parrot / Ayo (single-LLM static graph optimization)
      AlpaServe / MuxServe / Aegaeon (multi-LLM 但 workflow-agnostic)
2026  Scepsy ← 第一个同时勾满 "multi-LLM × workflow-aware × 自动分配 × 任意框架"

这不是在单 LLM 维度再抠百分之几，而是把问题 向上抬了一层 到 orchestration layer，类似于从 Linux CFS 到 Kubernetes HPA 的跃迁。

约束推导（Why not X?）

为何不可 per-LLM 独立自动扩缩（K8s HPA / AIBrix）？

  • 每个 LLM 的负载由 workflow 中其他 LLM 的输出驱动。生成器加速会增加验证器负载，触发验证器扩容，反过来又加速了生成器的 downstream → 级联振荡。数学表述：扩缩控制器是 PID-like 的负反馈系统，但 LLM 之间存在 未建模的前馈耦合，在稳态工况下稳定，但在负载瞬变时振荡。
  • 论文的实验（Fig. 6）直接证实了这一点。

为何不可纯时间/空间多路复用（AlpaServe/MuxServe/Aegaeon）？

  • 这些系统对每个 LLM 给定 固定 的多路复用策略，但 agentic workflow 的 per-LLM 负载随时间剧烈变动（beam search 的 GEN 调用次数 24-844×），固定策略必然在绝大部分时间 misallocate。
  • Aegaeon 的 prefill/decode 实例切换开销是 swapping-driven 的，无法利用 prefix cache，对 beam search 这种 长公共前缀 的 workload 损失尤其大（论文 Fig. 7 显示 14× 延迟 gap）。

为何不可 single-LLM workflow-aware 调度（Autellix/JITServe）？

  • 这些系统的建模假设是 整个 workflow 只用一个 LLM，每个请求对应一个优先级或一个 SLO 预估。它们无法表达"generator 和 verifier 的负载份额是多少"，因为它们的状态空间里没有"LLM 身份"这一维度。
  • 把它们硬套到多 LLM，只能把所有 LLM 看作同一个，丢失异构性。

为何不可静态图分析（Parrot/Ayo）？

  • 需要用户用特定 DSL 重写 workflow，丧失了 LangChain/AutoGen/Camel 生态。
  • 无法处理数据依赖的动态控制流（beam search 的"只对 top-k beam 继续扩展"这种分支）。

为何不可精确模型 per-request 的控制流？

  • 每个请求的 token 生成路径本质是 stochastic，精确建模需要 sequence-level 模拟，代价与运行 workflow 相当。论文观察到"相对份额稳定"这一 empirical fact，把精确动态模型替换成 steady-state aggregate 近似，这是关键的 engineering shortcut。

核心技术壁垒

一句话：Aggregate LLM Pipeline 抽象 + 三维联合搜索空间的剪枝 是整个系统能 work 的根本壁垒。

  • 不是"tracing"（HTTP proxy 是烂大街的工程）；
  • 不是"fractional GPU 放置"（vGPU/MPS 都能做）；
  • 而是 把 workflow 的动态行为用两个静态统计量 $(n_m, p_m)$ 压缩，并证明 (a) 相对份额分布窄（Fig. 3）(b) pipeline 公式 $L_w, T_w$ 可以精确预测各种 allocation 的效果。

这层抽象一旦建立，调度问题瞬间从"非凸 stochastic control"退化为"查表 + 算术 + 剪枝枚举"，后续所有工程实现都是基于这个简化后的 operational 模型。这是整个方案 2.4×/27× 提升的真正 sources。

具体支撑数据：Fig. 3 的"相对份额比绝对值稳定 4×"是这个壁垒能立住的 empirical proof。没有这个 4×，Aggregate Pipeline 就会是一个糟糕的预测器。

质疑假设

  1. "LLM 相对份额稳定" 的假设在什么条件下失效？
  2. 当 workflow 中某个 LLM 的调用模式与输入强相关（例如代码生成 agent 中 debugger LLM 仅在代码错误时被调用，不同 domain 的出错率差异极大），相对份额会有 bimodal 分布。论文 Discussion 承认此情况但未实现方案。
  3. 当 workflow 规模极小（<10 次 LLM 调用/请求），大数定律不生效，相对份额方差大。beam search @ 小 branching factor 可能出问题。
  4. 笔者不确定在 tool-heavy workflow（e.g. Sutradhara 那类 tool dispatch 为主的场景）下是否仍成立，论文没有评估。
    1. "non-LLM tool 时间 << LLM 时间"的假设？

          生态影响追踪

          论文太新（2026-04），尚无 downstream 采用数据。但架构上的潜在影响路径：

          • 对 K8s device plugin 生态：Scepsy 扩展了 NVIDIA device plugin 支持层次化 fractional placement。这方向与 NVIDIA GPU Operator + KAI scheduler + MIG 生态相邻，未来有并入标准 K8s GPU resource model 的可能。
          • 对 AIBrix / llm-d / vLLM Production Stack：这些开源 agentic-stack 都以 "per-LLM autoscaling" 为设计中心。Scepsy 的 workflow-aware scheduler 可作为其 control plane 的替代，也可作为"advisor"形式提出 LLM replica 建议。
          • 对 SGLang router：论文已经用了 SGLang Model Gateway 作为 request router，这是 downstream 协作的自然接口点。

          1. System Scope

          • Primary goal: Serving —— 不涉及训练/fine-tuning/数据处理。
          • Distributed: 单到多节点均可；实验规模 4 节点 × 4 A6000 = 16 GPU。理论上支持更大，但未验证 >100 GPU。
          • Online（latency-sensitive）—— 同时优化 latency 和 throughput，而不是 offline batch 吞吐。
          • Workload: 多 LLM agentic workflow，具体评估 beam search（inference-time scaling）和 RAG+reranker（检索增强）。

          2b. Data Movement Hotspots

          1. KV cache within engine —— 每 token 一次 read + write，对 HBM bandwidth 敏感。Scepsy 不优化此层（交给 vLLM 处理）。

              5. Scheduling & Resource Management

              • Batch formation: 由下游 engine（vLLM/SGLang）处理（continuous batching），Scepsy 不干预。
              • Memory management: per-replica 独立 KV cache；Scepsy 只决定 replica 的 GPU fraction（决定了 KV cache 上限）。
              • Idle avoidance: 通过 fractional GPU 把小 LLM pack 到同一 GPU，消除"整卡给小模型"的浪费。
              • Multi-tenancy: 支持多 workflow（egalitarian welfare 聚合），但未讨论恶意用户隔离。
              • Priority / SLO: 优化目标是 latency 和 target throughput，没有严格 SLO enforcement（不是 SLO-aware）。

              7d. Baselines & Fairness

              • K8s HPA: 公平 —— 同硬件、同 engine (vLLM)。但 HPA 的 scaling 策略 tuning 参数未说明（冷启动时间、scale-up 速度）—— 可能 under-tuned。
              • Aegaeon: 可能有点不公平 —— Aegaeon 设计为 prefill/decode 分离的多模型复用，在 beam search 这种 long-prefix workload 上天然吃亏（无 prefix cache）。论文承认此点。
              • Ayo: 用了旧版 vLLM v0.2（为了兼容性）。引擎老旧会拖累 Ayo，对 Scepsy 有利。这是一个需要注意的 fairness 点。

              Scenarios where baseline wins: Ayo 在 4 GPU 低 arrival rate 区有轻微优势（batching 更激进）。

              8. API & Usability

              • API compatibility: 完全透明 —— workflow 用原 framework（LangChain 等）编写，通过 HTTP proxy 无侵入接入。
              • Model format: 继承 vLLM/SGLang 支持（HF、GGUF）。
              • Deployment: Kubernetes only（具体是 MicroK8s）+ 扩展 NVIDIA device plugin + MPS。
              • Configuration: 相对 K8s HPA 简化 —— 用户只需指定 workflow 和 target arrival rate；GPU allocation 全自动。但需要初始 profiling 数据，这是额外 onboarding 成本。
