ForkKV: Scaling Multi-LoRA Agent Serving via Copy-on-Write Disaggregated KV Cache — L2 #

§1 TL;DR #

ForkKV disaggregates the KV cache into a massive shared base component ($bCache = xW$) and lightweight per-agent residual components ($rCache = xA_i$) using OS fork/CoW semantics managed by a DualRadixTree — achieving up to 3.04× throughput over prefix-caching baselines on multi-LoRA agent workloads with only 0.71% average quality degradation, by reducing per-agent memory footprint to $\approx r/n$ of the original.

§2 痛点 · 方法 · 结果 #

Q1 痛点 #

Multi-LoRA agent serving creates a critical memory bottleneck: KV cache divergence.

When multiple specialized agents (each with a different LoRA adapter) share a base model and process the same context, traditional prefix caching fails because unique LoRA activations produce different KV cache entries for identical text prefixes. Each agent must maintain an independent full-sized KV cache. Consequences:

  • Memory scales linearly with number of concurrent agents.
  • On Llama3-8B with 32K contexts, throughput drops 90.8% (ReAct) and 90.1% (MapReduce) as workflows scale from 1 to 8 (Fig 3).
  • GPU memory saturates rapidly, restricting batch parallelism.

Prior approaches are trapped in a dilemma: independent KV cache preserves accuracy but is memory-prohibitive; full KV reuse (ignoring adapter differences) causes severe quality degradation (21.95 F1-point drop on Llama3-8B/APIGen).

Q2 方法 #

ForkKV exploits the structural properties of LoRA to physically decouple the KV cache:

Insight: The LoRA projection $Y = xW + xA_iB_i$ naturally decomposes into a massive base cache $bCache = xW \in \mathbb{R}^{s \times n}$ and a lightweight residual cache $rCache = xA_i \in \mathbb{R}^{s \times r}$. Since rank $r \ll n$ (typically 16 vs 1024), bCache is ~64× larger than rCache.

Two mechanisms:

  1. DualRadixTree with Fork/CoW semantics: Manages bCache and rCache in separate radix trees with independent lifecycles. When a new agent launches: (Step 1) longest-prefix match inherits globally shared read-only bCache (like forking a parent process's pages); (Step 2) allocates exclusive memory for the agent's unique rCache (CoW footprint). Decoupled LRU eviction — if bCache is evicted but rCache survives, partial hit recomputes only base projection.
    1. ResidualAttention kernel: Fuses KV cache reconstruction into the attention computation within SRAM. Three stages: (i) on-the-fly Key reconstruction with deferred RoPE ($K = K_{base} + \text{RoPE}(K_{res} \cdot B_k)$); (ii) compute attention scores over shared tiles; (iii) fuse base and residual Value attention via matrix associativity ($\sum sm(QK^T)V_{base} + (\sum sm(QK^T)V_{res})B_v$) — pushing the up-projection out of the inner loop.

      2. 核心技术壁垒: The ResidualAttention kernel design that reconstructs disaggregated KV cache entirely within SRAM. The non-obvious insight is exploiting matrix associativity to separate base and residual attention scores, pushing the LoRA up-projection $B_v$ multiplication outside the inner loop — this eliminates both redundant computation and SRAM contention that would otherwise negate the memory savings. Naive HBM reconstruction or in-place updates both fail: the former wastes memory, the latter destroys intra-batch parallelism.

      Q3 结果 #

      • Throughput: 1.25–3.04× on ReAct, 1.68–2.60× on MapReduce vs prefix-caching baselines (vLLM, SGLang).
      • Quality: Only 0.71% average F1 degradation across 3 models × 2 datasets. Maximum drop: 1.60 points (Llama3-8B/APIGen).
      • Memory: 12.7× per-agent reduction; cache hit rate improved 6.93×; decode batch size 12.0× larger.
      • Memory ratio: $M_R = 1/N + r/n$ → as $N$ grows, per-agent cost approaches $r/n \approx 1.6\%$.
      • Sensitivity: Robust across LoRA ranks 8–32 (2.36–2.88× throughput) and varying output lengths (2.69–3.36×).

The decoupled eviction policy avoids the architectural mismatch of evicting bCache and rCache as a unit — partial hits gracefully recompute only the missing component.

      Paper's Figure 14, verbatim (caption: "Underlying causes of ForkKV performance gains. ForkKV reduces (a) the average per-agent memory usage, which consequently improves (b) the cache hit rate and increases (c) the average decode batch size.").

      The causal chain: 12.7× memory reduction → 6.93× cache hit rate improvement → 12.0× larger decode batch size → throughput gains. The throughput improvement is a downstream consequence of memory efficiency, not a direct scheduling optimization.

      Memory savings formalization #

      $$M_R = \frac{Mem_{disagg.}}{Mem_{unified}} = \frac{sn + N \cdot sr}{N \cdot sn} = \frac{1}{N} + \frac{r}{n}$$

      With typical $n = 1024$, $r = 16$: as $N$ grows large, $M_R \to r/n \approx 1.6\%$. Per-agent memory is dominated by the rank ratio, not the number of agents.

      System scope #

      • Stage coverage: Both prefill (chunked and non-chunked) and decode.
      • Serving vs training: Serving only. Multi-LoRA agent inference.
      • Parallelism: Single-GPU and dual-GPU (TP=2 for Qwen2.5-14B).
      • Deployment mode: Single node. Integrated into SGLang serving framework.

      Scheduler #

      • Base system: SGLang v0.5.6 scheduler extended with DualRadixTree queries.
      • Prefix matching: Longest-prefix match on base RadixTree for bCache inheritance.
      • Eviction: Decoupled LRU — independent eviction for base and residual trees.

      KV / Memory manager #

      • Allocation unit: PagedAttention-style blocks; separate pools for bCache and rCache.
      • bCache: Shared, read-only. Zero-copy inheritance via DualRadixTree.
      • rCache: Per-agent, CoW allocated. Small ($r/n$ of bCache per token).
      • Eviction: Independent LRU per tree. Partial hits: if bCache evicted but rCache survives, recompute only $xW$.
      • Reconstruction: ResidualAttention kernel fuses $bCache + rCache \times B_i$ within SRAM — no HBM materialization.

      Equation physical meaning #

      • $Y = xW + xA_iB_i = bCache + rCache \times B_i$: The LoRA output can be reconstructed from the large base projection plus a small residual up-projected by $B_i$. This is the algebraic foundation for disaggregation.
      • $M_R = 1/N + r/n$: Memory ratio converges to $r/n$ as agent count grows. With $r=16, n=1024$, this is 1.6% — the theoretical minimum per-agent cost.
      • $\sum sm(QK^T)V = \sum sm(QK^T)V_{base} + (\sum sm(QK^T)V_{res})B_v$: Matrix associativity allows pushing $B_v$ out of the inner attention loop. This is the key equation enabling efficient SRAM-only computation without redundant up-projection per sequence block.

      Monotonicity #

      $M_R$ is strictly decreasing in $N$ and linear in $r/n$. The optimum is at the boundary ($N \to \infty$, $r$ as small as possible). Higher LoRA ranks linearly increase per-agent cost — the paper validates this at $r \in \{8, 16, 32\}$ but does not test $r \geq 64$.

        §7 实现 cross-reference #

        Built on: SGLang v0.5.6. Approximately 3K lines of Python + custom Triton kernels.

        • Custom LoRA replacement module for linear projection layer separating adapter residuals from base activations.
        • DualRadixTree extends SGLang's native RadixCache into coordinated dual-tree architecture.
        • Scheduler adapted to orchestrate two-tiered KV cache pool across chunked prefill, non-chunked prefill, and decode phases.
        • ResidualAttention: custom Triton kernel adapted from SGLang's RadixAttention. Separate versions for prefill and decode.

        [实现未公开]

        关键实现细节:

        1. Deferred RoPE for residual cache: Since $rCache = xA_i \in \mathbb{R}^{s \times r}$ has dimension $r$ (not $n$), RoPE cannot be applied at cache time. Instead, RoPE is deferred to attention time: inside SRAM, the kernel performs $K_{lora} = \text{RoPE}(K_{res} \cdot B_k)$ before adding to $K_{base}$. This creates a design constraint where every attention computation must carry the sin-cos table and perform RoPE inside the kernel loop.
        2. Decoupled eviction with partial hits: When bCache is evicted but rCache survives (or vice versa), the system executes a partial hit — recomputing only the missing component. This avoids cascading eviction that would couple the pools and force low-contention pools to discard active cache. The partial hit logic is integrated into the DualRadixTree query.

          3. API & usability #

          • User-facing API: SGLang's existing API (OpenAI-compatible) — no API change required.
          • Config surface: LoRA rank, number of adapters. Disaggregation is transparent to users.
          • Migration cost: Low — extend existing SGLang deployment with ForkKV modifications.

          Deployment context #

          • Serving stage: Prefill + decode.
          • Concurrency regime: High concurrency multi-LoRA (8+ concurrent workflows, each with distinct adapters).
          • Hardware affinity: Evaluated on L40 and RTX 5000. Benefit scales with memory contention — more impactful on GPUs with less HBM.
          • Ecosystem integration: Implemented on SGLang — patch to RadixCache, scheduler, and attention kernel.
          • Migration path: Replace SGLang's RadixCache with DualRadixTree; replace attention kernel with ResidualAttention; load LoRA adapters normally.
