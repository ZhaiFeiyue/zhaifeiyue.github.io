为什么这组论文构成 SAGA 的核心参照系：SAGA 的创新是把 "workflow-as-unit" 从概念（SGH 的 scheduler continuum）落地为 serving 系统。它需要同时解决三个子问题——KV 缓存保留（vs Continuum 的 TTL）、缓存淘汰策略（vs PBKV 的预测驱动淘汰）、tool 延迟期间的资源管理（vs PASTE 的投机执行）。SideQuest 则代表了另一条路径：不在 serving 层解决缓存问题，而是让 LLM 自己做内存管理。

§2 本篇 vs 相关论文的 delta #

相对 Continuum (2511.02230) 的 delta #

SAGA 的核心 delta：从 per-request TTL 升级为 per-workflow DAG-predicted 淘汰。Continuum 的 memoryfulness factor $\eta$ 只捕获"维持程序顺序是否有助于加速"这一标量 [ref:2511.02230#核心技术壁垒]，而 SAGA 的 $P_{reuse}(s) = \sum_{u \in succ(v_s)} P(v_s \to u) \cdot overlap(s, u)$ 利用了 AEG 的完整 successor 结构和 transition probabilities [ref:2605.00528#核心技术壁垒]。代价是需要 framework integration（或 87% 准确率的 pattern inference）。

相对 PBKV (2605.06472) 的 delta #

维度PBKVSAGA
预测目标下一个 agent 类型（K 步分布）KV 缓存复用概率
预测器GraphSAGE (~350K params, 1.56ms)AEG transition probabilities + tool-type EMA
退化保证Lipschitz 遗憾界 → 退化至 LAE 而非 LRU无形式退化保证（退化至 prefix caching +96% TCT）

淘汰设计分层（退役确定性 + 评分概率性）加权线性组合（recency + reuse + size）
覆盖场景动态多 agent 工作流（跨 agent 缓存共享）单 agent 多 turn 工作流（session 内缓存保留）

PBKV 的 "确定性护栏 + 概率系统" 分层设计 [ref:2605.06472#核心技术壁垒] 在鲁棒性上优于 SAGA 的加权线性评分。PBKV 在预测精度从 0.94 到随机的整个谱上都严格优于 LRU [ref:2605.06472#预测误差鲁棒性]，而 SAGA 的 ablation 显示去除 AEG 后 TCT 退化 54% [ref:2605.00528#实验与数据]，意味着 SAGA 对预测质量的依赖更强。

SAGA 的优势在于 更丰富的决策信号：不仅预测"下一个 agent 是谁"，还预测"已有 KV 有多少比例会被复用"——后者是 agent 内部决策的直接函数。PBKV 跨 agent（工作流级），SAGA 跨 turn（session 级）。

相对 SideQuest (2602.22603) 的 delta #

SideQuest 用 LRM 辅助线程做语义级 eviction（56-65% token 降幅）[ref:2602.22603#Q3 结果]，但：

  • 操作粒度是 token-level cursor（intra-context），SAGA 是 session-level KV block（inter-turn）
  • SideQuest 需要 LoRA 微调 + 215 条训练数据，SAGA 无训练
  • SideQuest 的 serving 增益（SGLang +83.9% 吞吐）是因为释放 batch capacity；SAGA 的增益（1.64× TCT）是因为避免 cache regeneration

两者可组合：SAGA 跨 turn 保留 KV block，SideQuest 在单 turn 内压缩已保留 block 的内容。

相对 PASTE (2603.18897) 的 delta #

PASTE 的投机执行将 tool 等待时间减少 67% [ref:2603.18897#关键发现]，而 SAGA 通过 TTL 保留避免了 KV 重建。两者解决同一问题（tool-call idle period 的浪费）的不同面：

  • PASTE：重叠下一个 tool 的执行与当前 LLM 推理 → 减少 tool stall
  • SAGA：在 tool 执行期间保留 KV cache → 减少下一次 prefill 的重建开销

PASTE 的 pattern tuple $(C, T, f, p)$ [ref:2603.18897#核心技术壁垒] 和 SAGA 的 AEG 在概念上相近：都是从历史 trace 中提取 agent 行为模式用于预测。但 PASTE 预测的是 tool 类型+参数（用于投机执行），SAGA 预测的是 KV 复用概率（用于缓存决策）。

§3 可攻击面 #

攻击 1：AEG 的实用性与冷启动 #

SAGA 声称 AEG 可以无 framework hint 推断（87% 准确率）[ref:2605.00528#实验与数据]，但：

  • 87% 是在 SWE-bench 这种高度结构化 benchmark 上的数字。ReAct 循环的模式固定（think→act→observe），推断不难。
  • 对于更复杂的 workflow（multi-agent, tree-of-thought, 动态子任务生成），推断准确率可能大幅下降。论文自己承认"dynamic multi-agent frameworks requiring AEG re-inference per epoch"的退化 [ref:2605.00528#多agent考量]。
  • PBKV 只需 1K 训练轨迹即可达 0.94 精度 [ref:2605.06472#Q3 结果]，证明"预测 agent 调用"是可解问题——但 SAGA 的"预测 KV overlap"比"预测下一个 agent"更难验证正确性。

攻击 2：Bélády competitive ratio 的误导性 #

SAGA 报告 1.31× competitive ratio vs Bélády optimal [ref:2605.00528#Q3 结果]，但：

  • Bélády 是 offline optimal for page replacement，而 SAGA 的 cache space 是在多 worker、多 session 间共享的。真正的 optimal 应该是 joint scheduling + eviction 的全局最优，Bélády 只是单机 eviction 的 optimal。
  • Formal bound 是 20.5×（极度松散）[ref:2605.00528#作者证明]。1.31× 是 empirical average-case，在 adversarial workload 下可能远差。
  • Continuum 没有报告 competitive ratio，但其 8.18× 延迟改善（真实 SWE-agent）[ref:2511.02230#真实分布式SWE-Agent] 暗示在真实环境中的绝对收益可能更大——因为 Continuum 解决的是 queueing delay 而非纯 cache miss，而 queueing delay 在高负载下主导。

攻击 3：30% 吞吐损失的代价 #

SAGA 明确承认 ~30% throughput reduction vs BFS [ref:2605.00528#实验与数据]。这是 session-affinity 的直接代价——请求被路由到"正确的"worker 而非"空闲的"worker。在高吞吐需求场景下，这是一个严重的限制。

对比：

  • Continuum 没有报告吞吐损失（TTL 机制不影响路由）
  • PBKV 的保守预取设计"确保错误预测时无负面影响"[ref:2605.06472#核心技术壁垒]，没有吞吐代价
  • SideQuest 反而提升吞吐 +83.9%（通过释放 batch capacity）[ref:2602.22603#SGLang生产serving性能]

攻击 4：单模型、单 benchmark 类型的评估局限 #

SAGA 仅评估 Llama-3-70B-Instruct [ref:2605.00528#LLM backbone requirements]。未测试：

  • 更小模型（8B/14B）——KV cache 更小，缓存保留的边际价值更低
  • MoE 模型（DeepSeek-V3, Qwen3-32B）——routing-aware extensions 被标记为 limitation
  • 多模态 agent——图像 token 的 KV 特征不同

对比 PBKV 测试了 Qwen3-14B 和 Qwen3-32B [ref:2605.06472#LLM骨干要求]，Continuum 测试了 8B/70B/12B/355B 四个模型 [ref:2511.02230#LLM Backbone Requirements]。

§4 生态位 #

范式定位 #

SAGA 代表了 agent serving 从 "request-level scheduling" 到 "workflow-level scheduling" 的范式跃迁。这个方向的信号已经在多篇论文中出现：

  • SGH 从理论层面论证了"把 workflow 作为一等调度单元"的必要性（$|\mathcal{U}| \geq 1$ 的 scheduler continuum）[ref:2604.11378#破局]
  • PBKV 从缓存管理层面实现了跨 agent 的生命周期追踪 [ref:2605.06472#核心技术壁垒]
  • PASTE 从工具执行层面实现了 workflow pattern 的投机执行 [ref:2603.18897#核心技术壁垒]

SAGA 是这条线上最完整的系统实现：同时覆盖 KV 保留、分布式路由、公平性三个子问题。

采用证据与阻碍 #

正面信号：

  • Agent 工作负载的 KV 缓存问题已被广泛认可（Continuum 开源并被引用、PBKV/KVFlow 的出现）
  • 64-GPU 集群评估说明工程投入足够生产级

阻碍：

  • 代码未开源——严重阻碍社区采用
  • 需要 framework integration 或 pattern inference——部署门槛高于 Continuum（纯 vLLM 插件）
  • 30% 吞吐损失对高吞吐场景不可接受
  • 单模型评估降低了结论的泛化信心

竞争格局 #

在 "agent-aware KV cache management" 这个具体赛道上：


复杂度 ↑
    │  SAGA (cluster-wide, workflow-atomic)
    │     ↑ 完整但重
    │  PBKV (multi-workflow, prediction-driven)
    │     ↑ 理论保证强
    │  Continuum (single-instance, TTL-based)
    │     ↑ 简单可用
    │  vLLM+APC (prefix caching, no awareness)
    └──────────────────────────────────── 收益 →

Continuum 已开源且侵入性最低，可能率先被生产采用。PBKV 的 Lipschitz 退化保证使其在预测不可靠场景下更安全。SAGA 提供了最完整的视角但部署代价最高。

§5 未探索方向 #

  1. Continuum TTL + SAGA AEG 的混合：用 Continuum 的 cost-benefit TTL 做 fast-path（低开销，per-turn），用 SAGA 的 AEG 预测做 slow-path（高信息量，per-workflow）。在信息不足时退化到 TTL，在 workflow 模式清晰时升级到 AEG-guided。这组合了两者的优点而避免了 SAGA 对 AEG 质量的强依赖。
    1. SideQuest 语义压缩 + SAGA session 保留：SAGA 保留了完整 KV block 跨 turn，但长 session 的 KV 最终仍会超出 GPU 容量。SideQuest 的辅助线程可以在 SAGA 保留的 KV block 内部做语义压缩——保留结构但丢弃已过时的 tool response token。两个系统操作在不同粒度，组合后覆盖 "session 级保留 + turn 级压缩" 的完整谱。
      1. PBKV 的 Lipschitz 退化框架应用于 SAGA：SAGA 的 WA-LRU 缺乏形式退化保证。将 PBKV 的 "确定性护栏 + 概率系统" 分层设计 [ref:2605.06472#核心技术壁垒] 移植到 SAGA——工作流终止后的 cache 确定性回收作为护栏，AEG-guided eviction 作为概率层——可以在不牺牲 SAGA 信息优势的前提下获得形式化的退化保证。
        1. PASTE 投机执行 + SAGA 投机 prefetch 的联合优化：SAGA 已经在做 speculative KV prefetch（most-likely successor）[ref:2605.00528#关键実装细节]，PASTE 在做 speculative tool execution。如果两者共享 pattern 预测结果：当预测下一个 tool 是 run_test 时，PASTE 预执行 test，SAGA 同时 prefetch test-response 对应的 KV prefix。这消除了"投机执行返回后才开始 prefill"的串行依赖。
          1. CPU-GPU co-scheduling 扩展：2511.00739 揭示 CPU 工具执行占 E2E 延迟最高 88% [ref:2511.00739#核心发现]。SAGA 的 tool-call TTL 目前只考虑 GPU 侧的 KV 保留决策。如果将 CPU 工具执行的调度（COMB 微批、MAS 准入控制）纳入 SAGA 的统一调度，可以同时优化 GPU KV 保留和 CPU 工具执行——构建真正的 "heterogeneous workflow-aware scheduler"。
            1. 自适应 AFS 面向 DAG-structured workflows：SGH 提出 agent 执行应从 loop 走向 structured graph [ref:2604.11378#核心技术壁垒]。SAGA 的 AFS 目前假设 flat tenant 模型。当上游 agent runtime 采用 DAG 结构时，fairness 的定义应该从"per-task completion time"扩展为"per-DAG critical-path completion time"——这需要 AFS 感知 DAG 拓扑。
