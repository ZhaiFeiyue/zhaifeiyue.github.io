TL;DR
将 omni 推理拆为六段专门化流水线 (AuT → ViT → Thinker MoE → Talker MoE+Shared → Code Predictor → Code2Wav)，每段架构 (layers/d_model/heads/kv_heads/experts) 独立定制，总预算 30B-A3B 下做到 RTF<1、首包 234ms，并首次证明全模态无退化。

图的 7 个页签：

1. Top-level Pipeline — 文本/音频/图像/视频 → encoders → Thinker → Talker → Code Predictor → Code2Wav 的全链路

7. Code2Wav — RVQ 嵌入 → TransposedConv upsample → Residual Blocks → Causal 输出

Core Contribution

Qwen3-Omni 在 Thinker–Talker 架构上做了"端到端把全模态流水线彻底拆分为 6 个可独立调优的专门化子模型"，每个子模型的结构、维度、注意力形状、expert 数量都针对自己的角色精心定制。核心技术价值不在于"又一个多模态模型"，而在于第一次把"文本推理主干 / 语音主干 / 语音残差精修器 / 波形合成器"的职责彻底解耦，并在 30B-A3B 的总预算下让每个组件都恰好能在 vLLM + CUDA Graph 上跑到 RTF < 1，端到端首包延迟仅 234 ms。

Summary

Qwen3-Omni 的模型结构把一次 omni 推理拆成一条六段流水线，每段的 Transformer 骨架完全不同，维度从 2048→1024→1024 逐步收敛，token rate 从 context 级一路降到 codec 级 (12.5 Hz) 再升到音频采样率 (24 kHz)。

数据流如顶部架构图第 1 页 Top-level Pipeline 所示：Text / Audio / Image / Video 四路输入 → Tokenizer / AuT Encoder / ViT+DeepStack → Aggregate → Thinker (28L MoE AR, d=2048, 128E top-8) → Talker (20L MoE+Shared AR, d=1024, 128E top-6+Shared) → Code Predictor (5L Dense × 15 AR iters) → Code2Wav (Causal ConvNet) → Audio Output @24 kHz。同时 Thinker 产出的 text token 直接对外输出。

六段结构分别解决：多模态感知 (AuT/ViT) → 语言推理 (Thinker) → 声学粗粒度 (Talker) → 声学细粒度残差 (Code Predictor) → 波形渲染 (Code2Wav)。每一段都写死了自己的 hidden size、head 数、GQA ratio、expert 数，使得运行时的 memory-bandwidth 占用最小，且可以在 vLLM 下用不同的批次策略独立调度。

训练：S1 锁定 LLM 只训 encoder alignment；S2 全参 ~2T tokens；S3 长序列 32K。Post-training = SFT → Strong-to-Weak Distillation → GSPO。

关键结果：text 超越 GPT-4o 和同规模 text-only Qwen3；36 个 audio benchmark 中 32 个开源 SOTA、22 个总体 SOTA；audio 首包 234 ms，video 首包 547 ms，1→6 并发 TPS 仅降 29%。

Key Findings

  • 六段专门化 — 每个子模块都有独立的 (layers, d_model, heads, kv_heads, experts) 配置，精心匹配自己的角色
  • Thinker 的 GQA 比例 7:1 (28Q/4KV) + head_dim=128，是本模型 KV cache 小的关键：每 token 仅 4×128×2 = 1024 个参数 vs dense 28 heads × 128 × 2 = 7168 → 节省 7×
  • Talker 用 MoE + Shared-Expert 的罕见组合，routed 走 128E top-6 (expert hidden=384)，shared expert (hidden=768) 始终激活并用一个 sigmoid gate 做门控，参考 DeepSeek-V2 但加入了 per-token gate
  • Code Predictor 不是 MoE 也不是"一次出全部" — 而是 5-layer dense transformer + 15 AR 迭代 + 15 套独立 LM head / Embedding，每轮产出一个 residual codebook token 并 append 到序列；15 套独立 head 相当于 "每层残差一个专门 classifier"
  • AuT 的 Conv1D × 2 (k=3, s=2) + Temporal Merge = 8× 下采样 把 100 Hz 的 mel 降到 12.5 Hz 的 token rate，恰好对齐 codec rate
  • ViT 的 DeepStack — 从 8/16/24 三层取出中间特征、投影到 3584 维追加给 Thinker，等于在浅、中、深三个层次都给 Thinker "看了一眼"
  • 无退化：同 FLOPs 下，Omni-Base 在 text/vision 都 ≥ 单模态 base；audio 数据反而 +2.11 MMMU
  • MoE "免疫效应"：1→6 并发 TPS 仅降 29% (75→53)

Limitations

  • Vision 长视频受限于位置外推和 context length
  • Thinking 在 ASR/music 感知任务上反而不如 Instruct
  • 语音输出仅 10 种语言（而文本 119 种）
  • "无退化"仅在 30B-A3B 规模验证
  • Code Predictor 的 15 套 head/embed 带来一定参数冗余，未尝试 tied weights

Infrastructure Impact

  • Algorithm: "S1 adapter-first / encoder-second" + S2W Distillation + GSPO
  • Kernel: GQA with QK-Norm, MoE routing (128E top-8 / top-6+shared)，Conv1D streaming, causal ConvNet
  • Framework: vLLM chunked prefill + Thinker/Talker 异步 pipeline；CUDA Graph 加速 MTP+Code2Wav
  • LLM: 证明 "六段专门化" 比 "一个大 backbone" 在 omni 场景下更省算力
  • Agent: 119/19/10 语言的多语 voice agent；Thinker-Talker 解耦使 tool-call / RAG 能介入 text 路径

0. 顶层流水线 (Top-level Pipeline)

💡 滚轮缩放 · 拖动平移 · 顶部页签切换 · 下载 .drawio

设计初衷：传统 "单一大 LM 端到端出音频" 的做法（如 VALL-E）在高并发场景下会把 "文本推理 + 声学细节" 塞进同一个 KV cache，memory-bandwidth 立刻爆掉。Qwen3-Omni 的解法是把四个计算特征截然不同的子任务物理拆开：

子任务计算形状单独调优的维度
文本推理（长 context, 稀疏访存）长序列 ARThinker: 大 d=2048, MoE 128E, GQA 7:1
声学粗粒度（短 context, 实时性强）短序列 AR, 12.5 HzTalker: 小 d=1024, MoE+Shared, GQA 8:1
声学残差（固定 15 步，可 CUDA Graph）固定步长 ARCode Predictor: 5L dense, GQA 2:1
波形渲染（流式卷积）数据并行Code2Wav: causal ConvNet, 无 attention

这四种形状无法被同一套 hyperparameter 同时优化。六段专门化就是工程化的必然选择。

目标：把 16 kHz 原始波形 → 128 维 mel (hop 10ms → 100 Hz) → 12.5 Hz 的 token 流，每 token 3584 维，直接 concat 进 Thinker 的 input 流。

关键设计点 + 初衷：

设计点数值为什么这么设计

2. Vision Encoder — ViT + DeepStack (继承自 Qwen3-VL / SigLIP2)

关键设计点 + 初衷：

设计点数值为什么这么设计
Patch dim 1536 → 工作 dim 1152约 0.75× 收缩SigLIP2 原生 dim 1152，但 patch 层输出 1536 (为跨 SigLIP1/2 兼容)；一次性投到 1152 便于沿用 SigLIP 权重

4. Talker — 语音主干（MoE+Shared Expert AR, 20 层）

Talker 只预测 RVQ 的 zeroth codebook 和流式 hidden，其他 15 个 residual codebook 交给 Code Predictor。这是 Qwen3-Omni 和 VALL-E / Moshi 最根本的区别——Talker 只做"粗粒度声学语义"，不负责全频带细节。drawio 的 Talker 页包含三张子图：5.1 顶层数据流（三路输入融合 + 20 层 + Codec Head）/ 5.2 GQA Self-Attention (16Q/2KV) / 5.3 MoE + Shared Expert (routed 128E top-6 + shared + sigmoid gate)。

Routed expert hidden=384, Shared expert hidden=768shared 2× 大routed expert 专注 "高频变化" (说话人切换、语气剧烈波动)，数量多但每个小；shared expert 承担 "总是要做的基础声学变换"（RMSNorm-style baseline），一个就够但要大
Shared Gate 用 sigmoid 而非恒 1per-tokenDeepSeek-V2 的 shared expert 恒定权重；Qwen3-Omni 加 learnable per-token sigmoid gate。初衷：在某些 token (如 padding, silence) 让 shared 也被部分关闭，省 FLOPs；同时给模型一个 "我现在需不需要 baseline 修正" 的选择权
Codec Head 1024→3072输出 zeroth codebook logits3072 = codec codebook size (每个 codebook 的 code 数)。只预测 codebook 0，其余 1-15 交给 Code Predictor
设计初衷一句话：Talker 是"为 12.5 Hz streaming 定制的迷你 MoE" — 激进的 GQA 8:1 + 三路输入求和 + MoE+Shared 组合让它既能以 140 tok/s 的速度生成 zeroth code，又能对 Thinker 语义保持忠实。

5. Code Predictor (论文中的 MTP Module) — 残差精修器

这是最容易被误解的模块。论文叫 "MTP" (Multi-Token Prediction)，但它既不是 DeepSeek-V3 的 MTP（那是训练 trick），也不是"一次出全部 residuals"的并行 head，而是一个5 层 dense transformer 跑 15 次 AR 迭代、每次用一套独立的 head/embed 预测一个 residual codebook 的模块。这个设计在 drawio 里命名得更准确，叫 "Code Predictor"。drawio 的 Code Predictor 页包含两张子图：6.1 外循环（15 AR 迭代 + 15 套独立 LM head / embed）/ 6.2 单层 Dense Transformer 内部 (GQA 16Q/8KV + SwiGLU FFN 1024→3072)。

关键设计点 + 初衷：

设计点数值为什么这么设计
5 层 dense (不是 MoE)总参 ~80M残差精修不需要 expert 多样性 — 每一步都是"把上一个 residual 细化一点"，属于高度结构化的任务；dense 简单、fixed KV cache 可 CUDA Graph，单帧 14ms
GQA 16Q / 8 KV2:1序列极短 (最多 17 = 1 talker_hidden + 16 codes)，KV cache 不是瓶颈；GQA 2:1 换来每 head 的 K,V 更丰富，提升质量
15 套独立 LM Head / Embedding共 15×(1024×2048 + 2048×1024) ≈ 63M每个 residual codebook 的分布不同（codebook 1 编码大尺度声学细节，codebook 15 编码最细粒度噪声纹理）。共享 head 会导致靠后 codebook 质量塌陷；独立 head + 独立 embed table 是必需的
15 次 AR 迭代 (固定步长)每次输入长度 = 2 + i固定步长 → 可以 CUDA Graph 优化 → 实测 14ms/frame；若并行预测所有 15 个 codebook（如 SoundStream），质量大幅下降因为 codebook 间存在强依赖
summed_embed 反馈给 Talker16 codebook embed 加和Talker 下一步需要完整的"上一帧 codec 全貌"才能继续 AR；从 Code Predictor 把 16 个 code 的 embedding 求和后送回 Talker，闭环
KV cache 完全内部使用每帧重置Code Predictor 的 KV cache 只在单帧 15 轮 AR 内有效，每帧结束清空；因此可以用 CUDA Graph 固定 shape，完全没有动态内存分配
设计初衷一句话：Code Predictor 是一个"为 fixed-step CUDA Graph 定制的残差 AR 模块" — 5 层 dense + 15 套独立 head + 每帧 reset KV cache，这三个设计的共同目标就是让单帧 residual 生成的延迟控制在 20 ms 以内、且可以用 batch_size=N_concurrent 批处理。

关于"MTP"的命名：论文使用 "Multi-Token Prediction" 的说法，但这里的 "multi-token" 指"一帧内多个 codebook token"（而非 DeepSeek-V3 MTP 中的"跨时间步多 token"），含义截然不同，初读时极易混淆。drawio 用 "Code Predictor" 更准确。

6. Code2Wav — 波形渲染器（Causal ConvNet, 200M）

💡 滚轮缩放 · 拖动平移 · 顶部页签切换 · 下载 .drawio

关键设计点 + 初衷：

设计点数值为什么这么设计
Causal ConvNet 替代 DiT200MDiT vocoder 需要 block context (e.g. 8 帧=640 ms) 才能开始合成；Causal Conv 只需 1 帧即可产生输出。首包延迟从 640ms+ 降到 ~3ms
16 个 embedding 加和不是 concatconcat 会让后续 conv 通道数爆炸；加和使 conv 看到的通道始终固定
多阶段 TransposedConv~6 stages12.5 Hz → 24 kHz 需要 1920× 上采样；分多阶段 (2× 2× 2× 3× 5× 4×=480× 或类似) 让每阶段只做温和放大，避免棋盘伪影
Residual Blocks (HiFi-GAN style)Conv1d + activation + skip成熟的 vocoder 结构，hardware-accelerated 在各平台都可跑
Left-context-only (25 帧)仅左向依赖streaming 时不能依赖未来，每次多算 25 帧作为 "causal padding"，输出时裁掉这 25 帧
chunk_size = 10每 10 frame (~800ms) 输出不是每帧立刻输出 PCM（会让 IO 碎片化），而是累积 10 帧后打包；同时保持"首包仍 234ms"（首包只输出 1 chunk）
设计初衷一句话：Code2Wav 的本质是"用 ConvNet 的 streaming 友好性换掉 DiT 的质量优势"— 论文承认音质略逊，但论文强调延迟收益远超质量代价。

8j. 核心训练壁垒（最难复现的一件事）

**"S1 的 adapter-first / encoder-second 两步 + S2 起就混入 cross-modal data"

这对组合是整个 "无退化" 结论的工程命脉**。

  • S1 两步分训决定了 encoder 不去补偿 frozen LLM 的缺陷 → unfreeze 后不退化
  • S2 从头就混多模态数据，而非"先纯文本预训再补 VL/audio" → 让 LLM

表征空间天然容纳多模态，避免后期生硬适配

调度；长音频数据的具体构造（segment 长度、overlap）。**这些才是真正的

know-how 壁垒**。
