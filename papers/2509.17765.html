图的 7 个页签：

1. Top-level Pipeline — 文本/音频/图像/视频 → encoders → Thinker → Talker → Code Predictor → Code2Wav 的全链路

2. Audio Encoder (AuT) — Conv1D 下采样 + 32 层 encoder + downsample/temporal-merge/output-proj

3. Vision Encoder (ViT) — Patch projection + 27 层 encoder + DeepStack + Group 2×2 + MLP

4. Thinker — 28 层 MoE AR，内部展开 GQA Self-Attention 和 MoE FFN (128E top-8)

5. Talker — 20 层 MoE+Shared AR，内部展开 GQA 和 MoE+Shared Expert (128E top-6)

6. Code Predictor — 5 层 Dense Transformer 的单层细节 + 15 次 AR 迭代外循环

7. Code2Wav — RVQ 嵌入 → TransposedConv upsample → Residual Blocks → Causal 输出

Core Contribution

Qwen3-Omni 在 Thinker–Talker 架构上做了"端到端把全模态流水线彻底拆分为 6 个可独立调优的专门化子模型"，每个子模型的结构、维度、注意力形状、expert 数量都针对自己的角色精心定制。核心技术价值不在于"又一个多模态模型"，而在于第一次把"文本推理主干 / 语音主干 / 语音残差精修器 / 波形合成器"的职责彻底解耦，并在 30B-A3B 的总预算下让每个组件都恰好能在 vLLM + CUDA Graph 上跑到 RTF < 1，端到端首包延迟仅 234 ms。

Summary

Qwen3-Omni 的模型结构把一次 omni 推理拆成一条六段流水线，每段的 Transformer 骨架完全不同，维度从 2048→1024→1024 逐步收敛，token rate 从 context 级一路降到 codec 级 (12.5 Hz) 再升到音频采样率 (24 kHz)。总体数据流：

六段结构分别解决：多模态感知 (AuT/ViT) → 语言推理 (Thinker) → 声学粗粒度 (Talker) → 声学细粒度残差 (Code Predictor) → 波形渲染 (Code2Wav)。每一段都写死了自己的 hidden size、head 数、GQA ratio、expert 数，使得运行时的 memory-bandwidth 占用最小，且可以在 vLLM 下用不同的批次策略独立调度。

训练：S1 锁定 LLM 只训 encoder alignment；S2 全参 ~2T tokens；S3 长序列 32K。Post-training = SFT → Strong-to-Weak Distillation → GSPO。

关键结果：text 超越 GPT-4o 和同规模 text-only Qwen3；36 个 audio benchmark 中 32 个开源 SOTA、22 个总体 SOTA；audio 首包 234 ms，video 首包 547 ms，1→6 并发 TPS 仅降 29%。

Key Findings

  • 六段专门化 — 每个子模块都有独立的 (layers, d_model, heads, kv_heads, experts) 配置，精心匹配自己的角色
  • Thinker 的 GQA 比例 7:1 (28Q/4KV) + head_dim=128，是本模型 KV cache 小的关键：每 token 仅 4×128×2 = 1024 个参数 vs dense 28 heads × 128 × 2 = 7168 → 节省 7×
  • Talker 用 MoE + Shared-Expert 的罕见组合，routed 走 128E top-6 (expert hidden=384)，shared expert (hidden=768) 始终激活并用一个 sigmoid gate 做门控，参考 DeepSeek-V2 但加入了 per-token gate
  • Code Predictor 不是 MoE 也不是"一次出全部" — 而是 5-layer dense transformer + 15 AR 迭代 + 15 套独立 LM head / Embedding，每轮产出一个 residual codebook token 并 append 到序列；15 套独立 head 相当于 "每层残差一个专门 classifier"
  • AuT 的 Conv1D × 2 (k=3, s=2) + Temporal Merge = 8× 下采样 把 100 Hz 的 mel 降到 12.5 Hz 的 token rate，恰好对齐 codec rate
  • ViT 的 DeepStack — 从 8/16/24 三层取出中间特征、投影到 3584 维追加给 Thinker，等于在浅、中、深三个层次都给 Thinker "看了一眼"
  • 无退化：同 FLOPs 下，Omni-Base 在 text/vision 都 ≥ 单模态 base；audio 数据反而 +2.11 MMMU
  • MoE "免疫效应"：1→6 并发 TPS 仅降 29% (75→53)

Limitations

  • Vision 长视频受限于位置外推和 context length
  • Thinking 在 ASR/music 感知任务上反而不如 Instruct
  • 语音输出仅 10 种语言（而文本 119 种）
  • "无退化"仅在 30B-A3B 规模验证
  • Code Predictor 的 15 套 head/embed 带来一定参数冗余，未尝试 tied weights

Infrastructure Impact

  • Algorithm: "S1 adapter-first / encoder-second" + S2W Distillation + GSPO
  • Kernel: GQA with QK-Norm, MoE routing (128E top-8 / top-6+shared)，Conv1D streaming, causal ConvNet
  • Framework: vLLM chunked prefill + Thinker/Talker 异步 pipeline；CUDA Graph 加速 MTP+Code2Wav
  • LLM: 证明 "六段专门化" 比 "一个大 backbone" 在 omni 场景下更省算力
  • Agent: 119/19/10 语言的多语 voice agent；Thinker-Talker 解耦使 tool-call / RAG 能介入 text 路径

设计初衷：传统 "单一大 LM 端到端出音频" 的做法（如 VALL-E）在高并发场景下会把 "文本推理 + 声学细节" 塞进同一个 KV cache，memory-bandwidth 立刻爆掉。Qwen3-Omni 的解法是把四个计算特征截然不同的子任务物理拆开：

子任务计算形状单独调优的维度
文本推理（长 context, 稀疏访存）长序列 ARThinker: 大 d=2048, MoE 128E, GQA 7:1
声学粗粒度（短 context, 实时性强）短序列 AR, 12.5 HzTalker: 小 d=1024, MoE+Shared, GQA 8:1
声学残差（固定 15 步，可 CUDA Graph）固定步长 ARCode Predictor: 5L dense, GQA 2:1
波形渲染（流式卷积）数据并行Code2Wav: causal ConvNet, 无 attention

这四种形状无法被同一套 hyperparameter 同时优化。六段专门化就是工程化的必然选择。

1. Audio Encoder — AuT (Attention-based Audio Transformer)

目标：把 16 kHz 原始波形 → 128 维 mel (hop 10ms → 100 Hz) → 12.5 Hz 的 token 流，每 token 3584 维，直接 concat 进 Thinker 的 input 流。

│  QKV Projection  Linear(1280→1280) × 3    │
│  Scaled Dot-Product Attn                  │
│    heads=20, head_dim=64                  │  Flash-Attn + dynamic window (1-8s)
│  O Projection  Linear(1280→1280)          │
│  Residual Add                             │

关键设计点 + 初衷：

设计点数值为什么这么设计
替换 Whisper自研 AuT, 0.6B, 20M 小时Whisper 只优化 ASR 表征；AuT 训 10% audio-understanding 数据，通用音频 (音乐/环境音/情感) 更强
Conv1D×2 + Temporal Merge 下采样100 Hz → 12.5 Hz (8×)必须对齐 codec token rate 12.5 Hz，让 audio 输入和输出在同一时间粒度上，Thinker-Talker 时序对齐不需要额外转换
d=1280, heads=20, head_dim=64—Whisper-large 同配置，利于迁移现成 kernel；head_dim=64 让 flash-attn 友好
Dynamic window attention 1-8s—实时 prefill cache 的必需：全局 attn 无法做 block-wise streaming；固定窗口 attn 又在 offline 长音频上掉点。dynamic window 让同一组权重服务两种场景
FC expansion 4× (1280→5120)标准 FFN沿袭 Whisper encoder，不改结构以便热启动
GELU 而非 SwiGLU—音频 encoder 传统使用 GELU，Whisper 预训数据也是 GELU；换成 SwiGLU 需要重新预训，收益不匹配
Output proj 到 35841280→480→960→35843584 = Thinker 的"视觉-音频对齐隐层"，Thinker 在接收多模态时把所有外部模态都投到 3584 再下投到 2048
设计初衷一句话：AuT 的结构是 "Whisper-style encoder 骨架 + 12.5 Hz 强制下采样 + dynamic window attention"，每一处看起来保守的地方都是为了让同一组权重既能实时 streaming 又能做 40 分钟长音频。

关键设计点 + 初衷：

设计点数值为什么这么设计
Patch dim 1536 → 工作 dim 1152约 0.75× 收缩SigLIP2 原生 dim 1152，但 patch 层输出 1536 (为跨 SigLIP1/2 兼容)；一次性投到 1152 便于沿用 SigLIP 权重
head_dim = 72非 2 幂SigLIP2 原设计；72 = 1152/16，保留 16 heads。72 对 flash-attn 需要特殊 kernel 但已被 Triton 覆盖
FFN 4304 (不是 4608)≈ 3.74× expansionSigLIP2 原设计；非标准比例是为了让总参数落在 400M So400M 规模
SwiGLU 而非 GELU—视觉 encoder 更受益于 gated FFN；而 audio encoder 用 GELU 是热启动考虑
DeepStack（层 8/16/24）3 组 extra tokens不让 Thinker 只看最终层输出。浅层 (边缘/纹理) + 中层 (物体部件) + 深层 (语义) 三路信号同时喂给 Thinker，相当于 "跨层 skip 到 LM"。每组 token 独立投到 3584
Group 2×2 空间合并4× 下采样ViT 输出的 patch 数太多 (1k+)，直接塞 Thinker 会让 LM attention 成本暴涨。2×2 concat 到 4608 再投到 3584，相当于 "每 4 个 patch 合成一个 Thinker token"
同一 ViT 处理 image/video—Video 用 3D RoPE 把时间轴折叠到 2D RoPE 上，同一组权重；实现 image/video 统一
设计初衷一句话：Vision Encoder 故意保留 SigLIP2 的"奇怪维度" (head_dim=72, FFN=4304) 换来权重可以直接复用，然后通过 DeepStack + 2×2 Group 两个独立机制控制 "给 Thinker 的信息量" 和 "给 Thinker 的 token 数"，相当于把"表征质量"和"序列长度"两个旋钮解耦。

关键设计点 + 初衷：

设计点数值为什么这么设计
28 层, d=2048总激活 3B保持与 Qwen3-30B-A3B dense checkpoint 同骨架，便于权重热启动；d=2048 而非 d=3584 是为了 MoE 版本能通过加 expert 横向扩展
GQA 28Q / 4 KV, head_dim=128KV cache = 4·128·2 = 1024 / tok关键数字。若改 dense MHA 28 heads，则 KV cache = 28·128·2 = 7168，在 4K 序列下内存带宽瞬间爆炸。7:1 的 GQA 比例 是实验选出的 "质量-带宽" 最佳点（DeepSeek-V3 用 1:1 MLA 更激进但 kernel 复杂度高）
QK-Norm per headRMSNorm(128) on Q, KQwen3 引入。训练 30B+ 模型时 attention logits 容易发散；在 softmax 前对 Q, K 各自 RMS 归一，相当于把 attention score 限制到稳定范围。代价：多一次 per-head RMSNorm，但 fused kernel 下可忽略
vocab 152 064多语 BPE151 643 regular + ~400 special (包含 audio/image/video 分隔符 token)，与 Qwen3 同
MoE 128E, top-8稀疏 6.25%128E 提供足够表达力；top-8 比 top-2 更稳 (8 个专家加权投票)；实测和 dense 30B 性能持平
Expert hidden=768（非 2 幂）expert FFN = 2048·768·3 ≈ 4.7M128 个 expert × 4.7M = 600M per MoE FFN，28 层 × 600M ≈ 17B (MoE FFN 总参)。若 expert hidden 更大则总参爆炸。选 768 是"在 30B 总预算下能放下 128 个 expert 的最大单 expert 尺寸"
layer-24 hidden → Talkertap 在倒数第 4 层不是最后一层而是 layer 24 (倒数第 4 层)。直觉：最后几层专注 text logits，倒数第 4 层的 hidden 保留更多语义丰富度，更适合作 Talker 的语义 conditioning
"早期多模态混入"训练S2 起就混 text/audio/image/vid不是后期 fine-tuning 加多模态，而是 pretraining 阶段就混。这让 Thinker 的 representation space 天然支持多模态，也是 "无退化" 的关键
设计初衷一句话：Thinker 的每一个维度选择都指向同一个目标——让一个 30B 的 MoE 模型在 vLLM 下处理 4K-32K context 时，KV cache IO 不超过 dense 3B 的水平，同时用 top-8 稀疏激活抵消 dense-MoE 的质量损失。

关键设计点 + 初衷：

设计点数值为什么这么设计
d=1024（Thinker 的一半）激活参数 ~0.3B语音粗粒度 token 只需要覆盖韵律/情感/说话人，维度需求远小于语言推理。把 d 降一半意味着 KV cache/FLOPs 都降 4×
GQA 16Q / 2 KVKV cache = 2·128·2 = 512 / tok比 Thinker 的 GQA 还激进 (8:1)。原因：Talker 的 context 短 (单次对话几秒钟语音), KV cache IO 对质量影响小，激进 GQA 换来吞吐
三路输入加性融合（text / hidden / codec_embed）都投到 1024 后求和text 项提供语义；hidden 项提供语义 + 韵律连续性；codec_embed 项提供自回归上下文 (上一个 code)。不用 concat 而用 sum 是为了固定 context 宽度 (concat 会让 d=3072)
文本路径先经 SiLU 再投到 1024Lin→SiLU→Lin先非线性再投影，避免 text hidden 的分布直接污染 codec 空间
Codec Embedding vocab=307216 codebook × 192 index?每个 codec 层可能有 192 or 3072/16=192 codes (多 codebook 共享 embedding table)。Embed dim 1024 直接对齐主干
MoE 128E top-6激进 6/128=4.7% 稀疏top-6 比 Thinker 的 top-8 更稀：语音生成对 expert 多样性需求低，6 已足够
Routed expert hidden=384, Shared expert hidden=768shared 2× 大routed expert 专注 "高频变化" (说话人切换、语气剧烈波动)，数量多但每个小；shared expert 承担 "总是要做的基础声学变换"（RMSNorm-style baseline），一个就够但要大
Shared Gate 用 sigmoid 而非恒 1per-tokenDeepSeek-V2 的 shared expert 恒定权重；Qwen3-Omni 加 learnable per-token sigmoid gate。初衷：在某些 token (如 padding, silence) 让 shared 也被部分关闭，省 FLOPs；同时给模型一个 "我现在需不需要 baseline 修正" 的选择权
Codec Head 1024→3072输出 zeroth codebook logits3072 = codec codebook size (每个 codebook 的 code 数)。只预测 codebook 0，其余 1-15 交给 Code Predictor
设计初衷一句话：Talker 是"为 12.5 Hz streaming 定制的迷你 MoE" — 激进的 GQA 8:1 + 三路输入求和 + MoE+Shared 组合让它既能以 140 tok/s 的速度生成 zeroth code，又能对 Thinker 语义保持忠实。

5. Code Predictor (论文中的 MTP Module) — 残差精修器

这是最容易被误解的模块。论文叫 "MTP" (Multi-Token Prediction)，但它既不是 DeepSeek-V3 的 MTP（那是训练 trick），也不是"一次出全部 residuals"的并行 head，而是一个5 层 dense transformer 跑 15 次 AR 迭代、每次用一套独立的 head/embed 预测一个 residual codebook 的模块。这个设计在 drawio 里命名得更准确，叫 "Code Predictor"。

关键设计点 + 初衷：

设计点数值为什么这么设计
5 层 dense (不是 MoE)总参 ~80M残差精修不需要 expert 多样性 — 每一步都是"把上一个 residual 细化一点"，属于高度结构化的任务；dense 简单、fixed KV cache 可 CUDA Graph，单帧 14ms
GQA 16Q / 8 KV2:1序列极短 (最多 17 = 1 talker_hidden + 16 codes)，KV cache 不是瓶颈；GQA 2:1 换来每 head 的 K,V 更丰富，提升质量
15 套独立 LM Head / Embedding共 15×(1024×2048 + 2048×1024) ≈ 63M每个 residual codebook 的分布不同（codebook 1 编码大尺度声学细节，codebook 15 编码最细粒度噪声纹理）。共享 head 会导致靠后 codebook 质量塌陷；独立 head + 独立 embed table 是必需的
15 次 AR 迭代 (固定步长)每次输入长度 = 2 + i固定步长 → 可以 CUDA Graph 优化 → 实测 14ms/frame；若并行预测所有 15 个 codebook（如 SoundStream），质量大幅下降因为 codebook 间存在强依赖
summed_embed 反馈给 Talker16 codebook embed 加和Talker 下一步需要完整的"上一帧 codec 全貌"才能继续 AR；从 Code Predictor 把 16 个 code 的 embedding 求和后送回 Talker，闭环
KV cache 完全内部使用每帧重置Code Predictor 的 KV cache 只在单帧 15 轮 AR 内有效，每帧结束清空；因此可以用 CUDA Graph 固定 shape，完全没有动态内存分配
设计初衷一句话：Code Predictor 是一个"为 fixed-step CUDA Graph 定制的残差 AR 模块" — 5 层 dense + 15 套独立 head + 每帧 reset KV cache，这三个设计的共同目标就是让单帧 residual 生成的延迟控制在 20 ms 以内、且可以用 batch_size=N_concurrent 批处理。

关于"MTP"的命名：论文使用 "Multi-Token Prediction" 的说法，但这里的 "multi-token" 指"一帧内多个 codebook token"（而非 DeepSeek-V3 MTP 中的"跨时间步多 token"），含义截然不同，初读时极易混淆。drawio 用 "Code Predictor" 更准确。

6. Code2Wav — 波形渲染器（Causal ConvNet, 200M）


RVQ Codes  [1, 16, T_codec]                   # 16 个 codebook 层，T=帧数
  │
  ▼
16 RVQ Codebooks  (16 个独立 embedding table，与 Code Predictor 同步)
  │
  ▼
Lookup + Sum Embeddings  [1, emb_dim, T]      # 16 路 embed 相加（加性融合）
  │
  ▼
TransposedConv Upsample  多阶段              # 多次 stride>1 的反卷积
  │                                           # 12.5 Hz → 24 kHz 需要 1920× 上采样
  ▼                                           # 通过多阶段（~6 个）反卷积逐步放大
Residual Blocks (Conv1D + activation + skip) # HiFi-GAN 风格残差块
  │
  ▼
Final Conv  → [1, 1, T·up]                   # 收到 mono 波形
  │
  ▼
Trim Context  remove left_context=25         # 每次多算 25 帧用于 causal，输出时裁掉
  │
  ▼
PCM Output  float32 @ 24 kHz, chunk_size=10   # 每 10 帧 (= 800ms) 输出一个 chunk

关键设计点 + 初衷：

设计点数值为什么这么设计
Causal ConvNet 替代 DiT200MDiT vocoder 需要 block context (e.g. 8 帧=640 ms) 才能开始合成；Causal Conv 只需 1 帧即可产生输出。首包延迟从 640ms+ 降到 ~3ms
16 个 embedding 加和不是 concatconcat 会让后续 conv 通道数爆炸；加和使 conv 看到的通道始终固定
多阶段 TransposedConv~6 stages12.5 Hz → 24 kHz 需要 1920× 上采样；分多阶段 (2× 2× 2× 3× 5× 4×=480× 或类似) 让每阶段只做温和放大，避免棋盘伪影
Residual Blocks (HiFi-GAN style)Conv1d + activation + skip成熟的 vocoder 结构，hardware-accelerated 在各平台都可跑
Left-context-only (25 帧)仅左向依赖streaming 时不能依赖未来，每次多算 25 帧作为 "causal padding"，输出时裁掉这 25 帧
chunk_size = 10每 10 frame (~800ms) 输出不是每帧立刻输出 PCM（会让 IO 碎片化），而是累积 10 帧后打包；同时保持"首包仍 234ms"（首包只输出 1 chunk）
设计初衷一句话：Code2Wav 的本质是"用 ConvNet 的 streaming 友好性换掉 DiT 的质量优势"— 论文承认音质略逊，但论文强调延迟收益远超质量代价。

跨模块规律：

  • head_dim 统一为 128（视觉 encoder 72 除外，因为继承 SigLIP2）— 因为 NVIDIA / AMD flash-attn kernel 都最优化 head_dim=64/128；其他维度会降速 2-5×
  • Thinker GQA 7:1 最保守，Talker GQA 8:1 更激进 — 因为 context 长度和 KV cache 敏感度不同
  • Code Predictor 的 KV cache (2048) 反而最大 — 因为它层数最少，不能牺牲每 head 的 K/V 容量；但由于序列超短（≤17），总 KV footprint 仍很小
  • FFN expansion 差异显著：Thinker expert 只 1.5× (2048→768×2 via gate×up)；Talker routed 2× (1024→384×2)；Code Predictor 3× (1024→3072)。原因：MoE 的 expert 数量本身提供"FFN 容量"，单 expert 不需大；dense 层必须通过扩展 FFN 换容量

8. 训练 (保留原分析)

  • 三阶段预训：S1 锁 LLM 训 encoder alignment；S2 全参 ~2T tokens (text 0.57T + audio 0.77T + image 0.82T + video 0.05T + video-audio 0.05T)；S3 长序列 8K→32K
  • Post-training (Thinker): SFT → Off-policy Distillation → On-policy Distillation (teacher Qwen3-32B/235B) → GSPO
  • Post-training (Talker): 大规模 speech mapping → CPT → multilingual DPO → speaker fine-tuning
  • 训练壁垒：S1 的 "adapter-first, encoder-second" 两步训练 (LLM 冻结) 替代传统 "encoder+adapter 联合训练"；论文认为这是防止 encoder 补偿冻结 LLM 导致感知退化的关键

9. Performance (保留原分析)

  • Text: MMLU-Redux 89.3, GPQA 70.4 (>GPT-4o 69.7), AIME25 61.3, ZebraLogic 90.0
  • Audio: Librispeech WER 1.22 (>Seed-ASR 1.58), MMAU 77.5 (>Gemini-2.5-Pro)
  • Vision: MathVista 75.9, 与 Qwen2.5-VL-72B 相当
  • Speech Gen: Seed TTS test-en WER 1.39, 10 语言
  • Latency: Audio 首包 234 ms, Video 547 ms, RTF 0.47-0.66
