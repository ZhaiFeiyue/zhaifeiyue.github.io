2. 本篇 vs 相关论文的 delta #

ForkKV vs ICaRus [2603.13281]：ICaRus 将 Transformer 分解为 frozen logical encoder ($E_{\text{base}}$) 和 task-specific logical decoder ($D_{\text{task}}$)，所有 models 共享一份 KV cache。ForkKV 的核心区别：ICaRus 需要 retraining (fine-tune only decoder)，而 ForkKV 是 inference-time system optimization 无需模型修改——它利用 LoRA 的代数结构 ($Y = xW + xA_iB_i$) 在 serving 层面分离 cache。ICaRus 实现 "lossless" sharing (decoder 被训练适应 base encoder cache)，ForkKV 是 "near-lossless" (0.71% quality drop from sharing bCache without retraining) [2604.06370]。

两者面对的 trade-off 不同：ICaRus 要求 training investment 但获得 exact sharing；ForkKV 即时部署但接受微小 accuracy cost。在实际部署中，如果模型已按 ICaRus 方式训练，则不再需要 ForkKV——两者解决同一问题的不同层面。

ForkKV vs PrefillShare [2602.12029]：PrefillShare 也 freezes base model prefill 并训练 cache-conditioned decode modules。核心差异：PrefillShare 需要 cache-conditioned fine-tuning ($\mathcal{L}(\theta_\text{dec})$ trained to decode from "alien" cache)，而 ForkKV 不修改任何模型参数。PrefillShare 适用于 multi-model (不同 architecture 的 specialized models) 场景；ForkKV 限于 same-base-model + different-LoRA 场景。

矛盾点: PrefillShare 报告 4.5× lower p95 latency / 3.9× throughput，ForkKV 报告 up to 3.04× throughput [2604.06370]。但 PrefillShare 的 gain 部分来自 disaggregated P/D 架构 (DistServe-style) + eliminated redundant prefill，不纯是 cache sharing。ForkKV 的 gain 纯来自 memory efficiency → larger batch size。两者不直接可比。

ForkKV vs Harli [2511.11729]：Harli 在 decode GPU 上 co-locate PEFT finetuning，利用 decode 的 ~60% idle SM capacity。与 ForkKV 的联系：两者都在 LoRA/PEFT 场景下优化 GPU utilization，但不同维度——Harli 优化 compute (SM) utilization，ForkKV 优化 memory (KV cache) utilization。组合使用时，ForkKV 释放的内存可以给 Harli 更多空间 co-locate finetuning workloads。

ForkKV vs vLLM [2604.06370]：vLLM 的 PagedAttention + prefix caching 在 multi-LoRA 下效果为零——identical prefix 因 LoRA 不同产生不同 KV cache，throughput drops 90.8% scaling 1→8 workflows。ForkKV 的 DualRadixTree 在同一场景下实现 2.33× throughput by sharing bCache。这证明 monolithic KV cache assumption 是 multi-LoRA serving 的根本瓶颈。

3. 可攻击面 #

  1. 99.4% similarity 缺乏理论保证：cosine similarity >99.4% 是经验观察 [2604.06370]，无 worst-case bound。更高 rank ($r \geq 64$) 或更多样化的 adapters 可能打破此假设——论文仅测 rank ≤ 32。
    1. 低并发下性能退化：4 ReAct workflows 时 ForkKV 反而不如 baseline [2604.06370]——disaggregated KV 的计算开销 (deferred RoPE, SRAM reconstruction) 在内存不紧张时成为纯开销。论文提到 adaptive scheduling 但未实现。
      1. Deferred RoPE 的系统约束：$rCache = xA_i$ 维度为 $r$ (不兼容 RoPE $n$-dimensional rotation matrix)，迫使 RoPE 推迟到 attention time [2604.06370]。这意味着 (a) 每次 attention 必须携带 sin-cos table 到 SRAM，(b) 未来的 speculative decoding / paged KV offloading 需要额外处理 RoPE state，增加集成复杂度。
        1. Concurrent work LRAgent 的独立验证：Jeon et al. (2026) 独立提出相同 bCache/rCache 分解 [2604.06370]——convergent discovery 证明 idea 的自然性，但也意味着 ForkKV 的 novelty 在方法层面有限，差异化在 DualRadixTree + ResidualAttention kernel 工程。

          2. 4. 生态位 #

          ForkKV 占据 multi-LoRA agent serving 的精确 niche：当多个 LoRA-specialized agents 共享一个 base model 并处理 long shared context 时，它是唯一不需要模型修改即可获得 memory sharing 的系统方案。

          vs ICaRus/PrefillShare：需要 retraining/fine-tuning → higher accuracy guarantee, lower deployment flexibility。vs ForkKV：zero training cost → lower accuracy guarantee (0.71% drop), immediate deployment。

          采纳信号：Built on SGLang (3K lines)，OpenAI-compatible API 不变。来自 SJTU。未开源（截至论文时间）。

          5. 未探索方向 #

          1. ForkKV + ICaRus 两阶段方案：先用 ForkKV 即时部署获得 near-lossless sharing；后台异步做 ICaRus-style decoder fine-tuning；完成后热切换到 lossless mode。渐进式从 inference-time approximate 升级到 training-time exact。
            1. Adaptive disaggregation granularity：当前全层分解 bCache/rCache。观察到某些 layer 的 $x$ similarity 可能低于 99.4%（特别是后层）——选择性地对 high-divergence layers 回退到 per-agent full KV，在 memory 和 quality 间动态平衡。
              1. ForkKV for MLA (Multi-Latent Attention)：DeepSeek-V3 式 MLA 的 latent KV ($d_{kv} = 192$) 远小于标准 attention，LoRA 在 MLA 上的分解结构可能不同。验证 ForkKV 在 MLA 架构下的适用性是重要扩展。
                1. Cross-node ForkKV with disaggregated serving：将 bCache 放在 decode nodes (read-only shared)，rCache 本地生成——结合 P/D disaggregation 可以进一步减少 prefill redundancy。
