Microbenchmarking NVIDIA's Blackwell Architecture: An in-depth Architectural Analysis
Aaron Jarmusch, Sunita Chandrasekaran · arXiv: 2512.02189 · 2025-12
首篇 Blackwell B200 微架构实测：TMEM、tcgen05、FP4 三大创新的性能考古学

核心论点

NVIDIA 只发布高层 whitepaper，微架构参数严重缺失。用精心设计的 PTX 微基准可以"逆向工程"出 Blackwell 的真实性能参数。

核心方法

开发首个 Blackwell B200 PTX 微基准套件：dependency chain 隔离延迟、back-to-back issuing 测吞吐、PTX-to-SASS 翻译验证确保测量准确性。

核心结论

TMEM 延迟 -58%，tcgen05 延迟降低 2.9-11.6x，FP4 达 7.7 PFLOPS (96.3% 峰值)。LLM 推理快 1.57-1.59x，训练快 1.56x 且能效提升 42%。

🗺 研究背景和现状

A · 时代定位

2024-2025年，GPU 架构从"训练优先"转向"推理优先"。Hopper 已将训练性能推到很高水平，但 LLM 推理的爆发式增长使推理效率成为核心诉求。Blackwell 是 NVIDIA 首个明确以 post-training 和 inference efficiency 为设计重心的数据中心 GPU——这不是渐进式改良，而是架构哲学的转变。然而关键微架构参数完全未知，本文做的是硬件考古。

B · 领域定位

方向：GPU 微架构分析与硬件性能建模
子任务：通过微基准测试量化新 GPU 架构的指令延迟、内存带宽、计算吞吐等微架构参数
实际问题：kernel 开发者无法为 Blackwell 做 informed 的性能建模——例如不知道 TMEM 延迟是 100 cycles 还是 1000 cycles，无法决定是否值得重写 kernel

C · 核心痛点

①性能：现有方法能 profile 聚合指标，但无法区分 TMEM 访问和 L2 cache 访问的延迟差异
②效率：模拟器（Accel-Sim）需要微架构参数作为输入配置——而这些参数正是未知的（循环依赖）
③泛化：每代架构新增的硬件单元行为都需要从零测量，没有通用方法

D · 本文目标 & 设计空间约束

目标：能否用微基准测试精确量化 Blackwell B200 三大架构创新（TMEM、tcgen05、DE）的性能参数？

为何不可用 nvprof/nsight 直接测？ profiling 工具粒度不够细且有测量 overhead，无法隔离 TMEM 的 420 cycle 延迟。

为何不可用模拟器？ Accel-Sim 等需要微架构参数作为输入——经典循环依赖。

为何必须用 PTX？ CUDA 编译器会做指令重排/register 分配等优化，无法隔离特定硬件行为。PTX 是唯一能精确测量单条指令延迟的途径。

结论：排除以上方案后，剩余可行设计空间是"PTX 级微基准 + SASS 翻译验证"。

E · 提出框架

通过开发首个 Blackwell B200 PTX 微基准测试套件，使用 dependency chain 隔离特定硬件单元延迟，并通过 PTX-to-SASS 翻译映射验证测量准确性，实现对三大架构创新的系统性量化。现有方法只能观测聚合性能指标；本文方法直接操作 PTX 指令，绕过编译器优化干扰。

F · 技术挑战

  • 挑战：PTX 编译器可能引入意外优化 → 应对：逐条验证 SASS 翻译结果 已解决
  • 挑战：tcgen05 不同精度编译成完全不同的 SASS 指令 → 应对：建立完整 PTX-SASS 映射表 已解决
  • 挑战：TMEM cache-miss vs cache-hit 延迟混合 → 应对：pointer-chase 保证 cache-miss 部分解决
  • 挑战：FP6 缺乏软件工具链支持 → 应对：仅测量硬件层面吞吐 部分解决

🏗 核心架构图解

"Why not" 继续用 wgmma？

wgmma 是 warp-group 级指令（128 线程同步），调度粒度过粗。tcgen05 降至 warp 级（32 线程），消除 warp-group 同步空闲周期，调度自由度质变——每个 warp 可独立发射 MMA。

"Why" TMEM 设计？

传统路径 SMEM→RF→TC 延迟高、RF 容量有限。TMEM 新增 256KB/SM 专用片上存储（16 TB/s 读），accumulator 直接驻留 TMEM，消除每秒约 12 TB 的冗余数据搬运。

核心技术壁垒

对 PTX-to-SASS 翻译的深度逆向理解。发现 tcgen05.mma 不同精度编译成完全不同 SASS：FP4→OMMA（全新指令），FP32→HMMA，FP8→QMMA。ISA 层面逆向能力是整个测量可信的基础。

假设挑战

论文假设 NV-HBI 使双 die 对软件完全透明。但实际 NUMA-like 效应（跨 die 内存访问延迟差异）未被探索。多 GPU 场景下通信干扰也未系统考虑。CUDA 13.0 preliminary 支持可能仅反映硬件上限。

🔬 第一性原理
约束推导 Constraint

为什么不能用 nvprof/nsight 直接测？

对立观点：Profiling 工具已经足够详细

Profiler 报告的是聚合指标（如 SM occupancy、memory throughput），无法区分 TMEM 420 cycle 和 L2 cache ~100 cycle 的延迟差异。用 pointer-chase 微基准隔离才能测准单条指令延迟。此外 profiling 有 overhead，会干扰 pipeline 行为测量。

Profiling 看的是"平均体温"，微基准才能做"CT 扫描"。
约束推导 Constraint

为什么 FP32 accumulator 必然使吞吐减半？

对立观点：只是软件优化不够

FP16→FP16 吞吐 964.8 TFLOPS，FP16→FP32 仅 482.4 TFLOPS。精确 2x 差距说明这是硬件 datapath 宽度决定的：32-bit accumulator 需要 2x 的 register bandwidth 来写回结果。乘法单元是共享的，bottleneck 在 accumulator 写端口。

吞吐瓶颈不是乘法单元，而是 accumulator datapath——硬件限制无法软件绕过。
假设挑战 Assumption

FP4 的 8.2% perplexity 退化是否可接受？

论文假设：模型级平均退化反映实际退化

+8.2% 是模型级平均值，但实际 per-layer 退化分布未分析。某些 attention 层可能退化远超 8.2%，某些 FFN 层几乎无退化。"some layers tolerate FP4 while others need FP8" 只在 Discussion 一笔带过。MoE 模型中不同 expert 的退化分布也未知。

平均退化掩盖了分布——关键层的 catastrophic 退化才是实际部署的杀手。
设计选择 Design Choice

为什么 tcgen05 恒定延迟意味着"空间阵列"？

Hopper wgmma 延迟随 tile 宽度线性增长

Hopper: m64n64k16→32c, m64n128k16→64c, m64n256k16→128c，延迟与 N 维度线性相关，说明 pipeline 按时间展开。Blackwell: 不论 tile size，延迟恒定 ~11 cycles。唯一解释：Tensor Core 是空间展开的阵列，更大 tile 只是激活更多硬件单元，不增加 pipeline 深度。

恒定延迟是空间阵列设计的"指纹"——tile size 影响吞吐而非延迟。
设计选择 Design Choice

为什么 1.27x 加速比对所有共享精度一致？

FP8/FP16/BF16/TF32/INT8 的 B200/H200 比都是 1.27x

推算: SM 数量 148/132≈1.12x × 频率提升 ~1.13x ≈ 1.27x。这说明共享精度的 Tensor Core 架构没有质变，性能提升来自"更多 SM + 更高频"的线性缩放。真正的架构红利体现在 TMEM 延迟改善和 FP4/FP6 新精度上。

1.27x 一致比暴露了增量来源——SM 增量 × 频率提升的乘法效应。
假设挑战 Assumption

TMEM 420 cycle 延迟的代表性

论文在 cache-miss 场景下测量

420 cycle 是 pointer-chase 强制 cache-miss 的结果。实际 workload 中 TMEM 的 cache-hit 比例和混合延迟未给出。如果 Tensor Core workload 中 TMEM 访问大部分 cache-hit，实际延迟可能远低于 420 cycles。结论的外推性取决于 hit-rate。

Cache-miss 延迟是"worst case"基准——实际性能很可能更好，但多好未知。
📖 双栏批注主体
§III Blackwell Architecture Overview
¹ The Blackwell architecture represents NVIDIA's most significant architectural evolution for datacenter GPUs. The B200 GPU features a dual-die design connected via the NV-HBI (High-Bandwidth Interface), presenting a unified 192GB HBM3e memory space to software.

² Three key innovations define Blackwell: (1) Tensor Memory (TMEM), a 256KB per-SM dedicated on-chip memory for Tensor Core operations; (2) the tcgen05 instruction set, reducing execution granularity from warp-group to warp level; and (3) a hardware Decompression Engine supporting LZ4, Snappy, GZIP, and other formats.

³ The transition from warp-group to warp-level execution is not merely a granularity change — it represents a fundamental shift in scheduling freedom, where each warp can independently issue MMA operations.
研究动机

¹ 开篇定位 Blackwell 为"最重大的架构演进"。208B 晶体管的 dual-die 设计在单 GPU 产品中是首次。NV-HBI 使软件无需感知物理双 die。

核心方法

² 三大创新的层次：TMEM 解决数据搬运问题，tcgen05 解决调度粒度问题，DE 解决压缩数据传输问题。三者相互协同。

⚠ 难点提示

³ 理解 warp-group→warp 的意义需要知道：Hopper 的 wgmma 要求 4 个 warp 锁步同步，任何一个 warp stall 就会拖慢其他 3 个。Blackwell 消除了这种耦合。

🚀 科研改进方向

本文作为首篇 Blackwell 微架构实测论文，为社区提供了急需的底层数据。后续研究可从以下方向深入，将微基准数据转化为实际系统优化。

1. Per-Layer 混合精度策略 (Mixed-Precision Selection)

难度: ⭐⭐ · 适合: 量化方向研究者

FP4 的 +8.2% perplexity 退化要求 per-layer precision selection。利用本文 FP4/FP6/FP8 的精确吞吐数据，设计自适应精度分配算法。

  • 基于 layer sensitivity analysis 的精度分配（参考 GPTQ、AWQ）
  • 在 FP4/FP6/FP8 三个精度间做 Pareto 最优搜索
  • 利用 accumulator 吞吐减半的约束优化 accumulator 精度选择

2. TMEM-Aware Kernel 优化 (Tiling Strategy)

难度: ⭐⭐⭐ · 适合: 高性能计算/kernel 开发者

TMEM 最优 tile 从 32×32 变为 64×64，所有 GEMM/Attention kernel 需重新 tune。

  • 设计 TMEM 双缓冲 pipeline（利用 tcgen05.cp 异步传输）
  • Chained MMA 优化：保持中间结果在 TMEM，消除冗余数据搬运
  • Triton/CK 编译器 Blackwell 后端适配

3. Dual-Die NUMA 效应探索

难度: ⭐⭐⭐ · 适合: 系统架构研究者

论文未量化 NV-HBI 双 die 通信开销。需要设计跨 die 访问的微基准。

  • 测量跨 die 内存访问延迟 vs 本地 die 延迟
  • 多 SM 并发访问时的 NV-HBI 带宽饱和点
  • NUMA-aware kernel scheduling 策略

4. 解压引擎与推理 Pipeline 集成

难度: ⭐⭐ · 适合: 推理框架开发者

DE 在 Bitcomp 下达 462 GB/s 输出吞吐，可透明解压 HBM 中的压缩权重。

  • 设计 DE 与 GEMM kernel 的 overlap pipeline
  • 压缩格式选择优化（Bitcomp vs ANS vs LZ4 的吞吐-压缩率 Pareto）
  • 在 vLLM/TRT-LLM 中集成压缩模型加载

B · 实践部署上下文

推理：FP4 适合 decode 阶段（memory-bound），FP8 适合 prefill 阶段（compute-bound）。TMEM 对小 batch 推理延迟优化尤为关键（BS=1 延迟 12.3ms）。
训练：FP32 accumulator 吞吐减半是主要约束。推荐推理用 FP16 accumulator、训练用 FP32。
框架适配：vLLM/TRT-LLM 需要显式适配 tcgen05 + TMEM 数据通路，当前进度未知。FP6 硬件支持但缺软件工具链。
算力-带宽比：FP4 下 \(\text{Ops:Byte} = 7702.5 / 8.0 = 963\)，更多 workload 从 memory-bound 转向 compute-bound。

Blackwell 标志着 GPU 架构从"训练至上"到"推理优先"的范式转换。TMEM + tcgen05 的组合不仅是性能提升，更是数据通路哲学的重塑。未来 5 年的主航道是：如何在越来越低的精度下（FP4→FP2？），通过越来越聪明的硬件-软件协同设计，实现"free lunch"式的推理加速——不牺牲精度的前提下挖掘硬件红利。这篇论文提供了第一份可信的路线图。

什么是 TMEM？它与传统 Shared Memory 有何本质区别？

TMEM (Tensor Memory) 是 Blackwell 新增的每 SM 256KB 专用片上内存，2D 结构 (512×128×32bit)，专为 Tensor Core 操作设计。与 SMEM 的本质区别在于：(1) TMEM 只有 Tensor Core 可以直接读写，通用计算无法访问；(2) accumulator 可直接驻留 TMEM，消除经过 RF 的间接路径；(3) 提供 16 TB/s 读 / 8 TB/s 写的超高带宽。

tcgen05 指令相比 Hopper 的 wgmma 有哪些关键变化？这些变化意味着什么？

三大关键变化：(1) 执行粒度从 warp-group (128线程) 降至 warp (32线程)，每个 warp 可独立发射 MMA；(2) 操作数来源新增 TMEM 直接通路（SMEM+TMEM 替代 SMEM+RF）；(3) 延迟从随 tile 线性增长 (32-128 cycles) 变为恒定 (~11 cycles)，说明从时域流水线转为空间阵列设计。

为什么 FP32 accumulator 使 Tensor Core 吞吐精确减半？

FP16→FP16 吞吐 964.8 TFLOPS vs FP16→FP32 仅 482.4 TFLOPS，精确 2x 差距。瓶颈不在乘法单元（共享），而在 accumulator 写回 datapath——32-bit accumulator 需要 2x 的 register/TMEM 写端口带宽。这是硬件固有限制，无法通过软件绕过。

"恒定延迟"如何证明 Blackwell 采用了空间阵列设计？

Hopper wgmma 延迟: m64n64→32c, m64n128→64c, m64n256→128c（与 N 线性相关），说明计算按时间展开——更大 tile 需要更多 pipeline cycles。Blackwell tcgen05: 不论 m64n64 还是 m256n256，延迟恒定 ~11 cycles。唯一解释是计算按空间展开——更大 tile 激活更多硬件单元（并行度增加），但 pipeline 深度不变。

各精度之间 B200/H200 加速比为何恒定在 1.27x？

对 FP8/FP16/BF16/TF32/INT8 等共享精度，加速比一致为 1.27x。推算来源: SM 数量增加 148/132≈1.12x × 频率提升 ~1.13x ≈ 1.27x。这说明共享精度的 Tensor Core 架构没有质变——性能提升来自线性缩放。真正的架构红利体现在 TMEM 延迟改善和 FP4/FP6 新精度。

硬件解压引擎 (DE) 为什么呈现"decompression-throughput bounded"特征？

实验中输出吞吐稳定在 160-220 GB/s 不随压缩率变化，而输入吞吐从 173 GB/s (1x) 降到 0.85 GB/s (245x)。这证明 DE 内部有 ~200 GB/s 的解压吞吐天花板，是内部处理速度而非 HBM 输入带宽限制了整体表现。32KB chunk 最优 16 路并发，更大 chunk 消耗更多硬件资源，迫使降低并行度。

B200 训练 1.56x 提升的来源可以分解为哪些因素？

SM 增量 (1.09x) × CTA pairing (1.27x) × TMEM (1.26x) 的乘法效应 ≈ 1.56x。CTA pairing 通过相邻 CTA 共享操作数减少冗余搬运；TMEM 驻留中间结果降低 L2 thrashing 和 DRAM 访问。能效提升 42% (22.2 vs 15.6 tok/s/W) 尽管功耗高 14%。
