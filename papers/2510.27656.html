fabric-lib: RDMA Point-to-Point Communication for LLM Systems

粗读 Overview

Core Contribution

fabric-lib提供了一个可移植的RDMA点对点通信库，通过ImmCounter原语在不依赖消息顺序的前提下实现完成通知，统一抽象了ConnectX-7和AWS EFA等异构NIC硬件，在三个生产系统（KvCache传输、RL权重更新、MoE dispatch/combine）中实现了400 Gbps峰值吞吐和state-of-the-art延迟。

核心三问

Q1: 解决什么问题？

新兴LLM系统模式（disaggregated inference、MoE routing、异步RL fine-tuning）需要灵活的点对点通信，但现有RDMA方案存在严重的vendor lock-in问题。DeepEP依赖ConnectX独有的IBGDA，NVSHMEM在EFA上性能严重退化，Mooncake和NIXL缺乏EFA支持。云厂商部署不同RDMA实现（ConnectX用RC transport有序交付，AWS EFA用SRD协议无序交付），没有可行的跨provider方案。

Q2: 怎么解决的？

核心洞察：ConnectX RC和EFA SRD都支持可靠但无序的交付语义。就像建了一个"通用翻译层"——不管底层NIC说什么"方言"（RC/SRD），fabric-lib都用同一套"普通话"（reliable-but-unordered + ImmCounter）来对话。ImmCounter通过轮询completion queue并在收到immediate value时递增计数器来实现完成通知，完全不依赖消息顺序。库用Rust实现，透明管理多NIC聚合（EFA需要4×100Gbps才达400Gbps），暴露Send/Recv和WriteImm两类操作。

Q3: 效果如何？

  • 点对点吞吐：ConnectX-7和EFA均达到400 Gbps峰值
  • KvCache传输：层级传输被计算完全隐藏，128K序列TTFT增加仅1.9%（16735→17056ms）
  • RL权重更新：1T参数模型仅需1.3秒，比现有框架快100×+
  • MoE decode：ConnectX-7上超越DeepEP成为新SOTA，EFA上首次实现可行延迟（比pplx-kernels快3-6×）
  • 生产部署于Perplexity AI

逻辑故事还原

背景：

LLM推理进入disaggregated时代，MoE成为主流架构，RL训练需要频繁同步权重。这些场景都需要灵活的P2P通信，但集体通信库（NCCL/torch.distributed）的固定membership、同步初始化、操作排序等限制使其不适合。RDMA本应是最佳选择，但ConnectX和EFA的协议差异（有序vs无序、连接vs无连接）导致没有统一的跨硬件解决方案。

破局（类比）：

想象你要在两个说不同语言的国家之间建立邮政系统。一个国家（ConnectX）保证信件按发送顺序到达，另一个（EFA）只保证信件都能到但顺序随机。fabric-lib的做法是：我不依赖顺序，而是在每封信上贴一个编号（ImmCounter），收件人只需数清楚收到了几封编号为X的信就知道传输完成了。这样两个"国家"的邮政系统都能用同一套规则。

拆解：

1. TransferEngine核心（Section 3）：Rust实现，per-GPU worker线程，NUMA-aware分配，lock-free queue，透明多NIC管理。API包含Send/Recv（RPC风格）、WriteImm（单次/分页/scatter）、UVM Watcher（GPU→CPU通知，CUDA Graph兼容）

2. 三大应用系统：KvCache传输（Section 4，层级流水线overlap）、RL权重更新（Section 5，全集群带宽利用+4阶段流水线）、MoE dispatch/combine（Section 6，host proxy + NVLink/RDMA协同）

3. 硬件特化优化：EFA用libfabric + WR templating，ConnectX用libibverbs + WR chaining + relaxed ordering + 双QP分离（Send/Recv与WriteImm各一个避免干扰）

Key Details & Takeaway

Detail 1: ImmCounter的正确性依赖PCIe ordering

ImmCounter能在无序传输下正确工作的关键在于PCIe的写顺序保证：RDMA规范要求WriteImm的data payload在immediate value之前发出，PCIe switch保证指向同一设备的写按序到达。虽然immediate value目标是CPU而data目标是GPU（不同设备），但host-proxy架构巧妙避免了data race——CPU观察到ImmCount后的任何CPU→GPU操作（如GDRCopy flag更新）都会被PCIe switch排序在NIC→GPU的数据写之后。这是一个精妙的底层正确性论证。

Detail 2: MoE dispatch的两轮策略隐藏路由延迟

Dispatch先投机发送少量token到private buffer（不需要路由信息），同时交换路由信息。路由到达后再scatter剩余token到shared buffer。Private buffer的最优大小经验值为24-32 tokens（Figure 11）。这个设计将routing exchange的10-30µs延迟完全隐藏。

一句话总结： fabric-lib证明了"可靠但无序"是RDMA硬件的最大公约数，ImmCounter是在此语义上实现高效完成通知的关键原语，host proxy方案在当前硬件条件下不仅可行而且能超越GPU-initiated方案。

Summary

fabric-lib是Perplexity AI开源的可移植RDMA点对点通信库，解决了LLM系统中异构RDMA硬件的vendor lock-in问题。其核心洞察是ConnectX RC和AWS EFA SRD都支持可靠但无序的交付语义，因此可以构建一个不依赖消息顺序的统一抽象。库的关键创新是ImmCounter原语——通过WriteImm操作携带的32位immediate value在接收端递增计数器来实现完成通知，利用PCIe ordering保证正确性。TransferEngine用Rust实现，透明管理多NIC聚合（关键用于EFA的4×100Gbps场景），采用per-GPU worker + per-NIC domain的两级架构。

论文通过三个生产系统验证了设计：(1) KvCache传输利用UVM Watcher实现层级overlap，在Qwen3-235B上TTFT增加不到2%；(2) RL权重更新利用全集群NIC带宽和4阶段流水线，实现1T参数1.3秒传输，比现有框架快100×+；(3) MoE dispatch/combine尽管使用host proxy而非GPU-initiated RDMA，在ConnectX-7上仍超越DeepEP成为新SOTA，并在EFA上首次实现可行延迟。

这项工作的更深层意义在于证明了P2P通信是collectives的必要补充而非替代。在disaggregated inference、MoE和RL等新场景中，集体通信的固定membership、同步初始化和操作排序等约束已成为瓶颈，而基于RDMA的P2P通信通过更灵活的抽象提供了更好的解决方案。

Key Findings

1. 可靠但无序（reliable-but-unordered）是异构RDMA硬件的最大公约数：ConnectX RC可以忽略ordering，EFA SRD天然无序，基于此统一抽象避免了vendor lock-in

2. ImmCounter原语在无序传输下正确高效：利用PCIe write ordering保证和host-proxy架构避免data race，在不假设网络消息顺序的前提下实现完成通知

3. Host proxy方案在当代硬件上不输GPU-initiated RDMA：MoE decode在ConnectX-7上fabric-lib超越DeepEP（IBGDA），证明PCIe往返开销可被bulk transfer和流水线策略补偿

4. P2P权重传输利用全集群带宽：避免collective通信中Rank0瓶颈，1T参数模型1.3秒完成传输（256 GPU→128 GPU），比现有框架快100×+

5. EFA上首次实现可行的MoE decode：通过多NIC聚合和路由延迟隐藏策略，在AWS p5实例上实现比pplx-kernels快3-6×的端到端decode速度

6. 层级KvCache传输被计算完全隐藏：UVM Watcher + CUDA Graph兼容的层级触发，使disaggregated inference的TTFT开销几乎可忽略

Limitations

  • 仅支持ConnectX和EFA两种NIC：虽然论文声称RC兼容NIC容易适配，但缺乏eRDMA、Broadcom、AMD等NIC的实际验证和性能数据
  • MoE prefill性能不如DeepEP：DeepEP通过NVLink pre-accumulation和分批传输在prefill场景显著更优，fabric-lib的decode优化策略不能直接迁移到prefill
  • Host proxy随EP规模线性增长的CPU开销：Table 9显示EFA在EP64时post time达28µs p50，随着EP规模继续增大（如EP128/256），CPU开销可能成为瓶颈
  • MoE kernel的内存开销大：decode优化的kernel在prefill时因缺乏chunking导致buffer memory过大，限制了部分模型的部署可行性
  • 缺乏与NCCL P2P的直接对比：论文主要对比DeepEP和pplx-kernels，未系统比较torch.distributed的Send/Recv在相同场景下的性能
  • ImmCounter的PCIe ordering依赖假设：正确性论证依赖于NIC和PCIe switch的ordering行为，不同厂商的PCIe实现可能存在微妙差异
  • Anonymous authorship但标注Perplexity AI：论文声称production-deployed但无法独立验证具体生产规模和稳定性数据
  • 未讨论安全性和隔离性：RDMA one-sided Write直接操作远程GPU内存，multi-tenant环境下的安全隔离问题未涉及
  • NVLink内转发策略较简单：节点内仅在MoE场景使用NVLink，KvCache和RL场景未利用NVLink优化

Infrastructure Impact

  • Algorithm: 低 — 不改变算法本身，但P2P语义使disaggregated inference和异步RL成为可行的系统选择
  • Kernel: 中 — MoE dispatch/combine kernel设计（split kernel + host proxy + two-round dispatch）提供了新的kernel-network协同范式
  • Framework: 高 — 可集成到vLLM/SGLang/TensorRT-LLM等推理框架，为disaggregated serving和MoE routing提供底层通信原语；RL框架可直接采用P2P权重更新
  • LLM: 高 — 直接支持MoE模型（DeepSeek-V3/R1, Kimi-K2）的高效部署，使trillion参数级模型的RL训练周期显著缩短
  • Agent: 低 — 间接受益于更快的推理延迟
  • Cluster: 极高 — 打破RDMA vendor lock-in是cluster层面的核心贡献，使AWS EFA等云原生硬件首次可用于MoE推理，TransferEngine的多NIC聚合和NUMA-aware设计直接影响集群网络利用率
精读 Deep Analysis

Deep Analysis (cluster)

1. System Scope

fabric-lib定位为通信中间件，介于NIC驱动（libibverbs/libfabric）和上层LLM框架之间。它不是collective通信库的替代品，而是补充——专注于collectives无法高效覆盖的P2P场景。

系统边界明确：

  • 不管调度：由上层global scheduler决定prefiller/decoder分配
  • 不管路由：由MoE router产生expert assignment
  • 不管计算：只负责数据搬运，通过callback/flag与计算协调
  • 管连接：透明建立peer连接（ConnectX用UD交换RC握手，EFA无连接）
  • 管传输：sharding、load-balancing、completion tracking

与同类系统的定位对比：

  • vs NCCL：NCCL面向collectives，fabric-lib面向P2P
  • vs NIXL：功能相似但fabric-lib有成熟的EFA支持和生产验证
  • vs Mooncake Transfer Engine：Mooncake不支持EFA
  • vs DeepEP：DeepEP是specialized MoE kernel，fabric-lib是generic通信库

2. Topology & Physical Architecture

节点内拓扑：

  • Per-GPU DomainGroup worker线程，pin到NUMA node
  • 每个DomainGroup管理1-4个NIC Domain
  • ConnectX-7: 1 NIC/GPU (400 Gbps)
  • AWS EFA p5: 4 NIC/GPU (4×100 Gbps = 400 Gbps)
  • AWS EFA p5en: 2 NIC/GPU (2×200 Gbps = 400 Gbps)
  • NVLink用于节点内MoE token传输

节点间拓扑：

  • 全P2P mesh（每个GPU可直接Write到任何远程GPU）
  • 无需collective world的global membership
  • 限制：所有peer必须使用相同数量的NIC per GPU

关键设计：

  • NUMA-aware：线程pinning + 内存分配在正确NUMA节点
  • 多NIC聚合：对EFA尤其关键，透明sharding writes到多个NIC
  • GPUDirect RDMA：NIC直接访问GPU memory，无CPU中转
  • GDRCopy：低延迟CPU-GPU同步（UVM Watcher的实现基础）

3. Collective Communication

fabric-lib明确不做collectives，但论文系统地分析了collectives的局限性：

1. Fixed membership：NCCL需要预先知道所有参与者，disaggregated inference中prefiller/decoder动态伸缩不可行

2. Synchronized initialization：形成"world"需要全局协调，独立peer连接不可能

3. Operation ordering：所有参与者必须agree on operation sequence

4. Shape uniformity：跨参与者的transfer size必须一致

fabric-lib的P2P操作如何替代collectives：

  • AllGather → 每个训练GPU直接Write到推理GPU（RL weight transfer）
  • AlltoAll → scatter + barrier（MoE dispatch/combine）
  • Broadcast → 不需要，每个sender独立Write

4. Congestion & Traffic Engineering

带宽管理：

  • 多NIC sharding：单个large write可split到多NIC，paged/scatter/barrier操作自动shard
  • WR templating：预填充descriptor公共字段，减少CPU开销
  • WR chaining（ConnectX）：最多4个WR链接，减少doorbell ring次数
  • Relaxed PCIe ordering（ConnectX）：IBV_ACCESS_RELAXED_ORDERING允许NIC-GPU间乱序PCIe事务，降低延迟

拥塞控制：

  • 论文未讨论显式拥塞控制——依赖底层协议（RC的credit-based flow control, SRD的自有机制）
  • MoE场景：通过barrier同步确保不会over-write receive buffer
  • RL场景：one-sided Write，接收端无感知，无back-pressure机制
  • 潜在风险：大规模all-to-all（EP128+）时可能出现incast

流量工程：

  • MoE两轮dispatch策略：先小量投机发送隐藏路由延迟，再按路由精确scatter
  • NVLink vs RDMA分工：节点内用NVLink（带宽更高、延迟更低），节点间用RDMA
  • NVLink写序策略：先signal host再写NVLink peer，避免memory barrier阻塞关键路径

5. Storage & I/O Architecture

内存管理：

  • RDMA memory registration：MrDesc可序列化交换给peer，MrHandle本地使用
  • GPU memory + host memory统一注册
  • KvCache分页：128 tokens/page, 32KB/page，page indices间接引用
  • MoE buffer：dispatch receive buffer上限 N·T·max(R, E/N)，combine复用dispatch buffer
  • RL场景：GPU memory watermark控制in-flight tasks避免OOM

I/O路径：

  • Zero-copy: Write操作直接从source GPU DMA到destination GPU
  • UVM Watcher: CUDA Graph兼容的GPU→CPU通知（GDRCopy polling）
  • GDRCopy: 用于ImmCounter同步、UVM polling、host proxy与GPU的低延迟通信

7. Fault Tolerance & Reliability

KvCache传输的容错：

  • Heartbeat检测prefiller/decoder间传输层故障
  • Decoder发起的cancellation需要prefiller显式确认（防止remote write覆盖已释放的KV pages）
  • Prefiller无响应时decoder超时取消请求
  • Per-request cancellation token：停止未来传输 + 等待所有pending操作完成

传输层可靠性：

  • 底层依赖RC/SRD的可靠交付保证
  • ImmCounter提供应用层完成确认
  • 论文未讨论NIC故障恢复、网络分区、silent data corruption等问题

局限：

  • 无冗余传输或纠错机制
  • 单NIC故障可能导致DomainGroup吞吐下降
  • 缺乏checkpointing或传输断点续传

8. Cost & Efficiency

资源利用效率：

  • 全集群NIC带宽利用（RL场景vs collective的Rank0瓶颈）
  • 多NIC聚合达峰值带宽（EFA 4×100Gbps → 400Gbps）
  • 计算-通信overlap：KvCache层级流水线、RL 4阶段流水线、MoE dual-batch
  • CPU overhead可控：RDMA submit 23µs/call，UVM callback 6.3µs avg

对总拥有成本的影响：

  • 打破vendor lock-in → 可选择更具性价比的云实例（如AWS EFA vs ConnectX集群）
  • RL权重更新100×加速 → 训练GPU空闲等待时间大幅减少
  • Disaggregated inference → prefiller/decoder独立弹性伸缩，提高资源利用率
  • EFA MoE可行 → 首次使P5实例可用于DeepSeek-V3级MoE模型的实时推理

9. Infrastructure Impact

对集群网络基础设施的根本影响：

fabric-lib证明了一个重要的架构论点：LLM系统不应该只有collective通信。P2P primitives是现代LLM workload（disaggregated inference, MoE, RL）的必要基础设施组件。

对云provider的影响：

  • AWS EFA首次在MoE推理中可用 → 扩大EFA的适用场景
  • 降低从on-prem（ConnectX）迁移到云（EFA）的barrier
  • 可能推动云provider改进自有NIC对RDMA标准操作的支持

对框架生态的影响：

  • vLLM/SGLang可集成fabric-lib实现disaggregated serving
  • RL框架（OpenRLHF, veRL, NeMo-RL）可采用P2P权重更新
  • 可能催生更多基于P2P通信的新系统模式

10. Scalability & Future

当前规模：

  • 验证到64 GPU（8节点×8 GPU）MoE
  • RL场景256 training + 128 inference = 384 GPU
  • 未测试更大规模（如1000+ GPU）

扩展性瓶颈：

  • Host proxy CPU开销随EP线性增长（EP64已显现）
  • ConnectX RC每peer需2个QP → QP数量O(N²)，大规模可能成为问题
  • Barrier latency随参与者增加

未来方向（论文提到）：

  • GPU-Initiated RDMA：等p5en等实例成熟后可切换到GDA，进一步降低延迟
  • GB200 NVL72：宽NVLink domain可能使RDMA在MoE场景不再必要
  • 更多NIC支持：eRDMA、Broadcom、AMD等RC兼容NIC

未来方向（个人判断）：

  • 需要解决大规模all-to-all的incast问题
  • 可能需要adaptive sharding策略应对网络拥塞
  • Multi-tenant隔离和安全性将成为云部署的必要特性
  • 与UCCL等传输层优化的结合可能进一步提升性能
