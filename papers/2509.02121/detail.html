Batch Query Processing and Optimization for Agentic Workflows (Halo)

Junyi Shen, Noppanat Wadlom, Yao Lu (National University of Singapore) | 2025-09 | arXiv 2509.02121 Category: framework | Tags: agentic-workflow, batch-serving, query-optimization, dag-scheduling, cpu-gpu-pipelining, kv-cache-sharing Read: 2026-04-18

Core Contribution

Halo 是首个把数据库批查询优化思想系统性引入 agentic LLM workflow 的执行引擎：它把每个 workflow 建模为 DAG，把 batch 内所有 DAG 合并为一张 consolidated graph，并用基于 epoch 的动态规划求解器在 CPU 工具算子和 GPU LLM 算子之间做全局 placement / 排序 / 批处理，从而同时最大化 KV-cache 复用、模型驻留和 CPU–GPU pipeline overlap。

Summary

现有 LLM serving 栈（vLLM / SGLang / TRT-LLM）把每个请求当独立事件优化，而 LangGraph / AutoGen / AgentScope 等 agent framework 只做逻辑编排，没有全局调度视图。对于日益常见的 batch 分析式 agentic workloads——同一个 workflow 模板被复制到数百上千个市场、产品线、时间窗——这意味着 prompt 前缀、SQL 模板、中间摘要大量重复，但仍被当作独立 API 调用，导致 GPU 空转、tool 请求抖动、KV-cache fragmentation。

Halo 的做法是把整批 workflow DAG 编译成一张 structured plan DAG，节点要么是 GPU-LLM prefill/decode，要么是 CPU-bounded tool（SQL、HTTP、函数）。优化器在 epoch 粒度上决定「这轮把哪些 LLM 节点分配到哪个 GPU worker」，目标函数同时考虑 (a) CPU 预处理延迟 $T_{\text{prep}}$、(b) 模型切换惩罚 $T_{\text{model}}$、(c) 带 prefix-cache 折扣的推理延迟 $T_{\text{infer}}$，以及 epoch fragmentation 正则项。求解器用 DP + topological frontier pruning，比 MILP oracle 快 ~2300×，计划质量与 oracle 相同。运行时 Processor 用事件驱动 Coordinator + wavefront 执行 + opportunistic work-stealing + request coalescing 吸收运行时抖动。

在 6 个 workload（IMDb / FineWiki / TPC-H 的 diamond / chain / fanout 拓扑）×（Qwen3-14B/32B、GPT-OSS-20B）× 3×H200 平台上，Halo 相比 vLLM 朴素调用快 >400×（W1 71h→676s），相比 OpWise / LangGraph / AgentScope / Parrot 等 SOTA agentic 基线快 1.03–3.6×（batch）和 1.5–2.6×（online QPS）。消融显示 Request Coalescing 是最大贡献（W6 去掉后慢 2.54×），其次是 opportunistic execution。

Key Findings

  • 批量 agentic workload 的冗余来源于结构而非内容：同一个 workflow 模板会在不同 market/product/timeframe 参数下被复制 1024 次，前缀完全共享、SQL 模板完全共享，但现有 serving 引擎看不见 workflow-level 结构，因此只能做 request 级 prefix cache（碎片化）。
  • epoch 级 DP 可以在 2s 内达到 MILP 最优：把连续时间决策离散化到 epoch，并用 topological frontier 剪枝，状态数从指数级降到近线性，在 W1 上 2.24s vs MILP 1.44h，质量完全一致。
  • 多模型 agentic workflow 的主要 hidden cost 是 weight reload 而非 inference：Halo 的 $T_{\text{model}}$ 项显式把 model switch 成本纳入目标，促使调度器把同模型请求连续执行；这在 HEFT 贪心（W6 上慢 1.41×）中完全缺失。
  • Request Coalescing 的杠杆极大：高 fan-out agents 会产生大量相同 SQL / API call，coalesce 到单一物理执行后 W6 吞吐提升 2.54×——远超 KV-cache 复用带来的收益（8–20%）。
  • GPU-seconds（而不是瞬时利用率）才是正确的成本指标：OpWise 利用率呈「锯齿状」满载/空转切换，累计 28,889 unit·s；Halo 稳态运行累计只有 14,690 unit·s，是 2× 的真实成本差。
  • 在 3-GPU 单机设定下 scalability 接近线性：1→3 H200 worker，Halo 几乎线性加速，OpWise 因同步 barrier 边际收益递减。

Limitations

  • 单机多 GPU：无分布式 placement、cache migration、network-aware 调度。多节点部署会改变 cost model 中的 $T_{\text{prep}}$ 分布和 model weight broadcast 代价。
  • 固定逻辑 DAG：不允许 prompt rewriting、approximate sub-models、model cascading 等更高层优化（与 Palimpzest/LOTUS/DocETL 的逻辑重写路径正交）。
  • 静态 cost model 校准：profiling 虽然在线更新，但重大 workload 切换（模型、QPS、长度分布变化）时可能产生漂移，本文没有给出 drift detection + replanning 的收敛性保证。
  • 与 dynamic workflow / 循环 / 条件分支的集成：论文明确限定「static topology」，对 ReAct 式 planner-driven 的自决策循环未覆盖。

Infrastructure Impact

  • Algorithm: N/A（不涉及训练算法）。
  • Kernel: N/A，复用 vLLM 的 PagedAttention + FlashAttention kernel。
  • Framework: 核心影响——证明 serving engine 之上需要一层「workflow-aware query optimizer」，促使 vLLM / SGLang 向上暴露 prefix cache state 和 resident model state 接口。
  • LLM: N/A（模型无关）。
  • Agent: 重大影响——为 LangGraph / AutoGen / AgentScope 等 orchestration 框架指出下一步「plan-level co-optimization」方向，替代 black-box API 调用。

Deep Analysis (framework)

Phase 2 Inheritance: 时代定位 & 核心技术壁垒

时代定位：2024–2025 年 LLM serving 的单请求优化（PagedAttention、continuous batching、RadixAttention、speculative decoding、DistServe/Splitwise 的 P/D 分离）红利基本采收完毕，剩余瓶颈已经从「怎么让一次 forward pass 跑得快」上移到「怎么让上千个 workflow DAG 跑得少」。Halo 代表优化从 请求级 → workflow 级 → batch 级 的第三次抽象抬升，把 1970s database 的 query optimization 理论重新套用在 LLM workflow 上——这是 2025 年 agentic serving 进入「深水区」的标志之一（另一个标志是 KVFlow、Murakkab、Ayo 等同期工作）。

核心技术壁垒：epoch-based DP + topological frontier pruning。MILP 是 NP-complete 无法在线使用，HEFT 贪心又会陷入局部最优。Halo 的关键壁垒在于把连续时间离散化成 epoch 后还证明了 state space 可被 topological frontier 剪枝到近线性——这一步让 DP 状态数从 $2^{|V|}$ 降到实际可解的量级，同时保留 DP 的全局最优性。这个「可解性 × 最优性」的中间点是多数竞品（Ray / LangGraph / Parrot）没有命中的——他们要么放弃全局（贪心/actor isolation）要么放弃在线（offline planner）。Table 4 的 W1 上 2.24s vs MILP 1.44h 即是这个 barrier 的直接证据。

1. System Scope

  • Primary goal: offline batch inference + online serving for agentic workflows（不是 training）。
  • Scale: 单机多 GPU（3× H200 原型），不覆盖多节点。
  • Online / offline: 两者都支持。Batch 模式做 plan-then-run；online 用 epoch 滑动窗口处理连续流。
  • Workload: 多模型、多 agent、CPU-tool 混合的 analytics workload（IMDb、FineWiki、TPC-H 上的 RAG + SQL + 多步推理）。

2b. Data Movement Hotspots

  1. Model weight reload（GPU DRAM→HBM）：单模型 14–32B，reload 2–20s，每次 context switch 都罚这一次。Halo 用 $T_{\text{model}}$ 惩罚项显式最小化之。
  2. KV-cache state migration（GPU↔GPU）：当 DP 决定把 workflow lineage 迁移到另一个 worker 时要搬 KV。Halo 当前只是在 cost model 里「感知」不共享的代价，实际 migration 代价是 $T_{\text{infer}}$ 中 prefill 的一部分（重新 prefill）。
  3. CPU tool 结果 → GPU prompt 拼接（CPU→GPU，~MB 级）：每次 tool call 后把结果文本拼入下一个 LLM prompt，一次性拷贝，不是 bottleneck，但 tool 阻塞会造成 GPU 空转（C2: heterogeneous pipeline bubble 正是这里产生）。

    3c. 假设审计

    • ✅ 「workflow 拓扑静态」——在 enterprise SOP 场景成立，但在 ReAct / 自规划 agent 不成立。
    • ⚠️ 「prefix reuse 带来可观收益」——依赖 workload 有大量重复模板（TPC-H / IMDb / FineWiki），在长尾多样化的真实 agent 请求（如 customer support）上未必，本文未测。
    • ⚠️ 「epoch 离散化不损失最优性」——Table 4 在 W1/W6 上验证，但没给出理论 gap bound。如果 epoch 长度 vs 单节点耗时比例不合适，会损失并行度。
    • ⚠️ 「profiling 准确」——DBMS EXPLAIN 对 complex join 有 cost model drift；LLM 吞吐曲线对长生成的 decode 方差大。作者用 moving average 缓解但没量化 drift。

    3d. 核心技术壁垒重申

    DP + topological frontier pruning 的可解性证明是整个系统的 keystone。一旦 MILP 时间可接受（小规模 workload），Halo 的优势会缩小到 cost model 精度。反之在大规模（2000+ queries）上，frontier pruning 是唯一能在线使用的工具。

    3e. 设计绑定

    • 绑定 vLLM 风格 continuous batching（或 SGLang）——若切换到 TRT-LLM / batch=1 的 latency-optimized engine，cost model 需重校准。
    • 绑定「LLM 节点是全部 GPU 决策变量」假设——多模态或 tool-use 占 GPU 的场景（例如 vision encoder）需要扩展。
    • 绑定静态 DAG——不支持 ReAct 式动态规划 agent，这是最大的 scope 限制。
    • 绑定单机——跨节点 KV migration / model broadcast 会改变 cost model 中 $T_{\text{prep}}$、$T_{\text{model}}$ 的量纲。

    5. Scheduling & Resource Management

    • Batch formation: 每 epoch 由 DP solver 决定 $B_e \subseteq V \setminus D_e$ 和 $\phi_e: B_e \to W$ 注入映射（每 worker 一个 primary LLM 节点），epoch 内部运行时还有 continuous batching。
    • Memory: 依赖 vLLM 的 PagedAttention；Halo 只显式管 resident weights 和 KV signature，不重写 allocator。
    • Bubble avoidance: (a) cost 里加 $T_{\text{prep}}$ 让 solver 提前启动 CPU preparation；(b) CPU task 按到下一个未满足 LLM 节点的 DAG 深度排序；(c) runtime opportunistic exec 吃掉 I/O gap。
    • Priority / SLO: 当前只优化 makespan（$\mu \cdot \max_w T_w + (1-\mu) \sum_w T_w$），没有 per-request SLO。这是与 Murakkab 的主要区别。
    • Multi-tenancy: 未讨论。

    Metrics

    • E2E latency (batch): 完成 N=1024 queries 的 wall-clock。
    • Throughput (online QPS): 异步 stream 下最大可持续 QPS。

    • Solver time: 调度算法本身耗时。
    • Cumulative GPU usage $\int_0^T U(t)\,dt$（unit·s）：真实 GPU-second 成本代理。

    Fairness

    • 所有系统共用 vLLM 0.11.0 + Postgres 18.0 backend——公平。
    • baselines 因 static deployment 限制到 ≤3 model types，这是 baselines 的劣势（Halo 动态 load）。作者承认这是设计差异而非实验 unfair，但 Halo 动态模型切换的 overhead 已计入 $T_{\text{model}}$，而 baselines 的 pre-launch overhead 没计入对比——严格说这是对 Halo 不利的基准。

    算法 1 解读（部分）

    
Function Solve(D, H, e):
    if D = V: return (0, ∅)                    # 终止：所有节点完成
    if (D, H) ∈ M: return M[(D, H)]            # 记忆化命中
    F = GetFrontier(G, D)                      # 拓扑就绪集
    for B ⊆ F, φ: B → W:                       # 枚举 batch 和 worker 映射
        c_now = CostModel(B, φ, H, T_prep, T_model, T_infer)
        H_next = UpdateWorkerState(H, φ)
        (c_fut, Π_rest) = Solve(D∪B, H_next, e+1)
        if c_now + c_fut < best:
            best = c_now + c_fut
            plan = {(B, φ)} ∪ Π_rest
    M[(D, H)] = (best, plan)
    return M[(D, H)]

    关键点：

    • 状态 $(D, H)$——完成集 + worker 状态；记忆化跨 epoch 复用。
    • 剪枝：GetFrontier 只返回拓扑就绪节点，大大缩小 $B \subseteq \mathcal{F}$ 的枚举空间。
    • worker state $H$ 会按 $\phi$ 确定性更新——residual model + KV signature 的 transition 是确定的，所以 DP 合法。
    • 未剪枝部分：$\phi$ 的单射约束（$|B| \le |W|$）让 enumeration bounded。
    • 复杂度：状态数 $\le |\text{frontier}|^{|W|} \times |V|$，实际 W1 约数千次调用即收敛。
