Maestro: Workload-Aware Cross-Cluster Scheduling for LLM-Based Multi-Agent Systems #

1. TL;DR #

Serving LLM multi-agent workflows under fixed GPU budgets is bottlenecked by unpredictable per-stage output length, long-tail multi-model cold starts, and cross-cluster routing. Maestro predicts each stage's output length / KV footprint / tool-use intent, then drives a 3-tier scheduler (node colocation + cluster fitness routing + global preemptive SRTF), cutting KV-reservation HBM 67.2% and lifting high-contention SLO by 23.6 pp over EDF.

2. Q1 痛点 / Q2 方法 / Q3 结果 #

Q1 — 痛点 #

An LLM-MAS user query is not one request but a dependent workflow of stages (e.g. Planner → Coder → Reviewer), each an LLM call. Three coupled problems make naïve serving fail:

  1. Non-deterministic cost: decode length is unknown a priori, and latency/memory of a stage depends on its (unseen) output. Misestimating output length misestimates runtime and memory, causing head-of-line (HoL) blocking where a long reasoning step delays short interactive ones. The effect is amplified because length depends on agent role, tool type, and CoT vs non-CoT mode.
  2. Long-tail multi-model contention: a few models are hot, most are rare. Dedicating a GPU per model wastes capacity; on-demand loading pays tens-of-seconds cold starts that breach interactive SLOs; colocation relieves waste but intensifies memory contention, and KV cache grows with output length on top of static weights.
  3. Cross-cluster routing trade-off: the topologically nearest cluster minimizes RTT but may lack the resident model or KV headroom (→ cold start of tens of seconds), while a farther cluster may already be warm. Inter-region RTT (tens–hundreds of ms) is orders of magnitude smaller than a cold start (tens of s), so "nearest" is often wrong.

    4. Existing systems address these only partially: prediction-aware single-turn schedulers (FASTSERVE), multi-model multiplexers (MuxServe, QLM) with fixed partitioning, and static cross-cluster routers (SkyServe, AIBrix) oblivious to model readiness/KV affinity. None couple low-overhead stage-level cost estimation with cross-cluster allocation for agentic workflows.

    Q2 — 方法 #

    Maestro exposes five agent-level dimensions (role/workflow position, tool intent, predicted per-stage output length + KV footprint, remaining workflow time, per-cluster model readiness) and turns each stage into a scheduling unit passing through a 5-phase closed loop: observe → predict → schedule → node execute/manage-memory → profile. Three tiers:

    • Node level: memory-safe multi-model colocation via hierarchical weight residency (GPU → host RAM → local disk → remote) with LRU eviction (Algorithm 1), plus explicit memory accounting and elastic virtual-memory KV allocation (kvcached / CUDA VMM). When KV admission fails, a minimum-impact 5-level degradation plan (Algorithm 2) frees memory and reports a disruption cost back to the scheduler.
    • Cluster level: feasibility-aware fitness routing — filter nodes that can safely admit the inflated KV demand $R_{\text{need}}$, then rank feasible nodes by a compact score $S(N,T)$ combining network affinity, time-to-start, and disruption penalty (Algorithm 3).
    • Global level: two-level queueing with workflow-aware Shortest-Remaining-Time-First using predicted remaining workflow time, boundary preemption (only between LLM invocations), hysteresis to avoid oscillation, and aging to prevent starvation.

    核心技术壁垒: the single hardest-to-replicate insight is the tool-intent-first two-stage cost predictor wired as a scheduling signal. A calibrated (isotonic-regression) tool-call probability is used both directly by the scheduler and as an input feature to a per-role log-space output-length regressor, which then feeds every downstream decision (execution-time ordering Eq. 2, KV admission Eq. 3, SRTF remaining-time Eq. 7, and the routing safety margin via EWMA of prediction error). Reproducing Maestro is not about any one equation — it is about making a sub-network-RTT-cheap predictor accurate enough (MAE ↓19% vs Magnus) and calibrated enough that a whole scheduling stack can trust it.

    Q3 — 结果 #

    • Output-length MAE 165.43 tokens ($R^2=0.7774$), −19.2% vs Magnus, −30.9% vs BERT-MLP; linear baseline has negative $R^2$.
    • Five-model colocation on a single 40 GB A100 reaches ~122 GB virtual KV budget = 3.05× effective footprint = 205% overcommit = 67.2% less HBM than non-overcommitted reservation.
    • At high stress (λ=2.0, batch 0.8) SLO attainment 73.6% vs EDF 50.0% → +23.6 pp; interactive queueing delay −84.8% vs EDF.
    • Activation latency vs QLM: 0.11–2.50 s (Maestro) vs 36–52 s (QLM) across 0.6B–14B.

    4. 作者证明 #

    无形式化定理/证明 — Maestro is built on an analytical cost model (Eqs. 1–8) with empirical validation, not proofs. Notation and per-equation physical meaning below, followed by 6 consistency checks.

    Notation #

    SymbolMeaning
    $P(T),\ \hat L(T)$prompt length; predicted output length of stage $T$
    $\hat p_{\text{tool}}(T)$calibrated probability stage $T$ triggers a tool call
    $t_{\text{pre}}(P,M),\ t_{\text{dec}}(M)$profiled prefill latency; per-token decode latency for model $M$
    $\hat T_{\text{exec}}(T)$predicted stage execution time (Eq. 2)
    $\alpha(M)$model-specific KV memory per token
    $\hat R_{\text{kv}}(T),\ R_{\text{need}}(T)$predicted KV demand; margin-inflated demand
    $M_{\text{total}},M_{\text{kv}},M_{\text{res}},M^{k}_{\text{ctx}},M_{\text{other}}$total GPU mem; live KV; reserved non-KV; per-model warm context; other overhead
    $R^{\text{head}}_{\text{kv}}(N)$KV admission headroom on node $N$
    $T_{\text{act}},\text{BW}_{\text{tier}}$weight activation latency; storage-tier bandwidth
    $C_{\text{deg}}(N,T),\ c(e,a),\ c_{\text{int}}$disruption penalty; per-action restore cost; interrupt penalty
    $S(N,T),\ A(N,T),\ \lambda,\mu$fitness score; affinity; ready-time / disruption weights (default 1)
    $T_{\text{ready}},T_q,T_{\text{rem}},T_{\text{future}}$time-to-start; queueing delay; remaining/future workflow time

    Equations & physical meaning #

    • Cost model roots: $T_{\text{TTLT}} = T_{\text{prefill}}(P) + L \cdot t_{\text{decode}}$ — latency is prefill plus linear-in-output decode; large $L$ dominates, so misestimating $L$ misestimates runtime.
    • Eq. 1: $\hat{p}_{\text{tool}}(T) = \Pr[\text{tool}(T)=1 \mid x(T)]$ — a calibrated continuous signal (isotonic regression); set to 0 if no tools available.
    • Eq. 2: $\hat{T}_{\text{exec}}(T) = t_{\text{pre}}(P(T),M(T)) + t_{\text{dec}}(M(T)) \cdot \hat{L}(T)$ — same additive form as TTLT but driven by predicted $\hat L$; node-invariant for a fixed model-engine pair.
    • Eq. 3: $\hat{R}_{\text{kv}}(T) = \alpha(M(T)) \cdot ( P(T) + \hat{L}(T) )$ — KV scales linearly with total sequence length times per-model per-token constant.
    • Memory-safety invariant: $M_{\text{res}} = \sum_{k \in S} M^{k}_{\text{ctx}} + M_{\text{other}}$ and $M_{\text{kv}} + M_{\text{res}} \leq M_{\text{total}}$; headroom $R^{\text{head}}_{\text{kv}}(N) = M_{\text{total}} - M_{\text{res}} - M_{\text{kv}}$.
    • Activation model: $T_{\text{act}} \approx \text{Size} / \text{BW}_{\text{tier}}$ — weight bytes over tier bandwidth, unifying cross-tier switch costs.
    • Eq. 4: $C_{\text{deg}}(N,T) = \sum_{(e,a)\in P} c(e,a) + \mathbb{1}[I_{\text{active}}] \cdot c_{\text{int}}$ — aggregate restore latency of the degradation plan, plus a fixed penalty when an active execution is interrupted.
    • Eq. 5: $S(N,T) = A(N,T) - \lambda\, T_{\text{ready}}(N,T) - \mu\, C_{\text{deg}}(N,T)$ — reward affinity, penalize wait-time and disruption; pick $\arg\max_N S$.
    • Eq. 6: $T_{\text{ready}}(N,T) = T_{q}(N,T) + T_{\text{act}}(N,M(T))$ — time-to-start = queue wait + weight activation.
    • Eq. 7: $\hat{T}_{\text{rem}}(J,k) = \hat{T}_{\text{exec}}(T_k) + \hat{T}_{\text{future}}(J,k)$; Eq. 8: $\hat{T}_{\text{future}}(J,k) \approx T^{(0.50)}_{\text{next}}(\text{state}(J,k))$ — remaining workflow time = current stage + conditional-median of future stages, driving SRTF.
    • Safety margin: $R_{\text{need}}(T) = (1 + \rho) \cdot \hat{R}_{\text{kv}}(T)$, with $\rho$ set as a high quantile (90–95%) of EWMA relative underestimation, typically $\rho \in [0.1, 0.3]$.

    6 consistency checks #

    1. Dimensional (Eq. 2): $[t_{\text{pre}}] = s$, $[t_{\text{dec}} \cdot \hat L] = (s/\text{tok})\cdot\text{tok} = s$ — consistent time. ✓
    2. Eq. 3 vs §II: linear-in-$(P+\hat L)$ KV matches the §II "KV scales with $P+L$" statement; $\alpha(M)$ absorbs layers/heads/precision. ✓
    3. Safety invariant: $M_{\text{kv}} + M_{\text{res}} \le M_{\text{total}}$ with $R^{\text{head}}_{\text{kv}} = M_{\text{total}} - M_{\text{res}} - M_{\text{kv}} \ge 0$ is exactly the admission condition (headroom non-negative ⇔ feasible). ✓
    4. Routing argmin reduction: $\hat T_{\text{e2e}} \approx \text{RTT} + T_{\text{ready}} + \hat T_{\text{exec}} + \eta C_{\text{deg}}$; since $\hat T_{\text{exec}}$ is node-invariant for a fixed model-engine pair, dropping it yields the compact $S(N,T)$ in Eq. 5 — argmax over $S$ = argmin over $\hat T_{\text{e2e}}$. ✓
    5. Eq. 5 monotonicity: $\partial S/\partial A > 0$, $\partial S/\partial T_{\text{ready}} = -\lambda < 0$, $\partial S/\partial C_{\text{deg}} = -\mu < 0$ — the optimum is the max-affinity, min-wait, min-disruption node (boundary of the feasible set), matching Algorithm 3's greedy scan. ✓
    6. First-order mapping (Table V): summing the Virtual KV column $34.24+32.15+27.81+20.07+7.74 = 122.01$ GB on a 40 GB GPU gives $122.01/40 = 3.05\times$, overcommit $(122.01-40)/40 = 205\%$, and $1 - 40/122.01 = 67.2\%$ HBM saved — the headline number is recovered directly from the accounting, not a fitted sweep. ✓

      7. 实现 cross-reference #

      [实现未公开] — no code repository, commit, or artifact link is provided in the paper; the prototype is described only prose-level as an extension of vLLM v0.11.0's PagedAttention with an elastic paging mechanism, integrating kvcached (CUDA VMM) for the KV pool. Concrete anchors that a reimplementer would target:

      • Predictor: BERT/MiniLM sliding-window encoder → PCA → mean-pool for $x_{\text{sem}}$; isotonic-regression-calibrated tool-intent classifier; per-role LightGBM/GBDT regressor on $\log(1+L)$ with inverse transform (§III.B, Fig 6). Training: stratified temporal holdout, latest 20% test / 15% val for early stopping (§IV.A).
      • Node runtime: 5 readiness states (Running / Sleeping / CPU-resident / Disk-resident / Remote); Algorithm 1 cascading load-and-evict with per-tier LRU; Algorithm 2 greedy 5-level degradation sorted Idle→Sleeping→Active (§III.C).
      • Scheduler: Algorithm 3 filter-then-argmax on Eq. 5 with robust 5/95-percentile min–max normalization, default λ=µ=1; boundary-only preemption with hysteresis + per-job cooldown + EWMA-smoothed $T_q$ + aging (§III.D).

      核心技术壁垒 (§7 detail): the barrier is not any equation but the coupling — the calibrated tool probability must be good enough to serve simultaneously as (a) a direct scheduler signal, (b) a feature into the length regressor, and (c) an input to the EWMA-driven safety margin $\rho$. If the predictor is uncalibrated (high log-loss), the whole stack mis-admits KV and mis-orders SRTF; the paper's decisive choice of Maestro-Pred over the marginally-higher-accuracy MLP (Table III) is precisely because MLP's log-loss/calibration is worse. Replicating Maestro therefore means replicating a calibrated, sub-11ms, role-aware predictor and threading its outputs through four separate decision points — a systems-integration barrier more than an ML one.

      关键实现细节 (easy-to-miss):

      1. Long-lived prefix caching is disabled in the VMM elastic-KV mode (unless explicitly required) to keep pages reclaimable under multi-model colocation — a non-obvious correctness constraint that trades a common vLLM optimization for reclaimability.
      2. Preemption is boundary-only (between LLM invocations): the common path merely updates scheduler metadata and preserves KV allocations; only when memory pressure prevents preservation is the resume cost charged through $C_{\text{deg}}$ (Eq. 4). Missing this leads to over-counting preemption overhead or, worse, mid-decode interruption.
