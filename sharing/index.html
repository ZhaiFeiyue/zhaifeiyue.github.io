0 · 开场：Agent 不是"更长的聊天"

一句话定义：Agent = LLM + 循环(loop) + 工具(tools) + 记忆(memory)。它把 LLM 从"回答一个问题"变成"为达成目标，自主地反复思考、调用工具、读写记忆"。

核心转变无状态单轮问答 → 有状态多轮自主工具调用。 这一步转变，同时冲击了两件事：模型侧"怎么想/怎么做"（算法），系统侧"怎么把这种长尾、有状态、突发的负载跑得起、跑得快"（infra）。全场围绕这两条线展开。

为什么算法要变

单轮 prompt→answer 不再够用：需要规划、反思、工具编排、记忆管理、以及"该用多大的模型"。

为什么 infra 要变

长 session、持续增长的 KV cache、突发工具调用、不可迁移的 decode 状态——传统"无状态请求分发"的 serving 假设集体失效。

第二部分 · Agent 算法：5 个核心问题与解决方向

算法部分不铺技术清单，只抓 4 个真正驱动 agent 演化的问题，每个讲清"问题是什么 / 解决方向 / 到哪一步了"。核心是问题与方向。

① 长上下文用不好（lost-in-middle）→ 上下文管理 → Subagent

这是最主线的一条因果链。Agent 是多轮循环、上下文越滚越长，但学术界反复证明 "塞得下" ≠ "用得好"：

结论论文
信息放中间显著变差（U 型），多跳更明显Lost in the Middle & In-Between 2412.10079
光是上下文变长本身就拉低表现（即使能检索到）Context Length Alone Hurts 2510.05381
长上下文智能退化有临界阈值（RoPE 衰减+注意力弥散）Intelligence Degradation 2601.15300 · NoLiMa 2502.05167

解决方向：不硬塞，而是"选 / 压 / 换"上下文——学术打地基（数据合成 2404.16811、对比解码 2506.08371），工业做工程（Cursor DCD 上下文虚拟化、Claude Code 5 层 compaction 2604.14228、RLM 外部存储 2512.24601、PEEK 定向地图 2605.19932、CMV 结构无损裁剪 2602.22402）。

但压缩自带矛盾：压缩 vs prefix-cache我们实测（matplotlib-24149）：不压缩 prefix 命中爬到 96.95%；一开压缩就变锯齿——每次压缩当轮塌到 ~30%（改了历史前缀 → 其后缓存全失效）。业界解法是 Anthropic 在缓存查找之后于 API 层删（删而不失效）；我们在 sglang 上靠"晚压少压 + summary 钉死 offset + 只删尾不改头"，离线 61%→93.5%。2604.14228
方向的终点 = SubagentSubagent = 把子任务当一次工具调用：独立上下文、只回一条摘要。它同时做到 ①对抗 lost-in-middle（噪声不进主上下文）②任务级压缩（子轨迹坍缩成摘要，比 token 级更彻底）③prefix-safe（不改父前缀，绕开压缩伤缓存的矛盾）④可并行。Kimi Agent Swarm 再把它推成"模型原生的多代理并行"——用"多个隔离上下文并行"替代"一个巨型上下文"。（我们的实现见第四部分；半年 CC/Cursor 迭代主线：溢出→压缩→压缩伤缓存→subagent→记忆/导航）

② 降成本 → mixed-model 执行 → 跨模型 KV 共享（核心难题）

Cursor 引入了 mixed-model 执行：同一个 chat 里按问题难度切换 mode / model——简单步用小/快模型，难步才上强模型，从而大幅降成本；Claude Code 也有类似的多模型/多档位执行。

它引入的核心算法问题不同 model 权重不同 → 同样的 token 序列产生的 KV cache 不一样 → 在"模型切换点"，前面用 model A 建好的 KV 无法被 model B 复用，prefix cache 在切换处失效、要重新 prefill 整段历史。省了 token 单价，却可能赔上 KV 重算——mixed-model 的收益能不能落袋，取决于能否跨模型共享/复用 KV。

KB 调查到的解决方向（跨模型 KV 共享）：

方向做法出处
冻结共享 base，让 KV 天生相同PrefillShare：冻结 base prefill 模块 + cache-conditioned 微调 → N 个模型共享同一份 KV（O(N)→O(1)），disaggregated，吞吐 3.9×2602.12029

③ 隐藏等待（工具 stall）→ 投机执行 / 执行重叠

Agent 大量时间卡在等工具返回（CPU 侧可占端到端 88% 延迟）。核心是一轮里 GPU 和 tool 天然串行、互相干等。先看串行基线的执行序列——decode 生成完 tool_call 后，GPU 在整段 tool 执行期间完全空闲：

prefill   decode   tool exec   GPU 空闲/浪费   保温前缀

算法层 idea：投机执行PASTE：模式感知投机执行下一个工具 2603.18897；Speculative Tool Calls：小模型投机预测 tool call 并异步执行 2512.15834。原理同投机解码——"赌大概率的未来，赌对白赚、赌错丢弃"，从 token 级扩展到 tool 级。

工程落地：mori-scheduler 最近 PR(#157) 的三类执行重叠

投机执行是"赌未来"；mori-scheduler 走的是一条更稳、可叠加的路——不预测，直接把已知的 tool 延迟藏进 GPU 时间线。三类机制独立且可叠加，且大多 piggyback 现有引擎、无需改 sglang：

① fan-out：并发只读工具（等 max 不等 sum）

只读工具走保守白名单（safe-by-omission）；写类工具（bash/edit）仍串行保序，pod 状态与消息顺序逐字节不变——纯延迟优化。

② streaming dispatch：边 decode 边发工具

SSE aggregator 在某 tool_call 的累积 arguments 首次解析成合法 JSON 时触发 on_tool_ready，立刻启动只读 tool，future 存入 _INFLIGHT_TOOLS[session_id]；tools_node 再按原始顺序消费——与 fan-out 组合、结果保序，registry miss 就重跑（安全兜底）。

②的前提：toolcall streaming —— 引擎增量吐 arguments

overlap② 能"边 decode 边发工具"，前提是引擎按 OpenAI 流式协议把 tool_call 的 arguments 分片增量吐出（而不是等整轮结束一次性给）。客户端按 index 逐片累积，一旦 name 齐 + 累积 arguments 解析成合法 JSON 就立刻把快照交给 on_tool_ready 提前 dispatch（每个 tool_call 最多触发一次）：

sequenceDiagram participant E as 引擎 SGLang tool-call-parser participant C as 客户端 SSE aggregator participant T as Tool 只读 E->>C: delta·name = web_search E->>C: delta·arguments 片段 a E->>C: delta·arguments 片段 b（累积拼接中） E->>C: delta·arguments 片段 c Note over C: name 齐 + 累积 arguments 解析成合法 JSON ✓ C->>T: on_tool_ready 立刻 dispatch（decode 仍在继续） E->>C: 继续 stream 本轮其余 content 与其它 tool_call

即 arguments 分片 {"que → ry":"Ada → ..."} 逐片 += 累积，直到能 json.loads 成功才判定"这个 tool_call 齐了"。

serving 侧要求 & 坑需引擎支持流式结构化 tool_calls（SGLang --tool-call-parser，产出 delta.tool_calls[].function.arguments 分片）。反例：有些模型把 tool call 当纯文本吐——<tools>{...}</tools> 或 Kimi 原生 token <|tool_call_begin|>…<|tool_call_argument_begin|>{…}<|tool_call_end|>——它们走 content delta 而非结构化 tool_calls，得先 normalize，且"边流边判 arguments 是否已成完整 JSON"更难，提前 dispatch 的可靠性下降。好处是这条流式路径复用同一套 profiling（TTFT/TPOT）与 UI token 流（on_delta）——streaming 本就是投机执行/重叠的公共底座。

③ prefix keep-warm：保温与 toolcall 并行 overlap（2A prime / 2B KV-pin）

保温本身就是一次 GPU 活动，和 CPU 侧的 tool 执行天然并行：tools_node 在启动工具的同时并发触发保温（fire-and-forget），二者共享同一段 stall 窗口 —— 等于把 overlap② 的"藏工具"再叠一层"藏下一轮的冷 prefill"。
2A prime：stall 期间对当前窗口发一个 max_tokens=1 请求（带同一 x-mori-session-id 亲和到持有 KV 的 worker），best-effort 保住 RadixAttention 前缀，一个 done-callback 消费掉结果。2B KV-pin：用上游 session_id 字段 + --enable-session-radix-cache（配 priority 逐出策略）把前缀 KV 钉住不被逐出——2A 的鲁棒版，引擎无此特性则忽略该字段、安全独立合并。

一句话 & 对 MORI 意义fan-out 藏"多工具的 sum"、streaming 藏"decode 尾部"、keep-warm 藏"下一轮的冷 prefill"——三段被浪费的 GPU 时间线各有一招。关键设计：全部对只读工具生效、写类工具串行保序、结果按原序消费（纯延迟优化、语义不变）。这正是 MORI「同时握 serving + tool 两端控制权」才能做的 cross-layer 优化——通用 serving（把 LLM 当 API）做不到。mori-scheduler PR #157 · overlap_bench

④ 该用多大模型 / 怎么编排

模型级路由

不是每步都用最强模型（与 ② 互补：② 管"切换后 KV 怎么办"，这里管"何时切"）：RouteLLM 2406.18665 / OI-MAS 2601.04861 / Chimera 2603.22206；反直觉——最强模型可能是最差 planner AgentOpt 2604.06296。

Loop vs DAG / Harness

SGH 用 |U| 连续谱统一 70+ 项目 2604.11378；但生产验证"loop + 强 harness 已够用"——Claude Code "1.6% 决策 + 98.4% 基础设施" 2604.14228。护城河在 harness，不在 loop 结构。

⑤ 记忆之上：从 memory 到 knowledge

前四问都在"单个 agent 把一次任务做好"。但 agent 反复干活会沉淀经验，这里冒出一条常被混谈、却是不同问题的分界：memory（记忆）≠ knowledge（知识）。讲清这条边界，才知道每一侧该用什么机制去做。

归属私有，按 session / agent共享、可复用

脑科学类比海马体（快速编码情景）新皮层（巩固成语义 / 泛化）

一句话：memory→knowledge，就是海马体的情景记忆经"巩固"变成新皮层的语义知识——人类记忆本就是 constructive reconstruction（重建）而非逐字回放，这为"把经验抽象成知识"提供了理论合法性。AI Meets Brain 2512.23343

knowledge 解决什么问题

问题单个 agent 的 memory 是私有、情景、易过时、写入即固化、被动检索的。直接后果：①同样的坑反复踩（经验不复用）②无法跨任务 / 跨 agent 共享③轨迹越堆越长、噪声累积（正是 §① 的 lost-in-the-middle 根源）。knowledge 要解决的，就是把散落、重复、带噪的经验/来源，巩固成一份可泛化、可复用、可回查、去重、交叉链接的结构，让"想过一次的问题"沉淀下来、供后续推理直接调用。

怎么解决（三条机制，均有论文支撑）

① 沿"压缩轴"逐层抽象

Experience Compression Spectrum：memory / skill / rule 是同一条压缩轴上的层级，知识=把记忆沿轴往上蒸馏。它点出核心缺口 missing diagonal——几乎没有系统支持"自适应跨层级压缩"，且 memory 与 skill 社区互引 <1%。→ 解法就该是一条显式的分层蒸馏管线。2604.15877

② 巩固是"重建"，不是逐字存

AI Meets Brain：人脑走 extraction→updating→retrieval→utilization 闭环，海马体（情景）巩固进新皮层（语义），且是 constructive reconstruction 而非逐字回放。→ 知识必须经过提炼/更新/交叉链接，而不是把轨迹原样堆起来。2512.23343

③ 主动维护"定向知识"

PEEK：对反复访问的外部上下文，缓存可复用的 orientation knowledge（含什么 / 怎么组织 / 关键实体 / schema），常量大小、常驻。它由 cache policy 从轨迹自动生长——→ 知识应被主动构建与维护，而非被动检索。2605.19932

为什么这个结构天然契合 agent金字塔给的是"多粒度 + 渐进式展开（progressive disclosure）"：agent 先读常量大小的高层概览定位（省 context、抗 lost-in-middle），只对相关分支按需下钻到叶子取细节与出处——这正是 Ψ-RAG 的 agentic 检索器在做的事，也与 PEEK 的常驻 context map、上下文工程里的 progressive disclosure 同构。它同时满足 agent 的两种需求：叶子=可核查的事实细节，高层=可推理的主题结论。
诚实的边界论文证明的是"分层递归抽象"这一原理有效（多粒度检索 + 多跳推理更强），而非"恰好 N 层最优"——层数是工程选择（RAPTOR 树深由数据决定、GraphRAG 层级来自 Leiden）。另一处差异：RAPTOR/GraphRAG 的层级是聚类自动生长，而语义化的固定管线（原始→蒸馏→关联→综合）是按语义分层；两者共享的是同一个被验证的内核——"raw→逐层摘要的多级抽象"。
所以：一套"分层蒸馏 + 跨源关联 + 治理精修"的知识库设计是合理的把上面三条落地，自然推出一个知识系统应有的形态，也正是本次分享底层知识库的设计取向：①分层蒸馏（原始→蒸馏→关联→综合，对应压缩轴）②跨源去重 + 交叉链接（把孤立条目连成可推理的网）③出处可回查（每条断言带来源，知识才可信、可核）④时效巩固（陈旧内容进复审队列，对应"updating"闭环）⑤反向精修（上层发现不足→回炉重蒸，越往下越具体）。这几件正是"知识"区别于"一堆记忆条目"的必要条件——缺了任何一条，攒下的只是 memory，不是 knowledge。
对 serving / MORI 的启示memory 与 knowledge 对底层诉求不同：memory（私有、情景、append-heavy）→ 长上下文 + 检索 + KV offload；trend 是从被动检索走向主动可学习 AutoMem 2607.01224 · Mem-π 2605.21463 · MemSkill 2602.02474。knowledge（如 PEEK 的 context map）→ 常驻 system prompt、稳定前缀，正好是 §B1.5 说的"最该保护的 prefix 开头（保护开头比优化结尾更值钱）"。

B1.5 · 从无状态到有状态：agent client 主动操作服务端 KV

核心命题过去 LLM server 是无状态的"请求进→回复出"；但 Claude Code / Kimi Code / Cursor 这些 agent client 每一轮都在主动改写上下文，而这些改写直接决定服务端 KV cache 的复用 / 失效 / 搬迁。换句话说，client 的 slash 命令本质是一组"远程 KV cache 操作指令"——server 因此被迫从无状态转向有状态：必须记住每个 session 的 KV 落点、前缀指纹、并区分"读缓存"和"写缓存"。调查来自 mori-scheduler：Claude Code 反编译 + Kimi Code 源码 + Cursor 文档

轴 A · 身份轴（会话认识吗）

按 session_id 记住"上一轮落在哪个 worker（+dp rank）"，带 TTL 覆盖 agent 思考/工具间隙。回头客 → 路由回原 worker（那里握着温 KV），prefill 只传 delta；生客/过期/被全清换了 id → 冷路由。

轴 B · 内容轴（前缀缓存里有吗）

把 prompt 切块算指纹、查全局 KV 索引：命中足够长前缀 → 内容温（只重算尾部）；查不到 → 内容冷（整段重算）。
→ server 要维护跨实例的前缀指纹索引 + 块生命周期（保留/迁移/淘汰）。

最隐蔽的非对称 & 量化收益"身份温、内容冷"（换头/压缩）是最大的坑：会话亲和让你以为省了，其实前缀已废、这一轮照样贵——所以压缩的频率与切点直接决定命中率的周期性塌陷。反过来，把有状态调度做对，收益很实：mori-scheduler 实测 x-mori-session-id 会话亲和让 TTFT −76%。无状态 RR/P2C 在 agent 负载下会把温 KV 白白浪费掉。
对 MORI 的立身点agent 时代的 serving 是一条贯穿两端的链：client 的 harness 操作（A/B/C/D + 旁问/分叉）↔ server 的 KV 调度（两轴判新旧 + 读写解耦）必须协同。几条直接推论：① 保护前缀开头比优化结尾值钱（system/memory/tool 定义要稳）；② 压缩要晚、要少、多留尾；③ 把"读缓存/写缓存"解耦（skipCacheWrite 旁问）上升为 server 一等能力；④ session id 要稳定且在 TTL 内复用。这正是通用 serving 给不了、而 MORI 同时握着 serving + tool 两端控制权能做的差异化。

B6 · Low Latency Generation（TileRT / megakernel）

出发点（可直接拿来用）：Speed as the next scaling law。 AI 推理需求从"质量→吞吐→速度"演进，Agent / TTS / voice 都推动 near-BS=1、延迟优先的负载；更关键的是 Test-Time Scaling 让速度直接等于能力——固定延迟预算下，推理越快 = 越多 rollout / 越深推理链。所以速度本身成了一个 scaling 维度。TileRT / Speed-as-scaling-law blog

问题：小 BS decode 是 memory-bound，但执行边界把带宽浪费掉了

数量级差距8×H200 NVL 聚合带宽 ~38 TB/s、GLM-5.1 每 token 激活 ~42GB → 理论 ~1000 tok/s；实际传统框架只有几十 tok/s。根因不是算力，是 kernel 间的执行边界（launch/setup/teardown、cross-kernel barrier、global-mem 往返、TP sync）——BS≈1 时这些固定开销进入关键路径，形成 "memory pipeline bubbles"（Hazy Research 实测传统框架只用了 ~50% 带宽）。TileRT · Mirage MPK · Hazy "Look Ma No Bubbles"

为什么对 MoE 尤其关键：MoE 收益是"碗型"的

关键事实（多篇 2026 工作一致）：MoE decode 延迟 ∝ 一个 batch 激活的 experts 的并集大小，而不是 token 数——因为 decode 是 memory-bound，时间花在把 expert 权重从 HBM 搬进来；activation 流量 < 权重的 0.6%。METRO 2512.09277 · OEA 2511.02237 · ELDR 2607.00466

解法：两条腿（消边界 + 缩小 experts 并集）

系统侧：消除执行边界（persistent megakernel）

把小 BS 的理论带宽优势兑现出来。
· TileRT：AOT 把整模型展开成一个常驻 Engine Kernel + tile 级调度 + warp/GPU 三级特化，8×B200 上 DeepSeek-V3.2 600 tok/s tile-ai-tilert
· Fleet：chiplet-aware megakernel（Chiplet-task + M-major L2 复用），MI350 bs=1–16 比 vLLM 1.3–1.5× 2604.15379
· Mirage MPK：自动编译成 megakernel，1.2–6.7× 2512.22219

算法侧：缩小激活 experts 并集

从源头降低 memory-bound 延迟。
· METRO：均衡"激活 experts 数"而非 token 数，decode 延迟 −11–22% 2512.09277
· OEA：batch-aware 路由，让 token 搭"已加载 experts"的车 2511.02237
· ELDR：expert-locality 路由，同域请求并集少 17–21% 2607.00466

⚠ 两条腿缺一不可Fleet 明确指出：MoE 的 expert-route-per-token 会破坏 megakernel 的 M-major 权重协作前提（dense 才好协作）。所以 MoE 低延迟不能只靠 megakernel 消边界，还必须靠路由把 experts 并集压小——系统 + 算法两条腿一起，才拿得到小 BS 的延迟甜点。
对 MORI / infra 的启示低延迟 MoE serving = 小 BS memory-bound + persistent kernel（消边界）+ expert-locality 路由（缩并集）。MORI 作为 prefix/KV 调度器，天然可以再加一层 "expert-signature / locality-aware 路由"：把 expert 签名相近的请求聚到同一 worker，缩小每步激活的 experts 并集——与 ELDR 同向，且能和现有的 prefix 亲和路由叠加。

真正"推动指针"的三件事（按贡献排序）

  1. str_replace + 语法门（17%→30%，两次跑都 9/30，唯一可复现的稳定收益）：精确+唯一匹配、Python 语法检查、出错不写入。根治了行级 file_patch 造成的 NameError/SyntaxError 代码损坏。用 python_exec 片段实现 → 无需重建 sandbox 镜像。（借鉴 OpenHands ACI editor："编辑器质量 > 模型"）
  2. retry-on-empty + reproduce-first + max-turns 25→60（最大杠杆，30%→53%/63%）：头号失败模式是 empty-patch——"探索了半天但从没改代码"（约占一半，sympy 7/7）。针对它把空补丁从 ~15 降到 4。
  3. 评分器 per-repo 修复：in-sandbox grader 把 test_patch heredoc 里的 pytest 行误当命令、django FAIL_TO_PASS 是文档串导致 shell 崩——修好后回收了多例"假失败"（Lite 3 例、Verified 7 例），真实分数比原始更高。
诚实的边界N=30 单次跑有 ±3–5 的方差（我们见过 9/14/16/19）；str_replace 以下的增量特性在噪声地板附近，需要 N≥100 才能分辨信号。—— 这也是为什么我们更信 Verified-100 的 59%。

4.3 Subagent：agent-as-a-tool

子代理 = 把"另一个 agent loop"当一次工具调用：全新隔离的消息历史、跑到结束、只把最终摘要返回给父代理。子代理嘈杂的探索（读文件/grep/日志）留在子窗口里被丢弃 → 上下文经济 + 可并行。

两条正交的轴（核心心智模型）

上下文（默认独立、一次性、返回摘要；fork 是显式 opt-in 才继承父上下文）与 sandbox（借用父 pod / 独占自己的 pod）是相互独立的两个维度。一个"改完代码交回父代理"的子代理，上下文仍完全独立——它只是借用父的 sandbox 干活，再摘要回来（等价于一次 bash 调用）。

模式sandbox写仓库?典型用途
borrow-parent(write)共享父 pod，串行是（持久）实现者交接、P1 continue-fixing
borrow-parent(read)共享父 pod否（/tmp）轻量探索
own pod独立 pod是（自己的副本）best-of-N、P1 verifier

安全不变量：一个 pod 同时只能有一个写者（borrow 是串行的、父被 await 阻塞；并行写必须各用独立 pod）；MAX_SUBAGENT_DEPTH=1（子代理不能再生子代理）；一次性、不 resume。

P1 · 验证器重试（oracle-free）

模型只写复现，harness 判定：复现必须在未修复代码上先 FAIL、打补丁后 PASS 才算数；单调回退（新补丁只有让复现转 PASS 才保留，否则 revert）→ 免疫 fix-churn 和假 FAIL。绝不读 FAIL_TO_PASS。

P2 · best-of-N（oracle-free 选择）

N 个独立 pod 并行解，选择器用"复现是否 pre-fix FAIL + 更小 diff + lint 干净"等无 oracle 信号选赢家，只对赢家评一次分。

巧思串行子代理本身就是一个任务级上下文压缩原语：把子任务几十轮探索坍缩成一条摘要——"委派子任务"和"压缩该子任务轨迹"是同一个操作（与 §A2/token 级压缩正交）。

4.4 Subagent 对 Infra（MORI serving）的要求

子代理这个算法特性，直接给底层 serving 提了新需求——正好把本场"算法 ↔ infra"两条线接起来：

需求为什么状态
fork 的亲和路由fork 继承父上下文 → 必须路由到持有父历史 KV 的 worker（prefill+decode 都要，因 decode 亲和把历史 KV 累积在 decode worker）。用 x-mori-affinity-session-id=父 id 做路由键、x-mori-session-id=子 id 做 KV 归属键已实现（mori-sched 优先用亲和键）
独立子代理 = 无亲和默认独立上下文，KV 前缀不共享 → free/least-loaded 放置即可已有
EvictSession（按 session 急切驱逐 KV）一次性子代理返回即"死"，其 KV 前缀不该污染 radix cache⛔ 卡在 sglang（只有 LRU / 全局 flush，无按 session evict）
ClonePod（CoW 秒级克隆）best-of-N 需要从父状态种子化 N 个 podtar-relay 层已实现；containerd CoW 为北极星
子代理 pod 租约 < 父孤儿一次性 pod 应比父更早回收已实现（子 ~60s vs 父 ~90s）
对 MORI 的意义subagent 把"上下文独立性"和"亲和路由/急切驱逐"变成了 serving 的一等需求。fork 的双侧（prefill+decode）亲和、按 session 的 KV 驱逐、pod 差异化租约——这些正是 MORI 相对通用 serving 的差异化价值点。

第五部分 · 社区 vs 学术界的倾向性（例子③）

学术界

追求形式化保证（competitive ratio / Lipschitz 界）、命名新范式、benchmark 驱动、"预测 vs 观测"之争。

工业/社区

harness 工程、prompt caching 经济学、可复现、开源底座（vLLM/SGLang）、"够用就好"。

共识：request 级调度对 agent 不够，要 workflow/conversation 级。分歧：要不要预测不可观测量？要不要静态 DAG？—— 尚未收敛。
