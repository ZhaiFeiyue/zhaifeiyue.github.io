0 · 开场：Agent 不是"更长的聊天"

一句话定义：Agent = LLM + 循环(loop) + 工具(tools) + 记忆(memory)。它把 LLM 从"回答一个问题"变成"为达成目标，自主地反复思考、调用工具、读写记忆"。

核心转变无状态单轮问答 → 有状态多轮自主工具调用。 这一步转变，同时冲击了两件事：模型侧"怎么想/怎么做"（算法），系统侧"怎么把这种长尾、有状态、突发的负载跑得起、跑得快"（infra）。全场围绕这两条线展开。

为什么算法要变

单轮 prompt→answer 不再够用：需要规划、反思、工具编排、记忆管理、以及"该用多大的模型"。

为什么 infra 要变

长 session、持续增长的 KV cache、突发工具调用、不可迁移的 decode 状态——传统"无状态请求分发"的 serving 假设集体失效。

第二部分 · 支柱一：Agent 算法深挖

A1 · Agent Loop / ReAct：最小可用范式

最基础的 agent 就是一个循环：观察 → 推理 → 行动 → 观察…（ReAct）。模型每轮决定"继续想 / 调工具 / 结束"。

原理把"一次性生成答案"拆成"边想边做、用工具补足能力"。天然缺陷：隐式依赖、错误无界传播、难审计——这也是后面所有工程/理论工作的起点。

A2 · Harness 要解决什么（例子①）

agent loop 本身几行代码就能写出来；真正难的是围绕它的一整圈"基础设施"。

金句逆向分析 Claude Code 得到的比例：约 1.6% 是"决策逻辑"，98.4% 是确定性的基础设施（上下文管理、工具执行、权限、错误恢复、缓存）。2604.14228

Harness 的职责清单

① 上下文拼装与压缩 · ② 工具调用与结果回填 · ③ 权限/安全边界 · ④ 错误恢复与重试 · ⑤ prompt/KV 缓存复用 · ⑥ 可观测与审计。护城河在这里，不在 loop 结构本身。

最新工作SmoothAgent（lookahead context engineering）：把 summarization/offload/子代理隔离等"上下文变换"识别为可分段、与未来 token 无关，于是能异步提前执行、预备好变换后的 KV，直接替换而不阻塞主链路——TTFT 最高降 11.9×（基于 SGLang）。2607.00151 这类"上下文变换提前算"正把 harness 的上下文管理和 serving 调度打通。

A3 · Loop vs 结构化 DAG

把 agent 执行抽象成调度器五元组，用"就绪单元数 |U|"衡量：ReAct loop 是 |U|=1，结构化 DAG 是 |U|≥1（可并行）。

例子SGH 用 |U| 连续谱统一了 70+ 开源 agent 项目 2604.11378。但生产实践（Claude Code）证明：loop + 强 harness 已足够好用——DAG 的价值主要在有界恢复和可审计，不是必需。

A4 · 规划 / 反思 / 投机执行

  • 并行工具调用：一轮发多个 tool call，隐藏等待。
  • 投机式工具执行：在模型"思考"时提前跑下一个可能的工具，把 tool stall 藏进思考时间。PASTE 2603.18897（模式感知）/ Speculative Tool Calls 2512.15834（小模型投机）
原理投机 = "赌一个大概率的未来，赌对就白赚时间，赌错就丢弃"。从 token 级 → tool 级 → 资源级，是一条清晰的扩展谱。

A5 · 模型级路由：不是每步都用最强模型

方法思路出处

A6 · 记忆与技能

记忆正从"被动检索"走向"主动、可学习的记忆技能"：模型自己决定 encode 什么、何时召回、怎么组织。

主动记忆

AutoMem 把"记忆管理"当可训练技能，长程任务 2–4× 2607.01224；Proactive Memory 用独立记忆代理主动注入提醒 2607.08716。

模型自驱动 GC

SideQuest 让模型自己判断哪些 tool response 已过期并清理 2602.22603——证明"recency=重要性"的启发式在 agent 场景会崩。

第三部分 · 支柱二：Agent Infra 与调度

B1 · 为什么传统 serving 失效

负载特征变了

长 session、KV cache 持续增长、工具调用突发、decode 状态不可迁移。

瓶颈也变了

工具执行（CPU 侧）可占端到端最高 88% 延迟，且 GPU 越强、这个占比越突出。2511.00739

B2 · 例子②：mixed mode 下 KV cache 的"转换"

"mixed mode" 指 prefill 与 decode 混合/分离下，多轮 agent 的 KV cache 如何被处理。核心是 PD 分离 与 append-prefill。

flowchart LR subgraph T1[Turn 1] P1[Full Prefill
整段上下文] --> D1[Decode] end D1 --> KV[(KV cache
已建好)] subgraph T2[Turn 2+] KV --> AP[Append-Prefill
只追加新增部分] AP --> D2[Decode] end KV -. 复用/迁移/分层 .-> STORE[(GPU ↔ CPU ↔ SSD)]
关键数字Turn 2+ 的 append-prefill 对 decode 的干扰仅 ~2%（而 full prefill 高达 48%）→ 因此 Turn2+ 可以安全地和 decode 本地 co-locate，省掉 75% 跨节点 KV 传输。PPD 2603.13358

"转换"体现在三处：① 复用（prefix caching，system prompt+历史高度重复）；② 迁移（跨实例/跨层搬运 KV）；③ 生命周期（保留 / 淘汰 / 回迁）。这三件事，正是下面调度要管的。

B3 · 【重头】Agent 调度：从请求分发到工作流编排

分 5 条线讲，每条配 KB 里的代表系统。

3.1 路由/负载均衡的演进

无状态(RR / Random / P2C) → 一致性哈希 → cache-aware(Radix Tree) → session 亲和

核心矛盾：cache 亲和（把请求发去有它 KV 的实例）vs 负载均衡。

最新工作SMetric（balanced session-centric scheduling）：洞察到"只均衡每个 session 的首个请求"就够——首请求纯按负载均衡分发，后续请求走 cache-aware 最大化本地 KV 复用，再配一个全局 KV store 兜底。既不牺牲复用、又不压垮少数实例。colocation 下集群 TPS +10–16%，disagg 下 prefill TPS +2–34%。2607.08565

3.4 ⭐ 空闲感知调度 + 工具空闲窗口 offloading（2606.00866）

你点名的这篇Idleness is Relative：agent 调工具时 GPU 会"空"下来；但"空闲"不该是非黑即白的 busy/idle，而应是一个连续的相对量。
空闲度

近 k 步内，"执行工具的时间"占"推理+执行总时间"的比例。越接近 1 = 越可能正在等工具（GPU 越该让出）。

flowchart TB R[按全局 idleness ι 排序
所有 program] --> GATE{双层准入} GATE -->|GPU 满: 优先下沉
高 ι 且 Acting 的| CPU[CPU 层] GATE -->|GPU 空: 优先上提
低 ι 且待推理的| GPU[GPU 层] CPU -. 懒降级:只剩推理任务时
让它先跑完当前步 .-> GPU

为什么用"相对排序"而非"绝对预测"

工具时长是重尾的（跨 3+ 数量级，比如 python 一行 0.1s vs mypy 180s）——逐次预测绝对时长不可行；但"谁比谁更可能马上要用 GPU"这个相对序好估得多。

和邻居的区别

Continuum 用绝对时长 TTL决策；Autellix 用已获服务量；本文用"谁更可能马上消耗 GPU"，并把上层排队优先级下沉为 GPU/CPU 双层一致的准入+驱逐。

对 MORI 意义最大连续相对 idleness 排名 + 双层准入 + 类型化驱逐，本质就是 MORI scheduler 的立身机制。可延伸方向："相对排名 + 绝对上限"混合（借 Continuum 的 TTL 截止，降低 CPU 层长期滞留）。

B4 · 一条贯穿的争论：预测 vs 观测

预测派

造更好的预测器去估计不可观测的 decode 成本。
Maestro（预测器做信号总线）· Chimera（拥塞预测）· PBKV（预测 + Lipschitz 退化保证）

观测派

把调度单元抬高，让不可观测量"消失"。
ConServe：调度 conversation 而非 turn，只用两个可观测信号，零学习模型。

矛盾根源不是"谁的预测更准"，而是调度单元不同：调度 stage 就必须预测 decode 长度；调度 conversation 则把不确定性折叠掉了。

B5 · KV 分层 offload 与资源回收

  • 回收 idle 资源：把空闲 GPU 变成投机 drafter（SPECTRE 2605.08151）；decode 空闲 SM 跑 PEFT（Harli 2511.11729）

第四部分 · KimiPoC：我们自研的 Agent Loop 与 SWE 实战

前面讲的是"业界怎么做"；这一部分讲"我们自己怎么做的"——在自研 orchestrator + 自建 sandbox + 自部署 Kimi-K2.6 serving 上，做了一个完整的 coding agent，并把 SWE-bench 分数从基线一路打上去。所有结论都来自我们自己的真实跑测。

4.1 为什么要自研 Agent Loop

关键发现我们探针测过：开源 sglang serving 不支持 Moonshot 托管版的 swarm 多代理参数（发过去被静默忽略）。所以"多代理编排必须自己在 orchestrator 里实现"——这也直接呼应了 §A2 的结论：护城河在 harness/编排，不在模型接口。

自研 loop 用 LangGraph 骨架，三节点循环：

flowchart LR IN([task]) --> CC[condense_context
压缩/拼装上下文] CC --> LLM[llm_node
Kimi-K2.6 推理] LLM -->|工具调用| TN[tools_node
bash/read/str_replace/task...] TN --> CC LLM -->|完成| OUT([patch/结果])

组件：自研 orchestrator（agent_graph.py + swebench_runner.py）· 自建 sandbox（TCS pods）· 自部署 Kimi-K2.6（多 TP4 实例并行）。

4.2 SWE 提升实践：什么真正管用

基准配置Resolve
Lite hard-30基线（无 str_replace）5/30（17%）
Lite hard-30+ str_replace + 语法门9/30（30%）
Lite hard-30+ reproduce-first + retry-on-empty + max-turns19/30（63%，修正评分后）

对照：SWE-bench Lite 全量基线约 24.3%。harness 工程把 Kimi-K2.6 从 17% 拉到 ~60%。

真正"推动指针"的三件事（按贡献排序）

  1. str_replace + 语法门（17%→30%，两次跑都 9/30，唯一可复现的稳定收益）：精确+唯一匹配、Python 语法检查、出错不写入。根治了行级 file_patch 造成的 NameError/SyntaxError 代码损坏。用 python_exec 片段实现 → 无需重建 sandbox 镜像。（借鉴 OpenHands ACI editor："编辑器质量 > 模型"）
  2. retry-on-empty + reproduce-first + max-turns 25→60（最大杠杆，30%→53%/63%）：头号失败模式是 empty-patch——"探索了半天但从没改代码"（约占一半，sympy 7/7）。针对它把空补丁从 ~15 降到 4。
  3. 评分器 per-repo 修复：in-sandbox grader 把 test_patch heredoc 里的 pytest 行误当命令、django FAIL_TO_PASS 是文档串导致 shell 崩——修好后回收了多例"假失败"（Lite 3 例、Verified 7 例），真实分数比原始更高。
诚实的边界N=30 单次跑有 ±3–5 的方差（我们见过 9/14/16/19）；str_replace 以下的增量特性在噪声地板附近，需要 N≥100 才能分辨信号。—— 这也是为什么我们更信 Verified-100 的 59%。

4.3 Subagent：agent-as-a-tool

子代理 = 把"另一个 agent loop"当一次工具调用：全新隔离的消息历史、跑到结束、只把最终摘要返回给父代理。子代理嘈杂的探索（读文件/grep/日志）留在子窗口里被丢弃 → 上下文经济 + 可并行。

两条正交的轴（核心心智模型）

上下文（默认独立、一次性、返回摘要；fork 是显式 opt-in 才继承父上下文）与 sandbox（借用父 pod / 独占自己的 pod）是相互独立的两个维度。一个"改完代码交回父代理"的子代理，上下文仍完全独立——它只是借用父的 sandbox 干活，再摘要回来（等价于一次 bash 调用）。

模式sandbox写仓库?典型用途
borrow-parent(write)共享父 pod，串行是（持久）实现者交接、P1 continue-fixing
borrow-parent(read)共享父 pod否（/tmp）轻量探索
own pod独立 pod是（自己的副本）best-of-N、P1 verifier

安全不变量：一个 pod 同时只能有一个写者（borrow 是串行的、父被 await 阻塞；并行写必须各用独立 pod）；MAX_SUBAGENT_DEPTH=1（子代理不能再生子代理）；一次性、不 resume。

P1 · 验证器重试（oracle-free）

模型只写复现，harness 判定：复现必须在未修复代码上先 FAIL、打补丁后 PASS 才算数；单调回退（新补丁只有让复现转 PASS 才保留，否则 revert）→ 免疫 fix-churn 和假 FAIL。绝不读 FAIL_TO_PASS。

P2 · best-of-N（oracle-free 选择）

N 个独立 pod 并行解，选择器用"复现是否 pre-fix FAIL + 更小 diff + lint 干净"等无 oracle 信号选赢家，只对赢家评一次分。

巧思串行子代理本身就是一个任务级上下文压缩原语：把子任务几十轮探索坍缩成一条摘要——"委派子任务"和"压缩该子任务轨迹"是同一个操作（与 §A2/token 级压缩正交）。

4.4 Subagent 对 Infra（MORI serving）的要求

子代理这个算法特性，直接给底层 serving 提了新需求——正好把本场"算法 ↔ infra"两条线接起来：

需求为什么状态
fork 的亲和路由fork 继承父上下文 → 必须路由到持有父历史 KV 的 worker（prefill+decode 都要，因 decode 亲和把历史 KV 累积在 decode worker）。用 x-mori-affinity-session-id=父 id 做路由键、x-mori-session-id=子 id 做 KV 归属键已实现（mori-sched 优先用亲和键）
独立子代理 = 无亲和默认独立上下文，KV 前缀不共享 → free/least-loaded 放置即可已有
EvictSession（按 session 急切驱逐 KV）一次性子代理返回即"死"，其 KV 前缀不该污染 radix cache⛔ 卡在 sglang（只有 LRU / 全局 flush，无按 session evict）
ClonePod（CoW 秒级克隆）best-of-N 需要从父状态种子化 N 个 podtar-relay 层已实现；containerd CoW 为北极星
子代理 pod 租约 < 父孤儿一次性 pod 应比父更早回收已实现（子 ~60s vs 父 ~90s）
对 MORI 的意义subagent 把"上下文独立性"和"亲和路由/急切驱逐"变成了 serving 的一等需求。fork 的双侧（prefill+decode）亲和、按 session 的 KV 驱逐、pod 差异化租约——这些正是 MORI 相对通用 serving 的差异化价值点。
