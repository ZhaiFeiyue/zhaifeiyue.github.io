持续执行范式：Persistent Engine Kernel vs Megakernel

1. 主题缘起

2024–2026 年，LLM 推理工作负载从大 batch 训练时代的规则化执行模式，全面转向延迟敏感的微 batch serving。Token-by-token decoding 每次 forward 仅涉及少量数据但触发上百个 μs 级小操作——attention QKV、softmax、activation、LayerNorm、KV cache update 等 [L2:2604.17861]。GPU 的 kernel launch 机制仍为 ms 级大 kernel 设计，每次穿越 host-device 边界的固有开销在 3–7 μs [L2:2604.17861]。当操作本身仅需 3–10 μs compute 时，launch 开销占 30–100%，GPU profiler 呈典型"梳齿形"——计算 burst 之间充斥 host 提交空白期 [L2:2604.17861]。

以 8×H200 NVL 系统为例：聚合内存带宽 ~38 TB/s，GLM-5.1 每 decode token 激活参数量 ~42 GB，纯带宽约束下理论吞吐可达 ~1000 tok/s，而实际系统仅输出几十 tok/s——差距达一个数量级 [L2:tilert-speed-scaling-law]。根因不是算力不足，而是 GPU 反复执行 launch→load→compute→store→synchronize 循环，每次 kernel boundary 打断数据流、销毁局部性、强制重新同步 [L2:tilert-speed-scaling-law]。

与此同时，GPU 硬件从单 die 走向多 chiplet 架构——AMD MI300X/MI350 的 8 颗 XCD 各有 4 MB 私有 L2，跨 XCD 不相干 [L2:2604.15379]。CUDA/HIP 编程模型仍把 L2 当设备级共享资源暴露——LLM decode 时 8 颗 XCD 独立从 HBM 拉同一份权重到各自 L2，带宽被浪费 [L2:2604.15379]。FlashAttention2 的 round-robin 调度将同一 attention head 的 workgroup 打散到不同 XCD，导致 L2 命中率暴跌至 ~1% [L2:2511.02132]。

这两股压力——kernel 边界开销和 chiplet NUMA 效应——独立驱动研究者走向同一方向：持续执行（persistent execution），GPU 程序在操作之间不将控制权交还 host。2024–2026 年间三条独立路线同时涌现：AOT 编译的全模型 Engine Kernel [L2:tilert-speed-scaling-law]、JIT 动态 persistent dispatcher [L2:2604.17861]、chiplet-aware persistent megakernel [L2:2604.15379]。这一主题横跨 kernel（算子级优化、硬件调度、通信重叠）和 framework（编译策略、运行时编排、部署架构），是当前推理基础设施中最活跃的跨领域交汇点。

谱系包含三个关键分支点：

编译策略分叉。GPUOS 选择 JIT 保持动态性——NVRTC 模板编译 + dual-slot aliasing 零停机热更新 [L2:2604.17861]。Fleet 选择静态 task graph 预编译，获取跨 operator 寄存器分配等编译器优化空间 [L2:2604.15379]。TileRT 选择 AOT 全模型静态展开，获取最大优化边界 [L2:tilert-speed-scaling-law]。三者是灵活性-性能 Pareto 曲线上的不同采样点 [L3:tilert-speed-scaling-law]。

Chiplet 感知与持续执行的汇流。NUMA-aware Attention 首先识别了 chiplet 架构下 L2 分区的性能悬崖 [L2:2511.02132]。HipKittens 提供了 tile-level 的 XCD 调度算法（联合优化 L2+LLC，GEMM 加速 19%）[L2:2511.08083]。Fleet 将两者统合进 persistent megakernel：kernel 边界消除后 L2 状态在整个 decode 周期得以保留 [L2:2604.15379]。

通信策略从外部隔离到内部融合。ConCCL 通过 DMA offload 实现计算与通信的物理隔离 [L2:2412.14335]。TileRT 走向另一极端——通信直接嵌入 tile pipeline 内部，整个 attention 层从 host 视角仅为一次 Engine Kernel launch [L2:tilert-speed-scaling-law]。

5. 技术线交错

5.1 AOT 刚性 vs 运行时灵活性

TileRT 的 AOT 编译在编译期静态规划 GPU 间通信的 tile 对齐和数据依赖，运行时无法动态调整 GPU 角色分配或通信拓扑 [L2:tilert-speed-scaling-law]。每次模型架构更新需要 recompile——compiler IR、tile scheduler、通信原语、执行模型全部需要更新 [L3:tilert-speed-scaling-law]。

GPUOS 走向对立面——NVRTC JIT + dual-slot aliasing + version counter 实现零停机 operator 热更新，新 operator 从编译到全设备可调用仅需 ms 级 [L2:2604.17861]。但 template-based compilation 限制了表达能力，无法注入需要精细控制 shared memory layout 和 warp-level primitive 的自定义 kernel [L3:2604.17861]。

Fleet 介于两者之间——task graph 编译时确定，但运行时由 per-XCD 软件 scheduler 动态分派 [L2:2604.15379]。Mirage 超优化器尚未理解 Chiplet-task cost model，当前只能手动提供输入 task graph [L2:2604.15379]。

这三条路线映射到 ML 生态的不同阶段：GPUOS 的 JIT 方案适合 exploration-heavy 的研发阶段；Fleet 的 static+runtime-scheduler 适合 model-fixed 的部署阶段；TileRT 的全 AOT 适合 everything-fixed 的极致生产环境 [L3:2604.17861]。

5.2 NVLink 域约束 vs Chiplet NUMA 约束

TileRT 在 8×H200 NVL 域内工作，核心约束是 TP synchronization 的通信开销——GPU0 特化为 Sparse Indexer Worker、GPU1–7 为 MLA Workers，通信嵌入 tile pipeline [L2:tilert-speed-scaling-law]。Fleet 在单块 MI350 的 8-XCD 域内工作，核心约束是 L2 不相干——M-major windowed traversal 让同 XCD workers 共享权重列 [L2:2604.15379]。

两者的异构特化思路相通但作用域差异一个数量级：Fleet 是 XCD 粒度（同 GPU 内 8 个 XCD 分工），TileRT 是 GPU 粒度（8 块 GPU 分工）[L3:tilert-speed-scaling-law]。HipKittens 的 XCD 调度算法（Algorithm 1）证明 chiplet 感知在 kernel 层面可带来 19% GEMM 加速 [L2:2511.08083]，但只在单 kernel 级别——无法跨 kernel 维持 L2 affinity。Fleet 的 persistent megakernel 将此扩展到 decode 全路径 [L2:2604.15379]。

5.3 Kernel 级 tile 抽象 vs Framework 级 tile pipeline

HipKittens 定义 tile 作为 kernel 内部的编程抽象——typed tiles 在 register/shared/global memory 间流转，PyTorch 风格算子操作这些 tiles [L2:2511.08083]。TileRT 将 tile 提升为 framework 级调度抽象——compute、communication、async I/O 均为 tile-level task，在 GPU 内部持续推进，中间结果驻留寄存器和 shared memory [L2:tilert-speed-scaling-law]。

从 kernel-level tile 到 framework-level tile pipeline，抽象边界的上移伴随灵活性下降和优化空间扩大——这正是 kernel 和 framework 两个类别在持续执行范式下的核心交错。

5.4 计算-通信分离 vs 融合

ConCCL 证明将通信 offload 到 DMA 引擎（位于 IOD 层，不经过 L1/L2 cache）可完全消除 compute interference——通信和计算使用不同硬件执行器，互不干扰 [L2:2412.14335]。TileRT 采取"深度融合"策略——通信不经 NCCL 外部编排，直接嵌入 tile pipeline 作为 tile-level 通信任务执行 [L2:tilert-speed-scaling-law]。两种策略适用不同通信规模：ConCCL 在 ≥128 MB 时与 RCCL 持平 [L2:2412.14335]，TileRT 的 pipeline 内通信适合 tile 粒度的细粒度 broadcast/reduce。

6. 共识与分歧

6.1 共识

Inter-kernel overhead 是当前微 batch 推理的头号敌人。 GPUOS 量化了这一开销：null kernel launch 3–7 μs，100 ops/token 累计 300–700 μs 纯协调开销 [L2:2604.17861]。TileRT 从系统角度验证：理论 ~1000 tok/s vs 实际几十 tok/s，差距一个数量级 [L2:tilert-speed-scaling-law]。Fleet 从 chiplet 角度补充：每 token 近 250 次 launch，且每次 launch 都会 flush L2 以保证 device-sync 语义 [L2:2604.15379]。所有 6 篇工作都直接或间接地以消除 kernel 间协调开销为目标。

"Launch once, dispatch many"是持续执行的共识范式。 GPUOS 进程启动 launch 一次后永不退出 [L2:2604.17861]；Fleet megakernel 占满所有 CU 直到 decode 结束 [L2:2604.15379]；TileRT host 仅启动一次 Engine Kernel [L2:tilert-speed-scaling-law]。核心共识：问题不在"优化每次 launch"，而在"消除 launch 的概念" [L3:2604.17861]。

Chiplet 架构下 L2 cache 不可再被视为统一资源。 NUMA-aware Attention 证明 round-robin 调度在 MI300X 上导致 L2 命中率 ~1% [L2:2511.02132]；Fleet 证明无 chiplet-aware 调度时 decode 性能被 L2 复用率限死 [L2:2604.15379]；HipKittens 证明联合优化 L2+LLC 可带来 19% GEMM 加速 [L2:2511.08083]。

6.2 分歧

JIT vs AOT。 GPUOS 通过 NVRTC JIT 保持完全动态 shape/operator 支持 [L2:2604.17861]，TileRT 通过 AOT 静态展开获取最大编译优化空间 [L2:tilert-speed-scaling-law]。两者的根本假设不同——GPUOS 假设 workload 是"大量异构操作的动态混合流"，TileRT 假设 workload 是"固定模型在固定硬件上的确定性执行"。GPUOS 的 dual-slot aliasing 支持零停机热更新，TileRT 的模型结构变更需要 recompile——这是两条路线的根本性分叉 [L3:tilert-speed-scaling-law]。

Persistent kernel 的资源分配哲学。 GPUOS 极其保守——每 SM 仅 1 个 thread block（2–4% 总线程），定位"填谷"，大 GEMM 仍走传统路径 [L2:2604.17861]。Fleet 走向另一极端——persistent megakernel 占满所有 256 CU（减 3.1% scheduler），将所有 decode ops 融合 [L2:2604.15379]。技术根源在于 register pressure：Fleet 将所有 task 类型编进同一函数，寄存器用量取联合最大值，被迫只跑 1 wave/SIMD；GPUOS 因只处理微操作避免了这一约束 [L2:2604.15379]。

Fusion 的范围。 GPUOS 通过四维 filter 选择性路由微操作，大 GEMM 不走 persistent kernel [L2:2604.17861]。TileRT 将大 GEMM、attention、communication 全部纳入 persistent pipeline [L2:tilert-speed-scaling-law]。Fleet 覆盖 linear GEMM + attention + RMSNorm + SiLU，但在 compute-bound 区间（bs ≥ 64）反而不如专用 hipBLASLt [L2:2604.15379]。这揭示了一根结构性 trade-off 轴：跨算子持久化复用必须接受寄存器预算取并集，单 kernel 极致调优必须接受每次 launch 刷新 L2 [L3:tilert-speed-scaling-law]。

硬件阵营分裂。 GPUOS 深度绑定 NVIDIA（NVRTC、CUDA Driver API、PTX）[L2:2604.17861]。Fleet、NUMA-aware Attention、HipKittens 深度绑定 AMD CDNA（per-instruction scope 位、buffer_wbl2、HW_ID 寄存器）[L2:2604.15379] [L2:2511.02132] [L2:2511.08083]。TileRT 运行在 NVIDIA H200 上 [L2:tilert-speed-scaling-law]。当前持续执行范式完全沿厂商线分裂，无跨平台方案存在。

7. 根本性困难

7.1 动态模型架构 vs 静态编译

MoE 模型中每 token 的 expert routing 是 input-dependent 的，从根本上挑战了 AOT 全模型静态编译 [L3:tilert-speed-scaling-law]。TileRT 声称已服务 GLM-5.1（MoE 模型）上线 [L2:tilert-speed-scaling-law]，但未说明如何处理 routing 动态性。Fleet 的 M-major 协作假设在 MoE 下被直接破坏——不同 token 走不同 expert 权重 [L2:2604.15379]。GPUOS 的 template-based JIT 无法注入需要精细 shared memory 控制的 custom kernel [L3:2604.17861]。

预编译所有可能路径导致编译产物爆炸，在 kernel 内实现动态分发则本质回到运行时调度。需要 kernel 级的动态 expert 分发和 framework 级的 AOT/JIT 混合编译共同演进 [L3:tilert-speed-scaling-law]。预计难度：2–3 年。

7.2 Chiplet 执行的跨厂商统一

Fleet 的两级事件同步 + CDNA 三档 cache modifier 深度绑定 AMD ISA——buffer_wbl2 / flat_atomic_add sc0 sc1 / sc1=1 nt=1 的精确语义是 persistent megakernel 在 8-die GPU 上成立的唯一支点 [L2:2604.15379]。NVIDIA Blackwell 仅有粗粒度 _cs/_cg/_ca 修饰符，无 XCD-scope 概念 [L2:2604.15379]。HipKittens 和 ThunderKittens 已在 tile 接口层实现跨厂商统一 [L2:2511.08083]，但底层调度必须根本性重写。

跨厂商统一需要统一 cache scope 语义和同步原语——这是 ISA 级别分歧，非软件层可消弭。预计难度：3–5 年，取决于硬件厂商标准化意愿。

7.3 硬件极限下的执行稳定性

TileRT 在生产流量下面临的核心挑战不是峰值性能而是执行稳定性：KV 缓存碎片化导致内存局部性退化，MTP 的 draft/verify 路径持续动态重塑 pipeline，FP8 超长上下文下通信放大成为新瓶颈 [L2:tilert-speed-scaling-law]。Fleet 面临类似约束——register pressure 导致每次 L2 miss 直接 stall MFMA pipeline，L2 命中率从"锦上添花"升格为"生死攸关" [L2:2604.15379]。

接近硬件极限的持续执行系统没有安全裕度。传统 kernel-per-op 模型中单个 kernel 失败只影响一个操作；persistent kernel 中任何异常（XCD power event、L2 miss storm、NVLink 抖动）都可能导致整个 decode 失败。预计难度：持续性挑战。

7.4 Persistent kernel 与集体通信的多 GPU 共存

GPUOS 的全部评估在单 GPU 完成 [L2:2604.17861]，Fleet 同样仅评估单 GPU [L2:2604.15379]。Persistent kernel（每 SM 1 block 永不退出）与 NCCL collective kernel（需占用所有 SM）之间的资源竞争是严重未知。ConCCL 的 DMA offload 解决了通信侧——不再占 CU，GEMM 全速运行 [L2:2412.14335]——但其 CPU 编排在小传输 (<32 MB) 时慢 4× [L2:2412.14335]。

Production LLM serving 几乎必然是 multi-GPU（≥7B 模型需 TP≥2），persistent kernel 的 SM 常驻与 collective 的全 SM 需求之间缺乏硬件级隔离机制 [L3:2604.17861]。预计难度：1–2 年。

9. 邻接 topic

cluster-llm-deployment。持续执行当前局限于单卡或单 NVL 域。向集群规模扩展时（multi-node TP/PP），persistent kernel 与跨节点 collective、PD 分离架构、多租户调度的共存是自然延伸问题 [L3:tilert-speed-scaling-law]。Fleet 在 bs ≥ 64 被 vLLM 反超 [L2:2604.15379]，暗示集群级调度需要在 persistent 和 traditional 路径之间动态切换。

agent-system。Agent serving（多轮、小 batch、强延迟敏感）落在持续执行最优的 bs=1–16 区间 [L2:2604.15379]。TileRT 的"更快推理 = 更多 rollout = 更深推理链"叙事直接映射到 agent 场景的 test-time scaling 需求 [L2:tilert-speed-scaling-law]。两个 topic 的边界在 agent workload 的 KV cache 管理和 tool call 延迟处模糊化。

潜在演化：如果硬件层面解决了 launch overhead（如 NVIDIA GB10 已部分实现 [L2:2604.17861]），本 topic 可能收敛为纯 chiplet 优化方向。如果 chiplet 成为标准（Rubin Ultra 4 die），则可能拆分为 NVLink 域和 chiplet 域两个子 topic。
