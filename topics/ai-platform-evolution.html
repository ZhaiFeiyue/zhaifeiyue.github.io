AI 平台演进：从 MaaS 卖 Token 到 AaaS 卖结果 #

1. 主题缘起 #

2023–2026 年，AI 服务平台经历了三次范式跃迁：IaaS 卖 GPU 算力 → MaaS 卖 Token → AaaS 卖结果。每一次跃迁都伴随着计费单位的抽象化、风险承担方的转移、以及基础设施栈的结构性重组。

2026 Q1 出现了结构性拐点：Fortune 500 企业在 AI Agent 上的支出首次超过新 SaaS seat 支出。Per-seat 定价的市场份额从 21% 降至 15%，混合订阅+用量模式从 27% 升至 41%。Salesforce 推出 Agentic Work Unit (AWU) 将 token 消耗抽象化，按"一个 agent 完成的离散任务"计费——AWU 季度环比增长 57%，Agentforce 业务线已超 $1.8B 营收。

与此同时，Token 价格呈现经典的 Jevons Paradox：GPT-4 级模型的 token 价格从 $20/M (2022) 降至 $0.40/M (2025)——50 倍降幅，但企业 AI 支出从 $11.5B 暴涨至 $37B——320% 增长。更便宜的 token 催生了更多消耗：reasoning 模型消耗 5-20x 更多 token（chain-of-thought）、agent 工作流链式乘法增长、context window 持续膨胀。

本 topic 从平台架构师的视角，系统梳理 AI 服务平台从 MaaS 到 AaaS 的演进路径，覆盖平台组件、商业模式、基础设施适配三个维度。与 agent-scheduling topic 的关系：agent-scheduling 聚焦 serving 引擎内部的调度算法（Router + KV Cache 管理），本 topic 聚焦 serving 引擎之上的平台层（Gateway、Billing、Orchestration、Payment）和之下的集群管理层（GPU 管理、K8s 编排）。两者在 Router/Gateway 层存在接口——Model Router（本 topic）调用 Serving Router（agent-scheduling），形成两级路由。

2.1 关键转变机制 #

SaaS → MaaS：核心驱动力是"build vs buy 的数学被算力碾碎"。托管一个 1T+ 参数前沿模型的年化成本（GPU + 电力 + 工程师）远超通过 MaaS API 调用的成本。MaaS 提供商通过多租户架构将基础设施开销分摊到数百万用户。对平均企业而言，$0.20-$5.00/M tokens 的 API 调用 vs 数百万美元的自建成本——经济账一目了然。

MaaS → AaaS：核心驱动力是"Token 度量计算而非价值"。一个人类 30 秒完成的任务可能消耗数千 token，一个人类数小时的任务可能只需很少 token——compute cost 和 customer value 完全脱耦。AWU 的设计意图是"抽象掉 token 消耗，把定价绑定到客户真正关心的结果上"。

风险转移是最深层的变化：SaaS 时代用户承担 shelfware 风险（买了软件不用是你的问题），AaaS 时代厂商承担效果风险（agent 没完成任务不收费）。这消灭了"feature bloat"——厂商不再堆砌无用功能凑清单，只构建直接驱动 billable outcome 的能力。

3.2 设计原则 #

  1. Rate Limiting（执行）和 Metering（记账）分离：限流是实时决策，计量可以异步。Metering 走 Kafka/NATS 队列，不在推理热路径上增加延迟。
  2. 单一 OpenAI-compatible endpoint：屏蔽后端 provider 差异，所有现有 SDK/工具零改动接入。
  3. Tier-based 访问控制：free / standard / premium → 不同的模型访问权限、速率限制、SLA 保证。

    4.1 Router 层的两级分化 #

    MaaS 到 AaaS 的核心架构变化是 Router 从一级变为两级：

    级别职责决策信号实现方
    L1 Model Router选模型 + provider + region任务复杂度, 成本约束, SLA 要求平台层 (本 topic)
    L2 Serving Router选 worker + 管 KV CacheCache 驻留状态, prefix 匹配, 负载, session 亲和Serving 层 (agent-scheduling topic)

    L1 和 L2 之间需要双向信息传递：L1 告诉 L2 "这个 session 预计 20 turns"，L2 反馈 L1 "KV Cache 压力大，建议降级模型"。当前无标准接口。Bedrock 的 Prompt Routing 是 L1 的初步尝试，但完全不感知 L2 的 cache 状态。

    4.2 计费层的结构性升级 #

    6.1 GPU 集群管理 #

    推理服务用 K8s（服务型负载），训练可以 Slurm。关键组件：

    • NVIDIA GPU Operator（或 AMD 对应的 ROCm Device Plugin）：驱动管理 + 设备发现
    • KAI Scheduler：Gang scheduling + 拓扑感知 + 分数 GPU 共享
    • Kueue：多租户配额管理 + job queuing + 借用/抢占
    • KubeRay：分布式推理协调（TP/PP 跨节点）
    • Karpenter / KEDA：GPU-aware 弹性伸缩（基于 GPU 利用率/队列深度，而非 CPU）
    • Storage：RWX (NFS/CephFS) 共享模型权重 + Local NVMe 缓存 KV

    7. 共识与分歧 #

    共识 #

    1. Token 是过渡态，Outcome 是终态：Token 定价正在被 AWU / per-resolution 等结果定价取代，但底层仍需 token-level 的精细计量。计量和计费是两个独立问题——计量永远是 token 级，计费可以是 outcome 级。
      1. 智能路由是 MaaS 平台的核心 margin 杠杆：100-600x 的模型价差意味着路由策略直接决定 gross margin（AI 应用公司 50-60% vs 传统 SaaS 80-90%）。谁的路由更智能，谁的 margin 更高。
        1. 中间层 SaaS 被掏空：Agent 直接操作底层 API 和数据，跳过传统 UI-centric workflow。价值向编排层（上）和数据/基础设施层（下）两端集中。
          1. Agent 需要独立金融基础设施：x402、AgentCore、Credit Wallet 正在构建 agent 作为独立经济体的支付能力。2026 年 credit wallet 从实验走向标准基础设施。
            1. MaaS 最难改的两层（Router + Engine）恰好是 agent-scheduling topic 集中攻击的方向：平台上层（Gateway、Billing、Observability）的修改相对独立，但真正的竞争壁垒在 serving 层的 agent-aware scheduling。

              2. 分歧 #

              1. Token 定价 vs Outcome 定价的落地路径：纯 outcome 定价对厂商收入不可预测，纯 token 定价对用户价值不透明。市场可能收敛于 hybrid（基础平台费 + outcome 溢出），但最终比例未定。
                1. 平台 vs 垂直的竞争格局：通用 MaaS 平台（OpenAI、Bedrock）是否会被垂直 AaaS（Sierra AI for customer support、Harvey for legal）瓦解？Bessemer 认为"最成功的 AI 公司不是在收费技术，而是在收费完成的工作"。
                  1. 自建 vs 外购 inference：MaaS 推动"不要自建"，但 agent 的状态性要求（KV Cache 亲和、session sticky）让自建推理集群重新有了理由——你无法在第三方 MaaS API 上做 cache-aware routing。

                    2. 8. Open challenges #

                    8.1 L1-L2 Router 接口标准化 #

                    Model Router (L1) 和 Serving Router (L2) 之间缺乏标准化接口。L1 的 "session 预计 20 turns" 信息无法传递给 L2，L2 的 "KV Cache 压力过大" 信号无法反馈 L1。当前两者独立运作，形成跨层优化盲区。难度：1-2 年（工程问题 + 标准化博弈）。

                    8.2 跨模型 KV Cache 共享 #

                    Agent workflow 混用多模型时，每个模型需要独立 KV Cache，导致内存爆炸。ICaRus 和 PrefillShare 正在探索跨模型 KV 共享，但需要模型架构层面的适配。难度：2-3 年。

                    8.3 Credit Wallet 跨供应商互操作 #

                    单供应商 credit wallet 已经可行。但真实 agent workflow 跨多供应商——一个 wallet 管理多种 credit 类型、不同的 pooling/rollover 规则、跨供应商结算。需要生态级标准。难度：3-5 年。

                    8.4 Outcome 度量的客观性 #

                    Outcome-based 定价需要客观度量"任务是否成功完成"。当 agent 的结果和用户内部流程交织时，归因争议不可避免。需要可验证的 outcome attestation 机制。难度：2-3 年。

                    9. 邻接 topic #

                    Topic关系
                    agent-scheduling强关联——agent-scheduling 覆盖 L2 Serving Router + L5 Inference Engine 内部的调度算法，本 topic 覆盖之上的平台层和之下的集群管理层。两者在两级路由（L1 Model Router ↔ L2 Serving Router）处接口
                    cluster-llm-deployment强关联——cluster-llm-deployment 覆盖部署拓扑（xPyD、TP/EP/PP 选择），本 topic 的 L1 Infrastructure 层直接依赖其内容
                    agent-system中关联——agent-system 覆盖 agent 从模型到框架的全栈架构，本 topic 聚焦 agent 如何被"服务化"交付
                    agent-context-lifecycle弱关联——context lifecycle 影响 Metering（cached_tokens 需要被正确计量）和 Billing（compacted context 的计费归属）
