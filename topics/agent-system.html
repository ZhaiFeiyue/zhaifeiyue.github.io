Agent System：从请求推理到自主工作流的系统范式跃迁 #

1. 主题缘起 #

2025–2026 年，LLM 的主流使用模式正从单请求问答转向多轮自主工具调用。这一转变引发了贯穿 model、framework、agent、code 四个领域的系统性重构需求。

在 agent 领域，CPU 侧工具执行被发现占端到端延迟高达 88% [L2:2511.00739]，投机性工具执行将延迟降低 48.5% [L2:2603.18897]，而更深层的问题在于 KV cache 管理——end-of-turn eviction 导致 per-turn queueing delay 累积占总延迟 58.2% [L2:2511.02230]，LRU 在 agent 工作流中命中率仅 27–28% [L2:2605.06472]，启发式剪枝更导致模型崩溃（non-completion rate 60%+）[L2:2602.22603]。同时，对 Claude Code 512K 行源码的逆向分析揭示了生产级 agent 的核心范式——1.6% 决策逻辑 + 98.4% 确定性基础设施 [L2:2604.14228]。

在 framework 领域，serving 系统从请求级调度演进到 program-level 调度。KV cache 在 agentic workload 下的 middle-phase thrashing 导致 49% 延迟被浪费在重计算上 [L2:2601.22705]。SAGA 测量显示 38% 的集群执行时间花在重新生成被驱逐的 KV cache 上，GPU 利用率仅 42% [L2:2605.00528]。PD 分离架构的存储带宽瓶颈在 agentic workload 下尤其严重——KV cache 命中率高达 98.7%，prefill 从计算密集退化为 I/O 密集 [L2:2602.21548]。Halo 首次将数据库批查询优化引入 agentic workflow，GLM 的 multi-agent serving 达到 15.1× 吞吐提升 [L2:2509.02121] [L2:2511.01633]。vLLM Semantic Router 作为 Envoy ExtProc gRPC sidecar，通过 20+ 异构信号类型上的布尔表达式树决策，实现模型集群间的实时 per-request 智能路由 [L2:vllm-project-semantic-router]。

在 model 领域，两条架构路线同时为 agent serving 开辟空间。DeepSeek-V4 的 hybrid CSA+HCA attention 将 1M 上下文下 KV cache 压至 V3.2 的 10%，单 token 推理 FLOPs 仅 27% [L2:deepseek-v4]。Kimi K2 系列三代复用同一 1T MoE 主干，K2.6 将单次 rollout 推到 4000+ 工具调用、12–13 小时连续执行 [L2:kimi-k2-6]。Kimi Linear 的 KDA 线性注意力将 decode 吞吐提升 75%、KV cache 减少 75% [L2:2510.26692]——直接降低 agent 的 per-session 内存占用。

在 code 领域，OpenAI Agents SDK 和 Hermes Agent 分别代表了"声明式编排"和"自我进化闭环"两条工程路线，为 agent 系统提供了可落地的开发范式 [L2:openai-openai-agents-python] [L2:NousResearch-hermes-agent]。

这四个领域的技术演进共同指向一个核心命题：agentic workload 不是 LLM serving 的一种特殊模式，而是一个需要从模型训练、推理引擎、缓存策略、路由策略、调度策略到应用框架全栈重新设计的新系统范式。

          2. 5. 技术线交错 #

          模型能力驱动框架需求 #

          Kimi K2.6 的 4000+ 工具调用 / 12–13 小时连续执行 [L2:kimi-k2-6] 对推理基础设施提出了极端要求。DualPath 的分析显示 working set 随 tool call latency 以 $r^2$ 扩展、总成本以 $r^3$ 增长 [L2:2602.21548]。DeepSeek-V4 的 CSA+HCA 将 1M 下 KV cache 压至 V3.2 的 10% [L2:deepseek-v4]，若应用于 agent serving 可直接将 per-agent 的 GPU 内存占用降低一个数量级；Kimi Linear 的 KDA 层以 fixed-size recurrent state（128×128/head）替代累积 KV [L2:2510.26692]，从根本上消除了 KV cache 随 turn 数线性增长的问题——这对 Continuum/SAGA/PBKV 的 cache 管理策略构成"问题消失"的上游挑战。

          Agent KV Cache 管理成为独立研究集群 #

          2025-11 至 2026-05 的 8 个月内，5 篇论文从不同角度攻击同一问题——agent 多轮推理中的 KV cache 管理 [L3:agent]。Continuum 用时间统计 [L2:2511.02230]，SideQuest 用语义理解 [L2:2602.22603]，SAGA 用 DAG 结构预测 [L2:2605.00528]，PBKV 用 GNN 多步预测 [L2:2605.06472]，CMV 用会话级 DAG 版本控制 [L2:2602.22402]。这一集群的爆发表明 agent-aware KV 管理已从单点创新走向方案竞争阶段。

          Agent 运行时约束 framework 设计 #

          Claude Code 的 5 层 context compaction pipeline [L2:2604.14228] 揭示了生产级 agent 对 serving 系统的深层需求：prompt cache awareness、very long context（200K–1M tokens）、streaming tool dispatch、abort/recovery semantics。PASTE 的投机执行中间件要求 serving 引擎支持 authoritative vs speculative 任务的优先级隔离 [L2:2603.18897]。SAGA 的 session-affinity batching 要求引擎暴露 per-session KV cache 的跨 worker 迁移接口 [L2:2605.00528]。这些需求推动 vLLM/SGLang 从"单请求引擎"演进为"agent-aware runtime"。

          工具执行瓶颈重塑调度策略 #

          CPU-Centric Profiling 发现的 CPU 并行化饱和点（BS=128 时 throughput gain ratio ≈ 1）[L2:2511.00739] 直接影响了 Concur 和 ThunderAgent 的设计——两者都把 agent 的 tool 执行期视为一等调度信号 [L2:2601.22705] [L2:2602.13692]。Continuum 将 tool call 延迟的 CDF 作为 TTL 计算的核心输入——发现 cd 命令最慢 10% 占 94.1% 总延迟 [L2:2511.02230]，长尾工具延迟成为 KV 保留决策的关键变量。

          SDK 设计反映架构共识 #

          OpenAI Agents SDK 的 Runner 循环 + Guardrails + Handoffs 结构 [L2:openai-openai-agents-python] 与 Claude Code 的 queryLoop + Permission System + AgentTool 设计 [L2:2604.14228] 在抽象层级上高度同构，表明生产级 agent 的架构模式已经趋于收敛：中央循环 + 安全网关 + 工具分发 + 子代理委派。Hermes Agent 的 skill 自动创建和持久记忆则映射到 Claude Code 的 CLAUDE.md 4 级层次和 MCP 扩展机制 [L2:NousResearch-hermes-agent] [L2:2604.14228]。

          路由层作为独立系统层 #

          vLLM Semantic Router 在 agent stack 中引入了一个全新的基础设施级路由层——位于应用编排器之下、推理引擎之上 [L2:vllm-project-semantic-router]。这一层将"模型选择"从应用代码中剥离并下沉到基础设施，类似 Envoy 将负载均衡从应用代码中剥离。其信号驱动决策架构与 Sutradhara 的 orchestrator-engine co-design [L2:2601.12967] 和 Scepsy 的 aggregate pipeline scheduling [L2:2604.15186] 形成三层互补：Semantic Router 决定"哪个模型"，Sutradhara 优化"单模型上的 agentic pipeline"，Scepsy 优化"多模型间的 GPU 分配"。

          6. 共识与分歧 #

          共识 #

          共识 1：KV cache 管理是 agentic serving 的核心瓶颈。 从 KVFlow 的 STE 驱逐 [L2:2507.07400]、Continuum 的 TTL [L2:2511.02230]、Concur 的 AIMD [L2:2601.22705]、ThunderAgent 的指数衰减 [L2:2602.13692]、SideQuest 的语义 GC [L2:2602.22603]、DualPath 的双路径加载 [L2:2602.21548]、Helium 的 Templated Radix Tree [L2:2603.16104]、TokenDance 的 collective 去重 [L2:2604.03143]、SAGA 的 WA-LRU [L2:2605.00528]、到 PBKV 的 GraphSAGE 预测 [L2:2605.06472]——10 篇论文从不同角度攻击 GPU HBM 中的 KV cache。SAGA 量化表明 38% 集群时间花在 KV 重建 [L2:2605.00528]。

          共识 2：请求级调度在 agentic workload 下系统性失效。 Concur 证明 request-level admission 可能比无控制更差（某配置 0.72×）[L2:2601.22705]。ThunderAgent 测得 vLLM 的 KV-aware 路由导致 51% 跨节点内存不均衡 [L2:2602.13692]。KVFlow 证明 LRU 在循环 workflow 下产生反向驱逐信号 [L2:2507.07400]。SAGA 的 O(k²c) 退化分析给出了理论解释 [L2:2605.00528]。所有 framework 类论文一致认为需要 agent-level 或 program-level 的调度粒度。

          共识 3：Tool 执行是 agentic 系统的主要延迟源。 CPU-Centric Profiling 测得 35%–88% [L2:2511.00739]，PASTE 复现 35%–61% [L2:2603.18897]，Sutradhara 在 Microsoft 生产 trace 上测得 30%–80% [L2:2601.12967]，Continuum 发现排队延迟占 58.2% [L2:2511.02230]。四篇独立工作一致确认 CPU 侧是主要延迟贡献者。

          共识 4：安全正从附加检查上升为核心系统决策输入。 Semantic Router 将 jailbreak/PII/hallucination 检测作为路由决策的一等信号 [L2:vllm-project-semantic-router]。Claude Code 的 7 层安全管道将安全嵌入 agent runtime 核心循环 [L2:2604.14228]。Agents SDK 的 Guardrails 将安全检查作为并行执行组件 [L2:openai-openai-agents-python]。三个独立系统都将安全从"事后检查"提升为"事前决策输入"。

          分歧 #

          分歧 1：KV cache 应该保留多久？ Continuum 声称 TTL 应由 cost-benefit 模型决定——tool call 在 TTL 内返回则 KV 被保留 [L2:2511.02230]。但 SideQuest 的核心论点是 token 重要性非单调——某些 tool response 在第 t 轮看似无用但到第 t+n 轮可能重新变关键 [L2:2602.22603]。分歧根源：Continuum 的 SWE-Bench/BFCL 是 6–11 轮的短 horizon function-calling agent，SideQuest 的 FRAMES/BrowseComp 是 80+ 轮 120K+ token 的 deep research agent——时间维度对短 horizon 足够，语义维度对长 horizon 不可或缺 [L3:agent]。

          分歧 2：静态 DAG vs 动态预测。 SAGA 用 Agent Execution Graph（DAG + 转移概率）预测 KV 复用 [L2:2605.00528]。PBKV 明确批评"静态 DAG 假设无法处理运行时条件分支和重试循环"——在动态工作流上 PBKV 命中率 69% vs KVFlow 的 39.87% [L2:2605.06472]。但 SAGA 的 AEG 支持 backward retry edges 和 transition probabilities（非纯静态 DAG），在无 framework hints 时仍达 87% accuracy [L2:2605.00528]。分歧根源：两者面向不同的动态性程度——SAGA 面向结构化 coding agent（SWE-bench），PBKV 面向含重试循环的动态工作流。

          分歧 3：Agent Loop vs Structured Graph。 SGH 论证 Agent Loop 有三个结构性缺陷（隐式依赖、无界 recovery、不可审计）[L2:2604.11378]。但 Claude Code 恰恰是一个 Agent Loop 系统，通过 7 层安全管道 + append-only transcript 实现生产级可靠性 [L2:2604.14228]。分歧根源在于对 LLM 判断力的信任程度：SGH 假设 LLM 策略不可检查，Claude Code 假设模型有"good judgment"并用 harness 兜底 [L3:agent]。

          分歧 4：Reactive 优化 vs Proactive 控制。 PASTE 和 Sutradhara 采用 reactive 路线——在 tool 调用发生时通过投机执行或 prompt splitting 来隐藏延迟 [L2:2603.18897] [L2:2601.12967]。Concur、ThunderAgent 和 SAGA 采用 proactive 路线——通过准入控制、pause/restore 和 workflow-atomic 批调度预防资源争夺 [L2:2601.22705] [L2:2602.13692] [L2:2605.00528]。Reactive 假设引擎可被修改（需暴露内部接口），proactive 假设引擎是黑箱（只需使用指标）。

          分歧 5：GPU 侧优化 vs CPU-GPU 协同。 当 CPU 工具执行占 E2E 延迟 88% 时，GPU kernel 优化上限仅 12% [L2:2511.00739]。KV cache 优化攻击 GPU 侧，COMB/MAS 攻击 CPU-GPU 调度 [L2:2511.00739]。GPU 越强（H200/B200），CPU 瓶颈越突出——tool 推理占比从 Sys1 88% 降至 Sys2 77% [L2:2511.00739]。但 SideQuest 证明语义淘汰可使 SGLang 峰值吞吐提升 83.9% [L2:2602.22603]——表明 GPU 侧优化仍有巨大空间。

          7. Open challenges (根本性困难) #

          Challenge 1：Agent KV Cache 管理的信息论极限 #

          为何根本性困难：当前 5 种 KV cache 管理方案（TTL/语义/DAG/GNN/集体复用）分别在各自假设下工作，但缺乏统一的信息论框架来回答"KV cache 的最优淘汰策略的理论极限是什么"。SAGA 达到 Bélády 最优的 1.31× [L2:2605.00528]，但 Bélády 假设 perfect future knowledge，agent workload 的 future knowledge 天然受限于 LLM 决策的非确定性。PBKV 的 Lipschitz 界 $\mathcal{R}(B) \leq \frac{1}{2(1-\gamma)} \sum \epsilon_c^\gamma$ [L2:2605.06472] 是迈向理论极限的第一步，但尚未建立与模型表达力（attention 的 effective context window）之间的关系。

          需要哪些 category 协同：model 层的注意力机制选择（KDA 的 fixed-state 完全消除问题 [L2:2510.26692]，CSA/HCA 的压缩改变问题规模 [L2:deepseek-v4]）决定了 cache 管理的问题空间；agent 层的 workload 可预测性（SAGA 87% AEG accuracy [L2:2605.00528]，PBKV 0.94 1-step accuracy [L2:2605.06472]）决定了可达到的 competitive ratio；framework 层需要标准化的 cache 管理 API 让不同策略可插拔。

          预估难度：3–5 年。需要先建立 agentic workload 的信息论模型。

          Challenge 2：跨层全栈 SLO 保障 #

          为何根本性困难：agentic workload 的端到端延迟由 LLM 推理 + tool 执行 + orchestrator 逻辑三部分组成，三者的延迟分布特性完全不同——LLM 推理可预测，tool 执行变异系数超过 100% [L2:2601.12967]，orchestrator 逻辑依赖 LLM 实时决策。Scepsy 发现 agentic workflow 的绝对延迟抖动巨大（beam search 24–844 次 generator 调用）[L2:2604.15186]。Continuum 的 tool call CDF 分析表明单个 cd 命令最慢 10% 就贡献 94.1% 总延迟 [L2:2511.02230]——传统 TTFT/TPOT SLO 定义在此场景下失效。

          需要哪些 category 协同：framework 层需要 workflow-level SLO 语义；agent 层需暴露 tool 延迟预测信号（Continuum 的 CDF 估计 [L2:2511.02230] 和 SAGA 的 per-tool-type latency model [L2:2605.00528] 提供了初步方案）；model 层需稳定的 function calling 格式以支持 streaming dispatch。SGH 的 bounded termination 证明 [L2:2604.11378] 提供了理论基础，但尚无实验验证。

          预估难度：3–5 年。

          Challenge 3：Agent 安全性的多层联防 #

          为何根本性困难：Claude Code 的 93% permission approval rate 揭示了 approval fatigue [L2:2604.14228]。PASTE 的 side-effect management 依赖用户定义的 policy，602/20000（3%）投机操作被检测为有副作用 [L2:2603.18897]。SGH 的 validation gap：10 节点链的正确率可能仅 $0.95^{10} \approx 0.6$ [L2:2604.11378]。Semantic Router 的安全信号提取准确率尚无系统评估 [L2:vllm-project-semantic-router]。Hermes Agent 完全缺乏形式化安全护栏 [L2:NousResearch-hermes-agent]。

          需要哪些 category 协同：model 层需更强 alignment 以支持 values-over-rules 范式；agent 层需 side-effect 分类和 rollback 协议；framework 层需 sandbox 隔离和 audit trail（Agents SDK 已有 7+ sandbox 后端 [L2:openai-openai-agents-python]）；路由层需高精度安全信号提取 [L2:vllm-project-semantic-router]；code 层需开发者友好的安全 API。

          预估难度：5+ 年。需要模型 alignment、runtime 沙箱、路由安全信号、形式验证四条线交汇。

          Challenge 4：长程 Agent Session 的资源管理 #

          为何根本性困难：K2.6 的 4000+ 工具调用 / 12 小时 session [L2:kimi-k2-6] 使 KV cache 成为长期共享资源。DualPath 分析显示 working set 以 $r^2$ 扩展、总成本以 $r^3$ 增长 [L2:2602.21548]。ThunderAgent 证明 KV offload 到 CPU 因 PCIe 带宽不足而失效 [L2:2602.13692]。SideQuest 在 80+ 轮的 deep research agent 上实现 56–65% token 降幅 [L2:2602.22603]，但 non-completion rate 在极端条件下仍不为零。Concur 的 AIMD 对异构 agent cohort 的公平性未验证 [L2:2601.22705]。

          需要哪些 category 协同：framework 层需 hierarchical memory management；agent 层需暴露预期 session 长度和 tool 调用频率（SAGA 的 AEG [L2:2605.00528]、PBKV 的 GNN predictor [L2:2605.06472] 提供了预测信号）；model 层的 KV 压缩方案直接影响每 agent 的内存足迹——V4 的 CSA+HCA 将 KV 降至 10% [L2:deepseek-v4]，KDA 的 fixed state 消除增长 [L2:2510.26692]。

          预估难度：2–3 年。DualPath、ThunderAgent、SideQuest 已提供部分解决方案，但面向 100 小时级 session 的资源管理仍是未知领域。

          Challenge 5：Multi-Agent KV Cache 的 O(N) 问题 #

          为何根本性困难：当 N 个 agent 共享同一轮输出时，现有系统产生 N 份几乎相同的 KV cache 副本——TokenDance 测得 91%–97% 块级重叠 [L2:2604.03143]。SAGA 的 session-affinity routing 减少跨 worker 迁移但不解决同 worker 内的副本问题 [L2:2605.00528]。GLM 的 prefix caching 利用 Graph-CoT 的前缀共享但限于静态 prompt 部分 [L2:2511.01633]。Pancake 从记忆索引侧统一了跨 agent 查询 [L2:2602.21477]，但 KV cache 与 vector memory 仍然是独立的 HBM 消费者。TokenDance 将复用代价从 O(N) 摊销到 O(1)，但严格绑定同步 All-Gather 模式——异步 multi-agent 场景下的共享对齐更加困难 [L2:2604.03143]。

          预估难度：2–4 年。同步场景已有方案，异步场景仍是 open problem。

          9. 邻接 topic #

          PD-disaggregation：DualPath 的存储带宽瓶颈分析直接建立在 PD 分离架构之上 [L2:2602.21548]。ThunderAgent 已证明 PD 分离在 agent workload 下可能退化 [L2:2602.13692]。SAGA 的 session-affinity routing 部分缓解了 PD 分离下的 KV 迁移问题 [L2:2605.00528]。建议保持两个 topic 独立但交叉引用。

          attention-optimization：DeepSeek-V4 的 CSA+HCA [L2:deepseek-v4]、Kimi K2 的 MLA [L2:2507.20534]、Kimi Linear 的 KDA [L2:2510.26692] 直接影响 agentic serving 的 per-agent 内存足迹。TokenDance 的 collective reuse 深度依赖 RoPE 位置编码机制 [L2:2604.03143]。注意力机制的选择是 agent-system 的上游约束——特别是 KDA 的 O(1) KV 可能从根本上改变 agent cache 管理的问题定义。

          agent-safety：Claude Code 的 93% approval fatigue [L2:2604.14228]、SGH 的 validation gap [L2:2604.11378]、PASTE 的 side-effect management [L2:2603.18897]、Semantic Router 的安全信号 [L2:vllm-project-semantic-router]、以及 Agents SDK 的 Guardrails [L2:openai-openai-agents-python] 均触及安全边界。当安全相关论文积累到 5+ 篇时，建议拆出独立 agent-safety topic。

          RL-training-infra：K2.6 的长程 agentic RL [L2:kimi-k2-6] 和 Hermes Agent 的 Atropos RL 管线 [L2:NousResearch-hermes-agent] 连接了 agent-system 与 RL 训练基础设施。Agent serving 和 RL rollout 共享同一组 KV cache 管理挑战。

          model-routing / AI-gateway：Semantic Router 代表了一个潜在的独立 topic。当前仅 Semantic Router 一个实体直接聚焦此方向 [L2:vllm-project-semantic-router]，但其 Workload-Router-Pool 架构愿景和 Scepsy 的 multi-LLM pipeline [L2:2604.15186] 已呈现出模型集群级路由与调度的技术线索。如果相关实体积累到 3+ 篇，建议拆出。
