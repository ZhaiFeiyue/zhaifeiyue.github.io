Agent System：从请求推理到自主工作流的系统范式跃迁 #

1. 主题缘起 #

2025–2026 年，LLM 的主流使用模式正从单请求问答向多轮自主工具调用剧烈迁移。以 Cursor、Claude Code、Windsurf、Cline 为代表的编程 agent 成为最显著的工作负载形态——它们运行 12+ 小时的连续 session、在单次任务中产生 4000+ 工具调用、消耗 100K–262K tokens 的上下文窗口 [L2:kimi-k2-6] [L2:2604.14228]。这一转变引发了贯穿 model、framework、agent、code 四个领域的系统性重构需求。

应用层驱动的基础设施危机。对 Claude Code 512K 行源码的逆向分析首次揭示了生产级编程 agent 的核心范式——1.6% 决策逻辑 + 98.4% 确定性基础设施 [L2:2604.14228]。极简的 while-loop + AsyncGenerator 架构的成功证明：agent 的核心瓶颈不在"如何让 LLM 决策更好"，而在"如何让基础设施高效支撑 LLM 决策" [L3:2604.14228]。Claude Code 的 5 层渐进式上下文压缩管道（budget reduction → snip → microcompact → context collapse → auto-compact）暴露了编程 agent 对 serving 层的深层需求：prompt cache awareness、超长上下文（200K–1M tokens）、streaming tool dispatch、abort/recovery semantics [L2:2604.14228]。

CPU 侧工具执行被发现是隐藏的主导瓶颈。在 agent 领域，CPU 侧工具执行占端到端延迟高达 88% [L2:2511.00739]，file read/write、shell exec、web search 等 CPU-bound 操作使得 GPU kernel 优化的边际收益上限仅 12% [L2:2511.00739]。投机性工具执行方案将延迟降低 48.5% [L2:2603.18897]，但更深层的问题在于 KV cache 管理——end-of-turn eviction 导致 per-turn queueing delay 累积占总延迟 58.2% [L2:2511.02230]，标准 LRU 在 agent 工作流中命中率仅 27–28% [L2:2605.06472]，启发式剪枝更导致模型崩溃（non-completion rate 60%+）[L2:2602.22603]。

编程 agent 的工作负载特征与现有基础设施的根本错配。当前 serving 系统为 chatbot（短 session、独立请求、LRU-friendly 缓存）优化。编程 agent 则要求：session affinity（12h+ session 跨越数千轮 tool call）、KV persistence（工具返回后 KV 必须保留而非驱逐）、workflow atomicity（整个编程任务而非单次请求是调度原子）、bursty tool calls（文件操作/编译/测试 在 CPU 侧产生极端长尾延迟——cd 命令最慢 10% 占 94.1% 总延迟）[L2:2511.02230]。SAGA 测量显示 38% 的集群执行时间花在重新生成被驱逐的 KV cache 上，GPU 利用率仅 42% [L2:2605.00528]。

Framework 层正经历从请求级调度到程序级编排的范式转变。PD 分离架构的存储带宽瓶颈在 agentic workload 下尤其严重——KV cache 命中率高达 98.7%，prefill 从计算密集退化为 I/O 密集 [L2:2602.21548]。ThunderAgent 发现朴素 PD 分离在 agent 场景下退化（更低并发就触发 thrashing）[L3:2602.13692]。Halo 首次将数据库批查询优化引入 agentic workflow [L2:2509.02121]，GLM 的 multi-agent serving 达到 15.1× 吞吐提升 [L2:2511.01633]。vLLM Semantic Router 通过 20+ 异构信号类型上的布尔表达式树决策实现模型集群间的实时 per-request 智能路由 [L2:vllm-project-semantic-router]。

Model 层的两条架构路线同时为 agent serving 开辟空间。DeepSeek-V4 的 hybrid CSA+HCA attention 将 1M 上下文下 KV cache 压至 V3.2 的 10%，单 token 推理 FLOPs 仅 27% [L2:deepseek-v4]。Kimi K2.6 将单次 rollout 推到 4000+ 工具调用、12–13 小时连续执行 [L2:kimi-k2-6]。Kimi Linear 的 KDA 线性注意力将 decode 吞吐提升 75%、KV cache 减少 75% [L2:2510.26692]——直接降低编程 agent 的 per-session 内存占用。

Code 层提供了可落地的开发范式。OpenAI Agents SDK 和 Hermes Agent 分别代表"声明式编排"和"自我进化闭环"两条工程路线 [L2:openai-openai-agents-python] [L2:NousResearch-hermes-agent]。CMV 的 DAG 版本控制 + 三遍裁剪则代表了一种新兴的客户端 context 管理范式——将 serving 层无法感知的会话级优化下沉到 agent 本地 [L2:2602.22402]。

四个领域的技术演进共同指向：agentic workload 不是 LLM serving 的一种特殊模式，而是需要从模型训练、推理引擎、缓存策略、路由策略、调度策略到应用框架全栈重新设计的新系统范式。编程 agent 作为当前最具规模的 agentic workload，正在反向驱动整个基础设施栈的演进方向。

            2. 5. 技术线交错 #

            编程 Agent 的工作负载特征重塑全栈设计 #

            编程 agent（Cursor、Claude Code、Windsurf、Cline）已成为最具规模的 agentic workload，其特征对基础设施提出了系统性挑战：

            • 超长 session（12+ 小时）：K2.6 的 4000+ 工具调用 / 12–13 小时连续执行 [L2:kimi-k2-6] 对推理基础设施提出了极端的 KV cache 持久化要求。DualPath 分析显示 working set 随 tool call latency 以 $r^2$ 扩展、总成本以 $r^3$ 增长 [L2:2602.21548]。ThunderAgent 证明 KV offload 到 CPU 因 PCIe 带宽不足而失效 [L2:2602.13692] [L3:2602.13692]。
            • Bursty tool calls：文件读写、shell 执行、web 搜索产生极端长尾延迟分布。Continuum 的分析表明 cd 命令最慢 10% 占 94.1% 总延迟 [L2:2511.02230]——这使得标准的 TTFT/TPOT SLO 定义失效。CPU-Centric Profiling 发现 CPU 并行化饱和点在 BS=128 [L2:2511.00739]。
            • Context window 压力（100K–262K tokens）：Claude Code 的 5 层 compaction 管道是对 context 压力的直接回应——从 budget reduction（移除不重要 tool output）到 auto-compact（model-generated summary）的渐进策略 [L2:2604.14228]。SideQuest 在 80+ 轮的 deep research agent 上实现 56–65% token 降幅 [L2:2602.22603]，SAGA 的 session-affinity routing 确保 agent 不被分配到冷 worker [L2:2605.00528]。
            • Prefix caching 需求：编程 agent 的 system prompt + tool schemas 构成大量可缓存前缀。Claude Code 的 compaction 决策与 Anthropic API 的 cache_deleted_input_tokens 返回值深度耦合——compaction 本身需要考虑 prompt cache 的经济学 [L2:2604.14228] [L3:2604.14228]。

            从 Request-Level 到 Workflow-Level Scheduling #

            编程 agent 的模式驱动了调度范式的根本变革：

            • From LRU to workflow-aware KV cache management：KVFlow 的 steps-to-execution [L2:2507.07400] → SAGA 的 Agent Execution Graph [L2:2605.00528] → PBKV 的 GraphSAGE 多步预测 [L2:2605.06472]。核心洞察是 agent workload 的缓存访问模式有结构（DAG/graph），而非 chatbot 的随机/LRU-friendly 模式。
            • From static PD disaggregation to dynamic routing：PPD 发现 append-prefill（Turn 2+）仅造成 2% TPOT 劣化 [L2:2603.13358]，而 ThunderAgent 证明朴素 PD 分离在 agent workload 下导致"单侧 HBM 池变小，更低并发就触发 thrashing" [L2:2602.13692] [L3:2602.13692]。DualPath 的成功恰恰证明朴素 PD 分离确实退化，但优化过的 PD 分离（layerwise streaming + 带宽聚合 + VL QoS 隔离）在超大 MoE 规模可工作 [L2:2602.21548] [L3:2602.13692]。
            • From passive context to active context management：Claude Code 的 5 层 compaction 在 agent 内部主动管理 context [L2:2604.14228]；SideQuest 在 serving 层用 LLM 语义推理主动淘汰 [L2:2602.22603]；CMV 在用户端用 DAG 版本控制实现跨会话 context 复用 [L2:2602.22402]——三者代表了 agent 侧、serving 侧、用户侧三个不同层次的主动 context 管理。
            • 客户端 context 管理作为新调度层：CMV 揭示了一个重要的架构趋势——当 serving 引擎无法感知 agent 内部状态时，agent 应用本身承担一部分"调度"职责。Claude Code 的 compaction 决策本质上是一个客户端调度器（"什么时候压缩、压缩多少"的决策直接影响 serving 层的 KV cache 利用率）[L3:2604.14228]。

            模型能力驱动框架需求 #

            DeepSeek-V4 的 CSA+HCA 将 1M 下 KV cache 压至 V3.2 的 10% [L2:deepseek-v4]——若应用于 agent serving 可直接将 per-agent 的 GPU 内存占用降低一个数量级。Kimi Linear 的 KDA 层以 fixed-size recurrent state（128×128/head）替代累积 KV [L2:2510.26692]，从根本上消除了 KV cache 随 turn 数线性增长的问题。这对 Continuum/SAGA/PBKV 的 cache 管理策略构成"上游问题消失"的挑战——当 KDA 的 O(1) KV 取代 MLA 的 O(n) KV 时，精细的缓存管理策略是否仍有必要 [L3:model]？

            然而模型侧压缩的落地存在时间差。V4 的 CSA+HCA 仅验证文本模态 [L3:model]，K2.6 的 MLA 仍是 O(n)。在模型侧完全解决 KV 问题之前（预估 2–3 年），serving 层的 agent-aware KV 管理仍是必要的过渡方案。

            2025-11 至 2026-05 的 8 个月内，5 篇论文从不同信息源攻击同一问题——agent 多轮推理中的 KV cache 管理 [L3:agent]：

            Agent 运行时约束框架设计 #

            PASTE 的投机执行中间件要求 serving 引擎支持 authoritative vs speculative 任务的优先级隔离 [L2:2603.18897]。SAGA 的 session-affinity batching 要求引擎暴露 per-session KV cache 的跨 worker 迁移接口 [L2:2605.00528]。SideQuest 的辅助线程需要 serving 框架支持 KV cache 的 fork-and-share 机制 [L3:2602.22603]。这些需求推动 vLLM/SGLang 从"单请求引擎"演进为"agent-aware runtime"。

            SDK 设计反映架构共识 #

            OpenAI Agents SDK 的 Runner 循环 + Guardrails + Handoffs 结构 [L2:openai-openai-agents-python] 与 Claude Code 的 queryLoop + Permission System + AgentTool 设计 [L2:2604.14228] 在抽象层级上高度同构，表明生产级 agent 的架构模式已收敛：中央循环 + 安全网关 + 工具分发 + 子代理委派 [L3:code]。Hermes Agent 的 skill 自动创建和持久记忆则映射到 Claude Code 的 CLAUDE.md 4 级层次和 MCP 扩展机制 [L2:NousResearch-hermes-agent] [L2:2604.14228]。两条路线的核心对称性在于：OpenAI SDK 追求 可控性（Guardrails + HitL + RunState 序列化）但无学习能力，Hermes 追求 自治性（Skill patch + 持久记忆 + Atropos RL）但无安全护栏——尚无框架同时实现两者 [L3:code]。

            路由层作为独立系统层 #

            vLLM Semantic Router 在 agent stack 中引入了全新的基础设施级路由层——位于应用编排器之下、推理引擎之上 [L2:vllm-project-semantic-router]。这一层将"模型选择"从应用代码中剥离并下沉到基础设施，类似 Envoy 将负载均衡从应用代码中剥离。其信号驱动决策架构与 Sutradhara 的 orchestrator-engine co-design [L2:2601.12967] 和 Scepsy 的 aggregate pipeline scheduling [L2:2604.15186] 形成三层互补：Semantic Router 决定"哪个模型"，Sutradhara 优化"单模型上的 agentic pipeline"，Scepsy 优化"多模型间的 GPU 分配"。

            6. 共识与分歧 #

            共识 #

            共识 1：KV cache 管理是 agentic serving 的核心瓶颈。从 KVFlow 的 STE 驱逐 [L2:2507.07400]、Continuum 的 TTL [L2:2511.02230]、Concur 的 AIMD [L2:2601.22705]、ThunderAgent 的指数衰减 [L2:2602.13692]、SideQuest 的语义 GC [L2:2602.22603]、DualPath 的双路径加载 [L2:2602.21548]、Helium 的 Templated Radix Tree [L2:2603.16104]、TokenDance 的 collective 去重 [L2:2604.03143]、SAGA 的 WA-LRU [L2:2605.00528]、到 PBKV 的 GraphSAGE 预测 [L2:2605.06472]——10+ 篇论文从不同角度攻击 GPU HBM 中的 KV cache。SAGA 量化表明 38% 集群时间花在 KV 重建 [L2:2605.00528]。

            共识 2：请求级调度在 agentic workload 下系统性失效。Concur 证明 request-level admission 可能比无控制更差（某配置 0.72×）[L2:2601.22705]。ThunderAgent 测得 vLLM 的 KV-aware 路由导致 51% 跨节点内存不均衡 [L2:2602.13692]。KVFlow 证明 LRU 在循环 workflow 下产生反向驱逐信号 [L2:2507.07400]。SAGA 的 O(k²c) 退化分析给出了理论解释 [L2:2605.00528]。Halo、Helium、ThunderAgent、SAGA 一致认为需要 agent-level 或 program-level 的调度粒度 [L3:framework]。

            共识 3：Tool 执行是 agentic 系统的主要延迟源。CPU-Centric Profiling 测得 35%–88% [L2:2511.00739]，PASTE 复现 35%–61% [L2:2603.18897]，Sutradhara 在 Microsoft 生产 trace 上测得 30%–80% [L2:2601.12967]，Continuum 发现排队延迟占 58.2% [L2:2511.02230]。四篇独立工作一致确认 CPU 侧是主要延迟贡献者——GPU 越强（H200/B200），CPU 瓶颈越突出（Toolformer 推理占比从 Sys1 88% 降至 Sys2 77%）[L2:2511.00739]。

            共识 4：安全正从附加检查上升为核心系统决策输入。Semantic Router 将 jailbreak/PII/hallucination 检测作为路由决策的一等信号 [L2:vllm-project-semantic-router]。Claude Code 的 7 层安全管道将安全嵌入 agent runtime 核心循环——93% 的 approval rate 证明交互式确认不可靠，必须用 defense-in-depth 替代人类警觉性 [L2:2604.14228] [L3:2604.14228]。Agents SDK 的 Guardrails 将安全检查作为并行执行组件 [L2:openai-openai-agents-python]。三个独立系统都将安全从"事后检查"提升为"事前决策输入"。

            共识 5：编程 agent 的架构模式趋于收敛。Claude Code 的 "while-loop + harness"、OpenAI SDK 的 "Runner + Guardrails + Handoffs"、Hermes 的 "Agent loop + Skill + Memory" 在核心结构上同构——都采用中央循环 + 安全网关 + 工具分发 + 子代理委派 [L2:2604.14228] [L2:openai-openai-agents-python] [L2:NousResearch-hermes-agent] [L3:code]。SGH 的 scheduler 模型将此形式化为 $|\mathcal{U}|=1$ 的单就绪单元调度器 [L2:2604.11378]。

            分歧 #

            分歧 1：KV cache 应该保留多久——时间维度 vs 语义维度。Continuum 声称 TTL 应由 cost-benefit 模型决定——tool call 在 TTL 内返回则 KV 被保留 [L2:2511.02230]。SideQuest 的核心论点是 token 重要性非单调——某些 tool response 在第 t 轮看似无用但到第 t+n 轮可能重新变关键 [L2:2602.22603]。根源：实验 workload 不同。Continuum 的 SWE-Bench/BFCL 是 6–11 轮短 horizon function-calling agent，SideQuest 的 FRAMES/BrowseComp 是 80+ 轮 120K+ token 的 deep research agent——时间维度对短 horizon 足够，语义维度对长 horizon 不可或缺 [L3:agent]。SideQuest 在 coding agent benchmark（SWE-bench）上未评估 [L3:2602.22603]——这一关键盲区使两者无法直接比较。

            分歧 2：静态 DAG vs 动态预测。SAGA 用 Agent Execution Graph（DAG + 转移概率）预测 KV 复用 [L2:2605.00528]。PBKV 明确批评"静态 DAG 假设无法处理运行时条件分支和重试循环"——在动态工作流上 PBKV 命中率 69% vs KVFlow 的 39.87% [L2:2605.06472]。根源：SAGA 的 AEG 支持 backward retry edges 和 transition probabilities（非纯静态 DAG），在无 framework hints 时仍达 87% accuracy [L2:2605.00528]。PBKV 的批评更准确地针对 KVFlow 的纯静态距离，而非 SAGA 的概率化 DAG [L3:2605.06472]。两者面向不同的动态性程度——SAGA 面向结构化 coding agent（SWE-bench），PBKV 面向含重试循环的高动态工作流。

            分歧 3：Agent Loop vs Structured Graph。SGH 论证 Agent Loop 有三个结构性缺陷（隐式依赖、无界 recovery、不可审计）[L2:2604.11378]。但 Claude Code 恰恰是一个 Agent Loop 系统，通过 7 层安全管道 + append-only transcript 实现生产级可靠性 [L2:2604.14228]。根源：对 LLM 判断力的信任程度不同。SGH 假设 LLM 策略"不可检查"（non-det $\mathcal{P}$）；Claude Code 假设 frontier 模型有"good judgment"并用 harness 兜底 [L3:agent]。关键的可攻击面在于：Claude Code 的 "minimal scaffolding" 范式可能是 Claude-specific——如果 backbone 换为 Llama-70B 或 Qwen-72B，minimal scaffolding 是否仍然可行 [L3:2604.14228]？

            分歧 4：PD 分离在 agent workload 下是否可行。ThunderAgent 实验证明朴素 PD 分离在 agent workload 下退化——"单侧 HBM 池变小，更低并发就触发 thrashing" [L2:2602.13692]。然而 DualPath 的整个方案正是在 PD 分离架构内通过聚合 DE SNIC 带宽来解决 agent workload 的存储 I/O 瓶颈，在 1152 GPU 上实现近线性扩展 [L2:2602.21548]。根源：ThunderAgent 评测的是 vLLM 标准 PD 实现（无 QoS 隔离、无带宽聚合），DualPath 的方案带有 layerwise streaming + CNIC-centric flow isolation + InfiniBand VL QoS。两者的结论在各自硬件假设下都正确：朴素 PD 分离对 agent 退化，优化 PD 分离在超大规模可行 [L3:2602.13692]。PPD 提供了第三种视角——Turn 2+ 的 append-prefill 干扰仅 2%，可以安全 co-locate [L2:2603.13358]——暗示小模型场景下 PD 分离本身可能不必要。

            分歧 5：GPU 侧优化 vs CPU-GPU 协同。当 CPU 工具执行占 E2E 延迟 88% 时，GPU kernel 优化上限仅 12% [L2:2511.00739]。KV cache 优化攻击 GPU 侧，COMB/MAS 攻击 CPU-GPU 调度 [L2:2511.00739]。但 SideQuest 证明语义淘汰可使 SGLang 峰值吞吐提升 83.9% [L2:2602.22603]——表明 GPU 侧优化仍有巨大空间。根源：CPU 瓶颈和 GPU 瓶颈在不同并发度下交替成为主导约束——低并发 CPU 主导，高并发 GPU（内存）主导。

            分歧 6：淘汰决策的开销——极简 vs 重模型。PBKV 的预测器仅 ~350K 参数、1.18ms 推理 [L3:2605.06472]，SAGA 的 coordinator 开销仅 12.3ms + 3.1ms [L3:2602.22603]。而 SideQuest 的辅助线程每次生成 ~110-140 token 的管理推理——GPU 开销可能高 1-2 个数量级 [L3:2602.22603]。SideQuest 用"精度"换"开销"（non-completion rate 近零 vs heuristic 60%+），PBKV/SAGA 用"开销"换"覆盖度"（跨工作流全局操作）。

            7. Open challenges (根本性困难) #

            Challenge 1：Agent KV Cache 管理的信息论极限 #

            为何根本性困难：当前 5 种方案分别在各自假设下工作，但缺乏统一的信息论框架来回答"KV cache 最优淘汰策略的理论极限是什么"。SAGA 达到 Bélády 最优的 1.31× [L2:2605.00528]，但 Bélády 假设 perfect future knowledge，agent workload 的 future knowledge 天然受限于 LLM 决策的非确定性。PBKV 首次将 algorithms-with-predictions 框架引入此领域，其 Lipschitz 遗憾界 $\mathcal{R}(B) \leq \frac{1}{2(1-\gamma)} \sum \epsilon_c^\gamma$ 将预测误差与性能退化定量关联 [L2:2605.06472] [L3:2605.06472]——但尚未建立与模型表达力（attention 的 effective context window）之间的关系。

            需要哪些 category 协同：model 层的注意力机制选择（KDA 的 fixed-state 完全消除问题 [L2:2510.26692]，CSA/HCA 的压缩改变问题规模 [L2:deepseek-v4]）决定了 cache 管理的问题空间；agent 层的 workload 可预测性（SAGA 87% AEG accuracy [L2:2605.00528]，PBKV 0.94 1-step accuracy [L2:2605.06472]）决定了可达到的 competitive ratio；framework 层需标准化 cache 管理 API 让不同策略可插拔。

            预估难度：3–5 年。需先建立 agentic workload 的信息论模型。

            Challenge 2：跨层 Context 管理的一致性问题 #

            为何根本性困难：编程 agent 的 context 管理现在分散在三个独立层：(1) 应用层——Claude Code 的 5 层 compaction [L2:2604.14228]、CMV 的 DAG 裁剪 [L2:2602.22402]；(2) Serving 层——Continuum 的 KV TTL [L2:2511.02230]、SAGA 的 WA-LRU [L2:2605.00528]、SideQuest 的语义淘汰 [L2:2602.22603]；(3) 模型层——V4 CSA/HCA 的物理压缩 [L2:deepseek-v4]、KDA 的 recurrent state [L2:2510.26692]。三层独立优化可能互相冲突：agent 做 auto-compact 后 KV cache 前缀失效，serving 层的 prefix cache 策略被动失效 [L3:2604.14228]。如果 SideQuest 认为某 cursor 仍有语义价值，但 Continuum 的 TTL 已过期并驱逐了整个 session 的 KV cache，语义判断就被浪费了 [L3:2602.22603]。

            需要哪些 category 协同：需要 agent-serving 双向信号通道——agent 的 compaction 决策应考虑 serving 层的 cache 状态（如果当前 prefix 有 cache hit，推迟 compact；如果 cache 已被 evict，aggressively compact）。模型层的压缩方案决定了 serving 层 KV 的绝对大小和格式。

            预估难度：2–3 年。技术上直接可做（定义跨层 API），难点在标准化——需要 vLLM/SGLang/TensorRT 等框架协同定义接口。

            Challenge 3：跨层全栈 SLO 保障 #

            为何根本性困难：agentic workload 的端到端延迟由 LLM 推理 + tool 执行 + orchestrator 逻辑三部分组成，三者的延迟分布特性完全不同——LLM 推理可预测，tool 执行变异系数超过 100% [L2:2601.12967]，orchestrator 逻辑依赖 LLM 实时决策。Scepsy 发现 agentic workflow 的绝对延迟抖动巨大（beam search 24–844 次 generator 调用）[L2:2604.15186]。传统 TTFT/TPOT SLO 定义在编程 agent 场景下失效——"完成一个 SWE-bench 任务的延迟"无法分解为单请求 SLO 的简单聚合。

            需要哪些 category 协同：framework 层需要 workflow-level SLO 语义（HexGen-Flow 的 SLO 预算传播是最接近的尝试 [L2:2505.05286] [L3:2505.05286]）；agent 层需暴露 tool 延迟预测信号；model 层需稳定的 function calling 格式以支持 streaming dispatch。SGH 的 bounded termination 证明 [L2:2604.11378] 提供了理论基础但无实验验证。

            预估难度：3–5 年。

            Challenge 4：Agent 安全性的多层联防 #

            为何根本性困难：Claude Code 的 93% permission approval rate 揭示了 approval fatigue——交互式确认作为安全机制已经失效 [L2:2604.14228] [L3:2604.14228]。然而替代解释也存在：93% 批准率可能反映模型决策本身已非常准确，用户 approve 是理性而非疲劳（论文未提供 approval rate 随 session 时长递增的时序数据）[L3:2604.14228]。PASTE 的 side-effect management 依赖用户定义的 policy，3% 投机操作被检测为有副作用 [L2:2603.18897]。SGH 的 validation gap：10 节点链的正确率可能仅 $0.95^{10} \approx 0.6$ [L2:2604.11378]。Hermes Agent 完全缺乏形式化安全护栏 [L2:NousResearch-hermes-agent] [L3:code]。

            需要哪些 category 协同：model 层需更强 alignment 以支持 values-over-rules 范式（但这一范式可能是 Claude-specific [L3:2604.14228]）；agent 层需 side-effect 分类和 rollback 协议；framework 层需 sandbox 隔离和 audit trail（Agents SDK 已有 7+ sandbox 后端 [L2:openai-openai-agents-python]）；路由层需高精度安全信号提取 [L2:vllm-project-semantic-router]；code 层需开发者友好的安全 API。

            预估难度：5+ 年。需要模型 alignment、runtime 沙箱、路由安全信号、形式验证四条线交汇。

            Challenge 5：长程 Agent Session 的资源管理 #

            为何根本性困难：K2.6 的 4000+ 工具调用 / 12 小时 session [L2:kimi-k2-6] 使 KV cache 成为长期共享资源。DualPath 分析显示 working set 以 $r^2$ 扩展、总成本以 $r^3$ 增长 [L2:2602.21548]。ThunderAgent 证明 KV offload 到 CPU 因 PCIe 带宽不足而失效 [L2:2602.13692]——这意味着 GPU HBM 是不可逃逸的约束。SideQuest 在 80+ 轮的 deep research agent 上实现 56–65% token 降幅 [L2:2602.22603]，但其辅助线程的 GPU 开销在高并发场景可能成为新瓶颈 [L3:2602.22603]。面向 100 小时级 session 的资源管理仍是未知领域。

            需要哪些 category 协同：framework 层需 hierarchical memory management（SAGA 的 AEG [L2:2605.00528] 和 PBKV 的 GNN predictor [L2:2605.06472] 提供了预测信号，但仅验证短 session）；model 层的 KV 压缩方案直接影响每 agent 的内存足迹——V4 的 CSA+HCA 将 KV 降至 10% [L2:deepseek-v4]，KDA 的 fixed state 消除增长 [L2:2510.26692]；agent 层的客户端 compaction（Claude Code 5 层管道 [L2:2604.14228]）与 serving 层的 KV 管理需要协同而非独立运作。

            预估难度：2–3 年（短 horizon 已有方案），但面向 100h+ session 为 unknown territory。

            Challenge 6：Multi-Agent KV Cache 的 O(N) 问题 #

            为何根本性困难：当 N 个 agent 共享同一轮输出时，现有系统产生 N 份几乎相同的 KV cache 副本——TokenDance 测得 91%–97% 块级重叠 [L2:2604.03143]。SAGA 的 session-affinity routing 减少跨 worker 迁移但不解决同 worker 内的副本问题 [L2:2605.00528]。GLM 的 prefix caching 利用 Graph-CoT 的前缀共享但限于静态 prompt 部分 [L2:2511.01633]。TokenDance 将复用代价从 O(N) 摊销到 O(1)，但严格绑定同步 All-Gather 模式——异步 multi-agent 场景下的共享对齐更加困难 [L2:2604.03143]。Pancake 从记忆索引侧统一了跨 agent 查询 [L2:2602.21477]，但 KV cache 与 vector memory 仍然是独立的 HBM 消费者。

            预估难度：2–4 年。同步场景已有方案，异步场景仍是 open problem。
