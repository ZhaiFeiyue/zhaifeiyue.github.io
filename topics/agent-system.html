Agent System：从请求推理到自主工作流的系统范式跃迁 #

1. 主题缘起 #

2025–2026 年，LLM 的主流使用模式正从单请求问答转向多轮自主工具调用。这一转变引发了贯穿 model、framework、agent、code 四个领域的系统性重构需求。

在 agent 领域，CPU 侧工具执行被发现占端到端延迟高达 88% [L2:2511.00739]，而投机性工具执行将延迟降低 48.5% [L2:2603.18897]，表明 agent serving 的瓶颈已从 GPU 推理转向 tool dispatch 和 CPU-GPU 协同。同时，对 Claude Code 512K 行源码的逆向分析揭示了生产级 agent 的核心范式——1.6% 决策逻辑 + 98.4% 确定性基础设施 [L2:2604.14228]。

在 framework 领域，serving 系统从请求级调度演进到 program-level 调度。KV cache 在 agentic 工作负载下的 middle-phase thrashing 导致 49% 延迟被浪费在重计算上 [L2:2601.22705]，而 PD 分离架构的存储带宽瓶颈在 agentic workload 下尤其严重——KV cache 命中率高达 98.7%，prefill 从计算密集退化为 I/O 密集 [L2:2602.21548]。与此同时，异构模型集群（本地模型、私有模型、前沿模型混部）的兴起催生了一个全新的系统层——vLLM Semantic Router 作为 Envoy ExtProc gRPC sidecar，通过 20+ 异构信号类型上的布尔表达式树决策，实现模型集群间的实时 per-request 智能路由 [L2:vllm-project-semantic-router]，将 jailbreak/PII/hallucination 检测作为路由决策的一等输入 [L3:vllm-project-semantic-router]。

在 model 领域，1T 级 MoE 模型三代复用同一主干（K2→K2.5→K2.6），工程杠杆完全转向 agentic RL 和 orchestration——K2.6 将单次 rollout 推到 4000+ 工具调用、12–13 小时连续执行 [L2:kimi-k2-6]。

在 code 领域，OpenAI Agents SDK 和 Hermes Agent 分别代表了"声明式编排"和"自我进化闭环"两条工程路线，为 agent 系统提供了可落地的开发范式 [L2:openai-openai-agents-python] [L2:NousResearch-hermes-agent]。

这四个领域的技术演进共同指向一个核心命题：agentic workload 不是 LLM serving 的一种特殊模式，而是一个需要从模型训练、推理引擎、路由策略、调度策略到应用框架全栈重新设计的新系统范式。

          2. 5. 技术线交错 #

          模型能力驱动框架需求 #

          Kimi K2.6 的 4000+ 工具调用 / 12–13 小时连续执行 [L2:kimi-k2-6] 对推理基础设施提出了极端要求：KV cache 随 agent 步数单调增长（157 轮 × 429 token/轮 ≈ 67K tokens average）[L2:2602.21548]，存储 I/O 而非计算成为瓶颈。DualPath 的双路径 KV 加载正是为应对这种 cache-compute ratio 高达 22 GB/PFLOP 的 agentic workload 而设计 [L2:2602.21548]。同时，K2 系列的 MLA 注意力将每 token KV cache 压到 576 floats [L2:2507.20534]，这一架构选择直接影响 serving 系统的内存管理策略。

          Agent 运行时约束 framework 设计 #

          Claude Code 的 5 层 context compaction pipeline（budget reduction → snip → microcompact → context collapse → auto-compact）[L2:2604.14228] 揭示了生产级 agent 对 serving 系统的深层需求：prompt cache awareness、very long context support（200K–1M tokens）、streaming tool dispatch、abort/recovery semantics。PASTE 的投机执行中间件要求 serving 引擎支持 authoritative vs speculative 任务的优先级隔离 [L2:2603.18897]。这些需求推动 vLLM/SGLang 从"单请求引擎"演进为"agent-aware runtime"。

          工具执行瓶颈重塑调度策略 #

          CPU-Centric Profiling 发现的 CPU 并行化饱和点（BS=128 时 throughput gain ratio ≈ 1）[L2:2511.00739] 直接影响了 Concur 和 ThunderAgent 的设计——两者都把 agent 的 tool 执行期（Acting 状态）视为一等调度信号 [L2:2601.22705] [L2:2602.13692]。ThunderAgent 更进一步，将 tool 资源（Docker sandbox、端口）的生命周期管理与 LLM 调度统一 [L2:2602.13692]。

          SDK 设计反映架构共识 #

          OpenAI Agents SDK 的 Runner 循环 + Guardrails + Handoffs 结构 [L2:openai-openai-agents-python] 与 Claude Code 的 queryLoop + Permission System + AgentTool 设计 [L2:2604.14228] 在抽象层级上高度同构，表明生产级 agent 的架构模式已经趋于收敛：中央循环 + 安全网关 + 工具分发 + 子代理委派。Hermes Agent 的 skill 自动创建和持久记忆则映射到 Claude Code 的 CLAUDE.md 4 级层次和 MCP 扩展机制 [L2:NousResearch-hermes-agent] [L2:2604.14228]。

          路由层作为独立系统层 #

          vLLM Semantic Router 在 agent stack 中引入了一个全新的基础设施级路由层——位于应用编排器（Agents SDK、Claude Code、Hermes）之下、推理引擎（vLLM、SGLang）之上 [L3:vllm-project-semantic-router]。这一层将"模型选择"从应用代码中剥离并下沉到基础设施，类似 Envoy 将负载均衡从应用代码中剥离 [L2:vllm-project-semantic-router]。其信号驱动决策架构与 Sutradhara 的 orchestrator-engine co-design [L2:2601.12967] 和 Scepsy 的 aggregate pipeline scheduling [L2:2604.15186] 形成三层互补：Semantic Router 决定"哪个模型"，Sutradhara 优化"单模型上的 agentic pipeline"，Scepsy 优化"多模型间的 GPU 分配" [L3:vllm-project-semantic-router]。值得注意的是，Semantic Router 和 Claude Code 共享"infrastructure-heavy, decision-logic-light"的设计哲学——Claude Code 中 1.6% 是决策逻辑 [L2:2604.14228]，Semantic Router 中 DecisionEngine 的表达式树求值是极简逻辑核心，周围包裹了 Envoy 集成、K8s CRD、ML 推理 FFI 等大量基础设施 [L2:vllm-project-semantic-router]。

          6. 共识与分歧 #

          共识 #

          共识 1：Tool 执行是 agentic 系统的主要延迟源。 CPU-Centric Profiling 测得 tool 执行占 E2E 延迟 35%–88% [L2:2511.00739]，PASTE 在 3 种 agent × 3 种 benchmark 上复现了 35%–61% 的占比 [L2:2603.18897]，Sutradhara 在 Microsoft 生产 trace 上测得 30%–80% [L2:2601.12967]。三篇独立工作在不同系统和工作负载上得出一致结论。

          共识 2：请求级调度在 agentic workload 下系统性失效。 Concur 证明 request-level admission control 可能比无控制更差（某配置 0.72×）[L2:2601.22705]。ThunderAgent 测得 vLLM 的 KV-aware 路由导致 51% 跨节点内存不均衡 [L2:2602.13692]。KVFlow 证明 LRU 在循环 workflow 下产生反向驱逐信号 [L2:2507.07400]。所有 framework 类论文一致认为需要 agent-level 或 program-level 的调度粒度。

          共识 3：KV cache 管理是 agentic serving 的核心资源轴。 从 KVFlow 的 STE 驱逐 [L2:2507.07400]、Concur 的 AIMD 准入 [L2:2601.22705]、ThunderAgent 的指数衰减 [L2:2602.13692]、DualPath 的双路径加载 [L2:2602.21548]、到 TokenDance 的 collective 去重 [L2:2604.03143]，5 篇 framework 论文从不同角度攻击同一资源——GPU HBM 中的 KV cache。

          共识 4：安全正在从附加检查上升为核心系统决策输入。 Semantic Router 将 jailbreak/PII/hallucination 检测作为路由决策的一等信号类型——安全不是路由之后的 filter，而是决定请求发往哪个模型的输入维度 [L2:vllm-project-semantic-router] [L3:vllm-project-semantic-router]。Claude Code 的 7 层安全管道将安全嵌入 agent runtime 的核心循环 [L2:2604.14228]。Agents SDK 的 Guardrails 将安全检查作为 agent 框架的并行执行组件 [L2:openai-openai-agents-python]。三个独立系统——路由层、agent runtime、SDK 框架——都将安全从"事后检查"提升为"事前决策输入"。

          分歧 #

          分歧 1：Agent Loop vs Structured Graph。 SGH 论证 Agent Loop 有三个结构性缺陷（隐式依赖、无界 recovery、不可审计），认为只有显式 DAG 才能解决 [L2:2604.11378]。但 Claude Code 恰恰是一个 Agent Loop 系统，通过 7 层安全管道 + append-only transcript 实现了生产级可靠性 [L2:2604.14228]。分歧根源在于对 LLM 判断力的信任程度：SGH 假设 LLM 策略 $\mathcal{P}$ 不可检查 [L2:2604.11378]，Claude Code 假设模型有 "good judgment" 并用 harness 兜底 [L2:2604.14228]。

          分歧 2：Reactive 优化 vs Proactive 控制。 PASTE 和 Sutradhara 采用 reactive 路线——在 tool 调用发生时通过投机执行或 prompt splitting 来隐藏延迟 [L2:2603.18897] [L2:2601.12967]。Concur 和 ThunderAgent 采用 proactive 路线——通过准入控制和 pause/restore 预防资源争夺 [L2:2601.22705] [L2:2602.13692]。两条路线并非互斥，但在实现优先级上反映了不同的系统假设：reactive 假设 serving 引擎可被修改（需暴露 streaming callback、partial prefill 等内部接口），proactive 假设引擎是黑箱（只需 usage/hit-rate 指标）。

          分歧 3：CPU 瓶颈的持久性。 CPU-Centric Profiling 的结论强绑定于当前工具的实现（Python MP/MT、CPU-based FAISS flat search）[L2:2511.00739]。如果工具被 GPU 加速或被 LLM 替代（直接回答而非 RAG 检索），CPU 瓶颈可能消失 [L2:2511.00739]。但 PASTE 和 Sutradhara 的跨 benchmark 复现 [L2:2603.18897] [L2:2601.12967] 表明这一结论在 2025–2026 年窗口内有效。

          分歧 4：多模型协同的路由粒度。 Semantic Router 做逐请求的 signal-driven 路由决策，假设路由在单个请求级别是可分离的——20+ 信号提取 + 表达式树求值足以选出最优模型 [L2:vllm-project-semantic-router]。Helium 做 batch workflow 级别的查询计划优化，假设 batch 内的跨请求前缀复用是核心优化维度 [L2:2603.16104]。Scepsy 做 pipeline 级别的 GPU 资源分配，假设 LLM 的相对份额稳定性允许 ahead-of-time 分配 [L2:2604.15186]。三者代表了多模型协同的三种抽象层次 [L3:vllm-project-semantic-router]，分歧根源在于对工作负载可预测性的假设不同：SR 假设请求语义是可分析的，Helium 假设 workflow 结构是可解析的，Scepsy 假设 pipeline 行为是统计可稳定的。

          7. Open challenges (根本性困难) #

          Challenge 1：跨层全栈 SLO 保障 #

          为何根本性困难：agentic workload 的端到端延迟由 LLM 推理 + tool 执行 + orchestrator 逻辑三部分组成，三者的延迟分布特性完全不同——LLM 推理是计算密集且可预测的，tool 执行变异系数超过 100% [L2:2601.12967]，orchestrator 逻辑依赖 LLM 实时决策。没有任何单层优化能提供端到端 SLO 保障。Scepsy 发现 agentic workflow 的绝对延迟抖动巨大（beam search 24–844 次 generator 调用），使传统 TTFT/TPOT SLO 定义失效 [L2:2604.15186]。

          需要哪些 category 协同：framework 层需要定义 workflow-level SLO 语义（非请求级）；agent 层需要向 framework 暴露 tool 延迟预测信号；model 层需要稳定的 function calling 格式以支持 streaming dispatch [L2:2601.12967]。SGH 的 bounded termination 证明（$\sum_v \tau_v(b_v+1)$）[L2:2604.11378] 提供了理论基础，但尚无实验验证。

          预估难度：3–5 年。需要先建立 agentic SLO 的行业共识。

          Challenge 2：Agent 安全性的多层联防 #

          为何根本性困难：Claude Code 的 93% permission approval rate 揭示了 approval fatigue——交互式安全确认在大规模使用中不可靠 [L2:2604.14228]。安全不能依赖人类警觉性，但也不能完全自动化——PASTE 的 side-effect management 依赖用户定义的 policy，602/20000（3%）投机操作被检测为有副作用 [L2:2603.18897]。SGH 的 contract validation 存在 validation gap：10 个节点链的 semantic validation 正确率可能仅 $0.95^{10} \approx 0.6$ [L2:2604.11378]。Semantic Router 将安全检测作为路由信号（jailbreak/PII/hallucination）[L2:vllm-project-semantic-router]，提供了基础设施级的安全拦截层，但其信号提取的准确率尚无系统评估 [L3:vllm-project-semantic-router]。

          需要哪些 category 协同：model 层需要更强的 alignment 以支持 values-over-rules 范式 [L2:2604.14228]；agent 层需要 side-effect 分类和 rollback 协议；framework 层需要 sandbox 隔离和 audit trail；路由层需要高精度的安全信号提取 [L2:vllm-project-semantic-router]；code 层需要开发者友好的安全 API [L2:openai-openai-agents-python]。

          预估难度：5+ 年。需要模型 alignment、runtime 沙箱、路由安全信号、形式验证四条线交汇。

          Challenge 3：长程 Agent Session 的资源管理 #

          为何根本性困难：K2.6 的 4000+ 工具调用 / 12 小时 session [L2:kimi-k2-6] 使 KV cache 和 tool 资源成为长期共享资源而非短期消耗品。DualPath 的分析显示 working set 随 tool call latency 以 $r^2$ 扩展、总成本以 $r^3$ 增长 [L2:2602.21548]。ThunderAgent 证明 KV offload 到 CPU 因 PCIe 带宽不足而失效 [L2:2602.13692]，PD 分离在 agent workload 下因 HBM 池缩小而更早触发 thrashing [L2:2602.13692]。Concur 的 AIMD 在固定 hyperparameter 下工作，但对异构 agent cohort（短/长 agent 混合）的公平性未验证 [L2:2601.22705]。

          需要哪些 category 协同：framework 层需要 hierarchical memory management（GPU HBM → CPU DRAM → SSD → distributed storage）；agent 层需要向 framework 暴露预期 session 长度和 tool 调用频率；model 层的 KV cache 压缩方案（MLA、KDA）直接影响每 agent 的内存足迹 [L2:2507.20534]。

          预估难度：2–3 年。DualPath 和 ThunderAgent 已提供部分解决方案，但面向 100 小时级 session 的资源管理仍是未知领域。

          Challenge 4：Multi-Agent KV Cache 的 O(N) 问题 #

          为何根本性困难：当 N 个 agent 共享同一轮输出时，现有系统产生 N 份几乎相同的 KV cache 副本——TokenDance 测得 91%–97% 块级重叠 [L2:2604.03143]。Position-Independent Caching（PIC）虽能复用内容相同但位置不同的 KV，但 RoPE 旋转和 key-difference 分析被重复执行 N 次 [L2:2604.03143]。TokenDance 将复用代价从 O(N) 摊销到 O(1)，但严格绑定同步 All-Gather 通信模式——异步 multi-agent 场景下共享块的对齐更加困难。

          需要哪些 category 协同：model 层的注意力机制选择（RoPE vs NoPE vs ALiBi）决定了 PIC 的实现复杂度 [L2:2604.03143]；framework 层需要 round-aware 或 session-aware 的 KV 管理接口；agent 层需要标准化的 multi-agent 通信模式 [L2:2604.14228]。

          预估难度：2–4 年。同步场景已有 TokenDance 方案，异步场景仍是 open problem。

          Challenge 5：路由决策与 serving 状态的协同优化 #

          为何根本性困难：Semantic Router 的路由决策基于请求语义信号（jailbreak、domain、complexity 等 20+ 信号类型），但不感知目标模型实例的运行时状态——KV cache 负载、排队深度、GPU 利用率均不在其信号词汇表中 [L3:vllm-project-semantic-router]。Sutradhara 的 5 个 co-design API 可以暴露引擎内部状态 [L2:2601.12967]，DualPath 的 cache-compute ratio 分析 [L2:2602.21548] 和 Concur 的 AIMD 反馈 [L2:2601.22705] 提供了局部状态信号，但如何将这些信号标准化并输入路由层尚无方案。更根本的困难在于：路由决策需要在毫秒级完成 [L3:vllm-project-semantic-router]，而 serving 状态的采集和传播存在固有延迟——当路由器读到"实例 A 空闲"时，该实例可能已被后续请求占满。

          需要哪些 category 协同：framework 层需要标准化的引擎状态暴露接口（KV 压力、排队深度、TP 利用率）；路由层需要新增 engine-state 信号类型并整合到布尔表达式树中 [L3:vllm-project-semantic-router]；agent 层需要传递工作流上下文（session 长度、tool 调用频率、context 压缩状态）以支持 context-aware routing [L3:vllm-project-semantic-router]。

          预估难度：2–3 年。需要 serving 引擎和路由层之间的 API 标准化，Sutradhara 的 co-design 范式提供了方向但尚未扩展到多模型路由场景。

          9. 邻接 topic #

          PD-disaggregation：DualPath 的存储带宽瓶颈分析直接建立在 PD 分离架构之上 [L2:2602.21548]，agent-system 的 KV cache 管理与 PD 分离的 KV 传输路径高度交织。ThunderAgent 已证明 PD 分离在 agent workload 下可能退化 [L2:2602.13692]。建议保持两个 topic 独立但交叉引用。

          attention-optimization：Kimi K2 的 MLA 和 Kimi Linear 的 KDA 直接影响 agentic serving 的 per-agent 内存足迹 [L2:2507.20534]。TokenDance 的 collective reuse 深度依赖 RoPE 位置编码机制 [L2:2604.03143]。注意力机制的选择是 agent-system 的上游约束。

          agent-safety：Claude Code 的 7 层安全管道和 93% approval fatigue 数据 [L2:2604.14228]、SGH 的 validation gap 分析 [L2:2604.11378]、PASTE 的 side-effect management [L2:2603.18897]、以及 Semantic Router 的 jailbreak/PII/hallucination 安全信号 [L2:vllm-project-semantic-router] 均触及安全边界。Semantic Router 将安全定位为路由决策的输入维度 [L3:vllm-project-semantic-router]，与 Claude Code 的 runtime 安全管道和 Agents SDK 的 Guardrails 机制形成三层安全联防。当安全相关论文积累到 5+ 篇时，建议拆出独立 agent-safety topic。

          RL-training-infra：K2.6 的长程 agentic RL [L2:kimi-k2-6] 和 Concur 的 RL rollout 优化 [L2:2601.22705] 连接了 agent-system 与 RL 训练基础设施。Agent serving 和 RL rollout 共享同一组 KV cache 管理挑战。

          model-routing / AI-gateway：Semantic Router 代表了一个潜在的独立 topic——系统级 LLM 路由与 AI 网关。当前仅 Semantic Router 一个实体直接聚焦此方向 [L3:vllm-project-semantic-router]，但其 Workload-Router-Pool 架构愿景 [L2:vllm-project-semantic-router] 和 Scepsy 的 multi-LLM pipeline [L2:2604.15186] 已呈现出模型集群级路由与调度的技术线索。如果 routing/gateway 相关实体积累到 3+ 篇，建议拆出独立 model-routing topic。
