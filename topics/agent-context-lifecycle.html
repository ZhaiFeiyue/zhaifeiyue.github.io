Agent Context 生命周期：从 Lost in the Middle 到自顶向下的分析精化

1. 主题缘起

2025–2026 年，coding agent 成为 LLM 的主流使用模式。一个 agent 在执行复杂任务时，其 context 窗口会经历数十轮工具调用，累积大量文件内容、搜索结果、执行日志。这个过程暴露了两个互相交织的问题：

第一个是 Lost in the Middle——LLM 对 context 中间位置信息的利用能力显著低于开头和结尾。这个现象在 2023 年被 Liu et al. 首次系统性记录，到 2026 年仍未被根本解决，即使 context 窗口已扩展到 1M token。

第二个是 Context 缺乏生命周期管理——agent 的 context 是 append-only 的：skill 可以渐进式加载（progressive disclosure），但没有渐进式卸载（progressive removal）。当一个复杂任务在 high-level design 和 low-level execution 之间周期性切换时，上一个 phase 加载的 low-level 细节对当前 phase 的 high-level 思考不仅没有价值，还会成为噪音——而且恰好落入 attention 的中间死区。

这两个问题的交汇点指向一个更深的架构问题：如何设计一个 context 有完整生命周期（load → use → extract → evict）的 agent 系统，而不是让 context 无限膨胀直到模型在自己的记忆中迷失？

本文从 attention 机制层面的根因分析出发，经过系统层面的 context 管理方案，最终到达一个具体的工程实践——KB 知识库的层级化管道和自顶向下的分析精化机制。整个讨论链路是：


Attention 层: Lost in the Middle 的根因是什么？
    ↓
Attention 层: 能否用外挂 mask 解决？已有哪些方案？
    ↓
系统层: Agent 为什么尤其受害？context 缺少什么？
    ↓
架构层: 如何通过 subagent 实现 context 隔离？
    ↓
设计层: 当高层发现低层输出不足时，如何反向传递精化的指导？

2. Lost in the Middle：现象、根因、与 Agent 的关系

2.1 现象：U 型性能曲线

当 LLM 处理长 context 时，对信息的利用能力呈 U 型分布：开头（primacy）和结尾（recency）的信息被高效利用，中间位置的信息被严重低估。这不是个别模型的 bug——Liu et al. (2023) 的实验显示，GPT-3.5-Turbo 在 multi-document QA 任务中，当 relevant document 放在 context 中间时，性能下降超过 20 个百分点。更极端的是，在 20 和 30 document 设置下，middle 位置的性能甚至低于完全不给文档的 closed-book 基线（56.1%）。

三年后的 2026 年，这个现象仍未消失。RULER、LongBench v2、HELMET 等 benchmark 在 17 个长 context 模型上的测试表明：所有模型都展现 U 型曲线，更大的 context 窗口只是把死区移得更远，并没有消除它。正如一位研究者所言："Bigger context windows did not fix it. They just gave you more middle to lose things in."

2.2 根因：三层叠加

第一层：架构的先天缺陷（最根本）

2026 年 3 月发表的 Lost in the Middle at Birth: An Exact Theory of Transformer Position Bias 给出了严格的数学证明：U 型曲线在模型初始化时就已存在，与训练和位置编码无关。它是 causal decoder + residual connections 的固有几何性质。

具体机制：

  • Causal masking 保证 primacy bias：第一个 token 出现在所有后续 token 的注意力范围内，在代数上不可避免地获得更高权重。
  • Residual connections 锚定 recency bias：最后几个 token 通过残差路径获得最强的梯度信号。
  • 中间区域被卷积式稀释：在离散 Cesàro 矩阵的幂运算下，中间 token 的梯度路由被指数级压缩，形成一个 O(1/(H-1)!) 量级的"死区"。

论文的结论是：U 型偏差是拓扑层面的必然（"geometric birthright"），不是 bug，是 architecture 的 birthright。因此，修改 RoPE 等位置编码只是治标——要治本，训练范式必须被显式设计来克服这个初始化偏差。

第二层：训练过程的强化

  • Pre-training 的 next-token prediction 天然奖励 recency（最近的 token 与下一个 token 的统计相关性最强）。
  • Instruction tuning 天然奖励 primacy（system prompt / 指令总在开头）。
  • 中间位置在训练分布中既不是"指令"也不是"最近上下文"，两头不靠。

OpenReview 上的一篇 2025 年工作提出了另一个视角：Lost-in-the-Middle 不是"缺陷"，而是对不同信息检索需求的"适应"。Pre-training 同时包含需要 uniform recall（长期记忆需求）和优先最近信息（短期记忆需求）的任务。U 型曲线是这两种需求联合训练的产物。更大的模型分布 attention 更均匀，U 型效应减弱——但不会消失。

第三层：Attention Sink 效应

模型学会将多余的 softmax 概率质量 dump 到第一个 token（"attention sink"），进一步加剧了 primacy bias，同时压缩了分配给中间 token 的注意力预算。这是一个 learned artifact，但它叠加在架构层面的先天偏差之上。

2.3 Agent Coding 为什么尤其受害

Agent 编码场景有几个特殊性使 lost-in-middle 问题恶化：

场景特征为什么加剧 lost-in-middle
多轮对话conversation history 不断增长，早期的 research 结果被推到 middle
多文件探索读了 10 个文件，中间几个的内容被 attention 遗忘
工具调用结果search/grep 返回大量结果，关键信息可能在中间位置
context 膨胀agent 的 observation 累积远超人类手动 prompt 的长度

典型失败模式：agent 先读了 file A（开头），然后读了 file B、C、D（中间），最后读了 file E（结尾）。当需要综合 B/C/D 的信息做决策时，这些内容恰好在 attention 死区。

3. Attention 层的解法：从 Hard Mask 到 Soft Gate

3.1 一个自然的直觉：外挂 relevance mask

面对 lost-in-middle，一个直觉的想法是：在 softmax 之前，识别出哪些 token 与当前 generation 明确不相关，直接 mask 掉，只对剩下的 relevant token 算 softmax。这样：

  • softmax 的分母变小（更少的 token 竞争概率质量）
  • 不相关 token 不会偷走 attention 预算
  • 中间位置的 relevant token 有机会获得应有的 attention

3.2 核心矛盾：鸡生蛋问题

这个想法面临一个循环依赖：判断"相关/不相关"本身需要某种形式的 attention 计算。你还没算 QK^T，就不知道当前 query 跟哪些 key 相关。

此外，hard mask 还有几个严重风险：

风险说明
信息永久丢失soft attention 至少给每个 token 非零概率；hard mask = 0 意味着一旦判错就无法挽回
多跳推理断裂token A 看起来跟 query 不相关，但它是连接 token B（相关）和答案的桥梁
每层需要不同 masklayer 3 的"相关"和 layer 20 的"相关"可能完全不同，一个全局 mask 不够用
梯度被阻断hard binary mask 不可微，训练时梯度过不去

所有 2025–2026 年的成功方案都用 soft gating（sigmoid 连续值）而非 hard mask。以下是四条主要路线。

3.3 路线 1：用 Value 表征预测 mask — Dynamic Mask Attention (DMA)

DMA (ICLR 2026) 的核心思路是用 value representation（而非 QK 分数）来生成 content-aware 的 dynamic mask：


传统:  score = softmax(Q @ K^T) @ V
DMA:   mask  = sigmoid(f(V))        ← 用 V 的内容生成 mask
       score = softmax(Q @ K^T ⊙ mask) @ V

这打破了循环——V 是输入的线性投影，不依赖 attention 计算。mask 是 soft（连续重要性分数），保证可微分。效果：long-context 任务上最高 10x 加速，perplexity 和 accuracy 都优于 fixed sparse attention。

关键创新：mask 是 per-head、per-query 的——不同 attention head 学会关注不同的内容模式，不同 query 激活不同的 mask。这解决了"一个全局 mask 不够用"的问题。

3.4 路线 2：在 SDPA 输出端加 Gate — Gated Attention

Gated Attention (NeurIPS 2025, 已用于 Qwen3-Next) 选择了不同的位置：不在 softmax 前 mask，而是在 softmax 后加一个 query-dependent 的 sigmoid gate：


传统:  output = softmax(Q @ K^T / √d) @ V
Gated: output = [softmax(Q @ K^T / √d) @ V] ⊙ sigmoid(Q @ W_g)
                                                 ↑ query-dependent sparse gate

这个方案的优雅之处是：让 softmax 正常算，然后用 gate 来抑制不需要的信息，彻底绕开了"softmax 前怎么知道谁不相关"的问题。

效果：

  • 消除 attention sink（第一个 token 不再吸收多余概率）
  • RULER 长上下文评测提升 10+ 分
  • 训练更稳定，容忍更大 learning rate

该工作的 30 个变体对比实验揭示了 gating 有效的两个根本原因：(1) 在 value 投影和 output 投影之间引入非线性，打破了低秩瓶颈；(2) sigmoid 的天然稀疏性（输出集中在 0 和 1 附近）引入了 input-dependent sparsity，消除了 attention sink。

3.5 路线 3：双重 Gating + 自适应稀疏度 — Gated Sparse Attention (GSA)

GSA (2026) 把两个位置都加了 gate：


V' = V ⊙ sigmoid(h @ W_Vg)       ← value gate: 早期抑制无信息维度
O  = sparse_SDPA(Q, K, V')       ← 只对 top-k 重要位置算 attention
u  = O ⊙ sigmoid(h @ W_Og)       ← output gate: 丢弃无信息输出

关键创新是 adaptive sparsity controller：score 方差大 = 模型很确定谁重要 = 激进剪枝；方差小 = 不确定 = 保留更多 context。这让稀疏度随输入内容动态调整，而不是用固定的 top-k。

3.6 路线 4：特征维度稀疏 — Sparse Feature Attention (SFA)

SFA (2026) 换了一个正交的方向——不 mask token，而是把 Q/K 映射到 k-sparse code：


Q_sparse = top_k(Q)   # 每个 query 只激活 k 个维度
K_sparse = top_k(K)   # 每个 key 只激活 k 个维度
score 只在 overlap 维度上算，天然就是稀疏的

计算量从 O(n²d) 降到 O(n²k²/d)，当 k << d 时非常高效。SFA 与 token-level sparsity 正交，可以组合使用。

3.7 小结：不要对抗架构，要围绕它设计

四条路线的共同洞察：不要 hard mask，要 soft gate；不一定要在 softmax 前做，softmax 后做一样有效甚至更好。 lost-in-middle 是架构的拓扑必然，在 attention 层面的解法不是消除偏差，而是提供 content-aware 的补偿机制。

4. 系统层的问题：Context 没有 GC

4.1 Progressive Disclosure 的不对称

当前 agent 系统（包括 Cursor、各种 coding agent）在 context 管理上有一个结构性不对称：skill 和 context 可以渐进式加载（progressive disclosure），但没有渐进式卸载（progressive removal）。Context 生命周期是 append-only 的：


时间 →
[system prompt] [skill A] [skill B] [tool result 1] [tool result 2] ... [tool result N]
                ←———————————  只增不减  ———————————→

4.2 复杂任务的周期性模式

一个复杂的 coding 任务不是线性的，而是在 high-level 和 low-level 之间周期性切换的：


Phase 1: High-level design     → 需要：架构 skill, 需求摘要
Phase 2: Low-level execution   → 需要：具体 API 文档, 代码 skill
Phase 3: High-level review     → 需要：设计约束, 整体视角
Phase 4: Low-level fix         → 需要：debug skill, 错误上下文
         ↻ 循环

Phase 2 加载的具体 API 文档在 Phase 3 时不仅没有价值，还：

  • 占据 context 预算，挤压真正有用的信息
  • 落入 attention 的 middle 死区，产生噪音
  • 因为 lost-in-middle，反而可能让模型"注意到"这些过时细节而忽略当前目标

这本质上是一个 context 没有 GC（垃圾回收）的问题。

4.3 2025–2026 年的系统级解法

ContextBudget：预算感知的 Context 管理 (2026)

将 context 管理建模为预算约束的序列决策问题。用 RL 训练 agent 自己决定什么时候压缩、压缩多少。核心机制：(1) budget-conditioned inference state 在 append 新 observation 前暴露剩余 context headroom；(2) commit-block aggregation 让 agent 自适应控制压缩时机和力度。不是简单的 FIFO 淘汰，而是根据下游任务成功率来优化压缩策略。

Neural Paging：类 OS 的页面置换 (2026)

把 context 管理类比为操作系统的虚拟内存。LLM = CPU（只做推理），Page Controller = MMU（管理 context window）。context window 满了 → Page Controller 决定 EVICT 哪些 token；需要新信息 → Page Controller FETCH 进来。目标是近似 Semantic Belady's Algorithm（淘汰未来最久不会用到的 token）。关键是信息隐藏：Main LLM 以为自己有固定大小的 context window，Page Controller 透明地确保 relevant information 可用。

Unified Memory Agent (UMA)：端到端记忆管理 (2026)

用 RL 联合优化 memory operations（create, update, delete, reorganize）和 task execution。与 decoupled 两阶段方案（先训 memory 再训 QA）相比，联合训练的 UMA 在平均准确率上高出 10+ 个百分点（76.46% vs 66.29%）。原因是 decoupled 训练无法让 memory maintenance 预见下游 QA 需要什么信息，也无法让 QA 策略回溯影响上游 memory 决策。

CoDA：Context 解耦的层级 Agent (2025)

核心思路完全对应 agent coding 的周期性模式问题：将 monolithic 决策过程分解为两个 contextually-isolated 角色——Planner 和 Executor。两者由同一个模型扮演，但在不同的 context 下运行。

Executor 在 ephemeral workspace 中运行，每个 sub-task reset context，bounded by 单个文档；Planner 维护 strategic context，只看 concise summaries。关键：verbose raw documents 永远不进入 Planner 的 strategic context。Executor 是一个黑盒，抽象掉 noisy details，实现有效的 context decoupling。

CODEDELEGATOR：Ephemeral-Persistent 状态分离 (2026)

将 strategic planning 与 code implementation 通过 role separation 解耦。Persistent Delegator 处理 task decomposition 和 orchestration，从不写实现代码。Ephemeral Coders 在 isolated context 中执行 atomic sub-tasks：每个 Coder 以空 context 启动，只接收它的 spec，完成后所有 trace 随实例销毁。

核心原则 P2: Fresh Context, Focused Scope——通过 per sub-task reset context，完全防止 cross-sub-task pollution。Delegator 追踪 what is accomplished via workspace state，never how。

Amnesic Protocol：主动遗忘 (2025)

更激进的方案——把遗忘当作核心特性而非缺陷。Agent 的 context window 是 disposable RAM——它必须显式 "learn"（提取 artifact）和 "forget"（清除 context）来维持 focus。

操作原语：

  • stage_context(path) — 加载文件到工作内存
  • save_artifact(id, val) — 提取关键信息到 Backpack（外部持久存储）
  • unstage_context(path) — 显式清除文件（progressive removal）
  • switch_strategy(name) — 切换 persona（"Architect" → "Implementer"）

安全性由代码保证（Physical Invariants），不由 prompt 保证。如果 agent 试图在没有清除 memory 的情况下读取文件，系统物理阻止该操作。

GenericAgent：分层按需记忆 (2026)

围绕 "context information density maximization" 原则构建。四层机制：(1) 原子化工具描述，保持接口简洁；(2) 分层按需记忆，默认只在 context 放 high-level 索引，需要 drill-in 时才展开具体内容，用完后自动收回；(3) 自演化，把验证过的轨迹压缩成可复用的 SOP 和可执行代码；(4) 压缩+截断层，当 context 超预算时主动管理历史信息。

4.4 理想的 Context 生命周期

综合上述方案，理想的 context 生命周期应该是：


                    ┌──── skill/context scope ────┐
Phase 1 (design):  [arch_skill] [requirements]     → 产出: design.artifact
                    └──────── phase 结束，清除 ────┘

                    ┌──── skill/context scope ────┐
Phase 2 (impl):    [api_skill] [design.artifact]   → 产出: code + impl.artifact
                    └──────── phase 结束，清除 ────┘

                    ┌──── skill/context scope ────┐
Phase 3 (review):  [arch_skill] [design.artifact] [impl.artifact]
                    └──────── phase 结束，清除 ────┘

核心设计原则：

原则说明
Scoped loadingSkill 和 context 有明确的生命周期，不是"加载了就永远在"
Artifact extractionPhase 结束时，有价值的结论被提取为结构化 artifact，raw context 被丢弃
Phase-aware loading根据当前 phase 自动选择加载什么、卸载什么
Budget enforcement不是靠模型"自觉忽略"，而是物理上不让过时信息进入 context

5. KB Orchestrator：Subagent 作为 Context 隔离的实现

KB 知识库的 orchestrator 是上述 context 隔离理论的一个具体工程实现。它使用 Cursor 的 subagent（Task tool）机制来实现 ephemeral context：每个 subagent 启动时 context 是干净的，完成后只返回结构化的 metadata。

5.1 四个结构性防御

物理隔离（不靠模型自觉）

主 session 有严格的 Read allowlist/denylist：

  • 允许读取：INDEX.md（entity lookup）、LINKS.md（related discovery）、REVIEW.md（stale queue）、CHANGELOG.md（event history）、所有 entity 的 frontmatter（不读 body）
  • 禁止读取：skills/kb-L1-ingest/（只有 L1 subagent 读）、skills/kb-L2-distill/（只有 L2 subagent 读）、以此类推

Violation = skill leak into main session context。有 scanner (kb lint) 在每次 layer 完成后检查。

Ephemeral Context per Phase

每个 subagent 的 context 是 fresh 的——previous subagent 的 state 被 discard。这是 context 隔离的结构性防御。L1 subagent 读了原始 PDF、跑了所有 ingest 逻辑——这些全部随 subagent 销毁。L2 启动时看到的是干净的 context + 只有 {path, hash, source_type} 三个字段。

Metadata-only Handoff（信息漏斗）

层之间传递的不是 raw body，而是结构化的 frontmatter 级 metadata：

From → ToMetadata 传递内容
L1 → L2{path, hash, source_type} — 3 个字段
L2 → L3{path, hash, tags, category, related[]}
L3 → L4{path, hash, l4_targets=[category, topic1, topic2...]}

下一个 subagent 需要什么就自己去 Read。每次 phase 切换都经过一次信息压缩漏斗。

Orchestrator 本身保持 high-level

Orchestrator 的视野被限制在 frontmatter 级别——它永远处于 "high-level dispatch" 模式，具体的 low-level 内容被封装在 subagent 里。Orchestrator 不会因为 L1 阶段加载的 raw paper 内容而在 L3 的 relate 阶段被干扰。

6. 反向传递：当高层发现低层输出不足

6.1 前向管道的局限

KB 的 pipeline 是严格前向的：extract → classify → L1 → L2 → L3 → L4。当高层（如 L3）发现低层（如 L2）的输出不足时，唯一的选项是 pipeline halt。没有机制将 feedback 向下传播并触发重新运行。

真实场景：

  • L3 做 cross-paper synthesis 时，发现 L2 的 distill 把两个不同概念混为一谈
  • L3 需要某个 comparison 数据，但 L2 的 7-section distill 没有提取出来
  • L4 做 cross-category meta 时，发现不同 L3 之间有矛盾
  • L3 发现一个相关 entity 还没有 L1/L2，无法做有意义的 synthesis

6.2 L1 不可变性约束

一个关键的设计洞察是：L3/L4 不应该直接 trigger L1 re-run。原因：

L1 是忠于原文的——逐段复述，设计上不应有信息丢失。如果 L3 发现缺少某个信息，信息丢失点几乎一定在 L2 的压缩环节（从 L1 完整逐段复述到 7-section distill 的有损压缩），而不是 L1。如果 L1 做对了（faithful），那 L1 里已经有这个信息——只是 L2 distill 时没有提取出来。

这意味着反向传递遵循严格的逐级传递规则：


L4 ──► L3 only
L3 ──► L2 only
L2 ──► L1 only（唯一能触发 L1 re-run 的层）
L1 ──► source（不可变）

L3 没有资格直接评判 L1，因为 L3 看到的是 L2 的 distill 结果——如果信息不在 L2 里，应该先让 L2 重新做 distill。只有当 L2 re-run 时发现 L1 genuinely 缺失内容，才会触发 L1 re-run。

这个约束大幅简化了系统：

维度无约束逐级传递
可能的 backward 路径L2→L1, L3→L1, L3→L2, L4→L3, L4→L2, L4→L1L2→L1, L3→L2, L4→L3
决策复杂度orchestrator 需要判断回到哪一层每层只能回退到直接下级
L1 不变性弱（任何层都能触发重写）强（只有 L2 能触发）

6.3 Hybrid 重跑策略

根据 feedback 类型，选择不同的 agent 策略：

feedback typeagent 策略读 previous output?说明
clarificationpatch agentMUST read信息存在但被误解/简化/混淆。最常见——L2 不是"错了"，只是没有按上层需要的角度分析
insufficiencypatch agentMUST read信息确实缺失。需要回到 input 重新提取
errorfull re-runoptional reference分析根本性错误。不能信任 previous output
missing_depnew pipelinen/a相关 entity 缺少 L1/L2。启动独立 pipeline

关键：无论 patch 还是 full re-run，都是新 subagent（fresh context）。Patch agent 读 previous output 是通过 file read（结构化输入），不是通过继承上一个 agent 的 conversation history。这保持了 context 隔离的架构不变量。

7. 自顶向下的分析精化：对称的 Abstraction-Refinement

7.1 核心原则

KB 的 pyramid 是一个自下而上的抽象链：


L1 (concrete: faithful source copy)
  → L2 (compress: 7-section distill)
    → L3 (synthesize: cross-paper relate)
      → L4 (abstract: cross-category meta)

当信息不足时，feedback 流向相反方向（自顶向下）。但 feedback 不是 symptom report（"你缺了 X"），而是 analysis + guidance（"我分析了问题，你应该关注什么"）。每层向下传递时，guidance 变得更具体——因为每层理解自己 input 的结构比任何高层都好。

这形成了完美的对称：


构建时 (bottom-up):    具体 → 抽象     每层向上压缩信息
修复时 (top-down):     抽象 → 具体     每层向下精化指导

7.2 Analysis + Guidance，不是 Symptom Relay

feedback 包含两个部分：

  • Analysis：emitter 对问题的分析（在它自己的抽象级别上）
  • Guidance：对 target layer 的指导——告诉它应该关注什么，而不是告诉它应该怎么做

每层按自己的 design 来执行指导——L2 收到"clarify Paged Attention motivation"后，仍然按自己的 7-section template 做 distill，只是这次知道该重点提取什么。guidance 不改变 layer 的 design，只改变它的 focus。

7.3 翻译合约

每层向下传递 feedback 时，必须遵守翻译合约：你的 guidance 必须比你收到的 guidance 更具体。 这是 enforced 的——prompt 里明确要求"translate the abstract need into concrete terms that L(k-1) can act on"。

这防止了 feedback 在传递过程中退化为模糊的抽象描述。每一跳都增加精度。

7.5 Evidence Chain 作为 Routing Table：认识论基础

上述的分析精化机制建立在一个更根本的原则上：高层的理解必须建立在低层的证据链上，而不是凭空创造。

KB pyramid 的认识论结构是：


L4 的理解 = f(L3 evidence)     不是 L4 自己想的
L3 的理解 = f(L2 evidence)     不是 L3 自己想的
L2 的理解 = f(L1 evidence)     不是 L2 自己想的
L1 = source material            唯一的"事实创造"层

每一层的职责不同，但都建立在可追溯的证据链上：

层职责证据来源独特价值
L1忠实提取source 原文无损保存
L2分析压缩L1从 raw facts 中提炼 insight
L3跨源综合多个 L2发现 L2 之间的矛盾、互补、delta
L4跨域抽象多个 L3发现 L3 之间的 pattern、tension、evolution

L3 的独特价值不是"重述 L2"，而是在 L2 之间建立关系——发现哪些 L2 互相矛盾、哪些互补、哪些是同一问题的不同解法。但这些关系的每一个端点都必须 trace 到具体的 L2 citation。

这通过 [L2:] citation 格式强制执行。每条断言必须携带 citation，这不只是格式要求——citation chain 同时服务三个目的：


[ref:L2:paper-A#core-contribution] vs [ref:L2:paper-B#core-contribution]
         │                                      │
         ├── 1. 正确性审计: 读者可以验证这个 claim
         ├── 2. 矛盾发现: L3 的分析价值就在这些对比里
         └── 3. 反向修复路由: 如果 claim 错了, 精确定位到哪个 L2 的哪个 section

当用户说"L3 §4 的 comparison 不清楚"时，系统 parse L3 §4 的 citation，自动知道涉及哪些 L2 entity。反向修复的路由是确定性的——不需要 agent 运行时判断，证据链本身就是路由表。

7.6 大多数链在 L2 终止

因为 L1 是 faithful 的（逐段复述，信息都在），大多数 feedback chain 在 L2 就解决了：

  • L3 说"我需要 bandwidth 数据" → L2 重新读 L1，找到数据，重新 distill → done
  • L4 说"两个概念混淆" → L3 说"L2 缺乏 motivation 分析" → L2 重新读 L1，找到 motivation，重新 distill → done

只有极少数情况（L1 agent 在初次 ingest 时 genuinely 跳过了某段内容）才需要链延伸到 L1。这也验证了 L1 immutability 作为 strong invariant 的正确性。

7.7 Loop Safety

四个 guardrails 防止无限循环：

  1. Per-layer retry cap（max_retries=2）——每层在一次 pipeline 执行中最多被 re-run 2 次
  2. Chain depth cap（max_chain_depth=3）——refinement chain 不能超过 pyramid 的完整深度（L4→L3→L2→L1）
  3. Feedback dedup——如果 re-run 后同一个 (target_layer, focus_sections) tuple 重复出现，说明 fix 没生效，escalate to user
  4. Nested pipeline depth cap——missing_dep 触发的嵌套 pipeline 不能自身再触发 missing_dep（depth=1）

    5. 8. Open Challenges

    8.1 Attention 层：U 型偏差是否可根治？

    Lost in the Middle at Birth 的证明表明 U 型偏差是 causal decoder 的拓扑必然。这意味着在不改变基础架构的情况下（例如转向 bidirectional 或 non-autoregressive 模型），U 型偏差只能被补偿，不能被消除。

    Gated Attention 和 DMA 提供了有效的补偿，但它们增加了参数和计算量。开放问题：能否找到零开销的补偿方案？或者说，补偿 lost-in-middle 的最优 compute-quality tradeoff 在哪里？

    8.2 系统层：Context GC 的粒度问题

    现有方案在 context eviction 的粒度上差异很大：

    • Amnesic: file-level（stage/unstage 整个文件）
    • Neural Paging: token-level（精细但开销大）
    • CoDA/CODEDELEGATOR: phase-level（整个 subagent context 销毁）

    什么粒度最优？可能取决于任务类型：coding agent 可能适合 file-level（因为代码文件是自然的 context 单元），而 research agent 可能需要 section-level（因为一篇论文的不同部分在不同 phase 有不同价值）。

    8.3 分析精化链的质量保证

    自顶向下的 feedback 依赖每层 agent 的分析能力。如果某层 agent 的分析本身就是错的（例如 L4 错误地认为两个概念不同，实际上它们是同一个东西），feedback chain 会传播错误的修正指令。

    当前的 safety net 是 retry cap + dedup detection，但更根本的解决方案可能需要引入 feedback validation gate——在 re-run 之前验证 feedback 的合理性。

    8.4 从 KB 到通用 Agent：可迁移性

    KB orchestrator 的设计建立在 well-defined 的层级结构（L1/L2/L3/L4）之上。这个层级是人工设计的，不是自动发现的。对于通用 coding agent，如何自动识别 "phases" 和 "abstraction levels"？

    ContextBudget 和 UMA 试图通过 RL 来学习这一点，但目前只在简单任务上验证过。将分析精化链扩展到没有预定义层级的通用 agent 场景，仍然是一个开放问题。

    9. 邻接 topic

    Topic关系
    agent-system强关联——agent serving 的 context 管理是本 topic 的一个具体应用场景。agent-system 侧重 serving infrastructure（KV cache, scheduling, routing），本 topic 侧重 context lifecycle
    persistent-execution-paradigm弱关联——persistent kernel 消除了 GPU 端的 launch overhead，本 topic 消除了 LLM context 端的信息冗余。都是在对抗"不必要的重启"
