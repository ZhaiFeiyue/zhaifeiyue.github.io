Agent Context 生命周期：从注意力位置偏置到工作流感知的 KV Cache 管理 #

1. 主题缘起 #

2025–2026 年，coding agent 和 deep-research agent 成为 LLM 的主流使用模式。一个 agent 在执行复杂任务时，其 context 窗口会经历数十轮工具调用，累积大量文件内容、搜索结果、执行日志。这个过程暴露了四个分布在不同系统层级、但互相交织的问题：

第一，模型层的位置偏置。 LLM 对 context 中间位置信息的利用能力显著低于开头和结尾——即 Lost in the Middle 现象。该现象在 2023 年由 Liu et al. 首次系统性记录，到 2026 年在 NoLiMa 等更严格的 benchmark 上证实：即使标称 128K–1M context 的 frontier 模型，其有效利用长度仍不超过 8K [L2:2502.05167]。更严格的数学分析表明，U 型偏差可能是 causal decoder + residual connections 的固有几何性质 [L2:2601.15300]。

第二，agent 工作负载对 KV cache 的特殊需求。 Agent 的多轮 tool-call 间隙极短（平均 925ms–1923ms），但现有推理引擎的 end-of-turn eviction 策略导致下一轮必须重入队列，per-turn queueing delay 累积可占总延迟 58.2% [L2:2511.02230]。这催生了 2025–2026 年最活跃的研究方向之一：agent-aware KV cache 管理。

第三，框架层缺乏工作流感知的调度。 传统 serving 框架按单请求粒度调度，不理解 agent 工作流的 DAG 结构。多个 agent step 之间的 KV 复用、跨 step 的 context 共享、以及工作流级的公平调度，都需要框架从"请求级"跃迁到"程序级"编排 [L3:framework] [L2:2602.13692]。

第四，注意力架构的多元化使问题更加复杂。 从标准 softmax 到 compressed attention (CSA+HCA)、再到线性注意力 (KDA)，每种架构对 KV cache 的形状、大小、eviction 策略都有不同要求 [L3:model]。框架和 agent 系统必须同时适配多种注意力范式。

这四个问题的交汇点指向一个跨类别的主题：如何在模型层位置偏置的约束下，为 agent 工作负载设计有完整生命周期（load → use → evict → prefetch）的 context 管理系统？ 这个问题不能由任何单一类别独立回答——它需要模型层的注意力机制、agent 层的 KV cache 策略、以及框架层的工作流调度三者协同设计。

5. 技术线交错 #

5.1 模型层注意力设计如何约束框架层 KV cache 管理 #

注意力架构决定了 KV cache 的形状、大小和生命周期特征，直接约束框架层的缓存策略。

标准 MLA (K2 系列) 产生 576 floats/token 的 latent KV [L2:2507.20534]，KV cache 与序列长度线性增长。在 agent 多轮场景下，128K context 的 KV cache 可达数十 GB，直接触发 Continuum 所识别的 end-of-turn eviction 问题 [L2:2511.02230]。

KDA 线性注意力 (Kimi Linear) 用 fixed-size recurrent state 替代 per-token KV，cache 大小与序列长度无关（128×128/head）[L2:2510.26692]。这从根本上消除了 agent 多轮场景的 KV 膨胀问题——但框架层的 prefix caching 机制（如 KVFlow 的 STE [L2:2507.07400]、Helium 的 Radix Tree [L2:2603.16104]）假设 KV cache 是 append-only 的 token 序列，无法直接适配 recurrent state。

CSA+HCA 压缩注意力 (DeepSeek-V4) 做"先 4× 压缩再 sparse top-k 选择"，KV 条目数缩减为 O(n/m) [L2:deepseek-v4]。这要求 KVServe 等跨 DC 传输框架重新设计压缩策略——service-aware 压缩必须理解不同注意力层的压缩比差异 [L2:kvserve]。

5.2 Agent 工作负载模式如何驱动框架重新设计 #

Agent 工作负载有三个区别于交互式对话的关键特征，每个都要求框架层的响应：

特征 1：短间隙多轮 tool call。 间隙仅 925ms–1923ms [L2:2511.02230]，远短于传统对话的分钟级间隔。框架响应：Continuum 为每个 tool-call 计算最优 KV 保留时间 τ*，保留 KV 而不是每轮驱逐 [L2:2511.02230]；ThunderAgent 将 agent program 整体作为调度单元，避免 turn 间的调度开销 [L2:2602.13692]。

特征 2：DAG 结构的工作流。 Agent 的步骤间存在复杂的依赖关系（条件分支、重试循环、并行路径），不是简单的线性序列。框架响应：SAGA 用 Agent Execution Graph 形式化工作流为 DAG [L2:2605.00528]；Halo 借数据库 query optimizer 思路做 batch DAG 整合 [L2:2509.02121]。

特征 3：CPU 侧瓶颈。 工具执行（文件读写、搜索、编译）占 E2E 延迟 35%–88% [L2:2511.00739]。GPU 优化的上限因此被 CPU 侧限制。框架响应：PASTE 通过 pattern mining + 投机性工具执行，在 LLM "思考"时预执行下一个工具，E2E 延迟降 48.5% [L2:2603.18897]。

5.3 位置偏置如何从模型层传导到 agent 层 #

Lost in the Middle 的位置偏置在 agent 场景被放大。Agent 先读 file A（context 开头），然后读 file B、C、D（推入 middle），最后读 file E（context 末尾）。当需要综合 B/C/D 的信息时，这些内容恰好在注意力死区。

这不是简单的"单针检索"问题——NoLiMa 证明，即使去除字面匹配，13 个 frontier 模型的有效长度均 ≤8K [L2:2502.05167]。多跳场景更糟：证据片段之间的距离本身是独立的退化变量，且与准确率几乎线性反相关 [L2:2412.10079]。

这个传导链条解释了为什么 agent 的 KV cache 管理不能是简单的 LRU：被淘汰的 token 可能正是模型在注意力死区中遗漏的关键信息。SideQuest 的语义淘汰 [L2:2602.22603] 和 PBKV 的 GNN 预测 [L2:2605.06472] 本质上都是在尝试用"工作流语义"来弥补"注意力机制"的位置盲区。

5.4 生产系统架构如何反映这些张力 #

Claude Code 的逆向工程揭示了一个有趣的平衡：它选择了最简单的 Agent Loop 架构（while-loop + tool calls），但用 5 层 context compaction + 7 层安全管道来管理 context 膨胀 [L2:2604.14228]。核心范式是"1.6% 决策逻辑 + 98.4% 确定性基础设施"——模型负责判断，系统负责 context 生命周期管理 [L2:2604.14228]。

这与 SGH 的理论分析形成对照：SGH 认为 Agent Loop 有三个结构性缺陷（隐式依赖、无界 recovery、不可审计），主张用显式 DAG 替代 [L2:2604.11378]。两者反映了"信任模型判断力"与"结构化保证"之间的根本张力 [L3:agent]。

6. 共识与分歧 #

共识 1：End-of-turn eviction 对 agent 工作负载是灾难性的 #

所有 5 篇 agent KV cache 论文（Continuum、SideQuest、CMV、SAGA、PBKV）都以此为出发点。Continuum 测得 per-turn queueing delay 占总延迟 58.2% [L2:2511.02230]；SAGA 报告传统 LRU 的 competitive ratio 相比 Bélády 最优偏差巨大 [L2:2605.00528]；PBKV 测得 LRU 命中率仅 27% [L2:2605.06472]。无一论文为传统 end-of-turn eviction 辩护。

共识 2：位置偏置在多针/多跳场景远比单针严重 #

Gemini 2.5 Flash 在单针检索上已消除 LITM（1M context 100% 准确率）[L2:2511.05850]，但 NoLiMa 的去字面匹配 benchmark 将所有 frontier 模型有效长度压至 ≤8K [L2:2502.05167]，多跳 QA 中证据间距带来额外的乘性惩罚 [L2:2412.10079]。所有长上下文评测论文都同意：NIAH 式 benchmark 严重高估了模型的实际长上下文能力。

共识 3：KV cache 的形状正在多样化，框架必须适配 #

DeepSeek-V4 的压缩 KV (O(n/m)) [L2:deepseek-v4]、Kimi Linear 的 recurrent state (O(1)) [L2:2510.26692]、标准 MLA 的 per-token latent (O(n))——三种形状共存 [L3:model]。KVServe 的 service-aware 压缩 [L2:kvserve] 和 KVFlow 的 workflow-aware STE [L2:2507.07400] 都必须处理这种多样性。

分歧 1：KV 保留决策应基于时间还是语义？ #

Continuum 用时间维度的 cost-benefit 模型——tool call 在 TTL τ* 内返回则 KV 被保留 [L2:2511.02230]。SideQuest 认为 token 重要性是非单调的——某些 tool response 在第 t 轮看似无用但到第 t+n 轮可能重新变关键，必须用语义维度决定 [L2:2602.22603]。

根源：实验工作负载不同。Continuum 在 SWE-Bench/BFCL 上测试（6–11 turns），SideQuest 在 FRAMES/BrowseComp 上测试（80+ turns，120K+ tokens）[L3:agent]。短 horizon 下时间维度足够；长 horizon 下语义维度不可或缺。

分歧 2：静态 DAG 还是动态工作流更适合 KV 预测？ #

SAGA 用 Agent Execution Graph (DAG) 预测 KV 复用，假设工作流模式可建模为有向图（competitive ratio 1.31× vs Bélády）[L2:2605.00528]。PBKV 批评"静态 DAG 假设无法处理运行时条件分支和重试循环"，用 GNN 多步预测在动态工作流上达到 69% 命中率 vs 静态距离方法的 39.87% [L2:2605.06472]。

根源：SAGA 的 AEG 实际支持 backward retry edges 和 transition probabilities（不是纯静态 DAG），PBKV 的批评更准确地针对 KVFlow 而非 SAGA。但 SAGA 的 competitive ratio 是在 SWE-bench（相对结构化的 coding agent）上测得，高度动态的重试密集型工作流是否仍接近最优未经验证 [L3:agent]。

分歧 3：Lost in the Middle 是数据问题还是架构问题？ #

FilM/IN2 证明仅靠合成长上下文数据 + 指令微调就能修复位置偏置，不改注意力、不改位置编码 [L2:2404.16811]。但临界退化分析表明 Qwen2.5-7B 在 ~40% 标称长度处出现 cliff-like 衰退，RoPE 周期性和注意力弥散是更紧的约束 [L2:2601.15300]。Massive Values 分析揭示 RoPE 低频维度承载关键语义功能，破坏后 GSM8K 从 76.9% 暴跌至 4.0% [L2:2502.01563]。

根源：不同规模和不同任务类型下，数据约束和架构约束的相对重要性不同。FilM 的修复在 32K context 的 7B 模型上有效；但在 128K+ 的更大规模下，架构层面的约束（注意力弥散、RoPE 衰减）可能成为瓶颈 [L3:model]。

分歧 4：Agent Loop 是否"够用"？ #

SGH 认为 Agent Loop 有结构性缺陷（隐式依赖、无界 recovery、不可审计），只有显式 DAG 才能根本解决 [L2:2604.11378]。Claude Code 的生产成功恰恰建立在 Agent Loop 之上，通过确定性基础设施兜底 [L2:2604.14228]。

根源：对 LLM 判断力的信任程度不同。Claude Code 假设 frontier 模型有"good judgment"，用 harness 兜底；SGH 假设 LLM 决策不可检查。Claude Code 的 93% approval rate 恰恰暴露了 Agent Loop 的脆弱性——一旦模型犯错，防线只有人类审批 [L3:agent]。

7. 根本性困难 (Open Challenges) #

7.1 位置偏置与注意力表达力的信息论困境 #

Lost in the Middle 可能是 causal decoder + residual connections 的固有几何性质——U 型偏差在模型初始化时就存在 [L2:2601.15300]。Gated Attention 和 Attention Residuals 提供了有效补偿 [L2:2603.15031]，但补偿本身增加参数和计算量。

为什么困难：这不是"没人试"而是存在理论约束。在 causal masking 下，第一个 token 代数上不可避免地获得更高权重；residual connections 锚定 recency bias。要根治需要改变基础架构（bidirectional / non-autoregressive），但这与 LLM 的 autoregressive 生成范式冲突。线性注意力 (KDA) 通过 recurrent state 绕过了部分约束 [L2:2510.26692]，但 NoPE 方案需要 KDA 的 data-dependent gate 隐式补偿 RoPE 的语义功能——缺乏对应的机理分析 [L3:model]。

需要哪些类别协同：model（新注意力范式）+ algorithm（理论分析框架）+ framework（适配新 KV 形状）。

预估难度：5+ 年。零开销补偿方案可能不存在；最优 compute-quality tradeoff 尚未被理论刻画。

7.2 语义淘汰的 oracle 悖论 #

理想的 KV cache 淘汰应淘汰"未来最久不会用到的 token"（Semantic Bélády's Algorithm）。但判断"什么将来有用"本身需要理解任务的完整语义——这正是 agent 正在执行的推理过程。

为什么困难：SideQuest 用辅助 LRM 线程做语义判断（56–65% token 降幅）[L2:2602.22603]，但引入了对 LRM 能力的强依赖，且辅助线程本身消耗 GPU 资源。PBKV 的 Lipschitz 退化保证 [L2:2605.06472] 是目前唯一的形式化 regret bound，但其 GNN predictor 需要 per-workload 训练（~1K traces）。SAGA 的 AEG 预测在无 framework hints 时仍达 87% accuracy，但 TCT 退化 15.6% [L2:2605.00528]。

需要哪些类别协同：agent（语义理解）+ framework（高效淘汰执行）+ model（低开销语义信号提取）。

预估难度：2–3 年达到实用，但达到理论最优可能需要更长。部分方案（如 PBKV 的分层淘汰）已在特定工作负载上接近 Bélády 最优。

7.3 KV cache 形状碎片化与框架统一 #

三种主流注意力范式产生完全不同的 KV cache 形状：MLA 的 per-token latent [L3:model]、KDA 的 fixed-size recurrent state [L2:2510.26692]、CSA+HCA 的压缩条目 [L2:deepseek-v4]。现有的 KV cache 管理系统（Continuum、SAGA、PBKV）都假设标准的 per-token KV 格式。

为什么困难：框架层的 prefix caching（Radix Tree、STE）、eviction（TTL、LRU、GNN）、传输（KVServe 压缩）都依赖于 KV cache 是 token 序列这一假设。适配 recurrent state 需要重新定义"cache hit"的语义——一个 recurrent state 不能部分命中。适配压缩 KV 需要理解不同层的压缩比差异。

需要哪些类别协同：model（统一 KV 接口标准）+ framework（多形状 cache manager）+ agent（工作流级缓存策略适配）。

预估难度：1–2 年。DeepSeek-V4 和 Kimi Linear 的部署已在推动各自团队的框架适配，但跨团队的统一接口标准尚不存在。

7.4 Context 生命周期的粒度问题 #

现有方案在 context eviction 粒度上差异巨大：Continuum 是 turn-level（TTL 到期则整轮 KV 驱逐）[L2:2511.02230]，SideQuest 是 token-level（逐 token 语义淘汰）[L2:2602.22603]，CMV 是 session-level（三遍裁剪保留结构）[L2:2602.22402]，Claude Code 的 5 层 compaction 按 conversation segment 操作 [L2:2604.14228]。

为什么困难：最优粒度取决于任务类型——coding agent 可能适合 file-level（代码文件是自然的 context 单元），research agent 可能需要 paragraph-level（一篇论文的不同部分在不同阶段有不同价值），而多 agent 工作流可能需要 workflow-step-level。没有一种粒度能普遍最优，但框架支持多粒度的开销可能抵消收益。

需要哪些类别协同：agent（任务类型识别）+ framework（多粒度 eviction 机制）。

预估难度：1–2 年。组件技术已存在，需要的是系统集成。

8. 成熟度判断 #

整体评估：研究前沿 → 早期生产

  • 位置偏置诊断：成熟。LITM 现象已被充分刻画——从 2023 年的经验观察 [L2:2404.16811] 到 2026 年的理论分析 [L2:2601.15300]，从单针到多针/多跳 [L2:2412.10079] [L2:2502.05167]，diagnostic tooling 完备。
  • 位置偏置修复：部分成熟。数据驱动修复（FilM/IN2）在 ≤32K 有效 [L2:2404.16811]；训练免费解码方案（PCD）在特定设置下 +3–7% [L2:2506.08371]；frontier 模型已在单针上消除 LITM [L2:2511.05850]。但多针/多跳场景仍未解决。
  • Agent KV cache 管理：研究前沿，快速成熟中。2025 年 11 月 Continuum 首次提出 KV TTL [L2:2511.02230]，到 2026 年 5 月已有 5 个独立系统。Continuum 已开源为 vLLM 插件，生产可用 [L3:agent]。SAGA 和 PBKV 的工作流预测方法尚在论文阶段。
  • 工作流感知调度框架：早期生产。ThunderAgent 已开源 [L3:framework]，Helium 和 KVFlow 已有可复现的实现 [L3:framework]。但闭源系统（SAGA、Halo）展示的最佳数字不可复现。
  • 注意力架构多元化对框架的影响：早期探索。KDA 和 CSA+HCA 的框架适配仍在各自团队内部进行，跨团队的统一接口标准不存在。

趋势方向：快速加速。agent workload 的爆发式增长驱动了从模型到框架的全栈创新。2025 年 11 月到 2026 年 5 月的 7 个月内，该领域产出了 14 篇系统论文。主要商业 agent 系统（Claude Code、Cursor、Windsurf）的部署规模正在为这些研究提供真实工作负载验证。

9. 邻接 topic #

Topic关系
agent-system (agent serving infrastructure)强关联。agent-system 侧重 serving infrastructure 的全栈设计（KV cache 调度、集群路由、SLO 管理），本 topic 侧重 context 从创建到淘汰的完整生命周期。重叠区域是 KV cache 管理——SAGA、PBKV、Continuum 同时属于两个 topic。分界点：KV cache 的 scheduling 和 eviction policy 属于本 topic；KV cache 的 storage engine 和 network transfer 属于 agent-system。
attention-optimization (注意力机制优化)中等关联。Gated Attention、DMA、GSA 等注意力优化技术影响位置偏置的严重程度和 KV cache 的形状，但它们本身更关注 accuracy/efficiency tradeoff 而非 agent 场景。如果 attention-optimization topic 的 KV-compression 分支（CSA+HCA、KDA）与本 topic 的 KV-management 分支日益交叉，可考虑将 "KV 形状多样化的框架适配" 子主题独立为新 topic。
persistent-execution-paradigm (持久化执行)弱关联。TileRT/TokenSpeed 的 persistent Engine Kernel 消除 GPU 端的 kernel launch overhead [L3:framework]，本 topic 管理 LLM context 端的信息冗余。两者都在对抗"不必要的状态重建"，但操作层级不同（GPU kernel vs LLM context）。

潜在 topic 融合方向：如果 agent serving 系统开始在 KV cache 层面实现 context eviction（即 SAGA 的 WA-LRU 被集成到 vLLM/SGLang），那么 attention-level 的 KV 压缩（CSA+HCA）和 system-level 的 KV 淘汰（SAGA/PBKV）可能在 serving infrastructure 层面统一，推动本 topic 与 agent-system 的局部合并。
