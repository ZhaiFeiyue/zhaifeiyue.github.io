Static Compilation for LLM Inference：从 ASIC Graph Compile 到 Persistent Engine Kernel

1. 主题缘起

2018–2021 年间，AI 推理芯片（Habana Goya/Gaudi、Google TPU、Graphcore IPU）普遍采用 graph compilation 技术：编译期知道所有 tensor shape，静态规划执行流水，中间 activation 尽量驻留片上 SRAM。这一范式在 ASIC 上通过硬件支持（大 SRAM + 异构引擎 + 硬件 scheduler）获得了天然优势。

2024–2026 年，LLM 推理延迟成为核心指标后，TileRT 在通用 GPU 上重新实现了这一范式——AOT 编译 + 极致融合 + tile pipeline [L2:tilert-speed-scaling-law]。博客将其包装为"Persistent Engine Kernel"新范式，但从技术谱系看，底层原理与 2018 年 ASIC graph compile 同源：都是用编译期信息换运行时开销 [L3:tile-ai-tilert]。

这一 topic 横跨 kernel（算子融合、CUDA Graph、warp specialization）和 framework（graph compiler、静态调度、serving 系统架构），探讨的核心问题是：静态编译技术在 LLM inference 的适用边界和历史定位。

核心洞察：TileRT 不是一个新范式的发明，而是 ASIC graph compile 原理（静态编译 + 片上驻留 + 异构 overlap）在通用 GPU 上的软件复现。差异在于 GPU SMEM 极小（~228 KB vs ASIC 50+ MB），迫使采用 tile pipeline 作为 workaround [L2:tilert-speed-scaling-law]。

5. 技术线交错

5.1 ASIC 大 SRAM vs GPU 小 SMEM——同一目标的不同约束

ASIC 推理芯片拥有 50–900 MB 片上 SRAM，graph compiler 可以将完整 activation tensor 静态分配到 SRAM，中间结果全程不回 HBM——这是 trivial 的。

GPU shared memory 仅 ~228 KB/SM（比 ASIC SRAM 小 200–4000 倍）。TileRT 无法放完整 activation，必须将其切成 tile，每次只在 SMEM 放一个 tile 的数据，处理完传给下一个 stage 仍在 SMEM，再换下一个 tile——这就是"tile pipeline"存在的根本原因 [L2:tilert-speed-scaling-law]。

ASIC 做法GPU (TileRT) 做法
中间数据存储完整 tensor 在 SRAM1 tile 在 register/SMEM
编译器工作分配 tensor → SRAM 地址规划 tile pipeline stage + SMEM 分配
复杂度低（SRAM 够大就行）高（tile size 受限于 SMEM + register pressure）
等价条件SRAM ≥ max activation sizetile pipeline depth × tile SMEM ≤ SMEM capacity

5.2 硬件异构引擎 vs 软件 Warp Specialization

Gaudi 的 MME（矩阵引擎）+ TPC（通用引擎）+ DMA（数据搬运引擎）是物理上分离的，天然可以并行——graph compiler 只需排好时序。

TileRT 在同构的 NVIDIA SM 上，通过 warp group 分工模拟这种异构 [L2:tilert-speed-scaling-law]：


Gaudi 硬件:              TileRT 软件模拟:
  MME → GEMM              Warp Group B → Tensor Core compute
  TPC → Activation        (融合进同一个 warp group)
  DMA → 数据搬运           Warp Group A → TMA async copy
  NIC → 通信              Warp Group C → NVLink comm
  ↑ 物理并行               ↑ 软件并行（争抢同一 SM 资源）

ASIC 的 overlap 是硬件保证的（不同引擎各有独立执行单元），TileRT 的 overlap 需要精细管理 SMEM bank conflict、register pressure 和 warp scheduler 行为——工程复杂度高一个量级。

5.3 Auto Fusion (XLA/TVM) vs Manual Fusion (TileRT)——为什么自动编译器不够

2018–2021 年的自动 fusion compiler 有结构性限制：

XLA/TVM 的假设LLM bs=1 decode 的现实
GEMM 是 compute-bound → 不融合，调 cuBLASbs=1 时 GEMM 也 memory-bound → 应该融合
AllReduce 是黑盒 NCCL callAllReduce 时间和 compute 可比 → 需要 overlap
同构执行 → 所有线程做相同的事需要 warp specialization → 不同 warp 做不同事
Fusion 边界在 GEMM/control-flow 处切断需要跨 Norm+GEMM+Attention+AllReduce 全融合

TileRT 的解法是放弃自动化，手写 25+ 巨型 fused kernel（如 rmsnorm_projx_wqkvia 1095 行），代价是完全不通用 [L2:tile-ai-tilert]。

5.4 Graph Compile 的根本矛盾：静态编译 vs 动态 Serving

LLM serving 需要的动态性与 graph compile 的静态假设根本冲突：

Graph Compile 前提LLM Serving 现实
静态 shapeKV Cache 每 token 增长
静态计算图MoE 动态路由（input-dependent）
单次执行Continuous batching 动态拼 batch
确定性调度Preemption、priority scheduling
编译一次Prefix sharing 需要运行时判断

这是 XLA 在 LLM serving 走不通的根本原因，也是 vLLM/SGLang（动态 runtime + 单 kernel 优化）成为主流的原因。TileRT 通过限制场景（bs=1、模型固定、硬件固定）绕过了这些矛盾，但代价是极窄的适用范围 [L3:tile-ai-tilert]。

6. 共识与分歧

共识

Launch overhead 在 bs=1 decode 是真实瓶颈。 GPUOS 量化：null kernel launch 3–7 μs，100 ops/token 累计 300–700 μs [L2:2604.17861]。TileRT 验证：理论 ~1000 tok/s vs 实际几十 tok/s [L2:tilert-speed-scaling-law]。所有方案（CUDA Graphs、persistent kernel、megakernel、AOT engine）都以消除这一开销为目标。

静态编译可以有效消除 overhead，代价是动态性。 从 Gaudi graph compile 到 CUDA Graphs 到 TileRT，所有方案验证了同一结论：预先确定执行计划 → 运行时零调度 → latency 接近 compute 本身。

Prefill 阶段收益有限。 Prefill 是 compute-bound（大 seq_len GEMM），launch overhead 占比 <1%。静态编译/fusion 的主要收益在 decode 阶段（memory-bound、μs 级 kernel） [L2:tilert-speed-scaling-law]。

分歧

自动 vs 手动 fusion。 XLA/TVM 走自动 fusion（通用但浅），TileRT 走手动 fusion（极致但不通用）[L2:tile-ai-tilert]。中间地带（compiler hint + partial automation）目前空白。

"新范式" vs "旧酒新瓶"。 TileRT 博客将 Persistent Engine Kernel 定位为推理系统的范式转移 [L2:tilert-speed-scaling-law]。但从 ASIC graph compile 的历史视角看，核心原理（静态编译 + 片上驻留 + 异构 overlap）至少追溯到 2018 年 Goya/TPU，TileRT 的增量贡献是在通用 GPU 上用软件复现了这些效果。

是否需要 rewrite everything。 TileRT 路线要求全栈重写（compiler IR + tile scheduler + 通信原语）[L3:tile-ai-tilert]。对立观点：FlashAttention + CUDAGraph + 少量 fused kernel 已经覆盖了 80% 收益，剩余 20% 不值得全栈投入。代码实际也显示 TileRT 开源层正是用 CUDAGraph + fused ops 实现的 [L2:tile-ai-tilert]。

7. 根本性困难

7.1 静态编译 vs 动态 serving 的不可调和

KV Cache 动态增长、continuous batching、MoE 动态路由、prefix sharing 等 serving 核心特性都依赖运行时决策。Graph compile 路线必须放弃这些能力或引入复杂的 workaround（bucketed compilation、padding、conditional graph）。

目前无解：要么极端静态化（TileRT 的 bs=1 固定模型路线），要么放弃 graph compile（vLLM/SGLang 路线）。中间的 hybrid（静态骨架 + 动态插槽）是理论上的方向但无成熟实现 [L3:tilert-speed-scaling-law]。

7.2 Auto Fusion 到 GEMM+通信的扩展

XLA/TVM 的 fusion 止步于 GEMM 和通信边界。将自动 fusion 扩展到 bs=1 memory-bound GEMM 和 NVLink 通信需要：新的 cost model（识别 bs=1 GEMM 的 memory-bound 特性）、新的 codegen 能力（warp specialization）、新的通信抽象（tile-level broadcast/reduce）。TileLang/TileScale（TileRT 团队规划的编译器）可能是这个方向的尝试 [L2:tile-ai-tilert]。

7.3 GPU 片上内存的物理限制

GPU SMEM 容量受 die area 约束，短期内不会增长到 ASIC SRAM 水平。Tile pipeline 的深度受限于 SMEM 容量和 register pressure——这是物理限制而非工程限制。除非 GPU 架构发生根本变化（如 NVIDIA 引入大容量可编程 scratchpad），否则 tile pipeline 的复杂度无法从根本上简化。

8. 成熟度判断

路线成熟度趋势
ASIC graph compile (Gaudi/TPU)生产成熟，但 LLM serving 市场份额极低停滞——NVIDIA GPU 生态垄断
XLA for LLM inference不适用——结构性限制无法解决被 vLLM/SGLang 替代
CUDA Graphs生产就绪，vLLM 已集成稳定——作为 optimization pass 而非完整方案
vLLM/SGLang (动态路线)社区主流，生产广泛部署持续加速——占据通用 serving 市场
TileRT (极端静态路线)单一模型厂商生产 [L2:tile-ai-tilert]细分赛道——模型厂商自研引擎
Auto-compiler for LLM decode研究早期（TileLang/TileScale 规划中）观望——是否能自动生成 TileRT 级代码未知

整体判断：静态编译路线在 LLM inference 的适用范围极窄（bs=1、模型固定、硬件固定），不会成为通用方案。其价值存在于"模型厂商极致优化自有模型"的场景——类似 Google 用 TPU 服务自家模型。通用 serving 市场将继续由动态 runtime (vLLM/SGLang) 主导。
