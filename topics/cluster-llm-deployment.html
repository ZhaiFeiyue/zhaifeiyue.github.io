Cluster LLM 部署：存储、计算、执行与成本

§1 主题缘起

大规模 LLM 集群部署正在经历从"单节点单模型"到"千 GPU 级多模态流水线"的范式跃迁。这一跃迁受到四股力量的同时驱动。

存储维度：Agentic workload 的兴起使 KV-Cache 命中率极高（DeepSeek 生产 coding agent trace 显示 98.7% 命中率 [L2:2602.21548]），prefill 从计算密集退化为 I/O 密集。与此同时，PD 分离架构将所有存储读取集中在 prefill engine 的 SNIC 上，使存储带宽成为单点瓶颈 [L2:2602.21548]。新一代 hybrid attention 模型（KDA:MLA 混合）将单实例 KV 吞吐从 60 Gbps 压缩到 3 Gbps，使跨数据中心 PD 分离从不可行变为可行 [L2:2604.15039]。RL 训练中的权重传输同样面临 TB 级数据搬运的挑战——Ray object store 传 40 GB 需要 32 秒，而 GPU-direct RDMA 只需 0.2 秒 [L2:2604.09107]。

计算维度：硬件规格的代际提升未能自动转化为实际性能。MI300X 虽然理论 FP8 峰值 2615 TFLOPS 超过 H100 的 1979 TFLOPS，但实际利用率仅 45%（对比 NVIDIA 93%），根因是软件栈成熟度不足 [L2:2510.27583]。Blackwell B200 通过 TMEM 和第五代 Tensor Core 实现了从训练专用到推理优化的架构转换 [L2:2512.02189]，但 BSP 编程模型在分布式执行中引入三类性能税——kernel launch、全局同步、数据局部性丧失——消耗掉潜在的并行加速 [L2:2511.02168]。MoE 架构下，跨节点 all-to-all 通信几乎等于计算时间 [L2:2412.19437]。

执行效率维度：即使存储和硬件不是瓶颈，kernel 间的执行边界本身也在消耗一个数量级的性能。8×H200 NVL 聚合内存带宽 ~38 TB/s，GLM-5.1 每 decode token 激活参数量 ~42 GB，纯带宽约束下理论吞吐可达 ~1000 tok/s，而实际系统仅输出几十 tok/s [L2:tilert-speed-scaling-law]。根因不是算力不足，而是 GPU 反复执行 launch→load→compute→store→synchronize 循环，每次 kernel boundary 打断数据流、销毁局部性、强制重新同步 [L2:tilert-speed-scaling-law]。这一发现将优化目标从"如何让每个 kernel 更快"扩展到"如何消除 kernel 边界本身"，催生了 AOT 编译全模型为单个持久化 Engine Kernel 的新范式。

成本维度：SLO 达标率与吞吐量之间的 Pareto 前沿尚未被有效推进。Colocation 方案在 P99 TBT 上表现糟糕，disaggregation 方案的 GPU 利用率低至 0.2% [L2:2504.09285]。Agentic workflow 将单个用户查询拆成 19–21 次 LLM 调用，现有独立请求调度器既不懂阶段依赖也不懂异构 GPU 分配 [L2:2505.05286]。配置空间爆炸使手动调优需要数天 GPU 时间 [L2:2601.06288]。

这四个维度的矛盾在 2024–2026 年同时激化，催生了本主题的 18 篇跨 framework/hardware/cluster/kernel 四大类别的系统性工作。

关键谱系节点：

  • Three Taxes → Iris → TileRT 的持续执行演进链：BSP 模型是分布式 GPU 编程的事实标准 [L2:2511.02168]，Three Taxes 诊断了三类性能税，Iris 用 in-kernel primitive 在 tile 级别融合 compute-comm，而 TileRT 将这一思路推至逻辑极限——AOT 编译将整个模型静态展开为单个 Persistent Engine Kernel，host 仅启动一次，所有调度和通信在 GPU 内部持续推进 [L2:tilert-speed-scaling-law]。这条演进链从"消除单个 kernel 内的低效"走到"消除 kernel 边界本身"。
  • DeepSeek-V3 DualPipe 的 warp specialization 向 TileRT 的 GPU-level specialization 扩展：DualPipe 在 SM 级别用 20/132 SM（15%）专门分配给通信 [L2:2412.19437]，TileRT 将这一思路从 warp 级扩展到 GPU 级——GPU0 运行 Sparse Indexer Worker，GPU1–7 运行 MLA Workers，通信从 NCCL 外部编排迁入 tile pipeline 内部 [L2:tilert-speed-scaling-law]。
  • DualPath 与 TensorHub 独立发现同一洞察：集群中存在大量空闲 NIC 带宽可通过 DAG 拓扑聚合。DualPath 利用 DE 端 SNIC 空闲帮 PE 读 KV-Cache [L2:2602.21548]，TensorHub 利用正在接收权重的 rollout worker NIC 上行空闲帮后续 worker 读 weights [L3:2604.09107]。
  • DeepSeek-V3 DualPipe 是多条演进线的汇聚点：细粒度组件拆分 + 双向调度实现 all-to-all 完全隐藏 [L2:2412.19437]，其通信-计算重叠思想被 DynaServe、fabric-lib、TileRT 等后续工作继承。

§5 技术线交错

5.1 存储选择如何约束计算架构

KV-Cache 的存取方式直接决定了 GPU 的有效利用率。DualPath 的 CNIC-centric 数据通路设计本质上是对"当前 GPU 不支持 PCIe QoS"这一硬件缺陷的软件 workaround——所有 H2D/D2H 必须绕路走 RDMA NIC 才能实现流量隔离 [L2:2602.21548]。这意味着未来 GPU 如果原生支持 PCIe QoS lanes，整个 CNIC 绕路设计可以被简化。反之，PrfaaS 的层间 prefill pipelining 要求 execution engine 能在每层计算完成后立即发起 KV 传输 [L2:2604.15039]——这对现有 vLLM/SGLang 的"全 prefill 完成再传输"模式是侵入性改动。

5.2 硬件能力如何开启新框架设计

MI300X 的 192 GB HBM 允许更大模型单卡部署，减少 tensor parallelism 需求 [L2:2510.27583]。Blackwell 的 FP4 达到 7702.5 TFLOPS（96.3% 峰值）[L2:2512.02189]，使 ops:byte 比急剧升高，推动更多 workload 从 memory-bound 转向 compute-bound——这直接影响 AIConfigurator 的配置搜索空间，因为 disaggregated 配置在 memory-bound decode 场景价值更大 [L2:2601.06288]。NVSwitch 的全互联拓扑 vs Infinity Fabric 的 mesh 拓扑差异 [L2:gpu-arch-comparison-2025] 使得 fabric-lib 必须设计跨 vendor 的通信抽象 [L2:2510.27656]。

5.3 模型架构如何强制部署重新思考

DeepSeek-V3 的 256 routed experts + 64-way EP 使跨节点 all-to-all 通信时间几乎等于计算时间 [L2:2412.19437]。DualPipe 通过双向调度和 warp specialization 将 20/132 SM（15%）专门分配给通信才能实现完全隐藏 [L2:2412.19437]。Hybrid attention（KDA:MLA 混合）将 KV 吞吐压缩 4–13× [L2:2604.15039]，使 PrfaaS 的跨 DC 方案成为可能——但这反过来要求 PD 集群内必须同构才能做 P↔D 角色转换 [L2:2604.15039]。MoE 的动态 token 路由使静态 expert 放置注定产生 straggler，推动了运行时负载均衡算法的发展 [L2:moe-lb-interai25]。

5.4 持续执行范式如何跨越 framework-kernel-hardware 边界

TileRT 的 Persistent Engine Kernel 是 framework 层（系统设计）和 kernel 层（GPU 编程模型）深度融合的产物——AOT 编译器将整个 transformer 模型静态展开为单个 kernel，在编译期规划 tile 调度、warp 特化和通信时序 [L2:tilert-speed-scaling-law]。这一设计隐式依赖 hardware 层的能力：persistent kernel 通过软件 trick 占据 SM 不释放实现 GPU-resident 执行，如果硬件提供原生 persistent CTA 支持可以消除工程脆弱性；NVLink 上的 tile-level 通信目前基于 flag polling，原生硬件同步原语可以降低延迟 [L2:tilert-speed-scaling-law]。同时，Three Taxes 的 BSP 性能税诊断 [L2:2511.02168] 直接激发了 TileRT "消除所有 kernel 边界"的设计目标。这形成了一条从 cluster（性能税诊断）到 kernel（in-kernel fusion）到 framework（全模型 AOT）再到 hardware（硬件原语反推）的完整技术线交错。

5.5 调度器与通信库的纵向协同

fabric-lib 证明 P2P 通信是 collective 的必要补充 [L2:2510.27656]，而 TensorHub 的 ROS 利用 fabric-lib 类的 RDMA 引擎实现零 trainer stall [L3:2604.09107]。在推理侧，KVFlow 的 workflow-aware 驱逐需要 serving 层开始感知 application-layer graph [L2:2507.07400]，JITServe 则进一步要求调度器理解多阶段 compound 请求的 DAG 依赖 [L2:2504.20068]。TileRT 以相反方向推进这一趋势——将通信从外部编排"下沉"到 GPU-resident pipeline 内部，整个 attention 层从 host 视角仅为一次 Engine Kernel launch [L2:tilert-speed-scaling-law]。"应用语义下沉到基础设施"（KVFlow/JITServe）与"基础设施编排下沉到 GPU"（TileRT）是同一趋势的两个互补方向。

§6 共识与分歧

共识

  1. PD 分离是必要的基础架构：所有 18 篇工作中涉及推理 serving 的 8 篇（DualPath、PrfaaS、DynaServe、JITServe、KVFlow、HexGen-Flow、Scepsy、TileRT）均建立在 prefill 与 decode 分离或至少 decode-first 优化的前提上 [L2:2602.21548] [L2:2604.15039] [L2:2504.09285]。TileRT 虽以 decode 延迟为核心目标而非显式 PD 分离，其设计本质上是 decode-first 的极致优化 [L2:tilert-speed-scaling-law]。
    1. RDMA 是大规模 GPU 集群数据搬运的必需原语：DualPath [L2:2602.21548]、TensorHub [L2:2604.09107]、fabric-lib [L2:2510.27656] 三篇独立工作均选择 RDMA 作为核心传输机制。
      1. Agentic workload 正在重塑 serving 系统假设：KVFlow 证明 LRU 在 agentic workflow 中系统性驱逐热数据 [L2:2507.07400]，DualPath 展示 agentic trace 的 98.7% KV-Cache 命中率 [L2:2602.21548]，HexGen-Flow 指出单个 Text-to-SQL 查询触发约 20 次 LLM 调用 [L2:2505.05286]。TileRT 则从另一角度呼应：Agent/TTS/voice 场景推动近 BS=1 延迟优先工作负载成为主流，批处理摊销空间被压缩 [L2:tilert-speed-scaling-law]。
        1. 软件栈效率是当前 AMD 平台的主要瓶颈，而非硬件设计：MI300X 的频率-效率分离实验精确量化了软件效率 80–85% 和频率降低约 42% 各自的贡献 [L2:2510.27583]。
          1. Inter-kernel overhead 在低 batch size 下是性能主要瓶颈：Three Taxes 将 BSP 模型的开销分解为 kernel launch tax、bulk synchronous tax、data locality tax 三类 [L2:2511.02168]，TileRT 量化了 8×H200 NVL 上理论 ~1000 tok/s 与实际几十 tok/s 的量级差距并归因于 inter-kernel idle [L2:tilert-speed-scaling-law]，ConCCL 的 DMA offload 从另一角度验证了 compute interference 是 kernel 并发执行的关键障碍 [L2:2412.14335]。三篇工作从诊断、消除、融合三个层面达成共识。

            2. 分歧

            1. 存储层方案选择：DRAM pool vs SSD + NIC 聚合 vs 模型压缩。Mooncake 用分布式 DRAM pool 缓存 KV-Cache，但 DualPath 论证 DRAM 在 RL training rollout 时被 optimizer state offload 占用，且 working set 成本以 $r^3$ 扩展 [L2:2602.21548]，因此选择 SSD 存储 + NIC 带宽聚合路线。PrfaaS 则走第三条路——hybrid attention 压缩 KV 体积后用 commodity Ethernet 跨 DC 传输 [L2:2604.15039]。三者假设的硬件基础和成本结构完全不同。
              1. 通信融合粒度：kernel-level vs framework-level vs full-model AOT。Three Taxes 工作主张将通信逻辑直接嵌入 compute kernel 实现 tile-level pipeline [L2:2511.02168]，DeepSeek-V3 的 DualPipe 在 framework 层通过双向调度实现组件级重叠 [L2:2412.19437]，TileRT 则走到极端——AOT 将完整模型编译为单个 Persistent Engine Kernel，通信从 NCCL 外部编排迁入 tile pipeline 内部 [L2:tilert-speed-scaling-law]。三者在 MI300X/H800/H200 NVL 上分别获得 10–20%、约 60%+ 和（声称的）量级级别的改善，但设计哲学从增量到激进逐级升级——前者需重写单个 kernel，中者需全局调度逻辑，后者需从零设计 compiler IR、tile scheduler、通信原语和执行模型。
                1. GPU-initiated vs host-proxy RDMA。DeepEP 用 IBGDA（GPU-initiated RDMA）[L2:2510.27656]，fabric-lib 证明 host proxy 方案在 ConnectX-7 上反而超越 GPU-initiated（因 bulk transfer 和流水线策略补偿了 PCIe 往返延迟）[L2:2510.27656]。TileRT 走第三条路——彻底绕开 RDMA，通信嵌入 persistent kernel 内部直接通过 NVLink 在 GPU 间执行 [L2:tilert-speed-scaling-law]。这一分歧反映了不同通信范围的最佳策略不同：NVLink 域内持续执行、跨节点 host-proxy P2P、特殊硬件 GPU-initiated 各有适用场景。
                  1. FP8 vs FP16 的精度-性能权衡。MI300X 上 FP16 推理相对表现优于 FP8，因 FP16 的更大 memory footprint 恰好落在 MI300X 高带宽区间 [L2:2510.27583]。这与 Blackwell 上 FP4 达到 96.3% 峰值利用率但 perplexity 退化 8.2% 的发现 [L2:2512.02189] 形成对照——精度选择高度依赖硬件平台和工作负载特征。
                    1. AOT 静态编译 vs JIT 动态调度。TileRT 用 AOT 将完整 transformer 编译为单个 Engine Kernel，运行时不存在 host-device 交互 [L2:tilert-speed-scaling-law]。GPUOS 采取对立策略——JIT 动态 operator injection 保留 PyTorch eager semantics，大操作仍走传统 launch [L3:tilert-speed-scaling-law]。TileRT 的代价是放弃了动态性——模型结构变更需要 recompile，而 MoE routing 的 input-dependent 路径决策、KV cache 的动态增长、speculative decoding 的条件分支都对静态编译构成结构性挑战 [L3:tilert-speed-scaling-law]。Fleet 在 AMD MI350 上的 persistent megakernel 是两者之间的中间地带——per-operator 编译但 per-chiplet 软件调度 [L3:tilert-speed-scaling-law]。

                      2. §7 Open challenges (根本性困难)

                      7.1 KV-Cache 压缩与 attention 表达力的信息论瓶颈

                      Hybrid attention（linear + full-attention 混合）可将 KV 吞吐降低 4–13× [L2:2604.15039]，但这是模型架构层面的折衷——linear attention 丢失了 full-attention 的 long-range dependency 建模能力。没有理论工具量化"KV-Cache 压缩到什么程度会导致不可接受的下游任务退化"。这不是工程 gap 而是信息论问题：KV-Cache 是 attention 计算的中间表示，其最优压缩率取决于输入序列的互信息结构。解决此问题需要 model（注意力机制设计）和 framework（serving 系统）的联合优化，预估难度：3–5 年。

                      7.2 异构硬件下的通用性能可移植

                      DualPath 的 CNIC-centric 方案深度绑定 InfiniBand VL QoS [L2:2602.21548]；DeepSeek-V3 的 all-to-all kernel 绑定 H800 NVLink+IB 拓扑 [L2:2412.19437]；ConCCL 绑定 AMD MI300X 的 SDMA 引擎 [L2:2412.14335]；TileRT 的 Persistent Engine Kernel 深度绑定 8×H200 NVL 的 NVLink 域和 TMA 异步引擎 [L2:tilert-speed-scaling-law]。fabric-lib 尝试统一 ConnectX-7 和 EFA [L2:2510.27656]，但仍受限于两种 NIC。当前没有任何通信库或执行引擎能跨 InfiniBand、RoCE、EFA、CXL 和不同 GPU 架构提供一致的高性能语义。这需要 hardware（标准化接口）和 cluster（统一抽象）层的协同，预估难度：2–3 年。

                      7.3 动态 agentic workflow 的在线资源分配

                      KVFlow 假设 workflow graph 在运行时稳定可观测 [L2:2507.07400]，但 ReAct/Reflexion 等自主 agent 在运行时动态生成下一步，Agent Step Graph 无法在请求时刻给出完整拓扑。JITServe 用"保守估计 + 在线精化"部分缓解此问题 [L2:2504.20068]，但 compound 请求的依赖图本质上是 runtime-generated。Scepsy 的"相对份额稳定性"假设 [L2:2604.15186] 在 steady-state 下成立，对 workload 剧变时段可能失效。解决此问题需要 agent（运行时暴露结构信息）和 framework（adaptive 资源分配）的联合设计，预估难度：1–3 年。

                      7.4 千 GPU 级容错与一致性

                      DualPath 部署于 1152 GPU [L2:2602.21548]，TensorHub 部署于 1024 GPU [L2:2604.09107]，DeepSeek-V3 训练用 2048 GPU [L2:2412.19437]——但这些系统的容错机制均未充分披露。TensorHub 的中心化 reference server 在 10K+ GPU 规模下可能成为瓶颈 [L3:2604.09107]，DualPath 论文完全未讨论 server failure 对跨 DC 场景的影响。Pipeline replication 中 partial-replica 的故障恢复涉及 RDMA 层 4 秒保守超时后的状态清理 [L2:2604.09107]。这不是单一系统能解决的问题，需要 framework（分布式状态管理）+ cluster（网络容错）+ hardware（可靠性保证）的共同推进，预估难度：3–5 年。

                      7.5 AOT 静态编译 vs 动态工作负载的根本性张力

                      TileRT 的 AOT 全模型编译实现了 inter-kernel overhead 的彻底消除，但现代 LLM 的动态性不断增加：MoE routing 每 token 激活不同 expert（input-dependent），KV cache 随序列长度动态增长，speculative decoding 的 draft/verify 路径引入条件分支 [L3:tilert-speed-scaling-law]。AOT 编译必须预编译所有可能路径或在 kernel 内部实现动态分发——前者编译产物爆炸，后者本质上回到了运行时调度。Fleet 的 persistent megakernel 在 batch size ≥ 64 时被 hipBLASLt 反超 [L3:tilert-speed-scaling-law]，证明 compute-bound 区间单 kernel 极致调优优于跨 kernel 持久化。这不是工程问题——静态优化与动态负载之间存在根本性的 information-theoretic 张力：编译器在编译期能获取的信息严格少于运行时。解决此问题需要 framework（AOT/JIT 混合执行）+ kernel（adaptive fusion 粒度）+ hardware（原生 persistent CTA 支持）的协同创新，预估难度：2–4 年。

                      §9 邻接 topic

                      1. Agent Serving（agent 运行时优化）：与本 topic 在 agentic workload characterization 上深度重叠（DualPath/KVFlow/JITServe/HexGen-Flow 均以 agentic 场景为核心评估）。边界在于本 topic 聚焦"集群级资源管理"，agent-serving 更关注"agent 运行时的推理引擎优化"。两者可能合并为更广义的"Agentic Infrastructure"topic。
                        1. RDMA 通信（跨硬件可移植通信）：fabric-lib 和 TensorHub 的 RDMA 层工作同时属于本 topic（作为存储/计算的使能技术）和独立的通信 topic。当更多跨 vendor RDMA 工作出现时，可能需要独立的"RDMA-comms"topic。
                          1. MoE 推理优化：DeepSeek-V3、MoE 负载均衡、fabric-lib 的 MoE dispatch/combine 均涉及 MoE 特有的 expert parallelism 挑战。目前包含在本 topic 的计算维度中，但随着 MoE 架构成为主流，可能需要独立 topic。
                            1. GPU 架构演进：MI300X/Blackwell/GPU 对比三篇提供硬件基础约束。更深入的 ISA 级分析和下一代架构（CXL、photonic interconnect）属于独立的 hardware-evolution topic。
                              1. Persistent Kernel 执行范式：TileRT、GPUOS、Fleet 构成了一条从 per-op fusion 到 full-model persistent engine 的技术演进链 [L3:tilert-speed-scaling-law]。这一方向与本 topic 在"执行效率优化"上重叠，但其编译器设计、tile scheduling 理论、chiplet-aware 调度等问题跨越了 framework/kernel 边界，可能在更多相关工作出现后独立为"persistent-execution"topic。
