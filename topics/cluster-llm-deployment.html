Cluster LLM 部署：存储、计算、执行与成本 #

§1 主题缘起 #

大规模 LLM 集群部署正在经历从"单节点单模型"到"千 GPU 级多模态流水线"的范式跃迁。这一跃迁受到五股力量的同时驱动。

存储维度：Agentic workload 的兴起使 KV-Cache 命中率极高（DeepSeek 生产 coding agent trace 显示 98.7% 命中率 [L2:2602.21548]），prefill 从计算密集退化为 I/O 密集。PD 分离架构将所有存储读取集中在 prefill engine 的 SNIC 上，使存储带宽成为单点瓶颈 [L2:2602.21548]。新一代 hybrid attention 模型（KDA:MLA 混合）将单实例 KV 吞吐从 60 Gbps 压缩到 3 Gbps，使跨数据中心 PD 分离从不可行变为可行 [L2:2604.15039]。KV-Cache 传输瓶颈催生了多路径解决方案——KVServe 用 service-aware 压缩实现最高 10× KV 数据削减 [L2:kvserve]，PPD 发现 Turn 2+ 的 append-prefill 仅造成 2% TPOT 劣化（vs full prefill 的 48%），可安全回路到 decode 节点 [L2:2603.13358]。RL 训练中的权重传输同样面临 TB 级数据搬运挑战——Ray object store 传 40 GB 需要 32 秒，而 GPU-direct RDMA 只需 0.2 秒 [L2:2604.09107]。

计算维度：硬件规格的代际提升未能自动转化为实际性能。MI300X 虽然理论 FP8 峰值 2615 TFLOPS 超过 H100 的 1979 TFLOPS，但实际利用率仅 45%（对比 NVIDIA 93%），根因是软件栈成熟度不足 [L2:2510.27583]。Blackwell B200 通过 TMEM 和第五代 Tensor Core 实现了从训练专用到推理优化的架构转换，FP4 达到 96.3% 峰值利用率 [L2:2512.02189] [L2:2512.02189]。AMD CDNA4 (MI355X) 以 288 GB HBM3E 和 5 PFLOPS FP8 回应，但互联带宽落后 NVLink5 达 67% [L2:amd-cdna4-whitepaper] [L2:gpu-arch-comparison-2025]。BSP 编程模型在分布式执行中引入三类性能税——kernel launch、全局同步、数据局部性丧失——消耗掉潜在的并行加速 [L2:2511.02168]。MI300X 的 chiplet 架构进一步引入 NUMA 效应：FlashAttention2 在 8-XCD 上 L2 命中率仅 ~1%，swizzled head-first mapping 可恢复至 90–97% 并加速 50% [L2:2511.02132]。MoE 架构下，跨节点 all-to-all 通信几乎等于计算时间 [L2:2412.19437]。Ops:Byte 比率从 A100 的 153 (FP16) 攀升到 MI355X/B200 的 ~315/281 (FP16) 和 629/563 (FP8)，MXFP4 更达 1258 [L2:gpu-arch-comparison-2025]——除大矩阵 GEMM 外几乎所有操作 memory-bound，FP4 的真正价值在于用更少 bit 传输相同语义信息。

执行效率维度：即使存储和硬件不是瓶颈，kernel 间的执行边界本身也在消耗一个数量级的性能。8×H200 NVL 聚合内存带宽 ~38 TB/s，GLM-5.1 每 decode token 激活参数量 ~42 GB，纯带宽约束下理论吞吐可达 ~1000 tok/s，而实际系统仅输出几十 tok/s [L2:tilert-speed-scaling-law]。根因不是算力不足，而是 GPU 反复执行 launch→load→compute→store→synchronize 循环，每次 kernel boundary 打断数据流、销毁局部性、强制重新同步 [L2:tilert-speed-scaling-law]。这催生了三条并行的执行范式创新：TileRT 将整个 transformer AOT 编译为单个持久化 Engine Kernel [L2:tile-ai-tilert]，TokenSpeed 用 C++ FSM + Placement 编译器在 Blackwell 上超越 TRT-LLM [L2:tokenspeed]，Fleet 为 AMD MI350 多 die 设计了 hierarchical megakernel [ref:L2:2604.15379]。GPUOS 则走运行时路线——persistent kernel + lock-free ring buffer 将 per-op dispatch 从 3–7 μs 降至 <100 ns [L2:2604.17861]。GPU kernel 优化领域同期也从手工调优走向 AI-augmented 自动搜索——AVO 的 agentic evolution 超越 cuDNN 闭源峰值达 1668 TFLOPS [L3:kernel]，量化 attention 沿 INT8→INT4→FP4 路线在五代 SageAttention 中实现了从 2.1× 到 5× over FlashAttention2 的持续加速 [L3:kernel]。

成本维度：SLO 达标率与吞吐量之间的 Pareto 前沿尚未被有效推进。Colocation 方案在 P99 TBT 上表现糟糕，disaggregation 方案的 GPU 利用率低至 0.2% [L2:2504.09285]。MoE disaggregated serving 面临 3 类网络流量（KV-cache reuse、collective 通信、P2D 传输）的竞争，传统 FIFO 调度无法区分流量优先级 [L2:2603.17456]。ZeRO-Prefill 反转 MoE 数据流方向——从"路由激活到 expert"改为"流式传输 expert 权重到 GPU"——消除了 AllToAll 通信，MoE prefill 吞吐提升 1.35–1.59× [L2:2605.02960]。配置空间爆炸使手动调优需要数天 GPU 时间 [L2:2601.06288]。Justitia 引入 KV token-time（内存占用 × 时间）作为公平性度量，平均 JCT 减少 57.5% [ref:L2:2510.17015]。Harli 发现 decode 阶段 SM 利用率仅 ~40%，通过 inference+PEFT co-location 收割闲置算力实现 46% 额外推理吞吐——将"同 GPU 不同 workload 类型共存"作为新的利用率提升维度 [L2:2511.11729]。FaaSMoE 走更极端的解耦路线——将 MoE expert 部署为 stateless FaaS 函数实现多租户 expert 共享和 scale-to-zero，尝试用 serverless 范式替代 dedicated GPU serving [L2:2604.26881]。MARLIN 将优化目标从纯性能扩展到 sustainability——首次将碳排放、水消耗、电力成本与 TTFT 联合优化，在全球 geo-distributed DC 间做 game-theoretic routing [L2:2605.13496]。Parallax 则打破了"管控集群"这一基本假设——在 volunteer WAN GPU pool 上用 decentralized DHT + two-phase DP 实现异构推理，3.6× 超越 HexGen [L2:2509.26182]。

Agentic 基础设施维度：Agentic workflow 将单个用户查询拆成 19–21 次 LLM 调用 [L2:2505.05286]，但现有独立请求调度器既不懂阶段依赖也不懂异构 GPU 分配。高并发 agentic batch inference 中 KV-Cache thrashing 使中间阶段 49% 的延迟来自 KV 重计算，Concur 用 AIMD 准入控制将 KV 命中率从 35% 稳定在 73–96% [ref:L2:2601.22705]。ThunderAgent 引入"agentic program"作为一等调度单元，实现 1.48–3.58× serving 吞吐提升 [ref:L2:2602.13692]。Helium 借鉴数据库查询优化器思路，用 Templated Radix Tree 做 plan-level prefix reuse 优化 [ref:L2:2603.16104]。Halo 将 batch agentic workflow 整合为 DAG 统一调度，比 naive vLLM 加速 400× [ref:L2:2509.02121]。

这五个维度的矛盾在 2024–2026 年同时激化，催生了本主题 41 篇跨 framework/hardware/cluster/kernel 四大类别的系统性工作。

Framework 占绝对多数（30/41），反映当前优化空间主要在系统软件层。Framework 类别正经历从"高吞吐 batch serving"到"agentic workload 全生命周期编排"的范式转变 [L3:framework]——agentic serving 子方向涌现 6 篇（ThunderAgent、Helium、Halo、Concur、HexGen-Flow、KVFlow），MoE 执行优化分化为 ZeRO-Prefill（消除 AllToAll）、MFS（调度 AllToAll）和 FaaSMoE（expert-as-FaaS 去状态化）三条路线，超低延迟引擎从 TileRT 一枝独秀扩展为 TileRT/TokenSpeed/Fleet 三足鼎立 [L3:framework]。Harli 开辟了 inference+training co-location 的新子方向——收割 decode 结构性闲置 SM [L2:2511.11729]；Parallax 开辟了 decentralized/volunteer GPU serving 的新部署范式 [L2:2509.26182]。Kernel 类别从 1 篇扩展到 4 篇，反映 AMD chiplet 架构催生了大量 vendor-specific 优化（NUMA-Attn、HipKittens、ConCCL）和跨 vendor 运行时创新（GPUOS）[L3:kernel]。Hardware 增加了 CDNA4 白皮书，标志 AMD/NVIDIA 竞争进入 FP4/MXFP4 精度时代——两家 FP8 峰值算力已基本持平（~5 PFLOPS），竞争焦点从纯算力转向互联规模、显存容量、能效比和软件生态的全栈博弈 [L3:hardware]。Cluster 类别从 2 篇扩展到 3 篇——MARLIN 在 fabric-lib（intra-node data transfer）和 Three Taxes（intra-node kernel fusion）的基础上新增了 inter-DC meta-scheduling 层，使 cluster 类目覆盖从节点内到全球的完整调度栈 [L2:2605.13496]。fabric-lib 和 Three Taxes/Iris 恰好占据通信 taxonomy 的对角位置——inter-node + library-level + async P2P vs intra-node + in-kernel + fine-grained sync——交集区域（跨节点 in-kernel fusion）为空白 [L3:cluster]。

关键谱系节点：

  • Three Taxes → Iris → TileRT/TokenSpeed/Fleet 的持续执行演进分化：BSP 模型是分布式 GPU 编程的事实标准 [L2:2511.02168]，Three Taxes 诊断了三类性能税，Iris 用 in-kernel primitive 在 tile 级别融合 compute-comm。后续三个系统在不同硬件上发展了不同的极致优化路线——TileRT 在 NVIDIA NVL 上 AOT 编译全模型为单个 Engine Kernel [L2:tile-ai-tilert]，TokenSpeed 在 Blackwell 上用 Placement 编译器做 MLA 专用优化 [L2:tokenspeed]，Fleet 在 AMD MI350 上设计 chiplet-task 抽象和 hierarchical megakernel [ref:L2:2604.15379]。执行范式从统一范式分化为三条 vendor-specific 技术路线。
  • GPUOS vs TileRT 的运行时 vs 编译时张力：GPUOS 保留 PyTorch eager semantics，用 JIT 动态 operator injection 实现 <100 ns dispatch [L2:2604.17861]；TileRT 用 AOT 静态编译将运行时 host-device 交互降至零 [L2:tile-ai-tilert]。两者代表了 persistent kernel 范式的两种极端设计——轻量级 dispatcher vs 重量级 megakernel [L3:2604.17861]。
  • DualPath 与 TensorHub 独立发现同一洞察：集群中存在大量空闲 NIC 带宽可通过 DAG 拓扑聚合。DualPath 利用 DE 端 SNIC 空闲帮 PE 读 KV-Cache [L2:2602.21548]，TensorHub 利用 RDMA full-duplex 实现零 trainer stall 的 pipeline replication [L3:2604.09107]。ZeRO-Prefill 利用 prefill 计算窗口做背景 weight AllGather [L3:2605.02960]。三个系统在不同场景（推理 KV、训练权重、MoE expert）独立发现了同一优化原理。
  • DeepSeek-V3 DualPipe 是多条演进线的汇聚点：细粒度组件拆分 + 双向调度实现 all-to-all 完全隐藏 [L2:2412.19437]，其通信-计算重叠思想被 DynaServe、fabric-lib、TileRT、ZeRO-Prefill 等后续工作继承。ZeRO-Prefill 是 DualPipe 的 serving 等价物——将 prefill 大 batch 计算窗口替代 backward pass，用 NVLink D2D AllGather 替代 SM 内 warp specialization [L3:2605.02960]。
  • KVFlow → Helium → Concur 的 agentic cache 演进链：KVFlow 发现 LRU 在 agentic workflow 中系统性驱逐热数据，提出 workflow-aware STE [L2:2507.07400]；Helium 借鉴数据库查询优化器做 plan-level prefix reuse，超越 KVFlow 1.56× [ref:L2:2603.16104]；Concur 从拥塞控制角度将 KV-Cache 建模为有限共享资源，用 AIMD 做准入控制 [ref:L2:2601.22705]。三者分别从驱逐策略、预取策略、准入策略三个互补角度解决同一问题 [L3:2602.21548]。
  • CDNA3 chiplet NUMA 效应驱动两条 AMD-specific 优化链：NUMA-Attn 的 swizzled head-first mapping 解决了 L2 cache 局部性 [L2:2511.02132]，HipKittens 的 chiplet-aware XCD scheduling 进一步将 L2 命中率提升 19% [L2:2511.08083]，两者共同为 Fleet 的 hierarchical megakernel 提供了 chiplet 感知的基座。NVIDIA Blackwell 通过 NV-HBI 10 TB/s 在硬件层对软件透明地解决了 dual-die NUMA [L2:2512.02189]，两家在 chiplet 治理上的哲学差异（硬件透明 vs 软件显式）将长期影响上层系统设计 [L3:hardware]。

§5 技术线交错 #

5.1 存储选择如何约束计算架构 #

KV-Cache 的存取方式直接决定了 GPU 的有效利用率。DualPath 的 CNIC-centric 数据通路设计本质上是对"当前 GPU 不支持 PCIe QoS"这一硬件缺陷的软件 workaround [L2:2602.21548]。PrfaaS 的层间 prefill pipelining 要求 execution engine 能在每层计算完成后立即发起 KV 传输 [L2:2604.15039]。KVServe 的 service-aware 压缩在 PD 通路上引入了新的 trade-off：10× KV 压缩使低带宽链路可行，但需要 decode 端的解压开销 [L2:kvserve]。PPD 则揭示了一个更精细的约束——append-prefill 与 full prefill 的干扰模式完全不同，前者仅 2% TPOT 劣化，使 Turn 2+ 可安全 co-locate 于 decode 节点 [L2:2603.13358]。这意味着 PD 分离的最优策略不是二元选择，而是请求类型感知的动态路由——PPD 测量的 2% 干扰来自 Llama-3.1-8B + 4×H100，但 MoE 超大模型（DeepSeek-660B）的 I/O 密集 prefill 干扰模式完全不同——瓶颈不是计算干扰而是 storage I/O 带宽 [L3:2602.21548]。

5.2 硬件能力如何开启新框架设计 #

MI300X 的 192 GB HBM 允许更大模型单卡部署，减少 tensor parallelism 需求 [L2:2510.27583]。MI355X 将显存扩展到 288 GB、FP8 算力到 5 PFLOPS，MXFP4 达 10 PFLOPS [L2:amd-cdna4-whitepaper]。Blackwell 的 FP4 达到 9000 TFLOPS（实测 96.3% 峰值利用率 [L2:2512.02189]），但 FP32 accumulator 使 Tensor Core 吞吐减半（FP16→FP16 为 964.8 TFLOPS vs FP16→FP32 仅 482.4 TFLOPS）——accumulator datapath 是吞吐瓶颈而非乘法单元，推理用 FP16 accumulator、训练用 FP32 accumulator 成为硬件决定的最佳实践 [L3:hardware]。Ops:Byte 比率在 MI355X 上达 FP8 629、MXFP4 1258 [L2:gpu-arch-comparison-2025]——推动更多 workload 从 memory-bound 转向 compute-bound，直接影响 AIConfigurator 的配置搜索空间 [L2:2601.06288]。NVLink5 (1.8 TB/s) + NVL72 (72-GPU 域 130 TB/s) 与 Infinity Fabric (1075 GB/s, 8-GPU 域) 的 67% 带宽差距 [L2:amd-cdna4-whitepaper] 迫使 fabric-lib 设计跨 vendor 通信抽象 [L2:2510.27656]，也使 Fleet 必须在 chiplet 域内（而非跨节点）实现 persistent megakernel [ref:L2:2604.15379]。功耗趋势同样影响部署策略：MI355X 1400W 液冷 vs B200 1000W，NVIDIA FP8 能效 4.50 TFLOPS/W vs AMD 3.60 TFLOPS/W——额外 400W 仅换 ~9% 性能 [L3:hardware]。

5.3 模型架构如何强制部署重新思考 #

DeepSeek-V3 的 256 routed experts + 64-way EP 使跨节点 all-to-all 通信时间几乎等于计算时间 [L2:2412.19437]。DualPipe 通过双向调度和 warp specialization 将 20/132 SM（15%）专门分配给通信才能实现完全隐藏 [L2:2412.19437]。ZeRO-Prefill 走更激进的路线——完全消除 AllToAll，用背景 AllGather 流式传输 expert 权重 [L2:2605.02960]。MFS 则不消除 AllToAll，而是通过 RMLQ 三阶段调度隔离 AllToAll 与其他网络流量之间的竞争 [L2:2603.17456]。Hybrid attention（KDA:MLA 混合）将 KV 吞吐压缩 4–13× [L2:2604.15039]，使 PrfaaS 的跨 DC 方案成为可能——但这反过来要求 PD 集群内必须同构才能做 P↔D 角色转换 [L2:2604.15039]。MoE 的动态 token 路由使静态 expert 放置注定产生 straggler，推动了运行时负载均衡算法的发展 [L2:moe-lb-interai25]。

5.4 持续执行范式如何跨越 framework-kernel-hardware 边界 #

TileRT 的 Persistent Engine Kernel 是系统设计和 GPU 编程模型深度融合的产物——AOT 编译器将整个 transformer 模型静态展开为单个 kernel，在编译期规划 tile 调度、warp 特化和通信时序 [L2:tile-ai-tilert]。TokenSpeed 走不同的技术路线——用 C++ 有限状态机管理 operator 序列 + Placement 编译器做 memory layout 优化，保留了一定的运行时灵活性 [L2:tokenspeed]。Fleet 在 AMD MI350 上引入 chiplet-task 作为第四级执行抽象（高于 wavefront/workgroup/grid），将碎片化的 per-XCD 4 MB L2 转化为协作式共享 cache [ref:L2:2604.15379]。GPUOS 走最保守的路线——保留 PyTorch eager semantics，但在 persistent kernel 中用 lock-free ring buffer 接收动态注入的 operator，支持零停机热更新 [L2:2604.17861]。Four systems, four philosophies：TileRT 编译期全知，TokenSpeed 半编译半运行时，Fleet 硬件拓扑感知，GPUOS 运行时全动态。但它们共同依赖 hardware 层未提供的能力——原生 persistent CTA 支持、硬件级 tile 同步原语——当前均通过软件 trick 占据 SM 不释放来实现 GPU-resident 执行 [L2:tilert-speed-scaling-law]。Fleet 的 15.3× micro-op 加速仅适用于 launch-overhead-dominated 操作——大 GEMM 不受益，而 Fleet 的 persistent megakernel 在 batch size ≥ 64 时被 hipBLASLt 反超 [L3:2604.17861]。这暗示 runtime 优化与 kernel 计算优化在不同操作粒度上互补而非竞争 [L3:kernel]。

5.5 调度器与通信库的纵向协同 #

fabric-lib 证明 P2P 通信是 collective 的必要补充——collectives 的四大约束（fixed membership、synchronized initialization、operation ordering、shape uniformity）在 disaggregated inference/MoE 中已成瓶颈 [L2:2510.27656] [L3:cluster]。TensorHub 的 ROS 利用 RDMA 引擎实现零 trainer stall [L3:2604.09107]。在推理侧，KVFlow 的 workflow-aware 驱逐需要 serving 层开始感知 application-layer graph [L2:2507.07400]，JITServe 要求调度器理解多阶段 compound 请求的 DAG 依赖 [L2:2504.20068]，ThunderAgent 将"agentic program"提升为一等调度单元 [ref:L2:2602.13692]，Halo 用数据库 batch query optimizer 整合多 agent DAG [ref:L2:2509.02121]。MFS 在 disaggregated MoE 中发现三类网络流量互相干扰，用 RMLQ 做 Defer-and-Promote 调度 [L2:2603.17456]。TileRT 以相反方向推进——将通信从外部编排"下沉"到 GPU-resident pipeline 内部 [L2:tile-ai-tilert]。"应用语义下沉到基础设施"（KVFlow/JITServe/ThunderAgent/Halo）与"基础设施编排下沉到 GPU"（TileRT/Fleet/GPUOS）是同一趋势的两个互补方向。

5.6 AMD chiplet 架构催生跨层协同优化 #

MI300X/MI355X 的 8-XCD chiplet 架构在计算效率上引入了 NVIDIA 单 die 不存在的 NUMA 效应。NUMA-Attn 用 ~15 行 Triton 代码解决 attention 的 L2 miss 问题 [L2:2511.02132]，HipKittens 在更底层设计了 chiplet-aware scheduling 和 bank-conflict-free swizzles [L2:2511.08083]，ConCCL 利用 MI300X 独有的 SDMA 引擎做 DMA offload [L2:2412.14335]，Fleet 将 chiplet-task 提升为 programming model 级别的抽象 [ref:L2:2604.15379]。这形成了一条从硬件缺陷诊断（MI300X 45% 利用率 [L2:2510.27583]）到 kernel 层补丁（NUMA-Attn）到 DSL 工具链（HipKittens）到 runtime 架构（Fleet）的完整 AMD-specific 技术栈。NVIDIA Blackwell 通过 NV-HBI 10 TB/s 在硬件层对软件透明地解决了 dual-die NUMA [L2:2512.02189]，但 Blackwell dual-die NUMA 效应的实际影响仍未被系统量化——所有微基准仅在单 SM 或单 die 上测量 [L3:hardware]。两家在 chiplet 治理上的哲学差异（硬件透明 vs 软件显式）将长期影响上层系统设计。warp/wave specialization 的有效性本身也是 architecture-dependent：AVO 在 NVIDIA Blackwell 上使用 warp-specialized pipeline 达到 1668 TFLOPS [L3:kernel]，但 HipKittens 在 AMD CDNA4 上明确拒绝 wave specialization（仅达 80% peak），因 AMD 的 SIMD 寄存器分配是静态的，producer waves 空占寄存器却不贡献计算 [L3:kernel]。

5.7 集中式 vs 去中心化部署的根本张力 #

Parallax 与 DualPath/PrfaaS/MFS 代表了 LLM 部署架构的两极：Parallax 假设 volunteer WAN GPU pool + decentralized DHT 协调，用 pipeline parallelism only + cross-replica stitching 在异构公网环境中实现推理 [L2:2509.26182]；DualPath/PrfaaS/MFS 假设管控集群 + RDMA/NVLink + 中心化调度器，利用 tensor parallelism + KV-cache migration + continuous batching 实现高吞吐 [L2:2602.21548] [L2:2604.15039] [L2:2603.17456]。这一分歧不是简单的技术选择——decentralized 模式放弃了 TP（WAN 带宽不足）、continuous batching（分布式协调延迟太高）、KV migration（传输成本过高），换取的是对 GPU 供给异构性和动态性的容忍 [L2:2509.26182]。MARLIN 占据两者之间的位置——保持 DC 内部的集中式管控（vLLM/SGLang），但在 DC 之间用去中心化 multi-agent RL 做 meta-scheduling [L2:2605.13496]。PrfaaS 的跨 DC prefill offload [L2:2604.15039] 展示了在这两极之间可能存在一条中间路线——DC 内集中式执行 + DC 间选择性 offload，用 hybrid attention 压缩跨 DC 传输量。Parallax 的 7-GPU testbed [L2:2509.26182] vs DualPath 的 1152 GPU 生产部署 [L2:2602.21548] 的规模差距也体现了这两种范式的成熟度鸿沟。

5.8 GPU 利用率收割的三种正交维度 #

Harli、DualPath 和 ZeRO-Prefill 独立攻击了 decode 阶段结构性 underutilization 的三个资源维度：Harli 收割闲置 SM 算力（~40% 利用率 → inference+PEFT co-location，+46% 吞吐）[L2:2511.11729]，DualPath 收割闲置 SNIC 带宽（DE 空闲网卡帮 PE 加载 KV，+87% 离线吞吐）[L2:2602.21548]，ZeRO-Prefill 收割 prefill 计算窗口的背景传输能力（隐藏 expert weight AllGather）[L2:2605.02960]。三者的组合——同时收割 SM + SNIC + compute-window——理论上可行但面临 DRAM 带宽的统一调度挑战。Scepsy 的分数 GPU 分配 [L2:2604.15186] 和 PPD 的 append-prefill co-location [L2:2603.13358] 进一步证明了"闲置资源收割"是跨系统的共同设计模式——关键区别在于收割对象的维度和粒度。

§6 共识与分歧 #

共识 #

  1. PD 分离是必要的基础架构：所有 37 篇工作中涉及推理 serving 的系统均建立在 prefill 与 decode 分离或至少 decode-first 优化的前提上。DualPath [L2:2602.21548]、PrfaaS [L2:2604.15039]、DynaServe [L2:2504.09285]、PPD [L2:2603.13358]、KVServe [L2:kvserve] 五篇直接在 PD 架构上创新。PPD 的发现精化了这一共识：分离的粒度应该是请求类型感知的——Turn 1 full prefill 需要严格分离，而 Turn 2+ append-prefill 可安全 co-locate [L3:framework]。

      1. Agentic workload 正在重塑 serving 系统假设：至少 8 篇工作以 agentic 场景为核心评估——KVFlow [L2:2507.07400]、DualPath [L2:2602.21548]、HexGen-Flow [L2:2505.05286]、ThunderAgent [ref:L2:2602.13692]、Helium [ref:L2:2603.16104]、Halo [ref:L2:2509.02121]、Concur [ref:L2:2601.22705]、Justitia [ref:L2:2510.17015]。这些工作共同证明了 per-request 独立调度假设在 agentic workflow 中系统性失效。Framework 类别综述将此确认为从"高吞吐 batch serving"到"agentic workload 全生命周期编排"的范式转变 [L3:framework]。
        1. 软件栈效率是当前 AMD 平台的主要瓶颈，而非硬件设计：MI300X 的频率-效率分离实验精确量化了软件效率 80–85% 和频率降低约 42% 各自的贡献 [L2:2510.27583]。CDNA4 在 FP8 算力上追平 NVIDIA [L2:gpu-arch-comparison-2025]，但 HipKittens 的 BF16 GEMM 仅达 64.4% 峰值（vs NVIDIA ThunderKittens 69.9%）[L2:2511.08083]，显示软件差距在缩小但未消除。Hardware 类别综述确认理论 spec 与实际性能的差距根因是软件栈成熟度 [L3:hardware]。
          1. Inter-kernel overhead 在低 batch size 下是性能主要瓶颈：Three Taxes 将 BSP 模型的开销分解为 kernel launch tax、bulk synchronous tax、data locality tax 三类 [L2:2511.02168]，TileRT 量化了 8×H200 NVL 上理论 ~1000 tok/s 与实际几十 tok/s 的量级差距并归因于 inter-kernel idle [L2:tilert-speed-scaling-law]，GPUOS 的 15.3× micro-op 加速进一步证实了 launch overhead 在小操作中的主导地位 [L2:2604.17861]，ConCCL 的 DMA offload 从计算-通信干扰的角度验证了这一诊断 [L2:2412.14335]。Kernel 类别综述确认 runtime 优化与 kernel 计算优化在不同操作粒度上互补而非竞争 [L3:kernel]。
            1. "寻找并聚合空闲带宽"是跨系统的共同设计模式：DualPath 聚合 DE 空闲 SNIC [L2:2602.21548]，TensorHub 聚合 RDMA full-duplex [L3:2604.09107]，ZeRO-Prefill 利用 prefill 计算窗口做背景 weight AllGather [L3:2605.02960]。三个系统在不同场景（推理 KV、训练权重、MoE expert）独立发现了同一优化原理。

              2. 分歧 #

              1. 存储层方案选择：DRAM pool vs SSD + NIC 聚合 vs 模型压缩 vs KV 压缩。Mooncake 用分布式 DRAM pool，DualPath 选择 SSD 存储 + NIC 带宽聚合 [L2:2602.21548]，PrfaaS 走 hybrid attention 压缩 KV 体积后用 Ethernet 跨 DC 传输 [L2:2604.15039]，KVServe 走 service-aware 在线压缩 [L2:kvserve]。四者假设的硬件基础和成本结构完全不同——DualPath 需要 InfiniBand 双网隔离，PrfaaS 需要 hybrid attention 模型，KVServe 需要 decode 端解压预算。Framework 类别综述将此归入 PD 分离与 KV-Cache 生态主线的内部分支 [L3:framework]。
                1. 通信融合粒度的四级分化。Three Taxes 在 kernel 内嵌入 tile-level pipeline [L2:2511.02168]（增量改动）；DeepSeek-V3 DualPipe 在 framework 层双向调度实现组件级重叠 [L2:2412.19437]（框架级改动）；TileRT AOT 编译全模型为单个 Engine Kernel [L2:tile-ai-tilert]（全栈重写）；GPUOS 用 persistent kernel + JIT 动态注入保留 PyTorch eager semantics [L2:2604.17861]（运行时透明）。Fleet 在 AMD MI350 上的 persistent megakernel 是介于 kernel-level 和 full-model 之间的第五种策略——per-operator 编译但 per-chiplet 软件调度 [ref:L2:2604.15379]。Cluster 类别综述指出 BSP 的终结呈现两条路线——系统层用 P2P 绕行 vs kernel 层用 fine-grained sync 瓦解，两者在不同延迟量级下的最优策略不同 [L3:cluster]。
                  1. GPU-initiated vs host-proxy vs GPU-resident 通信。DeepEP 用 IBGDA（GPU-initiated RDMA）[L2:2510.27656]，fabric-lib 证明 host proxy 方案在 ConnectX-7 上反而超越 GPU-initiated（78.4 vs 73.8 tok/s）[L2:2510.27656]，TileRT 彻底绕开 RDMA，通信嵌入 persistent kernel 内部直接通过 NVLink 执行 [L2:tile-ai-tilert]。Cluster 类别综述将此归纳为 host proxy vs GPU-initiated RDMA 的 trade-off 轴，拐点取决于 EP 规模——EP64 时 host proxy post time 达 28μs 并随 EP 线性增长 [L3:cluster]。
                    1. MoE 通信：消除 AllToAll vs 优化 AllToAll vs 调度 AllToAll。ZeRO-Prefill 论证 EP AllToAll 是"零价值通信"应完全消除，用 AsyncEP weight streaming 替代 [L2:2605.02960]。DeepSeek-V3 DualPipe [L2:2412.19437] 和 fabric-lib [L2:2510.27656] 花大量精力优化 AllToAll 的通信效率。MFS 不改变 AllToAll 本身，而是通过 RMLQ 隔离其与其他网络流量的竞争 [L2:2603.17456]。分歧根源是 workload 差异：ZeRO-Prefill 面向 prefill-only 大 batch（compute window 长，可隐藏 AllGather），DualPipe 面向训练（per-token window 短，无法隐藏 AllGather），MFS 面向在线 SLO-constrained serving [L3:2605.02960]。
                      1. PPD vs DualPath/PrfaaS 的 PD 分离粒度之争。PPD 测量 append-prefill 仅造成 2% TPOT 劣化（Llama-3.1-8B + 4×H100）[L2:2603.13358]，论证 Turn 2+ 可安全 co-locate。DualPath 和 PrfaaS 假设 prefill 必须物理分离 [L2:2602.21548] [L2:2604.15039]。矛盾根源是规模和模型不同：小模型上 append-prefill 确实低干扰，但 MoE 超大模型（DeepSeek-660B）的 I/O 密集 prefill 干扰模式完全不同——瓶颈不是计算干扰而是 storage I/O 带宽 [L3:framework]。
                        1. Agentic cache 管理：驱逐策略 vs 预取策略 vs 准入控制。KVFlow 优化驱逐策略（STE 替代 LRU）[L2:2507.07400]，Helium 优化预取策略（plan-level prefix reuse via TRT）[ref:L2:2603.16104]，Concur 优化准入控制（AIMD 防止 thrashing）[ref:L2:2601.22705]。三者在不同负载区间最优——低负载下 I/O 是瓶颈（DualPath 优势），高并发下 thrashing 是瓶颈（Concur 优势），结构化 workflow 下 prefix reuse 是关键（Helium 优势）[L3:2602.21548]。
                          1. Warp/wave specialization 有效性的硬件依赖。AVO 在 NVIDIA Blackwell 上使用 warp-specialized pipeline 超越 cuDNN [L3:kernel]；HipKittens 在 AMD CDNA4 上明确拒绝 wave specialization [L2:2511.08083]。根因是 NVIDIA 的 warp register 分配支持动态调整各 warp group 的 register budget，而 AMD 的 SIMD 寄存器分配是编译时静态确定的。这一分歧不可跨 vendor 泛化，直接影响 TileRT/TokenSpeed（NVIDIA warp spec 有效）vs Fleet（AMD wave spec 无效）的设计选择 [L3:kernel]。
                            1. 集中式管控集群 vs 去中心化 volunteer pool 的部署范式之争。DualPath [L2:2602.21548]、PrfaaS [L2:2604.15039]、MFS [L2:2603.17456] 假设 RDMA/NVLink + 中心化调度器——获得 TP、continuous batching、KV migration 的全部能力。Parallax [L2:2509.26182] 假设 volunteer WAN GPU pool + DHT——放弃上述能力换取对 GPU 供给动态性的容忍。MARLIN [L2:2605.13496] 取折中位置——DC 内管控 + DC 间去中心化 multi-agent 博弈。分歧根源是对"GPU 可得性"的假设不同：管控集群假设固定容量下最大化利用率，volunteer pool 假设容量本身是动态变量。Parallax 的 3.6× 加速 [L2:2509.26182] 与 DualPath 的 1152 GPU 近线性扩展 [L2:2602.21548] 不可直接比较——它们解决不同假设下的不同问题。
                              1. Sustainability-aware vs throughput-maximization 的优化目标之争。MARLIN 首次将碳排放、水消耗、电力成本与 TTFT 联合优化，PHV 超越纯 throughput 方案 46.4% [L2:2605.13496]。但所有其他调度器（DynaServe [L2:2504.09285]、JITServe [L2:2504.20068]、MFS [L2:2603.17456]）仅以 throughput/latency/SLO 为目标。矛盾的核心：MARLIN-Balanced 的 TTFT 比纯 latency 方案劣化 23.61% [L2:2605.13496]——在 SLA < 1s 的 interactive serving 中这是不可接受的退化。当监管尚未强制 per-workload carbon footprint 披露时，sustainability 维度缺乏经济驱动力。PrfaaS 的跨 DC offload [L2:2604.15039] 虽未以 sustainability 为目标，但其"将 prefill 放到有空闲容量的远端 DC"的策略客观上与 MARLIN 的 carbon-aware routing 兼容——两者可层次化组合。
                                1. MoE 多租户共享：serverless FaaS vs dedicated GPU vs shared-instance。FaaSMoE 将 MoE expert 部署为 stateless FaaS 函数，追求 scale-to-zero + 多租户 expert 共享 [L2:2604.26881]。ZeRO-Prefill 走相反方向——让每个 GPU 持有完整 expert 集合，消除通信换取执行效率 [L2:2605.02960]。主流 vLLM/SGLang 则用 shared-instance continuous batching 天然服务多租户。三者的 trade-off 轴是"expert residency vs invocation overhead vs isolation"：FaaSMoE 最小 residency 但最高 invocation overhead（HTTP ms 级 vs GPU dispatch μs 级），ZeRO-Prefill 最大 residency 但零 invocation overhead，shared-instance 居中。FaaSMoE 的 CPU-only 评估 [L2:2604.26881] 使其定量结论不可外推到 GPU——但"MoE ≅ FaaS"的结构映射对未来 expert-level disaggregation 有概念启发价值 [L3:2604.26881]。

                                  2. §7 Open challenges (根本性困难) #

                                  7.1 KV-Cache 压缩与 attention 表达力的信息论瓶颈 #

                                  Hybrid attention（linear + full-attention 混合）可将 KV 吞吐降低 4–13× [L2:2604.15039]，KVServe 的 service-aware 压缩可达 10× [L2:kvserve]，但两者都是模型或 serving 层面的折衷。没有理论工具量化"KV-Cache 压缩到什么程度会导致不可接受的下游任务退化"。KV-Cache 是 attention 计算的中间表示，其最优压缩率取决于输入序列的互信息结构。解决此问题需要 model（注意力机制设计）和 framework（serving 系统）的联合优化。预估难度：3–5 年。

                                  7.2 异构硬件下的通用性能可移植 #

                                  DualPath 深度绑定 InfiniBand VL QoS [L2:2602.21548]；DeepSeek-V3 的 all-to-all kernel 绑定 H800 NVLink+IB 拓扑 [L2:2412.19437]；ConCCL 绑定 AMD MI300X 的 SDMA 引擎 [L2:2412.14335]；TileRT 深度绑定 8×H200 NVL 的 NVLink 域和 TMA 异步引擎 [L2:tilert-speed-scaling-law]；Fleet 绑定 MI350 的 chiplet-task 抽象 [ref:L2:2604.15379]；NUMA-Attn 绑定 MI300X 的 8-XCD 拓扑 [L2:2511.02132]。fabric-lib 尝试统一 ConnectX-7 和 EFA [L2:2510.27656]，HipKittens 尝试提供 AMD 的 tile-based DSL [L2:2511.08083]，但前者仅覆盖两种 NIC，后者仅覆盖 CDNA3/4。Kernel 类别综述确认 warp specialization 有效性本身就是 architecture-dependent [L3:kernel]。Hardware 类别综述揭示 chiplet 封装路线分歧（AMD 优先良率 + 工艺独立演进 vs NVIDIA 优先 die 内通信性能和软件透明性）将长期持续 [L3:hardware]。预估难度：2–3 年。

                                  7.3 动态 agentic workflow 的在线资源分配 #

                                  KVFlow 假设 workflow graph 在运行时稳定可观测 [L2:2507.07400]，但 ReAct/Reflexion 等自主 agent 在运行时动态生成下一步。Helium 的 Templated Radix Tree 需要模板化 workflow [ref:L2:2603.16104]——非结构化 agent 无法享受其优化。JITServe 用"保守估计 + 在线精化"部分缓解此问题 [L2:2504.20068]，但 compound 请求的依赖图本质上是 runtime-generated。ThunderAgent 的 program-aware scheduling 假设 agentic program 有可预测的 lifecycle [ref:L2:2602.13692]，对 open-ended agent（如 ReAct loop 无固定终止条件）失效。Concur 的 AIMD 是最少假设的方案——仅需观测 cache 命中率趋势 [ref:L2:2601.22705]——但无法利用 workflow 结构信息做预取。Scepsy 的"相对份额稳定性"假设在 steady-state 下成立，对 workload 剧变时段可能失效 [L2:2604.15186]。Framework 类别综述将"Agentic Training（Stage=training × Workload=agentic）"标记为首要空白——现有 RL 框架不理解 agent workflow 的 DAG 结构 [L3:framework]。预估难度：1–3 年。

                                  7.4 千 GPU 级容错与一致性 #

                                  DualPath 部署于 1152 GPU [L2:2602.21548]，TensorHub 部署于 1024 GPU [L2:2604.09107]，DeepSeek-V3 训练用 2048 GPU [L2:2412.19437]——但这些系统的容错机制均未充分披露。TensorHub 的中心化 reference server 在 10K+ GPU 规模下可能成为瓶颈 [L3:2604.09107]，DualPath 论文完全未讨论 server failure 对跨 DC 场景的影响 [L3:2602.21548]。ZeRO-Prefill 的 frontend T 阈值依赖 vLLM v0.11.0 scheduler 架构，非 portable [L3:2605.02960]。Pipeline replication 中 partial-replica 的故障恢复涉及 RDMA 层 4 秒保守超时后的状态清理 [L2:2604.09107]。预估难度：3–5 年。

                                  7.5 AOT 静态编译 vs 动态工作负载的根本性张力 #

                                  TileRT 的 AOT 全模型编译实现了 inter-kernel overhead 的彻底消除 [L2:tile-ai-tilert]，但现代 LLM 的动态性不断增加：MoE routing 每 token 激活不同 expert（input-dependent），KV cache 随序列长度动态增长，speculative decoding 的 draft/verify 路径引入条件分支。GPUOS 走 JIT 路线保留灵活性，但仅覆盖 micro-ops [L2:2604.17861]——其 template-based JIT 无法直接注入需要精细控制 shared memory layout 和 warp-level primitive 的 kernel（如 Flash-MLA、paged attention variant），此时要么 fallback 到传统 launch 恢复 3–7 μs 开销，要么等待模板库更新 [L3:2604.17861]。Fleet 的 persistent megakernel 在 batch size ≥ 64 时被 hipBLASLt 反超 [L3:2604.17861]，证明 compute-bound 区间单 kernel 极致调优优于跨 kernel 持久化。TokenSpeed 的 Placement 编译器在 TileRT 和 GPUOS 之间找到中间点——半编译半运行时 [L2:tokenspeed]。静态优化与动态负载之间存在根本性的信息论张力。GB10 上 GPUOS 加速仅 2.1–3.1× 暗示硬件厂商正在从 silicon 层面解决 launch overhead——未来架构（Rubin）可能使 persistent kernel 的软件方案变得不必要 [L3:2604.17861]。预估难度：2–4 年。

                                  7.6 MoE 通信范式的根本选择尚未收敛 #

                                  ZeRO-Prefill 完全消除 AllToAll 但引入 168 GB/layer 的 AllGather 数据量（vs AllToAll 的几十 MB），"零冗余"仅在完美 overlap 下成立 [L3:2605.02960]；DualPipe 的 warp specialization 将 15% SM 永久分配给通信 [L2:2412.19437]；MFS 的 RMLQ 需要精确的 network flow 分类 [L2:2603.17456]；MoE-LB 的 ILP 联合优化在 expert 数量增长时面临 NP-hard 复杂度 [L2:moe-lb-interai25]。四种策略在不同 workload（prefill-only vs decode vs training）和不同硬件（NVLink vs IB vs Ethernet）下各有最优区间，但没有统一框架能自动选择最优通信策略。预估难度：2–3 年。

                                  7.7 跨节点 in-kernel fusion 的空白地带 #

                                  Cluster 类别综述精确定位了通信 taxonomy 中的空白：fabric-lib 占据 inter-node + library-level + async P2P 位置，Three Taxes/Iris 占据 intra-node + in-kernel + fine-grained sync 位置，交集区域（inter-node × in-kernel × fine-grained sync）完全空白 [L3:cluster]。将 Iris 的 iris.load() 后端替换为 RDMA read/write 技术上可实现，但需处理 μs 级（RDMA）vs ns 级（intra-node）延迟的 spin-wait 效率问题。同时，Iris 仅在 AMD（Infinity Fabric）上验证，NVIDIA 的 Triton Distributed/CUTLASS 虽有类似能力但缺乏等价的系统化评估 [L3:cluster]。预估难度：1–2 年。

                                  7.8 Persistent kernel + dynamic batching 的融合 #

                                  TileRT/TokenSpeed 局限于 BS=1 [L2:tile-ai-tilert] [L2:tokenspeed]。将 persistent Engine Kernel 扩展到 dynamic batching（在 kernel 内部 batch 大小可变）理论上可行但需要重写 tile scheduler [L3:framework]。GPUOS 的 persistent kernel 与 NCCL collective kernel（需要占用所有 SM 做 AllReduce/AllGather）之间的资源竞争完全未验证——NCCL 期望 SM 全部可用，GPUOS 的 1 block/SM 可能导致 NCCL 性能退化 [L3:2604.17861]。预估难度：1–3 年。

                                  §9 邻接 topic #

                                  1. Agent Serving（agent 运行时优化）：与本 topic 在 agentic workload characterization 上深度重叠——ThunderAgent/Helium/Halo/Concur/KVFlow/HexGen-Flow 均以 agentic 场景为核心评估。边界在于本 topic 聚焦"集群级资源管理"，agent-serving 更关注"agent 运行时的推理引擎优化"。随着 agentic 工作在本 topic 中占比从 3/18 增长到 8/37，两者可能合并为更广义的"Agentic Infrastructure"topic。
                                    1. RDMA 通信（跨硬件可移植通信）：fabric-lib 和 TensorHub 的 RDMA 层工作同时属于本 topic 和独立通信 topic。MFS 的三阶段网络流量调度 [L2:2603.17456] 也处于通信与调度的交界。Cluster 类别综述指出 P2P 灵活性 vs collective 高效性的 trade-off 是核心张力 [L3:cluster]。
                                      1. MoE 推理优化：DeepSeek-V3、MoE-LB、fabric-lib、ZeRO-Prefill、MFS、FaaSMoE 六篇工作涉及 MoE 特有的 expert parallelism 挑战——从完全消除 AllToAll（ZeRO-Prefill）到优化 AllToAll 调度（MFS）到负载均衡（MoE-LB）到 serverless expert 解耦（FaaSMoE）。FaaSMoE 的"MoE ≅ FaaS"概念映射 [L2:2604.26881] 与 ZeRO-Prefill 的"所有 expert 本地化" [L2:2605.02960] 代表了 expert 管理哲学的两极。随着 MoE 架构成为主流，这些工作可能需要独立 topic。
                                        1. GPU 架构演进：MI300X/Blackwell/GPU 对比/CDNA4 白皮书四篇提供硬件基础约束。Hardware 类别综述识别出 MI355X 独立第三方实测、Blackwell dual-die NUMA 量化、MXFP6 端到端验证三大空白 [L3:hardware]。
                                          1. Persistent Kernel 执行范式：TileRT、TokenSpeed、Fleet、GPUOS 构成了从 per-op JIT 到 full-model AOT 的完整执行范式频谱。Kernel 类别综述将 GPUOS vs Fleet 定位为 persistent kernel 哲学的两极——轻量级 dispatcher vs 重量级 megakernel [L3:kernel] [L3:2604.17861]。
                                            1. AMD Kernel 工具链：NUMA-Attn、HipKittens、ConCCL、Fleet 构成了 AMD-specific 的 kernel 优化栈。Kernel 类别综述确认 AMD 上 FP4 attention 和 agent-driven kernel optimization 均为空白 [L3:kernel]。
                                              1. Attention 优化：Kernel 类别综述揭示 SageAttention 系列（INT8→INT4→FP4）和 SLA（95% sparse + linear hybrid）构成了 attention 加速的两条主线 [L3:kernel]。这些 kernel 层创新与本 topic 的 serving framework 有密切交互——量化 attention 的部署需要 framework 支持混合精度调度，稀疏 attention 需要 framework 感知 sparsity pattern。
                                                1. Geo-Distributed & Sustainability-Aware Scheduling：MARLIN [L2:2605.13496] 开辟了 carbon+water+energy 多目标的跨 DC 调度新维度。PrfaaS 的跨 DC prefill offload [L2:2604.15039] 提供了数据平面可行性验证。两者可层次化组合（MARLIN 选 DC → PrfaaS 决定 prefill placement → fabric-lib 执行传输），构成 geo-distributed LLM serving 的完整栈。当监管开始要求 per-workload carbon 披露时，这一方向的产业采纳可能加速。
                                                  1. Decentralized/Volunteer GPU Inference：Parallax [L2:2509.26182] 是目前唯一面向 volunteer WAN pool 的系统（Petals 除外）。其"放弃 TP/continuous batching 换取 GPU 供给弹性"的设计哲学与管控集群方案形成根本对立。如果 GPU 短缺持续或 edge inference 兴起，这一方向将获得更多关注。
                                                    1. Inference-Training Co-location：Harli [L2:2511.11729] 开辟了在同一 GPU 上共存 inference+training 的新维度。其方法论（SM profiling → interference prediction → GreenContext partitioning → CUDA VMM unified allocation）是当前唯一完整方案，但限于 PEFT + Ampere/Ada 架构。与 persistent kernel 环境的兼容性、Hopper/Blackwell 上的行为验证、多租户扩展均为开放问题。
