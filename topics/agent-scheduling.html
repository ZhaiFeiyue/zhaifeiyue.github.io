Agent 调度算法：从请求级负载均衡到工作流级 KV Cache 编排 #

1. 主题缘起 #

2025–2026 年，LLM 的主流使用模式从单请求问答转向多轮自主工具调用（agentic workload）。这一转变使传统负载均衡算法系统性失效：agent 的长 session、持续增长的 KV Cache、bursty 的工具调用模式、以及不可迁移的 decode 状态，要求调度系统从"无状态请求分发"跃迁到"有状态工作流编排"。

CPU 侧的瓶颈分析揭示工具执行（检索、代码执行、分子生成等）可占端到端延迟的 88%，且 CPU 并行化效率远低于 GPU 导致吞吐过早饱和 [L2:2511.00739]。与此同时，PD（Prefill/Decode）分离架构成为大规模 LLM serving 的主流部署模式——Prefill 计算密集、Decode 内存带宽密集——需要各自独立的调度策略。Agent 工作负载在 PD 分离架构下引入了额外挑战：多轮对话中 Turn 2+ 的 Append-Prefill（AP）操作需要在 prefill 节点和 decode 节点之间做动态路由决策 [L2:2603.13358]。

2026 年上半年，业界在四个层面同时发力：(1) 工作流级调度系统（SAGA、HexAGenT、Helium、Halo）把整个 agent workflow 作为一等调度单元 [L2:2605.00528] [L2:2605.16637] [L2:2603.16104] [L2:2509.02121]；(2) 预测性 KV Cache 管理（PBKV、Continuum、SideQuest）将 cache 生命周期从被动 LRU 升级为主动的前瞻式决策 [L2:2605.06472] [L2:2511.02230] [L2:2602.22603]；(3) 投机性执行（PASTE）利用控制流模式预测在 LLM 生成期间提前执行下一个工具调用，将 tool stall 隐藏在思考时间内 [L2:2603.18897]；(4) 客户端 context 管理（Claude Code 五层 compaction、Cursor 动态上下文发现）在客户端侧配合基础设施层做联合优化 [L2:2604.14228] [L2:blog-dynamic-context-discovery]。

本文系统梳理从经典无状态算法到前沿 agent 专用调度系统的演进脉络，覆盖 framework（serving 基础设施）、agent（context/cache 生命周期管理）、algorithm（路由和排序算法设计）三个 category。

关键分支点：

  1. Cache-Aware Routing 分化：从 Consistent Hash 分出 Radix Tree（精确匹配）和 PrefixHash（轻量近似），前者适合结构化 workflow，后者适合大规模分布式。

        1. 执行层优化独立成支：PASTE 的投机性工具执行和 CPU-Centric 的 COMB/MAS 不直接改变调度算法，但通过隐藏 tool stall 和重新平衡 CPU-GPU 利用率，改变了调度问题的约束结构。

          2. 5. 技术线交错 #

          算法设计 ↔ 系统实现的张力 #

          • 精确 cache-aware routing 需要 O(prefix_len) 查找，大规模部署中延迟不可接受。PrefixHash 以 O(log n) 的 xxhash 近似替代，但牺牲了细粒度匹配。llm-d 通过 ZMQ 订阅 KV block events 在精度和延迟之间找到中间地带 [L2:2507.07400]。Helium 的 Templated Radix Tree 以 552 KiB 代替 SGLang RadixCache 的 14.8 MiB（27× 缩减），因为 TRT 按模板静态结构而非 per-token 索引 [L2:2603.16104]。
          • Workflow 预测精度 vs 退化保证：PBKV 的 GraphSAGE 预测器达到 1-step 0.94 精度，但关键贡献是 Lipschitz 退化保证——预测失败时退化至生命周期感知策略而非 LRU [L2:2605.06472]。SAGA 的 AEG 模式推断在无框架提示时仍有 87% 准确率，但退化 15.6% TCT [L2:2605.00528]。PASTE 的 Top-1 准确率仅 27.8%，但通过多候选投机策略达到 93.8% overall hit rate——用资源换覆盖率 [L2:2603.18897]。

          模型架构 ↔ 基础设施调度的跨界 #

          • SideQuest 让模型自身管理 KV Cache：并行辅助线程 fork 主线程的 KV cache，语义推理哪些 tool response 已过期，输出结构化删除命令。仅需 215 个训练样本即可激活此能力。峰值 token 使用量降低 56-65%，SGLang 吞吐提升 83.9% [L2:2602.22603]。
          • Claude Code 的 microcompact 策略感知基础设施层 cache 状态：boundary messages 必须 defer 到 API 响应后获得实际 cache_deleted_input_tokens，形成客户端→基础设施的调度 hint 传递 [L2:2604.14228]。
          • SGH 将 Agent Loop 形式化为 |U|=1 的 scheduler：揭示主流 agent 框架本质上是单就绪单元调度器，结构化并行需要显式 DAG。但 Claude Code 的生产成功证明加上足够强的确定性 harness，Agent Loop 范式仍然可行 [L2:2604.11378] [L2:2604.14228]。

          PD 分离 ↔ 多轮 Agent 的结构冲突 #

          • PPD 揭示了核心洞察：append-prefill 对 decode 的干扰仅 2%（vs full prefill 的 48%），但无单一静态策略在所有场景占优——需要 per-request 动态路由 [L2:2603.13358]。
          • MFS 发现 disaggregated MoE serving 中三阶段通信（KV-cache 复用 + collective comm + P2D 传输）的网络争用使 TTFT 膨胀约 50%，提出 Reverse Multi-Level Queue 近似 Least-Laxity-First [L2:2603.17456]。
          • HexAGenT 将此冲突推向极致：在异构 A100/H100/H200 集群上联合优化 prefill 和 decode 的 placement，用 projected scaled-SLO risk 排序使 Req99 最大降低 80.5% [L2:2605.16637]。

          公平性 ↔ 效率的权衡 #

          • Justitia 用 KV token-time（显存占用 × 持续时间）度量成本，以 WFQ virtual finish time 排序，保证最坏延迟 ≤ 2c_max + C_max/M [L2:2510.17015]。
          • SAGA 的 Agent Fair Share 通过 Lyapunov drift 分析证明 bounded-deviation 保证——轻租户 SLO 达成率从 43.2% 跃升至 98.7% [L2:2605.00528]。

          工具执行 ↔ GPU 利用率的瓶颈转移 #

          • 当 CPU 工具执行占 E2E 延迟 88% 时，GPU kernel 优化上限仅 12% [L2:2511.00739]。PASTE 通过模式感知投机执行将 tool 等待时间减少 67%、实现 GPU-tool 10× overlap 提升 [L2:2603.18897]。Halo 的 CPU-GPU pipelining 和 request coalescing（最大贡献：去掉后慢 2.54×）从 batch 层面解决了相同问题 [L2:2509.02121]。三者在不同粒度（per-call / per-session / per-batch）攻击同一瓶颈。

          6. 共识与分歧 #

          共识 #

          1. Request-level 调度对 agent 工作负载不够：所有 2026 年论文一致认为需要 workflow/program-level 调度。SAGA 量化了 request-level 调度的代价——38% 执行时间浪费在 KV cache 再生、6.0× 端到端延迟膨胀 [L2:2605.00528]。HexAGenT 证明仅从 per-call 切换到 workflow-FCFS 就能降低 31.4% Req95 [L2:2605.16637]。Halo 对比显示 workflow-level 优化后比 vLLM 快 400× [L2:2509.02121]。
            1. KV Cache 是一等调度信号：Continuum（TTL 模型）、SAGA（Execution Graph 复用预测）、PBKV（GraphSAGE 评分）、BalanceRoute（KV 单调增长感知 F-score）、SideQuest（语义 GC）、KVFlow（Belady 适配）全部将 KV 状态作为核心调度输入 [L2:2511.02230] [L2:2605.00528] [L2:2605.06472] [L2:2605.06113] [L2:2602.22603] [L2:2507.07400]。
              1. 分层 Fallback 是标准范式：SGLang（sticky → cache_aware → P2C）、vLLM Router（consistent_hash → round_robin）、PBKV（退役缓存淘汰 → 评分淘汰 → LRU 兜底）均采用多级降级策略 [L2:2605.06472]。
                1. 工具执行是不可忽视的系统瓶颈：CPU-Centric 论文首次系统刻画（35%-88% E2E 延迟）[L2:2511.00739]，PASTE 通过投机执行将 tool stall 减少 67% [L2:2603.18897]，Halo 的 request coalescing 在 batch 分析场景消除了重复工具调用 [L2:2509.02121]。三篇独立确认了同一结论。

                  2. 分歧 #

                  1. 预测 vs 无预测路由：BalanceRoute 明确论证"二元终止分类器比完整长度回归更可靠，因为噪声增长快于信号"，BR-0（无预测）已能将不均衡降低 4.1× [L2:2605.06113]。而 SAGA 和 PBKV 投入大量工程在预测精度上（AEG 模式推断、GraphSAGE 多步预测）。两者在不同规模下各有优势：BR-0 在 G=4 时已足够好，但 BR-H 在 G=16 时额外带来 +12% [L2:2605.06113]。
                    1. TTL 策略 vs 预测性淘汰：Continuum 用 cost-benefit 模型设定有界 TTL [L2:2511.02230]，PBKV 用多步预测驱动淘汰评分 [L2:2605.06472]。Continuum 认为"per-turn queueing delay 是独立于数据传输的调度问题"（InferCept 的 preserve 策略对此无效）；PBKV 认为"LRU 的时间局部性假设与工作流级缓存复用的结构性不匹配"。两者解决的是同一问题的不同维度——TTL 解决"保留多久"，预测解决"保留哪些"。
                      1. 静态 DAG vs 动态工作流：SAGA 用 Agent Execution Graph（DAG + 转移概率）预测 KV 复用 [L2:2605.00528]。PBKV 批评"静态 DAG 假设无法处理运行时条件分支和重试循环"，其 GNN 在动态工作流上命中率 69% vs KVFlow（静态距离）的 39.87% [L2:2605.06472]。SGH 从理论层面支持静态 DAG 以获得有界终止和可审计性 [L2:2604.11378]，而 Claude Code 的生产成功证明 Agent Loop + 强 harness 仍然可行 [L2:2604.14228]。本质分歧在于对 LLM 决策可靠性的信任程度。
                        1. 中心式 vs 分布式调度决策：BalanceRoute 采用集中式 stateful proxy（需毫秒级全局状态维护）[L2:2605.06113]。HexAGenT 同样依赖 global scheduler [L2:2605.16637]。而 SGLang 的 router 和 vLLM Router 偏向去中心化的 per-request 决策。Helium 的 cost-based scheduler 在 batch offline 场景可以全局优化（达 MILP oracle 0.9% 内）[L2:2603.16104]，但 online 场景中集中式协调器可能成为瓶颈。
                          1. DB-style 查询优化 vs 程序感知调度：Helium 和 Halo 从数据系统视角将 workflow 视为可优化的 query plan [L2:2603.16104] [L2:2509.02121]；ThunderAgent 和 SAGA 从程序分析视角将 workflow 视为需要理解控制流的 program [L2:2602.13692] [L2:2605.00528]。DB 路径擅长结构化重复 batch（Halo request coalescing 2.54× 贡献），program 路径擅长动态非确定性 workflow。

                            2. 7. Open challenges (根本性困难) #

                            7.1 预测不可靠性的理论边界 #

                            Agent 工作流天然难预测——随机解码沿路径传播不确定性 [L2:2605.06472]。PBKV 的 Lipschitz 退化保证是第一步，但 competitive ratio 分析仍然缺失。SAGA 的理论 competitive ratio bound（20.5×）比经验值（1.31×）松弛 15 倍 [L2:2605.00528]。BalanceRoute 也未给出 F-score 贪心策略的近似比证明 [L2:2605.06113]。需要信息论层面的下界来理解"agent 调度能被优化到什么程度"。

                            涉及 category：algorithm（形式化保证）+ framework（实现可行性）。难度：3-5 年。

                            7.2 KV Cache 压缩 vs 注意力表达力的信息瓶颈 #

                            SideQuest 表明启发式 KV 压缩（H₂O、SnapKV）在 agentic 场景导致模型崩溃（non-completion rate 60%+），因为 token 重要性是非单调的 [L2:2602.22603]。但模型自驱动 GC 依赖 frontier-class 模型的"元认知"能力，不确定是否可扩展到更小模型。KV cache 压缩与注意力表达力之间存在理论层面的 information-bottleneck tradeoff，目前无 single-category 方案能完全解决。

                            涉及 category：agent（模型能力）+ algorithm（压缩理论）+ framework（serving 实现）。难度：未知。

                            7.3 客户端 Context 管理与基础设施调度的联合优化 #

                            Claude Code 的五层 compaction 和 Cursor 的 Dynamic Context Discovery 在客户端侧做优化 [L2:2604.14228] [L2:blog-dynamic-context-discovery]，SAGA/HexAGenT 在基础设施侧做 workflow-aware 调度。两者目前独立运作——理想的 agent serving 应同时优化两个层面，但缺少跨层信息传递的标准协议。Anthropic 的 cache_control breakpoint 是初步尝试，但远未形成生态标准。

                            涉及 category：agent（客户端架构）+ framework（serving 协议）。难度：1-2 年（工程问题）。

                            7.4 异构集群的 Workflow-Aware Placement #

                            HexAGenT 首次处理异构 GPU（A100/H100/H200）下的 workflow 调度，但 greedy joint planner 无近似比保证——问题本质是 online unrelated machines scheduling [L2:2605.16637]。HexGen-Flow 在较小规模验证了异构 GPU 派发的有效性（A100/A6000/L40 混用）[L2:2505.05286]。随着部署规模增长（100+ 节点、多代 GPU 混部），联合 placement 问题的搜索空间爆炸。Scepsy 的 aggregate pipeline 抽象能否推广到动态 DAG 仍未验证 [L2:2604.15186]。

                            涉及 category：algorithm（组合优化）+ framework（集群调度）。难度：2-3 年。

                            7.5 投机执行在高动态场景的可行边界 #

                            PASTE 在控制流模式稳定的 agent（coding: 55% edit→test，research: 51% search→fetch）上表现优异，但 Top-1 准确率仅 27.8%——模式不稳定的探索性任务中，投机执行的收益与资源浪费之间的平衡点尚不清晰 [L2:2603.18897]。将投机执行与工作流预测（PBKV GraphSAGE）结合——pattern 预测同时指导投机执行和缓存预取——是自然方向但尚无实现 [L3:agent]。

                            涉及 category：agent（执行策略）+ algorithm（预测模型）。难度：1-2 年。

                            7.6 Agent Loop vs 结构化 DAG 的统一理论 #

                            SGH 论证 Agent Loop 有三个结构性缺陷（隐式依赖、无界 recovery、不可审计），但估计 60-70% 的任务实际是线性链——此时 DAG 只比 loop 多出 bounded recovery 和审计价值 [L2:2604.11378]。需要一个理论框架来刻画"何时 DAG 结构的调度收益超过其实现复杂度（3300-6500 行 vs Agent Loop 数百行）"。

                            涉及 category：agent（运行时架构）+ framework（调度实现）。难度：2-3 年（需大规模实证）。
