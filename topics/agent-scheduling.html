Agent 调度算法：从请求级负载均衡到工作流级 KV Cache 编排 #

1. 主题缘起 #

2025–2026 年，LLM 的主流使用模式从单请求问答转向多轮自主工具调用（agentic workload）。这一转变使传统负载均衡算法系统性失效：agent 的长 session、持续增长的 KV Cache、bursty 的工具调用模式、以及不可迁移的 decode 状态，要求调度系统从"无状态请求分发"跃迁到"有状态工作流编排"。

同时，PD（Prefill/Decode）分离架构成为大规模 LLM serving 的主流部署模式。Prefill 和 Decode 两个阶段的资源特征截然不同——前者计算密集、后者内存带宽密集——需要各自独立的调度策略。Agent 工作负载在 PD 分离架构下引入了额外挑战：多轮对话中 Turn 2+ 的 Append-Prefill（AP）操作需要在 prefill 节点和 decode 节点之间做动态路由决策。

本文系统梳理从经典无状态算法到前沿 agent 专用调度系统的演进脉络，按 PD 分离架构分析 prefill 和 decode 各自的最优策略，并总结业界的收敛方向。

2. 无状态调度算法 #

无状态算法不维护任何请求历史或 engine 状态，每次决策独立。

2.1 Round Robin #

出处：经典操作系统调度算法。LLM serving 中被 vLLM、SGLang 等系统作为默认 baseline。

请求依次分配：W1 → W2 → W3 → W1 → ...

维度说明
优势O(1)，实现极简，完全公平
劣势不感知负载差异；长短请求混排严重不均衡；无 KV Cache 亲和性
Agent 适用度低——多轮 agent 的每轮请求可能落到不同 worker，KV Cache 命中率趋近于零
工业使用vLLM Router 默认策略；vLLM production-stack least_loaded 的 fallback

2.2 Random #

出处：经典随机算法。最大负载分析见 Raab & Steger, "Balls into Bins — A Simple and Tight Analysis", RANDOM 1998。

维度说明
优势O(1)，无协调开销，多 router 实例天然无需同步
劣势最大负载 O(log n / log log n)；无亲和性
Agent 适用度低——与 Round Robin 相同的 KV Cache 问题
工业使用vLLM Router 可选策略

2.3 Power of Two Choices (P2C) #

出处：Mitzenmacher, "The Power of Two Choices in Randomized Load Balancing", IEEE TPDS 2001 (论文)。源自其 1996 UC Berkeley PhD 论文。综述见 Mitzenmacher, Richa & Sitaraman, "The Power of Two Random Choices: A Survey", Handbook of Randomized Computing 2001。

算法：随机挑 2 个 worker → 选负载更低的。

维度说明
优势O(1)，最大负载从 O(log n / log log n) 降到 O(log log n)——指数级改善；d=2 是最优性价比点（d=3 只有常数改善）
劣势无 session 亲和性；负载信号如果只用请求数，不反映 KV Cache 内存压力；对 bursty 到达敏感
Agent 适用度中——单独用效果有限，但作为 cache-aware 路由的 fallback（冷启动 / 无先验信息时）表现很好
工业使用vLLM Router power_of_two；Ray Serve PrefixCacheAffinityRouter fallback；SGLang load_based 阶段用 P2C + 内存压力打分 work + λ * token_usage/(1-token_usage)

SGLang 的扩展打分公式中 token_usage/(1-token_usage) 是凸函数——内存占用趋近 1 时分数急剧上升，形成软隔离墙。

2.4 Least Connections / Least Loaded #

出处：经典网络负载均衡算法，可追溯到 1990s 的 F5/Nginx 实现。LLM 场景中 vLLM production-stack 的实现见 PR #890。

维度说明
优势简单直观的负载感知；选 running + queued 最少的 worker
劣势需要全局状态收集；不区分请求轻重；无 cache 亲和
Agent 适用度中低——请求数不等于真实负载，长 agent session 的 decode 持续占用内存和算力
工业使用vLLM production-stack least_loaded 策略

3. 有状态调度算法 #

维护 engine 状态（负载、缓存、历史）来做更智能的决策。

3.1 Consistent Hashing #

出处：Karger et al., "Consistent Hashing and Random Trees: Distributed Caching Protocols for Relieving Hot Spots on the World Wide Web", STOC 1997 (论文)。

算法：hash(routing_key) → 哈希环上找到目标 worker。增删 worker 只影响相邻区域，O(1/n) key 迁移。

维度说明
优势天然 session 亲和性；扩缩容迁移最小；O(log n) 查找
劣势不感知实际负载——热点 key 导致 worker 过载；不感知 KV Cache 真实状态
Agent 适用度中——提供粗粒度的 session 亲和，但 KV Cache 实际命中率不可控
工业使用vLLM Router consistent_hash（PD 模式 prefill 默认推荐）

3.2 Cache-Aware Routing（Radix Tree / Prefix Tree） #

出处：Zheng et al., "SGLang: Efficient Execution of Structured Language Model Programs", NeurIPS 2024 (论文)。提出 RadixAttention，用 radix tree 管理 KV Cache 并做 cache-aware scheduling。Router 级 cache-aware 负载均衡发布于 SGLang v0.4 (Dec 2024)。

算法：Router 维护 per-worker 的近似 radix tree → 匹配请求 prefix → 路由到缓存命中最高的 worker。

维度说明
优势最大化 KV Cache 命中率；减少重复 prefill 计算
劣势O(prefix_len) 查找开销；需同步 engine cache 状态（有延迟）；tree 维护成本
Agent 适用度高——agent 多轮中 system prompt + 历史 context 高度重复，cache-aware 路由是 prefill 侧的最优策略
工业使用SGLang RadixAttention 原生；vLLM Router cache_aware；Ray Serve PrefixCacheAffinityRouter；llm-d precise-prefix-cache-scorer（基于 ZMQ KV events 的精确 prefix 匹配）

3.3 PrefixHash（Consistent Hash + Prefix 感知） #

出处：SGLang model-gateway，PR #15935（2025）。

算法：提取请求前 N 个 token → xxhash → 一致性哈希环查找 → 负载过高时 fallback 到 least loaded。

维度说明
优势O(log n) 比 radix tree O(prefix_len) 快；bounded load balancing 防热点
劣势只看前 N token hash，粗粒度
Agent 适用度中——大规模部署中 cache-aware 的轻量替代
工业使用SGLang model-gateway（gRPC router）

3.4 DualMap（双哈希映射 + SLO 感知） #

出处：Chen et al., "DualMap: Enabling Both Cache Affinity and Load Balancing for Distributed LLM Serving", ICLR 2026 (论文)。

算法：两个独立哈希函数 f1, f2 → 映射到 2 个候选 worker → 优先选 cache 亲和的 → TTFT 超 SLO 时切换到负载更低的 → hotspot-aware rebalancing → dual-hash-ring 弹性扩缩。

维度说明
优势统一解决 cache 亲和 vs 负载均衡矛盾；SLO 驱动自适应切换
劣势实现复杂；需准确 TTFT 预测；hotspot tree 维护开销
Agent 适用度高——理论上结合了 P2C 和 consistent hash 的优点
工业使用学术前沿，暂未见大规模工业部署

3.5 Sticky Routing（Session 亲和） #

出处：经典 Web 负载均衡概念。LLM 场景中，SGLang SessionAware Router（Issue #25760）将其形式化为 routing_key + role + bucket 的三元组粘性。

维度说明
优势多轮对话 KV Cache 100% 复用；零迁移开销
劣势长 session 导致负载固化；不处理 worker 故障重路由
Agent 适用度极高——agent session 天然 sticky，KV Cache 迁移成本极高
工业使用SGLang SessionAware Router 第一级；PD 分离中 decode 阶段隐式 sticky（一旦请求落在 decode worker 就不再迁移）

3.6 BalanceRoute BR-H（Horizon-discounted 负载均衡） #

出处：Agrawal et al., "Tackling the Data-Parallel Load Balancing Bottleneck in LLM Serving: Practical Online Routing at Scale", arXiv 2605.06113 (论文)，2026 年 5 月。

算法：专注 PD 分离中 prefill→decode 路由。用 horizon H 步前瞻估算每个 worker 未来 H 步的负载，score = Σ discount^t * load(t) → 选分最低的 worker。配合 termination classifier 预测请求何时结束。

维度说明
优势唯一考虑 KV Cache 单调增长特性的前瞻性算法；毫秒级决策
劣势依赖输出长度预测；需 termination classifier 训练数据
Agent 适用度高——decode 阶段的前瞻性负载均衡
工业使用在 vllm-ascend 上验证（Qwen3-30B-A3B / DeepSeek-V3 W8A8）

3.7 SAL（Token-Aware Load Balancer） #

出处：Grigorescu et al., "Scheduling in LLM Serving: A Survey of Techniques and Systems", arXiv 2410.17840 (论文)。

维度说明
优势用 token 数而非请求数度量负载，更准确反映 GPU 资源消耗
劣势需估算输出长度；不考虑 cache 亲和
Agent 适用度中——drop-in replacement，可组合
工业使用论文验证阶段

3.8 LBGR（Learning-Based Greedy Routing） #

出处：Wu, Silwal & Zhang, "Randomization Boosts KV Caching, Learning Balances Query Load: A Joint Perspective", ICLR 2026 (论文)，开源。

算法：在线学习每个 worker 的端到端延迟模型 → 贪心选延迟最低的 worker。配合 RLT（Randomized Leaf Token）eviction 替代 LRU。

维度说明
优势自适应学习；统一 cache eviction 和 routing；理论 O(log n) competitive ratio
劣势学习收敛需时间；冷启动差
Agent 适用度中——动态工作负载下自适应路由
工业使用ICLR 2026，cache 命中率提升 6.92x

4. Agent 专用调度系统（2026 前沿） #

4.1 SAGA — Workflow 原子调度 #

出处：arXiv 2605.00528，2026 年 5 月。

核心：把整个 agent workflow 作为一等调度单元（而非单个 inference call）。

  • Agent Execution Graph：捕获 workflow 结构，预测跨 tool-call 边界的 KV 复用，达到 Belady 最优策略的 1.31x 以内
  • Session-affinity batching + work stealing：co-locate 相关请求并全局负载均衡
  • Agent Fair Share：以 task 完成时间为公平性指标，有 bounded-deviation 保证

效果：64-GPU 集群上 SWE-bench + WebArena，task 完成时间降低 1.64x，GPU 内存利用率提升 1.22x，99.2% SLO 达成率。

4.2 HexAGenT — Workflow + 异构感知调度 #

出处：arXiv 2605.16637，2026 年 5 月。

核心：在异构 PD 分离集群（A100/H100/H200）上，以 workflow 级 SLO 为目标。

  • 每个请求建模为 online-revealed DAG
  • 维护 workflow standalone completion horizon 估计
  • 按 projected risk 优先级排序 ready calls
  • 联合选择 prefill placement + decode placement + 本地队列优先级

效果：SLO scale 降低 20.1%(p95) 到 80.5%(p99)。

4.3 Continuum — KV Cache TTL 调度 #

出处：arXiv 2511.02230v2，2025 年 11 月。

核心：Agent tool call 间隙不立即驱逐 KV Cache，而是给一个 TTL。

  • 建模 prefill/reload 成本和 per-turn queueing 延迟收益
  • 根据工具调用时长分布和 GPU 内存压力计算最优 TTL
  • TTL 过期后安全驱逐，防止死锁

4.4 SideQuest — 模型自驱动 KV Cache GC #

出处：arXiv 2602.22603，2026 年 2 月。

核心：让大推理模型自己决定哪些 KV Cache 该删除，通过并行 auxiliary thread 执行压缩。模型输出结构化驱逐命令 {del_cursors: [0]}，仅需 215 个训练样本。峰值 token 使用量降低 65%。

论文明确提到 Cursor 的 Dynamic Context Discovery，认为其解决了"信息检索"问题但未解决"推理效率"问题。

4.5 PBKV — 预测性 Agent KV Cache 管理 #

出处：arXiv 2605.06472，2026 年 5 月。

核心：预测 agent workflow 未来几步会调用哪些 agent，据此决定 KV Cache eviction/prefetch。动态 workflow 上 1.85x 加速。

4.6 PPD — Decode 节点本地 Append-Prefill #

出处：arXiv 2603.13358，2026 年 3 月。

核心：多轮 agent 中 Turn 2+ 的 Append-Prefill 动态路由回 decode 节点本地执行，避免跨节点 KV 传输。路由参数 x∈[0,1] 控制 AP 操作路由比例，x=1 时全部在 decode 节点本地执行。Turn 2+ TTFT 降低 48-73%。

4.7 MFS — Multi-stage Flow Scheduling #

出处：arXiv 2603.17456，2026 年 3 月。

核心：PD 分离中多阶段 flow 的网络争用优化。近似 Least-Laxity-First，通过 Reverse Multi-Level Queue（RMLQ）动态提升优先级。TTFT SLO 达成率提升 1.2x-2.4x。

5. PD 分离下的 Agent 调度：Prefill vs Decode #

Agent 场景在 PD 分离架构下的核心挑战：

  • Prefill 端：首轮大 prompt + 后续轮 Append-Prefill（AP），prefix 重复度极高
  • Decode 端：长 session，KV Cache 持续增长不可迁移（sticky），bursty 工具调用

6. Claude Code 与 Cursor 的调度策略 #

两个最广泛使用的 coding agent 系统没有公开推理基础设施细节，但其客户端侧的 context 管理策略实质上是另一种形式的"调度"。

6.1 Claude Code：Prompt Caching 即调度 #

官方博客 "Prompt caching is everything" (原文) 揭示了核心策略：

  • 请求结构按 prefix 稳定性排序：System Prompt + Tool Schemas（全局缓存 1h TTL）→ CLAUDE.md + Project Context（滚动缓存 5min TTL）→ 历史 messages → 最新一轮（uncached）
  • Prefix match：API 侧字节级精确前缀匹配，任何位置修改使后续 cache 失效
  • Cache read 成本：0.1x base（90% 折扣），是 Claude Code 运行经济性的关键
  • 五层 compaction pipeline：Budget Reduction → Snip → Microcompact（cache-aware） → Context Collapse → Auto-Compact。每层成本递增，早期廉价层先执行
  • Compaction 复用 cache：compaction 时复用完全相同的 prefix（system prompt + tools + 历史），只在末尾追加 compaction 指令，确保整个 prefix 命中 cache

出处：arXiv 2604.14228，"Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems"，2026 年 4 月；arXiv 2602.22402，CMV 对 Claude Code 76 个会话的实证分析。

6.2 Cursor：Dynamic Context Discovery #

出处：SideQuest 论文（arXiv 2602.22603）引用 Katz 2026。

核心思路是上下文虚拟化：不把所有 tool 定义和代码文件塞进 context window，将它们抽象为可按需 lazy load 的虚拟资源。2026 年 4 月 Cursor 重建了并行 agent 编排界面，定位为 orchestration layer。

6.3 客户端 vs 基础设施的互补性 #

Claude Code 和 Cursor 的策略在客户端侧做 context 管理（compaction、lazy loading、prefix cache 优化），SAGA / HexAGenT 等在 serving 基础设施侧做 workflow-aware 调度。两者互补——理想的 agent serving 应同时优化两个层面。

8. 趋势总结 #

8.1 调度粒度的演进 #


2024:  request-level（vLLM continuous batching, round robin / P2C）
         ↓
2025:  prefix-level（SGLang RadixAttention, cache-aware routing）
         ↓
2026H1: workflow-level（SAGA / HexAGenT / ThunderAgent / Helium）
         + 客户端 context 管理（Claude Code 5层 compaction, Cursor 上下文虚拟化）
         + 模型自驱动内存管理（SideQuest）

核心共识是：request-level 调度不够，需要 workflow/program-level 调度。Agent 的多轮 tool-call 序列构成一个有结构的程序（而非独立请求流），调度系统必须理解这个结构才能做出正确的 KV Cache 保留/驱逐/复用决策。

8.2 分层 Fallback 成为标准范式 #

业界正收敛到 分层 fallback + cache/session 感知 的组合策略：


第一优先级：Session Sticky / Consistent Hash
  └── 多轮对话复用 KV Cache，避免迁移

第二优先级：Cache-Aware (Prefix Tree / KV Event)
  └── 新 session 或 sticky 失效时，找 cache 命中最高的 worker

第三优先级：Load-Aware (P2C / SAL / Least Loaded)
  └── 冷启动或 cache 无差异时，用负载感知做均衡

附加机制：
  ├── 条件路由（短请求跳过 PD 分离）
  ├── 背压（防止 decode 过载）
  └── SLO 驱动切换（TTFT 超限时牺牲 cache 亲和换负载均衡）

纯无状态算法（Round Robin、Random、单独的 P2C）在 agent 场景下效果显著不足。

8.3 KV Cache 成为一等调度信号 #

2026 年的 agent 调度系统一致地将 KV Cache 状态作为核心调度输入：

  • 内存占用率：SGLang 的 token_usage/(1-token_usage) 凸函数惩罚
  • Prefix 命中率：llm-d 通过 ZMQ 订阅每个 pod 的 KV block events，构建精确的 per-pod prefix index
  • KV Cache 增长预测：BR-H 用 horizon 前瞻 + termination classifier 预测未来负载
  • Workflow 级复用预测：SAGA 的 Execution Graph 预测跨 tool-call 边界的 KV 复用
  • TTL 控制：Continuum 的 cost-benefit 模型决定 tool call 间隙保留多久
  • 模型自驱动 GC：SideQuest 让模型自己决定哪些 KV 该删除

8.4 Prefill 和 Decode 策略持续分化 #

PD 分离下，Prefill 和 Decode 的调度策略正在持续分化：

维度PrefillDecode
核心目标最大化 KV Cache 命中（减少重复计算）均衡负载 + 保持 session sticky
最优算法Cache-Aware / PrefixHashSticky + BR-H / P2C+内存感知
状态信号prefix tree / KV eventsin-flight 请求数 + token_usage + termination 预测

8.5 客户端 Context 管理与基础设施调度的融合 #

Claude Code 的 prompt caching 策略和 Cursor 的 Dynamic Context Discovery 表明：agent 调度不仅发生在 serving 基础设施层（GPU 集群的请求路由），也发生在客户端层（context window 的内容管理）。两者正在趋向融合：

  • SideQuest 让模型在推理过程中自主管理 KV Cache（模型层 → 基础设施层的跨越）
  • Claude Code 的 microcompact 策略是 cache-aware 的（客户端层感知基础设施层的 cache 状态）
  • Anthropic prompt caching API 的 cache_control breakpoint 是客户端层向基础设施层传递调度 hint 的标准接口
