开源前沿 MoE 大模型：DeepSeek-V3 → V4 与 Kimi K2.6 的路线分野

§1 主题缘起

2024 年 12 月到 2026 年 4 月的 16 个月里，开源 MoE 大模型经历了两代跨越式演进，产生了两条截然不同的技术路线。这三款模型——DeepSeek-V3、DeepSeek-V4、Kimi K2.6——都达到了与闭源前沿可比的性能水平，但它们的路径选择揭示了 2026 年开源 LLM 竞争的根本张力。

DeepSeek-V3（2024-12） 是开源 MoE 模型的里程碑。671B/37B 的参数规模以仅 $5.576M 的训练成本 [L2:2412.19437] 实现了与 GPT-4o 可比的性能，DualPipe + FP8 training + 自定义 all-to-all kernel 的基础设施三件套定义了万亿参数 MoE 训练的工程范式。V3 的 MLA 将 KV cache 压缩到 512 维，auxiliary-loss-free load balancing 实现了 14.8T token 零 loss spike 的训练稳定性 [L2:2412.19437]。

DeepSeek-V4（2026-04） 代表了"architecture-first"的演进路线。从 V3 到 V4 的 16 个月间（中经 V3.2），DeepSeek 做了三个替换级升级：MLA → hybrid CSA+HCA，标准 residual → mHC（doubly stochastic 约束），AdamW → hybrid AdamW+Muon [L2:deepseek-v4]。这些架构变革的直接收益是 1M context 下 KV cache 降至 V3.2 的 7–10%、FLOPs 降至 10–27% [L2:deepseek-v4]。V4 相信长上下文推理效率的瓶颈在架构层，必须通过重新设计 attention 来突破。

Kimi K2.6（2026-04） 代表了"post-training-first"的演进路线。从 K2（2025-07）到 K2.5（2026-02）再到 K2.6（2026-04），Moonshot 三次迭代都复用同一 1.04T MoE + MLA 主干——config.json 逐字相同 [L2:kimi-k2-6]。K2.6 的全部创新在 post-training 和 orchestration：长程 agentic coding RL 将单次 rollout 推到 4000+ 工具调用、12–13 小时连续执行 [L2:kimi-k2-6]，Agent Swarm 扩容 3× 至 300 sub-agent × 4000 steps，INT4 默认量化让 1T 模型可在单 8×H200 节点部署 [L2:kimi-k2-6]。

这两条路线的分野不仅是工程选择的差异，更是对"下一个瓶颈在哪里"的不同判断：V4 认为瓶颈在 attention 架构的效率墙，K2.6 认为瓶颈在模型的行为边界。比较三者可以回答一个根本问题：当架构创新和训练创新各自接近收益递减点时，它们会在何处交汇？

三个模型共享 DeepSeekMoE 的核心框架（shared expert + fine-grained routed experts），但在 attention、训练策略、部署方案上走向了截然不同的方向。V3 是共同起点；V4 在架构维度做纵深突破；K2.6 在行为维度做极限拉伸。

值得注意的参数设计演进：V4 将 active experts 从 V3/K2.6 的 8 降到 6，同时将总 expert 数增加（V4-Pro 384 vs V3 256）。这一"每 token 激活更少 expert 但总 expert 更多"的策略降低了每 token 的通信和计算成本，同时通过更大的 expert pool 保持或扩大模型容量 [L2:deepseek-v4]。K2.6 保持 384 routed top-8 的配置不动，把 MoE routing 的优化空间完全让给了 post-training。

§4 三条路线的技术演进

4.1 Attention 架构：MLA → CSA+HCA vs MLA 复用

V3 引入的 MLA（Multi-head Latent Attention）通过低秩投影将 KV cache 从 n_h × d_h 压缩到 512 维 latent，推理时只需缓存 576 float/token（512 latent + 64 RoPE），相比 BF16 GQA8 约 14× 压缩 [L2:2412.19437]。这一设计在 128K context 下已足够高效。

V4 认为 MLA 在 1M context 下仍然不够——即使只有 576 float/token，1M token 的 KV cache 仍占数百 GB。V4 用 hybrid CSA+HCA 完全替换了 MLA [L2:deepseek-v4]：

  • CSA（Compressed Sparse Attention）：先将每 m=4 个连续 KV entries 通过 softmax 加权压缩为 1 个（含相邻块 overlap，实际融合 2m=8 个），再用 Lightning Indexer 做 top-k sparse selection（V4-Pro k=1024）。综合压缩率 ≈ 序列长度/256 [L2:deepseek-v4]。
  • HCA（Heavily Compressed Attention）：采用 m'=128 的极端压缩率，不做 overlap，但保持 dense attention。1M token 经 HCA 压缩后仅 ~7800 个 entry [L2:deepseek-v4]。
  • 两者交错排列（interleaved），CSA 提供精细化局部+稀疏全局关注，HCA 提供粗粒度完整全局覆盖。

K2.6 则完全保留了 MLA 不做任何改动——config.json 的 kv_lora_rank=512、qk_rope_head_dim=64、num_attention_heads=64 与 K2.5 逐字相同 [L2:kimi-k2-6]。

Trade-off 分析：V4 的 CSA+HCA 在 1M context 下将 KV cache 降到 V3.2 的 7–10%，FLOPs 降到 10–27%——这是 MLA 无论如何无法达到的效率水平 [L2:deepseek-v4]。但代价是架构复杂度急剧上升：Lightning Indexer 需要 FP4 精度下的 index score 计算、Partial RoPE 需要仅在最后 64 维应用且做反向补偿、Attention Sink 需要可学习 sink logit——每个细节的缺失都可能导致长上下文质量显著下降 [L2:deepseek-v4]。K2.6 证明了 MLA 在 262K context + 强 post-training 的组合下仍能在大部分 agentic 任务上与 V4 持平——SWE-Verified 80.2 vs V4 的 80.6 [L2:kimi-k2-6] [L2:deepseek-v4]，差距不到 1 分。这意味着 attention 架构升级的边际收益高度依赖于场景：1M context 的学术 benchmark 上 V4 大幅领先（MRCR/CorpusQA 超越 Gemini-3.1-Pro），但 262K 以内的 agentic 任务中 MLA 已经"够用"。

4.2 训练稳定性：从零 spike 到 Anticipatory Routing

V3 在 14.8T token 的预训练中实现了零 loss spike、零 rollback [L2:2412.19437]，关键技术是 auxiliary-loss-free load balancing——每 expert 维护标量 bias，仅影响 routing decision 不影响 gating value，step 末动态 ±γ 调整 [L2:2412.19437]。

V4 继承了这套稳定性框架并新增两个技术 [L2:deepseek-v4]：

  • Anticipatory Routing：将 backbone 和 routing network 的参数更新解耦——step t 时用当前参数 θ_t 计算特征，但 routing indices 来自 θ_{t-Δt}（提前 Δt 步计算并缓存）。基础设施优化将额外开销控制在 ~20%，且仅在 loss spike 自动检测触发时启用。
  • SwiGLU Clamping：线性分量 clamp 到 [-10, 10]，gate 上界 cap 到 10。

K2.6 沿用 K2/K2.5 的 MuonClip——对 Muon optimizer 的更新步长做 clip 约束 [L2:kimi-k2-6]。由于 K2.6 不改架构且复用 K2.5 checkpoint，训练稳定性本就不是其核心挑战。

为什么 V4 需要额外的稳定性技术？ 因为 compressed attention 引入了更复杂的信息流模式——CSA 的 KV 压缩通过 softmax 加权叠加了可学习位置偏置，HCA 的 128× 极端压缩使信息在残差流中的传递更加脆弱。这就是 V4 同时需要 mHC（Birkhoff polytope 约束保证残差流非扩张性）和 Anticipatory Routing（隔离 routing outlier 对 backbone 的冲击）的深层原因 [L2:deepseek-v4]。相比之下，K2.6 保留标准 MLA + 标准 residual，梯度行为与 V3 类似，MuonClip 足以应对。

4.3 Post-training 范式：Specialist + OPD vs 长程 Agentic RL

这是三个模型之间最深层的哲学分歧。

V3 使用简单的 SFT + RL 流程 [L2:2412.19437]，post-training 仅占 5K GPU hours、$0.01M——不到总训练成本的 0.2% [L2:2412.19437]。

V4 将 post-training 彻底重构为两阶段 [L2:deepseek-v4]：

  1. Specialist Training：每个领域模型经过 fine-tuning + GRPO RL 优化，支持 Non-think / Think High / Think Max 三种 reasoning effort 模式。难验证任务使用 Generative Reward Model（GRM）。
  2. On-Policy Distillation（OPD）：10+ 个 domain specialist 通过多 teacher OPD 合并为统一模型。关键创新是 full-vocabulary logit distillation——保留完整 logit 分布做 reverse KL divergence，摒弃先前的 token-level KL 估计（高方差、不稳定）。Teacher 调度支持无限数量的万亿参数 teacher，仅缓存最后一层 hidden states 实时重建 logits。

    3. K2.6 走了完全不同的路线——极端长程 agentic RL [L2:kimi-k2-6]。Blog 报告的 flagship case 展示了 4000+ 工具调用、12-13 小时连续自主执行、14 轮内部迭代。Reward 不依赖单次对错，而是最终性能指标（throughput、latency、test pass rate）+ 过程质量（是否用 flame graph 分析、是否正确识别瓶颈）[L2:kimi-k2-6]。

    核心 trade-off：V4 将 RL 视为领域特定工具，先训练 specialist 再通过蒸馏统一；K2.6 将 RL 视为端到端训练信号，直接通过长程 rollout 将行为能力刻入权重。V4 的 OPD 工程复杂性在于 teacher 调度（10+ 万亿参数 teacher、full-vocab logit 的存储和传输）和确定性保证（batch-invariant kernel 是 OPD 的技术前提 [L3:deepseek-v4]）。K2.6 的 compute 成本在于长程 rollout 本身——单条轨迹占用 GPU 4-13 小时，RL batch 难以推高 [L2:kimi-k2-6]。两者可能最终合流：先用长程 RL 训练 domain specialist，再用 OPD 合并 [L3:deepseek-v4]。

    从 V3 到 V4 的 post-training 演进幅度值得单独审视。V3 的 post-training 仅占 5K GPU hours、$0.01M——不到总成本 0.2% [L2:2412.19437]。V4 的 post-training 虽未公布具体 GPU hours，但 Specialist Training + OPD 的复杂度远超 V3：每个 domain specialist 需要独立的 SFT→GRPO 流程，OPD 阶段需要 10+ teacher 同步 inference + student training。这反映了一个行业趋势——pre-training 的边际收益递减后，post-training 正在成为主要成本中心和差异化来源。K2.6 则代表了这一趋势的极端——底模成本几乎为零（复用 K2.5 checkpoint），全部工程投入在 post-training 阶段。

    4.4 推理效率与部署

    V3 的推理部署采用 prefill/decode 分离：prefill 使用 TP4+SP+EP32+DP8（4 节点 32 GPU），decode 使用 TP4+SP+EP320+DP80（40 节点 320 GPU）[L2:2412.19437]。MLA 的 512d KV cache 使推理显存极低，MTP speculative decoding 带来 1.8× TPS（85-90% acceptance rate）[L2:2412.19437]。

    V4 的推理框架因 CSA+HCA 引入了前所未有的复杂度 [L2:deepseek-v4]：

    • 异构 KV Cache 布局：State Cache（固定大小，存 SWA 和 tail token）+ Classical KV Cache（CSA 和 HCA 的压缩 entry）+ Lightning Indexer 独立 embedding，不同层有不同 hit/eviction 策略。
    • On-Disk KV Cache 三种策略：Full SWA Caching（零冗余但 SSD 写密集）、Periodic Checkpointing（每 p 个 token checkpoint）、Zero SWA Caching（不存 SWA，恢复时重算）。
    • FP4 QAT：MoE expert 权重 FP4 → FP8 无损反量化 → FP8 计算，CSA indexer QK path 以 FP4 缓存。

    K2.6 的部署方案极其简洁 [L2:kimi-k2-6]：INT4 compressed-tensors 仅量化 routed MoE experts（~92% 参数），attention / shared expert / lm_head 保留 FP16。总权重 ~258 GB，单 8×H200 节点（1128 GB HBM）可舒适运行。MLA 的 576 float/token KV cache 在 262K 下仅 ~290 MB/seq [L2:kimi-k2-6]。

    Trade-off：V4 的推理方案更高效但更复杂——heterogeneous KV cache 打破了 PagedAttention 等标准框架的假设，需要定制推理引擎。K2.6 直接复用 vLLM/SGLang 生态（DeepseekV3ForCausalLM + compressed-tensors 分支），部署门槛极低 [L2:kimi-k2-6]。结果是 V4 能服务 1M context 但需要重建推理栈，K2.6 限于 262K 但开箱即用。

    4.6 Kernel 与基础设施

    V3 的基础设施定义了 MoE 训练的工程范式 [L2:2412.19437]：

    • DualPipe 双向 pipeline scheduling，pipeline bubble 减少 60%+ vs 1F1B
    • 自定义 all-to-all kernel（PTX 手写），20 SM/132 SM 专用于通信
    • FP8 混合精度训练框架（tile-wise 1×128 activation + block-wise 128×128 weight）
    • 内存优化完全消除 Tensor Parallelism

    V4 在 V3 基础上做了四个维度的升级 [L2:deepseek-v4]：

    1. TileLang DSL：替代数百个细粒度 Torch ATen 算子为少数融合 kernel，集成 Z3 SMT solver 做整数表达式形式分析，默认禁用 fast-math 保证数值可复现性。CPU 端验证开销从数十微秒降到 <1μs。
    2. Batch-invariant deterministic kernels：端到端替换 cuBLAS 为 DeepGEMM，放弃 split-k；attention backward 用独立累加 buffer 替代 atomicAdd；MoE backward 通过预处理 token 顺序保证确定性。这是 OPD full-vocab distillation 的技术前提 [L3:deepseek-v4]。
    3. MoE fused EP kernel：将 Dispatch/Linear-1/Linear-2/Combine 四阶段做 wave 粒度 fine-grained overlap，通用推理加速 1.50–1.73×，低延迟场景最高 1.96×（已开源为 DeepGEMM MegaMoE）。
    4. DSec Sandbox：生产级 Rust 沙箱平台，管理数十万并发沙箱实例，支持确定性重放。

      5. K2.6 在基础设施上采取完全不同的策略——复用现有生态 [L2:kimi-k2-6]：

      • 文本主干走 DeepSeek-V3 家族的 MLA kernel + sigmoid noaux_tc MoE routing
      • 新增 compressed-tensors INT4 解压与 GEMM 融合 kernel（vLLM/SGLang 已原生支持）
      • 推理调度复用 vLLM/SGLang 的 kimi_k25.py 模型文件，处理 compressed-tensors 分支即可加载

      V4 和 K2.6 在 kernel 层面的差异本质上反映了它们对"什么值得自研"的不同判断：V4 认为 compressed attention 架构需要从 kernel DSL 到 deterministic training 的全栈自研；K2.6 认为标准 MLA/MoE kernel 已足够成熟，工程资源应投入 RL 训练基础设施（长程 rollout 的 off-policy reuse、trajectory-level reward shaping、4-13h 的 fault-tolerant rollout 管理 [L2:kimi-k2-6]）。

      V4 的 kernel 创新对硬件设计也提出了四个具体建议 [L2:deepseek-v4]：(1) 通信带宽每 GBps 可隐藏 6.1 TFLOP/s 计算（C/B ≤ 6144 FLOPs/Byte）；(2) 极端 kernel fusion 驱动所有组件同时高负载，需充足功耗余量；(3) Push-based 通信需更低延迟的跨 GPU signaling；(4) 建议用低成本逐元素 activation 替代 SwiGLU（避免 exp/div 操作）。这些建议直接影响下一代 GPU 的 MoE 优化方向。

      §5 核心洞察

      洞察 1："架构创新 vs 训练/编排创新"是 2026 年开源 LLM 的根本路线分歧

      V4 证明架构创新仍有巨大空间——CSA+HCA 将 1M context 下的 KV cache 压到 MLA 的 1/10 [L2:deepseek-v4]，这不是微调能达到的。K2.6 证明同一架构可以通过 post-training 持续提升——Toolathlon 从 K2.5 的 27.8 跳到 50.0（+80%），MCPMark 从 29.5 到 55.9（+89%），仅靠长程 RL 和编排 [L2:kimi-k2-6]。

      这两条路线并不互斥，但它们对工程团队的能力要求截然不同：V4 路线需要 attention 架构设计 + kernel DSL 开发 + 确定性训练基础设施的全栈能力；K2.6 路线需要长程 RL 的 reward engineering + 大规模 rollout 基础设施 + 部署优化能力。未来的融合路径可能是：用 V4 式架构提供底层效率，用 K2.6 式 RL 提供行为能力 [L3:deepseek-v4]。

      洞察 2：Muon optimizer 正在成为万亿参数训练的新标配

      V4 和 K2.x 系列都采用了 Muon 变体，取代了纯 AdamW。但两者的实现路径不同：

      • V4 的 Hybrid Muon：10 次 Newton-Schulz 迭代分两阶段（前 8 次快速收敛 + 后 2 次稳定到奇异值=1），BF16 矩阵乘法即可稳定，MoE 梯度通信通过 BF16 随机舍入减半 [L2:deepseek-v4]。
      • K2.x 的 MuonClip：对更新步长做 clip 约束，与 token-level log-ratio clip + KL penalty + Toggle 组合使用 [L2:kimi-k2-6]。

      AdamW → Muon 的转变反映了万亿参数 MoE 训练对优化器的新需求：梯度近似正交化比简单的 momentum 更适合高维稀疏参数空间。两种 Muon 变体的收敛速度和稳定性比较尚无公开消融数据，但 V4 的 33T token 训练和 K2.x 的三代迭代都验证了 Muon 家族在万亿级规模上的可靠性。

      洞察 3：OPD vs 长程 RL 是 post-training 的核心 trade-off

      V4 的 OPD 和 K2.6 的长程 RL 代表了 post-training 的两种极端策略 [L2:deepseek-v4] [L2:kimi-k2-6]：

      维度V4 OPDK2.6 长程 RL
      核心思路多 teacher 蒸馏统一模型直接通过长 rollout 训练
      工程复杂性10+ teacher 调度、full-vocab logit 存储/传输、确定性 kernel 前提4-13h rollout 的 fault tolerance、trajectory reward shaping
      Compute 瓶颈Teacher inference + student training 交替单条轨迹占用 GPU 数小时
      适用范围可覆盖多领域（reasoning、code、knowledge 各 specialist）强依赖可执行 oracle（测试、性能指标）
      可扩展性Teacher 数量理论无上限Rollout 长度受 gradient signal 衰减限制

      V4 的 OPD 用 full-vocabulary KL loss 取代 token-level KL estimate，解决了高方差问题但带来了 logit 存储成本；K2.6 的长程 RL 把 reward horizon 从分钟推到小时，但 gradient signal 在 4000+ step 后的衰减程度未知 [L2:kimi-k2-6]。从 benchmark 看，两种策略在 SWE-Verified 上达到了几乎相同的水平（80.6 vs 80.2），但各自在不同维度上更强：V4 在需要深度推理的任务上领先（Codeforces 3206），K2.6 在需要持久执行的任务上领先（SWE-Bench Pro 58.6 全场最高）。

      洞察 4：1M context——杀手级应用尚待验证

      V4 是目前唯一原生支持 1M context 的开源模型，K2.6 止步于 262K [L2:deepseek-v4] [L2:kimi-k2-6]。V4 在 MRCR 1M 和 CorpusQA 上超越 Gemini-3.1-Pro [L2:deepseek-v4]，证明 1M 的学术能力已经到位。

      但 K2.6 用 262K 在大部分 agentic 任务中已经"够用"——SWE-Bench Pro、Terminal-Bench 2.0、Toolathlon 等核心 agent benchmark 中，K2.6 不逊于甚至超越 V4。这引发一个关键问题：1M context 的真正 killer app 是什么？ V4 的 Interleaved Thinking（跨 user message 保留推理内容）是一个候选——在 tool-calling 场景下充分利用 1M context 的连续推理链 [L2:deepseek-v4]。但这是否足以证明 CSA+HCA + mHC + on-disk KV cache 等一系列架构复杂性的合理性，仍待实际部署验证。

      V4 为 1M context 付出的工程代价是巨大的：异构 KV cache 布局、三种 on-disk SWA 策略、FP4 QAT、batch-invariant deterministic kernels [L2:deepseek-v4]。如果 262K 在 agentic workload 中就够用，那么 V4 的效率优势可能主要体现在成本端（更少的 KV cache GPU 占用 → 更高的 batch size → 更低的单位成本），而非能力端。

      洞察 5：端到端确定性在 V4 中从可选变为必须

      V3 没有显式强调确定性（使用 cuBLAS + standard attention）[L2:2412.19437]。V4 投入了大量工程在 batch-invariant attention、deterministic MoE backward、mHC MatMul 确定性上 [L2:deepseek-v4]。

      这不是学术洁癖。V4 的 OPD 要求 student 和 teacher 的 forward pass 在相同输入下产生完全相同的 logits——否则 full-vocabulary KL loss 的梯度估计会有不可控的方差。FP4 QAT 要求 rollout 和 inference 使用完全相同的量化权重——否则 train-deploy 一致性失效 [L3:deepseek-v4]。batch-invariant kernel 是 OPD pipeline 的技术前提，不是优化选项。

      K2.6 不需要这套确定性保证，因为它不做 full-vocab distillation——长程 RL 的 reward signal 来自外部 oracle（测试结果、性能指标），不依赖 forward pass 的 bitwise 一致性 [L2:kimi-k2-6]。这说明确定性需求是由 post-training 范式决定的：OPD 路线必须确定性，长程 RL 路线可以放松。

      洞察 6：部署复杂度是隐性成本

      维度DeepSeek-V4Kimi K2.6
      推理框架需要定制（异构 KV cache）复用 vLLM/SGLang
      KV Cache 管理State Cache + Classical KV + Indexer标准 MLA latent cache
      量化方案FP8 + FP4 QAT（训练时集成）INT4 compressed-tensors（post-training 量化）
      最小部署规模多节点（推测，未公布）单 8×H200 节点 (~258 GB)
      生态兼容性需要适配新 attention 架构直接走 DeepseekV3ForCausalLM 路径

      [L2:deepseek-v4] [L2:kimi-k2-6]

      V4 的推理方案更高效（1M context + 更低 KV cache 占用）但部署门槛更高。K2.6 的 INT4 routed-only 量化把 1T 模型的部署门槛从多节点拉到单节点 [L2:kimi-k2-6]，这对开源生态的可达性是质的变化。对于中小团队，K2.6 的部署友好度可能比 V4 的理论效率更有实际价值。

      洞察 7：V3 的遗产在两条路线中都被深度继承

      尽管 V4 和 K2.6 走向了不同方向，它们都深度依赖 V3 的遗产：

      • DeepSeekMoE：fine-grained routed experts + shared expert 的架构在三个模型中完全保留（V4 从 8 active 降到 6，但框架不变）[L2:2412.19437] [L2:deepseek-v4]
      • MTP：Multi-Token Prediction depth=1 在 V4 中沿用 [L2:deepseek-v4]
      • DualPipe：V4 的 pipeline 调度仍基于 DualPipe 框架，但 mHC 的引入需要调整 overlapped 1F1B stage 的时序（mHC 开销仅占 6.7% [L2:deepseek-v4]）
      • MLA 血统：K2.6 的 DeepseekV3ForCausalLM 文本主干直接继承 V3 架构，HF auto_map 中的类名甚至仍是 KimiK25ForConditionalGeneration [L2:kimi-k2-6]
      • 训练稳定性基线：V3 建立的 auxiliary-loss-free load balancing（dynamic bias ±γ）在两条路线中都被保留或增强

      V3 不仅是一个模型发布，更是一套工程范式。它在 2024-12 建立的 MoE 训练基础设施（DualPipe + FP8 + custom all-to-all）成为后续两代模型的共同地基 [L2:2412.19437]。V3 的另一个重要遗产是证明了 MoE 的 scaling path 可行性：37B active params 以 $5.576M 达到 GPT-4o 级别性能 [L2:2412.19437]，这个成本-性能基线激励了 V4 和 K2.x 系列的后续投入。

      §6 前瞻

      V5 会融合 K2.6 的长程 RL 吗？ V4 的 OPD 和 K2.6 的长程 RL 并不互斥。最自然的融合路径是：先用长程 agentic RL 训练 domain specialist（利用外部 oracle reward），再用 OPD full-vocab distillation 合并到统一模型 [L3:deepseek-v4]。V4 已经有完整的 specialist + OPD 基础设施，只需在 specialist training 阶段引入更长 horizon 的 RL。

      Kimi K3 会改架构吗？ K2 系列三次迭代都复用同一主干，说明瓶颈正从架构转向 RL/编排 [L2:kimi-k2-6]。但 Moonshot 内部已有 Kimi Linear（KDA linear attention），且 K2.6 在 262K 以上的 context 扩展空间有限。K3 可能采用 KDA + MLA hybrid 架构，或引入类似 V4 CSA 的 KV 压缩方案来突破 context 瓶颈——同时继承 K2.6 的长程 RL 训练方法论。

      MoE + Chiplet GPU 的协同机会。 V4 的 MoE fused EP kernel 已实现 expert 分割为 wave 的 fine-grained overlap，但假设 NVIDIA 的 monolithic L2 cache。AMD MI350/MI400 的 chiplet 架构（每颗 XCD 有私有 L2）与 MoE 的 expert 稀疏激活天然亲和——不同 expert 可驻留不同 XCD 的 L2，gating network 做 expert→XCD affinity routing [L3:deepseek-v4]。V4 的 MoE fused kernel 对硬件提出的四个建议（C/B ratio、功耗余量、push-based signaling、activation 简化）可以直接指导 chiplet-aware MoE kernel 的设计 [L2:deepseek-v4]。这是一个两个生态都未充分开发的方向。

      FP4 硬件支持将进一步改变效率格局。 V4 的 FP4 QAT 目前在 FP4→FP8 无损反量化后以 FP8 计算 [L2:deepseek-v4]。未来硬件原生支持 FP4 GEMM 时，V4 的 routed expert 理论效率可再提升约 1/3。K2.6 的 INT4 weight-only 量化也将受益于更高效的 INT4×FP16 GEMM kernel。两条路线在量化层面正在趋同——都走向"MoE routed experts 低精度、attention/shared expert 高精度"的混合精度部署。

      开源 vs 闭源的差距正在非均匀收窄。 V4-Pro-Max 在 Codeforces（3206 vs GPT-5.4 3168）、Putnam-2025（满分）、LiveCodeBench-v6（93.5 vs Gemini 91.7）等 reasoning 类 benchmark 上已经超越闭源 [L2:deepseek-v4]。K2.6 在 SWE-Bench Pro（58.6 领先全场）和 DeepSearchQA（f1 92.5 大幅领先 GPT 78.6）上登顶 [L2:kimi-k2-6]。但在复杂 agent 场景（Terminal-Bench 2.0 V4 67.9 / K2.6 66.7 vs GPT-5.4 75.1）和事实性知识（SimpleQA-Verified V4 57.9 vs Gemini 75.6）上仍有显著差距 [L2:deepseek-v4]。开源在"深度推理"和"领域 coding"上已持平或超越闭源，但在"广度知识"和"复杂多步 agent"上仍有 7-18 分的 gap，这可能需要更大的预训练数据量和更长的 RL rollout 来弥合。

      收敛点：当架构创新和 post-training 创新在同一个模型上叠加。 当前的分野源于工程资源的有限性——架构创新（V4 需要重建 attention + residual + optimizer + kernel）和 post-training 创新（K2.6 需要长程 RL 基础设施 + agent orchestration）各自消耗了大量工程带宽。但技术上没有理由两者不能共存。下一代前沿开源模型的竞争可能转向"谁能最先在 CSA+HCA 级别的高效架构上叠加 4000+ step 的长程 agentic RL"——这需要同时掌握 kernel DSL、确定性训练、RL rollout 基础设施和 agent orchestration 四个维度的工程能力。能做到这一点的团队，将定义 2027 年开源 LLM 的技术边界。

      从 V3 的 $5.576M 到 V4 的万亿参数规模再到 K2.6 的单节点部署，开源前沿 MoE 模型在 16 个月内完成了从"证明可行"到"分路线深耕"的跃迁。这两条路线的竞争和融合，将决定开源 LLM 在 reasoning、agent、long-context 三个核心维度上能走多远。
