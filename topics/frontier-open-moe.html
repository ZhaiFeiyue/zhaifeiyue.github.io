开源前沿 MoE 大模型：从 DeepSeek-V3 到 V4、Kimi K2 到 K2.6 的路线分野与基础设施演进 #

§1 主题缘起 #

2024 年 12 月到 2026 年 5 月的 18 个月里，开源 MoE 大模型经历了三代跨越式演进，催生了两条截然不同的技术路线——架构优先（DeepSeek）和训练/编排优先（Moonshot）。与此同时，为这些万亿参数模型服务的推理和训练基础设施也经历了同等量级的范式变迁：从单集群 RDMA serving 到跨数据中心 PD 分离 [ref:2604.15039#Q2方法]，从 request 级调度到 program-aware agentic 编排 [ref:2602.13692#summary]，从通用 kernel 到 persistent megakernel + chiplet-aware 调度 [ref:tile-ai-tilert#Q3结果] [ref:2604.15379#summary]。

模型侧，DeepSeek-V3（2024-12）以 671B/37B MoE + MLA + DualPipe 奠基了万亿参数训练的工程范式，仅 $5.576M 训练成本即达到 GPT-4o 可比性能 [ref:2412.19437#key-findings]。V4（2026-04）做了三个替换级升级——MLA → hybrid CSA+HCA、标准 residual → mHC、AdamW → hybrid Muon——将 1M context 下 KV cache 降至 V3.2 的 7–10%、FLOPs 降至 10–27% [ref:deepseek-v4#§2-核心三问]。Kimi K2（2025-07）到 K2.6（2026-04）的四代迭代则复用同一 1.04T MoE + MLA 主干——config.json 逐字不变 [ref:kimi-k2-6#key-findings]——把全部创新投入 MuonClip 优化器 [ref:2507.20534#summary]、多模态对接 [ref:2602.02276#key-findings]、长程 agentic RL（4000+ 工具调用、12–13 小时连续执行）[ref:kimi-k2-6#核心三问-q1q2q3]、以及 INT4 单节点部署 [ref:kimi-k2-6#key-findings]。Kimi Linear 则开辟了第三条路线——KDA 线性注意力在公平对比下首次全面超越全注意力 [ref:2510.26692#core-contribution]。

框架侧，这些模型的部署驱动了一整代新的基础设施创新。DualPath 把 PD 分离的存储带宽瓶颈从 prefill 端扩展到 decode 端聚合 [ref:2602.21548#核心三问]；PrfaaS 利用混合注意力将 KV 吞吐降 4–13× 使跨 DC Ethernet PD 分离成为现实 [ref:2604.15039#Q2方法]；ThunderAgent、Halo、KVFlow 把调度粒度从 per-request 提升到 per-program / per-workflow [ref:2602.13692#summary] [ref:2509.02121#summary] [ref:2507.07400#summary]；TileRT 将整个 DeepSeek-V3.2 模型 AOT 编译为单个 persistent Engine Kernel 达到 600 tok/s [ref:tile-ai-tilert#Q3结果]。

这两个维度——模型架构路线选择和服务基础设施演进——正在深度耦合。V4 的 CSA+HCA 需要定制推理栈（异构 KV cache 布局打破 PagedAttention 假设）[ref:deepseek-v4#§3-架构与方法]，K2.6 的长程 RL 需要 fault-tolerant rollout 管理和 TB 级权重分发 [ref:2604.09107#核心三问]，Kimi Linear 的混合注意力使跨 DC PD 分离成为可能 [ref:2604.15039#Q2方法]。理解开源前沿 MoE 模型的全貌，必须同时看到模型设计和基础设施这两面。

§5 技术线交错 #

5.1 Attention 架构选择如何约束推理基础设施 #

V4 的 CSA+HCA 在模型层实现了 1M context 下 KV cache 降至 MLA 的 1/10 [ref:deepseek-v4#§2-核心三问]，但代价是引入了异构 KV cache 布局——State Cache（固定大小，存 SWA 和 tail token）+ Classical KV Cache（CSA/HCA 压缩 entry）+ Lightning Indexer 独立 embedding [ref:deepseek-v4#§3-架构与方法]。这打破了 PagedAttention 等标准框架的假设，需要定制推理引擎。相比之下，K2.6 的标准 MLA 允许直接复用 vLLM/SGLang 生态的 DeepseekV3ForCausalLM 路径 [ref:kimi-k2-6#infrastructure-impact]。

Kimi Linear 的 KDA:MLA 3:1 hybrid 则开辟了第三种可能：KDA 层只有 O(1) fixed-size recurrent state（128×128/head），MLA 层有 per-token KV cache，综合 KV 吞吐降到标准 MLA 的 ~1/4 [ref:2510.26692#key-findings]。PrfaaS 正是利用了这一特性，将单实例 KV 吞吐从 ~60 Gbps（dense 模型）降到 ~3 Gbps，使跨 DC Ethernet PD 分离成为现实 [ref:2604.15039#Q2方法]。

5.2 MoE 稀疏性如何驱动通信优化 #

K2 的 sparsity 48（384 experts / top-8）比 sparsity 8 节省 1.69× FLOPs [ref:2507.20534#key-findings]，V4-Pro 则用 384 routed + top-6 进一步降低每 token 的 expert 激活数 [ref:deepseek-v4#§3-架构与方法]。但更高的稀疏度意味着更大的 all-to-all 通信压力。V3 的 DualPipe 用 20/132 SM 专用通信实现了完全隐藏 [ref:2412.19437#key-findings]，V4 的 MoE fused EP kernel 将 Dispatch/Linear-1/Linear-2/Combine 四阶段做 wave 粒度 fine-grained overlap，通用推理加速 1.50–1.73× [ref:deepseek-v4#§3-架构与方法]。ZeRO-Prefill 则从根本上反转了 EP 数据流方向——用 AsyncEP 把 expert weight 流式搬运到计算所在 GPU，完全消除 AllToAll [ref:2605.02960#核心三问]。V4 提出的 "C/B ≤ 6144 FLOPs/Byte" 准则将通信带宽与可隐藏计算量化挂钩 [ref:deepseek-v4#§3-架构与方法] [ref:framework#轴2-通信隐藏]。

FaaSMoE 提出了第三条截然不同的路线——将 MoE 的稀疏激活与 serverless FaaS 的 event-driven scale-to-zero 做结构映射，把 expert computation offload 为无状态函数调用，多租户共享同一 expert pool 消除 per-tenant 冗余驻留 [ref:2604.26881#Q2方法]。然而这条路线暴露了一个根本张力：FaaSMoE 的 CPU-only 原型（tinyFaaS + HTTP invocation）与 GPU-native MoE serving 的现实完全脱节——GPU 上 expert forward 耗时 ~0.1ms，而 FaaS HTTP round-trip 需 ~10ms，invocation overhead 是 computation 的 100×；且其 30 requests / 6 tenants 的微小评估规模和 "per-tenant full replication" 的过时 baseline 使定量结论不可外推 [ref:2604.26881#实验与数据]。这一张力恰好界定了 MoE serving 优化的两个极端：ZeRO-Prefill 追求"所有 expert 本地可用"以消除通信（GPU-native, single-tenant），FaaSMoE 追求"仅活跃 expert 存在"以消除冗余（serverless, multi-tenant）——两者设计假设相互矛盾，但其间的混合策略（如 GPU-native expert-as-a-service + popularity-aware tiering）可能是 multi-tenant MoE serving 的可行方向。

5.3 长程 Agentic RL 如何倒逼训练和推理框架 #

K2.6 的 4000+ 工具调用、12–13 小时连续执行 [ref:kimi-k2-6#核心三问-q1q2q3] 对框架层面提出了三个前所未有的要求：(1) Fault-tolerant rollout——单条轨迹占用 GPU 4-13 小时，任何中断都意味着巨大浪费；(2) TB 级权重分发——RL 的 trainer→rollout 每步需要将新版万亿参数权重传输到 rollout worker，TensorHub 的 Reference-Oriented Storage 解决了这一瓶颈（1024 GPU stall 减少 6.7×）[ref:2604.09107#核心三问]；(3) Agentic 推理调度——rollout 期间的 LLM 调用是多轮、短追加、DAG 依赖的 agentic workload，ThunderAgent 的 program-aware 调度（1.48–3.58× 吞吐提升）和 KVFlow 的 workflow-aware 缓存驱逐（1.83–2.19× 加速）都是直接为这类负载设计的 [ref:2602.13692#summary] [ref:2507.07400#summary]。

V4 的 OPD（On-Policy logit-level knowledge transfer, full-vocabulary KL divergence）则对框架有不同的约束：它要求 student 和 teacher 的 forward pass 在相同输入下产生完全相同的 logits，因此需要 batch-invariant deterministic kernels 作为技术前提 [ref:deepseek-v4#§3-架构与方法] [ref:model#轴6-post-training]。

5.4 硬件能力如何开启新的模型-框架协同 #

V4 的 FP4 QAT（MoE expert 权重 FP4→FP8 无损反量化→FP8 计算）[ref:deepseek-v4#§3-架构与方法] 和 K2.6 的 INT4 compressed-tensors [ref:kimi-k2-6#key-findings] 都采用了"MoE routed experts 低精度、attention/shared expert 高精度"的混合精度部署。Massive Values 研究表明这一策略有理论依据——Q/K 低频维度的 concentrated massive values 承载上下文知识理解的关键功能，保护这些大值的量化方法（如 AWQ）显著优于不保护的方法（如 GPTQ）[ref:2502.01563#key-findings]。

AMD MI350 的 chiplet 架构（8 颗 XCD 各自 4 MB 私有 L2）与 MoE 的 expert 稀疏激活天然亲和。Fleet 的 Chiplet-task 四级抽象把 "一颗 XCD + 其 L2" 作为显式作用域，在 LLM decode 上实现 1.3–1.56× 加速 [ref:2604.15379#summary]。V4 的 MoE fused kernel 对硬件提出的四个建议（C/B ratio、功耗余量、push-based signaling、activation 简化）可以直接指导 chiplet-aware MoE kernel 设计 [ref:deepseek-v4#§3-架构与方法]。

§6 共识与分歧 #

共识 1：DeepSeekMoE 的 shared + fine-grained routed experts 框架已成标准 #

DeepSeek-V3（1 shared + 256 routed top-8）[ref:2412.19437#key-tables]、V4-Pro（1 shared + 384 routed top-6）[ref:deepseek-v4#§3-架构与方法]、K2/K2.5/K2.6（1 shared + 384 routed top-8）[ref:2507.20534#summary] [ref:kimi-k2-6#summary]、Kimi Linear（1 shared + 256 routed top-8）[ref:2510.26692#summary] 全部采用此框架。四个独立团队（同一架构族）的验证表明 fine-grained MoE + shared expert 是当前万亿参数开源模型的事实标准。

共识 2：Muon 变体正在取代纯 AdamW #

V4 用 hybrid AdamW+Muon（10 次 Newton-Schulz 迭代）[ref:deepseek-v4#§3-架构与方法]，K2/K2.5/K2.6 用 MuonClip（QK-clip 约束）[ref:2507.20534#summary] [ref:kimi-k2-6#infrastructure-impact]，两条路线独立验证了 Muon 家族在万亿参数 MoE 中的可靠性。V3 是最后一个使用纯 AdamW 的前沿开源 MoE [ref:2412.19437#summary]。

共识 3：Post-training 正成为主要成本中心 #

V3 的 post-training 仅占 5K GPU hours / $0.01M（不到总成本 0.2%）[ref:2412.19437#key-tables]。V4 的 Specialist + OPD 需要 10+ domain specialist 各自走 SFT→GRPO 流程再做 full-vocab logit-level 合并 [ref:deepseek-v4#§3-架构与方法]。K2.6 则将全部工程投入 post-training（底模成本几乎为零，复用 K2.5 checkpoint）[ref:kimi-k2-6#summary]。

分歧 1：下一个瓶颈在 attention 架构还是行为能力？ #

V4 认为瓶颈在 attention 效率墙——CSA+HCA 将 1M context 下 FLOPs 降至 V3.2 的 27%，这不是 post-training 能达到的效率水平 [ref:deepseek-v4#§2-核心三问]。K2.6 用相同的 MLA 架构通过长程 RL 实现了 Toolathlon +80%、MCPMark +89% 的跳跃 [ref:kimi-k2-6#key-findings]。SWE-Verified 上两者几乎持平（80.6 vs 80.2）[ref:deepseek-v4#§2-核心三问] [ref:kimi-k2-6#key-findings]，说明在 262K 以内的 agentic 任务中两条路线收敛，但 1M 场景下差距显著。

分歧 2：位置编码——RoPE vs NoPE vs Partial RoPE #

K2 系列沿用 RoPE + YaRN 扩展到 262K [ref:2602.02276#summary]。Kimi Linear 完全放弃 RoPE，MLA 层用 NoPE，位置信息靠 KDA gates 提供，context 1M 且无外推问题 [ref:2510.26692#key-findings]。V4 用 Partial RoPE（仅最后 64 维），并在输出上施加反向补偿 [ref:deepseek-v4#§3-架构与方法]。Massive Values 研究揭示 RoPE 低频维度承载关键语义功能 [ref:2502.01563#key-findings]——Kimi Linear 的 NoPE 方案依赖 KDA 的 data-dependent gate 隐式补偿，但缺乏对应的机理分析 [ref:model#冲突1-位置编码]。

分歧 3：训练稳定性——MuonClip vs Anticipatory Routing #

K2 的 MuonClip(QK-clip) 针对 Muon 导致的 attention logit 爆炸，self-deactivating 且仅 12.7% heads 触发 [ref:2507.20534#key-findings]。V4 的 Anticipatory Routing 解耦 backbone 和 routing 参数更新，仅在 loss spike 时启用 [ref:deepseek-v4#§3-架构与方法]。两者针对不同 failure mode：MuonClip 针对 optimizer-level 的 QK 累加增长，Anticipatory Routing 针对 architecture-level 的 routing outlier 放大 [ref:model#冲突5-训练稳定性]。V4 同时使用 Muon + Anticipatory Routing + SwiGLU Clamping 三管齐下，说明万亿参数 MoE 中单一稳定性技术不够。

分歧 4：EP 通信策略——优化 AllToAll vs 消除 AllToAll #

V3 的 DualPipe 和 V4 的 MoE fused EP kernel 花大量精力优化 AllToAll 通信调度 [ref:2412.19437#核心三问] [ref:deepseek-v4#§3-架构与方法]。ZeRO-Prefill 论证 AllToAll 是"零价值通信"应被消除，用 AsyncEP 完全替代 [ref:2605.02960#核心三问]。根源在于 workload 差异：decode 和训练时 per-token compute window 太短无法隐藏 weight AllGather——AllToAll 仍优；大 batch prefill 时 compute window 足够长——weight streaming 更优 [ref:framework#冲突3-zeroprefill-vs-dualpipe]。

§7 根本性困难 #

困难 1：Compressed Attention 与通用推理框架的兼容性 #

V4 的 CSA+HCA 引入了异构 KV cache（State Cache + Classical KV + Indexer embedding）[ref:deepseek-v4#§3-架构与方法]，打破了 vLLM/SGLang 的 PagedAttention 假设。K2.6 能直接复用 vLLM 生态 [ref:kimi-k2-6#infrastructure-impact]，但 V4 需要定制推理栈。这不仅是工程量问题——它意味着每一次 compressed attention 架构升级都需要重写推理引擎、重新验证 correctness、重新优化 scheduling 策略。当前没有通用框架能同时高效支持 MLA、CSA+HCA、和 KDA hybrid 三种 KV cache layout。需要 model、framework 两个领域协同设计一个 extensible 的 attention backend 抽象。

部分尝试：TileRT 的 AOT 编译方式 [ref:tile-ai-tilert#Q2方法] 和 TokenSpeed 的 Placement 编译器 [ref:tokenspeed#核心三问] 展示了"整模编译"思路，但两者都局限于 BS=1 且不支持 dynamic batching。

预估难度：2–3 年。需要编译器团队和推理框架团队的深度协作。

困难 2：长程 RL 的 Gradient Signal 衰减 #

K2.6 将单次 rollout 推到 4000+ 工具调用、12–13 小时 [ref:kimi-k2-6#核心三问-q1q2q3]，但 blog 未披露 gradient signal 在 4000+ step 后的衰减程度 [ref:kimi-k2-6#limitations]。这是一个理论性的 credit assignment 瓶颈：在数千步的 trajectory 中，某个关键决策（如"是否使用 flame graph 分析性能瓶颈"）对最终 reward 的归因信号极其微弱。V4 的 OPD 绕过了这一问题（用 logit 级别的 teacher 信号），但 OPD 无法训练 agentic 行为（teacher 本身需要先具备该能力）。

部分尝试：K2.6 使用过程质量 reward（是否正确识别瓶颈、是否做了 profiling）[ref:kimi-k2-6#summary]；K2.5 的 PARL 用 CriticalSteps 度量过程效率 [ref:2602.02276#summary]。但这些都是 reward engineering 的 heuristic，没有理论保证。

预估难度：3–5 年。需要 RL 理论（credit assignment in long-horizon MDP）和系统工程（fault-tolerant rollout）的共同突破。

困难 3：端到端确定性训练在异构硬件上的扩展 #

V4 投入大量工程实现 batch-invariant deterministic kernels（替换 cuBLAS 为 DeepGEMM、放弃 split-k、用独立累加 buffer 替代 atomicAdd）[ref:deepseek-v4#§3-架构与方法]——这是 OPD full-vocab logit matching 的技术前提。但这套确定性保证深度绑定 NVIDIA H800 硬件。当 V4 迁移到 AMD MI350（chiplet 架构，8 颗 XCD 各自私有 L2）[ref:2604.15379#summary] 或其他加速器时，确定性保证需要从头重建。Fleet 的 chiplet-aware 调度已经展示了 L2 cache 行为的非确定性来源 [ref:2604.15379#summary]。

预估难度：未知。取决于硬件厂商是否提供确定性执行的 ISA 级保证。

困难 4：Agentic Serving 的 KV-Cache 生命周期管理 #

K2.6 的 300 sub-agent × 4000 步 Agent Swarm [ref:kimi-k2-6#key-findings] 和 V4 的 Interleaved Thinking（跨 user message 保留推理内容利用 1M context）[ref:deepseek-v4#§3-架构与方法] 都要求推理引擎在小时级时间跨度内管理海量 KV cache。当前最先进的解决方案——ThunderAgent 的 exponential time-decay caching [ref:2602.13692#summary]、KVFlow 的 steps-to-execution 优先级 [ref:2507.07400#summary]、Halo 的 epoch-level DP 优化 [ref:2509.02121#summary]——仍然各自为政，没有统一的 KV cache 生命周期管理框架能同时优化：多 agent 之间的 prefix 共享、agent 暂停/恢复时的 KV 保留/驱逐决策、跨节点 KV 迁移。

预估难度：1–2 年。技术上可行但需要 model（attention 架构决定 KV shape）和 framework（调度决定 KV 生命周期）的协同设计。

§8 成熟度判断 #

整体判断：Research-frontier，部分 Production-ready。

Production-ready 组件：

  • DeepSeekMoE 架构（shared + fine-grained routed experts）：四代模型验证，已成事实标准 [ref:2412.19437#key-findings] [ref:deepseek-v4#§3-架构与方法] [ref:2507.20534#summary]
  • MLA + 标准推理部署（K2.6 路线）：INT4 单 8×H200 节点可服务 1T 模型，vLLM/SGLang 原生支持 [ref:kimi-k2-6#key-findings]
  • DualPipe 训练框架：2048×H800 生产验证 [ref:2412.19437#key-findings]
  • MuonClip/Hybrid Muon 优化器：两条独立路线在 >30T token 训练中零 loss spike [ref:2507.20534#summary] [ref:deepseek-v4#§3-架构与方法]

Research-frontier 组件：

  • CSA+HCA 1M context（V4）：学术 benchmark 超越 Gemini-3.1-Pro [ref:deepseek-v4#§2-核心三问]，但推理栈定制需求限制了社区采用
  • 长程 agentic RL（K2.6）：展示了 4000+ step 连续执行 [ref:kimi-k2-6#key-findings]，但 RL recipe 不透明、可复现性低
  • KDA 线性注意力（Kimi Linear）：48B 验证 decode 吞吐 +75%、KV cache −75% [ref:2510.26692#key-findings]，但未到 100B+ 规模
  • Agentic serving 框架（ThunderAgent/Halo/KVFlow）：原型验证，尚未形成统一生态

趋势方向：加速中。 从 V3（2024-12）到 V4（2026-04）的 16 个月内，开源 MoE 在 Codeforces（3206 vs GPT-5.4 3168）、Putnam-2025（满分）、SWE-Bench Pro（K2.6 58.6 全场最高）等核心 benchmark 上已超越或追平闭源 [ref:deepseek-v4#§2-核心三问] [ref:kimi-k2-6#key-findings]。但在复杂 agent（Terminal-Bench 2.0 V4 67.9 / K2.6 66.7 vs GPT-5.4 75.1）和事实性知识（SimpleQA-Verified V4 57.9 vs Gemini 75.6）上仍有 7–18 分差距 [ref:deepseek-v4#§2-核心三问]。

§9 邻接 topic #

1. attention-optimization：Kimi Linear 的 KDA、V4 的 CSA+HCA、Attention Residuals 都属于广义的 attention 优化。但本 topic 聚焦于这些技术在开源前沿 MoE 模型中的集成应用，而 attention-optimization 更关注单一技术的理论和 kernel 实现。如果 KDA 被更多非 Kimi 模型采用，可能需要分离为独立 topic。

2. agent-serving：ThunderAgent、Halo、KVFlow 等 agentic serving 框架在本 topic 中作为支撑基础设施讨论。如果 agentic serving 研究扩展到非 MoE 模型（如 dense Llama 系列），应考虑独立为 agent-serving topic。当前两者高度耦合——K2.6 的长程 RL 直接驱动了 ThunderAgent 类系统的设计需求。

3. PD-disagg：PrfaaS 和 DualPath 在本 topic 中因直接服务于 MoE 模型（DeepSeek-V3/V4、Kimi Linear）而被纳入。PD 分离作为通用推理架构趋势，应有独立 topic 覆盖非 MoE 场景（如 dense model 的 PD 分离）。

4. multimodal-omni：K2.5 引入了 MoonViT-3D 多模态对接 [ref:2602.02276#key-findings]，但 MoE 多模态化在本 topic 中仅作为 K2 系列演进的一环。如果 V4 后续版本或 K3 引入原生多模态 MoE，可能需要与 multimodal-omni topic 合并相关章节。
