LLM-SLM 协作 Agent 系统：强弱模型组合的研究全景

1. 主题概述

2025–2026 年，"强弱模型协作"已从理论概念进入了系统化研究和产品化阶段。核心范式是：在 agent 系统中，client 侧部署一个小语言模型（SLM，Small Language Model，通常 1B–8B 参数），remote 侧部署一个大语言模型（LLM，Large Language Model，通常 70B+ 参数）及另一个 agent，两个 agent 通过协作协议共同完成任务。这一方向的核心动机是节省 token 成本和提升端到端效果——精心设计的大小模型协作系统可以在保持 95%+ 效果的同时大幅降低成本（5–30x）[1][2]。

该方向已有 3 篇专门的 survey[3][4][5]、十余篇顶会/顶刊论文、和多个工业产品在落地。学术界从 token 级（speculative decoding）、reasoning step 级（speculative thinking/reasoning）、task 级（MINIONS/SALE）三个粒度探索了协作机制；工业界则在 Cursor、Claude Code、Apple Intelligence、Google Speculative Cascades 等产品中实际部署了大小模型混合架构。

1.1 SLM 与 LLM 的定义

缩写全称典型参数量典型代表
SLMSmall Language Model1B–8BLlama 3.2 3B, Phi-4-Mini (3.8B), Gemma 3 4B, Qwen 2.5 7B
LLMLarge Language Model70B+GPT-4o, Claude Opus, Llama 3 70B/405B, DeepSeek-V3

在"强弱 agent 协作"语境里，SLM 是跑在 client/edge/手机/笔记本上的小模型（弱 agent 的底座），LLM 是跑在 cloud 上的大模型（强 agent 的底座）。

2. Client 端小模型 + Cloud 端大模型协作

这是与"client 侧 SLM + remote 侧 LLM + 两个 agent 协作"场景最直接相关的研究方向。

2.1 MINIONS：最接近的原型系统

MINIONS（arXiv 2502.15964，ICML 2025）[1] 来自 Stanford HazyResearch，是该方向最具代表性的工作。其核心设定是：一个小的 on-device 语言模型（如 Llama 3.2 3B/8B）可以访问本地数据，与一个 frontier 的 cloud-hosted 大模型（如 GPT-4o/Claude）协作完成涉及金融、医疗、科学推理的长文档任务。

两种协议对比：

  • Minion 协议（朴素版）：本地小模型和远端大模型简单地来回对话。因为只有本地模型读取完整 context，远端成本降低 30.4x，但效果仅恢复远端 baseline 的 87%（下降 9.4 个百分点）。根本原因是小模型在两方面能力不足：(1) 无法遵循远端模型的多步指令；(2) 长 context 推理能力差。
  • MinionS 协议（改进版）：远端大模型将任务分解为更简单的子任务，每个子任务作用于文档的较短 chunk，由本地小模型并行执行。MinionS 的成本降低 5.7x，效果恢复 97.9%。

关键发现：

  1. 3B 以下模型不够用。MINIONS 论文明确指出，他们的协作协议在 2024 年中（Llama 3.1 8B 和 GPT-4-turbo 发布）之前是不可行的，因为更早期的 SLM 连"按指令在文档片段上执行子任务"这种基础能力都做不好。至少需要 8B 级别的本地模型。
  2. 任务分解是关键。朴素对话效果差，必须让大模型将复杂任务拆成小模型能处理的粒度。
  3. 推理时计算的 scaling。重复采样、更细粒度的分解、context chunking 可以提升质量，但代价是更长的本地运行时间和更多的远端使用。

    4. 工业化：MINIONS 已由 Together.ai 推进工业化落地 [6]，提供了 WebGPU 浏览器端推理 + 云端协作的完整实现，甚至支持端到端加密通信（attestation verification + perfect forward secrecy）。开源实现在 https://github.com/HazyResearch/minions [7]。

    2.2 CLEAR：成本感知的边缘-云协作

    CLEAR（2025）[8] 提出了一种成本感知的 edge-cloud 协作推理框架。其核心思路是：边缘侧 SLM 先生成输出，一个基于强化学习训练的 cost-aware router 动态评估 SLM 输出质量，只在质量不达标时才将输出路由到云端 LLM 做精炼。框架还包含两个关键创新：(1) KV cache 管理系统，防止缓存驱逐并最小化冗余计算，通过限制并发云端请求和优化检索效率实现；(2) RL 训练的路由器，做出自适应路由决策以最小化云端使用同时保持输出质量。实验结果显示 CLEAR 在保持相近性能的同时实现了 46% 的成本降低，或在相近成本下实现了 15% 的性能提升。

    2.3 PicoSpec：流水线化的边缘-云推测解码

    PicoSpec（arXiv 2603.19133）[9] 是一个通用的、无需训练的推测解码框架，用于 LLM 边缘-云协作推理。其核心设计是一个异步流水线，将边缘侧 SLM 的 drafting 和云端 LLM 的 verification 解耦——边缘设备可以持续推测未来 token，无需等待云端反馈。为降低词汇表分布传输带来的通信延迟，PicoSpec 引入了分离拒绝采样 + 稀疏压缩，只需一次性传输压缩后的词汇表即可完成拒绝采样。在 NVIDIA Jetson AGX 上的实验显示，PicoSpec 相比云端纯自回归模型实现了最高 2.9x 加速，且对网络延迟具有显著的"延迟免疫"特性。

    2.4 Splitwise：自适应的边缘-云模型切分

    Splitwise（UCC 2025）[10] 提出了基于 Lyapunov 辅助深度强化学习（DRL）的框架，用于 LLM 在边缘和云环境之间的细粒度自适应分区。与传统层级切分不同，Splitwise 将 transformer 层分解为 attention heads 和 feed-forward sub-blocks，暴露出指数级更多的分区选择。层次化 DRL 策略在 Lyapunov 优化引导下，联合最小化延迟、能耗和精度退化，同时在随机工作负载和可变网络带宽下保证队列稳定性。在 Jetson Orin NX、Galaxy S23 和 Raspberry Pi 5 上的实验显示，Splitwise 相比现有方案降低延迟 1.4x–2.8x，能耗降低 41%，95 百分位延迟相比纯云端执行降低 53–61%。

    3. 强弱 Agent 协作（Multi-Agent 视角）

    3.1 小 Agent 能否打败大单体模型？

    "Can Small Agents Collaborate to Beat a Single Large LLM?"（arXiv 2601.11327）[11] 直接回答了这个问题。该研究引入了一个最小化设计的多 agent 系统，在 GAIA、GPQA、AIME、MuSiQue、HLE 五个 benchmark 上系统比较了小模型协作系统与大单体模型的性能。

    核心发现：

    1. 8B 多 agent 系统可以匹敌甚至超过 32B 单 agent。在 GAIA 上二者并列（23.0 vs 23.0），在 AIME 上 8B MAS 大幅领先（55.0 vs 45.0），仅在 GPQA 上略逊（58.6 vs 60.1）。
    2. 性能主要由 orchestrator 能力决定，sub-agent 大小影响较小。系统是 planner-limited 而非 executor-limited——这意味着协作架构应该让大模型做规划和验证，小模型做执行。
    3. 显式 thinking 应用于 orchestrator 时收益最大，应用于 sub-agent 时收益有限甚至为负。

      4. 3.2 SALE：通过策略拍卖扩展小 Agent

      SALE（Strategy Auctions for Workload Efficiency）（arXiv 2602.02751）[12] 受自由职业者市场启发，提出了一个 agent 框架。Agent 用简短的策略计划竞标任务，由系统化的成本-价值机制评分并通过共享拍卖记忆持续改进，实现按任务路由和不需要训练单独 router 的持续自我改进。

      核心发现：

      1. 小 agent 的性能随任务复杂度增加而不能 scale——在深度搜索和编码任务上，小 agent 单独使用效果显著不足。
      2. SALE 减少了对最大 agent 的依赖 53%，整体成本降低 35%，并持续改善最大 agent 的 pass@1——仅有可忽略的额外开销。
      3. 现有的基于任务描述的 router 要么性能不如最大 agent，要么无法降低成本——往往两者都失败，说明它们不适合 agentic 工作流。

        4. 3.3 BIGMAS：脑启发的图结构多 Agent 系统

        BIGMAS（Brain-Inspired Graph Multi-Agent Systems）（arXiv 2603.15371）[13] 提出了一种新框架，其中 GraphDesigner 根据问题动态构建 task-specific 的 agent 拓扑，agent 之间通过集中式共享工作区（centralized shared workspace）协调，而非点对点通信。

        关键发现：多 agent 架构增益与模型推理能力正交——即使底层模型本身很强，好的协作架构仍然有额外收益。BIGMAS 特别擅长弥补大推理模型在顺序多步任务上的规划失败。

        3.4 Graph-of-Agents (GoA)：可扩展的图结构协作

        Graph-of-Agents (GoA)（arXiv 2604.17148）[14] 提出了一种基于图的测试时协作框架。与 Mixture-of-Agents（MoA）将所有 agent 整合并聚合输出不同，GoA 只选择相关的 agent 子集作为图节点，通过有向消息传递实现层内通信——从更相关的 agent 向较不相关的 agent 流动。这种图结构实现了更好的可扩展性和更高效的 agent 间通信，可以扩展到 10 或 100 个 agent 的大 agent 池。

        3.5 四种多 Agent 方案的对比

        4. Reasoning 层面的大小模型协作

        这一方向的核心思路是：在推理过程中，小模型生成大部分内容，大模型在关键节点介入——验证、修正、或接管。粒度介于 token 级的 speculative decoding 和 task 级的 MINIONS 之间。

        4.1 Speculative Thinking：推理级的推测执行

        Speculative Thinking（arXiv 2504.12329）[15] 是一个无需训练的框架，让大推理模型在推理级别指导小模型。小模型生成大部分输出，但在特定结构性线索（如段落分隔符 \n\n 后跟 "wait"、"alternatively" 等反思性短语）处选择性地将控制权交给大模型。这些位置通常标记着内部修正，小模型在这些位置经常产生冗长输出，而大模型则更简洁、更有效地进行回溯。

        效果：在 32B 推理模型的辅助下，1.5B 模型在 MATH500 上的准确率从 83.2% 提升到 89.4%（+6.2%），同时平均输出长度从 5439 token 减少到 4583 token（减少 15.7%）。应用于非推理模型（Qwen-2.5-7B-Instruct）时，准确率从 74.0% 提升到 81.8%（+7.8%）。

        两种介入机制：(1) 反思触发：遇到 \n\n + 反思短语（"wait"、"alternatively"）时大模型接管生成 n₁ tokens；(2) 验证触发：遇到验证相关线索（"verify"、"double-check"）时大模型接管生成 n₂ tokens。

        4.2 SpecReason：推测推理

        SpecReason（2025）[16] 将较简单的中间推理步骤选择性地卸载给更小、更快的推测模型，同时保留更大但更慢的基础模型来高效验证这些推测步骤并引导推理过程走向正确轨迹。每个步骤通过两阶段解码：(1) 轻量推测器提议下一个推理步骤；(2) 基础模型评估提议步骤的语义价值。

        核心特征：高效验证——SpecReason 通过单次 prefill-only 前向传播评估每个推测步骤，而非对多个候选步骤做 reranking。与 speculative decoding 的组合可进一步减少延迟 19.4%–58.0%，精度损失仅 0.4–9.0%。SpecReason 在精度-延迟权衡的 Pareto 前沿上优于 speculative decoding。

        4.3 SpecCoT：推测性思维链

        SpecCoT（Speculative Chain-of-Thought）（EMNLP 2025 Findings）[17] 利用大小模型的互补优势。大模型提供初始指导并执行高效验证，小模型在整个过程中生成多个多样化的中间推理步骤。每一步中，小模型生成 N 个候选延续，大模型作为验证器 V 选择最有前途的延续，或在没有候选满足质量阈值时自己生成。验证极其高效——仅需单次前向传播输出一个 token 指示选择结果。

        4.4 MAI：多 Agent 推理

        MAI（Multi-Agent Inference）（arXiv 2604.04929）[18] 提出了一种多 agent 推理框架，让大模型保持短回复但将关键推理 token 从小模型转移到大模型。具体做法是：重用小模型生成的 cheap thinking tokens 作为大模型的上下文 prompt，让大模型参考这些推理过程来回答问题。由于 prefill 远比 decode 便宜，这种方式避免了大模型自己生成大量 thinking tokens。

        效果：MAI 实现了 2.4x 加速，比 speculative decoding 更实用——后者由于高 draft rejection rate（~40%）和严重的 token 级 context-switching 开销，实际比标准解码慢 4%。MAI 只需单次 8B 调用，原生兼容 vLLM 等优化引擎。

        4.5 四种推理协作方案的对比

        方案协作粒度大模型介入时机速度/效果权衡是否需要训练
        Speculative Thinking [15]推理段落级反思/验证线索准确率 ↑6.2%，长度 ↓15.7%否
        SpecReason [16]推理步骤级每步验证延迟 ↓19–58%，精度 ↓0.4–9%否

        5. 路由/级联系统（Routing & Cascade）

        这一方向的核心思路是：不是所有请求都需要大模型，通过智能路由将简单请求分配给小模型、复杂请求分配给大模型，或者先用小模型尝试、不够好再升级到大模型。

        5.1 Cascade Routing：统一路由与级联的理论最优框架

        Cascade Routing（ICML 2025）[19] 首次推导了级联策略的最优算法，证明了现有路由策略的最优性，并提出了将路由和级联整合为理论最优策略的 cascade routing 框架。通过分析，作者识别出好的质量估计器是模型选择范式成功的关键因素。实验显示 cascade routing 以大幅度持续优于单独的路由或级联方法。

        5.2 CASCADIA：部署与路由联合优化

        CASCADIA（arXiv 2506.04203）[20] 是一个专门为 LLM 设计的级联服务框架，使用 MILP 求解器优化部署（跨小/中/大模型的资源分配和并行策略），使用 Chebyshev 引导的方法优化路由（平衡延迟和质量）。系统支持自适应模型级联，根据模型大小自动选择最优并行策略（tensor parallelism、pipeline parallelism）。

        5.3 SCORE：约束感知的路由系统

        SCORE（Harvard, 2025）[21] 是一个路由系统，在满足用户指定的成本和延迟约束下最大化响应质量。对每个传入请求，SCORE 预测每个模型的响应质量和长度，选择最满足当前成本/延迟/质量需求的选项。通过持续调整这些决策，SCORE 平衡系统负载、执行预算限制、并通过及时、经济、准确的响应维持用户满意度。

        5.4 Speculative Cascades：混合级联与推测解码

        Speculative Cascades（Google Research, 2025）[22] 将级联和推测解码的优势结合。传统级联通过让小模型先处理、复杂任务才交给大模型来降低成本；传统推测解码通过小模型 draft + 大模型 verify 来加速。Speculative cascades 的创新在于：为了效率，有时会故意 defer 给小模型——在小模型能力足够时不调用大模型。在 Gemma 和 T5 模型上的测试显示，speculative cascades 相比单独的级联或推测解码实现了更好的成本-质量权衡。

        6. 端侧推理性能：SLM 跑在手机/笔记本上够用吗？

        这一节回答一个实际问题：如果 SLM 部署在端侧设备上，算力是否足以支撑 agent 工作负载？

        长 context 推理（>8K tokens）很高，prefill 慢不推荐勉强可以

        6.4 趋势

        端侧推理能力在快速改善。Gemma 4 E2B 的 MTP（Multi-Token Prediction）可以在端侧实现 2.2x 加速，Google 的 LiteRT-LM 在 Android GPU 上跑 E2B 已经 52 tok/s [24]。随着 SLM 能力持续提升（MINIONS 论文指出其协议在 2024 年中之前不可行 [1]），这个方向的价值会越来越大。

        7. 具体架构设计：LLM 出 Tool Call + SLM 组装结果

        在讨论的深入过程中，我们探索了一种具体的架构设计：LLM 负责生成 tool call 和决定如何调用工具（规划），SLM 负责将工具返回的结果组装拼凑成最终答案（合成）。

        7.1 架构描述

   ▼

        7.2 合理性分析

        Tool call 生成是 agent 里最难的部分，已有研究明确支持这一点：

        1. ToolOrchestra（2025，8B model 超越 GPT-5）[28] 的核心发现：tool call 的规划和编排（选哪个工具、传什么参数、什么顺序）需要强推理能力，但"40% 的 query 其实根本不需要调大模型"——一个 8B orchestrator 学会了什么时候该调大模型、什么时候不用。在 HLE 上，Orchestrator 得分 37.1%，超过 GPT-5 的 35.1%，同时效率提高 2.5x。
          1. "Can Small Agents Collaborate" [11] 的核心发现：agent 系统是 planner-limited 而不是 executor-limited。规划能力决定上限，执行者可以很小。
            1. 结果组装本质上是摘要/格式化，这恰好是 SLM 已经做得不错的任务。3–4B 模型做摘要和信息提取的能力已经足够（MINIONS 论文验证过 [1]）。

              2. 7.3 Token 经济学

              这一架构在 token 成本上特别有优势：

              传统全 LLM 架构：

              • LLM input: system prompt + user query + tool schemas ≈ 2K tokens
              • LLM output: tool call ≈ 200 tokens
              • [Tool 执行]
              • LLM input: 上面所有 + tool results (可能很长) ≈ 5–20K tokens ← 贵！
              • LLM output: 最终答案 ≈ 500 tokens
              • LLM 总消耗: ~8–23K tokens

              LLM + SLM 架构：

              • LLM input: system prompt + user query + tool schemas ≈ 2K tokens
              • LLM output: tool call ≈ 200 tokens
              • [Tool 执行]
              • SLM input: 简单指令 + tool results ≈ 5–20K tokens ← 免费/极便宜
              • SLM output: 最终答案 ≈ 500 tokens
              • LLM 总消耗: ~2.2K tokens（省了 75–90%）

              tool results 通常是最长的部分（搜索返回几十条结果、数据库查询返回大表、API 返回完整 JSON），让 SLM 消化这些长结果比让 LLM 消化便宜得多。

              7.4 关键挑战：闭环判断

              纯单向流（LLM→Tool→SLM→用户）有一个致命弱点：谁来判断工具结果够不够用？

              常见需要闭环的场景：

              • 搜索结果没有找到答案 → 需要换个 query 再搜
              • 第一个 API 的结果需要作为第二个 API 的输入
              • 工具返回了错误 → 需要修正参数重试

              SLM 在这些情况下很难做出正确判断，因为它不具备规划能力。三种可行的闭环方案：

              方案 A：SLM 判断 + 回调 LLM（混合闭环）——SLM 尝试判断结果是否充分，不够或不确定时回调 LLM 做下一轮规划。这与 CLEAR [8] 的做法一致：SLM 先尝试，quality router 判断不够才上云端。

              方案 B：LLM 一次性预规划全部工具调用链——LLM 一次生成多个并行或有依赖的 tool calls，全部执行后由 SLM 统一组装结果。这与 ParaManager (Agent-as-Tool) [29] 的做法一致：一次规划多个并行子任务，减少多轮交互。

              方案 C：LLM 同时输出规划 + 结果判断标准（差异化创新空间）——LLM 不只生成 tool call，还为 SLM 输出一个轻量级的结果验证/组装指令，例如成功标准（"结果中必须包含价格信息"）和失败时的备选方案。这在现有工作中是一个明确的空白，有明确的差异化创新空间。

              7.5 最接近的已有工作对比

              工作大模型做什么小模型做什么和此架构的差异
              ToolOrchestra [28]被当作"智能工具"按需调用做规划+编排（反过来了！）小模型做 orchestrator，大模型是 tool
              HTAA [30]做高层规划封装成 "agent tool" 处理子任务小模型包含了子任务执行和结果返回
              MINIONS [1]做任务分解处理长文档 chunk、返回结果最接近，但 SLM 处理的是文档不是 tool results
              MemEx [31]agent 规划sub-agent 异步执行 + scratchpad 组装用 Python scope 做结果组装而非 SLM
              AgentOpt [32]自动搜索最优模型组合每个 step 可以用不同大小的模型工具而非架构

              8. CPU 协同部署：GPU 服务器上利用空闲 CPU 跑 SLM

              在讨论中进一步探索了一个系统级优化：在云端 GPU 服务器上，CPU 通常利用率很低（10–25%），是否可以在空闲 CPU 上部署 8B SLM。

              8.1 GPU 服务器的资源使用画像

              典型 LLM serving 节点中，GPU 利用率 80–95%（核心瓶颈），但 CPU 仅 10–25%、RAM 仅 30–50%。LLM serving 的 CPU 主要做 tokenizer、采样、请求调度、KV cache 管理，这些都很轻量，大部分 CPU 核心确实是空着的。

              已有研究证实了这一观察：

              • OmniServe（arXiv 2603.12831）[33] 明确指出"LLM serving 集群中 CPU 资源严重未充分利用"，并利用空闲 CPU 做 best-effort 请求的 Attention 计算，SLO 达成率提升 1.48x，best-effort 吞吐提升 9.85x。
              • CaraServe [34] 利用空闲 CPU 在 GPU 加载 LoRA adapter 期间并行执行推理，隐藏冷启动延迟。
              • HyGen [35] 在 GPU serving 实例上 co-locate online/offline 工作负载，利用低流量期间收割空闲资源。

              8.2 CPU 跑 8B 的性能实测

              服务器 CPU核心数8B Q4 速度来源
              Xeon Platinum 8480+ (Sapphire Rapids) [36]56C/112T40 tok/s当前主流 GPU 服务器 CPU
              EPYC 9334 (Zen 4) [37]32C/64T x228 tok/s双路常见
              EPYC 9654 (Zen 4) [38]96C32 tok/s高端配置
              Xeon Gold 6312U [39]24C21 tok/s中端配置

              40 tok/s 的速度完全够做结果组装和摘要。8B Q4 模型仅需约 5 GB RAM，对 512GB+ 的 GPU 服务器来说微不足道。

              8.3 三个潜在陷阱

              陷阱 1：内存带宽竞争（最关键）

              CPU 推理的瓶颈不是算力，是内存带宽。8B Q4 模型 decode 阶段需要约 30–50 GB/s 的持续内存带宽。GPU serving 也依赖 host memory（KV cache page 管理、tokenizer batch processing、tensor parallelism 协调）。在 GPU serving 高负载时（大 batch prefill），SLM 推理可能慢 30–50%。

              对策：限制 SLM 使用的核心数（比如只用 8–16 个核心），并用 numactl 绑定到离 GPU 远的 NUMA node 上，避免带宽竞争。

              陷阱 2：NUMA 拓扑

              双路服务器（常见于 GPU 节点）有 NUMA 问题。GPU serving 通常绑定在 GPU 所在的 NUMA node 上，SLM 应该跑在另一个 NUMA node 的空闲核心上，用本地内存，两者互不干扰。

              
numactl --cpunodebind=1 --membind=1 \
  llama-server -m llama-8b-q4.gguf --threads 16 --port 8081

              陷阱 3：延迟考量

              在"LLM 出 tool call → SLM 组装结果"架构里，SLM 处理 2K token 的 tool results、生成 500 token 答案约需 ~17 秒（@ 30 tok/s），而云端 LLM 在 H100 上约 ~6 秒（@ 80 tok/s）。对于后台处理可以接受，对于实时对话可能需要优化（用更小的 3B 模型加速到 ~8 秒，或减少输出长度）。

              8.4 推荐的部署方案

              
┌──────────────────────────────────────────────────────┐
│                 GPU 服务器                              │
│                                                        │
│  NUMA Node 0                   NUMA Node 1             │
│  ┌────────────────────┐       ┌────────────────────┐   │
│  │ GPU Serving (vLLM) │       │ SLM Service        │   │
│  │ 大模型 (70B+)       │       │ 8B Q4 on CPU       │   │
│  │ CPU cores 0-31     │       │ CPU cores 32-47    │   │
│  │ GPU 0-3            │       │ 无 GPU              │   │
│  │ 本地 RAM            │       │ 本地 RAM (~8GB)     │   │

│          ├─────────────────────────────►│               │

              本质上是"free lunch"——利用已部署 GPU 服务器上闲置的 CPU 和 RAM 来跑 SLM，零额外硬件成本。大模型出 tool call 是 GPU-bound 的短任务，SLM 组装结果是 CPU-bound 的长任务，两者天然错开资源使用，形成很好的互补。

              9. 跨模型 Context 传递：Model Routing 的最后拼图

              在 model routing / cascade 系统中，切换模型意味着丢失前序模型积累的隐式理解。Cursor 的 Auto 模式就暴露了这个问题——中途切模型后，新模型"感觉变笨了"[56]。这不是新模型差，而是它缺少前序模型积累的三层状态：(1) 显式文本（chat history，可重放但昂贵）；(2) 推理中形成的隐式理解（如用户偏好、代码库结构，文本里没写但模型推断出了）；(3) KV cache 中的激活状态（每个模型内部表示空间完全不同，不可直接迁移）。

              9.1 Cursor Auto Routing 的教训

              Cursor 的 Auto 模式根据任务类型、容量状况、上下文复杂度在 turn 间切换模型 [56]。社区反馈显示切模型导致严重的 context 丢失："你可以用 Claude 3.7 编码，突然它把你切到 GPT O-1 mini，新模型完全不知道你的项目和之前做的事情"。

              更成熟的 Phase-Aware Routing [57] 根据编码阶段路由——plan → Opus/GPT-5（最贵但规划需要最强推理），implement → Sonnet/DeepSeek（性价比最优），test → DeepSeek/Kimi（便宜够用），document → Haiku/Flash（最便宜）——可带来 70–90% 的成本降低。但同样面临跨模型 context 丢失问题。

              9.2 解决方案层次

              层次 1：结构化文本记忆（最实用）

              Portable Agent Memory（arXiv 2605.11032）[47] 将 agent 积累的 context 序列化为五类结构化记忆——Episodic（事件历史）、Semantic（事实知识）、Procedural（操作流程）、Working（当前任务状态）、Identity（用户偏好）——用 Merkle-DAG 做防篡改校验，通过注入抗性 re-hydration 流水线安全传输。在 Claude → GPT-4 传输实验中，Transfer Continuity Score 从 no-memory baseline 的 0.28–0.45 提升到 0.83–0.92。

              CLO (Cross-LLM Orchestrator) [55] 是一个浏览器插件实现，实时捕获 reasoning state 为 canonical JSON schema，一键注入到新模型。

              层次 2：KV Cache 跨模型对齐（研究前沿）

              Latent Space Communication via K-V Cache Alignment（arXiv 2601.06123）[48] 学习一个全局共享 KV cache 表示空间，每个模型配两个 adapter（out-adapter 翻译到共享空间，in-adapter 翻译回来）。类似机器翻译中的"中间语言"概念。在 Gemma-2 系列上验证可行性，不修改预训练参数，adapter 参数开销随模型数线性增长。

              Cache-to-Cache (C2C)（arXiv 2510.03215）[49] 用 neural cache fuser 把 Sharer 模型的 KV cache 投影融合到 Receiver 模型空间。比 Text-to-Text 通信精度高 3.1–5.4%，延迟 2.5x 加速（跳过 decode→encode 往返）。

              KVCOMM（NeurIPS 2025, arXiv 2510.12872）[50] 无需训练的跨 context KV cache 复用。通过 anchor pool 存储已见样本的 KV 偏移量，推理时找最近 anchor 做插值来估计新 context 下的 KV cache。5-agent 场景下 7.8x 加速（TTFT 从 430ms 降到 55ms），70%+ 复用率，完全 training-free。

              QKVShare（arXiv 2605.03884）[54] 将 KV state 打包为量化 CacheCard，通过拓扑感知控制器分配 bits，在 agent 间传递并注入接收方模型。

              层次 3：模型自管理 Context（最前沿）

              StateLM（arXiv 2602.12108）[51] 让模型自己管理自己的 context——可以调用 deleteContext、readChunk、updateNote 等动作，主动裁剪无用历史并将关键信息写入持久笔记本。模型在切换前可以主动提炼知识到外部存储，切换后新模型从精炼笔记本恢复。

              对话历史压缩

              MT-OSC（arXiv 2604.08782）[52] 在后台顺序压缩 chat history，无需微调，针对大 token 节省同时保持准确性。C-DIC（ICLR 2026 提交）[53] 将对话视为交织的 contextual threads，维护可修订的 per-thread 压缩状态，通过 retrieve → revise → write-back 循环实现跨 turn 信息共享和纠错。

              9.3 与 LLM-SLM 协作架构的关系

              在本 topic 讨论的 "LLM 出 tool call + SLM 组装结果" 架构中，LLM→SLM 的切换天然避开了 context 丢失问题——SLM 不需要继承 LLM 的完整推理状态，只需接收结构化的 tool call 指令和原始 tool results。这比 Cursor 式的"同一角色换模型"干净得多。

              然而，跨模型 context 传递是 model routing/cascade 系统（第 5 节）落地的最后一块拼图。目前所有的 routing 系统都假设"切模型 = 丢 context"，如果能解决这个问题，routing 的价值会成倍放大——可以在每个 turn 甚至每个 sub-task 上自由选择最合适的模型，而不用担心状态丢失。

              9.4 跨模型 Context 传递方案对比

              方案层次是否需要训练传输保真度延迟开销适用范围

              12. 核心设计原则

              从本次调研覆盖的所有工作中提炼出的设计原则：

              1. Orchestrator > Executor。性能主要由编排者（大模型）的能力决定，执行者（小模型）的大小影响相对较小 [11][28]。架构应让大模型做规划和验证，小模型做执行。
                1. 任务分解是关键。朴素的大小模型对话效果不好（MINIONS 的 Minion 87% vs MinionS 97.9% [1]），必须做好任务分解。大模型把复杂任务拆成小模型能处理的粒度。
                  1. 小模型的下限很重要。3B 以下基本不可用，8B 是当前的 sweet spot [1]。随着 SLM 能力持续提升，这个方向的价值会越来越大。
                    1. 协作粒度可以分层：
                    2. Token 级：speculative decoding（最细，延迟优化）
                    3. Reasoning step 级：speculative thinking/reasoning [15][16][17]（中等，推理质量+速度）
                    4. Task 级：MINIONS/SALE [1][12]（最粗，成本优化）
                      1. 架构增益与模型能力正交。即使底层模型很强，好的协作架构仍然有额外收益 [13]。
                        1. GPU 服务器的闲置 CPU 是 free lunch。在已有 GPU serving 基础设施上零成本部署 SLM，大模型 GPU-bound 短任务与 SLM CPU-bound 长任务天然互补 [33][34]。
