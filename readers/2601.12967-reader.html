🗺 研究背景和现状

A · 时代定位

2025-2026年，LLM推理优化的"低垂果实"（continuous batching、PagedAttention、prefix caching、chunked prefill）已被充分采摘。但这些优化将每次LLM调用视为独立请求。当LLM进入agentic模式——多轮迭代LLM+工具调用——传统指标（TTFT, TPOT）不再反映用户体验。本文标志着serving优化从"单请求优化"进入"多迭代请求级优化"的新阶段。

B · 领域定位

方向：LLM推理serving系统 / Agentic工作负载优化
子任务：降低tool-based agentic推理的用户感知延迟（FTR）
实际问题：企业级AI助手（如Microsoft Copilot）在回答需要搜索、代码执行等工具调用的复杂问题时，用户等待时间可达十几秒

C · 核心痛点

现有agentic系统能正确完成多轮工具调用，但无法高效执行——三个层面的痛点：

  • ①性能：工具执行占FTR延迟30-80%（p99达83%），且延迟变异系数>100%
  • ②效率：50-80%的prompt内容在工具结果到达前即可用，但引擎闲等；decode完成后才开始解析工具调用
  • ③资源：并发agentic请求下，LRU驱逐策略导致KV cache级联失效，破坏本应有的上下文复用
根本原因：编排器（orchestrator）和LLM引擎作为解耦黑盒运行，通过不透明的request-response接口通信，阻止了跨层优化。

D · 本文目标

核心研究问题：能否通过编排器-引擎协同设计（co-design），在不修改模型架构和推理kernel的前提下，系统性降低agentic推理的FTR延迟？

这个问题之前未被解决，因为：(1) 现有框架（LangChain, AutoGen）将引擎视为黑盒；(2) 引擎（vLLM, SGLang）不感知请求的agentic语义；(3) 工具延迟的高变异性使预测型优化不实际。

D' · 设计空间约束

替代方案为何不可行
静态DAG编排（Parrot, Murakkab）Tool-based agent的执行路径运行时动态决定，无法预先构建DAG
工具延迟预测（Continuum）生产工具延迟CV > 100%（Figure 4b），预测精度无法指导调度
部分工具执行（Conveyor）生产工具需要完整参数才能执行，不支持partial execution
纯调度优化（Autellix）仅优化吞吐量，忽略工具延迟对FTR的贡献
KV offload（LMCache, Mooncake）解决存储层级但不解决编排瓶颈——可作为正交补充

结论：排除以上方案后，剩余可行设计空间是——在编排器和引擎之间建立轻量语义协调接口，让双方共享信息实现跨层优化。

E · 提出框架

SUTRADHARA——通过5个新API实现orchestrator-engine co-design。

核心思想：通过prompt splitting实现prefill与工具执行的并行，通过streaming dispatch在decode期间增量派发工具，通过语义标签指导KV cache优先级驱逐。

本质区别：现有方法将编排器和引擎视为独立黑盒；Sutradhara通过最小化API接口实现信息共享，用编排器的语义知识指导引擎的资源管理决策。

F · 技术挑战

🏗 核心架构图解

"Why not static DAG?" — 为什么不能预编译执行图？

Parrot和Murakkab通过将agentic工作流表示为静态DAG来优化。但tool-based agent的执行路径由LLM运行时动态决定——每轮调用哪些工具、参数是什么，都取决于上一轮的decode输出。中位迭代深度2次（尾部7次），中位fan-out 2（尾部20），这种高度动态性使静态DAG不可行。

"Why co-design?" — 为什么需要打破抽象边界？

编排器知道prompt结构（50-80%与工具无关）、工具调用格式（JSON数组）和迭代语义（哪些KV将被复用）。引擎知道GPU资源、KV cache状态和调度队列。双方信息互补但被黑盒接口隔离。5个API的最小化设计恰好桥接了这个信息鸿沟——不需要大规模重构，只需要精确的hint传递。

核心技术壁垒: Continuation Handle + KV Pinning

Prompt splitting的关键难点不在于"拆分"，而在于engine必须为partial prefill的KV块维护continuation handle——这些块被pinned（最高优先级），在等待工具结果期间不被驱逐。当extend_prefill()到达时，engine需要在正确位置拼接新内容到pinned KV cache上。这要求vLLM的PagedAttention KV manager新增"部分完成请求"状态——现有引擎设计中没有这个概念。

假设质疑: 工具调用格式与prompt分割

Sutradhara假设：(1) prompt可清晰分割为工具无关/依赖部分；(2) 工具调用以结构化JSON格式生成。如果编排器使用自由格式prompt（工具结果散布在各处），或模型使用自然语言描述工具调用，这两项假设可能失效。不确定的是：随着Agent协议（如MCP）标准化程度提高，这些假设的普适性可能增强还是减弱。

🔬 第一性原理

为什么不能预测工具延迟来优化调度？

反方观点：如果能预测工具延迟，就能更好地安排prefill和decode的时间

工具延迟是外部服务响应时间的函数，受网络条件、服务负载、查询复杂度影响。Figure 4b显示7种生产工具的变异系数 > 100%——同一工具在不同请求上的延迟差异可达10倍以上。即使使用复杂的预测模型（如Continuum），预测误差也太大以至于无法用于实时调度决策。

核心原理：外部工具延迟的不可预测性是agentic推理的本质特征，而非可以消除的缺陷。正确的策略不是预测，而是通过并行化来容忍（tolerate）延迟。

为什么LRU驱逐在agentic场景下必然失效？

反方观点：LRU在传统推理中工作良好，为什么agentic场景不行？

关键在于agentic请求的迭代间隙：iteration 1完成后，KV cache不会立即被iteration 2复用——中间有工具执行的等待期（数百毫秒到数秒）。在这段时间内，其他请求的LLM调用会进入engine，产生新的KV块，按LRU策略驱逐"最久未访问"的块——而这恰恰是即将被iteration 2复用的块。Figure 6展示了3个并发请求如何产生级联驱逐。

核心原理：LRU假设"最近访问的数据最可能被再次访问"——但agentic请求的时间结构打破了这个假设：最需要保留的KV块恰恰处于"等待工具"的非活跃状态。

为什么prompt splitting不会破坏attention的正确性？

疑问：将prompt拆分为两部分分别prefill，注意力计算还正确吗？

Prompt splitting的数学基础建立在causal attention的可分解性上。对于iteration $i+1$的prompt $P = [P_a \| P_b]$（$P_a$是工具无关前缀，$P_b$是工具输出后缀）：先对$P_a$执行prefill得到KV cache $\{K_a, V_a\}$，然后对$P_b$执行extend prefill时，attention计算中$P_b$的token可以attend到$P_a$的所有KV——因为它们的KV cache已经被计算并pinned。最终结果与一次性计算整个$P$完全等价。

核心原理：Causal attention的增量计算特性——先计算前缀的KV cache，再追加后缀时只需对新增token做attention，前缀的KV已被缓存。这与prefix caching的数学基础相同。

为什么streaming dispatch有效而partial execution无效？

对比：Conveyor也试图重叠decode和工具执行，为什么方法不同？

Conveyor尝试部分工具执行——在参数不完整时就开始执行工具。但生产中大部分工具（如API调用、代码执行）需要完整参数才能启动。Sutradhara的streaming dispatch解决的是不同问题：当LLM生成多个工具调用（JSON数组中的多个对象）时，第一个工具调用对象的`}`闭合后就可以立即派发，无需等待后续工具调用的生成。这利用了tool call fan-out（中位2，尾部20）的特征——通过提前派发已完成的工具调用来争取时间。

核心原理：区分"单个工具的参数不完整"（不可提前执行）和"多个工具中某个已完整"（可立即派发）——前者是Conveyor的思路，后者是Sutradhara的思路，且在生产中更实际。

5级驱逐优先级的设计逻辑

设计选择：为什么是RESPONSE→TOOL_OUTPUT→USER_QUERY→SYSTEM_PROMPT→PARTIAL_PREFILL？

优先级排序基于复用概率的递增：RESPONSE的token是用户最终输出，生成后不再被任何请求复用；TOOL_OUTPUT的复用取决于后续迭代是否共享系统prompt（如果系统prompt改变，前一轮的tool output KV就无法被复用）；USER_QUERY在同一请求的所有迭代间共享；SYSTEM_PROMPT可跨请求共享（同类工具调用的请求）；PARTIAL_PREFILL是正在被使用的活跃资源，驱逐会直接导致工作丢失。

核心原理：KV块的驱逐优先级应与其未来复用概率成反比——驱逐"已完成使命"的块，保留"即将被需要"的块。

假设挑战：单GPU serving的局限性

质疑：在P-D分离（prefill-decode disaggregation）架构下，Sutradhara还能工作吗？

Sutradhara的prompt splitting假设prefill和decode在同一GPU上执行——partial prefill的KV cache需要被pinned在原地等待extend。在P-D分离架构中，prefill节点和decode节点是不同的GPU集群，KV cache需要通过网络传输。此时prompt splitting面临新挑战：(1) partial prefill的KV需要被传输到decode节点还是保留在prefill节点？(2) extend_prefill应该在哪个节点执行？这些问题当前论文未解决。

这个假设在当前"P-D colocated"架构下完全成立；但随着P-D分离成为主流（如Mooncake、DistServe），Sutradhara需要重新设计prompt splitting的数据流——可能需要在prefill集群中实现"KV transfer with pinning guarantee"。

Sutradhara开创了orchestrator-engine co-design的研究方向。其核心贡献在于问题定义（将agentic推理的FTR延迟分解为三个可优化维度）和最小化API设计（5个API桥接两层信息鸿沟）。未来工作可从以下方向深化。

方向1: P-D分离架构下的Agentic Serving

方向2: 自适应Prompt分割策略

B · 实践部署上下文

Sutradhara适用于agentic推理的prefill阶段优化（通过并行化减少迭代间等待）和decode阶段优化（通过streaming dispatch减少工具派发延迟）。在A100 GPU上，中间迭代为prefill-bound（20K token prompt, ~100 decode tokens），最终迭代为decode-bound（~500 decode tokens）。prompt splitting主要加速prefill-bound的中间迭代，streaming dispatch加速decode→tool转换。

集成点：需要修改vLLM的v1 scheduler（添加优先级驱逐 + continuation handle）和构建自定义orchestrator。对于使用LangChain/AutoGen的现有部署，需要在orchestrator层添加prompt分析模块。

C · 生态影响追踪

论文来自Microsoft Research，基于内部agentic平台（类似Copilot的架构）。尚未公开开源，但其5-API设计理念有可能被vLLM/SGLang社区采纳。与Autellix（Berkeley）、Conveyor（Columbia）、Continuum（Berkeley/Stanford）形成竞争学术空间。如果Microsoft在Copilot中部署Sutradhara并开源API规范，有潜力成为agentic serving的事实标准接口。

资深教授判断：Agentic推理正从"能用"走向"好用"，serving系统必须从"单请求优化"进化为"多迭代请求级优化"。Sutradhara定义了orchestrator-engine co-design的范式——这是未来5年agentic serving的主线之一。真正的挑战在于：当agent变得更复杂（多agent协作、动态规划），co-design的API如何保持最小化而不膨胀？

📝 Worksheet 答题卡

1 FTR（First Token Rendered）与传统TTFT有什么本质区别？为什么在agentic场景下需要新指标？

核心论点：TTFT衡量单次LLM调用从prefill到第一个decode token的时间，而FTR衡量整个agentic迭代链（多轮LLM + 工具调用）到最终回复第一个token的总时间。在agentic场景下，中间迭代的所有TTFT加上工具延迟都被"隐藏"在FTR中，TTFT无法反映用户体验。

证据链：§1定义FTR → §3.3形式化 → §3.5 Finding 1证明工具延迟占FTR 30-80%

2 Prompt Splitting如何在不破坏attention正确性的前提下实现prefill与工具执行的并行？

核心论点：基于causal attention的增量计算特性——先对工具无关前缀$P_a$做prefill得到$\{K_a, V_a\}$，工具结果到达后对后缀$P_b$做extend prefill时，$P_b$的token可以attend到已缓存的$K_a, V_a$。数学上等价于一次性计算整个$P=[P_a \| P_b]$。

证据链：§4.2 Prompt Splitting设计 → §3.5 Finding 2 (50-80%工具无关) → Table 1 API定义

3 为什么LRU驱逐策略在并发agentic请求下导致KV cache级联失效？Sutradhara如何解决？

核心论点：Agentic请求的迭代间存在工具执行等待期，此时KV cache处于"不活跃但即将被复用"的状态。LRU基于访问时间驱逐，恰好驱逐了这些最需要保留的块。Sutradhara通过5级语义标签优先级驱逐（RESPONSE→TOOL_OUTPUT→USER_QUERY→SYSTEM_PROMPT→PARTIAL_PREFILL）替代LRU。

证据链：§3.5 Finding 3 + Figure 6（级联失效） → §4.4设计 + Figure 8（解决方案）

4 Streaming Tool Dispatch与Conveyor的Partial Tool Execution有何本质区别？为什么前者更适合生产环境？

核心论点：Conveyor尝试在工具参数不完整时开始执行（partial execution），但生产中大部分工具需要完整参数。Sutradhara的streaming dispatch解决的是不同问题——当LLM生成多个工具调用时，第一个完整工具对象闭合后立即派发，利用的是tool call fan-out（中位2，尾部20）特征。

证据链：§4.3 Streaming设计 → §3.4 fan-out分布 → §6 Related Work对比

5 5个Co-design API各自启用什么优化？为什么这5个API构成"最小化"接口？

核心论点：submit_partial_prefill和extend_prefill启用prompt splitting并行执行；register_streaming_callback启用streaming tool dispatch；tag_kv_blocks和set_reuse_priority启用语义cache管理。"最小化"体现在：每个API对应一项信息传递需求，没有冗余，且不需要深层引擎内部修改。

证据链：Table 1完整API列表 → §4.1-4.4各自使用的API → §4.5实现（3500 LOC）

6 消融实验中，Prompt Splitting贡献7.57%而Streaming Dispatch贡献4.2%——为什么PS贡献更大？

核心论点：PS将50-80%的prefill计算与工具执行重叠，直接减少了最大的等待瓶颈（工具执行期间的GPU闲置）。DS的贡献取决于tool call fan-out——只有当一次迭代生成多个工具调用时，提前派发第一个才有增益。在中位fan-out=2的情况下，DS的并行化空间有限。

证据链：Table 2消融数据 → §3.5 Finding 2 (50-80%工具无关) → §3.4 fan-out分布

7 Sutradhara在什么条件下效果最好？什么条件下优势减弱？

核心论点：效果最好：高tool fan-out、工具延迟占比大、KV cache压力高、中低QPS。优势减弱：低fan-out（iteration-heavy trace仅4% FTR改善）、工具延迟极短（本地工具）、单迭代请求、高QPS（引擎饱和时所有系统受排队主导）。

证据链：§5.2 QPS变化实验 → §5.2 Iteration-heavy对比 → Figure 11 Gemma结果

8 如果将Sutradhara的思想应用到P-D分离架构（prefill与decode在不同GPU集群），需要解决哪些新挑战？

核心论点：P-D分离下，partial prefill的KV cache在prefill节点生成，但decode需要在decode节点执行。新挑战：(1) partial KV需要传输到decode节点还是保留在prefill节点？(2) extend_prefill在哪个节点执行？(3) KV传输的pinning guarantee如何实现？(4) continuation handle需要跨节点可见。

证据链：§4.2假设单GPU → §4.5基于vLLM colocated PD → §7未来工作暗示
