L4 / category / framework

现状快照

2024–2026 年的 LLM 基础设施框架正经历从单请求优化到工作流级编排的范式跃迁。单次推理调用的低垂果实（continuous batching、PagedAttention、chunked prefill、P-D 分离）已被 vLLM/SGLang/TRT-LLM 充分采摘 [L2:2603.16104]，剩余优化空间从请求级下沉到算子/chiplet 级（持久化 megakernel、L2 分区感知）[L2:2604.15379]，同时向上抬升到agent 生命周期级（program-aware 调度、workflow DAG 优化）[L2:2602.13692]。训练侧，RLHF 框架从 Co-located 范式走向训练-推理解耦 [L2:2312.11819]，权重传输进一步从通信原语（NCCL/UCX）跃迁至引用级存储抽象——TensorHub 的 Reference-Oriented Storage 在 1024 GPU 上实现 6.7× stall 降低 [L2:2604.09107]；DualPipe 等细粒度 pipeline 并行把大规模 MoE 预训练的通信完全隐藏 [L2:2412.19437]。整个领域的主旋律是：把隐式的硬件/工作负载结构变成显式的一等调度资源。

主线与分支

主线一：RLHF/RL 训练框架——从 Co-located 到 Disaggregated 到 Reference-Oriented 传输

RLHF 训练涉及 Actor/Critic/Ref/Reward 四模型协同，Generation 阶段占 85%+ 时间 [L2:2312.11819]。FlexRLHF 提出 Interleaving 和 Disaggregated 两种放置策略，通过将训练和推理运行时物理分离，在 65B 模型上实现 11× 加速 [L2:2312.11819]。HybridFlow（开源为 veRL）进一步提出 hybrid programming model——inter-model 用 single-controller 灵活编排、intra-model 用 multi-controller 高效执行——配合 3D-HybridEngine 的 zero-redundancy resharding，在 7B–70B 模型上实现 1.53×–20.57× 吞吐提升 [L2:2409.19256]。TensorHub 针对训练完成后权重向数千 rollout worker 分发的带宽瓶颈，提出 Reference-Oriented Storage（ROS）——识别 RL 权重的三个可利用属性（数据并行冗余、推理不可变、版本短暂），直接复用 GPU 上已有权重 in place，引用服务器仅追踪轻量级引用 [L2:2604.09107]。Pipeline replication 利用 RDMA NIC 全双工特性，部分复制完成的 worker 立即成为新数据源，将 N-way fan-out 的总 stall 从二次方降至线性 [L2:2604.09107]。在 1T 模型 1024 GPU 上实现 6.7× stall 降低，跨 DC 场景达 19× 加速，仅需 ~40 行代码集成到 veRL [L2:2604.09107]。TensorHub 与 HybridFlow 形成垂直互补——HybridFlow 解决多模型编排和 3D-HybridEngine 内部的权重 resharding，TensorHub 解决跨 group 的权重同步，两者组合可同时消除 intra-actor resharding 和 inter-group weight sync 两个瓶颈 [L2:2604.09107] [L2:2409.19256]。

三者的共同趋势是将 RLHF 基础设施的优化粒度不断下沉——从"一视同仁"到"按角色差异化放置"（FlexRLHF/HybridFlow），再到"按数据属性差异化传输"（TensorHub）。这一思路已被 OpenRLHF 等后续框架吸收 [L2:2312.11819]。

主线二：大规模预训练基础设施——DualPipe 与 FP8 训练

DeepSeek-V3 的 DualPipe 通过双向 pipeline 调度 + 细粒度算子交错（attention/dispatch/MLP/combine），将 MoE 跨节点 all-to-all 通信完全隐藏在计算间隙中，pipeline bubble 比 1F1B 减少 60%+ [L2:2412.19437]。配合自研 all-to-all kernel（20 SM warp specialization）和首次在 671B 规模验证的 FP8 训练，总成本仅 $5.576M [L2:2412.19437]。代价是 2× 参数内存和高度耦合 H800 拓扑的通信 kernel [L2:2412.19437]，核心框架 HAI-LLM 未开源 [L2:2412.19437]。

主线三：Serving 架构演进——从 Colocation 到 Disaggregation 再到跨 DC

PD 分离的统一化：DynaServe 提出 micro-request 抽象，在任意 token 边界分割请求，将 colocation 和 disaggregation 统一为同一泛化执行空间的特例，配合两级调度实现 1.15×–3.07× serving capacity 提升 [L2:2504.09285]。

存储带宽瓶颈突破：DualPath 发现 agentic 工作负载下 PD 分离架构的 prefill engine 存储 NIC 持续饱和而 decode engine 的 NIC 空闲，通过增加 storage→DE→RDMA→PE 的第二条 KV 加载路径，配合 CNIC-centric 流量隔离和 InfiniBand VL QoS，在 1152 GPU 上实现近线性扩展 [L2:2602.21548]。其核心工程壁垒是"所有 H2D/D2H 绕路走 NIC"这一违反直觉但唯一可行的 PCIe QoS 方案 [L2:2602.21548]。

跨数据中心 PD 分离：PrfaaS 利用混合注意力（KDA:MLA=3:1）将单实例 KV 吞吐从 ~60 Gbps 压到 ~3 Gbps，使跨 DC commodity Ethernet 传输可行，配合长度阈值路由和双时间尺度调度实现 +54% 吞吐 [L2:2604.15039]。其层间 prefill pipelining 是核心工程壁垒 [L2:2604.15039]。

自动配置优化：AIConfigurator 通过算子级性能数据库 + 三种 serving mode 的数学建模，将配置搜索从数天 GPU 实验缩减到 <1 秒 CPU 计算，搜索加速 17 万–43 万倍 [L2:2601.06288]。

主线四：SLO 感知调度——从吞吐优先到 Goodput 优先

多类型 SLO：JITServe 是首个面向 latency/deadline/compound 三类 SLO 的调度系统，用 QRF 上界预测 + pattern-graph 匹配为 compound 请求分配 sub-deadline，通过 GMAX 算法实现 1.4×–6.3× goodput 提升，距 oracle 仅 3–9% [L2:2504.20068]。其理论贡献在于证明 SJF/EDF 的竞争比可任意差，而 GMAX 有常数竞争比 ≈1/8.55 [L2:2504.20068]。

公平调度：Justitia 将网络 WFQ 的 virtual-time 映射到 LLM serving，以 KV token-time（$pd + d^2/2$）作为成本度量，per-app MLP 预测需求后一次性确定优先级，平均 JCT 比 VTC 降低 57.5%，且 92% 应用更早完成 [L2:2510.17015]。其核心壁垒是 KV token-time 这个"量纲选择"——消融显示仅此度量就贡献 42.3% 增益 [L2:2510.17015]。

主线五：Agentic Workflow 调度——从请求级到 Program 级

这是 2025–2026 年增长最快的子领域，24 篇中有 11 篇直接涉及。

工作流 DAG 感知：HexGen-Flow 首次将"SLO 预算沿 workflow DAG 传播"系统化，通过两层调度（全局 workload-balanced dispatch + 本地 urgency 优先队列）在异构 GPU 上实现 1.42–1.56× P95 降低 [L2:2505.05286]。Halo 从数据库批查询优化角度切入，用 epoch-based DP 求解器（比 MILP 快 2300×，质量相同）做全局 placement，request coalescing 是最大贡献模块（W6 +154%）[L2:2509.02121]。Helium 进一步把 agentic workflow 重构为 query plan，通过 Templated Radix Tree + nested-sequence schedule 实现双层前缀复用，比 KVFlow 快 1.56× [L2:2603.16104]。

Orchestrator-Engine 协同设计：Sutradhara 打破 orchestrator 与 engine 的黑箱隔离，通过 5 个 thin API（partial prefill、streaming callback、KV tagging 等）实现 prefill-tool 并行和 streaming tool dispatch，FTR 降低 15% [L2:2601.12967]。Tool 执行占 FTR 延迟 30–80%、变异系数超 100% 这一实证发现 [L2:2601.12967]，直接挑战了"tool call 是轻量 I/O"的传统认知。

Program-aware 调度：ThunderAgent 将 agent workflow 抽象为 Agentic Program 元组，证明最优 pause 策略是 shortest-first（重计算代价 ∝ $c^2$），最优缓存衰减函数在 memoryless 假设下必为指数形式 [L2:2602.13692]。在 SWE-Agent/OpenHands/ToolOrchestra 上实现 1.48–3.58× 吞吐提升 [L2:2602.13692]。

多 LLM 聚合 Pipeline：Scepsy 发现虽然 agentic workflow 的绝对延迟抖动巨大，但每个 LLM 的相对执行时间占比高度稳定（集中度比绝对分布高 4×）[L2:2604.15186]，据此用 Aggregate LLM Pipeline 做 steady-state 预测，驱动 (fraction, TP, replica) 三维联合搜索，比 K8s autoscaler 最多 2.4× 吞吐、27× 延迟提升 [L2:2604.15186]。

主线六：KV Cache 管理——从 LRU 到语义感知

Workflow-aware 驱逐：KVFlow 用 Agent Step Graph 为每个 KV 节点计算 steps-to-execution（STE），以 min 聚合处理共享前缀，配合 proactive prefetching 和 status-aware scheduling，在 agentic workflow 下比 SGLang+HiCache 实现 1.83–2.19× 加速 [L2:2507.07400]。GLM 在 Graph-CoT 场景下设计四级优先级驱逐（共享 prompt > notebook > 历史 notebook > 临时信息），把 token 成本降 95.7%、吞吐升 15.1× [L2:2511.01633]。

Agent-level Admission Control：Concur 发现 middle-phase thrashing 这一新型性能病理——KV 占用率 pinned 在 100% 但命中率塌方、49% 延迟被重计算消耗 [L2:2601.22705]。通过将 KV cache 类比网络带宽、用 AIMD 做 agent-level 准入控制，在 Qwen3-32B 上实现 4.09× 加速 [L2:2601.22705]。其 dual-signal（usage + hit rate）gating 是关键——单用 usage 会在"有用地满"的健康高负载时错误减窗 [L2:2601.22705]。

多 Agent KV 去重：TokenDance 把 multi-agent All-Gather round 作为 KV 复用的一等单位，通过 collective reuse（N 个请求一次 RoPE + 一次 diff）和 Master-Mirror 块稀疏 diff 存储（11–17× 压缩），使同一 GPU 可支撑 2.7× 并发 agent 数 [L2:2604.03143]。

主线七：推理加速与硬件适配

Speculative Decoding 并行化：SSD/Saguaro 将 draft model 部署到独立 GPU，在 verification 期间预计算多个可能 outcome 的 speculation cache，消除 drafting 延迟。geometric fan-out cache topology 在 power-law cache miss 假设下有 closed-form 最优解，平均比最强 SD baseline 快 30% [L2:2603.03251]。

Chiplet 感知运行时：Fleet 在 CUDA/HIP 三层抽象之上补上 Chiplet-task 层（对应 AMD MI350 的 XCD + 4 MB 私有 L2），通过持久化 megakernel + 每 XCD 软件调度器 + 两级事件同步 + M-major 协作式 tile 遍历，在 Qwen3-8B decode 上比 vLLM 快 1.3–1.5× [L2:2604.15379]。其两级同步将每事件 fence 从 248 次压到 8 次（31× reduction）是 persistent megakernel 在 chiplet GPU 上成立的必要条件 [L2:2604.15379]。

MoE 负载均衡：联合优化 load imbalance 和 data movement 的 ILP + 多项式时间启发式算法，实现最高 12.5% 延迟减少和 2× 更频繁的重平衡 [L2:moe-lb-interai25]。

分支：Agent 外部记忆系统

Pancake 为多 agent 长期记忆设计了多层级动态 ANN 索引——三级 cluster cache + FSM 建模访问模式 + hybrid-graph 统一粗索引 + GPU-CPU 协同管理，将 memory-op 开销从占端到端 82% 压缩到 3.2%，端到端吞吐平均提升 4.29× [L2:2602.21477]。

核心 Trade-off 轴

轴一：Colocation vs Disaggregation vs 统一化

Colocation 简单但 prefill-decode 干扰严重（P99 TBT >300ms）[L2:2504.09285]；disaggregation 消除干扰但 GPU 利用率极低（MFU 低至 0.2%）[L2:2504.09285]；DynaServe 通过 micro-request 抽象统一两者 [L2:2504.09285]。DualPath 进一步发现 disaggregated 架构下存储带宽成为新瓶颈（prefill engine SNIC 饱和、decode engine SNIC 空闲），双路径加载聚合全部 SNIC 带宽 [L2:2602.21548]。PrfaaS 将分离推到跨 DC 尺度，但要求混合注意力架构将 KV 吞吐降 4–13× [L2:2604.15039]。

量化 trade-off：DynaServe 在 Azure Code 负载下 serving capacity 比 colocation 高 3×、比 disaggregation 高 1.37× [L2:2504.09285]。DualPath 在 DS 660B 上逼近零 I/O 的 Oracle 上界 [L2:2602.21548]。

轴二：短期公平 vs 长期公平（效率 vs 公平性）

VTC 追求时时刻刻等分资源（短期公平），但让本可更快完成的应用被无谓拖慢 [L2:2510.17015]。Justitia 用 virtual-time WFQ 实现长期公平——GPS 完成顺序近似 SJF 效率，同时保证最坏延迟界 $\le 2c_{\max}+C_{\max}/M$ [L2:2510.17015]。SRJF 效率最高但 elephant 应用在 mice 不断到来时线性饥饿 [L2:2510.17015]。

轴三：Reactive 驱逐 vs Proactive 准入控制

LRU 在 agentic workflow 下产生反向信号——"最近最少使用"的恰好是即将被复用的 [L2:2507.07400] [L2:2601.22705]。CPU offloading（HiCache）在高并发下因 PCIe 拥塞反而比重算更慢 [L2:2601.22705]。Request-level admission control 因不了解 agent 累积 KV 足迹而可能比无控制更差（某配置 0.72×）[L2:2601.22705]。Agent-level AIMD 准入是当前最有效方案 [L2:2601.22705]，但依赖 hit rate 信号在有共享前缀的工作负载下可靠 [L2:2601.22705]。

轴四：持久化 Megakernel vs Kernel-per-op

Fleet 的跨算子持久化复用在 memory-bound 区（bs ≤ 32）通过 L2 协作式复用获得显著收益（L2 hit 38.9% → 51.0%）[L2:2604.15379]，但 register pressure 导致 1 wave/SIMD，在 compute-bound 区（bs ≥ 64）反而比 hipBLASLt 慢 [L2:2604.15379]。这是结构性 trade-off：持久化方向必须接受寄存器预算取并集，单 kernel 方向必须接受每次 launch 刷新 L2 [L2:2604.15379]。

轴五：Push-then-Pull vs Reference-Oriented 权重传输

传统方案在 RL 权重传输中形成两极：distributed storage（Ray Plasma、parameter server）先接管数据所有权再提供读取——push-then-pull 模式将网络流量翻倍，Ray Plasma 40 GB 传输需 32s vs GPU-direct RDMA ~0.2s（~160× 差距）[L2:2604.09107]；collective communication（NCCL）避免中间存储但要求静态通信组和全局 barrier，任何节点故障导致全集群崩溃 [L2:2604.09107]。TensorHub 的 ROS 在两者间找到第三条路——无额外存储、无全局 barrier、保留拓扑感知，代价是要求权重满足"冗余、不可变、短暂"三属性 [L2:2604.09107]。

量化 trade-off：TensorHub RDMA 吞吐 22 GB/s（88% 理论值）vs NCCL 18.8 GB/s vs UCX 18.1 GB/s；1T 模型下总 GPU stall NCCL 5.3s × 1024 GPU vs TensorHub 3.1s × 256 GPU，6.7× 降低 [L2:2604.09107]。但 ROS 三属性假设在 expert parallelism（每 GPU 持有不同 shard，非等价副本）和异步 RL（trainer 可能在 rollout 使用旧权重时修改，非不可变）下部分失效——论文所有评估均在标准同步 PPO 下进行 [L2:2604.09107]。

贡献分离之难：同等规模（1T 参数）权重传输的延迟分解（基于 fabric-lib 数据）显示 FSDP unsharding 占 42%、跨 rank 同步占 29%、RDMA submit 仅占 2.1% [L3:2604.09107]。若该瓶颈分布在 TensorHub 场景中成立，pipeline replication 优化的可能是全链路中最小的环节——端到端 6.7× 优势可能主要来自消除 NCCL 全局 barrier 和 ROS publish 零开销。但 TensorHub 使用 Mooncake RDMA Direct 模式（长期 tensor 直接注册到 RNIC）绕过了 FSDP unsharding 路径 [L2:2604.09107]，使上述分解不能直接套用；而论文本身缺乏同等粒度的 breakdown，无法隔离 pipeline replication、消除全局 barrier、ROS publish 零开销三者的独立贡献。缺失的关键中间 baseline 是"Mooncake P2P direct"（不经 ROS 抽象，直接用 RDMA Write 做点对点传输）[L3:2604.09107]。

冲突与争议

争议一：PD 分离在 agent 场景是否有效？

ThunderAgent 实验发现 PD 分离在 agent 工作负载下退化——单侧 HBM 池变小，在更低并发就触发 thrashing [L2:2602.13692]。但 DualPath 和 PrfaaS 在 PD 分离基础上通过双路径加载和跨 DC offload 获得显著收益 [L2:2602.21548] [L2:2604.15039]。矛盾的根源在于：agent 场景的 KV cache 命中率极高（98.7%），prefill 从 compute-intensive 退化为 I/O-intensive [L2:2602.21548]，此时 PD 分离的价值取决于存储带宽是否足够（DualPath 解决）而非计算分离本身（ThunderAgent 观察到的问题）。

争议二：静态 DAG 假设是否足够？

Halo [L2:2509.02121]、Helium [L2:2603.16104]、KVFlow [L2:2507.07400] 都假设 workflow 拓扑静态可知。但 ReAct/Reflexion 等动态 agent 在运行时决定下一步，Agent Step Graph 无法在请求时刻给出完整拓扑 [L2:2507.07400]。ThunderAgent 和 Concur 选择不依赖 DAG 结构，而是在 agent 生命周期边界做调度 [L2:2602.13692] [L2:2601.22705]，但丧失了全局优化的能力。Scepsy 的 Aggregate Pipeline 用统计聚合绕过了这个问题 [L2:2604.15186]，但前提是"相对份额稳定"这一经验假设 [L2:2604.15186]。

争议三：KV cache hit rate 是否是好指标？

HiCache 达到 97% hit rate 但比 SGLang 慢 3× [L2:2601.22705]；Continuum 的 TTL pin 策略 hit rate 更高但 caching cost 被放大、吞吐反而低于 ThunderAgent [L2:2602.13692]；Helium 观察到 vLLM 与其 hit rate 相当（42.9%）但慢 38× [L2:2603.16104]。结论是：高 hit rate 是必要非充分条件，必须同时考虑 hit 的成本（offload 延迟）、miss 的成本（重计算）和调度效率 [L2:2602.13692]。

争议四：同步权重传输优化 vs 异步 RL 的 staleness tolerance

TensorHub 在同步 PPO 下极致优化权重传输——所有 rollout 拉取同一版本，pipeline replication 实现 6.7× stall 降低 [L2:2604.09107]。但异步 RL 框架（IMPALA、StreamRL、AReaL）选择完全不同的技术路线——允许 rollout 使用稍旧版本权重，通过 importance sampling ratio 校正来彻底绕开传输瓶颈。TensorHub 未对比任何异步 RL 框架作为 baseline [L2:2604.09107]。两条路线的根本分歧在于：同步方案追求传输效率的极致（ROS + pipeline replication），异步方案追求对传输延迟的完全容忍（staleness + off-policy correction）。在超大规模（>10K GPU）和跨 DC 场景下哪条路线胜出——或二者是否可以融合（staleness-aware 的 ROS 调度）——尚无直接对比实验。

未探索角落

  1. 动态 workflow 的在线 graph 更新：所有 workflow-aware 系统都假设拓扑在请求时刻已知 [L2:2507.07400] [L2:2509.02121]，ReAct/Reflexion 类动态 agent 的增量 graph 提示机制尚未被探索。
    1. 多模型异构 agent 的 GPU 分配：Scepsy 开创了多 LLM pipeline 建模 [L2:2604.15186]，但仅在 2 workflow × 16 A6000 上验证，H100/NVSwitch 大规模集群和 MoE 模型的场景完全空白。
      1. Chiplet-task 与 MoE 的结合：Fleet 的 M-major 协作假设在 MoE 下失效（不同 token 走不同 expert），但 Expert ↔ XCD 映射是天然的 chiplet-affinity 机会 [L2:2604.15379]。
        1. 跨层 KV 压缩与 agent serving 的组合：TokenDance 的 Master-Mirror 去重 [L2:2604.03143] 与 KIVI/H2O 等 per-cache 量化正交，但叠加后的 compounding gain 未被量化。
          1. Agent serving 的多租户公平性：Justitia 在 request 级证明了 WFQ 可行 [L2:2510.17015]，但在 agent-level 准入控制（Concur/ThunderAgent）下的跨租户隔离完全缺失 [L2:2601.22705] [L2:2602.13692]。
            1. 长 tool 延迟（分钟级）的处理：Sutradhara 的 tool 延迟变异系数 >100% [L2:2601.12967]，ThunderAgent 的指数衰减假设 memoryless [L2:2602.13692]——但真实远程 API 呈重尾分布，最优策略可能需要 Pareto-aware 修正。
              1. PrfaaS 的层间 pipelining 在开源框架中的复现：当前 vLLM/SGLang 的 PD 传输仍是"全部 prefill 完再发"[L2:2604.15039]，层间 streaming 是将 PrfaaS 的 +54% 收益兑现到开源生态的关键缺失。
                1. ROS 抽象扩展到 KV-Cache 传输：TensorHub 的 publish/replicate/unpublish 三原语在 KV-Cache 场景下部分成立——prefix cache 在多请求间高度冗余（DualPath 的 98.7% 命中率证实 [L2:2602.21548]），KV-Cache 在 decode 阶段不可变且版本短暂。若在 PD 分离架构中引入 ROS 消除中间存储层，pipeline replication 可使 PE→DE 传输中 DE-1 的已接收前缀立即服务 DE-2，但叠加效果未被量化 [L2:2604.09107] [L2:2602.21548]。关键挑战：KV-Cache 比权重大得多（64K context 下达数百 GB）且更新频率更高（每数十 token 写一次），ROS 的 mutability contract 需要适配 append-only 语义 [L3:2604.09107]。
                  1. Pipeline replication 在非 InfiniBand 硬件上的可移植性：TensorHub 的 pipeline replication 依赖 RDMA Read 单边语义和 NIC 全双工特性 [L2:2604.09107]。在 AWS EFA 等仅支持 Send/Recv 和 WriteImm 的硬件上，需要将 receiver-initiated 拉取重构为 sender-initiated 推送——核心逻辑（"部分完成的节点作为新数据源"）理论上可在 Write 语义上实现，但打破 InfiniBand 锁定的可移植方案尚未被验证。fabric-lib 的 TransferEngine 提供跨 ConnectX-7 和 EFA 的统一 RDMA 抽象，其 WriteImm + ImmCounter 机制天然适配 pipeline progress tracking，是实现可移植 TensorHub 的潜在基础层 [L3:2604.09107]。
                    1. 跨训练-推理统一的权重传输层：TensorHub 解决 RL trainer→rollout 权重传输 [L2:2604.09107]，DualPath 解决推理端 PE→DE 的 KV 传输 [L2:2602.21548]，两者本质上是同一"大规模数据搬运"问题的两个实例。LLM 生命周期中权重在 SFT→RL→rollout→serving→edge 多个系统间流动，每个边界都有独立传输机制，一个统一的"权重总线"抽象——以 ROS 的 publish/replicate 为语义核心——可消除重复数据搬运，但 serving 端权重"长期驻留"的特性需要扩展 ROS 的版本管理以支持零停机热更新 [L2:2604.09107]。更进一步，分层 ROS（GPU HBM → DRAM → SSD）可将 ZeRO-Offload 的 optimizer state 和 checkpoint 纳入同一引用语义，replicate 时自动选择最快可用源 [L3:2604.09107]。
                      1. TensorHub + ThunderAgent 联合优化 RL 循环：Pipeline replication 使不同 rollout worker 在不同时间完成权重更新——worker A 完成 replicate 可立即开始推理，worker B 仍在等待。若 ThunderAgent 的 global waiting queue 能订阅 TensorHub 的 progress counter 事件，可实现"权重到达即推理"的细粒度流水线——无需等待全部 worker 就绪，将 pipeline replication 的"增量就绪"特性暴露给调度层 [L3:2604.09107] [L2:2602.13692]。

                        2. 时代定位

                        Framework 类别正从"优化单次 forward pass"的成熟期进入"优化 agent 生命周期"的爆发期。三个代际标志：

                        • 2023–2024：PagedAttention（空间）→ Continuous Batching（时间）→ Chunked Prefill（交织）[L2:2603.16104]
                        • 2025：PD Disaggregation + Prefix Caching + SLO-aware Scheduling [L2:2504.09285] [L2:2504.20068]
                        • 2026：Agent-native Scheduling（program/round/workflow 级）+ Chiplet-aware Runtime + Cross-DC PD + Reference-Oriented Weight Transfer [L2:2602.13692] [L2:2604.15379] [L2:2604.15039] [L2:2604.09107]

                        与 adjacent categories 的交互：algorithm 类的 RLHF/GRPO 直接依赖 framework 类的 rollout 吞吐——TensorHub 的权重传输效率直接决定 RL 迭代速度 [L2:2604.09107] [L2:2602.13692]；cluster 类的 RDMA 通信原语（Mooncake、fabric-lib）是 TensorHub 等系统的底层数据平面 [L2:2604.09107]；kernel 类的 FlashAttention/FP8 GEMM 是 framework 的底层算力来源 [L2:2412.19437]；hardware 类的 chiplet 架构（MI350、Blackwell）正在迫使 framework 补上 L2 scope 这一缺失抽象 [L2:2604.15379]；agent 类的 LangGraph/AutoGen 等编排框架是 framework 优化的上层消费者，二者正从黑箱 API 走向 co-design [L2:2601.12967]。
