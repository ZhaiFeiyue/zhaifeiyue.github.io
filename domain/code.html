Code 领域全景：AI Agent 框架的架构分化与融合

现状快照

2026 年上半年，AI Agent 开源框架从「能跑」进入「能用」阶段，两条路线正在分化：以 Hermes Agent 为代表的「自治型 Agent」追求跨会话自我改进与平台无关部署 [L2:NousResearch-hermes-agent]，以 OpenAI Agents SDK 为代表的「编排型框架」则聚焦声明式多 Agent 协作、安全护栏和可中断恢复 [L2:openai-openai-agents-python]。两者都将 MCP 协议视为工具集成的一等公民，但在自治程度、架构粒度和 Provider 绑定策略上做出了截然不同的选择。

主线与分支

分支一：自我改进闭环——从「一次性执行器」到「持续进化助手」

Hermes Agent 的核心创新在于闭环学习架构：Agent 完成复杂任务后自动创建 Skill（YAML+Markdown），后续使用中发现不足会自动 patch 改进；MEMORY.md 和 USER.md 实现跨会话持久记忆与用户建模 [L2:NousResearch-hermes-agent]。这将 Agent 从「工具」变为「随时间变好的助理」，其价值在于经验的累积而非单次执行能力 [L2:NousResearch-hermes-agent]。

技能系统兼容 agentskills.io 开放标准，且项目同时包含 Atropos RL 环境和轨迹生成管线（batch_runner.py + trajectory_compressor.py），为训练下一代工具调用模型提供了完整的数据闭环 [L2:NousResearch-hermes-agent]。

分支二：声明式多 Agent 编排——从「手工编排」到「框架自动调度」

OpenAI Agents SDK 走的是另一条路：用声明式 @dataclass 定义 Agent 的指令、工具、护栏和委托关系，由 Runner 自动执行 turn loop [L2:openai-openai-agents-python]。Handoff 机制允许 Agent 间委托并继承完整对话历史，与 Agent-as-Tool（子任务调用、原 Agent 保持控制权）形成互补 [L2:openai-openai-agents-python]。

Guardrails（输入/输出护栏）与模型执行并行运行，不增加额外延迟；RunState 可序列化支持 human-in-the-loop 中断恢复 [L2:openai-openai-agents-python]。v0.14 新增的 Sandbox 系统支持 Docker/E2B/Modal 等 7+ 后端，为 Agent 提供隔离执行环境 [L2:openai-openai-agents-python]。

分支三：工具系统与 MCP 集成——两种不同的工具哲学

两个项目在工具系统设计上呈现明显差异。Hermes Agent 内置 40+ 工具，采用双路径执行：registry.dispatch 处理普通工具，_invoke_tool 处理需要 Agent 内部状态的有状态工具（todo/memory/session_search/delegate），避免了循环依赖但增加了维护负担 [L2:NousResearch-hermes-agent]。在 MCP 层面，Hermes 实现了双向集成——既是 MCP 客户端也是 MCP 服务端（mcp_serve.py 暴露给 Cursor/Claude Code）[L2:NousResearch-hermes-agent]。

OpenAI Agents SDK 的工具系统更结构化：Tool 是 Union 类型，包含 FunctionTool、WebSearchTool、CodeInterpreterTool、HostedMCPTool 等 12 种子类型 [L2:openai-openai-agents-python]。@function_tool 装饰器自动提取参数类型生成 JSON Schema，支持审批、动态启用和超时控制。MCP 作为一等公民原生集成，每次运行时自动发现并注册 MCP 工具 [L2:openai-openai-agents-python]。

分支四：多平台接入 vs 沙箱隔离——部署模型的两极

Hermes Agent 通过 Gateway 系统 + BasePlatformAdapter 抽象支持 Telegram/Discord/Slack/WhatsApp/飞书/钉钉等 20+ 平台，统一消息管道与授权体系 [L2:NousResearch-hermes-agent]。从 5 美元 VPS 到 GPU 集群均可运行，CLI 和 Gateway 共享同一个 AIAgent 实例。

OpenAI Agents SDK 没有内置多平台网关，但 v0.14 引入的 Sandbox 系统提供了另一个维度的部署能力：SandboxAgent 可以在 Docker、E2B、Modal、Cloudflare 等容器中执行，通过 Manifest 声明 Git/File/S3 挂载，配合 Shell/Filesystem/Patch 等 Capabilities 实现隔离工作空间 [L2:openai-openai-agents-python]。

核心 trade-off 轴

轴一：自治性 vs 可控性

Hermes Agent 追求高度自治——技能自动创建与改进、记忆自动积累、子代理自动委托 [L2:NousResearch-hermes-agent]。这带来了「随时间变好」的体验，但也意味着行为不完全可预测。

OpenAI Agents SDK 偏向可控性——Guardrails 可以在输入/输出层面阻断不安全行为（tripwire 触发即停），human-in-the-loop 可以在工具调用前暂停等待人工审批，RunState 序列化使得任何步骤都可以中断和恢复 [L2:openai-openai-agents-python]。

轴二：单体内聚 vs 模块化解耦

Hermes Agent 的核心 run_agent.py 达到 ~10,500 行，传输层、业务编排、错误恢复耦合在一起。好处是控制紧密、边界情况处理完整（错误分类→credential pool 轮换→fallback provider→jittered backoff→stale stream 检测→可中断重试），代价是可维护性挑战 [L2:NousResearch-hermes-agent]。

OpenAI Agents SDK 采用模块化设计，87K LOC 分布在 agent.py（941 行）、run.py（1859 行）、run_state.py（3304 行）、tool.py（1938 行）等独立模块，测试代码约 158K LOC（~1.8x 源码），配合 mypy strict + pyright 类型检查 [L2:openai-openai-agents-python]。但 run_state.py 的 3304 行状态序列化逻辑和 extensions 目录膨胀（7 种 sandbox provider + 5 种 session backend）也显示了复杂度的转移而非消除 [L2:openai-openai-agents-python]。

轴三：模型无关 vs Provider-First

Hermes Agent 从设计之初就追求模型无关，支持 200+ 模型，通过 api_mode 分支适配 OpenAI/Anthropic/Codex 三种 API 模式 [L2:NousResearch-hermes-agent]。代价是多 Provider 适配逻辑内置于 Agent 循环，导致核心文件膨胀。

OpenAI Agents SDK 采用 OpenAI-First 策略，内部数据格式统一为 OpenAI Responses API 格式，其他 LLM 通过 chatcmpl_converter.py 转换，LiteLLM/any-llm 作为 extensions 支持 100+ 模型 [L2:openai-openai-agents-python]。这使得 OpenAI 用户获得最佳体验，但非 OpenAI 场景存在来回格式转换的开销。

冲突与争议

数据结构选型：Dict vs Dataclass

两个项目在核心数据结构选型上走向了两极。Hermes Agent 大量使用 Dict 传递核心数据（messages、result、api_kwargs），无类型检查，字段名拼写错误不会报错 [L2:NousResearch-hermes-agent]。OpenAI Agents SDK 则以 @dataclass 为基础，Agent/RunState/RunResult 都有明确类型定义，支持 Generic[TContext] 泛型 [L2:openai-openai-agents-python]。在代码规模 ~400K 行的项目中，Dict 的灵活性可能已经变成了维护负担；但 OpenAI SDK 也没有选择 Pydantic，而是用更轻量的 dataclass + 手动 __post_init__ 验证 [L2:openai-openai-agents-python]。

工具执行路径的统一性

Hermes 的工具执行存在两条并行路径（registry vs agent 内联），逻辑有重复 [L2:NousResearch-hermes-agent]。OpenAI SDK 虽然工具类型更多（12 种 Tool 子类型），但通过统一的 tool_execution.py + asyncio.gather() 并行执行 [L2:openai-openai-agents-python]。两者都未完全解决有状态工具与无状态工具的边界问题。

未探索角落

  1. 跨框架 Agent 互操作：Hermes 的 MCP 双向集成和 ACP 适配器 [L2:NousResearch-hermes-agent] 与 OpenAI SDK 的 MCP 一等公民支持 [L2:openai-openai-agents-python] 暗示了 Agent 框架间互操作的可能，但目前没有项目真正实现跨框架 Agent 委托。
    1. 自我改进 + 安全护栏的融合：Hermes 有自我改进但缺少形式化护栏，OpenAI SDK 有护栏但缺少学习闭环。将两者融合——Agent 可以自主学习但受 Guardrails 约束——是一个明显的空白。
      1. Agent 行为的可观测性与可解释性：OpenAI SDK 内建 Tracing 系统（Trace → Span 层级结构）[L2:openai-openai-agents-python]，Hermes 有组件级日志但缺乏结构化追踪 [L2:NousResearch-hermes-agent]。Agent 决策链的可解释性（为什么选择这个工具、为什么委托给那个子 Agent）仍是开放问题。
        1. RL 训练闭环的规模化：Hermes 的 Atropos RL 环境和轨迹压缩管线 [L2:NousResearch-hermes-agent] 提供了概念验证，但从单 Agent 轨迹到大规模多 Agent 协作场景的 RL 训练数据生成尚无成熟方案。

          2. 时代定位

          Code 领域正处于 AI Agent 框架的架构分化期。2025 年的 Agent 框架主要解决「如何调用 LLM + 工具」的基本问题；2026 年的竞争焦点已转向更高层的系统性问题——持久化、学习、安全、多 Agent 协作、跨平台部署。

          与相邻领域的关系：

          • Agent 领域：code 领域的 Agent 框架是 agent 领域研究的工程载体，Hermes 的技能闭环和 OpenAI SDK 的编排模式都在将 agent 论文中的概念（self-improvement、multi-agent orchestration）落地为可运行代码。
          • Framework 领域：这些 Agent 框架建立在推理框架（OpenAI API、Anthropic API）之上，但正在成为独立的「Agent 运行时」层，类似于 Web 框架之于 HTTP 库的关系。

          下一个范式转折可能来自：Agent 框架与 RL 训练管线的深度融合——Agent 不仅在运行时自我改进（如 Hermes 的技能 patch），还能通过 RL 从轨迹数据中训练更好的底层模型。
