Agent 类别纵览

基于 4 篇 L2 蒸馏 (2511.00739, 2603.18897, 2604.11378, 2604.14228) 综合，2026-04-22

1. 现状快照

2025–2026 年 LLM agent 正处于从"Agent Loop 野蛮生长"到"结构化运行时沉淀"的拐点。生产级系统以 Claude Code 为代表，采用极简 while-loop + 庞大确定性基础设施的范式（1.6% 决策逻辑 vs 98.4% harness）[L2:2604.14228]。学术界则从系统性能和形式化两端同时推进：一方面揭示工具执行占端到端延迟高达 35%–88% 的 CPU 侧瓶颈 [L2:2511.00739]，另一方面尝试用 scheduler 理论统一描述 Agent Loop 与 graph executor 的连续谱 [L2:2604.11378]。投机性工具执行（speculative tool execution）作为新兴优化范式初露锋芒，将端到端延迟降低 48.5% [L2:2603.18897]。

2. 主线与分支

2.1 Agent 执行瓶颈：CPU 侧的被忽视战场

当 GPU 不断变强（RTX-6000 Pro → H200），agentic workload 的端到端瓶颈正从 GPU 推理转向 CPU 工具执行。RAG 检索（ENNS/FAISS）占 E2E 延迟 82%–89%，ChemCrow 分子生成占 85%–88%，SWE-Agent 的 Bash 执行在高端 GPU 系统上占比从 38% 飙升到 65% [L2:2511.00739]。CPU 并行化效率远低于 GPU：BS=64→128 时 H200 推理延迟仅增 1.06×，而 LexRank 摘要延迟增 2.0× [L2:2511.00739]。这一不对称性催生了两类调度优化：

  • COMB（微批重叠）：将大批次拆为 $B_{cap} \approx 1\text{-}2 \times \#\text{CPUs}$ 的微批并重叠 CPU/GPU 阶段，开放循环下服务延迟降低 3.9×（P50）[L2:2511.00739]。
  • MAS（混合调度）：为 CPU-heavy 和 GPU-heavy 请求设置独立弹性队列，保护少数请求类型的尾延迟（P50/P90 改善 2.37×/2.49×）[L2:2511.00739]。

领军实体：2511.00739（首次系统刻画）。PASTE [L2:2603.18897] 的 profiling 进一步印证了 tool 执行占总时间 35%–61% 的结论。

2.2 投机性工具执行：从被动等待到主动预测

Agent 的 "LLM→Tool→LLM→Tool" 串行循环中，tool 等待是延迟的主要来源。PASTE 提出 Pattern Tuple $(C, T, f, p)$ 抽象，将 tool 调用的控制流与数据流解耦 [L2:2603.18897]：

  • 控制流模式：coding agent 中 55% 的 file_editor 后紧跟 pytest，research agent 中 51% 的 search 后跟 web_fetch [L2:2603.18897]。
  • 数据流推导：95% 的 URL 参数可从前序 search 结果中符号推导，无需等待 LLM 生成 [L2:2603.18897]。
  • 风险感知调度：贪心优先级 $U(j) = p \cdot T / (c \cdot d)$ 在 slack 资源内最大化期望延迟收益 [L2:2603.18897]。

最终效果：E2E 延迟降低 48.5%，tool stall 减少 67%，overlap 提升 10×，且 97% 请求获得正加速 [L2:2603.18897]。资源开销极低（每减少 1s 延迟仅 0.02 core-sec CPU + 2.6 MB 内存）[L2:2603.18897]。

领军实体：2603.18897（PASTE）。与 2511.00739 的 COMB CPU-GPU 重叠思想在设计空间上有交集但攻击层次不同——COMB 在 serving 批调度层，PASTE 在单 agent session 的 tool dispatch 层。

2.3 Agent 运行时形式化：从 Loop 到 Scheduler 连续谱

2604.11378 用 scheduler 五元组 $\mathcal{E}=(\mathcal{S},\mathcal{U},\mathcal{P},\mathcal{O},\Delta)$ 统一描述所有 agent 执行系统，核心观察是 Agent Loop 本质是 $|\mathcal{U}| \leq 1$ 的 single-ready-unit scheduler [L2:2604.11378]。基于此提出 Structured Graph Harness（SGH）：

  • 静态 DAG + plan versioning：plan 在单个 version 内不可变，planning/execution/recovery 三层严格分离 [L2:2604.11378]。
  • 三级 recovery 协议：local_retry → local_patch → request_replan，跳级在 API 边界被直接拒绝 [L2:2604.11378]。
  • 有界终止证明：基于 10 状态节点状态机，证明 $\sum_v \tau_v(b_v+1) + \sum T_{\text{human}}$ 为时间上界 [L2:2604.11378]。

领军实体：2604.11378（SGH position paper）。但注意这是纯理论工作，无 prototype 无实验数据 [L2:2604.11378]。

2.4 生产级 Agent 架构剖析：Minimal Scaffolding + Maximal Harness

2604.14228 首次基于源码（Claude Code v2.1.88, ~512K 行 TypeScript）逆向工程分析生产级 coding agent，识别出 5 values → 13 design principles → implementation choices 的完整映射 [L2:2604.14228]。关键子系统包括：

  • 安全：7 层独立安全管道（tool pre-filtering → deny-first → permission modes → auto-mode classifier → shell sandbox → no-restore-on-resume → hook interception）[L2:2604.14228]。
  • 上下文管理：5 层渐进式压缩管道（budget reduction → snip → microcompact → context collapse → auto-compact）[L2:2604.14228]。
  • 扩展性：4 种机制按 context cost 分层（hooks 零成本 → skills 低 → plugins 中 → MCP 高）[L2:2604.14228]。
  • 委派：subagent 采用 summary-only return + worktree isolation，团队模式 token 消耗约 7× [L2:2604.14228]。

93% 的 permission prompt 被批准这一数据揭示了 approval fatigue 的系统性风险——安全机制必须独立于人类警觉性 [L2:2604.14228]。

领军实体：2604.14228（Claude Code 解剖）。

3. 核心 Trade-off 轴

轴 1：表达力 vs 可控性（Agent Loop ↔ Static DAG）

Agent Loop（$|\mathcal{U}|=1$, non-det）拥有最大表达力——LLM 可在运行时自由决定下一步，但代价是步骤依赖隐式、recovery 无边界、audit trail 不可构造 [L2:2604.11378]。SGH（$|\mathcal{U}| \geq 1$, det）通过静态 DAG 获得结构性并行和有界终止，但放弃了竞争性并行、递归子图展开和动态拓扑修改 [L2:2604.11378]。Claude Code 选择了 Agent Loop + maximal harness 的中间路线——不约束模型推理，但用确定性基础设施创造可信操作环境 [L2:2604.14228]。

量化参照：SGH 作者自估 60%–70% 任务实际是线性链，此时 graph gain 仅剩 bounded recovery + 审计 [L2:2604.11378]；而 Claude Code 的 values-over-rules 假设模型有 "good judgment"，直接依赖 Claude 的 alignment 质量 [L2:2604.14228]。

轴 2：GPU 优化 vs CPU-GPU 协同优化

传统 agentic serving 优化集中在 GPU kernel 和 KV-cache，但当 CPU 工具执行占 E2E 延迟 88% 时，GPU kernel 再快也只能改善剩余 12% [L2:2511.00739]。COMB/MAS 证明 CPU-aware 调度可带来 2–4× 的延迟改善 [L2:2511.00739]。PASTE 从另一角度攻击同一问题——通过投机执行将 tool 工作与 LLM 生成重叠，减少 67% 的 tool stall [L2:2603.18897]。

量化对比：COMB 在 serving 层面降低 3.9× 服务延迟（P50），PASTE 在 session 层面降低 48.5% E2E 延迟。两者可叠加 [L2:2511.00739] [L2:2603.18897]。

轴 3：Scaffolding 深度 vs 模型能力依赖

Claude Code 的 "minimal scaffolding" 范式（1.6% 决策逻辑）赌的是模型能力足够强，不需要显式规划图 [L2:2604.14228]。SGH 则赌 planner 能生成高质量 DAG，用结构化执行补偿模型不足 [L2:2604.11378]。这本质是一个随模型能力曲线移动的 trade-off：模型越强，scaffolding 的边际收益越低；模型越弱，越需要显式结构约束。

4. 冲突与争议

争议 1：Agent Loop 是否"够用"？

2604.11378 论证 Agent Loop 有三个结构性缺陷（隐式依赖、无界 recovery、不可审计），认为只有显式 DAG 才能根本解决 [L2:2604.11378]。但 2604.14228 展示的 Claude Code 恰恰是一个 Agent Loop 系统，通过 7 层安全管道 + 5 层 context compaction + append-only JSONL transcript 实现了高度的 production readiness [L2:2604.14228]。两者的分歧根源在于对 LLM 判断力的信任程度：SGH 假设 LLM 决策不可检查（non-det $\mathcal{P}$），Claude Code 假设模型有 "good judgment" 并用 harness 兜底。

当前状态：未解决。Claude Code 的生产成功（27% 任务是质变新工作）[L2:2604.14228] 支持 Agent Loop 的可行性，但缺乏可审计性的系统性数据。SGH 的理论优势尚无实验验证。

争议 2：CPU 瓶颈是持久性的还是过渡性的？

2511.00739 的核心前提是 CPU 工具执行主导 E2E 延迟 [L2:2511.00739]。但这一结论强绑定于当前工具的实现语言（Python MP/MT 受 GIL 限制）和部署形式（CPU-based FAISS flat search）。如果工具本身被 GPU 加速（GPU FAISS、GPU 文本处理）或被 LLM 替代（直接回答而非 RAG 检索），CPU 瓶颈可能消失 [L2:2511.00739]。PASTE 的 profiling 在不同 agent 和 benchmark 上复现了 35%–61% 的 tool 延迟占比 [L2:2603.18897]，部分交叉验证了这一结论的短期有效性。

争议 3：投机执行的模式稳定性

PASTE 假设 agent tool 调用存在稳定的控制流模式（55% edit→test, 51% search→fetch）[L2:2603.18897]，但这些统计来自 SWE-bench/MetaGPT/OpenHands 等结构化 benchmark。真实生产环境中用户行为多样性可能更大，custom tool 更多，模式更碎片化 [L2:2603.18897]。此外，Top-1 预测准确率仅 27.8%，PASTE 依赖多候选投机 + 93.8% overall hit rate 来补偿 [L2:2603.18897]。

5. 未探索角落

  • Multi-turn agent 的 KV-cache 状态管理与跨 turn 工具调度：2511.00739 明确未涉及 [L2:2511.00739]，PASTE 也仅在单 session 内优化 [L2:2603.18897]。跨 turn 的 KV-cache 复用（如 Sutradhara 的方向）与 CPU 工具调度的联合优化是空白区域。
  • SGH 的实验验证：2604.11378 提出了完整的 7 组对照实验 protocol (G0–G6)，但未执行 [L2:2604.11378]。$G_{\text{graph}}$（结构性并行带来的 wall-clock 收益）在真实 workload 上的量化仍是未知数。
  • COMB/MAS 的动态自适应：$B_{cap}$ 和 $E_{cap}$ 均为静态配置 [L2:2511.00739]，缺乏生产环境中 workload 组成动态变化时的 autoscaling 机制。
  • Agent 安全的系统性量化：Claude Code 报告 93% approval rate 暗示 approval fatigue [L2:2604.14228]，但缺乏对真实安全事件率的定量分析。已知有 2 个 pre-trust initialization ordering CVE [L2:2604.14228]，系统性安全审计尚未公开。
  • 弱模型上的 agent 架构：Claude Code 的 minimal scaffolding 范式 [L2:2604.14228] 和 SGH 的 planner 质量假设 [L2:2604.11378] 都未在 ≤8B 开源模型上验证。2511.00739 使用 ≤32B SLM 但未验证 70B+ 大模型 [L2:2511.00739]。模型能力曲线与最优架构选择的映射关系未被系统研究。
  • Cross-agent pattern 迁移：PASTE 的 pattern mining 冷启动问题——新 agent 或新 tool 集合缺乏历史 trace 时无模式可用 [L2:2603.18897]。跨 agent 的迁移学习未被探索。

6. 时代定位

Agent 类别在 2025–2026 年处于从"概念验证"到"生产基础设施"的关键过渡期。三个方向性力量正在塑造这个领域：

  1. GPU 强化加速 CPU 瓶颈暴露：H200/B200 时代的 GPU 越来越强，将瓶颈从 GPU 推理快速转移至 CPU 工具执行 [L2:2511.00739]。这要求 serving 框架从 "GPU throughput 最大化" 转向 "CPU-GPU balanced utilization" [L2:2511.00739]。
  2. Agent 运行时从 Loop 向 Graph 演进：LangGraph/AutoGen/CrewAI/TDP 都在往 graph 结构靠，但各解决局部问题 [L2:2604.11378]。SGH 的 scheduler 连续谱理论提供了统一坐标系。Claude Code 的成功则证明 Agent Loop + maximal harness 在当前模型能力下仍是可行的生产选择 [L2:2604.14228]。
  3. 投机执行范式崛起：PASTE 将 speculative decoding 的理念从 token 生成层扩展到 tool 执行层 [L2:2603.18897]，与 COMB 的 CPU-GPU 重叠思想形成互补 [L2:2511.00739]。

    4. 与邻近类别的交叉：agent 类别与 framework（serving 框架需要 CPU-aware 调度 + tool-calling middleware）、kernel（CPU 侧 FAISS/文本处理 kernel 优化）、model（SLM + 工具 vs 大模型的选型）紧密耦合。Agent 正在成为驱动整个 AI 基础设施栈重构的核心应用形态。
